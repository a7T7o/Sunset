### 3.1 `DialogueManager` 已具备乱码/解码文本路由
文件：`Assets/YYY_Scripts/Story/Managers/DialogueManager.cs`

关键事实：

- 有公开状态：

- 文本路由逻辑已存在：

  - 否则显示 `text`

这说明：
- “乱码链路尚未接入”这个旧问题已经被修补
- 但 `IsLanguageDecoded` 目前还是**临时状态存根**，未来应由正式剧情变量系统接管

### 3.2 `DialogueUI` 已具备验证场景自补线与显示状态机
文件：`Assets/YYY_Scripts/Story/UI/DialogueUI.cs`

关键事实：
- 事件订阅放在 `OnEnable()`，退订在 `OnDisable()`
- 使用 `EventBus.Subscribe(... owner: this)` 和 `EventBus.UnsubscribeAll(this)`
- `Awake()` 中会先调用 `ResolveValidationSceneReferences()`
- 仅当当前场景名是 `DialogueValidation` 时，才按固定对象名自动补线：
  - `DialoguePanel`
  - `SpeakerNameText`
  - `DialogueText`
  - `ContinueButton`
  - `PortraitImage`
  - `Background`
- 有普通对话 / 内心独白的 UI 状态切换：
  - 普通：白字、正常字体、恢复背景原始 Alpha、显示名字/头像
  - 独白：斜体、浅灰、背景 Alpha=0.5、隐藏名字/头像

这说明：
- 验证场景中序列化字段为空，不等于一定会 Null Reference
- 只要对象名没改，运行时可自补线
- 但正式场景**不能依赖这套自动补线**

### 3.3 `DialogueValidationBootstrap` 已存在且会运行时造样例数据
文件：`Assets/YYY_Scripts/Story/Managers/DialogueValidationBootstrap.cs`

关键事实：
- `Start()` 中：
  - 先判断是否处于 `DialogueValidation` 场景
  - 再判断 `playOnStart`
  - 然后自动调用 `PlaySampleDialogue(startDecoded)`
- `PlaySampleDialogue(bool isDecoded)` 中会：
  - 获取 `DialogueManager.Instance`
  - 设置 `manager.IsLanguageDecoded = isDecoded`
  - 调用 `manager.PlayDialogue(GetOrCreateRuntimeSequence())`
- 它运行时创建一个临时 `DialogueSequenceSO`
- 样例序列里有 3 个节点：
  1. 乱码节点（陌生旅人）
  2. 普通对话节点（艾拉）
  3. 内心独白节点
- 它还运行时创建了一个临时肖像 Sprite

这说明：
- 当前验证场景不依赖手工创建真实 `DialogueSequenceSO` 就能跑样例
- 这非常适合先做“系统闭环验证”
- 但它不是正式剧情接入方案，只是验证样例入口

### 3.4 `GameInputManager` 已最小接入对话输入锁
文件：`Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`

关键事实：
- 有 `_inputEnabled` 私有字段
- 有 `SetInputEnabled(bool enabled)`
- 对话开始时：`SetInputEnabled(false)`
- 对话结束时：`SetInputEnabled(true)`
- 当禁用输入时，会额外调用：
  - `playerMovement.SetMovementInput(Vector2.zero, false)`
- 订阅方式：
  - `EventBus.Subscribe<DialogueStartEvent>(OnDialogueStart, owner: this)`
  - `EventBus.Subscribe<DialogueEndEvent>(OnDialogueEnd, owner: this)`

这说明：
- 输入锁的**代码侧**已经接上了
- 但要确认“玩家真的在场景里被锁住”，仍需要真实场景验证

## 13. 关于“这个验证场景之后怎么用”

这是之前用户曾明确关心过的问题，所以交接里单独说清。

### 13.1 它不是正式主场景的替代品
不要把 `DialogueValidation.unity` 理解为“未来直接复制进主场景就完事”。

它更像：
- 一个隔离的验证沙盒
- 一个最小对话系统验收场
- 一个帮你确认系统链路无误的地方

### 13.2 正确用途
正确用途是：
1. 先在这里验证系统闭环
2. 确认对话播放/UI/输入锁都工作
3. 再把成熟做法迁移到正式场景和正式剧情资产上

### 13.3 不建议的做法
不建议：
- 直接把这个 scene 当主场景用
- 未经审视就把这里的临时布局/临时样例复制到正式生产内容里
- 把运行时样例 `DialogueValidationBootstrap` 当成正式剧情入口
