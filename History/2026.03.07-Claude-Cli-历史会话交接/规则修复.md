## 3. 这条治理线已经确认下来的核心结论

### 3.1 关于 Claude hooks 的最终现实定位

已经明确：
- `.claude/hooks` **不是** Kiro 的 askAgent / agentStop 语义执行层。
- `.claude/settings.json` / `.claude/settings.local.json` 本质是：
  - 权限表
  - hook 接线板
- 大多数 hook 只是：
  - marker
  - reminder
  - audit
  - 少量防呆

### 3.2 关于 `PreToolUse`

这个结论已经实锤，不是猜测：

1. 旧写法 `matcher: "Bash(rm*|git push -f*)"` 不可靠。
2. 可靠 wiring 是：`matcher: "Bash"`
3. 真正的危险命令判断放在 `.claude/hooks/pre-bash-block.sh` 内部做。
4. 当 hook 返回 `exit 2` 时，Claude Bash 工具会真实阻断该命令。
5. 实测证据包括：
   - `git push -f -h` 被阻断
   - `git push --force -h` 被阻断
   - `rm --version` 正常放行
   - probe log 证明 runner 确实调用了 repo 的 `PreToolUse` hook

### 3.3 关于 `Stop`

当前只确认到这里：
- `.claude/hooks/stop-update-memory.sh` 的**standalone 检测逻辑**已经成立。
- 它会检查已跟踪 `memory.md` 的 diff 中是否出现删除行。
- 命中删除行会 `exit 2`。
- 但：
  - `Stop` 生命周期本身是否真的触发？
  - `exit 2` 在 `Stop` 阶段是否具备真实阻断力？
  - 会不会只是提示 / 记录，不影响主流程退出？

这些还没拿到最终 runner 级结论。

### 3.4 关于 PlanMode

这个结论也已经固定：
- 本仓库默认**不主动进入 PlanMode**。
- 当前默认工作方式是 `acceptEdits`。
- 如果需要规划，正式产物应该直接回写到工作区文档，而不是回写到全局 `~/.claude/plans/`。
- 只有用户明确要求进入 PlanMode，且确认正式产物仍回写工作区时，才允许讨论这条路。

### 3.5 关于 memory

已经确定的硬规则：
- 代码修改、文档落盘、规则调整、架构决策、锐评核查、纯问答但形成结论，都必须记忆。
- 有子工作区时：**先子后父**。
- `memory.md` 必须 append-only。
- 写前读尾部，写后复查尾部。
- 允许分卷，不允许改历史块伪装成追加。

当前脚本关键逻辑：
- 读取 stdin 命令内容
- 归一化成单行
- 检查少量危险模式：
  - `rm -rf /`
  - `git reset --hard`
  - `git push -f`
  - `git push --force`
- 命中则：
  - stderr 输出 `[hook][block][pre-bash] matched dangerous shell pattern`
  - `exit 2`
- 未命中：`exit 0`

### 4.5 `.claude/hooks/stop-update-memory.sh`

当前脚本关键逻辑：
- 若不在 git worktree 内：打印提醒并 `exit 0`
- 扫描已修改的 `memory.md`
- 把 staged + unstaged diff 合并看是否有删除行
- 若有删除行：
  - stderr 输出 `[hook][warn][stop] append-only suspicion...`
  - `exit 2`
- 若没有删除行：
  - 提醒“没有检测到删除，但 memory 仍需人工确认”
  - `exit 0`

**注意**：它现在是一个“append-only 审计/防呆器”，不是已证实的自动 memory 更新器。
