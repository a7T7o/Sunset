## 二、这次确认过的真实状态

### 1. gateway 之前确实有隐性掉线问题

- 之前 `reader` / `reviewer` 虽然能回答，但日志里出现过：
  - `Gateway agent failed; falling back to embedded`
  - `gateway closed (1006 abnormal closure)`
- 这说明之前并不是 gateway 真稳定，而是 agent 在掉线后退回 embedded 模式兜底。

### 2. gateway 已重新拉起并验证监听

- 已重新启动本地 gateway。
- 当前监听端口已恢复：
  - `127.0.0.1:18789`
  - `[::1]:18789`
- Feishu WebSocket 也已重新建立。

### 3. `reader` / `reviewer` 当前都已实测能读到 `Sunset`

已完成实测：

- `reader` 能列出 `Sunset` 顶层目录
- `reviewer` 能列出 `Sunset` 顶层目录

这说明：

- 当前 `workspace` 指向已经正确
- `coding` profile 的读盘链路已可用
- 之前“看不到目录”的主问题已不再是当前阻塞

### 4. 浏览器基础设施已从“配置存在”推进到“可实际启动”

本次确认：

- Browser control service 已启动在 `18791`
- `openclaw` managed browser profile 已成功启动
- 当前检测到的浏览器为 Edge
- `openclaw` profile 当前可列出标签页

这意味着：

- 不是只有浏览器配置写在 config 里
- 而是本机托管浏览器这条链已经真正可用

### 5. Chrome / Edge relay 仍需手动完成最后一步

本次已做：

- 已执行扩展安装
- 扩展路径已准备好：`~\.openclaw\browser\chrome-extension`

但当前仍未完成的最后一步是：

- 在 Chrome / Edge 中手动加载 unpacked extension
- 手动点击扩展图标附着到具体标签页

所以目前结论是：

- `openclaw` 托管浏览器已可用
- 你自己的 Chrome / Edge 标签页接管链路还未完全落地

### 6. `main` 之前缺浏览器入口权限

这次补齐前，`main.tools.alsoAllow` 里只有：

- `web_search`
- `web_fetch`
- `image`
- `tts`

缺少：

- `browser`

这会导致即使本机浏览器已可用，`main` 也未必能直接调用浏览器工具。

本次已补：

- 为 `main` 增加 `browser` 工具权限

### 7. `web_search` 仍不是已解决项

当前仍然存在一个独立阻塞：

- `web_search` provider key 未配置
- 已确认真实报错：`missing_perplexity_api_key`

所以要明确：

- 浏览器能力 ≠ `web_search` 已恢复
- 现在能做的是浏览器驱动 / 网页读取 / `web_fetch`
- 不能假装说“联网搜索已经完全没问题了”
