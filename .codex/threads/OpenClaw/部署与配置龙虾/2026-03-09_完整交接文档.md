- `src/infra/json-files.ts`
- `src/infra/json-files.test.ts`

修复内容是：

- 原来 `writeTextAtomic()` 在 Windows 上执行 `rename(tmp, target)` 覆盖目标文件时，可能因为目标文件正被其他进程读取而失败

- `rename` 失败后按错误码判断
- Windows 风格可重试错误会 fallback 到 `copyFile`
- 测试覆盖了 `EPERM` 分支与非重试错误分支

## 五、关键踩坑与真正根因

## 5.1 “Unsupported type / Raw mode / apply_patch 警告”是工具用法问题，不是业务逻辑问题

线程里反复出现过：

- `Warning: apply_patch was requested via shell. Use the apply_patch tool instead of exec_command.`

这个已经证明：

- 补丁必须走 `apply_patch`
- 不能在 shell 里假装调用补丁

这属于执行层面的硬规矩。

## 5.2 “配置页面加载不出 / 保存异常 / 配置改了又像没改”不能只看 UI

线程里一个大教训是：

- OpenClaw UI 看起来能点，不代表配置一定已经正确落到运行态
- 有些真实状态必须回到 `openclaw.json` 和日志看

因此后续凡是遇到：

- 配置保存异常
- 工具权限不生效
- 页面和实际行为不一致

都应该优先交叉核对：

- `C:\Users\aTo\.openclaw\openclaw.json`
- Gateway 运行日志
- 对应 agent 的 session transcript

## 5.3 `builder` 的 “explicit allowlist” 提示不是 bug

这点上面已经提过一次，但值得单列：

- 用户看到这个提示时一度怀疑“是不是被搞花了”
- 真实情况是 builder 当前就是显式工具白名单

所以这行提示是**配置结果的说明**，不是异常。

## 5.4 “agent 看不到项目”不能靠 Junction 欺骗

线程里已经验证过：

- 把 workspace 做成指向 Sunset 的 Junction/目录链接，并不能绕过 OpenClaw 的路径策略

因此以后不要再在这条路上浪费时间。

正确方式只有两个关键词：

- 真实 workspace 根
- 正确工具权限

## 5.5 `main` 变慢，不是简单因为“队列堵了”

后期已经查清：

- `queueDepth = 0`
- 请求并没有堆在队列里
- 主要耗时出在首 token 之前

所以：

> 当前 `main` 的主要慢点不是排队，而是上游链路 jitter + 本地 prompt 负担。

## 5.6 `main` 的 prompt 文件过重，曾经确实放大了慢响应

线程里已经做过一轮很关键的软治理：

- 精简 `C:\Users\aTo\.openclaw\workspace\` 下的提示文件
- 软重置 `main` 会话

那一轮修完后，`main` 的 prompt token 确实明显下降过，体感一度改善。

这说明：

- prompt 体积不是虚假问题
- 但它不是唯一主因

## 5.7 当前 live binary 不接受 `agents.list[].params`

这是一个很容易重复踩的坑。

线程中已经证明：

- 虽然源码/文档中能看到更丰富的参数入口
- 但当前用户实际跑的这个版本，不接受 `agents.list[].params`

因此当前可靠做法是：

- 把模型参数放到 `agents.defaults.models["provider/model"].params`

不要再把时间浪费在给 `agents.list[].params` 配一堆无效值上。

## 5.8 `openai-responses` 和 `openai-completions` 的权衡已基本摸清

线程后期对上游做了真实压测，已经得到高价值结论：

- `openai-completions`
  - 更尊重请求的具体模型名
  - 但延迟抖动更明显
- `openai-responses`
  - 有时体感更快
  - 但会把 `gpt-5.4-xhigh-fast` / `gpt-5.4-xhigh` / `gpt-5.4-fast` 这类 suffix 折叠成 `gpt-5.4`

所以如果用户在意具体模型档位，当前还是更适合保留：

- `api = openai-completions`

## 八、接下来最合理的优先级

这里不写“所有可能做的事”，只写最合理的顺序。

## P0：保持当前系统别再回退

重点：

- 不要再随意重堆 `main` prompt
- 不要再把 heavy capability 全塞回 `main`
- 不要在没有必要时反复重启/折腾 gateway

## P1：决定 `main` 的稳定优先模型策略

这是当前体验层面的第一优先级。

推荐路线：

- 如果目标是入口稳定，优先把 `main` 和 `pm` 切到 `gpt-5.4-xhigh`
- `reader` 保持 `xhigh-fast`
- `reviewer` 保持 `high`
- `builder` 保持 `xhigh`

这一步做完，才能更可靠地继续工作流建设。

## P2：让 `pm` 真正上岗

具体不是“再建一个 agent”，而是：

- 明确 `main` 和飞书都该如何把高层任务交给 `pm`
- 让 `pm` 负责拆任务与汇总

## P3：建立第一版半自动派单协议

目标不是一上来全自动，而是：

1. 用户给高层任务
2. `pm` 产出派单草案
3. `reader` / `reviewer` 并行
