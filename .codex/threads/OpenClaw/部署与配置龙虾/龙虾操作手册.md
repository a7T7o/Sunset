# 龙虾操作手册

## 一、当前最稳的入口

### 入口 1：`main`

适合：

- 本地调试
- 检查配置
- 验证浏览器/agent/gateway
- 紧急救火

### 入口 2：飞书

适合：

- 日常正式使用
- 远程给出任务
- 收取汇总结果

这两个入口都可用，当前不要把它们看成互相替代。

---

## 二、当前 agent 分工

- `main`：总入口 / 收口 / 排障 / 汇总
- `pm`：任务拆分、优先级整理、项目经理视角汇总
- `reader`：代码阅读、找入口、梳调用链
- `reviewer`：风险审查、回归风险、测试缺口
- `builder`：边界明确后的最小实施

---

## 三、当前最稳的浏览器用法

### 方案 A：托管浏览器 `openclaw`

这是当前最稳、最先可用的方案。

已确认可用能力：

- 启动浏览器
- 列出标签页
- 后续可用于打开页面、读取页面、做浏览器操作

### 方案 B：接管你自己的 Chrome / Edge 标签页

当前还差最后一步手动操作：

1. Chrome 或 Edge 打开扩展管理页
2. 开启开发者模式
3. 选择 `Load unpacked`
4. 加载 `~\.openclaw\browser\chrome-extension`
5. 固定扩展并在目标标签页点击附着

---

## 四、当前最稳的工作流

1. 你在 `main` 或飞书提出高层目标
2. `main` 先帮你整理成派单草案
3. 子任务分别发给：
   - `reader`
   - `reviewer`
   - `pm`
4. `builder` 只在边界明确后再启动
5. 结果再回到 `main` 或飞书汇总

---

## 五、当前不要误判的点

### 1. “浏览器能力可用”不等于“联网搜索全恢复”

当前：

- 托管浏览器已可用
- `web_search` 仍缺 provider key

### 2. “多 agent 都存在”不等于“自动编排已落地”

当前：

- 多 agent 已在
- 自动派单还没正式做完

### 3. “agent 能回答”不等于“gateway 一直稳定”

本次已经修过一次 gateway 掉线后退回 embedded 的问题，后续若再感觉不对，优先先查 gateway 是否仍监听 `18789`
