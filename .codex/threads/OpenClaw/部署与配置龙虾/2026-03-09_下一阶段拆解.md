# 下一阶段拆解（2026-03-09）

## 阶段 A：浏览器接入收尾

### 目标
- 让 Chrome / Edge 不只是 relay 端口可认证
- 而是真正能控制用户当前浏览器标签页

### 完成标准
- Chrome 至少有 1 个附着标签页
- Edge 至少有 1 个附着标签页
- `pnpm openclaw browser tabs --browser-profile chrome` 能列出标签页
- `pnpm openclaw browser tabs --browser-profile edge` 能列出标签页

### 需要用户手动完成
- 浏览器 UI 中加载 unpacked extension
- 在标签页点击扩展图标附着

## 阶段 B：搜索 provider 决策

### 目标
- 决定要不要启用真正的 `web_search`

### 路径 1：启用
- 补一个真实搜索 key
- 复测 `main` 和 Feishu 的 `web_search`

### 路径 2：暂不启用
- 明确把联网策略固定为 `browser + web_fetch`
- 后续任务提示词里不再假设 `web_search` 可用

## 阶段 C：入口统一

### 目标
- 明确谁是总控入口
- 明确 `main` 和 Feishu 的职责边界

### 当前建议
- `main`：本地桌面总控入口
- Feishu：移动侧/远程总控入口
- 两者都应该进入同一套 PM 协议，而不是各自走各自逻辑

### 完成标准
- 能清楚回答一句话：
  - “我在 main 发总任务，会发生什么？”
  - “我在 Feishu 发总任务，会发生什么？”

## 阶段 D：半自动派单

### 目标
- 先让 `pm` 成为稳定的任务拆解器
- 再让 `reader/reviewer/builder` 接固定格式子任务

### 最小落地形态
1. 用户在 `main` 或 Feishu 发高层任务
2. `pm` 先输出拆解单
3. 用户确认后，再把拆解单分别发给 `reader/reviewer/builder`
4. `pm` 回收结果并汇总

### 这样做的原因
- 风险低
- 可解释
- 出问题时容易定位
- 不会一上来就把系统搞成黑盒

## 阶段 E：自动派单试点

### 前提
- 阶段 A、B、C、D 已稳定

### 试点目标
- 仅对“只读分析型任务”开放自动派单
- 不允许自动修改代码
- 不允许多个 agent 同时修改同一系统

### 首批适合试点的任务
- 项目结构扫描
- 调用链梳理
- 架构风险分析
- 存档链路分析

## 当前建议

- 现在就可以开始阶段 C 和阶段 D 的文档设计
- 真正自动化执行，放到阶段 A 和阶段 B 稳定之后
