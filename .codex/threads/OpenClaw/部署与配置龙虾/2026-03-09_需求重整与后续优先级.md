# 需求重整与后续优先级（2026-03-09）

## 一、这次重新分析的目的

这份文档的目的不是重复“主入口提速已经完成”，而是重新校准一件更重要的事：

> 我们修完速度之后，真正要继续推进的系统到底是什么？

你这次的提醒是对的：

- 不能只盯着“把飞书绑定到 pm”这一个动作
- 还要重新思考 **main 窗口、飞书入口、龙虾本体、多 agent 角色、自动化流程** 之间的关系
- 否则会把“局部正确的技术动作”误当成“整体正确的工作流”

所以这份文档要解决四个问题：

1. 你现在真正的需求是什么
2. 我们当前的真实处境是什么
3. 我们现在应该把系统整理成什么样
4. 后续优先级应该如何排

---

## 二、重新理解后的真实需求

这次重新看，你的需求不是“飞书替代 main”，也不是“main 替代飞书”。

你的真实需求是：

### 1. `main` 窗口和飞书都要可用

你要的是一个**可切换的双入口系统**，而不是单入口系统。

原因很明确：

- 飞书适合日常发指令、远程使用、工作流推进
- `main` 窗口适合本地调试、配置核对、手动验证、紧急救火
- 如果只保留一个入口，系统韧性会下降

所以正确理解不是“二选一”，而是：

> **两者都应该可用，而且要能在工作场景里自由切换。**

### 2. 飞书应该承载一套完整自动化流程

你真正想要的不是“飞书里也能聊两句”，而是：

- 在飞书中给出一条高层任务
- 龙虾读懂任务并开始分工
- 不同角色各自处理自己的子任务
- 最终汇总给你

也就是说：

> 飞书不是装饰性渠道，而应该是你日常工作流的正式入口。

### 3. 本质上操作对象仍然是“龙虾 / OpenClaw 系统本体”

这点你说得也对。

飞书自动化流程的本质，不是“飞书系统在自动化”，而是：

- 飞书只是消息表面
- 真正干活的是龙虾 / OpenClaw 本体
- 真正要配置清楚的是：
  - 入口如何路由
  - 谁负责总控
  - 谁负责分析
  - 谁负责执行
  - 结果如何沉淀

所以问题的本质仍然是：

> **龙虾的窗口与配置如何组织成一套长期可用的工作系统。**

### 4. 你要的不是“多 agent 存在”，而是“多 agent 可工作”

这两者差别很大。

你真正需要的是：

- 不是只看到 `pm / reader / reviewer / builder` 这几个名字存在
- 而是它们之间已经形成可执行的协作秩序

也就是说，你要的是：

- 总控入口明确
- 分工关系明确
- 派单机制明确
- 汇总机制明确
- memory / tasks / progress 有沉淀机制

## 三、我们当前的真实处境

下面这部分不是理想状态，而是当前已经核对过的真实状态。

### 1. `main` 和飞书当前都可用，但只是“两个表面”

目前：

- `main` 窗口可用
- 飞书私聊可用
- 它们都能和龙虾说话

但要注意：

> “都可用”不等于“都已经进入正确工作流”。

当前它们更像是：

- 两个已经恢复正常的入口表面
- 但后面的调度逻辑还没真正搭起来

### 2. `main` 现在是轻量聊天入口，不是总控调度核心

经过前一轮修复后，`main` 被有意收缩成了轻量入口。

这一步是合理的，因为不这么做，主入口会继续被重上下文拖死。

当前的 `main` 更适合：

- 本地聊天
- 本地核对
- 配置调试
- 手动验证
- 系统救火

而不适合：

- 承载所有重分析能力
- 承载所有浏览器能力
- 承载完整多 agent 自动编排

所以当前 `main` 的定位已经变化：

> 它现在是“轻入口”，不是“万能总控核心”。

### 3. 飞书是可用主表面，但还不是自动化总控入口

当前飞书已经可用，但它还没有成为真正的 orchestrator surface。

也就是说，现在飞书并不是：

- 你发一条任务
- `pm` 自动接手
- 其他角色自动并行
- 最后自动汇总

当前它只是：

- 一个已经连通的聊天入口
- 后面还是默认逻辑

### 4. 多 agent 已经存在，但尚未进入自动工作流

当前系统里已经有：

- `pm`
- `reader`
- `reviewer`
- `builder`

但“存在”不代表“已经自动协作”。

当前问题在于：

- 还没有路由规则把某个入口正式交给 `pm`
- 还没有让 `pm` 成为真正值班的总控角色
- 还没有建立清晰的分工执行链

所以当前本质仍然是：

> **多 agent 已配置，编排层未落地。**

### 5. 浏览器基础设施在，但还没进入主工作流

当前浏览器相关配置与 profile 已经保留。

但它还没有变成你可以稳定依赖的工作流部件。

这说明：

- 浏览器能力“存在”
- 但浏览器能力“还没有被组织进体系”

### 6. 当前系统的核心问题不是“入口断了”，而是“角色关系还没整理完”

目前系统已经从“坏掉”恢复为“可用”。

所以现在真正的问题已经不是：

- 飞书能不能连上
- 网页能不能打开
- 龙虾能不能回话

而是：

- 两个入口各自承担什么职责
- 谁是总控
- 谁是工作者
- 自动化从哪里开始
- 哪些能力该进主流程，哪些能力不该再塞回 `main`

## 四、现在应该怎么理解“main”和“飞书”的关系

这一部分是这次重整里最关键的判断。

## 1. 最合适的理解不是“谁替代谁”，而是“表面层 vs 控制层”

更合理的系统结构应该是这样：

### 表面层（Surface）

这是你与龙虾交互的地方：

- `main` 窗口
- 飞书

### 控制层（Control / Orchestration）

这是决定任务怎么拆、谁去做、结果怎么汇总的地方：

- `pm`

### 执行层（Workers）

这是各自承担专业子任务的地方：

- `reader`
- `reviewer`
- `builder`
- 未来的浏览器 / research 角色

### 沉淀层（Memory / Task Ledger）

这是保存记忆与工作状态的地方：

- `memory`
- `tasks`
- `progress`
- handoff / report

所以正确理解应该是：

> `main` 和飞书都只是“表面层”；真正要搭建的是后面的控制层和执行层。

## 2. `main` 应该继续保留，而且必须保留

我认为 `main` 不应该被废掉，也不应该完全让位给飞书。

原因：

- `main` 是本地救火入口
- `main` 是配置调试入口
- `main` 是飞书失效时的后备入口
- `main` 是查看系统真实状态最方便的入口

所以更合理的结论是：

> `main` 必须保留，而且要保持快、轻、稳。

## 3. 飞书应该成为“日常工作总入口”

如果从使用频率和场景看，飞书更适合作为：

- 日常任务入口
- 远程工作入口
- 高层指令入口
- 自动化流程入口

所以飞书不应该只是“也能聊”，而应该变成：

> **日常工作入口**。

## 4. 这不意味着飞书要绕开龙虾本体

你提醒得很对：

- 真正需要整理的是龙虾系统本身
- 飞书只是把任务送进来
- 本体系统仍然是核心

因此正确说法不是：

- “飞书自己做自动化”

而是：

- “龙虾以飞书作为主要任务入口来触发自动化工作流”

---

## 五、我们现在“应该是什么情况”

如果按照更合理的中间态设计，现在最应该达到的状态是：

### 1. `main` 和飞书都能用

- `main`：本地轻入口 / 调试 / 配置 / 手动验证 / 紧急救火
- 飞书：日常总入口 / 远程派单 / 自动化流程触发

### 2. `pm` 成为逻辑上的总控中心

不管消息从哪里来，最终真正负责“拆任务、汇总结果、管理节奏”的都应该是 `pm`。

### 3. worker agent 不直接面向用户，而面向 `pm`

- `reader`：只读分析
- `reviewer`：风险审查
- `builder`：明确边界后的实施
- `research`（未来）：联网 / 浏览器 / 后台巡检

### 4. memory / tasks / progress 进入正式工作流

每次工作不能只停在聊天记录里，而要落地为：

- 已完成了什么
- 当前卡在哪里
- 下一步该做什么
- 谁负责什么

---

## 六、我们最终要做到什么情况

最终目标不是“main 和飞书都能聊天”，而是下面这套结构：

### 目标态：双入口、单系统、分层协作

#### 第一层：双入口
- `main` 窗口：本地入口
- 飞书：日常入口

#### 第二层：单系统
- 本质上仍然是同一个龙虾 / OpenClaw 系统在工作
- 不是飞书一套、网页一套

#### 第三层：分层协作
- `pm` 负责总控
- `reader` / `reviewer` / `builder` / `research` 负责执行
- 结果统一回给你
- 同步进入 memory / tasks / progress

换句话说，最终目标是：

> 你在飞书或本地入口发一条高层任务，龙虾识别任务、调用正确角色、并行推进、最终汇总，并且把工作状态落盘。

这才是你真正要的“持续工作系统”。

## 七、后续工作优先级（按重要性排序）

下面这部分是最重要的执行结论。

## P0：保持 `main` 继续轻量、稳定、可救火

### 目标
不让我们刚救回来的 `main` 再次变重、变慢、失控。

### 原因
- `main` 是本地救火入口
- 是系统韧性的关键
- 不能为了自动化再把所有能力塞回去

### 原则
- `main` 保持轻入口定位
- 不重新堆重工具、重技能、重上下文

---

## P1：把飞书直聊正式绑定给 `pm`

### 目标
让飞书不再只是默认聊天入口，而是正式成为“总控任务入口”。

### 为什么是第一优先级
因为你已经明确：

- 飞书是更核心的人机面
- 飞书应该承载完整自动化流程
- 总控角色应该是 `pm`

### 这一步的结果应该是
- 你在飞书发任务
- 不是 `main` 在接
- 而是 `pm` 在接

### 注意
这一步并不等于废掉 `main`。

而是：
- 飞书 -> `pm`
- `main` -> 保留为本地轻入口

这是当前阶段最合理的双入口分工。

---

## P2：明确 `pm` 的派单模板与工作边界

### 目标
让 `pm` 不是“名字叫 PM”，而是真的具备总控行为。

### 需要明确的内容
- 什么时候派给 `reader`
- 什么时候派给 `reviewer`
- 什么时候允许进 `builder`
- 什么时候需要联网 / 浏览器角色
- 最终汇总模板是什么

### 核心原则
- 分析并行
- 修改串行
- 未经确认不扩大影响面

---

## P3：建立“飞书任务 -> `pm` -> worker”的第一版编排链

### 目标
实现真正的第一版可用工作流。

### 不要求一步到位自动魔法
第一版不用追求“全自动黑盒编排”，而要先实现：

- 飞书收任务
- `pm` 做拆分
- `reader` / `reviewer` 并行
- `pm` 汇总
- `builder` 在确认后再介入

### 这一步的价值
这一步一旦成了，系统就从“多 agent 配置存在”升级为“多 agent 真正开始工作”。

---

## P4：补一个专门的浏览器 / research 角色

### 目标
把联网、搜资料、读网页、后台巡检这些能力，从 `main` 中分离出来。

### 为什么重要
你真正需要浏览器能力，但我们已经确认：

- 这些能力不应该再回塞 `main`
- 否则主入口很容易重新变重

### 更合理的做法
建立独立角色，例如：
- `research`
- `navigator`

交给 `pm` 按需调用。

---

## P5：修正 `builder` 的工作区与实施边界

### 目标
让 `builder` 真正能够进入 Sunset 的实施闭环。

### 当前问题
当前 `builder` 还不是面向 Sunset 真正落地实施的理想配置。

### 这一步要做的事
- 确定 `builder` 的正确工作区
- 确定它的修改边界
- 确定它何时允许改文件
- 确定验证和回传方式

---

## P6：建立 tasks / progress / memory 的正式沉淀机制

### 目标
把系统从“会聊天”升级为“会持续工作”。

### 这一步的意义
如果没有这层沉淀：
- 任务会断
- 角色切换会断
- 场景切换会断
- 过几轮又回到“全靠人脑记”

### 这一步应该落地为
- tasks
- progress
- report
- memory 更新纪律

---

## P7：再决定是否让 `main` 也拥有“切换到 PM 模式”的能力

### 说明
这一步是后续增强项，不是当前第一优先级。

### 原因
你希望 `main` 和飞书都可以自由切换，这个诉求本身是合理的。

但当前更稳的推进路径是：

1. 先让飞书成为正式总控入口
2. 先把 `pm` 编排链跑通
3. 再决定 `main` 是否也要直达 `pm`

### 为什么不建议现在直接一步做满
因为现在最怕的是：
- 还没把角色关系理清
- 又把 `main` 拉回复杂状态
- 再次把主入口拖慢

所以这一项应该放在后面，而不是先做。

---

## 八、结论

这次重新整理后，结论已经非常清楚：

### 1. 你的需求不是“飞书替代 main”
而是：

- `main` 和飞书都要可用
- 两者都属于龙虾系统的表面层
- 你需要的是“可切换双入口”

### 2. 飞书应该成为日常工作总入口
也就是：

- 日常派单
- 高层指令
- 自动化流程触发
- 未来的持续工作入口

### 3. `main` 必须保留为轻量本地控制台
它的职责不是承担所有工作，而是：

- 本地调试
- 配置核对
- 手动验证
- 紧急救火

### 4. 当前真正缺的不是入口，而是“控制层”
也就是：

- `pm` 还没有真正上岗
- worker 还没有真正形成编排链
- 浏览器能力还没进正式流程
- tasks / progress / memory 还没形成闭环

### 5. 当前最正确的下一步不是继续改 `main`
而是：

> **先把飞书直聊绑定给 `pm`，让 `pm` 真正接管总控入口。**

这一步一旦完成，整个系统就会从“恢复可用”进入“开始形成真正工作流”的阶段。
