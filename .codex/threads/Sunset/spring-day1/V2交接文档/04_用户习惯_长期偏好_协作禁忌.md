# 用户习惯、长期偏好与协作禁忌

## 1. 用户对文档位置的要求
1. 线程记忆必须落在项目内：
   - `D:\Unity\Unity_learning\Sunset\.codex\threads\Sunset\spring-day1\memory_0.md`
2. 工作区文档和线程交接要分层：
   - 工作区正文在 `.kiro/specs/...`
   - 代际交接正文在 `.codex/threads/.../V2交接文档/`
3. 不能把交接文档错误写到 `.kiro`。
4. 不能把全部线程交接都堆在 `.codex/threads` 根层共享目录。

## 2. 用户对主线 / 支线纪律的要求
1. 用户极度在意主线是否漂移。
2. “修报错 / 查问题 / 补验证 / hygiene / 纠偏”通常都只是主线支撑动作，不代表换题。
3. 用户不接受线程在没有明确切题的情况下，自己把主线换成别的东西。
4. 用户喜欢“这一轮就做这一刀”，不喜欢顺手叠第二刀。

## 3. 用户对中文透明度的要求
1. 普通说明一律中文。
2. 路径、命令、代码标识可以保留原文。
3. 用户要求回复透明，不要“梦游式复述背景”，也不要泛泛而谈。
4. 用户更喜欢：
   - 结论先行
   - 边界清楚
   - changed paths 清楚
   - blocker 明确

## 4. 用户对 UI / 观感的长期偏好
这些是 V2 不能丢的硬标准：

### 任务卡 / Prompt
1. 不接受“能显示就算完成”。
2. 用户要的是有日历感、撕页感、正式 RPG UI 感的任务页，不是开发者测试条。
3. 一任务一页，完成时要有逐条完成动画与翻页承接。
4. 文字变多时必须自适应，下面内容要跟着移动，不能重叠。
5. 对话出来时，任务提示应该隐藏；对话结束后，UI 应有渐入回归，而不是硬切。

### 对话体验
1. 用户要沉浸式：
   - 对话时其他 UI 要隐藏或压制
   - 时间静止时其他内容不应继续乱动
2. 跳过对话不要“任意键”，用户明确改成 `T` 键。
3. 血条 / 回复类表现更偏好“逐渐显示、逐渐变化”，而不是突兀弹出。

### 工作台 UI
1. 工作台 UI 必须悬浮在工作台附近，并跟随工作台，而不是跟随镜头漂。
2. 显示方向只有上下两种：
   - 默认上方
   - 如果玩家在工作台上方交互，就显示在下方
3. 用户不接受“相对运动 / 漂移感”。
4. 左侧必须有可滚动的配方选择区，而且图标真的要看得见。
5. 右侧要显示：
   - 工具名称
   - 简介
   - 材料需求
   - 数量控制
   - 制作按钮 / 进度
6. 右键不能透过工作台 UI 去点地面触发导航。
7. 工作台 UI 区域只允许左键进行 UI 交互。
8. 数据应以 `SO` 为唯一来源，不要在 UI 里塞平行硬编码。

## 5. 用户对 prompt / 交接 / 治理格式的偏好
1. 用户经常要求“按固定回执格式回复”，必须严格照做。
2. 用户不喜欢你“解释了很多，但没按要求回核心字段”。
3. 用户喜欢把长任务切成明确委托文档，然后要求线程必须完整读取。
4. 用户会要求“直接执行某个 prompt 文件”，这时必须以 live 文档为准。

## 6. 用户最讨厌的行为
1. 没有自检、把编译错误留给 Unity / 用户发现
2. 明明没复刻对，却说“我看过了是一样的”
3. 把旧口径当 live 事实
4. scope 漂移
5. 没按固定格式回执
6. 擅自删治理证据、委托文档或样式快照
7. 自作主张认领别的线程文件

## 7. 用户最常纠正的问题
1. 你是不是又把支撑子任务当成换主线了？
2. 你是不是只回答了最后一句，没有交代主线状态？
3. 你是不是以为“能跑/能显示”就等于“好看/对味/可验收”？
4. 你是不是没先自己检查编译与基本闭环？
5. 你是不是没有真正按 prompt 全量吸收要求？
