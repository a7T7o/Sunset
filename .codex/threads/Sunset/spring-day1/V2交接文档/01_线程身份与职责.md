# 线程身份与职责

## 最初创建动机
`spring-day1` 最初是为 `900_开篇 / spring-day1-implementation` 服务的业务线程，目的是把春 1 日从“规划/文档”推进到“真实可运行的首段剧情与后续 Day1 教学链”。

最早核心目标有两层：
1. 把 `0.0.2` 的剧情基础脊柱真正落进项目主干。
2. 围绕 Day1 首日体验，逐步把对话、任务提示、工作台、教学链和验收口径接起来。

## 历史上承担过的主要职责
1. `0.0.2` 首段剧情骨架迁入 `main`：
   - `StoryPhase`
   - `StoryManager`
   - `DialogueSequenceSO` 推进字段
   - `DialogueManager` 完成事件
   - `NPCDialogueInteractable` 首段 / follow-up 分流
2. Day1 运行链接线：
   - 和 `NPC001` 的首段对话
   - 首次解码 / follow-up
   - 导演层骨架、测试骨架、调试骨架
3. Day1 用户体验收口：
   - Prompt / 任务卡
   - 工作台交互与配方
   - UI 层级压制
   - 与 NPC / 工作台近距交互提示的最小接轨
4. 线程后期的 hygiene / ownership 纠偏：
   - own checkpoint 与 hot-file blocker 分离
   - 撤回错认 owner 与越权删证据的判断

## 当前真正身份
到本轮为止，`spring-day1` 的真正身份已经从“继续一刀刀写 Day1”切换成：

> “为 `spring-day1V2` 准备一套高密度、可接班、可避免重蹈覆辙的交接包。”

也就是说，当前线程最重要的价值不是再多做一点功能，而是把：
- 已接受成果
- 未接受的部分
- 用户硬性偏好
- 他线边界
- 当前 blocker

一次写清楚。

## 与其他线程的边界

### 与场景搭建线程
- 场景搭建线程负责“空间承载物”和场景布局稳定。
- `spring-day1` 只负责把剧情 / UI / 交互逻辑挂到已存在的承载点上。
- 工作台对象、床位、农田、树木、晚餐承载点的空间质量，原则上不由 `spring-day1` 一线吞并。

### 与 NPC 线程
- NPC 线程负责漫游、路径、区域、行为节奏等长期体验。
- `spring-day1` 只在 Day1 对话 / 教学链需要时，提出“对话期间冻结、面向玩家、恢复漫游”的接轨要求。
- 不能把整个 NPC 行为树和导航问题并到 Day1 线程里。

### 与导航 / 放置 / 农田线程
- 如果 Day1 运行态被这些系统的 compile red 或 runtime red 阻断，要记为 external blocker。
- 不要因为“Day1 被卡住”就擅自认领这些系统的尾账。

### 与治理线程
- 治理线程负责 prompt、索引、委托、卫生委托、规则快照。
- `spring-day1` 可以引用这些文档，但不能自发删改它们来“帮治理收口”。

## 哪些事项虽然相关，但不应再由本线继续吞并
1. `Primary.unity` 的 scene mixed 清分
2. 他线 debug 物体、玩家/NPC/相机位移、shared root 编译红字总清扫
3. 治理证据删除
4. 任何“顺手把 shared root 清干净”的冲动

## 一句话给 V2 的身份提醒
`spring-day1V2` 接手时，要把自己当成“延续 Day1 业务与体验判断的线程”，而不是“什么都来一把的 shared root 清洁工”。
