# 关键节点、分叉路与判废记录

## 关键节点 1：`0.0.2` 基础脊柱必须先进 `main`

### 当时的分叉
- 一条路：继续停留在 carrier / 文档层。
- 另一条路：把最小基础脊柱真正迁入 `main`。

### 最终判定
选择了后者，这个判断是对的。

### 为什么重要
如果没有 `83d809a9` 这一步，后续所有 Day1 首段剧情、follow-up、导演层、工作台与任务链都会失去真实挂点。

---

## 关键节点 2：Day1 线程不能把“能编译 / 有代码”包装成“体验已通过”

### 误判是怎么发生的
- 线程一度过分关注“代码有没有写上”“脚本能不能过 validate”“逻辑大致能不能走”。
- 但用户的反馈已经多次明确：体验、观感、层级、交互感才是决定性验收项。

### 后来为什么改判
- 用户连续多轮指出：
  - 工作台 UI 审美差
  - 左列没显示
  - 位置不对
  - 任务卡效果不对
  - 浮动 / 漂移感不对
  - 对话与任务提示互相挡
- 这说明“代码上有东西”根本不是完成定义。

### 当前应以什么为准
- 任何和 UI/体验直接相关的东西，都必须按“用户真实游玩观感”算数。

---

## 关键节点 3：工作台 UI 不能只看运行时生成结果，必须以 prefab 为视觉基线

### 误判是怎么发生的
- 线程一度用“我读了 prefab / 我理解了样式 / 运行时结构大致能对上”为理由，误以为自己已经复刻用户精修过的 UI。
- 实际上用户明确指出：运行态看到的内容和 `Assets/222_Prefabs/UI/Spring-day1` 里的预制体根本不是一回事。

### 改判原因
- 用户明确指出：
  - edit 模式静态下看不到运行时创建的那堆节点；
  - 运行态字体、尺寸、排版、位置、滚动区全部不对；
  - “哪怕照着抄都行”，结果也没抄对。

### 当前正确口径
1. 视觉样式以预制体为准。
2. 运行时代码只能在不破坏视觉基线的前提下做功能层重构。
3. 任何“我觉得差不多”都不算证据，必须以 live 对照为准。

---

## 关键节点 4：两份 UI 脚本不是独立闭包，最小代码闭包是 7 文件

### 旧误判
把 `SpringDay1PromptOverlay.cs + SpringDay1WorkbenchCraftingOverlay.cs` 当成当前可单独白名单收口的完整面。

### 为什么后来改判
- live preflight 直接证明这两份文件依赖其它 Day1 接口：
  - `SpringDay1UiLayerUtility.cs`
  - `SpringDay1WorldHintBubble.cs`
  - `SpringDay1Director.cs`
  - `CraftingStationInteractable.cs`
  - `NPCDialogueInteractable.cs`

### 当前应继承的结论
如果 V2 再做 Day1 UI / 工作台行为续工，不要再幻想“两份 UI 脚本单独热修就够了”。

---

## 关键节点 5：`Primary.unity` 必须改判为 mixed hot-file blocker

### 误判是怎么发生的
- 线程一度倾向把 `Primary.unity` 里的剧情相关改动继续视为自己的一部分。
- 但 live 证据显示，它不是纯 Day1 own scene。

### 当前 mixed 的核心原因
`Primary.unity` 同时混有：
1. Day1 自己的剧情侧块：
   - `StoryManager`
   - `startLanguageDecoded`
   - `Anvil_0` 参数
2. 他线内容：
   - 相机位移
   - Player 位移
   - NPC debug override

### 当前正确结论
- `Primary.unity` 不再由 `spring-day1` claim owner。
- 只能在交接里写成 blocker，不能继续 scene 续写。
