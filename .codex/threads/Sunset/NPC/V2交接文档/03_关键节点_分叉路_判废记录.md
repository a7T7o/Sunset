# NPC 关键节点、分叉路与判废记录

## 1. 最大的早期误区：把“carrier 在”误当成“生产现场已稳”

### 旧口径

- 只要 `codex/npc-roam-phase2-*` 还在，就默认 NPC 现场还在。

### 后来为什么改判

- `Primary.unity` 真正依赖的是 `main` 工作树里的运行时资产。
- carrier 在，不等于 `main` 已拥有生产场景依赖。

### 当前应以什么为准

- 一律先看 live `main` 的真实资源闭合与当前 working tree 状态。

## 2. 最大的职责误判：把导航缺陷继续算到 NPC 头上

### 误判是如何发生的

- 用户在描述“玩家推着 NPC 走”“NPC 和 NPC 互顶”时，我一度把完整系统修正一起包给 NPC 线程。

### 为什么后来改判

- 只读代码后能明确区分：
  - NPC 负责消费现有导航结果与自身状态语义
  - 导航线程负责动态避让、让行、局部求解和交通裁决

### 当前必须保留的判断逻辑

- 以后只要再出现“玩家推着 NPC 走”，先问：
  - 这是导航裁决失败？
  - 还是 NPC 自身消费结果失败？
  - 不能先把锅一起背回 NPC 线。

## 3. 气泡返工里最值得保留的原始逻辑

### 错路

- 只看数值是否改了，不看用户是否能明显看见差异。

### 正确逻辑

- 用户在这条线上最在意的是：
  - 留白
  - 尾巴方向
  - 可读性
  - 动效节奏
  - 整体是否自然、专业、好看

### 必须传给 `NPCV2` 的经验

- 面对 UI / 气泡类需求，不要满足于“参数改了”。
- 要以“用户肉眼是否一眼看出改善”为标准。

## 4. `003` 的关键分叉：压测 NPC 还是正式 NPC

### 旧状态

- `003` 一度因为 `NPCBubbleStressTalker.startOnEnable = true`，在 Play 时自动关闭自己的 roam。

### 为什么这是问题

- 这会把“验证样本”误变成“正式场景里的异常 NPC”。

### 改判后口径

- `003` 应该是：
  - 默认仍是正式 NPC
  - 同时长期保留为验证样本入口
  - 测试模式必须显式开启

### 当前结论

- 这条判定已经推进到 prefab、生成器、scene integration tool 和 inspector 入口层，不应再回退。

## 5. 文档治理里的判废记录

### 为什么判废

- 用户明确要求文件少而精。
- 多份平行文档会让下一代线程继续在文档森林里迷路。

### 当前准则

- 导航协作文档只维护：

## 6. 关于当前 blocker 的关键改判

### 最容易出错的旧判断

- “Unity / MCP 可用，所以现在就能进 `Primary.unity` 继续做。”

### 为什么这是错的

- 当前真正阻塞不是 Editor 状态，而是 Git 磁盘态：
  - `Assets/000_Scenes/Primary.unity` 仍是 dirty
  - 这轮 live 复核里它的 `git diff --stat` 是 `76 insertions / 4 deletions`
  - 同时没有 active lock / 明确独占写归属

### 当前必须坚持的判断逻辑

- scene 开工条件必须同时满足：
  1. Git dirty 归属明确
  2. A 类热区窗口明确
  3. Unity 在 Edit Mode 且非 compile / reload
  4. 没有新的 shared hot-zone 串扰

## 7. 明确不要再走的路

1. 不要再把旧 `mcp-unity` / `8080` 口径当 live 事实
2. 不要再把 `codex/npc-roam-phase2-003` 当现实执行现场
3. 不要再在 shared root mixed dirty 时，顺手把 scene 集成硬做掉
4. 不要再把“只差一点”当理由去吞并导航线程主战场
5. 不要再把“已经有 2.0 设计”误说成“scene 集成已经完成”

## 8. 当前最值得保留的判断结论

这条线程最有价值的不是某个单点参数，而是以下 4 个稳定结论：

1. NPC 当前下一刀应该是 scene 集成，不是继续微调表现
2. 导航与 NPC 的边界已经清楚，不能再混
3. `003` 的测试模式必须显式开启，不能再污染正式场景
4. 当前 blocker 已经足够清楚，因此现在交接不会失真
