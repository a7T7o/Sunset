# NPC 主线与支线迁移编年

## 阶段 1：2026-03-16，先把 NPC 从“缺失现场”救回来

### 当时目标

- 恢复 NPC phase2 运行时代码、Prefab、动画、profile 和生成器链路。

### 当时动作

- 从历史 `codex/npc-*` 分支回填 NPC 二阶段代码与资产。
- 同时为了解除编译阻断，顺带处理了 `FarmToolPreview` 兼容口。

### 结果

- 001 / 002 / 003 恢复为可漫游、可聊天、可继续开发的状态。

### 对后续的影响

- 这一步把“NPC 系统存在吗”从问题变成既成事实。
- 也埋下了后续 branch/worktree 漂移审计的种子，因为当时成果并不都落在 clean `main`。

## 阶段 2：2026-03-17，资源链路补齐与 Git 现场纠偏

### 当时目标

- 查清 Prefab / Sprite / Animator 为什么又丢。
- 查清为什么这条线程口径是 NPC，但 Git 现场却漂到 farm 分支。

### 当时动作

- 修复 `Assets/Sprites/NPC/001~003.png.meta`，让 fileID 与 Prefab / 动画重新闭合。
- 只读审计 `git reflog`、`git worktree list`、`git branch -vv`。

### 结果

- 证实真正的 Sprite 丢失根因是 `.meta` 不匹配。
- 证实 NPC 成果一度挂到了 farm 分支，不是稳定落在 NPC carrier 上。

### 对后续的影响

- 后面所有“carrier / continuation branch / main-only”讨论，都是从这次漂移审计延伸出来的。

## 阶段 3：2026-03-20，确认生产场景对 NPC 运行时资产的真实依赖

### 当时目标

- 不是只看 carrier 在不在，而是确认 `main` 里的 `Primary.unity` 到底依赖了哪些 NPC 资产。

### 当时动作

- 对照 `Primary.unity` 的 YAML 引用与历史 NPC 资产，恢复主场景真正依赖的 Prefab / meta / profile / runtime 脚本。

### 结果

- 证实生产场景确实直接引用了 NPC phase2 资源。
- 这轮之后，“NPC 运行时资产已经回到 `main`”成为稳定事实。

### 对后续的影响

- 这一步让 NPC 后续可以真正按 `main-only` 继续推进。

## 阶段 4：2026-03-22，主线从“恢复”切到“返工与定边界”

### 当时目标

- 解决用户直观看到的两类问题：
  - 气泡压脸、样式不到位
  - NPC 是整身 trigger、被玩家顶得太容易

### 当时动作

- 在 `main` 上直接改 `NPCBubblePresenter`、`NPCAutoRoamController`、生成器和三份 Prefab。
- 同时只读审计导航能力边界。

### 结果

- 气泡和碰撞基线在 `main` 上真正落地。
- 也明确写死了职责边界：
  - NPC 只认 NPC 自己的代码
  - 导航底座增强必须由导航线程认领

### 对后续的影响

- 这是整条线程第一次真正从“补文件”进入“用户体验返工”。
- 也是“不要再把导航和 NPC 混着讲”的重要转折点。

## 阶段 5：2026-03-23，长时间 live 迭代，把气泡与 `003` 验证样本真正做顺

### 当时目标

- 把气泡做成可 live 证明的版本。
- 让 `003` 从一次性压测对象，升级成长期验证样本。

### 当时动作

- 连续迭代 `NPCBubblePresenter` 的位置、尾巴方向、内边距、自适应宽高、字数换行逻辑。
- 新增并正规化 `NPCBubbleStressTalker`。
- 补 `NPCBubbleStressTalkerEditor` 与 `NPCAutoRoamControllerEditor`。
- 新增 `NPCSceneIntegrationTool` 和生成器角色分流逻辑。

### 结果

- `003` 既能承担验证样本职责，又不再默认污染正式漫游。
- 气泡从“测试味 UI”推进到可用审美基线。

### 对后续的影响

- 这一步确立了 NPC 后续的两个现实方向：
  - `003` 要长期保留为验证样本
  - 真正下一刀应该进入 scene 集成，而不是无限修气泡

## 阶段 6：2026-03-25，2.0.0 产品化入口成型

### 当时目标

- 不再只修现象，要把 NPC 拉进 2.0 产品化阶段。

### 当时动作

- 建立 `2.0.0进一步落地` 工作区。
- 收口唯一导航契约。
- 新建需求拆分、场景化设计、交互反应设计、系统实施主表。
- 为 `001 / 002 / 003` 建立独立角色化 roam profile。

### 结果

- NPC 不再只是“能动会说”的测试系统。
- 2.0 的角色化、关系、双气泡、受击兼容方向都有了正式入口。

### 对后续的影响

- `NPCV2` 不需要再重新定义大方向，只需要决定“先落哪一刀实现”。

## 阶段 7：2026-03-25 到 2026-03-26，线程主动停在安全边界，准备交接

### 当时目标

- 判断是否还能继续安全推进。
- 如果不能，就把 scene blocker 和接手条件写清楚。

### 当时动作

- 只读复核 shared root、`Primary.unity`、Unity/MCP 基线和热区状态。
- 将“现在不能直接进 `Primary.unity`”写入 memory。
- 评估是否已满足代际交接条件。

### 结果

- 证实当前真正阻塞是：
  - `Primary.unity` 仍 dirty
  - 没有 active lock / 独占写归属
  - shared root 同时处于高活跃 mixed dirty 现场

### 对后续的影响

- 这使得当前线程虽然不能继续写 scene，但已经足够稳定，可直接交给 `NPCV2` 接棒。

## 哪些支线后来升级成主线的一部分

1. `003` 压测入口最初只是测试支线，后来升级成正式验证样本体系
2. 生成器分流和 scene integration tool 最初只是“减少手修”，后来升级成 scene 集成前置基础设施
3. 2.0.0 文档起初像设计支线，后来升级成当前主线的下一代蓝图

## 哪些支线只是噪声、误导或临时绕路

1. 继续把 `codex/npc-roam-phase2-003` 当执行现场
2. 把导航主战场问题继续吞进 NPC 线程
3. 在 scene 窗口未建立前反复讨论“能不能先顺手进 `Primary.unity`”
4. 用更多平行文档代替唯一契约

## 编年一句话总结

这条线程的主线迁移顺序非常清楚：

- 恢复系统
- 修资源链
- 纠 Git 漂移
- main-only 返工
- live 验证样本化
- 2.0 产品化入口
- scene 前置 blocker 写实
- 正式交接给 `NPCV2`
