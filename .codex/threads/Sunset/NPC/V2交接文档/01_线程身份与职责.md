# NPC 线程身份与职责

## 最初创建动机

这条线程最初不是为了做 2.0 产品化设计，而是为了把 Sunset 的 NPC 体系从缺资产、缺 Prefab、缺运行时代码、缺动画引用的混乱现场里救回来，并重新建立：

- NPC 模板生成器
- 001 / 002 / 003 预制体
- 自动漫游
- 环境气泡
- 二阶段行为闭环

也就是说，`NPC` 的原始职责是“把 NPC 从不可运行恢复到可继续开发”。

## 历史上承担过的主要职责

这条线程历史上真正承担过 6 类职责：

1. NPC phase2 运行时资产与脚本恢复
2. PNG `.meta` / Sprite / AnimatorController / Prefab 引用链修复
3. main-only 口径下的气泡视觉、碰撞体与 prefab 基线返工
4. `003` 测试模式正规化，以及生成器 / scene integration tool / inspector 入口收口
5. `2.0.0进一步落地` 文档与角色化 profile 起步
6. scene 写窗口未成立时的边界守门、阻塞写实和交接准备

## 当前真正身份

当前这条线程的真正身份已经发生迁移：

- 从“继续迭代 NPC 功能”
- 转成“为 `NPCV2` 准备可直接接班的稳定现场说明”

换句话说，`NPC` 当前更像一条“主线沉淀与代际交接线程”。

## 当前主要负责范围

`NPCV2` 接棒后，仍应把下面这些视为 NPC 主责范围：

1. `Assets/YYY_Scripts/Controller/NPC/` 下属于 NPC 自身表现、测试模式、角色表达的代码
2. `Assets/222_Prefabs/NPC/001.prefab`、`002.prefab`、`003.prefab`
3. `Assets/111_Data/NPC/` 下的 NPC 角色化 roam profile
4. `Assets/Editor/NPCPrefabGeneratorTool.cs`
5. `Assets/Editor/NPCSceneIntegrationTool.cs`
6. `Assets/Editor/NPCAutoRoamControllerEditor.cs`
7. NPC 2.0.0 设计文档与导航契约文档
8. `Primary.unity` 中属于 NPC 的最小 scene 集成切片：
   - `HomeAnchor`
   - 活动范围
   - 路线 / 停留点
   - 验证样本与正式 NPC 的摆位

## 当前明确不负责范围

`NPC` 线当前明确不负责下面这些内容：

1. `NavGrid2D`、`NavigationLocalAvoidanceSolver`、交通裁决、局部避障、动态占位等导航底座核心
2. `PlayerAutoNavigator.cs` 的主循环、玩家自动导航动态避让与路口礼让
3. `Primary.unity` 当前 dirty 的全局归属判定与 shared root 总治理
4. 与 NPC 无关的共享字体、公共 UI、`TagManager.asset`、`GameInputManager.cs`
5. SpringDay1 侧的外部编译阻断与故事系统问题

## 与其他线程的边界

### 与导航线程的边界

NPC 线程负责：

- NPC 如何消费导航结果
- NPC 自己的状态语义、验证样本、联调验收口径

导航线程负责：

- 玩家 / NPC / NPC 之间的动态避让与真实交通闭环

### 与治理线程的边界

NPC 线程负责：

- 说清楚自己当前现场与 blocker

治理线程负责：

- shared root 占用治理
- 热文件窗口裁定
- 并发模型口径维护

### 与其他业务线程的边界

NPC 线程不应再吞并：

- 玩家输入链
- 放置系统
- SpringDay1 叙事 / UI 主战场
- 字体资源清理

## 哪些事项虽然相关，但不应再由本线继续吞并

1. 为了验证导航而去改导航主循环
2. 为了 scene 集成而顺手治理 shared root 整体脏现场
3. 为了表现验收而继续无限微调气泡参数
4. 为了“先做完再说”去抢写 `Primary.unity` 热区

## 当前最准确的一句话

`NPC` 线程已经完成“恢复、整理、定边界、留路线”的 V1 工作；`NPCV2` 不应再回到“先证明 NPC 系统是不是存在”的老问题，而应直接接“scene 集成前置确认 + 真实落点”。
