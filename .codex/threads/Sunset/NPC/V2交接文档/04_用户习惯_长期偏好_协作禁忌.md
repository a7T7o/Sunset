# 用户习惯、长期偏好与协作禁忌

## 1. 输出与沟通风格

### 稳定偏好

- 说明文字默认中文
- 不要端着，不要绕，最好直接说人话
- 复杂事情可以详细，但先把结论说清
- 最小回执时就按格式，不要顺手加戏

### 用户特别在意

- 你有没有真正理解他刚刚新增或修正的需求
- 你有没有把“旧的正确要求保留，新要求只覆盖有冲突部分”处理清楚

## 2. 主线纪律

### 稳定偏好

- 不要因为中间插入了阻塞处理、治理 prompt、只读核查，就把主线忘了
- 每一轮都要知道自己现在到底是在：
  - 推主线
  - 清 blocker
  - 还是只读审计

### 用户最反感的表现

- 像梦游一样反复忘记自己该做什么
- 只盯最后一句，而丢掉整条历史主线

## 3. main-only 与现场纪律

### 稳定偏好

- 当前真实实现现场默认就是 `main`
- 旧 `codex/*` 分支只可读参考，不应继续当现实执行现场

### 协作禁忌

- 不要擅自切离 `main`
- 不要在 shared root 热区未明确归属时继续写 scene
- 不要在只读委托里顺手做额外事

## 4. Unity / MCP 使用习惯

### 稳定偏好

- 进入 Unity / MCP live 写前，要说清楚：
  - 需要什么实例
  - 最多占多久
  - 只做什么
  - 做完退回什么状态

### 硬禁忌

- 进了 Play Mode 不退回 Edit Mode
- 把 Unity 留在运行态让别人收拾
- 在用户明确说“先别用 MCP / 场景先留给我”时继续打扰 Editor

## 5. 审美标准

### 这不是软要求，是硬验收项

用户对气泡、字体、间距、尾巴方向、动效节奏、整体专业感极度敏感。

真正的验收标准不是：

- 能显示
- 参数变了
- 逻辑通了

而是：

- 看起来真的明显更自然
- 可读
- 不压脸
- 留白合理
- 有你自己的审美判断

### 已经形成的具体偏好

- 箭头必须明确朝下指向 NPC
- 动效可以轻微，但要有生命感
- 文字大小、框体大小、内边距要真实可见，不要只做微调

## 6. 文档治理偏好

### 稳定偏好

- 文件数量少而精
- 同主题尽量维护唯一正式文件
- 不要堆“新版、终版、补充版、二稿”

### 当前直接适用

- 导航协作文档必须只有一个当前版本
- 2.0.0 的设计主文件族可以少，但每份都要像回事

## 7. 责任边界偏好

### 稳定偏好

- 你要主动区分“这是我该认的，还是别的线程该认的”
- 不能把导航、物理、scene 治理、shared root 占用一锅端回自己头上

### 用户明确纠正过的点

- NPC 就只认 NPC 自己的代码和体验
- 导航就认导航核心
- 你要给导航线程写专业 prompt 和回执，但不要替它写代码

## 8. 记忆与连续性偏好

- 不要假设“模型记住了”
- 稳定结论要落 memory
- 交接、prompt、契约、回执都应能被下一代线程直接拿来继续

## 9. 一句话协作禁忌总结

用户最讨厌的，不是你遇到难题，而是：

- 忘主线
- 不认边界
- 不讲人话
- 改了像没改
- 没有审美判断
- 在共享现场里留下别人还得替你收拾的运行态或脏态
