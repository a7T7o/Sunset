## 二、你必须先消化的已判定结论

下面这些，不是让你照抄措辞，而是你必须真正理解并在你的总方案里重新论证出来。

### 1. 用户最初真实顺序一直很稳定

不是：

- 先做一个更通用、更正式、更稳的版本

而是：

1. **先把用户手调 prefab 逐项照抄回运行时**
2. **再在这个正确视觉结果上补体验增强**
3. **最后才讨论如何抽成长期可复用方案**

也就是：

**先抄 -> 再稳 -> 再抽**

### 2. 当前 runtime UI 和手调 prefab 不是同一套东西

而且这不是“感觉不像”，是结构上就决定了不会一样。

因为当前运行入口不是直接实例化用户手调后的两个 prefab，而是：

- `new GameObject`
- `AddComponent`
- `BuildUi()`

这意味着：

- 代码运行时重新造了一套 UI
- 用户后来手调并保存到 prefab 的参数不会自动回流到这套 runtime UI

所以现在的现实不是“prefab 稍微偏了一点”，而是：

**prefab formal-face 和 runtime face 双源并存。**

### 3. 你前面方向不算全错，但顺序错了

你前面提到：

- 纯 `BuildUi()` 长期不合理
- 应该走 `prefab + 绑定代码 + 数据/规则`

这部分方向本身可以保留。

但错在：

- 你在还没把 Phase 1 做对之前，就已经开始往长期抽象滑

这会让用户感受到：

- 你不是在升级
- 你是在跳过他已经调好的结果，重新发明一版

### 4. 正确答案不是三选一，而是分层

不是：

- 只用 prefab
- 或只用运行时代码
- 或只用一套通用代码

而是：

#### 第一层：Prefab 负责视觉模板

负责：

- 层级
- RectTransform
- 锚点
- 字体 / 字号 / 样式 / 留白
- 背景、纸张、按钮底板、视口结构
- 左右栏骨架、双页结构、滚动容器等固定视觉框架

并且这层必须以用户手调 prefab 为唯一真源。

#### 第二层：代码负责动态行为

负责：

- 自适应布局重算
- 内容灌入后的 rebuild
- 滚动内容生成
- 翻页动效
- 制作按钮 / 进度条状态机
- 离台小进度的显隐与刷新

但这层不该再去重新长整棵 UI 壳。

#### 第三层：业务数据负责不同工作台的内容差异

负责：

- 配方
- 材料
- 状态
- 工作台类型差异
- 文案和内容来源

也就是：

**视觉靠 prefab，行为靠代码，差异靠数据。**

### 5. 纯 `BuildUi()` 在这个场景里不合理，但纯 prefab 也不够

你必须能说清楚这两个判断为什么能同时成立。

#### 纯 `BuildUi()` 不合理的原因

不是因为运行时代码 UI 永远不行，

而是因为当前视觉真源来自：

- 用户亲手在运行时微调后的实际结果

在这种前提下，如果代码继续“自己再猜一遍 UI 长什么样”，几乎一定偏。

#### 纯 prefab 也不够的原因

因为用户要的不是静态壳子，而是：

- 自适应
- 日历撕页
- ScrollRect / Viewport / Mask 正确链路
- 制作状态机
- 离台小进度
- 固定锚定不漂

这些都需要代码。

所以你必须能把“为什么不能全靠代码”和“为什么也不能全靠 prefab”同时讲透。
