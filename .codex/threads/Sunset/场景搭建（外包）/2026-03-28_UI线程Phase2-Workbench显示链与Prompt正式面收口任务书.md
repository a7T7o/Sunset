## 二、这轮唯一主刀

### A. Workbench：把“能点但看不见”和“右侧材料区还像半成品”一起收掉

你不能再把 Workbench 只理解成“左边显示一下就行”。

这轮 Workbench 要一起过线的其实是**同一面板的两个失败点**：

1. **左列 RecipeColumn 真实可见**
   - 不是 hierarchy 里有
   - 不是日志里说生成了
   - 而是玩家在屏幕上真的看得到：
     - recipe 行
     - 图标
     - 文字
     - 至少当前可选项

2. **右侧 Materials / 所需材料 区域回到正式样子**
   - 现在用户明确看出来：右侧“所需材料”这一块还是没收好
   - 问题不只是数据数值，而是：
     - 信息层级
     - 留白
     - 行距
     - 数字 / 材料项的呈现方式
     - 和整个面板的协调性
   - 这轮不要求你顺手开新系统，但要求你把它从“半成品感”收回到 **正式 UI 感**

### B. Prompt：任务栏重新回到用户手调 prefab 的 formal-face

这轮 Prompt 不是只要求“不再乱飞”。

而是要让用户一眼看上去，不会再觉得它是“临时拼装面”。

重点仍然是：

- 标题条
- 引导文
- 任务行
- 小方块 / 勾选块
- 底部说明区
- 卡片留白和层级

全部都要重新向用户手调 prefab 靠拢。

## 三、这轮你必须怎么判断问题，不要再猜错

### 1. Workbench 左列的判断口径

用户已经明确补充：

- 左侧什么都没有显示
- 但可以点击、可以切换
- 右侧内容会变化

这已经把问题性质说得很清楚了：

**逻辑和内容在，真正坏的是显示链 / 配置链。**

所以你这轮排查重点必须压在：

- `RecipeColumn`
- `Viewport`
- `Content`
- `Mask / ScrollRect`
- recipe item 节点父子关系
- icon `Image` 是否被正确绑定 / 激活 / 赋 sprite
- 文本颜色 / alpha / size / overflow / mask
- item 行是否被挤出 viewport
- item 行是否长到了错误父节点
- item 行是否尺寸为 0 或被 layout 吃没

你这轮收的不是“节点存在”，而是：

**左列在最终 GameView 里真实可见。**

### 2. Workbench 右侧 Materials 的判断口径

用户这次已经不只是抱怨“左边空”。

还明确指出：

**右侧所需材料这一块，样式也没有做好。**

所以你不能把右侧当“数据在就算过”。

这轮右侧至少要做到：

- `所需材料` 标题层级清楚
- 材料项 / 数量 / 预计耗时 / 数量区 / 按钮区之间关系清楚
- 不再有明显“半成品占位块感”
- 和用户手调 prefab 的整体正式面协调

注意：

- 这轮不是让你开新交互系统
- 也不是让你把按钮状态机全部重构
- 只是要求你把当前**右侧正式面**收回到用户可接受的程度

### 3. Prompt 的判断口径

Prompt 这轮仍然不能走“横向改壳体”。

用户又强调了一次偏好：

- 自适应只允许 **上面内容变多，下面往下推**
- 不要左右宽度乱变
- 不要整体位置、宽高、脸型又被 runtime 改掉

所以 Prompt 这轮还是要守这个边界：

- 外壳 formal-face 继续以手调 prefab 为真源
- 动态布局只做内容层纵向下推
- 不准再把整体卡片改回代码味很重的 runtime 壳

## 七、这轮之外，但必须记住的后续需求

这部分**现在不要落实现**，只允许记入你自己的工作区 `memory.md`，作为后续单独切片。

用户刚补的要求非常重要，后面必须做，但**不是这一轮**：

1. 现在 gizmos 下 NPC 靠近时头顶只有一个小点，这种提示几乎不可感知，不算过线。
2. 交互提示必须小巧、灵巧、专业，且视觉与逻辑一致。
3. 同一时刻只能触发**最近的一个**可交互目标提示：
   - NPC
   - 工作台
   - 其他可交互物
   都要统一仲裁。
4. 不能出现 NPC 和工作台同时都提示。
5. 如果都走同一个 `E` 键交互，那么真正响应的也必须是最近的那个。
6. NPC 上 `E` 交互时，剧情对话优先级高于非正式气泡聊天。
7. 聊天气泡速度后续要放慢一点，调到正常可读速度。

你这轮只需要把这段需求记入工作区记忆，等待后续单独立项。
