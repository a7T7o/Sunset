# Sunset

一个基于 Unity 6 的 2D 农场模拟项目，当前已落地时间推进、背包与 Hotbar、农田与工具、放置与箱子、存档、导航、遮挡，以及 `spring-day1` 剧情验证等核心链路。  
我在其中主要负责项目推进、交互方向把控、验收标准建立，以及多智能体协作条件下的任务收口与质量治理。

## 项目概览

`Sunset` 不是单点功能展示型 Demo，而是一个持续开发中的多系统联动项目。

当前项目的复杂度主要来自几类真实耦合：

- 农田系统需要同时和工具、种子、树苗、放置、输入链、Tooltip 与运行时状态联动。
- 箱子系统不只是容器 UI，还牵涉库存实例态、拖拽交换、放置、存档和动态对象重建。
- 导航与遮挡不是单脚本调参，而是要兼顾场景挂载、运行时行为、动态代理和体感验收。
- `spring-day1` 剧情验证不只是对话弹窗，还涉及导演层、阶段推进、UI 表现、工作台与床交互、运行时状态切换。

因此，这个项目的推进重点一直不只是“继续加功能”，而是把多条系统主线逐步收口到可验证、可交接、可持续迭代的状态。

## 项目亮点

- 多系统真实联动：时间、库存、农田、放置、箱子、导航、遮挡和剧情并不是分散孤岛，而是在真实场景中互相影响。
- 进入项目化阶段：重点不再只是功能堆叠，而是体验一致性、问题收口、验证方式和持续迭代能力。
- 有明确的质量边界：很多问题不是“能跑就算完成”，而是要进一步确认可复现、可回退、可验收、可接手。

## 我的作用

我在这个项目中更接近下面四类职责的组合，而不是“包办全部实现的人”：

- 项目推进负责人：持续定义主线目标、阶段边界、任务顺序和恢复点。
- 产品 / 交互方向把控者：对放置体验、Tooltip、导航体感、UI 呈现、剧情观感等持续纠偏。
- 质量门槛建立者：把“功能可用”进一步收紧为“可验证、可复现、可回退、可交接”。
- 多智能体协作调度者：为不同线程定义任务边界、交付格式、回执方式和风险控制规则。

## 核心系统一览

- 时间 / 季节 / 运行时状态：负责昼夜推进、阶段切换和部分系统驱动。
- 背包 / Hotbar / 物品：覆盖物品显示、快捷栏、拖拽、装备、Tooltip 和部分实例态链路。
- 农田 / 工具 / 放置：涵盖耕作、播种、树苗、预览、放置边界和交互反馈。
- 箱子 / 存档 / 动态对象：处理容器交互、库存同步、实例态与持久化之间的关系。
- 导航 / 遮挡 / 世界交互：负责移动体寻路、动态规避、树木遮挡与场景级行为验证。
- `spring-day1` 剧情验证链：用于验证导演层、对话节奏、教学链、工作台和睡觉收束等流程。

## 我如何推进这个项目

这个项目的推进方式，核心不是“想到什么就让 AI 先写”，而是以下四步循环：

1. 先定义阶段目标和真正的问题边界。
2. 用工作区文档、主线说明和记忆体系固定上下文。
3. 让不同线程或不同智能体按明确职责推进实现、验证和复核。
4. 用现场验收、专项验证和最小 checkpoint 持续纠偏。

这里的重点不是“使用了 AI”，而是把 AI 协作变成一种可控的项目推进手段，而不是新的混乱来源。

## 代表性推进案例

### 1. 农田交互从局部修补升级为全局交互治理

面对放置、Toolbar、抖动锁定、耐久、精力、Tooltip、箱子递归等交错问题，没有继续把它们拆成一堆松散小 bug，而是把问题提升成统一交互主线来重新收口。

### 2. `spring-day1` 从单点剧情验证推进到导演层承载

关注点不只停在“对话能不能弹出”，而是继续推进导演层、工作台、床交互、低精力反馈、回血表现和阶段收束，让剧情验证真正具备完整走通的骨架。

### 3. 导航问题从局部 patch 重新提升到结构问题

当现场体验反馈“玩家面对动态 NPC 仍像推土机”时，没有接受表面统一规则就算收工，而是要求回到 live 场景审计、承载对象裁决、共享执行层和最终行为终验。

## 当前状态

当前项目更适合被描述为：

- 持续开发中的中高复杂度 Unity 项目。
- 已经具备多条真实系统主链，不是空壳 Demo。
- 正在从“功能实现”进一步走向“体验打磨、系统收口与项目化治理”。

这意味着它不是一个已经上线封板的商业成品，但也远不只是概念验证。
