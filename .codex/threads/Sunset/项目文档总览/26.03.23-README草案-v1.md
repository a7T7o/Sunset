# README 草案 v1

> 这是一份面向仓库根 `README.md` 的第一版草案，用于求职展示与项目说明，不是最终发布稿。

## Sunset

一个基于 Unity 6 的 2D 农场模拟项目。  
当前项目已经落地了时间推进、背包与 Hotbar、农田与工具、放置与箱子、存档、导航、遮挡、以及 spring-day1 剧情验证等多条真实系统主链；我在其中主要负责项目推进、交互方向把控、验收标准建立，以及多智能体协作治理。

## 项目概览

`Sunset` 不是只做单一功能的课堂练习型项目，而是一个持续演进中的多系统耦合项目。

项目当前的复杂度主要来自几类真实联动：

- 农田系统要同时和工具、种子、树苗、放置、输入链、Tooltip、运行时状态联动。
- 箱子系统不只是容器 UI，还牵涉库存实例态、拖拽交换、放置、存档和对象重建。
- 导航与遮挡不是单脚本调参，而是需要兼顾场景挂载、运行时行为、动态代理和体感验收。
- spring-day1 剧情线不只是对话播放，还涉及导演层、阶段推进、UI 表现、工作台与床交互、运行时状态切换。

因此，这个项目的推进重点一直不只是“继续加功能”，而是把多条系统主线逐步收口到可验证、可交接、可持续迭代的状态。

## 当前系统版图

当前已经形成或正在持续打磨的主线主要包括：

- 时间 / 季节 / 运行时状态：负责昼夜推进、阶段切换和部分系统驱动。
- 背包 / Hotbar / 物品：覆盖物品显示、快捷栏、拖拽、装备、Tooltip 和部分实例态链路。
- 农田 / 工具 / 放置：涵盖种植、耕作、树苗、预览、放置边界和交互反馈。
- 箱子 / 存档 / 动态对象：处理容器交互、库存同步、实例态与持久化之间的关系。
- 导航 / 遮挡 / 世界交互：负责移动体寻路、动态规避、树木遮挡与场景级行为验证。
- spring-day1 剧情验证链：用于验证导演层、对话节奏、教学链、工作台和睡觉收束等流程。

## 为什么这个项目值得看

这个项目的价值不只在于“系统多”，而在于它已经进入了一个更真实的开发阶段：

- 不是只写功能，还在持续处理系统边界冲突和交互一致性问题。
- 不是只做静态文档，还在建立可追溯的工作区、记忆和验收体系。
- 不是单线程推进，而是在多 AI 协作的条件下管理并发、热文件、回执和阶段收口。

换句话说，这个项目同时体现了产品推进、交互打磨、复杂系统整合和协作治理能力。

## 我的角色

我在这个项目中最核心的角色不是“单纯提需求的人”，也不是“把所有代码都亲自写完的人”，而是更接近下面四类职责的组合：

- 项目推进负责人：持续定义主线目标、阶段边界、任务顺序和恢复点。
- 产品 / 交互方向把控者：对玩法边界、Tooltip、放置体验、导航体感、剧情观感等做持续纠偏。
- 质量门槛建立者：要求把“能跑”进一步收紧为“可验证、可复现、可回退、可交接”。
- 多智能体协作调度者：为不同线程定义职责、Prompt、回执格式、锁与 checkpoint 规则，控制并发推进风险。

## 我如何推进这个项目

这个项目的推进方式，核心不是“想到什么就让 AI 先写”，而是四步循环：

1. 先定义阶段目标和真正的问题边界。
2. 用工作区、主文档和记忆把上下文固定下来。
3. 让不同线程或不同智能体按明确职责推进。
4. 用现场验收、专项验证和收口规则持续纠偏。

这套方式的重点不是“把 AI 用得很多”，而是把 AI 协作变成一种可控的项目推进手段。

## 代表性推进案例

### 1. 农田交互从局部修补升级为全局交互治理

面对放置、Toolbar、抖动锁定、耐久、精力、Tooltip、箱子递归等交错问题，没有继续把它们拆成一堆松散小 bug，而是把主线升级为 `全局交互V3`，先完成 docs-first 的需求收口和统一任务清单，再转入阻塞清除式实现。

### 2. spring-day1 从单点剧情验证推进到导演层承载

不仅关注对话能不能弹出，还继续推进导演层、工作台、床交互、低精力反馈、回血表现、运行时阶段收束等链路，让剧情验证真正开始具备“可完整走通”的骨架。

### 3. 导航问题从局部 patch 重新提升到结构问题

当用户现场反馈“玩家面对动态 NPC 仍像推土机”时，没有接受“共享了一部分规则就算统一导航”的假收口，而是要求重新做 live 场景审计、承载对象裁决、S4 共享路径执行层收口和最终行为终验。

## 多 AI 协作与治理方式

这个项目的一个差异化特点，是在真实开发中引入了多智能体协作。

这里的重点不在“使用了 AI”，而在于如何把 AI 从单次工具使用，推进到可管理的协作体系：

- 通过线程分工减少上下文混乱。
- 通过 Prompt 和回执格式约束交付内容。
- 通过工作区记忆和主文档减少历史丢失。
- 通过 Git 白名单同步、热文件控制和 live 验证窗口约束风险。

这种方式让项目在复杂度上升后，仍能保持一定程度的可追溯和可收口。

## 当前状态

当前项目更适合被描述为：

- 持续开发中的中高复杂度 Unity 项目。
- 已经具备多条真实系统主链，不是空壳 demo。
- 正在从“功能实现”进一步走向“体验打磨、系统收口与项目化治理”。

这意味着它不是一个已经上线封板的商业成品，但也远不只是概念验证。
