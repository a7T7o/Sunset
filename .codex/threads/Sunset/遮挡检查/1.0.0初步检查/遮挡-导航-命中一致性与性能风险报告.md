# 遮挡-导航-命中一致性与性能风险报告

## 1. 一致性审查

## 1.1 已对齐部分
- 玩家中心点使用 Collider 中心（符合规范）：
  - `OcclusionManager` 取玩家中心与 bounds：`Assets/YYY_Scripts/Service/Rendering/OcclusionManager.cs:430-439`
  - `GameInputManager` 点击逻辑也优先用玩家 Collider 中心：`Assets/YYY_Scripts/Controller/Input/GameInputManager.cs:985-987`

- 树木状态变化会联动导航刷新（符合树木规范）：
  - `TreeController` 状态变化后触发 `OnRequestGridRefresh`：`Assets/YYY_Scripts/Controller/TreeController.cs:1607-1611`, `1655`

## 1.2 存在偏差（重点）
- **命中链策略分裂**：
  - 工具命中 `PlayerToolHitEmitter` 使用 `GetColliderBounds()`（`Assets/YYY_Scripts/Combat/PlayerToolHitEmitter.cs:223`）
  - 点击交互 `GameInputManager` 使用 `GetBounds()`（Sprite bounds）（`Assets/YYY_Scripts/Controller/Input/GameInputManager.cs:1011-1013`）

这会导致同一目标“可点到但砍不到”或“砍到但点交互判定边缘不同”的体验不一致。

## 2. 性能风险评估

## 2.1 高风险
1. **遮挡检测全量遍历**
   - `DetectOcclusion` 每周期遍历 `registeredOccluders`（`OcclusionManager.cs:455-563`）。
   - 物体规模增大时，CPU 开销线性上升。

2. **像素采样开销**
   - `ContainsPointPrecise` 包含坐标转换和 `GetPixel`（`OcclusionTransparency.cs:354-435`）。
   - `CalculateOcclusionRatioPrecise` 多点采样会进一步放大成本（`443-464`）。

3. **导航网格全图重建**
   - `NavGrid2D.RefreshGrid -> RebuildGrid` 是全量重建（`NavGrid2D.cs:92-95`, `119-163`）。
   - 树木状态频繁变更时，容易形成“遮挡+导航”双热点叠加。

## 2.2 中风险
4. **树林智能边缘逻辑复杂**
   - 凸包、边界距离、保底机制共存（`OcclusionManager.cs:912-1109`），正确性强但维护成本高。

5. **调试输出放大问题**
   - 多个脚本在 debug 开启时有高频日志，可能掩盖真实性能问题（尤其编辑器环境）。

## 2.3 低风险
6. **透明更新有短路保护**
   - `SetOccluding` 首行状态相同直接 return（`OcclusionTransparency.cs:145`），避免无效写入。

---

## 3. 风险优先级（建议）

- P0：统一命中判定基准（至少保证“工具命中”与“点击交互”视觉边界一致）。
- P1：限制高密度场景下像素采样范围，避免全量启用。
- P1：降低导航全图重建触发频率（尤其连续状态变化时）。
- P2：对遮挡对象做更积极的空间预筛选，减少每周期候选集合。

---

## 4. 结论

当前遮挡系统功能完整度高，且与树木、预览、导航形成了真实生产级联动；
但系统体验的一致性风险主要集中在“命中判定标准分裂”，性能风险主要集中在“全量扫描 + 可选像素采样 + 全图导航重建”三者叠加。
