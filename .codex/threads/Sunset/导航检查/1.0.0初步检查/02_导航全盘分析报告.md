## A. 逻辑框架（架构视图）

### A1. 主路径：输入驱动导航
1. 右键输入进入 `HandleRightClickAutoNav()`（GameInputManager.cs:929）
2. 用 `OverlapPointAll` 收集点击命中（1035）
3. 若命中 `IInteractable`：按优先级/距离筛选后 `HandleInteractable(...)`（1121）
4. 调 `autoNavigator.FollowTarget(...)` 发起跟随导航（1152）
5. 到达后再做一次距离复核 `TryInteractWithDistanceCheck(...)`（1188）
6. 若无交互对象：`autoNavigator.SetDestination(world)`（1113）

这条链明确了“导航只是中间层，真正行为落点在交互判定和业务执行”。

### A2. 执行器路径：PlayerAutoNavigator
- `SetDestination` / `FollowTarget` 统一收口到 `BuildPath()`（PlayerAutoNavigator.cs:110,136,496）
- `BuildPath()` 调 `NavGrid2D.TryFindPath(...)`（546）
- 寻路后做 `SmoothPath()` 与 `CleanupPathBehindPlayer()`（570,605）
- 移动过程有 LOS 复核：`HasLineOfSight` + `CircleCast`（410,441）

### A3. 网格内核路径：NavGrid2D
- `RebuildGrid()` 是地图状态快照入口（NavGrid2D.cs:119）
- 先 `Physics2D.SyncTransforms()`（124）
- 再按采样逻辑判定格子阻挡（`OverlapCircleAll`，266）
- `TryFindPath` 执行 A*（300），并在 open 集合上反复排序（328）

## B. 引用关系与耦合图（代码事实）

### B1. 谁调用了导航
- 输入模块：GameInputManager -> `SetDestination` / `FollowTarget`
- 放置模块：PlacementManager -> PlacementNavigator -> PlayerAutoNavigator
- 农田流程：GameInputManager 的农田协程链也走 autoNavigator

### B2. 谁触发了网格重建
- 树木：TreeController 在碰撞更新后触发 `OnRequestGridRefresh`（TreeController.cs:1655）
- 箱子：ChestController 在初始化/状态变化后触发（ChestController.cs:556,558,685,686）

### B3. 谁“读取”了导航内部数据
- CloudShadowManager 读取 NavGrid 信息用于渲染区域推导，但通过反射读取私有字段（CloudShadowManager.cs:334,336）

## C. 处理方式分析（做得好的地方）

1. **AutoSync 关闭 + 手动 SyncTransforms**
   - 配置是 `m_AutoSyncTransforms: 0`（Physics2DSettings.asset:46）
   - 导航重建前显式同步（NavGrid2D.cs:124）
   - 这是对性能与一致性的平衡做法：平时省开销，关键点手动对齐。

2. **交互统一接口化（IInteractable）**
   - 输入层不再写死“作物/箱子/树”分支，改为统一协议
   - 作物收获已接入（CropController.cs:33,157）

3. **放置导航采用 ClosestPoint 一致方案**
   - 目标计算、到达判定、近距判定统一到同一个几何模型（PlacementNavigator.cs:123,212,245）
   - 明显减少“看起来到了但逻辑判没到”的错位问题。

4. **动态障碍刷新链明确**
   - 树/箱子都能显式发导航刷新事件，避免长期脏网格。

## D. 性能全盘分析（热点与成本）

### D1. 网格重建成本
- 热点：`OverlapCircleAll`（NavGrid2D.cs:266）
- 含义：每次重建都要做大量物理查询，地图越大、采样越密，成本越高。

### D2. A* 热点
- 热点：`open.Sort(...)`（NavGrid2D.cs:328）
- 含义：路径搜索迭代时反复全量排序，长路径/复杂障碍下会放大 CPU 开销。

### D3. 运动期额外检测
- `HasLineOfSight` + `CircleCast`（PlayerAutoNavigator.cs:410,441）
- 含义：路径平滑与动态重算质量更高，但会引入持续物理检测成本。

### D4. 刷新风暴风险
- `OnRequestGridRefresh` 是静态事件，且多模块可触发（NavGrid2D.cs:37）
- 当树成长、箱子移动、动态重建密集发生时，可能短时间反复重建网格，造成帧抖动。

## E. 风险分级（按影响排序）

### P1（高）- 渲染对导航私有字段反射耦合
- 证据：CloudShadowManager.cs:334,336
- 风险：字段改名/封装调整后，渲染区域推导可能失效且编译期无保护。

### P1（高）- 网格重建触发缺乏“合并节流”
- 证据：静态刷新事件 + 多源触发
- 风险：性能抖动集中在“状态频繁变更帧段”。

### P2（中）- 输入总线过重
- 证据：GameInputManager 同时管理输入、筛选、导航、交互、农田协程（929~1828 多段）
- 风险：后续功能叠加时可维护性与回归成本上升。

### P2（中）- 导航正确性强依赖刷新时机
- 证据：箱子处已有“Load 先于 Start 的场景说明”（ChestController.cs:589 注释）
- 风险：生命周期边缘场景处理不当会出现短时脏网格。

## F. 建议路线（不做超范围改造，仅给落地优先级）

1. **先做刷新合并（最高性价比）**
   - 对 `OnRequestGridRefresh` 触发端做“本帧只重建一次”或“短时间窗口合并”。

2. **再做 A* open 容器优化**
   - 将 `open.Sort` 方案替换为更低复杂度的数据结构。

3. **去反射耦合**
   - NavGrid2D 暴露稳定只读接口（边界/尺寸），CloudShadowManager 不再反射私有字段。

4. **输入职责收口**
   - 逐步把 `GameInputManager` 内导航/交互策略下沉为独立协作模块。

## G. 验证建议（你可直接用来安排下一轮）

- 场景级验证：
  - 箱子放置/推动后，导航阻挡是否同步变化
  - 树成长阶段变化后，路径是否绕行最新碰撞形态
  - 放置导航到达判定是否稳定触发
- 压力级验证：
  - 使用 `NavGrid2DStressTest` 场景对重建频率和寻路频率做分离压测
- 监控指标：
  - 单次 RebuildGrid 耗时
  - 每秒重建次数
  - TryFindPath 平均耗时与峰值

## H. 总结结论

你项目当前导航系统已经形成“统一交互协议 + 执行器 + 网格内核 + 动态刷新”的完整链路，架构方向是对的，历史关键问题（放置到达一致性、动态障碍同步）也已有修复基础。当前最值得优先处理的不是功能缺失，而是**性能稳定性（重建触发与寻路热点）和跨系统硬耦合（反射读取私有字段）**。
