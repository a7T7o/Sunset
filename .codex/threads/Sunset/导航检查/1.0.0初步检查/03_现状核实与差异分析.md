## 2. 我确认无误的当前主链

### 2.1 输入主链

当前项目里的主导航入口仍然是：

`GameInputManager.HandleRightClickAutoNav()`  
→ 命中筛选（`Physics2D.OverlapPointAll` + `IInteractable`/Tag/Layer）  
→ `HandleInteractable()`  
→ `PlayerAutoNavigator.FollowTarget()` / `SetDestination()`  
→ 到达后 `TryInteractWithDistanceCheck()`

这条链现在有三个明确特征：

1. 交互优先于纯导航。
2. `IInteractable` 已经成为统一契约，而不是继续在 `GameInputManager` 里写死“箱子/作物/NPC”分支。
3. 交互距离计算已经整体收口到 `ClosestPoint` 思路，而不是目标中心点。

### 2.2 执行器主链

`PlayerAutoNavigator` 当前不是简单“点到点移动器”，而是一个带状态隔离和几何补偿的执行层：

- `FollowTarget()` 里会先 `ForceCancel()`，再通过 `_navToken` 隔离旧回调。
- 目标点不再直接使用 `target.position`，而是 `CalculateClosestApproachPoint()` → `GetTargetAnchorPoint()`。
- 停止距离不再硬编码，而是优先从 `IInteractable.InteractionDistance` 推导。
- 路径构建之后还会做：
  - `SmoothPath()`
  - `CleanupPathBehindPlayer()`
  - `HasLineOfSight()` 视线跳点
  - `CheckAndHandleStuck()` 卡住重建

我认为这说明当前导航执行层已经明显偏“实战型补丁累积版本”，不是一个纯净但脆弱的初版。

### 2.3 网格主链

`NavGrid2D` 的核心事实：

- 主场景 `Primary.unity` 实挂在 `Systems` 根下。
- 场景参数为：
  - `autoDetectWorldBounds = true`
  - `cellSize = 0.5`
  - `probeRadius = 0.5`
  - `obstacleMask = 0`
  - `obstacleTags = Interactable / Building / Tree / Rock / Placed`
- `RebuildGrid()` 会先调用 `Physics2D.SyncTransforms()`，这和 `ProjectSettings/Physics2DSettings.asset` 中 `m_AutoSyncTransforms: 0` 是配套关系。

换句话说，当前导航阻挡判断主要依赖“Tag + 物理探测”，不是 LayerMask 主导。

## 5. 我这次新增确认出的关键问题

### P1：当前实现并不是真正的 Zero GC

这是我认为本轮最需要被单独钉住的点。

代码里存在两个容易误判的地方：

- `NavGrid2D.IsPointBlocked()` 使用的是 `Physics2D.OverlapCircleAll(...)`
- `GameInputManager.HandleRightClickAutoNav()` / 收获检测使用的是 `Physics2D.OverlapPointAll(...)`

虽然 `NavGrid2D` 里有 `_colliderCache`，但当前写法是“先拿到 `OverlapCircleAll` 返回的新数组，再复制到缓存里”。  
缓存避免的只是后续临时容器反复创建，不是 API 本身返回数组的分配。

官方文档也明确写了：

- [Physics2D.OverlapCircleAll](https://docs.unity3d.com/ScriptReference/Physics2D.OverlapCircleAll.html)  
  “This function will allocate memory for the returned Collider2D array.”
- [Physics2D.OverlapPointAll](https://docs.unity3d.com/ScriptReference/Physics2D.OverlapPointAll.html)  
  同样说明会为返回数组分配内存。

这会带来两个直接结论：

1. `NavGrid2DStressTest.cs` 当前的“Zero GC 压测”目标，与 `NavGrid2D.cs` 的真实实现不一致。
2. `RebuildGrid()` 在大地图重建时，真正的热点不只是 CPU，还有大量数组分配压力。

旧稿把热点主要落在 `open.Sort()` 和查询次数上，但从这次核实看，GC 风险至少应该和排序热点同级，甚至更靠前。

### P1：刷新链仍然缺少统一节流

这部分旧稿判断仍然成立，而且我这次更确认它是“结构性问题”：

- `TreeController` 触发刷新：延迟 0.2 秒 + 可取消前一次 `Invoke`
- `ChestController` 触发刷新：初始化一帧后直接发事件，状态切换时也会发
- `NavGrid2D` 自己对静态事件没有全局合并逻辑

也就是说：

- 单个树对象内部有“自我合并”
- 但多个树 / 箱子 / 其他未来动态障碍同时变化时，系统层没有“本帧只重建一次”的统一门

所以旧稿说“刷新风暴”是对的，但这次我会把它定义得更具体：

> 当前缺的不是单模块延迟，而是跨模块、跨对象、跨帧段的统一刷新调度器。

### P1：文档和代码已经出现一处可确认的漂移

`chest-interaction.md` 第 91-93 行写的是：

- 点击箱子 → 自动导航到交互距离 → 到达后自动打开
- 导航目标点 = Collider 底部中心偏移

但现代码已经不是这套逻辑：

- `GameInputManager.GetTargetAnchor()` 使用 `Collider.ClosestPoint(playerPos)`
- `PlayerAutoNavigator.GetTargetAnchorPoint()` 也使用 `ClosestPoint`
- `PlacementNavigator` 同样基于 `Bounds.ClosestPoint()`

所以箱子系统文档现在已经落后于代码演进。

这不是功能 bug，但它会带来两个问题：

1. 后续人如果按 steering 文档判断“正确行为”，会误以为代码偏了。
2. 导航整改时容易把已经统一好的 `ClosestPoint` 链路又改回去。

### P2：`GameInputManager` 仍然过重，但它已经不是纯粹“坏味道”

旧稿对 `GameInputManager` 的担心没有错，但我这次判断更细一点：

- 它确实很重，既做输入、又做交互、又做农田状态机、又做导航桥接。
- 但当前不是“完全无序的大杂烩”，而是已经把通用交互契约、农田队列、放置模式切换逐步往结构化方向推了一半。

因此我认为它的真实状态是：

> 已经超重，但还没烂到必须先拆才能继续做事。

如果后续要整改导航，优先级仍应排在：

1. 先修 `NavGrid2D` 的查询/刷新问题
2. 再考虑把 `GameInputManager` 的导航桥接层抽出来

## 6. 我对当前系统成熟度的判断

如果只从“有没有形成完整闭环”来看，当前导航系统已经不是脆弱原型，而是成熟度中等偏上的业务系统：

- 有统一交互契约
- 有运行时最近点逻辑
- 有放置导航专用几何模型
- 有树/箱子动态障碍刷新链
- 有场景实挂
- 有 Prefab 大规模实际接入
- 还有专门的 Editor 辅助面板

但如果从“是否已经进入稳定维护期”来看，我的结论仍然是否：

- 运行正确性已经基本可用
- 性能与治理一致性还没真正收口

更直白地说：

> 现在最需要处理的已经不是“导航能不能用”，而是“导航什么时候会抖、为什么会抖、文档和实现是否还能对齐”。

## 7. 我建议后续主线怎么接

如果这个线程继续往下做，我建议下一轮直接按下面顺序推进：

1. 先做一次“查询分配审计”
   - 把 `OverlapCircleAll` / `OverlapPointAll` 的调用点全部列清
   - 区分“高频每格调用”“每帧调用”“每次输入调用”
2. 再做“刷新调度器方案”
   - 目标不是改树或箱子单模块
   - 而是在 `NavGrid2D` 层补统一合并入口
3. 最后做“文档回写”
   - 至少修正 `chest-interaction.md` 对箱子导航目标点的描述
   - 同时把这次子工作区 memory 作为后续 requirements/design 的起点

## 8. 一句话结论

从我这次重新核实的结果看，线程旧稿抓住了大框架，但漏掉了三个对后续决策更关键的事实：

- 当前主场景里 `CloudShadowManager` 的反射路径是潜在风险，不是当下主风险
- `PlacementNavigator` 是运行时创建，不是场景静态节点
- `NavGrid2D` 当前并不是真正 Zero GC，这比旧稿里写的排序热点更值得优先处理

这三个点，会直接改变后续整改的优先级排序。
