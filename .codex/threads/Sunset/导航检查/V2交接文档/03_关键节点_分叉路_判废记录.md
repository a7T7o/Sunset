# 关键节点、分叉路与判废记录

## 关键节点 1：共享基础设施落地后，仍被用户一票打回
- 旧判断：
  - “共享契约、注册表、solver 接上了，应该已经开始变好”
- 为什么这是误判：
  - 结构接线不等于真实右键入口的行为已经自然通过
- 当前改判：
  - 任何结构 checkpoint 都必须接受体验线单独审判

## 关键节点 2：把“最坏回归已压掉”过早外推成总体体验恢复
- 旧判断：
  - 某轮 `RealInputPlayerAvoidsMovingNpc` 看起来过了，就能宣布主问题被压掉
- 为什么这是误判：
  - 用户现场仍明确感知到：
    - 保护罩
    - 很远就停
    - 被围抽搐
- 当前改判：
  - 用户现场高于局部 runner 成果
  - 单场通过不能代表整条体验线已过

## 关键节点 3：曾把第一责任点错误压成“只剩 NPC 提前停摆”
- 旧判断：
  - 玩家推着 NPC 走已压掉，现在只剩 NPC 自己的到达/停止语义
- 为什么这是误判：
  - 用户随后 fresh 复测直接否定了这条外推
- 当前改判：
  - 直到 5 组同轮 fresh live 跑实之前，不能把问题过早缩成单一 NPC runtime 问题

## 关键节点 4：`TrafficArbiter / MotionCommand` runtime 接线被判废
- 旧路线：
  - 把 runtime 往 `TrafficArbiter + MotionCommand` 新链上推
- 为什么判废：
  - 用户和治理都明确指出：
    - 结构线与体验线混验
    - 新骨架没有转化成可验收结果
  - 当前 runtime 更诚实、更稳定的基线是旧 solver 直出链
- 当前结论：
  - 不要让 `导航检查V2` 再回头把它当本轮主刀

## 关键节点 5：继续泛调 solver 参数已被事实证明收益极低
- 旧路线：
  - 遇到推土机、保护罩、抽搐，就继续调：
    - avoidance shell
    - forward fix
    - slowDownWeight
    - repath threshold
- 为什么判废：
  - 最新有效收敛说明：
    - `ShouldRepath` 已经会出现
    - 但 detour owner 没稳定接管执行层
  - 在握手断裂没修前，继续调 solver 只是让 Layer C 更会算，不能让 Layer D/E 真执行
- 当前结论：
  - solver 不是完全不能碰
  - 但它不再是当前第一责任点

## 关键节点 6：detour 底座“看起来存在”，但 runtime 闭环其实未成立
- 容易误读的表象：
  - `NavigationPathExecutor2D` 已经有：
    - `TryCreateDetour`
    - `TryClearDetourAndRecover`
    - override waypoint state
- 为什么这会误导：
  - runtime controller 当前没有真正把 `TryClearDetourAndRecover()` 接进去
  - 玩家/NPC controller 里仍有多处旧分支直接把 detour 意图吞掉
- 当前改判：
  - 不能因为底座 API 存在，就宣称 detour lifecycle 已完成

## 明确判废的方向
1. 把 `TrafficArbiter / MotionCommand` 当成当前 runtime 主线
2. 把 `probe` 或单组 runner 绿灯当成最终体验验收
3. 在 `ShouldRepath` 已经出现后继续泛调 solver 当主刀
4. 让导航线程顺手清 `Primary.unity` 或 unrelated mixed dirty
5. 遇到 compile blocker 或 external dirty 就停止导航热区推进

## 必须保留原始判断逻辑，而不能只留结论的点
1. 为什么“shared root 已可编译”不等于“可以 claim real-input 通过”
2. 为什么“detour API 存在”不等于“owner 已稳定接管执行层”
3. 为什么“用户现场否定”可以直接推翻本线程上一轮的乐观外推
4. 为什么当前不该再漂回 solver 参数或大架构复盘

## 给 `导航检查V2` 的分叉提醒
- 接手后最容易重蹈覆辙的不是“代码看不懂”，而是：
  - 又被症状带着回去调 solver 参数
  - 又被结构执念带回 `TrafficArbiter / MotionCommand`
  - 又因为底座里有 clear/recover API 就误判成 runtime 已接好
