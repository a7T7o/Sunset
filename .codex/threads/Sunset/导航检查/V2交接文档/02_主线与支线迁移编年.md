# 主线与支线迁移编年

## 阶段 0：从“局部 bug”升级为“统一导航重构”
- 当时目标：
  - 建一个不会迷失的唯一主文档，把导航问题正式定性为系统级分裂，而不是零散补丁
- 当时动作：
  - 建 `统一导航重构阶段设计与执行主表`
  - 把工作区记忆体系立起来
- 结果：
  - 导航工作不再只是“修某个现象”，而是进入 S0-S8 长线
- 对后续影响：
  - 这一步是后面所有共享基础设施落地的前提

## 阶段 1：共享导航核心第一批落地
- 当时目标：
  - 把玩家/NPC 至少拉到共享代理契约与共享局部避让入口上
- 当时动作：
  - 扩展 `INavigationUnit`
  - 新增 `NavigationAgentSnapshot`
  - 新增 `NavigationAgentRegistry`
  - 新增 `NavigationAvoidanceRules`
  - 新增 `NavigationLocalAvoidanceSolver`
  - 修改 `PlayerAutoNavigator`
  - 修改 `NPCAutoRoamController`
- 结果：
  - 玩家与 NPC 开始共享局部避让输入源
- 对后续影响：
  - 结构上前进了一大步
  - 但体感线并没有因此自动通过

## 阶段 2：第一次体验性打回，发现“共享已接线 != 行为通过”
- 当时目标：
  - 解决“玩家右键导航仍推着 NPC 走”
- 当时动作：
  - 重新审视共享规则
  - 增加“自动导航玩家遇 moving NPC 默认让行”
  - 给 solver 加更强减前冲与侧绕
- 结果：
  - 发现仅靠规则层与 solver 微调不够
- 对后续影响：
  - 线程第一次被强行从“结构已做”拉回“体验未过”

## 阶段 3：发现玩家最终运动层会吞减速语义
- 当时目标：
  - 搞清楚为什么前面明明有减速/避让意图，玩家还是继续顶着走
- 当时动作：
  - 修改 `PlayerMovement`
  - 让导航输入不再被重新归一化吃掉减速
  - 增加临时动态绕行点
- 结果：
  - 证明问题不是单点，而是规则层 + 执行层共同导致
- 对后续影响：
  - 主线开始从“单调 solver”迁到“执行层闭环”

## 阶段 4：卫生与 hot-file 边界建立
- 当时目标：
  - 避免 `Primary.unity` 继续承载导航中间态
- 当时动作：
  - 清掉 scene 上属于导航 own 的部分序列化残留
  - 后续改成只读审计 `Primary.unity`
- 结果：
  - Scene 不再是导航线程默认落点
- 对后续影响：
  - 后续可以更清楚地区分代码热区与 mixed scene dirty

## 阶段 5：共享路径执行层真实退壳
- 当时目标：
  - 把 `BuildPath / RebuildPath / stuck / detour lifecycle` 逐步迁进 `NavigationPathExecutor2D`
- 当时动作：
  - 分阶段把：
    - `BuildPath / RebuildPath / ActualDestination / 路径后处理`
    - `stuck / repath / 恢复入口`
    - `detour create / clear / recover`
    往共享执行层迁
- 结果：
  - 底座能力确实比最早强很多
- 对后续影响：
  - 也正因如此，后面才能确认“底座有 API，不代表 runtime 真闭环”

## 阶段 6：错误地把结构 checkpoint 过早外推成体验过关
- 当时目标：
  - 用 runner 和 live 证明“最坏回归已压掉”
- 当时动作：
  - 做过一轮把 runtime 撤回到旧 solver 直出链的 selective restore
  - 一度拿到看起来不错的单场结果
- 结果：
  - 用户现场复测明确否定了“体验已回正”
- 对后续影响：
  - 线程被迫承认：
    - probe 绿灯不等于右键体感过关
    - 单场局部成功不能顶替真实玩法

## 阶段 7：same-round fresh 5 组 live 把责任点重新压窄
- 当时目标：
  - 不再拿 external blocker、旧 runner 或少报 dirty 当借口
- 当时动作：
  - 先做 hygiene 报实
  - 再补 `RealInputPlayerAvoidsMovingNpc / RealInputPlayerSingleNpcNear / RealInputPlayerCrowdPass / NpcAvoidsPlayer / NpcNpcCrossing`
- 结果：
  - 5 组都没过
  - 但收敛出了一个非常关键的新结论：
    - real-input 现场里 `shouldRepath=True` 会出现
    - 但 detour 长期仍然是 `false`
- 对后续影响：
  - 主线第一次稳定收束到“shared detour 不落地”

## 阶段 8：执行层握手链路开颅审计
- 当时目标：
  - 停止一切继续调 solver 参数的冲动
  - 直接交代 `ShouldRepath -> Detour` 为什么在 controller 层掉链子
- 当时动作：
  - 逐行核对：
    - `PlayerAutoNavigator`
    - `NPCAutoRoamController`
    - `NavigationLocalAvoidanceSolver`
    - `NavigationPathExecutor2D`
    - `PlayerMovement`
- 结果：
  - 叙事终于稳定到当前版本：
    - solver 已经会产出 detour 意图
    - 但 controller 没把它保活成 owner
    - runtime 继续退回 old path / stuck / blocked steering
- 对后续影响：
  - 已经满足进入下一代交接的条件
  - 因为当前不是“多种怀疑并行”，而是“单一第一阻塞点足够清楚”

## 哪些支线后来升级成主线的一部分
1. hygiene 审计
   - 一开始只是支撑动作
   - 后来变成必须保证交接不失真的边界前置
2. real-input runner
   - 一开始像验证工具支线
   - 后来成为体验主线的硬证据入口
3. detour lifecycle 迁移
   - 一开始像结构层工作
   - 后来直接成为“为什么执行层没接住意图”的一部分

## 哪些支线后来被证明只是噪声或绕路
1. 继续泛调 solver 权重
2. 重新争论 `TrafficArbiter / MotionCommand`
3. 把 compile blocker 或外部 dirty 当作默认停车位
4. 拿 probe 绿灯或单次局部 live 顶用户体感
