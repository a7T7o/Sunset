# 用户习惯、长期偏好与协作禁忌

## 用户对文档位置的要求
1. 工作区正文、prompt、memory 要按 Sunset 现行工作区规则落盘
2. 线程交接正文必须写到：
   - `D:\Unity\Unity_learning\Sunset\.codex\threads\Sunset\导航检查\V2交接文档\`
3. 不允许把交接正文写到 `.kiro`
4. 不允许把所有线程交接正文堆到 `.codex/threads` 根层共享目录

## 用户对主线/支线纪律的要求
1. 不明确换线时，默认仍服务原主线
2. 修 compile、修 runner、修 hygiene 都只是支撑子任务，不代表主线换了
3. 一轮只能打一刀时，就不要顺手扩成多刀
4. 验收失败后的续工 prompt 必须锁一个硬切片，不接受再次泛化

## 用户对中文透明度的要求
1. 需要中文、直白、硬核
2. 不要模糊话术，不要把“差不多”说成“通过”
3. 不要少报 `changed_paths`
4. 不要把 own / non-own / mixed dirty 混在一起说
5. 如果某件事只是推断，要明确写“这是推断”和依据

## 用户对 MCP / live / 核查方式的偏好
1. 用户强烈要求用 MCP 做 live Scene / prefab / component 核查
2. 不喜欢在明明可以用 MCP 时还让他先截图
3. live 必须短窗、证据够了就停、结束后回 `Edit Mode`
4. 不允许把 `probe` 绿灯包装成真实右键导航体验已通过

## 用户对 prompt、交接、治理格式的偏好
1. 喜欢固定格式、最小回执、可复制 prompt
2. 不接受“写了一堆解释，但没按约定格式答”
3. 对交接文档要求很高：
   - 不是摘要
   - 要能让下一代线程直接接班

## 用户最讨厌的行为
1. 一直在一堆参数里打转
2. 说“方向没错、骨架已写”来代替用户能验收到的结果
3. 不用 MCP、非要让用户补截图
4. 少报 dirty、漏报 hot file
5. 拿 external blocker 当停车位
6. 把结构进展和体验进展混在一起
7. 明明现场还在推土机，却提前 claim “最坏回归已压掉”

## 用户最常纠正的问题
1. “不要再泛调 solver 参数”
2. “不要再回漂到大架构宣讲”
3. “只围绕当前唯一主刀继续”
4. “先把 hygiene 报实”
5. “必须 fresh real-input，不准拿旧轮结果顶账”

## 已形成稳定习惯的命名、路径、协作口径
1. 线程继任命名按 `V2/V3/...` 递增，不得出现 `V2V2`
2. `Primary.unity` 被明确当作 A 类热文件处理
3. 导航工作区固定在：
   - `D:\Unity\Unity_learning\Sunset\.kiro\specs\屎山修复\导航检查\`
4. 用户认可的当前收敛表述必须够窄、够硬：
   - 例如当前这一轮能接受的口径就是：
     - `ShouldRepath` 已出现
     - detour owner 没稳定接管执行层

## 给 `导航检查V2` 的协作禁忌提醒
- 如果你下一轮又写出：
  - “继续观察”
  - “还需要再看看”
  - “可能是 solver 参数还要微调”
  - “也许需要回到大架构重审”
  这大概率会立刻踩用户雷。
