# 用户习惯、长期偏好与协作禁忌

## 1. 文档位置偏好

- 需求、设计、阶段日志、工作区记忆：
  - 写在 `.kiro/specs/...`
- 线程记忆：
  - 写在 `.codex/threads/.../memory_0.md`
- 重型代际交接文档：
  - 写在当前线程自己的 `.codex/threads/<分组>/<线程名>/V2交接文档/`
- 用户非常在意“东西写错地方”：
  - 不要把交接正文写进 `.kiro`
  - 不要把线程记忆写到全局 `C:` 下

## 2. 对主线 / 支线纪律的要求

- 默认不接受你因为一句插入式需求就丢掉原主线；
- 如果只是修阻塞、补文档、验收、排障，必须说清这是服务主线的子任务；
- 用户很讨厌“做着做着换题了，自己还没意识到”。

## 3. 对中文与透明度的要求

- 默认中文；
- 只在路径、命令、代码标识、Unity 类型名上保留英文；
- 不接受“装懂式简略结论”；
- 更喜欢你把：
  - 当前在干什么
  - 为什么这么判
  - 还有什么没做
  直接说清。

## 4. 对 prompt / 交接 / 治理格式的偏好

- 喜欢固定格式、最小回执、可复制执行；
- 但不接受空壳式最小回执；
- 真到需要交接时，用户要的是能彻底接班的重型交接，不是几段摘要。

## 5. 用户最讨厌的行为

1. 凭印象写历史
2. 擅自改写用户原话，特别是需求原文
3. 把局部 live pass 说成整条线已绿
4. 把 mixed-owner 文件静默当成自己独占
5. 没搞清边界就大面积改代码
6. 一边说“我懂了”，一边又改变了早已定好的交互规则
7. 自作主张加新快捷键、改旧输入边界
8. 用治理废话掩盖当前真实状态

## 6. 用户最常纠正的问题

1. “只是手持”不等于“正在交互”
2. Toolbar 与背包第一行是同一个热槽事实
3. placeable 手感必须服从视觉与操作一致性
4. 打开 UI 这类静止时间操作，不应偷偷改动其他系统状态
5. 要有大局观，不接受“当前只先这么凑合”
6. placeable / chest / farm 不是 Unity 试玩工程，要按长期运行规范设计

## 7. 已形成稳定习惯的命名与协作口径

- 线程名继续沿版本号递增：
  - `农田交互修复V2 -> 农田交互修复V3`
- 线程语义可以升级：
  - 当前虽然线程名还是 `农田交互修复V2`，但语义已升级为 `全局交互V3`
- 文档采用 docs-first 或 log-first 时，用户认可“先写清楚再做”，但前提是别写空话。

## 8. 用户对交互体验的硬偏好

- 逻辑必须服从视觉与手感一致；
- “看起来放下了但其实没成功”是高等级错误；
- “角色走到了，但其实交互距离还不对”也是高等级错误；
- 一旦涉及 UI / placeable / chest / 工具反馈，用户更看重：
  - 体感
  - 一致性
  - 不误导
  而不是你有没有局部把代码跑通。

## 9. 用户最讨厌你说但又做不到的话

- “这轮已经完全好了”
- “只剩一点小问题”
- “这不是大问题”
- “先这样，后面再说”

如果你说了，就要拿得出能验的证据；拿不出来，宁可老实写成：

- 哪一部分已恢复
- 哪一部分还没恢复
- 哪一部分只是验证工具尾差

## 10. 给继任线程的协作提醒

- 新线程不要试图“重新教育用户需求是什么”；
- 应该先假定：
  - 用户的体验判断是有效信号
  - 文档里保留下来的情绪表达也是有效信号
- 尤其新增 6 条需求，不要缩写成温和版，因为里面包含了很多真实验收标准。
