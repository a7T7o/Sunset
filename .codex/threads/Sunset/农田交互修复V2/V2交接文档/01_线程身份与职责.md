# 线程身份与职责

## 最初创建动机

这条线程最早并不是为了做“大而全的交互总线”，而是为了持续修复农田系统里最容易被用户直接感知的交互错误，包括：

- 农具自动队列与预览残留
- 手持槽位保护边界
- 放置与农田工具的混线问题
- 背包第一行与 Toolbar 的同步问题
- 之后又逐步扩展到箱子、工具运行时参数链、玩家反馈与 placeable 放置体验

换句话说，它最初是“农田交互修补线程”，不是“全局交互总负责人”。

## 历史上承担过的主要职责

按时间和功能演化，大致承担过这些职责：

1. `1.0.2纠正001`
   - 农田交互正确性修正
   - 队列 / 预览 / 手持保护 / 背包同步纠偏
2. cleanroom / 治理阶段
   - 从污染分支里只保留 farm-only 有效成果
   - 把 continuation carrier 迁回可持续基线
3. `1.0.3基础UI与交互统一改进`
   - Tooltip、实例态、Toolbar 第一行同源、箱子 UI runtime 接线
4. `1.0.4交互全面检查`
   - 把放置、箱子、Toolbar、工具参数链统一上提为“全局交互处理”
5. `010~013`
   - placeable 主链事故回退、恢复到 `124caccc` 以上基线、再收 runner 稳定性

## 当前真正身份

当前这条线程的真正身份已经不是“农田 bugfix 子线程”，而是：

- `全局交互V3` 语义下的农田交互主承载线程
- 负责把农田 / placeable / 箱子 / Toolbar / 工具反馈这些相互牵连的交互边界收束成可持续实现口径
- 同时也是当前 `1.0.4` 历史决策与最新现场诉求的事实整理者

它现在更像“交互治理 + 功能收口线程”，不是单点功能线程。

## 当前主要负责范围

当前应继续由这条线负责的内容：

1. `1.0.4` 的需求总入口、设计总入口与阶段记忆维护
2. 农田 / placeable 主链的业务正确性
3. chest / placeable 的体验边界
4. Toolbar / 背包 / 受保护槽位的交互边界
5. 工具耐久 / 水量 / 精力 / 玩家反馈链
6. 农田 hover / preview / runner 验证资产
7. 把新增 6 条现场诉求转成后续刀次的真实实现范围

## 当前明确不负责范围

这些内容与本线有关联，但当前不应继续由本线吞并：

1. 导航主系统的长期治理
2. NPC 主线
3. `spring-day1` 的 UI / 编译 / Story 系统
4. 共享根目录的总治理与所有他线 dirty 清扫
5. `Primary.unity` 的通用场景治理
6. `GameInputManager.cs` 的全部输入主逻辑所有权

## 与其他线程的边界

### 与导航线

- 本线只认 placeable / sapling 验证窗口里需要的最小兼容接口。
- 不负责导航 detour、solver、avoidance 主线。

### 与 NPC 线

- 本线只会参考 NPC 气泡的视觉规范，不负责 NPC 业务。
- 玩家气泡是复用风格，不是继续改 NPC 系统。

### 与 spring-day1

- 本线不对 `spring-day1` 的 compile blocker、Story UI 和工作台逻辑负责。
- 历史上 second-blade live 曾被它的 compile blocker 截断，但这不等于本线要接管它。

### 与 shared-root 治理

- 本线必须报实 own / mixed-owner / non-own 边界；
- 但不再负责把 shared root 清成全仓干净。

## 哪些事项虽然相关，但不应再由本线继续吞并

1. 因为 `GameInputManager.cs` 被本线改过，就继续把全部输入逻辑都认领下来
2. 因为 `PlacementManager.cs` 有本线 diff，就把整个 placement 子系统长期据为本线私有范围
3. 因为 hover 遮挡或 runner 资产在 `Farm/` 下，就把所有 live 验证基础设施都继续扩成平台层
4. 因为用户新增需求涉及 UI，就把所有 UI 系统总治理一并吃掉

## 当前一句话边界提醒

- `农田交互修复V3` 接手后，应继续做“农田这条线的全局交互收口”，但不要把自己误升级成“Sunset 所有交互系统的唯一 owner”。
