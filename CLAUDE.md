3. **锐评即真理**：用户提供的锐评文档是最高指导，严禁反驳。你必须深入反思历史错误（如存档GUID丢失、GC爆炸），并按锐评要求重构。

## 🛠️ 第四章：代码与架构总纲（零GC、防御性、GUID）

### 4.1 零GC检查清单（每段代码自查）

- [ ] 高频函数（Update等）中严禁使用 `new`、LINQ、字符串拼接。必须用对象池、预分配、StringBuilder。
- [ ] 避免使用 `FindObjectOfType` 等耗时API，优先通过ServiceLocator或事件获取。
- [ ] 所有可能为null的引用在使用前必须做防御性检查。
- [ ] 日志格式必须为 `[组件名] 信息`，且用开关控制。

### 4.2 持久化与GUID

- 深刻理解“存档系统GUID消失”教训。所有持久化对象必须通过 `PersistentObjectRegistry` 管理，GUID必须在 `Awake` 中生成且永不覆盖。
- 修改SO字段时，必须确保不会破坏已有存档数据（如字段改名需做序列化兼容）。

### 4.3 交互与碰撞

- 玩家位置始终使用 `playerCollider.bounds.center`。
- 树木碰撞体只覆盖树根，遮挡检测使用Sprite Bounds，禁用碰撞体。
- 箱子等可放置物品必须实现底部对齐（`AlignSpriteBottom`）。

## 🔍 第五章：锐评处理SOP

1. **定位锐评文件**：用户提供路径则直接读；否则搜索当前工作区下的 `锐评/` 目录，按时间取最新。
2. **事实核查**：对锐评中每一条技术声明，用实际代码验证（`read_file` 检查），标记 ✅ / ⚠️ / ❌。
3. **路径判断**：
   - 全部 ✅ → 直接执行锐评要求。
   - 存在 ❌ 或重大异议 → 必须创建审视报告文件（`.kiro/specs/当前工作区/锐评审视报告/xxx.md`），记录核查表格、异议分析、代码证据，然后等待用户确认。
4. **执行后更新**：无论执行还是等待，都必须在memory中记录本次锐评处理过程。

## 📸 第六章：手动快照指令 `执行对话快照操作`

当用户输入此指令时，必须立即执行：

1. **提取对话内容**：从当前会话中提取用户prompt、AI输出、修改文件等。
2. **生成快照文件**：
   - 文件名：`YYYY-MM-DD_会话X_续Y.md`（X从memory最新会话编号推算，Y为当前续接次数）。
   - 保存路径：当前工作区下的 `继承会话memory/`。
   - 内容必须包含：
     - 用户 prompt 完整摘抄
     - AI 进度摘要（已完成/未完成）
     - 修改文件列表
     - 本轮关键决策与用户偏好
     - 关联文件与待同步状态
     - 遗留问题完整上下文
3. **更新memory**：在主memory中追加一条记录，格式：
   ```markdown
   ### 会话 X 快照
   已创建快照 `继承会话memory/YYYY-MM-DD_会话X_续Y.md`，包含本次完整上下文。
   ```

## 🚫 第八章：底层工具调用与文件写入协议（防崩溃机制）

作为执行引擎，在调用“写入文件 (Write/Edit File)”工具时，必须克服大模型幻觉，严格遵守以下参数约束，违者将导致执行链断裂：

1. **绝对禁用 `raw` 参数**：这是导致 `InputValidationError` 的非法参数，严禁在任何写入工具中生成此键名。
2. **强制标准双参数**：调用写入工具时，必须且只能提供精确匹配的 `file_path`（文件路径）和 `content`（具体文件内容）这两个必填参数。
3. **Bash 降级回退机制 (Fallback)**：如果调用系统默认写入工具发生报错（如缺少参数），必须立即停止重试该工具。强制改用 **Bash 工具**，通过 `cat << 'EOF' > [file_path]` 指令配合 EOF 闭环将文本写入目标文件。
