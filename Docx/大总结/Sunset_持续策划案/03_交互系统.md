# Sunset 交互系统：输入、状态边界、背包、放置与存储

## 1. 文档目标

本卷用于统一 `Sunset` 当前交互层的正式口径，重点明确输入分发、状态边界、背包 / Toolbar、农田工具、放置系统、箱子系统与工具动作锁之间的关系。

`Sunset` 当前交互复杂度的真正来源，不是按钮多，而是以下几条链路长期并行存在：

1. 世界交互。
2. UI 交互。
3. 导航交互。
4. 工具动作交互。
5. 剧情 / 教学锁定交互。

本卷的任务，就是把这些链路压回同一套状态语言，避免继续依赖临时特判。

## 2. 交互层总原则

### 2.1 当前正式目标

当前交互层的统一目标有四条：

1. 玩家在任意时刻都只有一个清晰主状态。
2. 世界输入与 UI 输入不能互相打穿。
3. 放置、农田、导航、背包、箱子必须共享同一套取消与恢复逻辑。
4. 红色无效状态必须真正做到“什么都不触发”。

### 2.2 当前统一原则

当前正式口径应始终遵守以下原则：

1. `红色 = 无效 = 不触发动作、不触发导航、不偷跑执行`。
2. 玩家始终保留撤回权，`WASD / 右键 / ESC` 必须能在约定状态下取消当前链路。
3. 面板打开的默认语义是“暂停”，不是“偷偷继续跑逻辑”。
4. 工具装备、工具使用、动画播放和实际结算必须分层，而不能混在一个输入判断里。

## 3. 输入分发与全局状态锁

### 3.1 当前核心承载体

当前输入与交互边界主要由以下入口共同承载：

- `Controller/Input/GameInputManager.cs`
- `PlayerMovement.cs`
- `PlayerInteraction.cs`
- `ToolActionLockManager.cs`
- `SprintStateManager.cs`
- `PlayerToolFeedbackService.cs`

### 3.2 当前输入层职责

输入层当前至少负责以下事项：

1. 左键、右键、滚轮、数字键与面板快捷键分发。
2. 工具态、普通交互态与剧情态的互斥判断。
3. 放置系统、农田系统与普通工具使用的分流。
4. 面板打开时对世界输入、移动输入和滚轮输入的阻断。
5. 工具动作播放期间的行动锁、方向缓存与收尾恢复。

### 3.3 当前全局锁的正式理解

当前项目中至少存在三类必须分开理解的锁：

1. `面板锁`：打开背包、箱子或其他面板后，世界输入被阻断。
2. `动作锁`：工具动画执行中，动作链路进入保护态，不允许被普通输入直接打断。
3. `剧情锁`：对话、CG、教学提示、NPC 引导等脚本化片段会临时接管输入权限。

这三类锁的意义不同，不能继续用“现在不能动”一句话糊成同一件事。

## 4. 背包、Toolbar 与物品操作

### 4.1 当前库存结构

当前库存的正式结构不是两套独立背包，而是：

1. 主背包显示层为 `36` 格。
2. 其中 `0 ~ 11` 同时作为 `12` 格 Toolbar / Hotbar 的映射区。
3. Toolbar 与背包第一行共用同一批数据，不允许双份库存。

也就是说，当前正确口径不是“背包一套、快捷栏一套”，而是“同源数据，两种表现”。

### 4.2 当前快捷栏输入边界

当前 Toolbar 输入边界已经明确统一为：

1. 数字键 `1 ~ 5` 只负责前五格直选。
2. 滚轮在全部 `12` 个快捷栏槽位内循环。
3. Toolbar 选中变化会立即驱动当前手持工具 / 武器更新。

这个口径已经在 `InventoryService.HotbarWidth = 12`、`HotbarDirectSelectCount = 5` 以及交互矩阵里收口，不应再回退到旧的 `8` 格或“数字键全选 12 格”的混合说法。

### 4.3 当前背包交互状态机

背包交互当前已经不是“拖一下换位置”的简化版本，而是至少明确包含以下状态：

1. `Idle`
2. `HeldByShift`
3. `HeldByCtrl`
4. `Dragging`

并且已经明确支持以下操作：

1. 普通拖拽交换。
2. `Shift` 二分拿取与连续二分。
3. `Ctrl` 单个拿取、持续拿取与快速装备。
4. 面板外松开丢弃。
5. `ESC` 取消当前拿取状态。

### 4.4 当前物品丢弃区域口径

当前背包系统已经明确：

1. 真正的背包区域是 `Main + Top`。
2. `Background` 只是背景装饰，不算有效放置区。
3. 在 `Main / Top` 外松开手持物品时，才进入丢弃逻辑。

这条规则很重要，因为它决定了“误点背景”与“主动丢弃”在交互上是两回事。

### 4.5 当前跨场景玩家状态合同

最近一轮交互层新增的重点，不是又多了一个按钮，而是玩家运行态已经开始跨场景保留：

1. `PersistentPlayerSceneBridge` 负责让 `Primary / Town / Home` 只保留唯一玩家实例。
2. 背包数据、Hotbar 选中槽位与当前工具态会在切场前捕获、切场后恢复到新场景的本地场景服务上。
3. `GameInputManager`、`HotbarSelectionService`、`PlayerAutoNavigator` 会在新场景重新绑定，不再继续指向旧场景对象。
4. 门触发已加入“入场锚点 + 入场缓冲”，避免落地瞬间被原门再次吞回。

这意味着当前交互层已经不再只服务“单场景点击”，而是开始服务“同一条玩法链跨场景继续成立”。

## 5. Toolbar、手持工具与动作触发

### 5.1 当前正式职责分离

工具栏链路当前已经明确拆成两段：

1. `HotbarSelectionService` 负责选中变化与装备更新。
2. `GameInputManager` 负责检测世界点击并触发对应动作动画。

这意味着：

1. 点击 Toolbar 选中工具时，就已经完成装备。
2. 随后左键点击世界时，只负责触发使用动作。
3. 点击 UI 不会触发世界使用动作，但不会再阻断“装备本身”。

这条分离规则是当前工具栏交互稳定下来的关键。

### 5.2 当前工具动作映射

当前工具使用至少已经明确映射到以下动画状态：

- 斧头 / 镰刀 -> `Slice`
- 镐子 / 锄头 -> `Crush`
- 长剑 -> `Pierce`
- 洒水壶 -> `Watering`
- 鱼竿 -> `Fish`

工具动作当前不是孤立动画，而是与以下内容同步：

1. 玩家本体动画。
2. 手持工具动画。
3. 工具显隐与排序。
4. 动作完成后的实际结算。

### 5.3 动作锁的当前意义

当前动作锁不是单纯“播动画时别乱按”，而是要保证：

1. 执行动作时不被无关输入抢占。
2. 方向切换不会导致动作抽搐或错帧。
3. 手持工具在动作结束时不会闪现或残留。
4. 同一动作的视觉阶段与实际结算阶段保持一致。

### 5.4 当前工具态与场景重绑口径

当前工具动作链已经明确：玩家跨场景时不常驻整套本地场景工具服务，而是迁移运行态结果：

1. 当前背包快照和 Hotbar 选中态先被保留。
2. 新场景里的 `InventoryService / HotbarSelectionService / GameInputManager` 再接住这些状态。
3. 当前手持工具与动作入口在新场景重新挂接，而不是继续吃旧引用。

这条口径的价值，是让 `Town -> Primary -> Home` 的场景切换不会再因为“玩家还是同一个、服务却换了一套”而天然断掉工具链。

## 6. 放置系统

### 6.1 当前放置系统的正式定位

放置系统已经不是“点地上生成物体”的单步功能，而是一套完整状态机，当前主要承载于：

- `PlacementManager.cs`
- `PlacementValidator.cs`
- `PlacementPreview.cs`
- `PlacementNavigator.cs`
- `PlacementGridCalculator.cs`
- `PlacementLayerDetector.cs`
- `PlacementExecutionTransaction.cs`

### 6.2 当前放置主状态

当前放置链路应按以下状态理解：

1. `Idle`：未进入放置模式。
2. `Preview`：预览跟随鼠标。
3. `Locked`：目标点已锁定，等待导航或执行。
4. `Navigating`：玩家正在前往放置点。
5. `Executing`：放置动作进入执行保护。

如果只讨论“已进入放置模式后的主流转”，则核心是 `Preview -> Locked -> Navigating -> Executing`，`Idle` 只是外层基态。

### 6.3 当前放置合法性判断

当前放置合法性已经明确拆成几层：

1. 坐标是否落在有效格子中心。
2. 玩家当前 Layer 与目标 Layer 是否一致。
3. 目标格是否被树、石头、建筑、水域、玩家或其他可放置物占用。
4. 特殊物品是否满足附加规则。

当前正式语义是：

1. 绿色只表示“该位置本身允许放置”。
2. 绿色不等于“玩家现在立刻就能放下去”。
3. 近距离与远距离的差别，由导航接近层处理，而不是让颜色承担双重语义。

### 6.4 当前放置交互规则

当前放置系统已经形成稳定矩阵：

1. 绿色近距离左键：锁定后直接执行放置。
2. 绿色远距离左键：锁定后导航，到点执行放置。
3. 红色左键：什么都不做。
4. 导航中再次点绿色：取消旧导航，重锁新目标。
5. 导航中点红色：取消导航，恢复到跟随预览。
6. `WASD` 或特定取消输入：退出锁定或打断导航。

当前项目里，放置系统真正成熟的地方，不是预览能显示，而是这些取消、重定向和无效输入语义已经被写成正式规则。

### 6.5 放置与特殊对象

当前放置系统已经明确要对以下对象单列处理：

1. 树苗。
2. 箱子。
3. 工作台。
4. 多格物体或特殊碰撞体物体。

这说明当前放置系统不是“万物共用一条 if 判断”，而是已经承认不同放置物对齐、碰撞和交互后的行为并不相同。

## 7. 农田工具交互

### 7.1 农田系统不是放置系统的附属

当前农田工具链已经形成独立的 `FarmNavState` 流程，至少包含：

1. `Idle`
2. `Preview`
3. `Locked`
4. `Navigating`
5. `Executing`

虽然状态名字与放置系统相近，但农田链路有自己的规则边界，因为它同时要处理：

1. 锄地。
2. 浇水。
3. 播种。
4. 清除枯萎作物。
5. 与农田 `1 + 8` 预览和三层显示的关系。

### 7.2 当前农田交互口径

当前农田交互正式口径已经明确：

1. 可交互区域显示有效预览。
2. 不可交互区域必须给出明确的无效反馈，而不是静默无显示。
3. 红色无效预览与有效预览必须共享同一套显示逻辑，不允许一个显示完整预览、一个只剩临时白框。
4. 队列预览、正在执行的预览与实际落地层必须分离，不能互相提前清掉。

### 7.3 农田与放置的共同原则

农田与放置虽然是两套系统，但已经共用三条底层原则：

1. 红色无效时不触发动作。
2. 打开面板时暂停预览，而不是直接破坏当前链路。
3. 导航中点击新目标时，要么重新导航，要么取消恢复，不允许半残留状态。

### 7.4 当前 `Town / Home` 场景合同

最近一轮农田与场景交互的真实推进，已经把两个场景的合同拉开了：

1. `Town`
   - 当前已补齐 `FarmTileManager.layerTilemaps[0]` 的 `groundTilemap` 等关键引用。
   - 锄地预览与真实执行都已把 `groundTilemap` 当成硬前置，不再因为场景合同缺口直接爆红。
   - 这意味着 `Town` 当前可以被视为完整本地场景交互场之一。
2. `Home`
   - 当前只有门合同、持久化玩家承接、固定镜头与最小输入 fallback。
   - 它还没有补齐农田、放置、完整 UI 与本地交互底座，不能被误判成和 `Primary / Town` 同级的完整玩法场。

## 8. 箱子、存储与面板互斥

### 8.1 当前箱子系统正式状态机

箱子系统当前已经明确拆成三态面板机：

1. `NoPanel`
2. `BackpackOpen`
3. `BoxOpen`

它的核心不是“能打开箱子 UI”，而是：

1. 箱子 UI 打开时，背包主页与顶栏如何隐藏。
2. 按 `Tab / ESC / 页面键` 时，箱子 UI 与背包 UI 如何互相切换。
3. 面板打开后，世界输入如何彻底禁用。

### 8.2 当前箱子与背包关系

当前箱子 UI 的正式理解是：

1. `Up` 区域显示箱子内容。
2. `Down` 区域显示玩家完整背包视图。
3. Toolbar 的第一行本质上仍来自同一批背包数据。
4. 箱子容量通过 `1×12 / 2×12 / 3×12 / 4×12` 等布局承载。

这意味着箱子系统不是额外再造一套库存逻辑，而是把“容器视图 + 玩家库存视图 + 互斥面板行为”缝成一体。

### 8.3 当前箱子交互的重点

当前箱子系统最关键的正式问题有三类：

1. 面板互斥与输入阻断。
2. 箱子槽位与背包槽位的拖拽转移。
3. 箱子与放置、导航、世界点击的共享边界。

因此，箱子系统的价值不只是存东西，而是把 UI 状态、库存状态和世界交互的边界全都逼到了同一处。

## 9. 交互层的跨系统统一口径

### 9.1 当前真正收口的不是单个 Bug，而是边界语言

当前交互层已经逐渐形成一套统一语言：

1. `Preview / Locked / Navigating / Executing` 用于描述世界交互链。
2. `NoPanel / BackpackOpen / BoxOpen` 用于描述面板层。
3. `Idle / HeldByShift / HeldByCtrl / Dragging` 用于描述库存操作层。

这套统一语言的意义是，后续新增功能不必每次重新发明状态名。

### 9.2 当前已经被证明最重要的边界

结合已有规则文档与问题修复史，当前最关键的交互边界包括：

1. 红色无效输入不能偷跑逻辑。
2. 面板打开后必须禁止世界输入。
3. 导航中重定向与取消必须清晰可预测。
4. 工具装备与工具使用必须分离。
5. 预览层、队列层与实际落地层不能互相污染。
6. 背包第一行、Toolbar、箱子 Down 区必须坚持同源数据口径。

## 10. 全局优先级与打断规则

### 10.1 输入归属优先级

当前交互层不是“所有点击都先进同一个入口再临时判断”，而是已经形成固定归属顺序：

| 优先级 | 交互入口 | 适用对象 | 正式结果 |
|---|---|---|---|
| P0 | 面板 / 剧情锁 | 背包、箱子、设置、脚本化剧情片段 | 世界输入先被阻断或暂停，避免 UI 与世界链路互相打穿 |
| P1 | 农田链 | 锄头、水壶、种子、农田收获 | 农田工具由 `FarmNavState` 接管；收获检测在左键链路内保持最高优先级 |
| P2 | 放置链 | `PlaceableItemData`、树苗、箱子、工作台等 | 由放置状态机单独处理，不穿透到通用工具逻辑 |
| P3 | 世界交互链 | `IInteractable`、箱子、工作台、NPC 等 | 按 `InteractionPriority` 降序、同优先级按距离升序选择目标 |
| P4 | 通用工具 / 武器链 | 斧头、镐子、长剑等普通动作 | 只在前几层都未命中时才进入普通攻击 / 工具动作 |

这条顺序直接决定了三个正式结论：

1. 农田工具不会误入放置链或普通工具链。
2. 箱子、工作台、NPC 等世界交互对象不是“谁先被射线扫到就算谁”，而是按交互优先级和距离共同决策。
3. 面板或剧情锁一旦生效，世界点击不能继续偷跑到底层逻辑。

### 10.2 全局状态优先级表

当前交互层已经形成可复用的状态层级：

| 状态层 | 代表状态 | 世界点击 | 导航 / 预览 | 统一语义 |
|---|---|---|---|---|
| 脚本接管层 | 对话、CG、教学片段、剧情锁 | 禁止 | 由脚本接管 | 最高优先级，先保证剧情链完整 |
| 面板互斥层 | `BackpackOpen`、`BoxOpen` | 禁止 | 暂停但不强制清空 | 面板 = 暂停，不是偷偷继续结算 |
| 执行保护层 | `Executing`、`ToolActionLock` | 忽略重入 | 导航与方向输入进入保护 / 缓存 | 动画阶段与实际结算必须保持一致 |
| 锁定导航层 | `Locked`、`Navigating` | 允许按规则改写目标 | 允许重定向或取消 | 玩家始终保留撤回权 |
| 预览跟随层 | `Preview` | 允许 | 允许 | 负责持续反馈当前可操作位置 |
| 空闲层 | `Idle`、`NoPanel` | 允许 | 无 | 回到正常世界交互基态 |

### 10.3 取消、暂停与恢复矩阵

当前项目最关键的不是“能不能进入某个状态”，而是进入后能否稳定退出：

| 输入 / 场景 | 目标状态 | 正式结果 |
|---|---|---|
| `WASD` 手动移动 | 农田 / 放置 `Locked`、`Navigating` | 立即取消导航，恢复跟随预览或退出放置锁定，再把控制权还给玩家 |
| `WASD` 手动移动 | `Executing` / 工具动作锁 | 不打断动作本体，只缓存方向，待动作结束后消费 |
| `右键` / `ESC` | 农田 `Locked`、`Navigating` | 取消导航，恢复到 `Preview` |
| `右键` / `ESC` | 放置 `Preview`、`Locked` | 退出放置模式，回到 `Idle` |
| `右键` / `ESC` | 放置 `Navigating` | 取消导航，回到 `Preview`，再由玩家决定是否退出 |
| `Tab` / 其他面板热键 | 面板关闭时 | 打开面板，同时暂停预览更新，不直接破坏已有链路 |
| `ESC` | 面板打开时 | 先关闭面板，不顺手取消农田 / 放置状态 |
| 导航中点击绿色目标 | 农田 / 放置链 | 取消旧导航，锁定新目标，重新导航 |
| 导航中点击红色目标 | 农田 / 放置链 | 取消导航，解锁预览，恢复跟随 |
| 滚轮 / 数字键切换工具 | 农田非 `Idle` | 取消当前导航，再切到新工具对应状态 |
| 滚轮 / 数字键切换工具 | 工具动作锁定中 | 先缓存切换请求，待动作结束后再消费，避免动画闪断 |

### 10.4 当前回归与验收基线

以下内容已经构成当前交互卷的正式回归基线：

| 编号 | 场景 | 你怎么做 | 通过标准 | 失败像什么 |
|---|---|---|---|---|
| A1 | 红色无效预览 | 在红色农田格或红色放置格左键点击 | 不触发动画、不触发导航、不生成任何落地结果 | 红色位置仍然开始走路、播放动作或落地生成 |
| A2 | 面板暂停不中断 | 农田 / 放置预览存在时打开背包，再关闭 | 面板打开期间世界输入被阻断；关闭后能恢复原状态 | 打开面板直接清空链路，或关闭后状态丢失 |
| A3 | 导航中绿色重定向 | 农田 / 放置导航中改点另一个绿色目标 | 旧导航被取消，新目标重新锁定并继续导航 | 旧导航残留、双重导航、目标错位 |
| A4 | 导航中红色取消 | 农田 / 放置导航中点击红色目标 | 立刻取消导航并恢复跟随预览 | 红色点击无效但旧导航仍然继续 |
| A5 | 箱子与背包互斥 | 世界状态打开箱子、`Tab` 切回背包、`ESC` 关闭 | `NoPanel / BackpackOpen / BoxOpen` 三态切换稳定，世界输入只在 `NoPanel` 可用 | 箱子关闭后背包打不开，或面板开着还能点世界 |
| A6 | 背包 / Hotbar 同源 | 拖拽、滚轮、数字键、箱子 Down 区来回转移 | 背包首行、Toolbar、箱子 Down 区始终是同一批数据 | 出现双份库存、双选中或只刷新一边 |
| B1 | 工具动作锁 | 工具动作过程中连续滚轮、数字键、换方向 | 动作不中断、方向不卡帧、工具不闪烁；动作结束后再消费缓存输入 | 动画抽搐、工具残影、动作被切断 |
| B2 | 世界交互排序 | 让多个 `IInteractable` 重叠，重复点击 | 由交互优先级决定候选，同优先级距离近者优先 | 目标乱跳、明明更近或更重要却选不中 |
| B3 | 作物收获优先级 | 在可收获作物上左键，同时持有普通工具 | 先执行收获，不被普通工具动作覆盖 | 对着成熟作物仍然挥工具、不进入收获 |
| B4 | 拿取状态收口 | 背包 / 箱子里拿起物品后按 `ESC` 或拖到外部 | `ESC` 回源；仅在有效外部区域才进入丢弃 | 背景误丢弃、拿起状态无法取消 |

## 11. 当前正式收口

当前交互层的正式收口可以概括为四条：

1. 世界输入、面板输入、剧情输入已经被压回统一优先级，不再是互相打穿的散乱状态。
2. 农田、放置、箱子三条高复杂度链路已经共享“红色无效、绿色重定向、撤回权在玩家手里、面板暂停不中断”的统一语言。
3. `Toolbar` 装备更新、工具动作执行、动画锁保护、库存同源显示与跨场景工具重绑这几层已经被明确拆开，避免一处输入同时负责所有事情。
4. 当前交互卷已经能够同时服务单场景边界收口与 `Primary / Town / Home` 的跨场景运行合同，不再只是历史问题汇总。
