# Sunset 表现层：场景、UI、动画、遮挡与导航

## 1. 文档目标

本卷用于统一 `Sunset` 当前表现层的正式口径，明确场景分层、UI 表达、人物与工具动画、导航避让、遮挡透明、昼夜光影与世界提示的职责边界，为后续场景搭建、交互验收与表现精修提供统一依据。

## 2. 表现层总结构

### 2.1 当前承载范围

`Sunset` 当前表现层由六个互相咬合的部分组成：

1. 场景与 Tilemap 层级
2. UI 与世界提示
3. 玩家 / 工具 / NPC 动画
4. 导航与避让
5. 遮挡透明、云影与动态排序
6. 昼夜光影与环境提示

### 2.2 当前直接场景入口

- `Assets/000_Scenes/Primary.unity`
- `Assets/000_Scenes/SceneBuild_01.unity`
- `Assets/000_Scenes/Home.unity`
- `Assets/000_Scenes/Town.unity`

### 2.3 当前表现层的统一目标

当前表现层不是独立美术外壳，而是承担以下设计目标：

1. 让资源、农田、道路、水域、建筑与 NPC 落点形成可读的空间结构。
2. 让玩家的操作反馈、提示反馈与剧情提示进入同一套视觉秩序。
3. 让导航、遮挡、排序和碰撞在玩家视角下保持一致，不出现“能看到但不能走”“看不见却被挡住”的割裂感。
4. 让人物动作、工具动作与劳动反馈形成稳定的节奏表达。

### 2.4 当前跨场景表现合同

最近一轮表现层已经不再只服务单张场景，而是开始服务 `Primary / Town / Home` 的承接关系：

1. `Primary / Town` 当前仍是完整本地场景表现基线，承接 HUD、交互提示、角色落位、工作台与农田主体验证。
2. `Home` 当前已被收成固定镜头、黑底、室内承接壳，不再沿用默认跟随镜头逻辑。
3. 持久化玩家进入新场景后，核心表现层要优先保证“镜头、提示、工具状态、背包显示”继续可读，而不是重新把玩家当成新对象生成一遍。

## 3. 场景与 Tilemap 表达

### 3.1 当前层级基础

根据 `第一阶段完结报告`、`世界物体设计.md` 与场景现状，当前项目已经明确采用多层 Tilemap 结构，至少包含：

- `Ground`
- `Detail`
- `Farmland`
- `Paths`
- `Water`
- `Cliff`

当前口径下，环境层、碰撞层、排序层与功能层不是一层混用，而是分开组织。农田与普通地表也不是同一承载层。

### 3.2 场景层的当前职责

当前场景层同时承担：

1. 资源分布与采集区域落点。
2. 动态障碍物与可走区域的空间基础。
3. 农田、道路、水域、建筑和功能区域的区分。
4. NPC 日常站位、漫游点与交互气泡的空间承载。
5. 遮挡、导航、动态排序与提示气泡的共同基底。

### 3.3 当前场景表达的正式要求

当前场景搭建应遵守以下口径：

1. 地形表达优先保证功能分层清晰，再做装饰堆叠。
2. 树木、石头、建筑等世界物体的可见范围、碰撞范围和排序表现必须统一。
3. 交互入口必须在世界空间中有明确落点，不能只存在脚本逻辑而没有视觉锚点。

## 4. UI 与世界提示

### 4.1 当前 UI 架构

根据 `UI系统与总系统设计规划.md`，当前 UI 层已经形成较明确的总架构：

- `PackagePanel`
- `Main`
- `Top`
- 常驻 `ToolBar`
- 多页签切换
- 快捷键与 Toggle 同步

### 4.2 当前 UI 承载内容

当前 UI 层已经明确承载：

- 背包页
- 配方页
- 地图页
- 关系页
- 设置页
- Tooltip
- 时间与状态显示位
- `spring-day1` 的 `PromptOverlay`
- `spring-day1` 的 `StatusOverlay`
- 工作台 Overlay
- 世界空间气泡提示

### 4.3 当前 UI 表现口径

当前 UI 表达遵循三层结构：

1. 常驻 HUD：时间、状态条、工具栏等持续信息。
2. 功能面板：背包、配方、关系、设置等结构化信息。
3. 世界提示：NPC 提示气泡、交互提示、`spring-day1` 世界锚点提示。

这意味着 `Sunset` 的提示设计不依赖单一大面板，而是同时使用 HUD、面板和世界提示三条表达通路。

### 4.4 当前玩家面可读性收口

最近一轮 UI 与世界提示的真实推进，已经把几条最直接影响体验的口径收得更清楚：

1. `DialogueUI` 当前继续文本已统一回到空格推进，不再依赖旧的 `T` 键或鼠标点按推进。
2. `ItemTooltip` 已和即时状态条分流：
   - 状态条负责立即显示当前耐久 / 水量 / 精力等运行态信息
   - tooltip 负责悬停延迟显示，且被限制在 UI 面板范围内，不再乱飘到场景主视面
3. `Workbench` Overlay 当前已经不再只是“能打开”，而是开始承接：
   - 多 recipe 排队
   - 进度条文本
   - 悬浮进度卡聚合
   - 左列 row 自愈与运行态几何恢复
4. 玩家 / NPC 气泡当前都在往“圆角主体 + 明确尾巴 + 正式对话与轻反馈可区分”的统一壳体推进。

## 5. 玩家、工具与 NPC 动画

### 5.1 当前动画基础

根据 `000_动画系统完整文档.md` 与阶段报告，当前玩家动画至少已明确：

- 状态：`Idle / Walk / Run / Carry / Collect / Hit / Slice / Pierce / Crush / Fish / Watering / Death`
- 方向：`Down / Side / Up`
- Side 方向使用翻转

### 5.2 当前动画表现的核心成果

当前动画层最关键的成果，是人物与工具动作的同步问题已经被正式解决。当前采用的核心口径是：

1. 不再依赖纯 `normalizedTime` 对齐。
2. 使用“帧索引硬锁”作为人物与工具同步基准。
3. 同步时同时处理状态参数、方向、显隐、排序与翻转。

这套方案直接对应 `LayerAnimSync.cs`、`PlayerAnimController.cs` 与 `PlayerToolController.cs` 的协作关系。

### 5.3 当前动画工具链

当前与动画直接对应的 Editor 工具包括：

- `LayerAnimSetupTool.cs`
- `SliceAnimControllerTool.cs`
- `ToolAnimationGeneratorTool.cs`
- `ToolAnimationPipeline.cs`
- `TriDirectionalAnimGenerator.cs`
- `Tool_003_BatchAnimTransitions.cs`

它们共同承担三件事：

1. 三方向动画资产组织。
2. 工具动作控制器生成与过渡配置。
3. 动画资源批处理与同步成本压缩。

## 6. 导航、避让与遮挡联动

### 6.1 当前导航承载体

当前导航与避让层直接承载于：

- `NavGrid2D.cs`
- `NavigationPathExecutor2D.cs`
- `NavigationLocalAvoidanceSolver.cs`
- `NavigationAvoidanceRules.cs`
- `NavigationAgentRegistry.cs`
- `PlayerAutoNavigator.cs`
- `NPCAutoRoamController.cs`

### 6.2 当前导航口径

根据 `000_遮挡与导航系统完整文档.md`，当前导航层已经明确：

1. 使用 A* 八方向寻路。
2. 采用严格对角线检测。
3. 具备世界边界自动检测。
4. 具备动态障碍物刷新能力。
5. 玩家与 NPC 都依赖同一组基础导航能力。

### 6.3 当前遮挡与排序口径

当前遮挡与动态排序层直接承载于：

- `OcclusionManager.cs`
- `OcclusionTransparency.cs`
- `CloudShadowManager.cs`
- `DynamicSortingOrder.cs`

当前正式口径是：

1. 玩家位于物体下方且发生视觉重叠时，遮挡物进入透明处理。
2. 树木、建筑、石头等遮挡物需要兼顾可读性与空间压迫感，不能简单做“全挡住”或“永远透明”。
3. 树木成长、树冠连通、云影覆盖与动态排序属于同一表现问题链，而不是分散的小特效。

### 6.4 导航与遮挡必须联动的原因

当前项目中，导航与遮挡必须作为一组规则共同维护，原因包括：

1. 放置、采集、制作等交互都依赖“玩家能走到、能看见、能点到”的统一判断。
2. 树木成长会同步改变遮挡、碰撞与可走区域。
3. 玩家肉眼看到的距离感必须和系统的交互距离、导航停靠距离保持一致。

2. `Primary` 与 `Town` 当前都已接入云影，并按场景分别缓存运行态，不再共用一套临时重抽状态。
3. 历史孤儿云清理、预览云防序列化与场景级恢复链已经补齐，避免云影残留越积越多。
4. 云影运行态已经开始接入正式存档链，保证“玩家看到的云层状态”不再只停留在当前会话里。

## 8. 当前统一视觉语言

### 8.1 场景空间表达

当前场景表达已经形成三条正式要求：

1. 可走区域、可互动区域、可遮挡区域必须同时可读，不能只有一种信息在画面里成立。
2. 农田、道路、水域、悬崖、建筑、资源点必须通过层级、轮廓和落点清楚分开，避免“看得见但不知道能不能用”。
3. 交互入口必须落在世界空间里有锚点的位置，例如工作台、床铺、资源点、NPC 站位，而不是只在逻辑层存在。

### 8.2 UI 与世界提示表达

当前 UI 与提示层采用三条并行通路：

1. 常驻 HUD 负责时间、状态条、工具栏这类持续信息，不抢占世界中心注意力。
2. 功能面板负责背包、配方、关系、地图等结构化信息，强调完整阅读与拖拽操作。
3. 世界提示负责 `PromptOverlay`、交互提示、NPC 气泡和 Day1 世界锚点提示，强调就地反馈。

这套视觉语言要求：

1. 临时提示只说当前最需要知道的一件事，不把剧情提示、系统提示、交互提示堆成一屏。
2. 玩家气泡与 NPC 气泡保持同一产品节奏，但必须一眼可分。
3. 状态条、工作台 Overlay、Day1 提示层与常规背包 UI 在视觉上属于同一项目，而不是四种不同风格的临时界面。

### 8.3 动画与动作反馈表达

当前动画层的正式口径已经非常明确：

1. 人物与工具动作必须以“帧索引硬锁”保持零帧级同步，而不是接受 1~2 帧拖影。
2. 工具动作播放期间，方向不变时不重置动画，避免方向切换抽搐。
3. 工具显隐、排序和翻转必须跟人物动作同步处理，不能在动作结束瞬间闪烁。
4. 工具渲染层当前正式跟随规则是“位于人物下方一层”，即 `toolRenderer.sortingOrder = playerRenderer.sortingOrder - 1`。

### 8.4 导航、遮挡与光影表达

当前表现层对导航与遮挡不是两套松散规则，而是一套空间可读性规则：

1. 玩家处于物体下方且与精灵视觉重叠时，遮挡物进入透明处理；当前遮挡透明度基线为 `0.3`。
2. 树木、建筑、石头等遮挡对象必须同时兼顾“挡住空间压迫感”和“不给玩家视线造成彻底断层”。
3. 导航停靠距离、可见距离、交互距离在玩家视角下必须大体一致，不能肉眼已经走到边上却被系统判定过远。
4. 夜间阅读压力由全局光影、局部点光与世界提示共同承担，而不是单纯把画面压暗。

## 9. 当前正式验收基线

| 编号 | 表现子项 | 你怎么验 | 通过标准 | 失败像什么 |
|---|---|---|---|---|
| A1 | Tilemap 分层 | 在主场景检查地表、农田、水域、悬崖、道路与建筑 | 功能层与装饰层清楚分离，交互入口位置可读 | 地表与功能区混层，玩家分不清可用区域 |
| A2 | 工具同步 | 挥斧、挥镐、浇水、刺击连续测试 | 工具与人物动作无拖影、无闪烁、无错层 | 工具慢半拍、结束闪一下、排序跳变 |
| A3 | 导航与遮挡联动 | 走到树木、建筑下方并贴边导航 | 看得见、走得通、点得到三件事保持一致 | 透明触发和导航停靠互相打架 |
| A4 | HUD 与世界提示 | 触发 Day1、工作台、交互提示与常驻状态条 | HUD 不抢戏，世界提示只说当前焦点，层级清晰 | 多种提示重叠、互遮、信息抢焦点 |
| A5 | 夜间可读性 | 进入夜间并观察提示、局部照明与路径识别 | 夜间氛围变暗但关键交互对象仍可稳定识别 | 夜里整体糊成一片，只能靠猜位置操作 |
| A6 | NPC / 对话气泡 | 观察玩家气泡、NPC 气泡、轻反馈气泡 | 同一套节奏下可一眼区分“谁在说”“是正式对话还是轻反馈” | 气泡风格混乱、方向不清、身份感弱 |

## 10. 当前正式收口

当前表现层的正式收口可以概括为四条：

1. 场景、UI、动画、导航、遮挡和光影已经不是分散小系统，而是共同服务“空间可读、反馈清晰、节奏统一”的同一目标。
2. 动画层已经通过帧索引硬锁与动作锁保护把人物 / 工具同步问题收稳，不再停留在“差不多对齐”。
3. 遮挡、导航、云影与跨场景镜头承接已经开始围绕同一套“玩家在任何场景都要读得清”的标准收口。
4. 当前表现卷已经具备正式验收基线，可以同时服务场景搭建、UI 精修、云影终验与跨场景体验回归。
