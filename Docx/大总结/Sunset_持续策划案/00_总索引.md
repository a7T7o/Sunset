# Sunset 策划总索引

## 1. 文档定位

本套文档作为 `Sunset` 项目的统一策划总文档，负责承接项目的系统设计、规则口径、阶段记录、模块进度与后续迭代更新。

后续凡是进入正式开发链的设计决策、规则调整、结构修订、阶段结论与完成状态，均以本套文档作为持续回写入口。

## 2. 当前固定事实

- 项目名称：`Sunset`
- 项目类型：Unity 6 2D 像素风奇幻生活模拟 RPG
- 项目核心叙事：失忆工匠流落废弃聚落，通过采集、制作、建造逐步复兴聚落，并在过程中找回记忆
- 核心玩法闭环：资源积累 -> 经营生产 -> 能力成长 -> 事件 / 探索解锁 -> 聚落复兴反哺

## 3. 记录原则

### 3.1 只写真实存在的内容

这里只记录三类东西：

1. 已经在项目里出现过的设计正文、需求、规则、验收、迭代记录。
2. 已经在运行时代码、数据资产、预制体、Editor 工具里有承载的内容。
3. 已经在 workspaces、threads、Docx 分类库中被明确提出、持续推进或反复修正的内容。

### 3.2 状态口径

- `已形成基础运行时`
- `已形成设计库并多轮迭代`
- `已进入规则收口阶段`

### 3.3 更新顺序

后续维护默认按这个顺序执行：

1. 先改对应主题正文。
2. 再改 `08_进度总表.md`。
3. 如果这次是新主题，补进本文件索引。
4. 最后在对应线程 / 工作区 memory 里记录“这次补录了什么、还缺什么”。

## 6. 文档编排原则

1. 全卷按设计主题编排，不按线程历史编排。
2. 每卷优先落系统目标、当前口径、关键规则、实际承载体与后续接续位。
3. 历史稿、阶段稿、专项修复稿只作为来源，不直接替代当前正文。
4. 当前正文以“项目实际使用口径”为最高优先级，避免同一模块出现多套平行说法。

## 7. 当前维护要求

1. 新增系统设计时，优先补对应主题卷，不另起一份平行总稿。
2. 规则修正时，先改正文，再改进度总表。
3. 如果某个模块仍处在专项 workspaces 内快速迭代，正文至少要同步写清当前主口径和未收口项。
4. 后续开发继续沿用本套文档，不再把正式内容长期留在零散锐评、一次性委托和临时总结里。
