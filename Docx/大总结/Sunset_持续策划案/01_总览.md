# Sunset 项目总览、世界观与核心闭环

## 1. 文档目标

本卷用于统一 `Sunset` 的项目基础口径，明确项目定位、叙事前提、阶段线与核心循环，作为后续各专题卷的总前提。

## 2. 项目是什么

`Sunset` 是一个基于 Unity 6 开发的 2D 像素风奇幻生活模拟 RPG。

从当前仓库中的项目简介、历史规划、阶段报告、`spring-day1` 切片和三大系统版图可以确认，项目当前主轴是“生活经营 + 资源成长 + 剧情推进”的复合循环。

当前固定口径：

- 题材外壳：奇幻生活模拟
- 行为基础：采集、种植、制作、建造、交互、探索
- 长线目标：逐步复兴聚落
- 叙事前提：失忆工匠在废弃聚落中重新建立秩序，并在经营过程中找回自己的记忆

### 3.1 早期阶段

从 `Docx/大总结/第一阶段完结报告.md` 可以明确看出，项目在第一阶段已经把“核心框架与基础系统”作为主目标，重点完成的是：

- 时间系统
- 地图与环境系统
- 角色系统
- 物品与背包系统

- 导航系统

### 3.2 中期阶段

从 `Docx/Plan/第二阶段初步规划.md` 与 `种田系统完整设计方案.md` 可以看到，中期明显进入了：

- 种植系统
- 场景与资源点
- 生命值 / 战斗
- 场景完善

### 3.3 当前阶段补记

从 `.kiro/specs` 和 `.codex/threads` 的第一批 live 吸收来看，当前项目已经不再是“哪个系统先搭起来”的早期阶段，而是多条核心线同时进入收平阶段：

- `spring-day1` 已不再是“主线能不能搭起来”的早期验证，而是进入 `Town -> Primary -> Home` 主链已成、导演尾差和 runtime 终验待补的阶段。
- 农田、放置、工具与背包链继续做公共交互边界、状态机与场景合同收口，其中 placement 公共链已被明确切成独立问题，不再混成剧情禁用语义。
- `UI` 当前重点不是继续堆面板，而是 `Workbench / PromptOverlay / ProximityService / task-list` 的玩家面集成、刷新风暴止血与 packaged/live 证据。
- `NPC` 已从“角色能出现”推进到 resident 化、`NPC_Hand` 真源、关系页结构与自漫游 / avoidance 纠偏。
- 导航与跨场景桥接问题已从泛导航 bug 压成 `scene contract / runtime bridge / driver` 三层责任。
- `Town / Primary / Home` 三场景 persistent baseline 已接入正式运行链，当前主要剩 live 门链与体验尾差。
- 策划侧工具链、验证菜单与 AI 协同治理继续服务这些主线收口，而不再只是独立工具堆叠。

### 3.4 当前阶段的主验证面补记

当前项目最近一轮最重要的验证面，已经从“单系统能不能跑”推进到“多场景合同能不能一起成立、同一批 runtime 状态能不能被完整接住”：

1. `Town` 负责入村围观、原生常驻居民消费导演 cue、围观散场后先回家再恢复漫游。
2. `Primary` 继续承接旧屋安置、工作台闪回、农田教学与第一次真实制作。
3. `Home` 已接进 `Home <-> Primary` 门合同、固定镜头黑底和持久化玩家承接，但当前仍不是完整本地场景玩法场。
4. 背包、Hotbar、工具态与云影运行态已经开始按“跨场景保留 / 新场景重绑”这套口径推进。

这意味着当前主风险已经不是“底座没做完”，而是“主线已成后，导演尾账、placement 公共链和 live 终验还没完全收平”。

### 3.5 当前阶段的主控拍板与主风险补记

本节用于补记当前阶段已经形成并持续生效的项目推进方式和真实主风险。

当前可以确认的项目级变化包括：

1. 主线已持续收束在 `spring-day1` 的跨场景承接上，当前正式口径为 `Town -> Primary -> Home`，并且 `day1` 已从“搭骨架”进入“收导演尾差和 runtime 终验”的阶段。
2. 多条关键设计方向已被正式拍板并开始反向约束线程推进，包括：
   - NPC 从 runtime 临时承载改判为常驻居民化
   - 玩家走向跨场景唯一实例持久化
   - `Home` 镜头按固定构图、黑底、尊重手摆的室内口径继续推进
   - UI 结构按“核心 UI 全局、场景扩展本地”收束
   - 高频验证口径按 `CLI first, MCP last-resort` 固定
3. 当前主风险不再是“有没有底座”，而是：
   - 导演尾账还没彻底吃平
   - placement 公共链仍需继续与剧情链脱钩收口
   - 多场景 live 黑盒终验还差最后一轮稳定票
4. `Codex规则落地`、规范快照和回执体系当前只负责把这些边界、验证顺序和停发条件落成执行层，不再代替 `day1 / UI / NPC / 导航 / 存档` 本身的业务事实。

这条推进方式的意义，是把项目从“多系统并行推进”进一步收束到“围绕同一条连续世界主线完成承接、居民化、持久化与终验”的正式口径。

## 4. 世界观与叙事前提

当前项目采用的叙事前提为：

玩家扮演一名失忆工匠，在一处已经衰败、失去活力的聚落中醒来。玩家需要通过采集资源、恢复生产、解锁设施、与 NPC 建立连接，逐步让聚落恢复基本功能，并在这一过程中重新拼回自己的过去。

这一叙事前提在项目中的作用：

1. 它天然允许“从零开始”的生活模拟起步。
2. 它天然解释了为什么一开始很多设施、资源链、社会关系都不完整。
3. 它把经营和叙事绑在了一起，因为聚落复兴本身就是玩法目标。

## 5. 项目核心玩法闭环

当前核心玩法闭环统一表述为：

`资源积累 -> 经营生产 -> 能力成长 -> 事件 / 探索解锁 -> 聚落复兴反哺`

### 5.1 资源积累

当前能直接承载这一层的实际内容包括：

- 树木 / 石头 / 矿物 / 资源点
- 农田 / 作物 / 种子
- 世界物体 / 掉落物 / 拾取
- 工具命中与消耗

### 5.2 经营生产

当前与这一层直接相连的内容包括：

- 背包与快捷栏
- 箱子与存储
- 工作台与配方
- 可放置物体
- 锁箱 / 钥匙 / 容器交互

### 5.3 能力成长

当前项目里的成长不是只靠数值面板，而是同时体现在：

- 工具等级
- 材料等级
- 资源成长层级
- 配方和工作台的扩展
- 交互能力与行动范围的扩大

### 5.4 事件 / 探索解锁

这一层目前最明确的承载体是：

- `spring-day1` 剧情引导
- `Town -> Primary -> Home` 场景切换与入场锚点
- NPC 互动、带路与围观承接
- 触发器与教学系统
- 正式对话消费后的常驻居民 / informal 回落入口
- 日常气泡与关系成长入口

### 5.5 聚落复兴反哺

这部分当前还没有完全整合成一份统一长文，但它已经作为项目主目标持续存在：

- 设施修复与工作台开放
- NPC 与日常秩序恢复
- 场景从“可生存”向“可经营”演进
- 后续剧情和记忆回溯有了明确世界承载

## 7. 当前项目主设计特征

### 7.1 先定规则，再做内容扩张

当前项目的突出特征，是很多内容先以规则、交互边界和数据结构的形式落地，再继续并入玩法与表现层。

例如：

- 放置系统先写完整规则汇总、状态机和快照机制。
- 箱子系统先写交互场景矩阵、面板状态机和验收指南。
- `spring-day1` 先写 18 项需求和 P0/P1/P2 任务拆解。
- NPC 不是先做“多角色越多越好”，而是先拆“场景落点 / 日常 / 双气泡 / 关系成长 / 命中兼容”四条主线。

### 7.2 边界修复本身构成设计推进的一部分

从当前工作区结构可以看出，`Sunset` 很多设计不是一稿写死，而是在反复修：

- 农田系统多轮迭代
- 放置系统 V1 -> 重构 -> V3 -> 联动 -> 对齐修复 -> 开发规范 -> 多问题修复
- 箱子系统 核心 -> 样式与交互 -> 综合修复 -> UI 完善 -> V2 -> V3
- NPC 从初步规划到进一步落地，再到全面清盘与第二轮清盘

### 7.3 当前切片已经进入跨场景运行合同阶段

最近一轮开发最明显的变化，是项目主切片已经不再只验证“某个系统能不能单独工作”，而是开始验证：

1. 同一个玩家能否跨 `Primary / Town / Home` 保持唯一实例与当前玩法状态。
2. 同一条 `spring-day1` 叙事能否在 `Town` 完成围观承接、在 `Primary` 继续旧屋安置与教学、在 `Home` 留下可扩张的室内合同。
3. 视觉层、存档层和交互层是否能围绕同一批运行态继续工作，而不是切场后各自重置。

### 7.4 当前阶段的收口条件补记

当前阶段的另一个显著特征，是项目已经先形成统一收口条件，再继续扩写各条线：

1. `spring-day1`、NPC、UI、存档与云影，不再允许各自按“局部最优”扩写，而是必须共同服务 `Town -> Primary -> Home` 的跨场景连续体验。
2. `resident 化`、`持久化玩家`、`Home` 场景定位、UI “核心全局 / 场景扩展本地”、`CLI first` 等判断，已经不只是讨论结果，而是会直接改变后续施工边界。
3. 当前能否继续推进，不再只看“功能有没有做出来”，还要看围观、门链、工具态、Workbench、Prompt 与玩家面反馈有没有一起收平。
4. 因此当前最该补的是 end-to-end runtime / live 终验，而不是再平行拉开新的大面。

这使 `Sunset` 当前的设计推进，已经从普通需求沟通转向“策划主控先定义主线与收口条件，再由线程执行并通过终验继续收束”的阶段。

## 8. 当前继续扩展重点

当前总览卷后续重点继续落在四条主线上：

1. `Town / Primary / Home` 三场景在主线切片里的正式分工与验收口径。
2. 聚落复兴的阶段化总表、角色入驻与反哺收益结构。
3. 资源博弈、成长与风险收益的统一数值表达。
4. 各阶段历史口径继续向当前跨场景正式口径统一。
