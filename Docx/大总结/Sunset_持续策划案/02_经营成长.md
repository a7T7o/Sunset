# Sunset 生活经营、资源成长与时间季节

## 1. 文档目标

本卷用于统一 `Sunset` 当前生活经营与成长线的正式口径，明确时间、季节、资源、农田、制作、工具门槛与行动成本之间的完整关系。

`Sunset` 的经营线不是把“采集、种田、制作”并排摆放，而是要形成一条可持续推进的成长闭环：

1. 时间与季节决定世界节奏。
2. 资源节点与农田系统决定玩家当下能做什么。
3. 制作链与工作台决定资源如何转化为更高阶能力。
4. 精力、耐久与工具等级决定一次行动值不值得做、能做到哪一步。

## 2. 经营成长总闭环

### 2.1 当前经营闭环

当前项目的经营成长主线应按以下闭环理解：

`时间推进 -> 季节变化 -> 资源刷新 / 外观变化 -> 采集与耕作 -> 制作与转化 -> 工具升级 -> 解锁更高阶资源 -> 反哺下一轮经营`

这条闭环的核心不是“刷数值”，而是让玩家在每一天的劳动里持续面对三类判断：

1. 现在该把时间花在采集、耕种还是制作上。
2. 当前工具和材料等级允许我处理哪些资源，哪些只能先路过。
3. 这轮资源应该直接消耗，还是先压进工作台链做更长期的成长。

### 2.2 当前经营线的正式构成

当前经营线由五个互相咬合的模块组成：

1. 时间、季节、天气与昼夜。
2. 树木、石头、矿物等世界资源节点。
3. 农田、作物、湿润与收获循环。
4. 配方、工作台、箱子与资源转化链。
5. 工具等级、材料等级、精力与耐久成本。

`Sunset` 当前真正有价值的地方，不是某一项单独功能，而是这些模块已经开始围绕同一套成长门槛组织。

## 3. 时间、季节与世界节奏

### 3.1 当前时间系统职责

当前时间线已明确由以下运行时入口共同承载：

- `TimeManager.cs`
- `SeasonManager.cs`
- `WeatherSystem.cs`
- `DayNightManager.cs`

时间系统当前承担的不是单纯计时，而是全局节奏控制器，至少负责：

1. 推进分钟、小时、天的流逝。
2. 广播 `OnMinuteChanged / OnHourChanged / OnDayChanged` 等系统事件。
3. 驱动作物按天生长、湿润重置与阶段更新。
4. 触发季节切换、天气状态与表现层变化。
5. 为 `spring-day1` 这类脚本化剧情切片提供时间控制入口。

### 3.2 当前季节结构

当前项目已经明确存在两层季节口径：

1. 基础四季：
   - `Spring`
   - `Summer`
   - `Autumn`
   - `Winter`
2. 植被与表现层使用的五阶段过渡：
   - 早春
   - 晚春早夏
   - 晚夏早秋
   - 晚秋
   - 冬季

这意味着 `Sunset` 的季节不是简单换皮，而是把“经营节奏层”与“植被表现层”拆成两套精度不同的系统：前者按四季做规则，后者按五阶段做世界反馈。

### 3.3 季节对经营线的影响

当前季节系统已经明确影响以下内容：

1. 作物是否允许种植以及生长推进是否成立。
2. 树木与植被的季节外观、挂冰 / 融化 / 枯萎等状态变化。
3. 昼夜光照、点光源与世界可读性。
4. 场景氛围、资源观察感与玩家对“现在该做什么”的主观判断。

对 `Sunset` 来说，季节最重要的价值不是“好看”，而是让同一张地图在不同阶段呈现不同的生产意味与资源意味。

### 3.4 当前四季经营重点

当前四季在 `Sunset` 里已经不只是世界背景，而是经营节奏的分工器。按照现有 `spring-day1`、农田、树木、天气与季节规则，当前正式口径可归纳为：

1. 春季是开荒和第一轮生产底座期。
   - `spring-day1` 的教学链本身就把开垦、播种、浇水、收木材和第一次工作台制作排在一起。
   - 这一阶段的重点不是高阶产出，而是把农田、基础资源和工作台入口全部接通。
2. 夏季是稳定经营与持续扩张期。
   - 时间系统、作物成长、树木成长和工作台转化在这一阶段能形成最稳定的连续生产。
   - 经营重点会自然转向扩大耕地、补足材料池和推进下一轮资源深化。
3. 秋季是收获、整理与储备期。
   - 五阶段植被在这一段会从晚夏早秋逐步过渡到晚秋，世界反馈会明显提醒玩家从“扩产”转向“收口”。
   - 这一阶段的核心不是再开新线，而是把作物、木材、矿材和半成品尽量压进可过冬的存量。
4. 冬季是经营重心转向制作、存储与整理的阶段。
   - 当前规则已经明确：冬季树苗不能继续作为正常种植入口，树苗会死亡，天气会触发枯萎 / 挂冰 / 融化等特殊反馈。
   - 这意味着冬季不会鼓励继续做高频种植扩张，而更适合作为制作链、箱子整理、资源盘点和探索准备期。

## 4. 资源节点、采集门槛与材料层级

### 4.1 树木系统

当前树木系统已经明确采用 `Stage0 ~ Stage5` 的六阶段结构，并且不再是静态装饰物。

树木成长主线当前正式口径如下：

| 阶段 | 名称 | 成长天数 | 血量 | 主要工具 | 经营含义 |
|------|------|----------|------|----------|----------|
| 0 | 树苗 | 1 天 | 0 | 锄头 | 可被挖出、可重布置 |
| 1 | 幼苗 | 2 天 | 4 | 斧头 | 初级木材来源 |
| 2 | 小树 | 2 天 | 9 | 斧头 | 开始参与碰撞与遮挡 |
| 3 | 中树 | 4 天 | 17 | 斧头 | 进入独立树桩阶段 |
| 4 | 大树 | 5 天 | 28 | 生铁斧及以上 | 中高阶木材门槛 |
| 5 | 巨树 | 稳定终态 | 40 | 黄铜斧及以上 | 高阶采集门槛 |

从树苗到巨树共计 `14` 天，这个时间长度直接决定了树木不仅是资源点，也是长期经营资源。

### 4.2 树木不是单纯掉落物

树木当前同时承担五类职责：

1. 木材与树桩掉落来源。
2. 动态障碍物与导航边界。
3. 遮挡透明、排序与场景分层测试对象。
4. 五阶段季节表现的主要载体。
5. 放置、锄地、种植等系统的空间冲突对象。

因此，树木在 `Sunset` 中不是“被砍掉就结束”的采集点，而是经营线、表现线与交互线共同依赖的世界节点。

### 4.3 石头、矿石与掉落规则

当前石头 / 矿石线至少已经通过以下入口落地：

- `StoneController.cs`
- `RockController.cs`
- `StoneStageConfig.cs`
- `StoneDropConfig.cs`
- `DropTable.cs`
- `ResourceNode.cs`

当前正式口径里，矿物采集不是“能不能敲”，而是“能不能敲出目标矿物”：

1. 所有镐子都能获得基础石料。
2. 低等级镐子可以命中高等级矿，但只能拿到石料，拿不到对应矿物。
3. 真正决定成长门槛的是工具等级与矿物层级的匹配关系。

当前矿物层级已明确包括：

- `C2` 铁矿，需要石镐及以上。
- `C1` 铜矿，需要生铁镐及以上。
- `C3` 金矿，需要钢镐及以上。

这套规则的价值在于：即使玩家“够得着”矿点，也不代表已经具备完整收益资格。

### 4.4 材料等级系统

当前材料等级已明确按六级组织：

1. 木 `Wood`
2. 石 `Stone`
3. 生铁 `Iron`
4. 黄铜 `Brass`
5. 钢 `Steel`
6. 金 `Gold`

材料等级当前同时作用于：

1. 工具可处理的资源范围。
2. 武器表现与战斗效果。
3. SO 命名与资源组织。
4. 工作台升级后的制作目标。

这意味着 `Sunset` 的成长核心并不是传统角色等级，而是“材料等级 -> 工具等级 -> 可处理资源层级 -> 新材料”的正反馈链。

## 5. 农田、作物与农业循环

### 5.1 当前农田主循环

当前农田系统的正式核心循环为：

`锄地 -> 播种 -> 浇水 -> 等待生长 -> 收获 -> 回到下一轮播种`

这条循环并不是停留在设计层，而是已经明确落到了：

- `FarmTileManager.cs`
- `FarmTileData.cs`
- `CropController.cs`
- `CropInstance.cs`
- `CropManager.cs`
- `FarmVisualManager.cs`
- `FarmToolPreview.cs`
- `FarmlandBorderManager.cs`
- `SoilMoistureState.cs`
- `CropStageConfig.cs`
- `LayerTilemaps.cs`

### 5.2 当前农田结构

当前农田系统已经明确不是“单块 Tile 切换”，而是同时管理三层内容：

1. 地块数据层：记录是否已耕作、是否已种植、当前湿润状态。
2. 预览 / 队列层：记录锄地、浇水、播种等操作的当前可执行状态。
3. 实际表现层：用 Tilemap、边界、作物实例和排序层把状态真正落到场景。

在农田交互重构中，项目已经把耕地的视觉组织进一步压成 `1 + 8` 邻接逻辑，也就是“中心格 + 八方向边界”共同决定耕地预览、队列预览与实际落地层的表现关系。

这个设计的意义是：农田不再是纯九宫格占位，而是把逻辑中心格、视觉边界和批量生产场景下的预览一致性彻底分开处理。

### 5.3 当前湿润系统

当前农田湿润状态已明确不是二元开关，而是至少包含三种视觉语义：

1. 干燥。
2. 湿润带水洼。
3. 湿润深色。

当前正式口径为：

- 浇水后短时间内表现为“湿润带水洼”。
- 经过一定时间后进入“湿润深色”。
- 到第二天重置或蒸发后回到干燥。

这让浇水不只是一个是否生效的布尔值，而是会在同一天内留下明显的经营反馈。

### 5.4 当前作物推进口径

当前作物系统已明确支持以下设计元素：

1. 多生长阶段推进。
2. 按天推进生长，而不是按实时秒表推进。
3. 连续缺水会停滞，继续缺水则会枯萎。
4. 支持可重复收获作物。
5. 支持收获品质判定，品质影响售价而非外观。

当前与作物成长最相关的正式字段已经包括：

- `harvestAmountRange`
- `isReHarvestable`
- `reHarvestDays`
- `maxHarvestCount`
- 作物品质 `Quality`

这说明 `Sunset` 的作物不是“成熟一次就结束”的一次性资源，而是已经预留了长期经营价值与收益差异。

### 5.5 农田在经营线中的真实地位

农田系统当前不仅负责种植本身，还承担：

1. 新手教学的第一条劳动主线。
2. 精力消耗与劳动反馈的最直观承载位。
3. 时间系统“按天推进”最容易被玩家感知的出口。
4. 资源转化链中最稳定、最可预测的一类产出来源。

## 6. 制作、工作台与资源转化

### 6.1 当前生产链承载体

当前与制作和资源转化直接相关的入口包括：

- `.kiro/specs/制作台系统`
- `.kiro/specs/箱子系统`
- `RecipeData.cs`
- `WorkstationData.cs`
- `StorageData.cs`
- `KeyLockData.cs`
- `CraftingStationInteractable.cs`
- `Assets/YYY_Scripts/Service/Crafting/*`
- `Assets/YYY_Scripts/UI/Crafting/*`

### 6.2 工作台的正式定位

当前工作台在 `Sunset` 中不是普通 UI 容器，而是成长门槛装置，至少承担：

1. 把基础资源加工成中间材料或进阶工具。
2. 把“我拿到了资源”转化为“我具备了更高能力”。
3. 通过工作台等级、配方解锁与交互流程组织长期成长曲线。
4. 与剧情切片中的工作台闪回、基础制作配方解锁形成叙事连接。

### 6.3 箱子、钥匙与资源风险

`StorageData` 与 `KeyLockData` 的存在，意味着箱子系统当前不仅承担存储，还已经接进了经营风险设计：

1. 箱子容量影响资源周转能力。
2. 锁与钥匙影响容器收益的获取门槛。
3. 存储系统与背包、放置、工作台共同组成资源流转链。

因此，制作台和箱子不是两条分离小功能，而是一条“资源获取 -> 资源存储 -> 资源转化”的完整经营链。

### 6.4 当前工作台成长链的正式口径

当前制作链在项目里已经明确分成两层，而不是“一个工作台脚本包办所有内容”：

1. 配方入口层：由 `CraftingStation` 决定配方应该在哪个制作入口显示和执行。
2. 世界设施层：由 `WorkstationType` 决定这类设施在场景里的大类、倍率、燃料和放置语义。

从当前代码和历史制作台工作区可确认，配方入口层已经至少存在以下站点：

- `Workbench`
- `Furnace`
- `AlchemyTable`
- `Grill`
- `CookingPot`
- `AnvilForge`
- `MagicTower`

而世界设施层则已经明确预留了以下大类：

- `Crafting`
- `Smelting`
- `Cooking`
- `Pharmacy`
- `Sawmill`
- `Heating`

这套双层结构非常关键，因为它说明 `Sunset` 的制作链不是平铺按钮页，而是已经开始按“设施身份”和“配方入口”拆分成长曲线。

在当前正式口径里，各站点的职责可以这样理解：

1. `Workbench` 是最早进入玩家经营视野的基础制作入口，也是 `spring-day1` 里第一次正式把“资源 -> 配方 -> 成品”串起来的叙事锚点。
2. `Furnace` 承担矿石向锭类材料的冶炼入口，并且在 `WorkstationData` 里已经预留了燃料、槽位和制作倍率这套生产参数。
3. `AnvilForge` 把金属深化从通用工作台里进一步拆出来，意味着后续锻造链与基础制作链不会长期混在一个入口里。
4. `CookingPot` 与 `Grill` 把作物、食材和恢复类资源接进制作系统，让农业线不只停在收获，而能继续进入消耗品加工。
5. `AlchemyTable` 与 `MagicTower` 说明项目已经给药品和特殊转化留出了正式入口，不需要未来再把所有特殊配方硬塞回通用工作台。

需要明确的是：当前这条工作台成长链已经有数据层、服务层和 `spring-day1` 交互锚点，但并不等于所有站点都已完成最终资源配置和场景闭环。它的正式意义在于，经营成长线的“分站点扩张”方向已经被写死，不再是散乱的未来设想。

### 6.5 当前资源流向的正式口径

当前 `Sunset` 的资源并不是“采到就结束”，而是已经开始围绕工作台链形成多段转化：

1. 木材线：
   - 树木提供最早期、最稳定的基础材料来源。
   - 木材既要支撑基础制作，也承担后续锯木、建造和更复杂设施链的前置资源角色。
2. 矿物线：
   - 石料、铁矿、铜矿、金矿先由世界节点产出，再通过工具等级门槛筛选玩家能否拿到真正矿物收益。
   - 进入制作链后，矿石首先应该流向 `Furnace` 这类冶炼入口，再转向 `Workbench / AnvilForge` 这样的进阶制作入口。
3. 农业与食物线：
   - 作物先通过农田系统形成稳定产出，再向“直接食用”或“进入 `CookingPot / Grill` 深加工”两条线分流。
   - 这让农业系统既能承担恢复，也能继续进入更复杂的食物与经营收益链。
4. 存储与周转线：
   - 箱子系统不是附属背包，而是把一次性采集变成长期囤积、分批生产和跨天调度的关键节点。
   - 锁与钥匙又在这条周转线上加入了收益门槛与风险收益差异。
5. 成长回环线：
   - 配方产物最终又会反过来提升工具等级、材料等级和可处理资源范围。
   - 这意味着一次制作的真正价值，不是得到一个单品，而是把下一轮能进入的资源层级再往上推。

因此，当前经营线最准确的理解不是“采集系统 + 农田系统 + 制作系统并列”，而是：

`世界资源 -> 背包 / 箱子周转 -> 工作台分流转化 -> 更高阶工具 / 消耗品 / 中间材料 -> 解锁下一轮资源获取`

## 7. 行动成本、精力与耐久

### 7.1 成本系统的正式意义

当前精力与耐久系统的作用，不是单纯限制玩家，而是为经营决策建立成本。

当前经营线里，至少以下动作已经与精力或耐久发生明确关系：

1. 锄地。
2. 浇水。
3. 砍树。
4. 挖矿。
5. 工具与武器的重复使用。

### 7.2 当前成本口径

已有设计材料已经明确给出部分动作成本基线：

- 锄地消耗精力 `3`
- 浇水消耗精力 `2`
- 低等级工具面对高等级资源时，可能无法造成有效收益，但仍消耗精力与耐久

这条规则非常关键，因为它直接防止了“拿错工具也能白嫖推进”的问题。

### 7.3 当前成长节奏的核心判断

`Sunset` 当前成长节奏的正式核心不应理解为“等级越高数字越大”，而应理解为：

1. 时间在推着玩家做阶段规划。
2. 工具等级在限定玩家的可处理范围。
3. 精力和耐久在限定玩家每天的有效行动数。
4. 工作台和配方在决定资源是否能转化成长期收益。

这四条共同成立时，生活经营线才会真正出现“取舍感”和“第二天还想继续”的推进感。

## 8. 当前正式收口

当前 `Sunset` 的经营成长线已经形成相对完整的底层骨架：

1. 时间、季节与天气已经具备全局驱动职责。
2. 树木、矿物、材料等级与工具门槛已经形成成长阶梯。
3. 农田循环已经具备独立状态层、预览层和成长层。
4. 制作台、配方、箱子与锁系统已经开始接入长期经营线。
5. 四季经营重点、工作台双层结构与资源转化方向已经能写成稳定口径，而不再只是散点设想。

当前继续扩展重点主要集中在三类表格化内容：

1. 各类资源与工作台的阶段解锁表。
2. 各站点的资源配置、燃料约束与资产闭环表。
3. 作物、矿物、木材与中间材料的正式经济流向图。
