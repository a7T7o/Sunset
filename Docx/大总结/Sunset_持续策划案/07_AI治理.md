# Sunset AI 协同治理、规则层与线程体系

## 1. 文档目标

本卷用于统一 `Sunset` 当前 AI 协同治理体系的正式口径，明确规则层、工作区层、线程层、memory 分层、hooks、locks、thread-state、skills 与 Unity MCP 的职责划分，以及项目当前采用的标准协作流程。

## 2. 治理层的当前职责

`Sunset` 当前治理层的核心任务，不是简单记录“用了 AI”，而是把单人策划主导、多线程 AI 执行、多轮回归验收压回同一套工业化秩序。

当前治理层主要解决五类问题：

1. 用户的大白话长需求，如何被收束成真正可执行的切片，而不是继续放大歧义。
2. 多个线程 / 工作区并行推进时，谁在改什么、何时开工、何时收口、何时停车，如何一眼说清。
3. Scene、Prefab、热脚本、Unity / MCP live 写入等高风险现场，如何避免互撞与误收口。
4. 规则、工作区、memory、线程状态、用户汇报、治理回执之间，如何维持一致口径。
5. 用户直聊业务线程、治理线程审回执、线程完成后一刀交验，如何严格区分角色和输出格式。

## 3. 当前正式治理架构

### 3.1 规则栈

当前治理不是单靠某一份说明文档，而是有固定的规则栈：

| 层级 | 载体 | 当前职责 |
|---|---|---|
| 第 1 层 | 用户当前裁定 | 决定本轮真正要做什么、停什么、先什么 |
| 第 2 层 | `AGENTS.md` | 项目 live 规则总闸，定义主线锚定、技能前置、memory、汇报和 thread-state 硬要求 |
| 第 3 层 | `Steering规则区优化/当前运行基线与开发规则` | 维护当前规范快照、唯一状态说明、现行入口与实时口径 |
| 第 4 层 | `.kiro/specs/Codex规则落地` | 把典狱长、看守长、分发、回执、thread-state 尾巴、强制 skills 闸门等治理机制落成正式正文 |
| 第 5 层 | `.kiro/steering/*` | 提供按领域拆分的通用规则，如 UI、动画、物品、存档、文档、评审、场景修改等 |

这套规则栈的作用，是让业务线程在任何一轮施工前都先知道三件事：

1. 这轮真正服从谁。
2. 这轮应该先读什么。
3. 这轮不能越过哪些硬边界。

### 3.2 工作区栈

当前项目不是把所有治理内容塞在一个目录里，而是分成三层工作区：

| 工作区 | 当前职责 | 典型内容 |
|---|---|---|
| `Steering规则区优化` | 维护 live 基线、当前唯一状态、现行入口、历史口径纠偏 | 规范快照、唯一状态说明、活文档入口、迁移收口材料 |
| `Codex规则落地` | 把治理机制真正变成执行规则 | 典狱长 / 看守长规范、治理分发与回执规范、thread-state 接线尾巴、强制 skills 闸门 |
| 业务工作区 | 承接具体玩法和系统规则 | `spring-day1`、`NPC`、`SO设计系统与工具`、`农田系统`、`物品放置系统` 等 |

这意味着治理不是悬空挂在业务之上，而是通过“治理工作区 + 业务工作区”双线并行，持续回写到真实开发现场。

### 3.3 Memory 与审计栈

当前项目的上下文承接已经形成固定层级：

| 层级 | 载体 | 作用 | 当前口径 |
|---|---|---|---|
| 业务层 | 子工作区 `memory.md` | 记录最贴近当前功能的真实结论 | 先写子工作区，避免细节只留在聊天里 |
| 汇总层 | 父工作区 `memory.md` | 汇总阶段结论与跨子线依赖 | 负责把分散切片收回到主工作区 |
| 线程层 | `.codex/threads/Sunset/.../memory_0.md` | 记录线程视角的主线、阻塞、恢复点与阶段判断 | 保证同一线程下次能无缝接手 |
| 审计层 | `C:\Users\aTo\.codex\memories\skill-trigger-log.md` | 记录本轮实际触发了哪些 skills、为什么触发、用户是否可见 | 不替代业务 memory，只负责行为审计 |
| 能力层 | `C:\Users\aTo\.codex\memories\global-skill-registry.md`、`global-learnings` | 记录当前可用 skills 能力面与稳定经验 | 让 skills 和长期偏好不再散落在单轮对话里 |

当前正式顺序已经明确为：

1. 先子工作区。
2. 再父工作区。
3. 必要时补全局审计层。
4. 最后回写线程记忆。

### 3.4 现场约束层

当前治理现场由三套硬约束共同承载：

| 约束层 | 当前载体 | 作用 |
|---|---|---|
| Hook 层 | `.kiro/hooks/smart-assistant.kiro.hook`、`memory-update-check.kiro.hook`、`git-quick-commit.kiro.hook`、`git-preflight.kiro.hook` | 负责规则路由、memory 检查、Git preflight 等前置动作 |
| Lock 层 | `.kiro/locks/shared-root-queue.md`、`shared-root-branch-occupancy.md`、`mcp-single-instance-occupancy.md`、`mcp-hot-zones.md`、`mcp-live-baseline.md` | 管理 shared root、MCP 单实例与高危写入现场 |
| Thread-state 层 | `Begin-Slice.ps1`、`Ready-To-Sync.ps1`、`Park-Slice.ps1`、`Show-Active-Ownership.ps1` | 把线程从“准备施工、正在施工、可收口、已停车”的状态显式落盘 |

其中 `thread-state` 的意义尤其关键：

1. 线程不是靠聊天宣称自己在施工，而是要真的登记 `ACTIVE / READY / PARKED`。
2. 治理位不再只靠 memory 猜谁在动文件，而是能直接看现场状态。
3. 业务线程直聊场景下也必须自己跑这套状态，而不是等治理位提醒。

## 4. 当前角色与输出口径分工

### 4.1 业务线程

业务线程负责：

1. 按命中的工作区和规则栈施工。
2. 把用户的大白话需求转成真正可执行的切片和验收口径。
3. 对用户先交“保底六点卡”式的人话汇报，再补技术审计层。

### 4.2 典狱长模式

`典狱长` 不是验收位，而是治理总闸。它当前负责：

1. 先审线程回执是否说清“做成什么 / 还剩什么 / 下一步”。
2. 再判线程是继续施工、停给用户验收、停给用户分析，还是无需继续。
3. 只有确认该继续时，才生成下一轮 prompt 或批次分发。

### 4.3 看守长模式

`看守长` 是业务线程的一刀交接位，不是治理分发位。它当前负责：

1. 把线程最近完成的一刀整理成正式验收包。
2. 明确线程已自验、仍需用户终验的点和本轮不在范围内的点。
3. 给出验收清单矩阵、最少必测包、快捷回执单和完整版回执单。

### 4.4 三种固定输出口径

当前已经正式分离的三种输出口径是：

1. 用户可读汇报：先说当前主线、已完成、未完成、当前阶段、下一步、用户现在要不要做事。
2. 治理最小回执：说明 changed paths、风险点、验证状态、是否需要 Unity / MCP live 写。
3. 验收交接：在看守长模式下把某一刀收成可直接执行的验收包。

## 5. 当前标准推进流程

当前项目已经形成固定推进链：

1. 用户以大白话长文本提出需求，先由业务线程做 `skills-governor + 工作区路由` 的前置核查。
2. 根据主线命中业务工作区，并补读对应 `steering`、`memory`、活跃规范快照与治理规则。
3. 把需求压成单轮可验的纵切片，明确目标路径、完成定义和不在本轮范围内的内容。
4. 准备真实施工前执行 `Begin-Slice`，把本轮 owned paths 和切片目标登记到现场状态层。
5. 施工过程中用业务规则、Editor 工具、skills 和 Unity MCP 推进实现、验证与回归。
6. 准备白名单收口前执行 `Ready-To-Sync`；如果本轮先停、让位或阻塞，则执行 `Park-Slice`。
7. 对用户先交保底六点卡；若已到终验阶段，则进入看守长模式；若需要继续分发，则交由典狱长模式裁定。

这条流程里，当前最有代表性的治理动作包括：

1. `保底六点卡`：防止线程只交技术 dump，不说人话结果。
2. `回执卡`：让治理位能用固定字段快速判定是否可继续。
3. `停发裁定`：避免线程自己的尾账还没收干净，就被继续推去叠下一刀。

## 6. Skills、本地化适配与 Unity MCP

### 6.1 当前已接入生产链的核心 skills

当前被正式纳入 `Sunset` 协作链的 skills，已经不止“会不会用”，而是各自绑定了明确职责：

- `skills-governor`：做实质性任务前置核查、skill 路由和收尾审计。
- `sunset-startup-guard`：在项目侧统一核主线、工作区、shared root、MCP 占用与首个同伴 skill。
- `sunset-workspace-router`：把任务路由到正确工作区、memory 和现行入口。
- `sunset-scene-audit`：在 Scene / Prefab / Inspector 风险修改前做现场审计。
- `sunset-review-router`：处理锐评、审视报告和差异采纳。
- `sunset-unity-validation-loop`：统一 Unity 侧验证顺序。
- `sunset-prompt-slice-guard`：把下一轮 prompt 压回单刀可验的纵切片。
- `sunset-no-red-handoff`：保证代码 / Unity 现场不留红错离场。
- `acceptance-warden-mode`、`sunset-acceptance-handoff`：收线程终验包。
- `sunset-warden-mode`、`sunset-governance-dispatch-protocol`、`sunset-thread-wakeup-coordinator`：负责治理分发、回收、唤醒与停发。
- `sunset-ui-evidence-capture`、`sunset-lock-steward`、`sunset-doc-encoding-auditor`、`sunset-release-snapshot`、`sunset-rapid-incident-triage`：分别处理 UI 证据、热文件锁、编码健康、版本快照和快速事故定责。

### 6.2 Skills 的本地化适配方式

当前项目并不是把 skills 当成通用外挂，而是做了明确的本地化适配：

1. 每个关键 skill 都绑定 `Sunset` 的工作区入口、活文档、证据目录或现场约束，而不是只给抽象建议。
2. 业务线程第一条 `commentary` 必须显式说明当前命中了哪些 skills，避免“用了但用户看不见”。
3. 本轮 skill 触发会写入 `skill-trigger-log.md`，形成独立审计层。
4. 当前可用的 skill 面会同步回写到 `global-skill-registry.md`，避免后续重复造轮子。

### 6.3 `Skills和MCP` 线的实际贡献

`Skills和MCP` 这条历史线程已经真实完成过以下工作：

1. 把项目专用 skills 独立迁移并落根。
2. 补齐 `sunset-unity-validation-loop`，把 Unity 验证闭环从零散操作收成正式流程。
3. 清退旧桥口径和旧端口口径，避免多个 MCP 说法并存。
4. 把 MCP live 基线、候选方案与接入方式写成正式文档。
5. 把 skills 与 MCP 的使用方式吸收到当前治理体系，而不是停留在实验阶段。

### 6.4 当前 Unity MCP 基线

当前 Unity MCP 的正式口径已经很清楚：

1. 当前唯一 MCP server 名是 `unityMCP`。
2. 当前唯一 HTTP 端点是 `http://127.0.0.1:8888/mcp`。
3. live 成立必须同时经过配置层、服务层、会话层和实例层核验。
4. MCP 使用同时受 `mcp-single-instance-occupancy.md`、`mcp-hot-zones.md` 和 `mcp-live-baseline.md` 约束。
5. 旧桥已退出当前活口径，后续验证默认只测新桥 `unityMCP`。

## 7. 当前正式收口

当前 AI 治理卷的正式收口可以概括为六条：

1. `Sunset` 已经形成稳定的规则栈、工作区栈、memory 栈和现场状态栈，不再只靠聊天上下文勉强延续。
2. 业务线程、典狱长、看守长三种角色已经彻底分流，继续施工、治理分发和用户终验不再混写。
3. `hooks + locks + thread-state` 已经把并发、多线程和 live 现场约束从“口头提醒”升级为显式机制。
4. skills 已经被本地化进项目生产链，并且有显式触发展示、触发日志和能力注册表承接。
5. Unity MCP 已经从“能不能连上”升级为“何时算 live、怎样纳入正式验证闭环”的工程规则。
6. 当前这卷已经能够真实代表你在 `Sunset` 里对 AI 协同开发的主控能力，而不是泛泛写一句“会用 AI”。
