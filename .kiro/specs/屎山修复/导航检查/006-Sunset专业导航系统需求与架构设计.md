## 2. 这次设计真正要解决的，不只是“导航过没过”

### 2.1 用户真实目标

当前用户要的不是：

- `probe` 里三条 case 通过
- 玩家不再把 NPC 顶着走
- NPC-NPC 会车偶尔能成功

当前用户真正要的是：

- 一套能长期支撑 `Sunset` 的专业导航底座
- 一套对 2D 俯视角项目高度适配的移动基础设施
- 一套后续能直接承接：
  - 玩家自动导航
  - NPC 漫游
  - 怪物追击
  - 宠物跟随
  - 活体单位之间的自然礼让与会车
  的统一系统

### 2.2 当前验收标准必须升级

旧标准过于低级：

- “没卡死”
- “受控验证能过”
- “没有直接重叠”

新标准必须拆成两条：

1. 功能验收线
   - 受控场景稳定通过
   - 没有死锁、没有爆栈、没有明显逻辑回退

2. 体验验收线
   - 玩家面对 NPC 不再呈现明显推土机感
   - 活体单位不再带明显大圆壳体感
   - 单位遇到冲突时表现像“交通参与者”，不是像“磁铁球”

只过功能线，不等于通过导航验收。

## 3. 当前导航系统的根问题重新定性

### 3.1 当前不是“统一动态导航系统”

当前项目真实存在的是一套混合系统：

1. 静态建路层  
   `NavGrid2D`

2. 共享执行层  
   `NavigationPathExecutor2D`

3. 启发式局部避让层  
   `NavigationLocalAvoidanceSolver`

4. 玩家包装层  
   `PlayerAutoNavigator`

5. NPC 包装层  
   `NPCAutoRoamController`

它共享了一部分底座，但没有真正共享完整的动态交通语义。

### 3.2 当前反复爆病相的根因

当前问题不是一个参数问题，而是 4 个结构问题叠加：

1. 静态地貌认知和动态活体交通没有严格断层
2. 微观避让通过补丁方式反向污染了宏观路径执行
3. 玩家与 NPC 的最终运动语义没有真正统一
4. 当前系统更偏“强目标推进”，而不是“交通裁决优先”

### 3.3 当前最关键的错误起点

当前系统最初的偏差，不是“完全走错”，而是：

- 它把“统一导航”理解成了“统一执行层 + 后补动态避让”

而不是：

- 先定义统一动态交通规则，再让执行层和行为层共同服从这套规则

这就是为什么它一路会长出：

- 推土机
- 不推进
- 慢蹭
- detour 清理风暴
- detour 清理后恢复原目标失败

## 4. 新导航系统的总目标

新系统的目标不是“算法更花哨”，而是做到下面 7 件事。

### 4.1 空间真相一致

- 系统必须尊重单位真实占地，而不是把所有活体都近似成夸张统一大圆壳
- 可以允许 broad-phase 用简化近似，但 narrow-phase 和 clearance 判定必须尽量贴合脚底真实 footprint

### 4.2 宏观与微观断层

- 宏观寻路只管静态世界
- 微观避让只管动态世界
- 二者可以协同，但不能混成一套模糊逻辑

### 4.3 活体交通语义成立

系统必须支持明确的：

- `Proceed`
- `Yield`
- `Wait`
- `SidePass`
- `Follow`
- `Chase`
- `Recover`

而不是只会输出一个“修过的方向向量”

### 4.4 玩家、NPC、怪物、宠物共享同一底座

这些单位可以有不同脑子，但必须共用：

- 同一静态建路服务
- 同一动态上下文采样
- 同一交通裁决标准
- 同一运动学收口原则

### 4.5 真实玩法体验优先

系统不能只服务受控 `probe`。

它必须对真实玩法负责：

- 玩家右键任意点位
- NPC 漫游中间态
- 多活体交错
- 场景静态障碍混合
- 后续战斗追击/跟随

### 4.6 状态生命周期可解释

任何中间态必须能被明确解释：

- 为什么停
- 为什么让
- 为什么绕
- 为什么恢复

不能再出现“看上去像随机卡住，其实是 5 层补丁叠加”的黑盒状态。

### 4.7 迁移可执行

这不是推倒重来文艺稿。

新设计必须明确：

- 哪些现有资产继续用
- 哪些资产只保留骨架
- 哪些语义必须彻底废止

## 5. 红线与硬约束

以下内容作为新导航系统的硬红线。

### 5.1 动态活体不得再写进静态网格阻挡语义

活体单位可以参与：

- 局部交通裁决
- 动态占位预测
- 近距 clearance

但不得再直接当作静态网格里的长期不可走障碍。

否则永远会继续制造“双层壳体感”。

### 5.2 微观避让不得直接污染宏观路径主语义

动态避让不能再以“修改主路径列表”为默认策略。

允许存在临时中间目标，但它必须满足：

- 生命周期独立
- 不反向改写主路径走廊
- 退出后自动回归主走廊
- 不再触发 clear/rebuild 风暴

换句话说：

- 问题不在“绝不能有临时中间目标”
- 问题在“不能再把它写成主路径状态的一部分”

### 5.3 玩家与 NPC 的最终运动学语义必须统一

这里的“统一”不是强制同一个函数实现，而是：

- 同样接受 `FinalVelocity / FinalMotionIntent`
- 同样受物理世界约束
- 同样遵守停让和碰撞响应原则

是否全部用 `linearVelocity`，是实现选择。  
但“不再允许一个单位是速度推进，另一个单位是强制挪位而彼此手感完全不同”是硬要求。

### 5.4 体验语义优先于数学补丁

未来不允许再出现这种决策顺序：

- 先目标前冲
- 再拿一堆 `Lerp` 权重去修方向

正确顺序应是：

- 先判断当前该不该前冲
- 再决定是让行、等待还是通过
- 最后才计算运动向量

### 5.5 受控 probe 不能替代真实体验验收

以后任何“导航已完成”的说法，必须明确它通过的是：

- `probe` 口径
- 还是自由玩法口径

不能混说。

## 6. 现有代码资产审计：哪些保留，哪些重构，哪些淘汰

### 6.1 可继承资产

以下资产不是问题本身，可以继承：

1. `NavGrid2D`
   - 继续作为静态地貌服务
   - 但要净化数据源和职责边界

2. `NavigationAgentRegistry`
   - 继续作为动态单位注册表/上下文采样入口

3. `NavigationPathExecutor2D`
   - 可以保留“路径状态 / path index / progress / stuck 监测”这些通用执行骨架
   - 但要剥离被 detour 污染的旧语义

4. 当前 `probe` 与测试资产
   - 继续保留为功能回归层
   - 但必须降级为“功能线验证”，不再冒充最终体验线

### 6.2 需重构资产

1. `NavigationAvoidanceRules`
   - 从“谁让谁”简表，升级成真正的交通裁决规则层

2. `NavigationLocalAvoidanceSolver`
   - 从启发式魔法权重求解器，升级为“上下文感知 + 交通状态驱动”的微观运动求解层

3. `PlayerAutoNavigator`
   - 从“强目标推进 + 避让补丁包装”，收口成“玩家脑层 + 通用导航意图输出”

4. `NPCAutoRoamController`
   - 从“NPC 漫游 + 路径执行 + 避让补丁 + 运动收口混装”，拆成脑层与运动底盘之间的更清晰责任关系

### 6.3 必须废止的旧语义

1. 活体直接作为静态网格阻挡
2. detour 直接写进主路径语义
3. 玩家 / NPC 长期维持两套明显不对称的运动学体感
4. 把体验问题继续伪装成“只是参数没调好”

## 7. Sunset 目标导航架构

本项目推荐采用 5 层架构，而不是继续堆一层共享脚本。

## 7.1 Layer A：意图与行为脑层

职责：

- 决定“我是谁、我要去哪、为什么要去”

典型来源：

- 玩家右键点地
- NPC 漫游
- 怪物追击
- 宠物跟随

输出：

- `Target`
- `DesiredSpeed`
- `BehaviorMode`
- `PriorityProfile`

限制：

- 不知道 A*
- 不知道碰撞细节
- 不知道 detour
- 不直接操纵 Rigidbody

### 7.2 Layer B：宏观静态路径层

职责：

- 提供静态走廊
- 回答“大方向怎么去”

输入：

- 当前静态地图
- 起点
- 终点

输出：

- `PathCorridor`

红线：

- 只认静态世界
- 不认活体临时交通
- 不因动态单位改写世界可走性

### 7.3 Layer C：动态上下文与交通裁决层

职责：

- 收集周围动态实体
- 判断当前交通语义

这是新系统真正的核心层。

它必须回答：

- 前方是谁
- 对方在干什么
- 我们是谁优先
- 当前空间足够错身还是必须停让
- 当前应该：
  - `Proceed`
  - `Yield`
  - `Wait`
  - `SidePass`
  - `Follow`
  - `Chase`
  - `Recover`

它输出的首先应该是“交通决策”，其次才是“方向向量”。

### 7.4 Layer D：微观运动求解层

职责：

- 在 Layer C 已给出交通裁决后
- 结合真实 footprint / clearance / moving context
- 求解下一帧 `FinalVelocity` 或 `FinalMotionIntent`

它不再承担：

- 决定谁让谁
- 决定是否应该停车
- 决定是否该切行为模式

这些必须在 Layer C 已经裁决完。

### 7.5 Layer E：统一运动执行层

职责：

- 把 `FinalVelocity / FinalMotionIntent` 统一落到物理世界
- 驱动动画
- 对外暴露统一运动状态

强约束：

- 玩家和 NPC 最终都要从这一层收口语义
- 即便内部实现不同，也不能再让用户明显感到“一个像推速度，一个像瞬移挪位置”

## 8. 空间模型设计

### 8.1 Broad-phase 与 narrow-phase 分离

为了性能和手感兼顾，动态活体应采用两层空间模型：

1. broad-phase
   - 用较轻量的近似快速筛邻居

2. narrow-phase
   - 用更贴合脚底 footprint 的模型算真实 clearance

这意味着：

- 圆形近似可以保留在粗筛阶段
- 但不能再把粗筛半径直接当成最终体感边界

### 8.2 脚底 footprint 才是体验真相

在 `Sunset` 这种 2D 俯视角项目里，真正重要的是脚底占地，而不是整个人物 sprite 的最大外接圆。

新系统必须围绕：

- foot box
- foot ellipse
- 或 shape-aware footprint

来定义 clearance，而不是继续用夸张统一圆环统治所有行为。

### 8.3 静态世界与动态世界必须双通道

静态世界处理：

- 墙
- 建筑
- 树
- 岩石
- 永久不可走区域

动态世界处理：

- 玩家
- NPC
- 怪物
- 宠物

同一对象不得同时在两条通道里担任相同语义的阻挡者。

## 9. 交通语义设计

### 9.1 所有活体单位必须共享一套基础交通词典

基础词典如下：

1. `Proceed`
   - 正常沿走廊前进

2. `Yield`
   - 有冲突，当前单位主动让行

3. `Wait`
   - 当前不该继续向前，速度应明确归零

4. `SidePass`
   - 当前允许小幅横向错身通过

5. `Follow`
   - 保持目标与间距，不触发追尾推挤

6. `Chase`
   - 面向动态目标接近，但仍受交通规则约束

7. `Recover`
   - 冲突解除后回归原走廊/原目标

### 9.2 玩家不是天然的“只会前冲”

玩家自动导航不能再默认：

- 目标永远优先于交通

而必须支持：

- 自动让行
- 暂停等待
- 再恢复前进

否则用户永远会继续看到推土机感。

### 9.3 NPC 不是天然的“永远礼让”

NPC 也不能一律弱势。

新系统必须支持：

- 玩家让行 NPC
- NPC 让行玩家
- NPC-NPC 会车裁决
- 后续怪物追击时的压迫优先级
- 宠物跟随时的柔性从属优先级

优先级不是唯一规则，但必须是规则的一部分。

## 10. 对未来系统的具体适配要求

### 10.1 玩家自动导航

要求：

- 面对移动 NPC 时能自然停、让、再走
- 不再像磁铁球贴边滑
- 不再出现“绕一下立刻猛冲回去”的强裂变体感

### 10.2 NPC 漫游

要求：

- 会车自然
- 遇玩家不再只是卡停或磨边
- 暂停/起步/转向的中间态不会破坏交通语义

### 10.3 怪物追击

要求：

- 有更强目标性
- 但仍遵守统一运动学和交通系统
- 不允许为了“更凶”重新走回推土机架构

### 10.4 宠物跟随

要求：

- 能保持队形与间距
- 不追尾玩家
- 不把玩家当静态墙
- 不和 NPC 持续互推

## 11. 验收口径

## 11.1 功能线验收

至少包含：

1. 玩家绕移动 NPC
2. NPC 绕玩家
3. NPC-NPC 会车
4. 玩家穿混合静态障碍 + 漫游 NPC
5. 怪物追击基础样例
6. 宠物跟随基础样例

### 11.2 体验线验收

必须人工观察并记录：

1. 是否仍有明显大圆壳体感
2. 是否仍像推土机贴边滑
3. 是否仍出现“停一下又猛冲”的违和恢复
4. 是否明显像活体交通，而不是数学避障

### 11.3 回归线验收

必须长期保留：

- `probe` 回归
- 真实玩法回归
- 文档化复测步骤

## 12. 迁移策略

新系统不能一口气推翻全部代码，应分阶段迁移。

### 阶段 1：语义净化

目标：

- 把“当前架构到底哪里错”正式钉死
- 明确剥离静态/动态阻挡语义
- 停止继续给旧启发式系统补无限 patch

阶段产物：

- 当前文档
- 更新后的主表与迁移路线

### 阶段 2：底座分层

目标：

- 保留可继承资产
- 拆出新的职责边界

重点：

- `NavGrid2D` 降级为纯静态路径服务
- 执行层保留通用骨架
- 交通裁决层单独成层

### 阶段 3：交通语义上线

目标：

- 正式引入 `Proceed / Yield / Wait / SidePass / Recover`

阶段结果：

- 玩家与 NPC 不再只是输出方向修正
- 而是先输出交通决策，再输出运动意图

### 阶段 4：统一运动学收口

目标：

- 玩家 / NPC / 后续怪物 / 宠物 在最终运动学层面达成统一语义

### 阶段 5：体验线验收

目标：

- 不只看 `probe`
- 正式验证真实玩法手感

## 13. 给后续代码线程的边界说明

后续代码线程不该直接把本文件翻译成“写哪些函数”。

后续代码线程该从本文件里拿走的是：

1. 哪些旧语义必须停止
2. 哪些层级必须新增或拆分
3. 哪些现有资产可以利用
4. 当前每一阶段该产出什么验证证据

代码线程必须避免两种错误：

1. 继续在旧启发式系统上无限补丁
2. 完全无视当前资产，另起一套与项目脱节的空中楼阁

## 14. 当前最终判断

### 14.1 对当前实现的判词

当前实现不是“完全没价值”。

它已经完成了两件有价值的事：

1. 共享执行层和共享动态单位上下文已经出现
2. 受控场景已经从系统级失败拉回到了可通过

但它仍然不是项目最终需要的导航底座。

### 14.2 对后续方向的判词

从现在开始，导航线不应再以：

- “继续调参数”
- “继续补一个 detour bug”
- “继续证明 probe 能过”

作为主线。

从现在开始，导航线的主线应改为：

- 建立 `Sunset` 的专业导航基础设施

也就是：

- 以项目级需求为起点
- 以统一交通语义为核心
- 以未来扩展能力为约束
- 以功能线 + 体验线双验收为标准

## 15. 下一步唯一推荐动作
