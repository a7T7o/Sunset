# 2026-04-05｜导航统一执行内核方向锚定

## 1. 这份文档解决什么

这份文档只解决一件事：

- 在不推翻当前玩家已认可导航版本的前提下，
- 给后续“玩家 / NPC / 其他可移动单位共用统一导航接口”的改造，
- 定一个不会变成黑洞的落地方向。

这不是“大重构动员令”，而是约束文档。

---

## 2. 当前客观事实

### 2.1 已经统一的层

当前项目里，下面这些底层已经是共享的：

- `NavGrid2D`
- `NavigationPathExecutor2D`
- `NavigationAgentRegistry`
- `NavigationLocalAvoidanceSolver`
- `INavigationUnit`

也就是说：

- 玩家和 NPC 已经共用同一张静态导航网格
- 已经共用路径数据结构、waypoint 推进基础件和局部避让底层

### 2.2 还没有统一的层

真正没有统一的是“拿到路径以后每帧怎么走”这一层。

当前是两套：

- 玩家：
  - `Assets/YYY_Scripts/Service/Player/PlayerAutoNavigator.cs`
  - `Assets/YYY_Scripts/Service/Player/PlayerMovement.cs`
- NPC：
  - `Assets/YYY_Scripts/Controller/NPC/NPCAutoRoamController.cs`
  - `Assets/YYY_Scripts/Controller/NPC/NPCMotionController.cs`

### 2.3 现在真正的问题

现在的问题不是“导航图不统一”，而是：

- 同一张图
- 两套 path-following / traversal 执行逻辑
- 玩家修好的 bug，NPC 还会再长一遍

这就是当前重复返工和体验不一致的根因。

---

## 3. 唯一正确的统一边界

### 3.1 要统一什么

只统一中间这一层：

- `Traversal / Path Following / Step Execution Core`

这层应只负责：

- 建路后的 waypoint 推进
- shared avoidance
- bounds enforcement
- traversal soft-pass
- no-progress / stuck / oscillation 判定
- rebuild / recover / arrived / blocked 的统一输出

### 3.2 不统一什么

下面这些不应该硬并成一套大控制器：

- 目标来源
- 业务状态机
- 动画与动作层
- 完成语义

具体说：

- 玩家保留：
  - 右键输入
  - 点击终点完成合同
  - follow target
  - 玩家专属 blocked input / 到点判定
- NPC 保留：
  - roam 选点
  - short pause / long pause
  - ambient chat
  - move cycle / interruption

### 3.3 结论

正确方向不是：

- “统一整个 PlayerAutoNavigator 和 NPCAutoRoamController”

正确方向是：

- “统一 traversal 执行内核”

---

## 4. 目标结构

后续应形成三层：

### A. 目标来源层

谁发目标点，谁决定何时开始。

例子：

- 玩家：右键点击传目标
- NPC：漫游状态机 / 剧情系统传目标

### B. 统一 traversal 执行层

输入：

- 当前导航体位置
- 当前路径 / 目标点
- 单位半径 / 占位规则
- 可选完成距离 / 特殊合同参数

输出：

- `Move`
- `BlockedMove`
- `HardStop`
- `Arrived`
- `Rebuild`
- `Interrupt`

### C. 运动落地层

谁真正把速度/位移交给物理和动画，谁继续各自保留：

- 玩家继续走 `PlayerMovement`
- NPC 继续走 `NPCMotionController`

---

## 5. 实施顺序

这是唯一推荐顺序，不能反过来。

### Phase 1：先抽窄核心，不动玩家业务语义

先从现有玩家/NPC 代码中抽出共享 traversal 执行步骤，但：

- 不改玩家右键完成定义
- 不改 NPC roam/chat 业务定义
- 不改场景配置

这一阶段只做“逻辑抽心”。

### Phase 2：先让 NPC 接入共享内核

原因：

- 当前玩家版本已被用户认可
- NPC 是当前重复长 bug 最严重的一侧

所以应先做：

- NPC 的 `TickMoving()` 改为走共享 traversal core
- NPC 状态机只保留“什么时候给目标”

### Phase 3：NPC 跑稳后，再让玩家接入同一内核

玩家不该先动。

必须在 NPC 稳定后再做：

- 玩家把路径执行部分切到同一个 core
- 但继续保留玩家的 arrival contract

### Phase 4：再开放给其他可移动体

例如：

- 剧情单位
- 特殊 follower
- 其他会移动的交互体

这一步必须在玩家/NPC 统一 core 稳住以后。

---

## 6. 明确禁止的错误路线

### 禁止 1：直接把 NPC 改成 PlayerAutoNavigator

原因：

- 玩家文件里混有大量玩家专属完成语义
- 强塞给 NPC 只会变成 `if (isNpc) ... else ...`

结果一定会变屎山。

### 禁止 2：继续长期维持双实现，只互相抄 bugfix

原因：

- 这就是当前反复返工的来源
- 会导致“玩家一套绿、NPC 一套红”

### 禁止 3：一刀大重构，把目标来源、状态机、运动落地全并了

原因：

- 范围太大
- 风险不可控
- 会直接伤到已经认可的玩家导航版本

---

## 7. 当前最小落地切口

后续真正开工时，第一刀只允许做下面这些：

1. 新增一个共享 traversal core 文件或窄类
2. 先把 NPC 当前 `TickMoving()` 里与 path-following 直接相关的逻辑搬进去
3. 保留 NPC 的：
   - `TryBeginMove`
   - `EnterShortPause`
   - `EnterLongPause`
   - `ambient chat`
   - interruption snapshot
4. 不动玩家右键完成语义
5. 不动 scene / binder / inspector 配置

如果第一刀超出这些边界，就说明开始失控了，必须停。

---

## 8. 完成定义

只有满足下面 4 条，才算这条统一方向真的站住：

1. 玩家当前已认可版本不回退
2. NPC 不再出现“玩家修好了、NPC 还要再补一遍”的 path-following 同类 bug
3. 玩家与 NPC 的 traversal 行为在同一类场景下表现一致
4. 上层差异只剩：
   - 谁给目标
   - 什么时候算到点
   - 谁来播动画和状态机

---

## 9. 风险判断

### 这条路为什么是可实施的

因为当前并不是从零统一。

已经统一的底层足够多，真正需要抽的只有中层 traversal step。

### 这条路什么时候会变黑洞

只会在两种情况下变黑洞：

1. 试图统一整个 controller
2. 在第一刀里同时改玩家业务语义 + NPC 状态机 + 运动落地

只要守住本文档的边界，这条路不是黑洞。

## 11. 一句话总判断

“玩家 / NPC / 所有可移动体共用统一导航接口”这个目标是对的；

但真正应该统一的是：

- `traversal 执行内核`

而不是：

- `整个玩家控制器`
- `整个 NPC 控制器`

后续只要严格按这个边界推进，就是可落地、可分阶段、可回退的方案。
