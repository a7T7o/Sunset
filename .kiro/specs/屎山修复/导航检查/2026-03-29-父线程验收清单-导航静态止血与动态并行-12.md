# 2026-03-29-父线程验收清单-导航静态止血与动态并行-12

本清单只服务父线程当前这一阶段的双线并行验收：

1. 父线程自己验静态点导航止血
2. `导航检查V2` 继续验动态避让/高保真矩阵

目标不是重新讲大架构，  
而是把“哪些证据算静态有效、哪些算动态有效、哪些一出现就整轮作废”钉死。

---

## 一、总原则

这轮先看两件事：

1. 普通地面点导航的契约有没有继续被混错
2. 静态与动态 live 是否继续互相污染

如果这两件事没有先讲清，  
后面的 pass / fail 一律降级。

---

## 二、静态线验收门

### Gate S0：契约门

父线程当前正式契约固定为：

1. 普通地面点导航使用“玩家实际占位中心”语义
2. 跟随交互目标才使用 `ClosestPoint + stopRadius`
3. 这两套不得继续混用

当前代码触点固定看：

1. `PlayerAutoNavigator.GetPlayerPosition()`
2. `PlayerAutoNavigator.GetPathRequestDestination()`
3. `PlayerAutoNavigator.CompleteArrival()`

如果下一轮这些点被回退，  
静态线直接判回归。

### Gate S1：静态 runner 起跑门

当前静态验证工具固定看：

1. `NavigationStaticPointValidationRunner.cs`
2. `NavigationStaticPointValidationMenu.cs`

最低有效起跑信号必须同时满足：

1. 出现 `runtime_launch_request=MarkerFile` 或等价静态启动信号
2. 出现 `runner_started`
3. 出现 `accepted_case_count >= 1`
4. `case_start` 中的 `navTarget` 与 `target` 一致

如果只看到 `case_start` 之前的日志，  
还不能宣称静态线已测完。

### Gate S2：静态样本门

当前静态样本筛选口径固定为：

1. 允许固定偏移候选里只保留开放地面样本
2. 过滤逻辑以“路径长度比 + 直线横向偏移”为准
3. 不再使用 `pathProbeBuffer.Count > 2` 这种错误口径

当前 fresh 已知事实：

1. `(1.60, 0.00)` 与 `(0.00, 1.60)` 已被接受
2. `(1.15, 1.15)` 与 `(-1.40, 0.20)` 因 `lengthRatio / maxLateralDeviation` 超线被拒绝

如果后续样本再次变成 `accepted_case_count=0`，  
必须先回答是筛选回退，还是现场被污染。

### Gate S3：静态结果门

静态结果只有在同时满足下面 3 条时才算有效：

1. 出现 `case_end` 或 `all_completed`
2. `centerDistance` 是主验收值
3. 整个 `case_start -> case_end/all_completed` 窗口内没有外部 live 干扰

如果没有 `case_end`，  
只能判“起跑已成功，结果未闭环”。

### Gate S4：静态污染否决门

只要在静态 case 窗口内出现下面任一条，本轮静态结果直接作废：

1. `[NavValidation]`
2. `MCP-FOR-UNITY: [ExecuteMenuItem] Handling menu item command`
3. `[SpringUiEvidence]`
4. `[DialogueDebugMenu]`
5. 其他会改玩家位置、输入、对话、UI 截图或 probe setup 的外部菜单/runner 信号

当前 fresh 已证实：

1. 静态 runner 可以干净起跑
2. 但 shared Unity 现场仍会被外部 `ExecuteMenuItem` 抢占
3. 因此当前静态线最新结论是：
   - `契约修正已进代码`
   - `静态验证工具已能找到样本`
   - `fresh 最终结果仍需独占 live 窗口才能成立`

---

## 三、动态线验收门

动态线仍由 `导航检查V2` 负责，  
父线程这轮只看 4 件事：

1. 不准再把静态点契约当动态问题去乱修
2. 不准再触碰父线程当前静态工具链：
   - `PlayerAutoNavigator.cs` 的静态点契约触点
   - `NavigationStaticPointValidationRunner.cs`
   - `NavigationStaticPointValidationMenu.cs`
3. 不准在父线程静态 live 窗口里继续排 `execute_menu_item / Play / pending action`
4. 继续按高保真动态矩阵口径收窄真正的动态坏相

动态线下一轮如果要 claim 有效，  
必须先保证没有踩回静态 runtime 触点。

---

## 四、父线程下一轮审查顺序

顺序固定为：

1. 先看静态契约触点有没有回退
2. 再看静态 runner 是否拿到有效起跑信号
3. 再看静态 live 是否被外部菜单污染
4. 然后才看动态线回执是否尊重边界
5. 最后才看 `changed_paths / dirty / sync`

---

## 五、当前阶段结论

当前最准确的状态不是：

- “静态已经修好”
- 也不是“静态还是完全没动”

而是：

1. 静态点导航的目标语义修正已经落进代码
2. 静态验证工具已经从黑盒变成可解释
3. 当前 fresh live 的第一阻塞不再是样本筛选错误
4. 当前 fresh live 的第一阻塞是 shared Unity 里有外部菜单命令抢占执行窗口

这意味着下一轮父线程验收时，  
必须先判“证据是否被污染”，  
再判“静态导航本体是不是还坏”。
