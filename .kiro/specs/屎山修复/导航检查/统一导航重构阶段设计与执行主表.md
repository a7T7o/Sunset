## 3. 当前问题重新定性

## 3.1 现状不是一个统一导航系统

当前项目里真正存在的是三套彼此耦合但没有统一收口的导航现实：

1. **静态路径规划现实**
   - 位于 `D:\Unity\Unity_learning\Sunset\Assets\YYY_Scripts\Service\Navigation\NavGrid2D.cs`
   - 负责世界可走性与 A* 路径搜索
   - 适合静态或准静态障碍

2. **玩家导航现实**
   - 位于 `D:\Unity\Unity_learning\Sunset\Assets\YYY_Scripts\Service\Player\PlayerAutoNavigator.cs`
   - 自己维护目标、路径、局部避障、卡住重建与停止判定

3. **NPC 漫游现实**
   - 位于 `D:\Unity\Unity_learning\Sunset\Assets\YYY_Scripts\Controller\NPC\NPCAutoRoamController.cs`
   - 自己维护随机选点、路径跟随、卡住恢复、漫游节奏与最终移动

这意味着：

- 玩家和 NPC 不是在“同一套导航系统”中移动
- 只是碰巧共享了同一个静态网格路径服务

## 3.2 当前现场暴露出的根问题

从最初导航审计、后续代码推进和当前现场症状综合看，真正的问题已经不是“某个路径点算得不准”，而是下面三件事同时成立：

1. **静态路径层与动态代理层混在一起**
   - `NavGrid2D` 只适合静态/准静态障碍
   - 但我们一直在试图用它解释移动 NPC 带来的阻挡问题

2. **玩家与 NPC 各自维护一套局部逻辑**
   - 玩家侧补一点
   - NPC 侧补一点
   - 卡住再各自重建一点
   - 这是典型的“屎山上雕花”前兆

3. **最终运动语义并不一致**
   - 玩家主要通过 `PlayerMovement` 的 `rb.linearVelocity`
   - NPC 主要通过 `NPCAutoRoamController` 的 `rb.MovePosition(...)`
   - 即便路径一致，最终接触层的空间结果也可能不一致

## 3.3 当前必须接受的结论

**结论：需要统一导航核心。**

但这个“统一”不是：

- 所有对象共用一个超大脚本

而是：

- 所有移动体共用同一套导航基础设施  
- 各自保留自己的行为脑与运动适配器

### 3.4.4 当前真实架构

当前 live + 代码综合后的真实架构，不是“目标中的 5 层都已成立”，而是下面这套过渡结构：

1. **静态路径层**
   - `NavGrid2D`
   - live 承载对象：`Primary/2_World/Systems`
2. **共享动态代理层**
   - `NavigationAgentRegistry`
   - `NavigationAvoidanceRules`
   - `NavigationLocalAvoidanceSolver`
   - 当前主要作为代码层公共逻辑被玩家/NPC 两侧调用
3. **玩家私有执行层**
   - `GameInputManager -> PlayerAutoNavigator -> PlayerMovement`
   - 仍由玩家自己维护路径推进、replan、arrival、blocked-state
4. **NPC 私有执行层**
   - `NPCAutoRoamController -> NPCMotionController`
   - 仍由 NPC 自己维护 roam brain、路径推进、stuck、pause 和最终位移
5. **Scene 承载层**
   - `Systems` 仍是“静态路径 + 世界生成 + 输入入口”的混装节点

所以当前真实状态应该被描述为：

> “共享局部避让工具已经接入，但玩家与 NPC 仍旧运行在两套私有执行器上；静态路径承载与 Scene 语义也还没有收束。”

### 3.4.5 目标架构

目标不是继续往 `Systems` 里塞更多导航脚本，而是把 live Scene 与代码结构一起对齐到下面这套形态：

1. **NavigationRoot**
   - 专门承载导航系统
   - 至少应承载：
     - `NavGrid2D`
     - 未来需要落到 scene/service carrier 的导航核心组件
2. **Systems**
   - 回到世界生成 / 世界调试 / 其他系统服务
   - 不再混装导航核心
3. **共享路径执行层**
   - 统一承接：
     - path
     - pathIndex
     - corridor / waypoint 推进
     - stuck
     - replan
     - arrival
4. **行为脑层**
   - 玩家输入脑只发“去哪里”
   - NPC 漫游脑只发“去哪里”
5. **运动适配层**
   - 玩家 / NPC 只消费统一执行层给出的最终运动意图
   - 不再各自决定“遇到阻挡时到底怎么顶、怎么蹭、怎么停”

### 3.4.6 为什么十多轮迭代后现场仍然几乎没有本质变化

当前可以明确写下来的原因至少有五条：

1. **S1-S3 落了，S4 没落**
   - 共享 solver 已经有了
   - 但共享路径执行层还没有成立
2. **共享逻辑被嵌进了两套旧执行器，而不是替换掉旧执行器**
   - 结果就是“共享一点、私有一点”
3. **最终运动语义仍然分裂**
   - 玩家与 NPC 没有真正服从同一个 blocked / yield / stop 语义
4. **Scene 承载没有跟着架构一起收束**
   - `Systems` 继续混装，导致 live 现场仍然像旧系统，而不是新架构
5. **文本理解被错当成运行态事实**
   - `005-gemini-1.md` 对主表的理解是“目标架构长什么样”
   - 但它不等于现场已经做到

### 3.4.9 `NavigationRoot` 迁移五段分析（2026-03-23 / 002-prompt-2）

#### 1. 原有配置

当前 live Scene 中，`Primary/2_World/Systems` 仍是 `2_World` 下唯一子物体，并同时承担：

1. `NavGrid2D` 静态路径承载
2. `WorldSpawnService` 世界生成服务
3. `WorldSpawnDebug` 世界生成调试
4. `GameInputManager` 输入 / 玩家导航入口

同时，玩家 `PlayerAutoNavigator.navGrid` 与 3 个 live NPC `NPCAutoRoamController.navGrid` 都显式引用这个 `Systems/NavGrid2D`。

#### 2. 问题原因

`NavGrid2D` 继续挂在 `Systems` 上，已经不利于语义收口，原因有五条：

1. Scene 承载把“静态路径底座”与“世界生成 / 调试 / 输入入口”混在同一节点，导航核心没有独立边界。
2. 玩家与 NPC 都已经共享同一个 `NavGrid2D`，但承载对象名字仍然暗示“泛系统桶”，不利于后续继续挂共享导航核心。
3. 当前真正要推进的是 `NavigationRoot + S4 共享路径执行层`，如果承载仍留在 `Systems`，后续只会继续把导航实现往旧桶里塞。
4. 当前用户验收关注的是“统一导航系统是否真正成立”，而不是“某个系统桶里又多了几个导航脚本”。
5. 如果不先把承载语义收回来，后续 live 核查仍会持续出现“代码层想统一，Scene 层仍像旧系统”的错位。

#### 3. 建议修改

本轮默认直接建立独立承载对象 `Primary/2_World/NavigationRoot`，并把导航核心从 `Systems` 中剥离出来，最低落地要求如下：

1. 在 `2_World` 下新建 `NavigationRoot`
2. 在 `NavigationRoot` 上挂载新的 `NavGrid2D`
3. 把旧 `Systems/NavGrid2D` 的关键配置复制到 `NavigationRoot/NavGrid2D`
