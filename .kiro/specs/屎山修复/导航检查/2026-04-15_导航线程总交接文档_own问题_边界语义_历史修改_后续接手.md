# 2026-04-15｜导航线程总交接文档｜own 问题、边界语义、历史修改、后续接手

## 0. 文档定位

这不是“给单一线程的一段续工 prompt”，而是一份正式总交接文档。

它的用途是：

1. 把导航线程到目前为止到底改了什么、没改什么、为什么还没过线，彻底讲清楚。
2. 把导航、`Day1`、`NPC locomotion body` 之间的边界、交叉污染点、禁止再碰点讲清楚。
3. 把用户最新裁定、用户给 `Day1` 的语义、`Day1 -> 导航` 的 prompt 边界，以及 `NPC` 线程的接口判断，一次性收成一个统一入口。
4. 让后续接手者不需要再从长聊天、散 memory、旧 prompt 和现场截图里重新拼故事。

这份文档不是在宣称“导航已经修好”。

当前最准确的总判断是：

- 导航 own 现在已经做过多轮止血和一部分结构修正。
- 但玩家可感知的问题还没有真正过线。
- 用户最新把问题压成 3 个主根：
  1. `NPC 漫游走的都是坏点`
  2. `NPC 的静态导航计算还是垃圾`
  3. `NPC 阻塞卡死了不会自救`
- 这个归纳是对的，而且比导航线程前几轮自己的拆法更接近主根。

## 1. 用户最新裁定与语义基线

### 1.1 用户对导航 own 的最新主判断

用户已经明确把当前导航问题压成 3 个主根：

1. `坏点`
   - 当前 roam 目标点、detour 点、重规划点、最终落点，经常贴在静态边界或坏位置附近。
2. `静态导航垃圾`
   - 当前静态世界可达性判断仍然太像“先走过去再说，走不过去再补救”，而不是“先证明这条路本来就安全”。
3. `不会自救`
   - 一旦局部阻塞、互卡或贴边磨步形成，小系统会反复抖动、原地修同一小段，而不是进入稳定脱困。

这 3 条不是情绪描述，而是当前最有价值的施工拆解。

### 1.2 用户给 Day1 的剧情语义裁定

这些内容虽然不属于导航 own，但对导航后续接手极关键，因为它们决定了导航到底该接哪一段：

1. `opening`
   - 进入 `Town` 后，所有 NPC 包括 `001/002` 都应先传送到用户配置好的起点。
   - 对话框真正弹出前，应先从起点走到终点。
   - 最多等 5 秒；5 秒没到位就直接 snap 到终点。
   - 任意对白进行中，不再继续走位。
   - 对话结束后才放人。

2. `opening 结束后`
   - `001/002` 继续吃 `house lead / 进 Primary` 这条剧情语义。
   - `003` 不应再特殊化；应和 `101~203` 一样进入同一条 resident movement contract。
   - “走回 anchor/home 并恢复日常 roam”应由导航/NPC locomotion 执行，不该再由 `Day1` 自己手搓位移。

3. `Healing / Workbench / Farming`
   - `001/002` 只应出现在 `Primary`。
   - `Town resident` 不应因为剧情 workaround 被整体冻结。
   - 用户明确否定了“为了避免 bug 而冻结 Town resident”这条路线。

4. `dinner`
   - dinner 的 staged contract 与 opening 同口径。
   - 所有 NPC 包括 `001/002` 都一样：
     - 先到起点
     - 走向终点
     - 最多等待 5 秒
     - 超时 snap
     - 对话期间冻结
     - 对话结束后放人

5. `free time / night`
   - `19:30` 进入 free time
   - `20:00` resident 自主回 home
   - `21:00` 还没到位的允许 snap 到 home
   - `26:00` forced sleep
   - 用户认可这套口径，并倾向认为它应是“日常默认逻辑”，不应写成 Day1 私房逻辑

## 6. 为什么现在还没修完：按 3 个主根解释

### 6.1 主根一：`NPC 漫游走的都是坏点`

当前已做：

1. autonomous roam 候选不再完全 first-fit。
2. 候选评分会参考：
   - 开阔度
   - 与其他 agent 间距
   - 一部分 body clearance
   - 一部分 correction cost

为什么还不够：

1. 这只是“几个候选里挑相对好一点的”。
2. 还没有做到“任何目标点都必须先过统一安全合同”。
3. 当前仍会出现：
   - detour 点贴边
   - 重规划点贴边
   - 路径点压在静态碰撞边缘
   - 最终落点太靠近树/桶/墙/围栏

用户截图直接支持这一点。

### 6.2 主根二：`NPC 的静态导航计算还是垃圾`

当前已做：

1. next-step constraint 加了 body-clearance margin。
2. static obstacle steering 更偏侧绕。
3. avoidance 壳体余量稍微放大。

为什么还不够：

1. 当前很多地方仍是“路径先成立，再靠后处理纠偏”。
2. 也就是说，路径成立时对身体宽度和静态边界的尊重还不够硬。
3. 这就导致：
   - 树根边、桶边、墙角这种复杂边界会被系统先认为“可以试着过去”
   - 然后再在执行层不断被 steering / constraint / avoidance 拉扯

所以当前正确说法不是“没算碰撞体”，而是“算了，但算得还不够前置，也不够硬”。

### 6.3 主根三：`NPC 阻塞卡死了不会自救`

当前已做：

1. blocked/stuck 的止血
2. oscillation 的止血
3. retarget 的止血
4. decision reuse 的止血

为什么还不够：

1. 这些更像“减少炸锅”和“减少抖动”。
2. 还不是一套完整的脱困流程。

完整脱困至少要能分流：

1. 静态卡死
   - 短后退
   - 侧移采样
   - 临时拉黑当前坏边/坏点
   - 再重规划
2. 动态互卡
   - 短让路
   - 限时等待
   - 失败后重规划

当前系统仍没有把这套明确做完整，所以才会出现“原地磨、原地抽、反复修同一小段”。
