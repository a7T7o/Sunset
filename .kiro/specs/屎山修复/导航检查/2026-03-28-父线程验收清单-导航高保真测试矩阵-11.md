## 三、硬验收门

### Gate 0：口径完整性

必须全部满足：

- 明确区分 `Raw Real Input`
- 明确区分 `SuppressedNpcInteractions`
- 如用到 `Synthetic / Direct`，也明确区分
- 明确写出“主结论以 raw 为准”
- 没有把 suppressed 或 synthetic 冒充玩家体验结论

任一缺失：

- 直接判 `测试口径未站稳`

### Gate 1：玩家预期先行

必须能在报告前半段看到：

- 普通地面点导航什么叫“真的到点”
- 静止 NPC 什么叫“正常绕开而不是顶过去”
- moving NPC 什么叫“不是太早太僵硬”
- crowd 什么叫“短让行后回主线”

如果没有这层玩家预期，  
后面的数字一律降级为“技术记录”，不能判为高保真测试。

### Gate 2：锚点分层

必须显式区分：

- 点击点
- `Transform.position`
- `Rigidbody2D.position`
- `Collider.bounds.center`

并明确回答：

- 哪个才是普通地面点导航的应验收锚点
- 哪些现有 probe 还在用错锚点

如果它仍把 `Transform / Rigidbody` 的 reached 直接当“玩家到点”，  
本轮直接判 `契约未收口`。

### Gate 3：P0 矩阵完成度

报告中必须明确列出并完成：

1. 普通地面点导航 raw 精度组
2. 单个静止 NPC raw 组
3. 单个移动 NPC raw 组
4. crowd raw 组
5. NPC 自身两条
6. raw vs suppressed 对照组

允许它说明某个 case 当前补不进 runner，  
但它必须同时交：

- 为什么补不进
- 覆盖缺口是什么
- 这如何影响当前结论

如果没有这种说明，只是少跑，  
直接判 `矩阵不完整`。

### Gate 4：指标完整度

这轮至少要看到下面这些指标被明确输出或汇总：

#### 普通地面点导航

- 点击点到：
  - `Transform.position`
  - `Rigidbody2D.position`
  - `Collider.bounds.center`
  的 `deltaX / deltaY / distance`

#### 玩家避让动作

- `pathMoveFrames`
- `detourMoveFrames`
- `blockedInputFrames`
- `hardStopFrames`
- `actionChanges`

#### 避让触发

- `blockOnsetEdgeDistance`
- 或等价的“首次明显避让触发距离”

#### 结果与代价

- `minEdgeClearance`
- `maxPlayerLateralOffset`
- `npcPushDisplacement`

#### 入口污染

- `pendingAutoInteractionAfterClick`
- 对话/交互污染信号

如果它拿一堆 pass，但这些关键指标根本没出来，  
直接判 `数据不够验收`。

### Gate 5：三类坏相是否逐条回答

父线程下一轮必须明确能从报告里看出：

1. 普通地面点导航停偏现在还在不在
2. 静止 NPC 推土机现在还在不在
3. moving NPC 过早、僵硬绕行现在还在不在

要求不是“都修好了”，  
而是“每条都被明确回答，有证据，不偷换”。

### Gate 6：第一责任点是否收窄

它必须给出：

- 当前最窄第一责任点
- 为什么是它
- 为什么不是另外两类坏相先打

如果报告结尾还是：

- “问题很多，后面继续优化”
- “先继续多跑”
- “可能这里也有、那里也有”

这种级别，  
直接判 `责任点未压窄`。

## 四、快速否决项

下一轮只要出现下面任一条，直接不通过：

1. 只交聊天回执，不交测试报告文档
2. 把 `SuppressedNpcInteractions` 结果写成主结论
3. 继续把普通点导航的到达判定建立在错误锚点上，却不自知
4. 只强调 `pass=True`，不回答 3 类玩家坏相
5. 又回漂去 solver / 大架构 / cleanup 大谈
6. 在“测试矩阵”名义下顺手改了一堆 runtime 行为，却没明确说明
7. 一堆日志很多，但没有场景预期、没有分层、没有结论边界
8. 没有显式说明哪些旧 probe/pass 需要降级理解

## 五、可接受完成定义

### 结局 A：可接受

可以通过的最低条件：

1. `P0` 矩阵主体完成
2. raw / suppressed / synthetic 分层清楚
3. 普通点导航锚点偏差被量化
4. 三类坏相逐条回答
5. 第一责任点重新压窄
6. 没有明显 runtime 漂移或 unrelated 施工

这不等于“导航修好了”，  
只等于：

> 测试底座终于站稳，下一刀可以更准

### 结局 B：部分通过

只在以下情况可接受部分通过：

1. `P0` 基本完成，但有一个缺口被清楚定义
2. 缺口理由具体、边界清楚
3. 当前第一责任点已经明显比现在更窄

这类结果只能判：

- `checkpoint：测试底座接近成立`

不能判：

- `已完成`

### 结局 C：不通过

出现以下任一情况：

1. 还是只给局部漂亮数据
2. 还是混口径
3. 还是不回答玩家坏相
4. 还是没有把锚点契约收进测试口径

则直接不通过。
