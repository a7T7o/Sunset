# 2026-04-01-导航检查V2-NPC漫游异常中断与鬼畜止血-20

这轮不要继续把主刀留在 `PlayerAutoNavigator.cs` 的 sidestep 微调，也不要继续把问题表述成一般性的 `crowd stall`。

用户刚给出的最新真实现场，已经把新的第一痛点钉死了：

1. 静态点导航 / 右键到点偏上，这条线仍未解决，但这轮不是它的 runtime 主刀；
2. 更灾难的新坏相是：当 NPC 和玩家都要去同一个地方，或者多个 NPC 在近距区域同时移动时，会出现互相挤、互相卡、反复鬼畜、长时间不肯中断的情况；
3. 用户已经明确给出新的设计裁定：
   - 这说到底不是“再多调一点避让权重”；
   - 而是 NPC 的漫游导航缺少“异常即中断”的契约；
   - 当前漫游链一旦进入鬼畜 / 被打断 / 多方互相拖扯，不应该无限次重试、重建、复活；
   - 对 NPC 漫游来说，**一次明确异常就应该取消当前这次漫游导航**；
   - 并且这个“中断”必须留一个明确口子，后续 NPC 侧系统会接手写中断后的反应，例如被玩家打断、被 NPC 阻挡后的关系反馈或行为分支。

所以这轮唯一主刀改成：

- 只收 `NPCAutoRoamController.cs` 这条 **NPC 漫游异常中断与抛因口**；
- 目标不是“NPC 一定绕过去到终点”，而是“NPC 漫游一旦进入异常互卡，不再无限鬼畜，而是立即结束本次漫游段，并把原因向上抛出”。

## 二、这轮唯一主刀

只做这一件事：

把 `NPCAutoRoamController.cs` 的 **漫游导航异常中断契约** 补出来。

一句话翻译成玩家体验：

- NPC 漫游时如果和玩家 / 其他 NPC 互相卡成鬼畜，不会继续无限顶、无限抖、无限改道；
- 它会立刻放弃当前这次漫游段；
- 现场不再出现“两个家伙在一个点附近互相挤到天荒地老”的灾难级表演。

## 三、这轮必须建立的最小契约

你这轮至少要把下面 3 件事同时做出来：

### 1. 明确的中断原因

在 `NPCAutoRoamController.cs` 里补一个最小可用的“漫游导航中断原因”表达。

可以是：

- `enum`
- 结构化快照
- 或 `event Action<...>`

但必须至少能报清：

1. 这次是不是 roam move 的异常中断；
2. 中断原因是什么；
3. 是否由玩家 / NPC / 共享避让异常 / stuck 异常触发；
4. 当前目标点、当前位置、阻挡方 id 或 blocker 线索（如果当下拿得到）。

这不是为了做漂亮架构，而是为了给后续 NPC 逻辑留真正可接的口子。

### 2. 一次异常就取消当前漫游段

对 **普通漫游 move** 来说，不再允许“进入异常后无限次继续挣扎”。

只要满足“当前这次漫游段已经不再是正常可执行导航”的第一次明确信号，就应该：

1. 立刻取消本次 roam move；
2. 清掉当前 move 的路径 / detour / 继续追终点执念；
3. 停止当前运动；
4. 进入 roam 的中断后状态；
5. 触发上面的中断口。

注意：

- 这条规则是针对 **roam move**；
- 不是要求你把 `DebugMoveTo(...)`、剧情强制移动、probe 驱动的显式目标移动一起一刀切改掉；
- 这轮优先守住“漫游异常即中断”，不要顺手改坏现有 probe 驱动路径。

### 3. 不再无限重建同一段漫游目标

当前最可疑的问题就是：

- `TryHandleSharedAvoidance(...)`
- `TryReleaseSharedAvoidanceDetour(...)`
- `CheckAndHandleStuck(...)`

这几条链在 roam move 里把“已经异常”的现场继续当成“可恢复路径”，于是可以反复 detour / recover / rebuild。

你这轮必须把它改成：

- **roam move 一旦命中第一条异常信号，就不再继续同段恢复**

不要再让“重试次数没到 / 还能 rebuild / 还能 detour”顶替掉“这段 roam 已经该中断”的判断。
