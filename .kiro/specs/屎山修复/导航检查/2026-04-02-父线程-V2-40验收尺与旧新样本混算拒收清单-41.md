## 2. 下一次回执的固定审查顺序

### 第一关：先审新 blocker 是否真的 fresh 塌缩过

必须先看：

1. 它有没有先 clear console + request compile + 读最新 log
2. 它有没有只围绕：
   - `PlayerNpcChatSessionService`
   - `SetConversationBubbleFocus`
   去塌缩
3. 最后有没有明确落成：
   - `cleared`
   - `stale`
   - `active`

如果只写一句：

- “后来被新的 compile gate 截停了”

但没有 fresh exact lines，继续拒收。

### 第二关：再审它汇报的样本是不是全部来自“当前最终代码”

必须先看：

1. 它有没有明确写：
   - `这轮汇报的样本是否全部来自当前最终代码连续重跑`
2. 如果不是，它有没有把：
   - 历史 carry 样本
   - 当前最终代码样本
   分开列

只要它继续把：

- `fail -> fail -> pass`
- `fail -> pass -> pass`

直接写成“当前已稳定”，继续拒收。

### 第三关：再审当前最终代码矩阵是否真的跑满

必须至少看见：

1. `EndpointNpcOccupied raw ×3`
2. `Crowd raw ×3`
3. `Ground raw matrix ×1`
4. `SingleNpcNear raw ×2`
5. `MovingNpc raw ×1`
6. `NpcAvoidsPlayer ×1`
7. `NpcNpcCrossing ×1`

缺任何一项，都不能 claim `-40` 已完整收口。

### 第四关：最后才审它有没有再次偷换体验

继续固定追问：

1. `Ground raw matrix pass=True` 为什么仍不能直接代表用户可视停位已关闭？
2. 如果 `EndpointNpcOccupied` 或 `Crowd` 转绿，它现在凭什么说这已经等于真实右键体验过线？
