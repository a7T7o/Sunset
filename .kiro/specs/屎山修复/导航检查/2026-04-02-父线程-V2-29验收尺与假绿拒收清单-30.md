## 2. 目前已经钉死的拒收事实

基于当前代码现场，下面这些点已经被钉死，后续不允许再模糊化。

### 2.1 dedicated endpoint 当前的 green 定义不可接受

当前 dedicated endpoint case 的核心逻辑是：

- `endpointArrivalTolerance = combinedRadius + 0.35`
- `reachedByCenter = playerCenterDistance <= endpointArrivalTolerance`
- `reachedByBlockedShell = playerRigidbodyDistance <= endpointArrivalTolerance`
- `playerReached = !IsActive && (reachedByCenter || reachedByBlockedShell)`

这意味着：

- 只要玩家进入了一个很大的 endpoint shell
- 即使 `playerCenterDistance` 还在 `1.0+`
- 也可能被算成 `playerReached`

这和“普通点导航真实到点”不是一回事。

### 2.2 `endpointArrivalMode=center` 当前标签已失真

如果回执里再次出现类似：

- `playerCenterDistance=1.06`
- `endpointArrivalMode=center`

那这一条默认直接判为：

- **标签失真**

因为当前工程里其它点导航口径仍沿用：

- `PlayerPointArrivalCenterTolerance = 0.35`

所以 `1.06` 不可能被称为真正的 center 到点。

### 2.3 toolchain 仍未完整接回也不能放行

如果下一次回执里：

- runner 逻辑改了
- 但 `NavigationLiveValidationMenu.cs` 里仍没有：
  - dedicated endpoint 的 `PendingAction`
  - dedicated endpoint 的菜单项
  - dedicated endpoint 的 `ExecuteAction(...)`

那这条专案仍然只算：

- `runner-only checkpoint`

不能 claim toolchain 收口。

## 3. 下一次回执的强制验收顺序

后续收 `-29` 回执时，父线程必须固定按下面顺序审，不能跳步。

### 第一关：先审 green 定义是不是假的

先问 4 件事：

1. `pass=True` 现在的必要条件是什么？
2. `playerCenterDistance > 0.35` 时，还会不会被判成 pass？
3. `blockedShell / holdOutsideOccupiedEndpoint` 现在是不是仍被并进 green？
4. `endpointArrivalMode` 有没有被重命名、拆层或重判？

只要这 4 条有任一条仍含糊，这轮就继续拒收。

### 第二关：再审 outcome 分类有没有真落

后续必须看到 dedicated endpoint 至少拆成：

1. `caseValid`
2. `outcome`

其中 `outcome` 至少要能覆盖：

1. `ReachedClickPoint`
2. `StableHoldOutsideOccupiedEndpoint`
3. `InteractionHijack`
4. `Bulldoze`
5. `Oscillation`
6. `Linger`

如果它还是只有：

- `pass`
- `endpointArrivalMode`

那就说明这轮只是换了一层说法，没有真正把 dedicated endpoint 变成可裁证据。

### 第三关：再审 raw click 是否还是导航点击

后续 raw dedicated endpoint case 每条都必须带：

1. `playerClickIssued`
2. `pendingAutoInteractionAfterClick`
3. `pendingAutoInteractionTarget`

如果任一 raw case 出现：

- `pendingAutoInteractionAfterClick = true`

那默认先判：

- `InteractionHijack`

不允许把这种样本静默算进 dedicated endpoint 的成功样本里。

### 第四关：再审 toolchain 是否完整

除了 runner 本体，还要审：

- `NavigationLiveValidationMenu.cs`

只有当：

1. dedicated endpoint 的 `PendingAction` 存在
2. 菜单项存在
3. `ExecuteAction(...)` 分发存在

这条专案才算真正接回标准 live validation toolchain。

### 第五关：最后才审 fresh matrix

在前四关都过之后，才审 fresh matrix 是否够。

如果前四关没过，跑再多 fresh 也不能绿。

## 5. 仍然不能放行的两种假完成说法

后续只要 `V2` 用下面任一叙事，我都继续拒收。

### 假完成说法 A

- “玩家虽然没有到点击点，但稳定停在 occupied endpoint 外，这也可以先算 pass”

当前拒收原因：

- 这不是 point-arrival pass
- 这最多是：
  - `StableHoldOutsideOccupiedEndpoint`
  - 或 `semantic_pending`

除非产品明文改口，否则不能进 green。

### 假完成说法 B

- “这条专案先 green 掉，剩下只等 full matrix”

当前拒收原因：

- 如果 green 定义本身是假的，full matrix 只会把假绿扩散到更大范围

所以顺序不能反：

1. 先修 green 定义
2. 再补 fresh matrix
3. 最后才谈这条专案能不能算站住
