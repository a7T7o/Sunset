## 三、当前拒收的硬证据

### 1. runtime 语义偷换仍在代码里

当前 [PlayerAutoNavigator.cs](/D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Service/Player/PlayerAutoNavigator.cs) 的 `GetPlayerPosition()` 仍保留这条特判：

```csharp
if (targetTransform == null)
{
    if (playerRigidbody != null)
    {
        return playerRigidbody.position;
    }

    if (player != null)
    {
        return (Vector2)player.position;
    }
}
```

这意味着：

- 普通点导航当前仍优先按 `Rigidbody/Transform` 收口；
- 而不是统一按玩家实际占位中心收口。

父线程当前拒收的不是“这段代码一定永远不允许存在”，而是：

- 在 static fresh 仍明确 fail 的前提下，
- 不能把这条特判包装成“普通点契约已经修好”。

### 2. 场景几何本身存在 `Collider.center` 上偏事实

[Primary.unity](/D:/Unity/Unity_learning/Sunset/Assets/000_Scenes/Primary.unity) 里玩家 `BoxCollider2D` 当前硬证据是：

- `m_Offset: {x: -0.007229537, y: 1.2013028}`

这和 `Case2` 的：

- `centerDistance≈1.120`
- `rigidbodyDistance≈0.081`

高度一致。

所以用户体感里的“右键点地，人总是落在点的上方/下方，不是正对点”不是幻觉，而是当前场景几何语义与 runtime 收口语义错层的真实结果。

### 3. static runner 本身当前不是第一责任点

父线程当前不接受“问题都是 static runner 工具链在误报”。

因为父线程已经把工具链推进到了：

所以现在的拒收重点不再是：
