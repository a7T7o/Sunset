# 2026-04-02-父线程-V2-35验收尺与自清恢复前置拒收清单-36

## 1. 这份文档的目的

这是父线程下一次收 `-35` 回执时的固定验收尺。

它专门防两件事：

1. `V2` 继续把 compile gate 写成“等别人修”的默认停车位
2. `V2` 没做完自清恢复动作，就又回 PAN 或又停在 blocker checkpoint

---

## 2. 当前已经钉死的新增事实

父线程已经替 `V2` 查掉了这些常见源码侧借口：

1. `InventorySlotUI.cs / ToolbarSlotUI.cs` 当前工作树都真实有方法本体
2. 仓库里没有第二份同名类
3. 调用点与定义点标识符字节一致
4. 文件括号结构正常
5. 文件无 NUL 脏字节

因此下次再回来时，不接受它继续只说：

- `当前文件是好的，但 Unity 没吃进去`

而不做进一步自清恢复动作。

---

## 3. 下一次回执的固定审查顺序

### 第一关：先审它有没有做完整套“自清恢复”

必须先看：

1. clear console
2. request compile
3. 读最新 log
4. reimport / refresh
5. 再 compile
6. 再读 log
7. editor 级恢复动作
8. 最后一次 compile + log

缺任何一步，都不算过第一关。

### 第二关：再审 gate 分类是否被压实

必须明确落到三者之一：

1. `cleared`
2. `active-but-self-recoverable-failed`
3. `stale`

如果仍写泛泛的 `active blocker`，但没有恢复动作链和最新证据，继续拒收。

### 第三关：如果 gate 清掉，再审它有没有继续做 PAN

如果 gate 已 `cleared`，却没有继续：

1. `EndpointNpcOccupied raw ×3`
2. `Crowd raw ×3`
3. `Ground raw matrix ×1`

以及后续 PAN 改刀，继续拒收。

### 第四关：最后才审体验改善

只有前 3 关过后，才审：

1. endpoint
2. crowd
3. ground 可视停位

是否至少两条出现真实改善。

---

## 4. 下一次回执的最小通过条件

如果它想把 `-35` 判成“这轮站住”，最少必须同时满足：

1. 自清恢复动作完整做完
2. gate 分类已塌缩
3. 如果 gate 已 cleared，已继续做 PAN 大闭环
4. 如果 gate 仍 active，已给出足够证据证明它是无法自清的外部 live blocker

---

## 5. 继续拒收的三种假完成

### 假完成 A

- “我确认它是真 gate，所以这轮先停”

拒收条件：

- 没做完整套自清恢复动作

### 假完成 B

- “gate 清了，但这轮先只报 compile truth”

拒收条件：

- gate 已 cleared 却没继续 PAN

### 假完成 C

- “PAN 有结构绿，所以右键停位偏上可以先算关”

拒收条件：

- 没有 `Transform / Rigidbody / Foot / 可视真值`

---

## 6. 父线程下一次固定追问

1. 你这轮完整做了哪些自清恢复动作？
2. gate 最后到底是 `cleared / active-but-self-recoverable-failed / stale` 哪一种？
3. 如果 gate 清了，你从哪一步开始继续 PAN 大闭环？
4. 如果 gate 没清，你凭什么说它现在已经是无法自清的外部 blocker？
5. 你现在还能不能把“右键停位偏上”写成已关闭？

答不上来，就继续拒收。
