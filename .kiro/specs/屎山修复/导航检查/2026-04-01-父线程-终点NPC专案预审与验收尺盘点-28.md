## 2. 为什么 crowd 代理不能继续顶

`Crowd raw` 当前混了至少三层语义：

1. corridor 识别
2. 多 blocker 通过期的 detour / blocked-input
3. 终点前 lingering

这会带来两个治理问题：

1. 只要它继续 fail，`V2` 就很容易把责任过早压成“只剩一个 crowd slow-crawl 点”；
2. 用户真正抱怨的“终点附近有一个静止 NPC，玩家一直像想撞开它一样死避让”，会一直被 crowd 通道问题淹掉。

所以 dedicated case 的目的不是“多跑一个花样 case”，而是把下面这条体验单独拎出来：

- 主路径基本干净
- 终点附近有一个静止 NPC 占位
- 玩家右键后，坏相是否仍表现成：
  - 死避让
  - 反复 detour
  - lingering 不收口
  - 或继续推着 NPC 走

## 3. 建议的新 case 名称与边界

### 建议名称

- 代码名：
  - `RealInputPlayerGoalNpcOccupancy`
- 菜单名：
  - `Run Real Input Goal NPC Occupancy Validation`
  - `Run Raw Real Input Goal NPC Occupancy Validation`

### 边界

这条 case 只回答：

- 玩家右键到一个“终点附近被单个静止 NPC 占位”的点时，arrival blocker / finalize / lingering 到底怎么表现。

它**不回答**：

- 双 NPC corridor
- moving NPC 会车
- NPC roam
- static 点偏上
- 普通 ground contract

因此它应该是 standalone 单跑，不要直接塞进当前 `RunAll` 自动链。

原因很简单：

- 当前 `RunAll` 仍按
  - `AvoidsMovingNpc -> SingleNpcNear -> Crowd -> NpcAvoidsPlayer -> NpcNpcCrossing`
  这条老链自动推进；
- dedicated 终点占位 case 还在定义阶段，先单跑更不容易把旧矩阵搅混。

## 4. 最小场景布置建议

### 核心设计目标

要同时满足两件事：

1. 点击仍然是“玩家真实右键导航”
2. 但不要退化成“你其实点中了 NPC collider，于是先进入交互测试”

所以建议的最小布置不是把 NPC 直接摆在点击点正中心，而是：

- 点击点保持在一块干净地面上
- 静止 NPC 占位在点击点附近的 arrival shell 里
- 让这条 case 更像“终点被占了”，而不是“点击命中了 NPC”

### 建议布置

可先沿用当前 ground/single/crowd 共用的直线路径：

- `playerStart = (-9.15, 4.45)`
- `playerTarget = (-5.80, 4.45)`

单个静止 NPC 不放在 target 正中心，而是放在 target 邻近壳层内，例如：

- 首选尝试：
  - `npcAStart = (-5.45, 4.45)`
- 允许微调窗口：
  - `npcAStart.x` 落在 `[-5.52, -5.38]`
  - `npcAStart.y` 落在 `[4.38, 4.52]`

其它 NPC 继续远离测试区：

- `npcB = (-3.5, 8.0)`
- `npcC = (-1.5, 8.5)`

### 为什么这样摆

这样摆有三个好处：

1. 主路径大部分是干净的，不会退化成 `SingleNpcNear` 的“中段挡路”；
2. target 仍在 ground 上，raw 右键更容易保持为导航点击；
3. 目标点离 NPC 足够近，仍能触发终点 arrival blocker / finalize 邻域，而不是完全绕开。

### 原始点击有效性闸门

这条 case 的第一闸不是 pass/fail，而是 **case 是否有效**：

- `playerClickIssued == true`
- `pendingAutoInteractionAfterClick == false`

如果 raw case 一上来就出现：

- `pendingAutoInteractionAfterClick == true`

那它当前回答的是“交互命中劫持”，不是“终点占位 lingering”。

这时正确做法不是硬拿它当 dedicated 终点占位证据，而是：

1. 报成 `interaction-hijack`
2. 轻微平移 `npcAStart`
3. 直到 raw click 仍是导航点击，再开始看 arrival blocker

## 7. dedicated case 的分型口径

这条 case 当前最有价值的，不是立刻吹成 green，而是把失败类型拆清楚。

### A. `Fail-InteractionHijack`

判定：

- `playerClickIssued == false`
  或
- `pendingAutoInteractionAfterClick == true`

解释：

- 当前 raw 右键先被 NPC 交互吃掉，case 还没进入 arrival blocker 语义。

### B. `Fail-Bulldoze`

判定建议：

- `npcPushDisplacement > 0.08`

解释：

- 终点 occupancy 仍被玩家硬顶着走，属于最直接的玩家可感知失败。

### C. `Fail-Oscillation`

判定建议：

- `directionFlips > 1`
  或
- `actionChanges > 6`

解释：

- 终点附近仍在来回改向、犹豫、鬼畜，不是正常收口。

### D. `Fail-Linger`

判定建议：

- 到 timeout 仍未收口
  且
- `playerCenterDistance > 0.35`
  且
- `blockedInputFrames + detourMoveFrames` 明显偏高

解释：

- 玩家不是正常到点，也不是稳定停住，而是在终点前 lingering。

### E. `Pass-Reached`

判定建议：

- `playerReached == true`
- `playerCenterDistance <= 0.35`
- `npcPushDisplacement <= 0.05`
- `directionFlips <= 1`

解释：

- 玩家仍能以可接受方式到点，没有把 NPC 顶走，也没有明显鬼畜。

### F. `Pass-StableHold`（只作为可选增强口径）

如果产品接受“终点被占位时，玩家稳定停在占位前沿，不强行完成”，那这条需要一个小增强指标：

- `stableHoldDuration`

也就是：

- 玩家已经进入目标邻域
- 速度接近 0
- 不再来回 flip
- 不再推 NPC
- 持续稳定一小段时间

如果这条增强指标暂时不做，那么当前 dedicated case 就先不要把“稳定停住”直接吹成 green，只把它报成：

- `diagnostic_valid but semantics_pending`

这比假装已经确定产品答案更稳。

这条 case 的 tick 不需要发明新体系，建议只在 `SingleNpcNear` 基础上做最小分化：

1. player 与 npc 的位置采样，继续沿用：
   - foot position
   - center position
   - edge clearance
2. 继续复用：
   - `UpdateNpcPushMetrics(...)`
   - `ObservePlayerNavigationAction()`
3. 新 case 更应额外看：
   - `pendingAutoInteractionAfterClick`
   - `requested/resolved`
   - `lastAction`
4. 超时时一定要把 fail 分类写进 summary，而不是只给一个 `pass=False`

推荐 summary 里直接带：

- `classification=InteractionHijack/Bulldoze/Oscillation/Linger/Reached/StableHoldPending`

这样父线程和用户下一轮都不用再从长 details 里猜。

## 9. 最小验证矩阵建议

这条 case 一旦补出，建议最小跑法是：

1. `Ground raw ×1`
   - 只是 guardrail，确认没把 ground contract 打坏
2. `SingleNpcNear raw ×1`
   - 只是 guardrail，确认没把单 NPC 重新打坏
3. `GoalNpcOccupancy raw ×3`
   - 主证据
4. 如果 raw 出现 interaction hijack，再补：
   - `GoalNpcOccupancy suppressed ×1`
   用来判是交互噪声还是 arrival blocker 本体
5. `Crowd raw ×1`
   - 只看双 NPC / corridor 有没有被带坏

不建议第一次就把它并进 `RunAll`，避免一次把三种坏相搅成一锅。

## 11. 当前结论

这轮父线程能正式站住的判断只有一句：

- dedicated 的“终点有 NPC 停留” case 现在确实缺失；
- 它的最小正确落点是 `NavigationLiveValidationRunner + NavigationLiveValidationMenu`；
- `Crowd raw` 不能再长期代理它；
