## 2. 当前正式判断

当前导航线的事实必须同时承认两件事：

1. 旧系统补丁线已经完成了它该完成的使命：
   - 证明共享执行层方向不是错的
   - 证明受控 probe 可以从系统级失败拉回到可通过
   - 证明 `NavigationRoot + 共享执行底座 + 局部避让入口` 这套过渡资产有保留价值
2. 但这仍然不等于项目最终需要的专业导航底座已经建成：
   - 当前依然是“静态建路 + 启发式局部避让 + 行为层补丁包装”的混合方案
   - 当前 probe 通过，不等于真实玩法体验通过
   - 当前还没有真正完成“统一动态交通语义”的迁移

因此，从现在开始，这条线的正式主语固定为：

> 不再继续把旧系统 patch 到“勉强能用”，而是分阶段迁移到 `Sunset` 的专业导航底座。

## 3. 下一步唯一推荐动作

下一步不推荐直接继续改 solver 参数，也不推荐先大改 `GameInputManager`、`Primary.unity` 或未来怪物/宠物逻辑。

下一步唯一推荐动作是：

> 立即进入 `S0 基线冻结与施工蓝图阶段`，把“保留资产 / 必须废止的旧语义 / 第一批热区 / 第一批最小运行 slice / 双验收基线”全部钉死，然后再进入代码迁移。

这个动作的直接目的不是再写一篇大文档，而是避免后续重新掉回：

1. 一边改代码一边重新讨论方向
2. 旧 patch 逻辑和新底座逻辑长期并存
3. probe 全绿但真实玩法持续不自然
4. 到后面才发现热区误碰太多、回滚成本太高

## 4. 后续开发的 5 个主方向

后续所有实现都必须围绕下面 5 条主方向展开：

### 4.1 静态世界与动态交通彻底断层

- 静态路径层只回答“地图怎么走”
- 动态交通层只回答“活体之间现在该怎么过”
- 任何活体都不再同时扮演静态障碍和动态 blocker 两种同语义角色

### 4.2 交通裁决先于方向修正

- 先决定 `Proceed / Yield / Wait / SidePass / Follow / Chase / Recover`
- 再根据裁决输出运动意图
- 不再允许“先往目标冲，再靠权重把结果拽回来”

### 4.3 最终运动学语义统一

- 玩家、NPC、后续怪物、宠物可以有不同 brain
- 但最终都必须收口到统一的运动状态语义
- `停、让、绕、恢复` 不能再表现成完全不同的两套体感

### 4.4 功能线与体验线双验收

- `probe` 通过只是功能线
- 玩家现场右键导航、NPC 漫游中间态、多人会车体感属于体验线
- 后续任何“已完成”都必须明确是通过了哪一条线

### 4.5 未来移动体可直接接入

- 怪物追击
- 宠物跟随
- 牲畜活动
- 后续新增移动体

这些都不应再要求重新发明一整套导航闭环，而应只需要：

1. 新 brain
2. 新参数
3. 必要的优先级 profile

## 6. 分阶段详细执行口径

### S0 基线冻结与施工蓝图

阶段目标：  
把“从今天起到底按什么迁移”钉死，禁止再边开发边重新解释主线。

必须产物：

1. 本路线图文档
2. 现有资产继承清单
3. 旧语义废止清单
4. 第一批热区 / 非热区清单
5. 功能线 / 体验线双基线

关键任务：

1. 明确当前可继承资产：
   - `NavGrid2D`
   - `NavigationAgentRegistry`
   - `NavigationPathExecutor2D`
   - 当前 probe / tests / live runner
2. 明确必须废止的旧语义：
   - 动态活体直接充当静态网格阻挡
   - detour 直接写入主路径语义
   - 玩家/NPC 各自维持私有完整导航闭环
3. 定义第一批最小 slice：
   - 只先围绕玩家自动导航与 NPC 漫游
   - 只先覆盖 `Proceed / Yield / Wait / Recover`
4. 明确第一阶段不碰项：
   - 怪物追击
   - 宠物跟随
   - NPC 场景化落点 / Home Anchor 最终摆位
   - `GameInputManager.cs` 深改
   - 非必要 `Primary.unity` 热修改

验收：

1. 后续代码线程能明确回答“这一刀属于哪一层迁移”
2. 不再出现“补了行为 bug 但不知道是不是偏离主线”
3. 所有后续阶段都能直接从本阶段产物往下接

风险与注意：

1. `S0` 不是拖延代码，而是为后续所有代码避免返工
2. 如果 `S0` 做得模糊，后面每一刀都会重新回到“旧系统补丁”轨道

### S1 代理契约与动态上下文净化

阶段目标：  
让所有移动体都能被还原成统一动态快照，先解决“系统到底看到谁、如何看到”的问题。

必须产物：

1. 稳定的代理快照字段定义
2. 移动体类型与优先级 profile 口径
3. moving / sleeping / paused / following 等状态语义

关键任务：

1. 收紧 `INavigationUnit` 与 `NavigationAgentSnapshot`
2. 明确 `Player / NPC / Monster / Pet` 的快照映射方式
3. 把控制器私有状态翻译成统一公共状态，而不是让 solver 直接读控制器细节
4. 给动态上下文采样建立只读边界，避免采样器反向操纵控制器

验收：

1. 任意移动体都能在运行态被还原为同一类快照
2. 交通裁决层不需要知道“这个对象背后是哪个控制器”
3. 后续新增移动体不需要新增一套特殊判断分支

风险与注意：

1. 这一阶段最容易偷懒的错误是“继续往快照里塞控制器私货”
2. 快照要足够表达交通语义，但不能重新长成万能大对象

### S2 交通裁决层 MVP

阶段目标：  
把“谁该让谁、什么时候停、什么时候恢复”从补丁逻辑升级成正式的一层。

必须产物：

1. 交通状态机最小集合：
   - `Proceed`
   - `Yield`
   - `Wait`
   - `Recover`
2. 基础优先级矩阵
3. 状态惯性 / 最小时长 / 解锁条件

关键任务：

1. 先做玩家 vs NPC、NPC vs 玩家、NPC vs NPC 三类冲突裁决
2. 引入交通记忆，避免边界条件每帧横跳
3. 明确 `Recover` 完成判据，不允许阻挡一消失就瞬切猛冲
4. 把“是否该前冲”与“如何修正方向”彻底拆开

验收：

1. 玩家和 NPC 面对同类冲突时，得到的是同语义裁决
2. 运行态里能清楚解释角色现在为什么停、为什么让、为什么恢复
3. 不再出现靠调一个系数解释所有交通问题的情况

风险与注意：

1. 不要急着把 `SidePass / Follow / Chase` 一次性全做深
2. MVP 先把主交通词典站稳，再往上扩

### S3 微观运动求解层替换

阶段目标：  
让局部运动求解真正服从交通裁决，而不是继续做“壳层 steering 魔法”。

必须产物：

1. broad-phase / narrow-phase 双层空间模型
2. shape-aware footprint clearance 的最小可用实现
3. 由交通裁决驱动的 `FinalVelocity / FinalMotionIntent`

关键任务：

1. 保留轻量 broad-phase 粗筛
2. 用 `ClosestPoint`、脚底 footprint 或轻量 shape-aware 查询做 narrow-phase
3. 先把“明显大圆壳体感”压下去
4. 保证性能边界可控，不把运行态变成高频重几何运算

验收：

1. 玩家与 NPC 的 interaction shell 不再明显夸张
2. 真实近距错身不再像隔着隐形大球
3. 日志与现场体感都能对应到脚底真实占地，而不是抽象半径魔法

风险与注意：

1. 不要为了追求几何完美把性能炸掉
2. 也不要以性能为借口继续让“圆壳假象”统治体验

### S4 共享路径执行层二次净化

阶段目标：  
保留 `NavigationPathExecutor2D` 的价值，但把它从旧 detour 污染语义里真正洗干净。

必须产物：

1. 纯静态 `PathCorridor`
2. 独立生命周期的临时中间目标 / 动态绕行意图
3. 明确的 stuck / repath / arrival 契约

关键任务：

1. 把 detour 从主路径状态语义中彻底隔离
2. 保留对旧 probe 的回归兼容，但不再让 probe 绑架正式架构
3. 让 `PathExecutor` 只负责走廊推进和重建路径入口，不决定谁该让谁
4. 明确动态阻挡触发重规划与交通等待的分界线

验收：

1. detour 不再引发 clear/rebuild 风暴
2. 恢复原目标路径不再依赖包装层私货补洞
3. 玩家和 NPC 共享同一套执行骨架而不共享历史 bug

风险与注意：

1. 不要把旧 patch 逻辑一股脑塞进新 executor
2. executor 是通用底座，不是第二个交通裁决层

### S5 统一运动执行层

阶段目标：  
让玩家和 NPC 在最终运动学层面真正收口，而不是表面共用导航、体感仍然割裂。

必须产物：

1. 统一运动状态语义
2. 加速 / 减速 / 刹停 / 恢复推进过渡
3. 面向动画与表现层的统一运动输出

关键任务：

1. 明确 `PlayerMovement` 与 `NPCMotionController` 的统一契约
2. 决定最终是统一输出 `FinalVelocity`、`MotionIntent` 还是更高一层状态
3. 确保 `Yield / Wait / Recover` 的体感自然，不再像瞬时撞墙

验收：

1. 玩家与 NPC 的停让、恢复推进手感不再明显割裂
2. 不会再出现一个像推土机、一个像被脚本挪位
3. 动画与位移的节奏关系可解释、可调参

风险与注意：

1. 这一阶段不是机械统一 API，而是统一语义
2. 如果只统一表面接口、体感仍然不同，就不算过

### S6 玩家 / NPC 行为脑接入与旧闭环下线

阶段目标：  
让玩家和 NPC 只保留 brain 职责，退出私有导航黑盒。

必须产物：

1. 玩家 brain 的导航意图输出口径
2. NPC 漫游 brain 的导航意图输出口径
3. 旧私有 stuck / repath / detour 闭环下线清单

关键任务：

1. `PlayerAutoNavigator` 退化为玩家导航 brain / 外观桥接层
2. `NPCAutoRoamController` 退化为 NPC 漫游 brain
3. 旧的私有导航主循环、私有 detour 逻辑逐项下线

验收：

1. 玩家 brain 只回答“我要去哪里、以什么模式去”
2. NPC brain 只回答“我要去哪、为什么动/停”
3. 旧逻辑移除后，功能线和体验线都不回退

风险与注意：

1. 不要过早去碰 `GameInputManager.cs`
2. 不要把“行为脑退化”做成“能力丢失”

### S7 其他移动体接入

阶段目标：  
把怪物、宠物、牲畜等纳入同一底座，验证架构不是只对玩家/NPC 有效。

必须产物：

1. 新移动体 profile 矩阵
2. `Chase / Follow` 等扩展交通语义
3. 最小接线样例

关键任务：

1. 怪物追击先接 `Chase`
2. 宠物跟随先接 `Follow`
3. 牲畜活动接柔性让行与低优先级 profile

验收：

1. 新移动体接入时不需要再造一套导航系统
2. 只通过参数和 brain 差异就能成立基本行为

风险与注意：

1. 这阶段必须在玩家/NPC 底座稳定后再开
2. 不允许用“先硬写一套怪物特判”回避底座问题

### S8 双验收与封板

阶段目标：  
把“功能过了”和“体验过了”都正式封板，结束这条迁移线。

必须产物：

1. 功能线验收报告
2. 体验线验收报告
3. 回归线文档与复测步骤
4. 旧语义下线清单

功能线必须覆盖：

1. 玩家绕移动 NPC
2. NPC 绕玩家
3. NPC-NPC 会车
4. 玩家穿混合静态障碍 + 漫游 NPC
5. 怪物追击基础样例
6. 宠物跟随基础样例

体验线必须回答：

1. 是否还有明显推土机感
2. 是否还有明显大圆壳体感
3. 是否还有停一下再猛冲的违和恢复
4. 是否已经像活体交通而不是数学避障

封板条件：

1. 功能线通过
2. 体验线通过
3. 回归线可重复
4. 旧语义明确下线

## 8. 后续开发中的硬注意事项

1. 不要再把“继续调参数”当主线。
2. 不要再把动态活体继续写进静态网格阻挡语义。
3. 不要再让 detour 继续担任主路径状态的一部分。

## 9. 什么时候才算真正完成

只有同时满足下面 4 条，才允许说“Sunset 专业导航底座迁移完成”：

1. `006` 中定义的层级职责已经在代码和运行态里成立
2. 玩家 / NPC / 未来移动体共享同一底座，而不是共享几段脚本
3. 功能线通过，且体验线也通过
4. 旧补丁语义已经明确下线，不再作为运行时暗桩继续存在

如果只满足下面任意一种，都还不能算完成：

1. probe 全绿，但真实玩法体感仍差
