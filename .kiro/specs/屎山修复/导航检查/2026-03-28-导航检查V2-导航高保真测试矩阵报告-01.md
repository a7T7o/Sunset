# 2026-03-28 导航检查V2 高保真测试矩阵报告 - 01

## 0. 本轮目的

这轮不宣称“导航体验已经修好”，只回答三件事：

1. 现在真实玩家坏相到底还剩哪些，而且有多稳定。
2. 现有 `probe/pass` 到底哪些还能信，哪些必须降级。
3. 下一刀最窄第一责任点应该先锁哪里。

---

## 1. 玩家预期

### 1.1 普通地面点导航

- 玩家点普通地面点时，真正应验收的是“玩家实际占位中心”到点。
- 玩家不应该出现“点击点与真实占位中心对齐了，但因为 `Transform/Rigidbody` 在脚下位置，所以被误判成没到点”。
- 普通地面点导航不应再混入 `ClosestPoint + stopRadius` 的交互跟随语义。

### 1.2 单个静止 NPC

- 玩家点击静止 NPC 后方时，应该识别 blocker，做出真实绕行或明确停让。
- 不应该出现“玩家把静止 NPC 顶着推过去，自己还没真正到点”的推土机式结果。

### 1.3 单个移动 NPC

- 玩家应在合适距离开始规避，而不是大老远就僵硬偏航。
- 规避后应回到主线并到点，而不是超大侧偏、长 timeout、或最后仍没到点。

### 1.4 crowd

- 玩家应表现为“短暂让行后回主线”。
- 不应该出现“始终不真正绕行，只是在主路径上越来越慢，最后拖着走到 timeout 还没到点”。

### 1.5 NPC 自身两条

- `NpcAvoidsPlayer` 与 `NpcNpcCrossing` 这轮只作为护栏。
- 目的不是扩回 NPC 主线，而是确认这轮测试/验证层补口没有把 NPC 既有基线带坏。

## 2. 入口分层与锚点分层

### 2.1 入口分层

- `Raw Real Input`
  - 真右键语义
  - 本轮主结论来源
- `SuppressedNpcInteractions`
  - 只作对照组
  - 用来切开“点击入口污染”和“导航本体问题”
- `Synthetic / Direct`
  - 本轮不拿来做体验结论
  - 只允许作为低层旁证

### 2.2 锚点分层

- 点击点 / 目标点
- `Transform.position`
- `Rigidbody2D.position`
- `Collider.bounds.center`
- 历史脚底锚点（`GetActorFootPosition(...) => rigidbody.position`）

### 2.3 本轮锚点裁定

- 普通地面点导航的应验收锚点：`Collider.bounds.center`
- 历史运行/验证锚点：`Transform.position / Rigidbody2D.position / 脚底`
- 仍在用错锚点的现状：
  - `NavigationLiveValidationRunner` 里历史 `GetActorFootPosition(...)` 与部分 `playerReached` 口径
  - 所以旧的 `pass=True` 不能直接翻译成“玩家真实占位中心已到点”

## 3. 测试环境与测试层补口

### 3.1 本轮真实数据源

- 主数据源：`[NavValidation]`
- 辅助旁证：`[NavStaticValidation]`
- 主结论不认 `NavStaticValidation` 的 `pass/fail`，它只作为噪声/旁证记录

### 3.2 本轮必要测试层补口

- 已补：
  - `NavigationStaticPointValidationRunner.cs`
- 原因：
  - static point runner 会在 live navigation 已有 pending action 时，把 live pending action 吞掉，导致 crowd 矩阵被测试层互踩打断。
- 补口内容：
  - static runner 在检测到 `Sunset.NavigationLiveValidation.PendingAction` 时退让，不再清空 live pending action。
- 影响边界：
  - 只影响测试启动顺序
  - 不影响导航 runtime 语义

### 3.3 本轮 live 噪声

- 出现过 `NavStaticValidation` 自动起跑与 `no_valid_ground_click_cases`
- 出现过 `DialogueDebugMenu.StepSpringDay1Validation` 的外部菜单噪声
- 因此：
  - player raw 矩阵主结论以 `Editor.log` 中最终 `scenario_end / scenario details` 为准
  - NPC 两条改为“先进入 Play，再在 Play 中直接触发 probe”的方式补齐

## 4.1 P0-1 普通地面点导航精度

### raw real input ground matrix

- 完成情况：`6/6`
- 总结：
  - `scenario_end=RealInputPlayerGroundPointMatrix pass=True clickMode=RawRightClick reachedCases=6/6 accurateCenterCases=6/6 positiveCenterBiasCases=0/6 maxColliderDistance=0.192`
- 稳定事实：
  - `Transform/Rigidbody` 相对点击点稳定下偏约 `1.098 ~ 1.239`
  - `Collider.bounds.center` 相对点击点误差约 `0.126 ~ 0.192`
  - `pathRequest` 与 `resolvedDestination` 均直接等于点击点
  - `navigationOffset` 稳定约 `(-0.01, +1.20)`

### suppressed ground matrix

- 完成情况：`6/6`
- 总结：
  - `scenario_end=RealInputPlayerGroundPointMatrix pass=True clickMode=SuppressedNpcInteractions reachedCases=6/6 accurateCenterCases=6/6 positiveCenterBiasCases=0/6 maxColliderDistance=0.192`
- 与 raw 对照结论：
  - 数值几乎一致
  - 普通地面点导航的偏差不是点击入口污染，而是锚点/验收契约问题

### ground matrix 结论

- 普通地面点导航“点 A 停在 A 下方约 1.2”的视觉感受是真实存在的
- 但如果按本轮已接受的“玩家实际占位中心”语义验收，`Collider.bounds.center` 对点是稳定成立的
- 所以普通点导航当前更像：
  - 结构性锚点契约错层
  - 而不是第一优先级 runtime 失效

## 5. 对十个必答问题的正面回答

### 5.1 普通地面点导航现在到底是不是停偏

- 是，`Transform/Rigidbody` 相对点击点稳定下偏约 `1.10 ~ 1.24`。

### 5.2 如果停偏，偏差是偶发还是稳定方向性偏差

- 是稳定偏差，不是偶发。
- 但它体现在历史脚底/刚体锚点，不体现在 `Collider.bounds.center`。

### 5.3 哪个锚点才是普通点导航的应验收锚点

- `Collider.bounds.center`

### 5.4 现有哪些 probe 还在用错误锚点

- `NavigationLiveValidationRunner` 的历史 `GetActorFootPosition(...)`
- 以及仍按 `Transform/Rigidbody/脚底` 判 `playerReached` 的历史口径

### 5.5 静止 NPC 现在到底还会不会被推土机式顶过去

- 会，而且是稳定会。
- raw ×3 全部表现为：
  - `npcPushDisplacement≈2.29`
  - `detourMoveFrames=0`
  - `playerReached=False`

### 5.6 moving NPC 现在到底是不是过早、过僵硬地绕

- 是，而且当前已经不只是“过早、过僵硬”，而是 raw ×3 全部失败。
- onset 偏早、侧偏过大、最后仍没到点。

### 5.7 这 3 类坏相里，哪一类是当前第一责任点

- 第一责任点：`单个静止 NPC / passive NPC blocker` 识别与处理链。

### 5.8 这个第一责任点为什么比另外两类更值得先打

- 因为它最稳定、最窄、最不需要额外解释：
  - raw ×3 和 suppressed 对照都一致失败
  - detour 完全不产生
  - 推土机结果极其稳定
- 相比之下：
  - 普通地面点主要是锚点契约问题
  - moving/crowd 虽然也坏，但形态更宽，可能叠加了更大的回主线/动态规避问题

### 5.9 哪些现有 pass 结果可以继续信

- `ground matrix` 的 `Collider.bounds.center` 对点结论可以继续信
- `NpcAvoidsPlayer ×2`
- `NpcNpcCrossing ×2`

### 5.10 哪些现有 pass 结果必须降级理解

- 任何基于 `Transform/Rigidbody/脚底 reached` 的普通点 `pass`
- 任何旧的 `SingleNpcNear / MovingNpc / Crowd` 局部 pass，只要没有与本轮 fresh raw 矩阵对齐，都必须降级
- `SuppressedNpcInteractions` 只能当对照组，不能顶替 raw 玩家体验结论

## 6. 当前最窄第一责任点

### 6.1 裁定

当前最窄第一责任点不是 solver 大调，不是 `TrafficArbiter / MotionCommand`，也不是 scene/UI。

当前最窄第一责任点是：

> `PlayerAutoNavigator.cs` 中 passive/static NPC blocker 命中后，玩家仍持续停留在 `PathMove` 主路径、没有进入有效 detour / 停让 / blocker 升级，最终把 NPC 顶走并且自己还没到点的那条处理链

### 6.2 证据

- `SingleNpcNear raw ×3`
  - `detourMoveFrames=0`
  - `hardStopFrames=0`
  - `actionChanges=1`
  - `npcPushDisplacement≈2.29`
  - `playerReached=False`
- `SingleNpcNear suppressed ×1`
  - 结果几乎相同
- `Crowd raw ×3`
  - 同样基本不进入 detour
  - 长时间 `PathMove` 蹭行后未到点
- 这说明现在最先该打的不是“detour 怎么更漂亮”，而是：
  - 为什么 passive blocker 命中了，主链却没有进入有效的 blocker 响应

### 6.3 为什么不是普通地面点契约先打

- 因为按当前已接受的“玩家实际占位中心”语义，ground matrix 已经证明 `Collider.bounds.center` 是稳定对点的。
- 普通点现在更像“验收锚点与视觉/脚底锚点错层”，不是本轮最先该进 runtime 修复的一刀。

### 6.4 为什么不是 moving/crowd 先打

- moving/crowd 也确实失败，但它们的形态更宽：
  - onset 过早
  - 回主线失败
  - 大侧偏
  - crowd stall
- static single 更窄：
  - 单一 blocker
  - 稳定推土机
  - raw/suppressed 同形
  - 因果链更容易先钉死

## 7. 哪些 probe / pass 口径需要降级

### 7.1 必须降级

- 以 `Transform.position / Rigidbody.position / 脚底 reached` 作为普通点到点证明的旧口径
- 任何没有 raw fresh 支撑的历史 `SingleNpcNear / MovingNpc / Crowd pass`
- 把 `SuppressedNpcInteractions` 结果直接翻译成“真实玩家体验已过”的口径

### 7.2 可以保留，但要加注释

- `GroundPointMatrix pass=True`
  - 只能表示：
    - `Collider.bounds.center` 对点成立
  - 不能表示：
    - `Transform/Rigidbody` 已无偏差
- `NPC 自身两条 pass`
  - 可以作为护栏
  - 不能反推玩家侧本体已经没问题

## 8. 本轮结论

### 8.1 现在仍然存在的玩家可感知问题

1. 普通地面点导航：
   - 玩家视觉上仍会看到脚底/刚体停在点击点下方约 `1.2`
   - 但按已接受的实际占位中心语义，中心对点是成立的
2. 单个静止 NPC：
   - 稳定推土机式顶过去
3. 单个移动 NPC：
   - onset 偏早、侧偏过大、最后常常仍没到点
4. crowd：
   - 不是短让行回主线，而是长时间蹭行后 still fail

### 8.2 当前阶段判断

- 这轮已经完成“高保真测试矩阵接管”
- 当前不再允许拿局部 `pass=True` 代替体验结论
- 下一轮实现施工应该从：
  - `PlayerAutoNavigator.cs`
  - passive/static NPC blocker 处理链
  重新开刀

### 8.3 一句话总结

- ground 这边主要是契约错层；
- 真正最该先打的 runtime 第一责任点，是 `PlayerAutoNavigator` 对 passive/static NPC blocker 的失效响应链。
