# `000-gemini锐评-1.0.md` 边界补记

## 文件定位

本文件不是新的上位设计，也不是导航实现施工单。  
它只服务一个目的：

> 把 `000-gemini锐评-1.0.md` 当前为何判 `Path B`、哪些点可以吸收、哪些点不能直接变成施工指令，正式钉死。

## 当前冻结结论

### 1. 可以吸收的部分

以下内容可以作为 `导航V2` 的认知提醒继续保留：

1. 当前导航失败不只是 solver 参数问题，而是控制流与职责切分的问题。
2. `PlayerAutoNavigator` / `NPCAutoRoamController` 仍然职责过重，这个诊断与现有代码事实相符。
3. 交通语义应先于方向修正，状态惯性与交通记忆是真实需要。
4. shape-aware / `ClosestPoint` 方向是对的，但它在 Sunset 当前语境里属于 `S3` 的求解层深化，而不是一句话就能取代所有现有逻辑。

### 2. 不能直接转成施工硬令的部分

以下内容不能因为 `Path B` 就被直接带入实现施工：

1. 不能把“必须新建唯一 `DynamicNavigationAgent`”当成当前唯一合法形态。
2. 不能把“绝对禁止 detour / 临时中间目标”写成硬红线，因为 `006` 允许生命周期独立的临时中间目标存在。
3. 不能把“统一运动学语义”偷换成“必须统一到 `linearVelocity`”。
4. 不能把“controller 过重”直接偷换成“当前就应该立刻把 controller 清空并删除所有过渡资产”。

### 3. 正确使用方式

`000-gemini锐评-1.0.md` 在 Sunset 当前语境里的正确使用方式只有一种：

1. 它可以继续作为问题意识输入，用来提醒 `导航V2` 不要再掉回“大架构讲得很对，于是直接越级开刀”的老路。
2. 它不能升格为高于 `006/007` 的上位法。
3. 它不能直接生成导航实现施工指令。
4. 后续如果 `导航V2` 进入实现判断，仍必须回到：
   - `006-Sunset专业导航系统需求与架构设计.md`
   - `007-Sunset专业导航底座后续开发路线图.md`
   - 当轮 live / 治理委托
   - 当前代码热区事实
