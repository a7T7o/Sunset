## 1. 文档定位

这不是一份新的锐评。  
这也不是 `006/007` 的替代品。  
它是一份给 `导航V2` 使用的**开发宪法**，作用只有一个：

> 把 `006/007` 的上位设计、当前 live 代码现实、已接受的阶段判断，以及从失败里收缩出来的硬边界，收成一套后续每一刀都必须服从的执行口径。

它要解决的不是“最终理想架构是什么”，而是下面这件事：

> 在已经有 `006/007`、已经有多轮 prompt、已经有多轮 runtime 失败与纠偏的前提下，`导航V2` 之后到底该按什么宪法继续开发，才不会再次退回“聊天里看起来很懂、代码里继续漂”的循环。

### 1.1 与其他文档的关系

三层关系必须固定：

1. `006-Sunset专业导航系统需求与架构设计.md`
   - 回答“最终目标架构是什么”
2. `007-Sunset专业导航底座后续开发路线图.md`
   - 回答“阶段顺序是什么、S0-S8 怎么走”
3. **本文**
   - 回答“从现在开始，V2 开发必须遵守什么、禁止什么、当前先做什么、什么条件下才能进入下一阶段”

也就是说：

- `006` 是目标蓝图
- `007` 是迁移路线
- **本文是 V2 的现行开发宪法**

### 1.2 优先级

后续如果出现冲突，默认按这个顺序判断：

1. 用户当前明确指令
2. `006 + 007`
3. **本文**
4. 已接受的 live 事实 / 已接受 checkpoint
5. 当轮 prompt / 回执
6. Gemini / 外部锐评材料

禁止再发生这种事：

- 把一份新锐评说得更锋利，就抬到 `006/007` 上面
- 把一轮 prompt 写得更具体，就默默覆盖本文
- 把一条 probe 绿灯，当成架构已经闭环

### 1.3 本文不是代码级防火墙

本文负责收口开发口径，不负责替代编译期硬约束。

当前仓库里已经存在一部分代码级隔离资产，例如：

- `INavigationUnit`
- `NavigationAgentRegistry`
- `NavigationPathExecutor2D`

这说明 Sunset 现在不是“完全没有代码级隔离”，但这也不等于物理隔离墙已经足够强。

因此从现在起，`导航V2` 必须同时承认两件事：

1. 不能把“已有 `002`”偷换成“只靠 markdown 宪法就够了”
2. 也不能把 `ITrafficArbiter`、`ILocomotionReceiver` 这类未来接口名，偷换成当前所有 `P0` runtime 切片的统一前置门槛

正确做法应是：

- 文档宪法继续负责钉边界
- 进入结构性热区时，同刀补齐所需的接口 / 契约 / 状态资产
- 让代码级硬约束和本文并行生效，而不是二选一

### 1.4 与 `004 / 005` 的关系

从 `v1.1` 之后，`导航V2` 的现行资产不再只有本文一份。

它们之间的关系固定为：

1. `002`
   - 负责开发宪法
   - 回答“当前开发必须遵守什么”
2. `004-导航V2接班准入与自治规约.md`
   - 负责自治边界
   - 回答“`导航V2` 自己是谁、能决定什么、何时必须上报”
3. `005-导航V2偏差账本.md`
   - 负责实时记账
   - 回答“当前真实状态、偏差、收敛和冻结项是什么”

也就是说：

- 本文不再独自承担全部 V2 任务
- 本文负责宪法
- `004` 负责自治
- `005` 负责状态

## 2. 当前统一判断

`导航V2` 当前面对的，不是“缺设计”，而是“缺一份把设计和现实真正接起来的开发宪法”。

### 2.1 现在已经成立的事实

以下事实必须同时承认：

1. `006/007` 仍然有效，没有被推翻。
2. 当前代码里已经不是“什么都没有”：
   - 共享代理契约已存在
   - 共享注册表已存在
   - `NavigationPathExecutor2D` 已经存在
   - detour create / clear / recover 资产已经存在
   - detour clear hysteresis / recovery cooldown 的代码与测试基线已经存在
3. 但这依然不等于专业导航底座已成立：
   - 玩家 / NPC 的私有闭环没有完全下线
   - 控制流仍会被 controller 私有分支吞掉
   - 玩家侧拿到过有效窗口，不等于 NPC 侧也成立
   - probe / 单场 green，不等于真实体验闭环

### 2.2 当前真正的主问题

当前最真实的问题，不再是“solver 参数还不够聪明”，而是：

> `ShouldRepath / Detour` 这类共享层意图已经出现，但在 `PlayerAutoNavigator / NPCAutoRoamController / NavigationPathExecutor2D` 的执行层握手里，仍然会被旧分支吞掉、打断或提前释放。

因此，V2 当前的核心语义必须固定为：

> 继续把共享交通语义从“有资产”推进到“稳定统治执行层”。

### 2.3 当前为什么还需要这份文档

因为到现在为止，真正指导开发的内容已经散落在四种地方：

1. `006/007`
2. `导航检查` 的多轮治理 prompt
3. `导航V2` 的自省与锐评审核结论
4. 用户、父线程、子线程之间的大量聊天共识

如果不把它们重新压成一份正式约束，后面就会继续出现：

1. 上位设计很大，但当前刀口没人能复述清楚
2. 结构进展和体验进展混说
3. prompt 比文档更像真正的规范
4. 每一轮都像在“重新发明本轮应该信什么”

## 3. 导航V2 的 10 条宪法

以下 10 条，从现在起视为 `导航V2` 的硬约束。

### 宪法 1：用户现场体验高于 probe 成绩

任何“通过”都必须先区分：

1. 受控功能线通过
2. 真实体验线通过

禁止再把：

- `runner` 绿灯
- `minClearance` 达标
- 某一条 case 过了

包装成“导航已经自然了”。

### 宪法 2：结构线与体验线必须分开报

每一轮必须明确区分：

1. 结构线证据：
   - 哪个 owner / lifecycle / shared layer 真迁进去了
2. 体验线证据：
   - 用户现场到底好没好
   - 哪个 live 过了 / 没过

禁止再把：

- “detour owner 迁进共享层了”

偷换成：

- “所以玩家已经不推 NPC 了”

### 宪法 3：交通裁决先于方向修正

`导航V2` 以后所有实现都必须服从一个顺序：

1. 先判断当前是不是该前进、该等待、该让行、该恢复
2. 再求运动方向和速度

禁止继续把系统写成：

1. 先朝目标猛冲
2. 再靠一堆 steering / lerp / scale 去补救

### 宪法 4：Detour 是独立生命周期，不是主路径的一部分

Detour 可以存在，但它只能是：

- 短期空间意图
- 临时 override
- 独立 owner 生命周期

禁止再次回到：

- 把 detour 节点塞进主路径
- 用 detour 反向改写 corridor 语义
- 让 detour clear / rebuild / recover 风暴重新长出来

### 宪法 5：共享层一旦拥有 owner，controller 不能再静默一票否决

这是当前最关键的一条。

一旦共享层已经明确产生：

- detour owner
- release / recover 状态
- 交通状态

那么 controller 的私有分支就不能再靠：

- cooldown
- rebuild
- stop
- early-return
- 局部 fallback

静默吞掉它。

### 宪法 6：第一责任点未钉死前，不准乱补

当前 `导航V2` 的一大教训就是：

- 很多 runtime fail 看上去像同一个问题
- 实际上同一文件里可能有 2 到 4 段相邻分支都在吞状态

因此从现在起：

1. 先压窄第一责任点
2. 再打最小补口
3. 再做最小 fresh

禁止再回到：

- 一边诊断一边顺手改三段
- 改完之后也说不清到底是哪一刀生效

### 宪法 7：cleanup 不是主线，只能是同刀收尾

除非命中 shared-root mixed hot 面卡死，否则：

- own dirty / whitelist sync / owner 报实

都只能作为当前这一刀的收尾，不得升格成替代主线。

尤其禁止：

- runtime 主问题没钉死，先去清 `Primary.unity`
- live 还没过，先去清字体
- 子线程不卡在自己的问题上，先被治理拉去做大扫除

### 宪法 8：mixed hot 面必须显式报实，不能靠感觉吞并

对以下对象，必须显式报实 owner：

1. `Primary.unity`
2. `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`
3. `ProjectSettings/TagManager.asset`
4. shared root 上的 singleton / hot scene / hot asset

禁止：

- 因为“像是我碰过”就整包认领
- 因为“现在刚好 dirty”就顺手 cleanup

### 宪法 9：Gemini 只提供问题意识，不提供现成施工权

`000-gemini锐评-1.0.md` 的 `Path B` 和 `1.1` 的 `Path C` 已经说明得足够清楚：

- 可以吸收：
  - 控制流崩坏
  - Detour 独立化
  - 状态惯性
  - shape-aware
- 不可直接照做：
  - 立即重开 `TrafficArbiter / MotionCommand`
  - 立即硬切 `DynamicNavigationAgent`
  - 立即全量 `linearVelocity`

### 宪法 10：V2 每一刀都必须能回答“这刀让哪个阶段前进了”

以后任何一刀，都必须明确回答：

1. 这刀属于 `S0-S8` 的哪个阶段
2. 这刀推进的是：
   - 架构净化
   - 执行层统治
   - 运动学统一
   - 体验验证
3. 这刀没推进什么

如果一刀连自己属于哪一阶段都说不清，就说明它又开始漂了。

## 5. V2 之后的阶段推进顺序

`导航V2` 后续不能自由发挥，必须按下面顺序推进。

### P0：先把当前用户可见阻塞打穿

这不是 `007` 新阶段，而是当前现实的**阶段前门**。

必须先把这两条压住：

1. `NpcAvoidsPlayer` 的 NPC 侧 runtime 继续压窄并最终过线
2. `HomeAnchor` 的运行中补口链接上

原因很简单：

- 如果用户眼前这两个最直观的问题都还没过去
- 再讨论下一阶段，都会继续失真

补充边界：

- `P0` 当前指的是**前门阶段**，也就是先把用户眼前最直观的阻塞压离混战态
- `P0` 不是“下一代结构退壳已经完成”的退出判定
- 从 `P0` 第一刀开始，就必须同步维护最小状态账本 / 偏差账本

这里说的“最小状态账本”不是再开一份泛大设计，而是要求实时记录：

1. 本轮切的是哪条 hot path
2. 哪个共享 owner / lifecycle 已接管
3. 哪些旧闭环仍残留
4. 本轮明确不碰什么

也就是说，账本必须从 `P0` 同步启动，不能等 `P0` 整体稳住后再做事后尸检。

### P1：补当前中间态架构增补

这是当前中间态架构增补层，不再等价于“第一次开始记账”的时机。

在 `P0` 至少有一条最小 slice 稳定下来之后，必须把前面实时维护的最小账本升级为正式中间态资产，明确回答：

1. 当前代码现实和 `006/007` 的差距
2. 哪些共享 owner 已经真正接管
3. 哪些旧闭环仍然活着
4. 下一阶段真正能动的点是什么

### P2：重新回到 `S2 -> S3 -> S4`

只有当：

1. 当前用户可见阻塞不再处于混战态
2. 中间态架构增补已经补上

才允许重新回到：

- `S2` 交通裁决层形式化
- `S3` shape-aware 微观求解层
- `S4` 共享执行层净化

补充边界：

- 从 `P2` 开始，或任何一轮准备宣称“某个结构阶段可以退出”时，不能只看“用户可见阻塞稳定”
- 还必须补充当前阶段 hot path 的客观依赖切断指标，例如：
  - controller 是否仍在吞共享 owner / release / recover
  - 玩家 / NPC brain 是否仍直接承担私有 detour / repath 主循环
  - 关键热点脚本对旧闭环资产的直接依赖是否已切断
- 这些指标应按当前阶段的 owned hot path 制定，不能偷换成“全仓某个 API 调用数归零”这种失真的总量口号

### P3：再谈 `S5 / S6`

在 `S2-S4` 没有站稳前，不允许再把：

- 统一运动学收口
- brain 退壳
- 旧私有闭环下线

重新写成“马上下一轮就该大动刀”的东西。

## 6. V2 的阶段完成定义

以后 `导航V2` 的“完成”不能再只靠一句感觉。

至少要满足这四层：

### 6.1 文档层

必须有：

1. 当前上位依据
2. 当前中间态判断
3. 当前禁止漂移边界

### 6.2 代码层

必须能明确指出：

1. 这一刀改了哪一段 owner / lifecycle / branch
2. 这一刀不再改什么

### 6.3 验证层

至少有一条真实证据：

1. compile / test
2. targeted live / real input

补充边界：

- 当前仓库已经有 detour clear hysteresis / recovery cooldown 的代码与 EditMode 测试基线
- 但未来交通裁决状态机（例如 `Yield / Wait / Recover` 记忆）的正式测试集仍未建立
- 因此后续不允许再把现状写成“完全没有 Hysteresis 基准”，也不允许把现有 detour 测试偷换成“交通状态机测试已经齐备”

### 6.4 收口层

必须回答：

1. `changed_paths`
2. `当前 own 路径是否 clean`
3. 当前残留 blocker 是什么

如果一轮没有第 6.4 层，就不允许把它包装成“这刀已经闭环”。

## 7. V2 的禁止漂移清单

从现在起，以下几类漂移默认视为违规：

1. 再次把结构进展冒充体验过线
2. 再次把一份锐评升格成施工上位法
3. 在第一责任点未钉死前就直接改多段 runtime
4. 用户现场问题没过，就先去做 broad cleanup
5. 在 mixed hot 面上凭感觉吞 owner
6. 把 `NPC` 的 Inspector / scene 问题和导航 runtime 混成一锅
7. 把 `导航V2` 又写回“大讨论工作区”，不产出任何可约束未来开发的资产

## 9. 当前最终判断

### 9.1 这份文档为什么现在必须写

因为前面那轮讨论已经证明了一件事：

> 只靠聊天、回执、prompt 和 memory，不足以支撑 `导航V2` 继续长期推进。

现在正是产出的最佳时机，不是因为系统已经完美，而是因为：

1. 上位设计已经有了
2. 真实失败样本已经够多了
3. 当前活跃子线已经压到足够窄
4. 再不写统一文档，后续仍会继续靠聊天共识推进

### 9.2 `导航V2` 之后的角色

`导航V2` 从现在开始，不应只被定义成：

- 锐评审核工作区

它还应被定义成：

- **V2 开发现行宪法工作区**

也就是：

1. 不直接代替 `导航检查V2` 实现线程
2. 但负责给后续 V2 开发提供：
   - 统一口径
   - 禁止漂移边界
   - 阶段推进方向

### 9.3 当前一句话总口径

`006` 负责告诉我们最终要去哪里，`007` 负责告诉我们阶段怎么走，**本文负责确保从现在开始每一刀都不再漂离那条路。**
