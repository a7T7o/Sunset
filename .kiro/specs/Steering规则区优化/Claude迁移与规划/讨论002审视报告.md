## 1) 结论摘要（最影响效率的 4 个根因）

1. **最高优先级规则发生硬冲突**：`CLAUDE.md` 第1章“锐评即真理、严禁反驳”与 `.kiro/steering/code-reaper-review.md` 的“独立验证、不盲从、路径C中断”冲突，导致执行端在关键时刻必须自相矛盾。
2. **工具名/能力表述不一致，制造无效迭代**：`CLAUDE.md` 第8章提到 `Update` 工具与分块策略；而当前 Claude Code 的可用工具是 `Read/Edit/Write/Bash/...`（无 `Update`），会诱发“按规则执行却找不到工具”的假报错与来回修正。
3. **Unity 验证链路措辞容易把人引到终端 batchmode 灾难**：规则体系整体偏好 MCP（Unity Editor API），但历史文本里出现“终端命令/命令行验证”措辞时，会误导到危险路径。
4. **Hook 体系处于“配置存在但实现为空”的失效态**：`.claude/settings.local.json` 声明了多个 hooks，但对应 `.claude/hooks/*.sh` 为空文件，导致“大家以为自动化在跑、实际上没有”，产生 memory 漏记、规则未注入、压缩快照缺失等后果。
