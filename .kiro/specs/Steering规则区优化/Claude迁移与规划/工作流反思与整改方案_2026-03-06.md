# 工作流反思与整改方案（最终版，2026-03-06）

## 1. 最终纠偏结论

这次问题的本质，已经可以下最终判断：

1. **错误主体是我自己的工具使用手法，不是用户误解。**
2. **我之前最大的误判，是把 Claude Code 的 `.claude/hooks` 当成了接近 Kiro Hook 的语义执行层。**
3. **本项目真正需要迁移的核心，不是 Kiro 的“继承/压缩支线”，而是“每次对话结束后的强制收尾纪律”。**

因此，本最终版方案不再把“靠 shell hook 阻断错误”作为主修方向，而改为：
- 把纪律内化为 `CLAUDE.md` + steering + 工具调用 SOP；
- 把 `.claude/hooks` 降级为“提醒 / 审计 / 少量防呆”的辅助层；
- 把 `memory-update-check.kiro.hook` 的迁移重点，锁定为：**每一次进程、每一次对话结束后，都必须执行 memory 记录流程。**

## 2. 已确认的硬事实

### 2.1 Kiro Hook 的本质

根据 `hook事件触发README.md`：
- Kiro Hook 是 **事件 → askAgent / runCommand** 的机制；
- `promptSubmit` 会在用户发消息前向智能体注入提示；
- `agentStop` 会在智能体执行完成后触发收尾逻辑；
- `preToolUse` 才是 Kiro 里真正有机会阻止工具的事件。

这说明：**Kiro Hook 本质上是 IDE 内部的智能体调度机制，不是简单 shell 回调。**

### 2.2 `memory-update-check.kiro.hook` 的真正含义

`memory-update-check.kiro.hook` 的主干语义不是“继承恢复”，而是：
- `agentStop` 触发；
- 执行结束后，判断并完成 memory 更新；
- 强调“所有对话都必须记录”；
- 有子工作区时先子后主；
- 压缩/继承快照只是 Kiro 对话框场景下的附属分支，不是这条规则的核心灵魂。

**对 Claude CLI 迁移来说，应保留的是“结束后必记录”，而不是执着复刻 Kiro 的继承分支。**

### 2.3 当前 `.claude/hooks` 在这个项目里实际做了什么

根据 `.claude/settings.json`、`.claude/settings.local.json` 与现有脚本内容，当前配置应拆成两层理解：

- `.claude/settings.json`：仓库级 **permissions + hooks 接线**
- `.claude/settings.local.json`：本机侧补充 **allow 白名单**
- `.claude/hooks/*.sh`：被接上的实际脚本

当前 wiring 是：

1. `UserPromptSubmit` → `.claude/hooks/domain-rule-injector.sh`
2. `PreToolUse`（仅匹配部分 Bash）→ `.claude/hooks/pre-bash-block.sh`
3. `PostToolUse`（Edit|Write）→ `.claude/hooks/post-edit-format.sh`
4. `PreCompact` → `.claude/hooks/pre-compact-snapshot.sh`
5. `Stop` → `.claude/hooks/stop-update-memory.sh`

但脚本现状非常明确：
- `domain-rule-injector.sh`：只打印一行 marker。
- `post-edit-format.sh`：只打印一行 marker。
- `pre-compact-snapshot.sh`：只打印一行 marker。
- `stop-update-memory.sh`：只打印一行 marker。
- `pre-bash-block.sh`：只做少量模式匹配；且命中后当前脚本是 `exit 0`，并不构成可证明的硬阻断。

结论：**当前 `.claude/settings*.json + .claude/hooks/*.sh` 本质上是“权限表 + 接线板 + 终端提示脚本集合”，不是 Kiro 那种语义级 Hook 工作流。**

## 3. 用人话解释：现在这些 `.claude/hooks` 到底干了什么

### 3.1 `domain-rule-injector.sh`
- **实际作用**：用户发消息时，终端打印一句“hook active”。
- **没有做到**：不会自动读取 steering，不会把规则真正注入到我的思考里。

### 3.2 `pre-bash-block.sh`
- **实际作用**：在部分 Bash 命令前检查字符串里有没有极少数危险模式。
- **没有做到**：不能代表 Kiro 的 `preToolUse` 语义审查；当前也不能把它当成可靠总闸门。

### 3.3 `post-edit-format.sh`
- **实际作用**：文件 Edit/Write 后打印一句“done”。
- **没有做到**：没有格式化，也没有校验。

### 3.4 `pre-compact-snapshot.sh`
- **实际作用**：压缩前打印一句 marker。
- **没有做到**：没有创建快照，也不该再作为本迁移目标的主线。

### 3.5 `stop-update-memory.sh`
- **实际作用**：会话停止时打印一句 marker。
- **没有做到**：没有自动更新 memory，也不能替代我自己的收尾纪律。

一句话总括：

**当前 `.claude/hooks` 几乎都只是“提示你我到了哪个生命周期节点”，不是“帮我完成 Kiro 规则迁移”。**

## 6. 最终版标准作业流（SOP）

### 6.1 执行前（强制）
1. 先界定本轮任务边界与产物。
2. 按用户问题路由需要的 steering 文档。
3. 先读证据，再给计划，再开始动作。

### 6.2 执行中（强制）
1. 所有结论都必须有 Read/Grep 证据支撑。
2. 锐评必须走 `code-reaper-review.md` 的 A/B/C 分流。
3. 不得超出用户批准范围顺手扩写。
4. 不得把 `.claude/hooks` 当成语义护栏；真正护栏是我自己的操作纪律。

### 6.3 执行结束前（硬规则）
**每一次进程、每一次对话，在结束前都必须执行 memory 流程。**

流程固定如下：

1. 先判断当前是否存在明确子工作区。
2. 有子工作区：
   - 先更新子 `memory.md`（详细记录）；
   - 再更新父 / 主 `memory.md`（摘要记录）。
3. 无子工作区：
   - 仅更新主 `memory.md`。
4. 纯问答、方案讨论、锐评审视、规则梳理，同样必须记忆；不允许以“无代码改动”为理由跳过。
5. 本 Claude CLI 迁移版 **不再把继承快照作为主流程要求**。

这一步不是“建议”，而是交付前最后一道硬工序。

## 7. append-only 的最终防复发机制（工具调用手法版）

### 7.1 写前步骤（必须）
1. 先读取目标 `memory.md` 当前内容，至少确认末尾结构。
2. 确认旧会话块仍完整存在。
3. 明确本次是“新增会话块”，不是“更新旧会话块”。

### 7.2 写入动作（必须）
1. 只能在 **EOF（文件末尾）之后追加** 新内容。
2. 禁止选中旧会话块做替换。
3. 禁止以“格式整理”为名改写历史会话。
4. 若需分卷，允许“`memory.md` → `memory_N.md` 归档 + 新建 `memory.md` 前情提要 + 追加新记录”；但这不等于改写历史内容。

### 7.3 写后校验（必须）
1. 重新读取目标 memory 尾部。
2. 确认旧记录还在。
3. 确认新记录只出现在尾部。
4. 确认父 / 子顺序正确（有子则先子后父）。

**从现在起，append-only 的第一保障是这套手法，不是幻想中的外部 hook 阻断。**

## 9. `.claude/hooks` 如何最大限度贴近 Kiro，但不再自欺

### 9.1 能贴近的部分

1. **生命周期提醒**
- `UserPromptSubmit` 可以提醒“进入新一轮消息”。
- `Stop` 可以提醒“会话即将结束”。

2. **少量防呆**
- `PreToolUse` 可以对 Bash 做有限的危险命令筛查。

3. **审计与日志**
- `PostToolUse` 可以在未来接真实 formatter / checker / log。

### 9.2 贴近不了的部分

1. **不能等价替代 Kiro 的 askAgent 路由能力**
- 它不会像 Kiro 那样把一段前置语义真正注入为我的执行任务。

2. **不能等价替代 Kiro 的 agentStop 语义收尾能力**
- 它不会自动帮我完成“判断是否记忆 + 写入子父 memory + 选择正确粒度”的语义动作。

3. **不能承担主治理责任**
- 最多是辅助层，不能替代 `CLAUDE.md` 和我的操作纪律。

### 9.3 最现实的完善方向

- **方向 A**：把所有 `.claude/hooks` 的命名/注释改成“marker / reminder / audit”语义，防止误导。
- **方向 B**：保留 `pre-bash-block.sh` 作为“尽力而为的 Bash 防呆层”，但不宣称它是绝对闸门。
- **方向 C**：把真正的 Kiro 迁移重点全部前移到 `CLAUDE.md + steering + 工具手法`。

## 12. 补强结论：工作区先行 / 代办先行必须升级为硬门槛

### 12.1 新暴露出的流程缺口

本轮又暴露出一个之前没有钉死的漏洞：

- 另一终端在还没先锚定 `.kiro/specs/` 工作区时，就进入了 Claude PlanMode；
- 正式规划内容先写到了 `C:\Users\aTo\.claude\plans\groovy-hatching-pond.md`；
- 这虽然符合 Claude 自身的全局计划文件行为，但**不符合本项目的工作区先行制度**。

这说明：

1. 仅仅写“先定位当前最相关工作区”还不够。
2. 只要没有把“禁止把全局 plans 当项目正式产物”写成硬门槛，流程就仍然会被系统默认行为带偏。
3. “发现错误流程本身先记代办”这件事，也还没有被我执行成肌肉记忆。

### 12.2 新增纠偏结论

因此最终版方案还必须再补一层：

- **工作区先行**：任何计划、报告、审视、任务拆解、整改方案，都必须先绑定到 `.kiro/specs/` 内的明确工作区。
- **代办先行**：凡是暴露出“暂不立刻修 / 后续处理 / 流程漏洞 / 制度缺口”的事项，必须先写入 `000_代办`，再继续分析或延后。
- **PlanMode 降级**：若 Claude PlanMode 的正式产物天然先落到用户主目录 plans，则在本仓库内不应把它当正式规划载体；正式规划仍要回落到工作区内文档。

### 12.3 防复发 SOP（新增）

以后凡遇到“要开始规划 / 设计 / 长报告”的任务，顺序必须固定为：

1. **先找工作区**：确认当前任务属于哪个 `.kiro/specs/.../`。
2. **没有工作区就先建工作区**：禁止先在项目外目录落正式文档。
3. **若发现这是后续治理项/遗留项**：先记 `000_代办`。
4. **再读该工作区 memory / 三件套 / 锐评**。
5. **最后才开始产出 design/tasks/报告**。

也就是说，今后正确顺序不是：
- 先 PlanMode → 再想工作区

而必须是：
- **先工作区 / 先代办 → 再规划落盘**。

### 12.4 当前已落实的修补动作

本次已立即完成三件事：

1. 在 `000_代办/Steering规则区优化/` 下新建 `TD_Claude迁移与规划.md`，把这次“工作区先行缺口”与“代办先行缺口”正式入账。
2. 在主代办 `TD_000_Steering规则区优化.md` 中补入子代办索引与 P0 汇总项。
3. 在 `CLAUDE.md` 中新增“工作区先行硬门槛”，明确禁止把 `~/.claude/plans/` 视为本仓库正式规划产物目录。

### 12.5 补强后的最终现实结论

到这一步，Claude CLI 迁移的主防线已进一步明确为：

1. **工作区先行**
2. **代办先行**
3. **证据先行**
4. **memory 收尾必做**

其中：
- hooks 仍只是辅助层；
- PlanMode 也不是项目正式产物层；
- 真正不会漂移的，必须是 `.kiro/specs/` 内部的工作区文档体系。

## 13. 补充收口：acceptEdits 常驻 + hooks 二次实测

### 13.1 已落实动作

- `.claude/settings.local.json` 已移除 `EnterPlanMode`，仓库默认保持 `acceptEdits` 工作方式。
- `.claude/settings.json` 已把 `PreToolUse.matcher` 补齐为同时覆盖 `git push -f*` 与 `git push --force*`。
- `.claude/hooks/pre-bash-block.sh` 已改为命中危险模式 `exit 2`。
- `.claude/hooks/stop-update-memory.sh` 已改为基于 `git diff` 的 append-only 检查器：发现已跟踪 `memory.md` 有删除行时告警并 `exit 2`。

### 13.2 二次实测结论

- 单脚本层面：`pre-bash-block.sh` 对 `git push -f` / `git push --force` 均返回 `2`，说明脚本自身 gate 语义已成立。
- runner 层面：对 `git push -f -h`、`git push --force -h`、`rm --version` 的 Bash 安全探针仍未看到 hook 输出；临时 probe log 也未生成，因此**当前没有证据证明 repo `PreToolUse` hook 已被 runner 调用**。
- stop hook 层面：在当前仓库状态下 standalone 执行 `stop-update-memory.sh` 返回 `0`，说明“无 memory 删除行”路径正常；但 `Stop` 生命周期是否具备真实阻断力，仍不能高估。

### 13.3 因此本仓库的最终执行口径

1. **默认不进入 PlanMode**，直接保持 `acceptEdits`。
2. **需要规划时也不依赖全局 plans**，而是直接回落到当前工作区文档。
3. **hooks 只算辅助层**：即便脚本改成 `exit 2`，在 runner 真实调用未证实前，仍不得把它当成绝对闸门。
4. **memory 收尾纪律继续由文档与工具手法兜底**：先子后父、append-only、写后复查。

## 14. 最新修正：PreToolUse 的 runner 真实语义已经拿到实证

### 14.1 为什么要再修正一次

前一版结论里，我还保留了一个审慎判断：
- `pre-bash-block.sh` 脚本本身已经具备 `exit 2` gate 语义；
- 但是否真的被当前 Claude runner 调用，还没有拿到硬证据。

现在这个判断需要更新，因为本轮已经补到了 runner 级实证。

### 14.2 新发现：旧 matcher 写法本身就不可靠

继续补查资料后，确认当前 Claude Code 的 hook matcher **不应依赖** `Bash(rm*|git push -f*)` 这种“按 Bash 参数做模式匹配”的写法。

更稳妥、且已被本轮实测验证的做法是：

```json
"matcher": "Bash"
```

然后把危险命令识别全部放进 `.claude/hooks/pre-bash-block.sh` 自己判断。

也就是说，真正可靠的分层应是：
- `settings.json`：负责把 Bash 调用统一送进 hook
- `pre-bash-block.sh`：负责识别危险模式
- `exit 2`：负责把危险命令升级成 runner 级阻断

### 14.3 本轮 runner 实证结果

在把 `PreToolUse.matcher` 改成 `"Bash"` 后，本轮做了 4 个受控验证：

1. `git push -f -h`
   - Bash 工具直接返回 `PreToolUse:Bash hook error`
   - 没有继续输出 git help
2. `git push --force -h`
   - 结果同上
3. `rm --version`
   - 命令正常执行
4. 临时 probe log
   - hook 确实写出了 3 条调用记录

这四条证据合在一起，已经足够得出更强结论：

- **当前 runner 确实会调用 repo `PreToolUse` hook**；
- **当 hook 返回 `exit 2` 时，Bash 调用会被真实阻断**；
- **普通 Bash 命令在脚本未命中危险模式时会被正常放行**。

### 14.4 因而本仓库现在的最稳实现口径

1. `PreToolUse` 这条链路现在已经不是“猜测可用”，而是**runner 级已证实可用**。
2. 可靠 wiring 方式不再是 `Bash(rm*|...)`，而是：
   - `matcher: "Bash"`
   - hook 内部做命令过滤
3. 因此 `.claude/hooks/pre-bash-block.sh` 现在的现实定位应上调为：
   - **已证实生效的 Bash 防呆/阻断层**
   - 但仍只覆盖脚本里列出的少量危险模式，不能夸大成“全能安全网”
4. `Stop` hook 仍然维持原判断：
   - standalone append-only 检查器已经成立；
   - 但 `Stop` 生命周期下 `exit 2` 的真实阻断力，仍未直接实测，不能高估。

### 14.5 修正后的最终执行口径

到这一步，本仓库的治理主轴进一步收紧为：

1. **默认 `acceptEdits`，不主动 PlanMode**
2. **正式规划永远回写工作区文档**
3. **`PreToolUse` 已证实可用于 Bash 危险命令防呆，但只靠 `matcher: "Bash"` + hook 内过滤才可靠**
4. **`Stop` 继续只当 append-only 审计/提醒层，直到 runner 级阻断语义被直接证实**
5. **真正不会漂移的主防线仍然是 `CLAUDE.md` + steering + memory 收尾 SOP**
