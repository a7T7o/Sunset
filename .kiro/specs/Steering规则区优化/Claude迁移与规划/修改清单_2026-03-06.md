### 3.1 `CLAUDE.md`（修改 / 高影响：全局执行协议）

**修改前**
- “锐评即真理”：要求无条件服从锐评并按锐评重构。
- “事实核查”工具名使用旧术语：`read_file`。
- “写入协议”偏向“禁 raw / 参数约束 + Bash fallback 写整文件”。
- 没有 Unity MCP 的“权限沙盒/验证闭环”专章。

**修改后**
- 将锐评处理改为“锐评优先级最高，但拒绝盲从”，强制执行 `.kiro/steering/code-reaper-review.md` 的事实核查与 A/B/C 路径分流。
- “事实核查”工具名对齐 Claude Code：`Read`/`Grep`。
- 第八章升级为“大文件分块写入协议”，新增硬规则：
  - 已存在文件：只允许 `Edit`，禁止 `Write` 覆盖。
  - 新文件：`Write` 单次 50-80 行，后续用 `Edit` 分块续写。
  - 出现 `raw` 报错：立刻停止重试，拆块。
- 新增第九章：MCP 自动化沙盒与 Unity 权限控制协议（Create-only 自动化放行、No-modify 拦截转交、禁止 batchmode 拉 Unity、MCP 编译/测试闭环、禁止 PlayMode MCP 等）。

**我做了什么（动作）**
- 将“锐评无条件服从”改为“锐评输入最高优先级，但必须先事实核查再执行”，把“盲从制造缺陷”列为禁止项。
- 将旧工具术语替换为 Claude Code 工具集合（`read_file`→`Read/Grep`）。
- 把“写入稳定性”从“参数正确性”升级为“可操作的分块写入 SOP”，并对 `Write` 覆盖行为加硬性红线。
- 把 Unity MCP 的边界与验证流程制度化写进全局协议。

**影响/风险**
- 影响面覆盖后续所有对话：它会改变“是否要生成审视报告”“是否允许覆盖文件”“如何跑验证”的默认行为。

### 3.4 `.claude/hooks/pre-bash-block.sh`（新增内容 / 防御性：危险 Bash 拦截）

**修改前**
- 空文件。

**修改后（核心逻辑）**
- 从 stdin 读取命令 → 归一化空白 → 用 `grep -Eiq` 匹配危险模式：
  - `rm -rf /`
  - `git reset --hard`
  - `git push -f/--force`
- 命中则输出 `⛔ blocked...` 并 `exit 0`，否则输出 `allow`。

**我做了什么（动作）**
- 增加了一层“命令级黑名单”防线，尝试阻断最明显的不可逆/破坏性 Bash。

**影响/风险**
- 这是“兜底拦截”，不是完整沙箱：正则覆盖有限，且 `exit 0` 的语义依赖 hook runner 的“输出即阻断”约定（需要你确认 Claude hooks 实际行为与预期一致）。

### 3.5 `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`（修改 / 高影响：运行时行为变更）

**修改前**
- 已有“UI 面板打开时禁用移动/工具使用”的 gating（`IsAnyPanelOpen()` 相关）。

**修改后（新增代码块摘要）**

  - `private bool _inputEnabled = true;`
  - `public void SetInputEnabled(bool enabled)`：禁用时把 `playerMovement` 输入归零。
- 新增事件订阅生命周期：
  - `OnEnable()`：订阅 `DialogueStartEvent/DialogueEndEvent`

- 在关键输入处理点加 gate：
  - `HandleMovement()`：`if (!_inputEnabled) { SetMovementInput(Vector2.zero,false); return; }`

**我做了什么（动作）**
- 以“最小接入”的方式把对话事件接到输入系统：对话开始 → 禁输入；对话结束 → 恢复输入。

**影响/风险**
- 这是主线控制器的行为改动：如果事件触发链路或订阅时机不一致，可能导致输入被意外锁死/误放开。
