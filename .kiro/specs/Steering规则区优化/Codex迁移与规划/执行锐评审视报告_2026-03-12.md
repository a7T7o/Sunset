# 执行锐评审视报告（2026-03-12）

## 1. 审视对象

本次审视对象是“要求立即执行 `NPC` 第二刀真实修正，并在同轮内完成真实点击验收与失败即回滚”的执行锐评。

## 2. 路径判定

本次判定为 **Path C**。

原因不是“第二刀方向错误”，而是这份执行锐评同时混入了两类不同性质的要求：

1. 一部分是事实成立且应立即采纳的执行前修正；
2. 另一部分则把需要用户参与的真实点击复验、以及基于复验结果的回滚动作，错误地压缩成同一轮内必须自动完成的闭环。

这会把高风险外部状态写入与尚未发生的用户交互结果混成一条不可验证、也不可诚实承诺的执行链。

## 3. 事实核查表

| 锐评主张 | 当前事实 | 结论 |
|---|---|---|
| 第一刀单改 `git_branch` 已失败，不能继续停在预案层 | 当前治理文档与验证矩阵已确认第一刀失败样本成立 | ✅ 事实正确 |
| 第二刀对象已经锁到 rollout 恢复层 + `threads.git_sha` | 当前候选彻查报告、T80、第二刀预案已全部锁定为该组合 | ✅ 事实正确 |
| `active-workspace-roots` 不能再写成恒定实时值，执行前必须改成采样值口径 | 该字段是动态全局 UI 状态；用户本轮再次给出与上轮不同的直读值 | ✅ 事实正确 |
| 这轮必须直接执行第二刀写入 | 从技术上可执行，但属于高风险本地状态修正，必须与真实点击复验分段处理 | ⚠️ 方向正确，执行闭环表述过度压缩 |
| 这轮必须在同一轮内完成真实点击验收 | 真实点击需要用户在客户端执行，模型不能伪造该结果 | ❌ 执行条件不成立 |
| 如果第二刀仍失败，必须立即回滚，不准拖 | 只有在真实点击结果已返回后，才能判断是否回滚；不能在尚未收到复验结果前预设已失败并自动回滚 | ❌ 时序错误 |

## 4. 我认同的部分

### 4.1 执行前口径修正必须采纳

- `active-workspace-roots` 只能写成采样时刻值；
- 不能继续写成“当前恒定实时值”；
- 第二刀相关文档在真正执行前，必须再次直读一次当前值并留档。

### 4.2 第二刀不该继续停在纯分析层

- 当前证据链已经足够支撑第二刀最小写入；
- rollout 恢复层 + `threads.git_sha` 已经不是“候选方向”，而是当前唯一合理的主执行入口。

## 5. 我不采纳的部分

### 5.1 不采纳“同轮内必须完成真实点击验收”

- 这一步依赖用户在 `Codex` 客户端中真实点击线程；
- 模型无法替用户完成该动作，也不能诚实地把“待用户点击”写成“本轮已完成验收”。

### 5.2 不采纳“失败即回滚也必须同轮自动完成”

- 回滚决策的前提是已经拿到真实点击结果；
- 如果还没有用户复验结果，就不存在“已失败”的事实基础；
- 因此只能写成：
  - 第二刀执行后进入“待用户两条路径复验”；
  - 一旦用户回报失败现象，再立刻执行预置好的回滚。

## 6. 独立判断

当前正确的执行顺序应当是：

1. 先把动态值口径收紧成“采样值”；
2. 再做第二刀真实写入；
3. 写后立即完成本地回读校验；
4. 把 `T74` 更新为“第二刀已执行，待用户两条路径复验”；
5. 等用户返回真实点击结果后，再做通过/失败判定；
6. 若失败，再执行回滚并形成失败样本证据。

也就是说：

- 第二刀可以开始执行；
- 但“执行写入”和“用户真实复验/是否回滚”必须拆成两段；
- 锐评想推进执行，这个方向是对的；
- 它把用户参与的验收和回滚压成同轮自动闭环，这一点不符合当前能力边界。

## 7. 建议采用的最小修正版

把原执行锐评收紧为以下口径后，才适合作为下一轮真实执行指令：

1. 执行前先补采样值说明；
2. 真实执行 `NPC` 第二刀写入；
3. 完成本地回读校验；
4. 更新执行记录与 `T74`；
5. 向用户明确两条点击复验路径；
6. 收到用户复验结果后，再决定是否触发回滚。

## 8. 审视结论

这份执行锐评不是整体错误。

它的核心推进方向成立：

- 不能继续停在分析层；
- 第二刀应进入真实执行。

但它把“可由模型立即完成的高风险写入”和“必须由用户参与提供结果的真实点击验收/失败回滚”压成了同一轮必达闭环，属于执行时序与能力边界错误。

因此本次结论为 **Path C**：先出审视报告，停止把它原样当成可直接执行的完整指令。
