# Codex工作流与技巧手册（Sunset 项目版）

**创建日期**: 2026-03-07

---

## 1. 手册定位

这份手册只回答一件事：

> 在 Sunset 项目里，Codex 应该怎样工作，才能既遵守 `.kiro/steering`，又不把自己假装成 Kiro 或 Claude Hook 系统。

它不是新的业务规则正文，也不替代 `.kiro/steering`。

---

## 2. Codex 的总原则

### 2.1 三条总纲

1. **Steering 是唯一规则源**：遇到业务规则、工作区规则、锐评规则、场景改动规则，一律回到 `.kiro/steering/`。
2. **工作区是正式产物层**：计划、设计、报告、分析、迁移方案都应写回 `.kiro/specs/` 当前工作区。
3. **Memory 收尾是硬纪律**：结束前必须更新 memory；有子工作区时先子后父。

### 2.2 Codex 不做的事

- 不假设自己拥有 Kiro 的 `promptSubmit` / `agentStop` / `preToolUse` 原生 Hook 语义。
- 不假设 `.claude/hooks` 的能力能自动平移到 Codex。
- 不把 `update_plan` 当成项目正式规划文档。
- 不编造当前环境不存在的工具能力。

## 3. 会话启动 SOP

### Step 1：先锚定工作区

- 如果当前任务已经有明确工作区：先读该工作区 `memory.md`。
- 如果当前任务没有工作区，但需要落盘规划/报告/设计：先创建或确认工作区，再继续。

### Step 2：按关键词路由规则

优先参考下列路由：

| 任务类型 | 优先读取 |
|---|---|
| 工作区、memory、快照、分卷 | `.kiro/steering/workspace-memory.md` |
| 锐评、审查、审视报告 | `.kiro/steering/code-reaper-review.md` |
| 场景、Prefab、Inspector、SO 修改 | `.kiro/steering/scene-modification-rule.md` |
| 文档写作、沟通节奏 | `.kiro/steering/rules.md` + `communication.md` |
| 领域系统问题 | 对应 canonical 领域文档 |

### Step 3：证据先行

- 任何结论前，先读取相关文档、代码、配置或日志。
- 不对未读取内容做断言。

### Step 4：决定是否使用 `update_plan`

适合使用 `update_plan` 的情况：

- 任务非单步可完成。
- 需要多阶段推进。
- 用户一次要做多件事。
- 你需要给用户明确进度感。

不适合使用的情况：

- 纯问答、单文件小修、一次性读取后即可回答的问题。

### Step 5：正式规划落在工作区文档

- `update_plan` 只用于当前会话追踪。
- 正式的规划、拆解、设计结论，要写回工作区文档，而不是停留在工具状态里。

## 4. Codex 工具使用技巧

### 4.1 `shell`

适合：

- 搜索文件与文本（优先 `rg` / `rg --files`）
- 读取文档
- 生成补丁
- 跑局部验证命令

技巧：

- 先成组读取，再做判断。
- 读大文件时，优先按主题精准抽读，不要一口气扫全仓。
- 写入前先确认目标文件当前状态，尤其是 `memory.md` 尾部。

### 4.2 `update_plan`

适合：

- 让用户知道当前阶段、下一步和是否已完成。

技巧：

- 计划项要短、可验证、可完成。
- 始终保持最多一个 `in_progress`。
- 不要把显而易见的单步任务硬拆成大计划。

### 4.3 `web`

适合：

- 最新信息
- 官方文档核验
- 需要精确来源或链接的内容
- 高时效 / 高风险问题

技巧：

- 技术问题优先使用官方文档或一手资料。
- 如果信息可能已经变化，就不要只凭记忆回答。

### 4.4 Unity MCP

适合：

- Unity 编辑器内的受支持操作
- 查看控制台日志
- 重新编译脚本
- 运行 EditMode 测试

技巧：

- 高风险修改（现有场景物体、Prefab 配置、Inspector、SO 核心数据）先审视、后确认。
- 不要用 shell 假装替代编辑器原生命令。

## 5. Kiro / Claude 经验在 Codex 中的落地方式

### 5.1 能直接继承的部分

- 工作区先行
- 证据先行
- Steering 路由思路
- Memory append-only 纪律
- “正式产物回写工作区”的治理原则

### 5.2 需要降级为手动 SOP 的部分

- 自动 prompt 路由
- 自动 memory 更新提醒
- 自动压缩快照与继承恢复
- 生命周期 Hook 驱动的自动治理

### 5.3 需要 Codex 化重写的部分

- 会话前导：通过简短 preamble 告知用户下一步，而不是依赖 Hook 注入。
- 计划跟踪：通过 `update_plan` 提供状态，而不是依赖外部 plan 文件。
- 最终交付：通过简洁、路径可追踪的最终回复交付，而不是依赖特定终端标记。

## 6. Memory 收尾 SOP（Codex 版）

### 6.1 何时必须记忆

以下情况都必须更新 memory：

- 改了代码
- 新建或修改了工作区文档
- 形成了架构结论或治理结论
- 做了审查 / 锐评核查
- 即使是纯问答，但产出了可追踪结论

### 6.2 更新顺序

1. 有子工作区：先子后父。
2. 多层嵌套：从最内层开始逐级补。
3. 无子工作区：更新当前主工作区 `memory.md`。

### 6.3 追加手法

1. 写前先读目标 `memory.md` 尾部。
2. 确认旧记录仍在。
3. 只在 EOF 之后追加。
4. 写后再次检查尾部，确认不是覆盖历史。

### 6.4 Codex 额外提醒

- 不要因为“只是写文档”就跳过 memory。
- 不要把计划工具状态当成已完成收尾。
- 如果创建了新工作区，`memory.md` 也要一起建。

## 7. Codex 输出与沟通技巧

### 7.1 对用户的即时更新

- 工具调用前先发一句简短 preamble。
- 一次说明一组动作，不要每读一个文件都单独报备。
- 中途若任务较长，给一句 8-10 字左右的进度感更新。

### 7.2 最终回复

- 中文、直接、少废话。
- 优先交代：做了什么、改了哪些文件、还有什么待继续。
- 路径要可追踪；在 Codex Desktop 中优先使用绝对路径。

### 7.3 不再强行沿用的旧习惯

- 不要求输出 Kiro 的“输出完毕标记”。
- 不把 Claude 专用 Hook 语义写成 Codex 必备流程。
- 不把历史终端的约束原样照搬到当前平台。

## 8. 常见误区清单

1. **把 `update_plan` 当正式设计文档**
   - 修正：正式内容回写工作区文件。

2. **把 Kiro Hook 当 Codex 默认存在**
   - 修正：改成手动 SOP 或显式转交。

3. **忘记记 memory**
   - 修正：把 memory 收尾视为最终回答前的固定步骤。

4. **没读证据就下判断**
   - 修正：先读再说，特别是规则类、迁移类、配置类任务。

5. **顺手改太多**
   - 修正：只做当前任务要求的最小变更。
