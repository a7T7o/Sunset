## 2. 字段纠偏边界

## 2.1 可以自动修

以下字段适合自动纠偏：

- `threads.cwd`
  - 条件：项目内存在显式线程路由映射，且目标目录真实存在。
  - 原因：它本质是线程默认工作目录投影，修正后不会改聊天原文。
- `.codex-global-state.json` 中的 `electron-saved-workspace-roots`
  - 条件：只是补齐缺失的已知工作区根，不删除现有根。
  - 原因：它更像工作区清单，而不是线程历史语义。

## 2.2 只能谨慎修

以下字段不适合默认自动改，只能在严格条件下修：

- `threads.git_branch`
  - 允许修正的前提：
    1. 已存在显式线程路由映射；
    2. `threads.cwd` 已与期望目录一致；
    3. 当前真实 worktree 分支与映射期望分支一致；
    4. 当前线程属于长期固定线程，而不是临时线程；
    5. 本次修正有状态库备份。
  - 只提示、不自动改的情况：
    - `cwd` 仍不可信；
    - 当前真实分支不唯一或无法获取；
    - 线程标题 / `thread_name` 历史明显漂移；
    - 用户当前正在别的工作区根活动，容易误伤。
- `.codex-global-state.json` 中的 `active-workspace-roots`
  - 这是全局当前激活工作区状态，不是单线程专有字段。
  - 后续可以做“谨慎纠偏候选”，但不应在第一版守护器里默认自动改。

## 2.3 默认绝不能动

以下对象默认不进入自动纠偏：

- `session_index.jsonl` 的历史 `thread_name`
- `threads.title`
- `threads.rollout_path`
- `threads.first_user_message`
- `sessions/*.jsonl`
- `archived_sessions/*.jsonl`
- `WindowsApps` 下的 Codex 安装本体

原因很简单：这些对象更接近线程历史语义或聊天原文，一旦改坏，恢复成本远高于显示层收益。

## 3. 路线选择

## 3.1 结论：优先扩展现有 `provider-bridge`

当前倾向性结论是：

- **选扩展现有 `provider-bridge`**
- **暂不新建第二桥**

## 3.2 为什么选它

现有 `provider-bridge` 已经具备这些基础能力：

- 备份 `state_5.sqlite`
- 一次性修复
- 后台守护
- 状态日志
- `cwd` 规范化
- `has_user_event` 回补
- `model_provider` 桥接

如果直接另起第二桥，会立刻引入三类新风险：

- 两个守护器同时看同一份状态库，竞争写入；
- 备份、日志、状态文件分裂成两套；
- 用户以后更难判断“到底是谁在改状态”。

## 3.3 为什么暂不选新建第二桥

暂不新建第二桥，不是因为第二桥永远不行，而是因为当前最重要的是**先把状态治理的写入口统一**。

在现在这个阶段，新建第二桥的坏处明显大于好处：

- 多一层竞争；
- 多一套配置；
- 多一套启停入口；
- 更难复盘。

---

## 4. 最小实现范围

后续扩展 `provider-bridge` 的最小实现范围应当是：

1. 增加项目级线程路由映射读取能力；
2. 增加线程承接审计模式；
3. 增加“只报告，不修正”的差异分类；
4. 在审计稳定后，再增加 `threads.cwd` 的继续纠偏；
5. `threads.git_branch` 第一版默认只提示，不自动修；
6. `active-workspace-roots` 第一版只观察，不自动改。

也就是说，第一刀不是“自动乱改更多字段”，而是：

- 先让它更会看；
- 再让它更会报；
- 最后才让它在小范围内安全地修。
