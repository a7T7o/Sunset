## 12. 真实线程承接修复阶段补充协议

### 12.1 分层完成态收紧

- 协议层 / 只读治理层：可认为已成立；
- 修复设计层：已进入；
- 真实线程承接层：尚未成立；
- 用户验收层：尚未成立。

当前禁止再把整体描述成“整体到 L4”；`L4` 只适用于“某个样本已经完成文档同步态”，不适用于整个承接系统。

### 12.2 当前问题的四层拆解

| 层级 | 当前问题 | 当前结论 |
|------|------|------|
| Git / worktree 真实层 | worktree 是否还在、分支是否还在 | `NPC` 与 `farm` 的真实 worktree 和真实分支都还在；根仓库报 `branch already used by worktree` 反而证明 worktree 仍存在。 |
| 状态库投影层 | 线程恢复依赖的元数据是否过时 | `cwd` 已对到 worktree，但 `git_branch = main` 仍是核心旧投影；`NPC` 最优先，`farm` 也有同类结构。 |
| UI / workspace 承接层 | 点击线程后为何掉回 `Sunset/main` | 当前更像“线程元数据旧值 + 状态投影旧值 + UI 刷新失配”叠加，而不是单一字段问题。 |
| 用户验收层 | 用户眼里什么时候才算恢复 | 只有点开线程就回正确现场、且不再依赖重启，才算真正恢复。 |

### 12.3 可修复对象清单

#### C1. 已确认可只读核定
- `.codex-global-state.json`
- `state_5.sqlite`
- `session_index.jsonl`
- `git worktree list`
- 功能 worktree 真实分支与 HEAD
- 线程记忆实际位置
- 用户已反馈的 UI 分组 / 线程点击现象

#### C2. 已确认可做受控修正预案
- 单线程 `state_5.sqlite` 字段候选：
  - `cwd`
  - `git_branch`
  - `git_sha`
  - `rollout_path`
- 单线程承接路由映射
- 线程目标 memory 归位策略
- 审计脚本 / 基线脚本 / 最小修复预案脚本

#### C3. 当前仍属客户端黑箱，不可承诺
- provider 切换后一定不漂
- 单次脚本执行后 UI 一定无重启刷新
- 不依赖任何 UI 操作就能立刻全恢复

### 12.4 两个问题的协议答案

#### A. 到哪一阶段才能验收到“无需重启/重启后都正常”
- 只读治理阶段：不能；
- 状态投影修正准备阶段：不能；
- 单线程真实修复阶段：可能只让单线程成立；
- 用户真实闭环验收阶段：才允许对外确认“重启/不重启都能回正确现场”。

#### B. 当前是否存在“一跑即恢复 UI”的脚本式能力
- 当前没有已证实可承诺的能力；
- 当前最接近的方案是：
  1. 建立单线程最小修复入口；
  2. 对单线程状态投影字段做受控修正；
  3. 用用户真实点击矩阵验收；
  4. 根据结果再判断是否能扩成更通用的修复脚本。
- 若失败，默认回退到“只读核定 + 重启后人工复验”的 fallback。
