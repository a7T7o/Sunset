# Codex现场判定与完成态协议

**日期**: 2026-03-12  
**文档定位**: 用于统一回答“现在该信哪个现场”“什么才算做完了”“冲突时怎么判断和怎么说”。

---

## 1. 现场定义

### 1.1 根仓库现场

指项目根目录仓库的真实 Git 现场，例如：

- `D:\Unity\Unity_learning\Sunset`
- `git branch --show-current`
- `git rev-parse HEAD`

它适用于：

- 治理线程；
- 总览线程；
- 只读审计线程；
- 真正绑定 `main` 的长期线程。

### 1.2 线程 worktree 现场

指某条长期功能线程对应的真实物理 worktree 现场，例如：

- `D:\Unity\Unity_learning\Sunset_worktrees\NPC`
- `D:\Unity\Unity_learning\Sunset_worktrees\farm-10.2.2-patch002`

它适用于：

- 功能实现线程；
- 需要隔离视图和文件树的长期开发线程；
- 任何“我这条线程现在到底在哪工作”的判断。

### 1.3 UI 激活工作区现场

指客户端当前激活工作区根与相关前端状态，例如：

- `.codex-global-state.json` 中的 `active-workspace-roots`
- 已保存工作区根
- 客户端侧边栏当前所处的工作区视图

它适用于：

- 解释客户端为什么显示这样；
- 判断线程为什么会误回根仓库；
- 守护式纠偏是否该触发。

它**不是**当前真实 Git 现场的最高真相源。

### 1.4 用户验收现场

指用户当前真实打开、真实观察、真实点击和真实验收的那份工程与那块视图。

在 Unity 项目里，它至少包含：

- 用户当前打开的工程路径；
- 当前打开的场景 / 资源 / Inspector；
- 用户眼前能否真的看到结果；
- 是否真的能点、能交互、能验收。

它是可视化任务的一等公民。

## 2. 优先级协议

### 2.1 默认优先级

当任务涉及真实交付、线程归属或用户混乱时，默认优先级如下：

1. **用户验收现场**
2. **线程 worktree 现场**
3. **根仓库现场**
4. **UI 激活工作区现场**
5. **状态库 / 线程记忆 / 历史标题等投影线索**

### 2.2 为什么这样排

- 用户验收现场优先，因为“用户眼前是否成立”决定当前是否真的交付；
- 线程 worktree 现场次之，因为它决定当前线程应该在哪工作；
- 根仓库现场再次之，因为它是治理主路，但不必然等于当前线程执行现场；
- UI 激活工作区更像“客户端显示现场”，重要但不能压过真实 Git 和用户验收；
- 状态库与记忆更多是证据与投影，不应替代当前真相。

### 2.3 特殊说明

如果当前任务明确是治理 / 审计 / 规则任务，而不是功能线程实现，则：

- 根仓库现场与治理线程现场通常合并为第一执行现场；
- 但一旦用户反馈“我眼前看到的不是你说的状态”，仍应把用户验收现场提升回第一优先级。

## 3. 冲突时的决策树

### 3.1 第一步：先判断任务类型

必须先回答：

- 这是治理线程还是功能线程？
- 这是只读审计还是要落地产物？
- 这是纯 Git / 文档问题，还是 Unity 可视验收问题？

### 3.2 第二步：锁定用户当前验收现场

优先确认：

- 用户当前打开的是哪份工程；
- 用户当前是看根仓库还是某个 worktree；
- 用户当前到底在哪个路径和分支上验收。

### 3.3 第三步：核对线程真实 worktree 现场

固定核对：

- 当前 `cwd`
- 当前 `git branch --show-current`
- 当前 `git rev-parse --show-toplevel`
- `git worktree list`

### 3.4 第四步：再读 UI 激活工作区与状态投影

只在前面三步完成后，再补读：

- `.codex-global-state.json`
- `state_5.sqlite`
- `session_index.jsonl`
- 线程路由映射

### 3.5 第五步：给结论时必须分层

当冲突存在时，回答不能只给一句“你现在在 X”。

必须拆成：

- 根仓库现场是什么；
- 当前线程 worktree 现场是什么；
- 用户当前验收现场是什么；
- 当前主线应跟哪一层；
- 最小下一步是什么。

## 4. 只读核查顺序

### 4.1 最小核查顺序

1. `Get-Location`
2. `git branch --show-current`
3. `git rev-parse --show-toplevel`
4. `git worktree list`
5. 用户当前打开工程 / 当前可见视图
6. `.codex-global-state.json`
7. `state_5.sqlite`
8. `session_index.jsonl`
9. 线程路由映射与线程记忆路径

### 4.2 为什么不能反过来

如果先看 UI、先看状态库、先看记忆，很容易得到“线程看起来属于某处”的投影结论；  
但当前真实执行现场和用户当前验收现场，可能根本不是那个位置。

## 5. 自动纠偏边界

### 5.1 可以自动修的

- `threads.cwd`
  - 仅限有显式线程路由映射，且目标目录真实存在；
- `electron-saved-workspace-roots`
  - 仅限补齐，不做危险删除。

### 5.2 只能谨慎修的

- `threads.git_branch`
  - 必须满足：
    1. 线程真实 worktree 已确定；
    2. 当前真实分支可唯一确认；
    3. 有状态库备份；
    4. 本轮明确不是临时线程；
    5. 不会覆盖用户当前验收现场。
- `active-workspace-roots`
  - 只能在协议已经明确、回归样本已经通过时谨慎修。

### 5.3 默认绝不能动的

- 聊天原文；
- `sessions/*.jsonl`；
- `archived_sessions/*.jsonl`；
- `thread_name`；
- `title`；
- `rollout_path`；
- `WindowsApps` 下的 Codex 本体。

## 6. 完成态分级协议

### 6.1 L0：存在态

定义：

- 某内容在某处存在；
- 可能是代码、资源、文档或快照。

不等于：

- Git 已记录；
- 当前线程可见；
- 用户可验收。

### 6.2 L1：Git 记录态

定义：

- 内容已被某个分支或提交记录下来；
- 可追溯、可比较、可回退。

不等于：

- 当前线程现场已经对上；
- 用户当前打开工程可见。

### 6.3 L2：线程现场态

定义：

- 当前线程所在目录、worktree、分支、必要文件视图都已对齐；
- 当前线程真的能在自己应该工作的现场继续推进。

不等于：

- 用户当前打开工程已经可见。

### 6.4 L3：用户验收态

定义：

- 用户当前打开的工程和视图里，结果已经可见、可点、可验收。

这是 Unity 可视任务的最低“可交付”标准。

### 6.5 L4：文档同步态

定义：

- 子工作区、父工作区、线程记忆都已更新；
- 文档里明确写清来源分支、同步状态、恢复点与风险边界。

### 6.6 L5：最终闭环态

定义：

- 用户验收成立；
- 文档同步完成；
- 回滚口径明确；
- 当前主线已能稳定交接；
- 不再依赖单次偶然状态维持。

### 6.7 禁止混说

后续禁止再把：

- “存在态”
- “Git 记录态”
- “线程现场态”
- “用户验收态”
- “文档同步态”

混成一句“做完了”。

## 7. 默认回答模板

### 7.1 当用户问“我现在到底在哪”

默认按这个顺序回答：

1. **先现场**：你当前打开的是哪份工程 / 我当前真实 `cwd` 是什么；
2. **再主线**：当前主线绑定哪个 worktree / 分支；
3. **再分层**：根仓库、线程现场、UI 激活根分别是什么；
4. **再下一步**：现在应该继续哪里，不该继续哪里；
5. **最后才讲规则**：为什么以后要这么判断。

### 7.2 当用户问“是不是已经做完了”

默认按完成态分级回答：

1. 目前成立到哪一层；
2. 还没成立哪一层；
3. 当前最大缺口是什么；
4. 下一步最小动作是什么。

### 7.3 当用户已经混乱或焦虑

默认不要先讲抽象治理原则。

先给：

- 当前真实现场；
- 当前该信哪一层；
- 当前主线有没有恢复；
- 当前最小下一步。

---

## 8. 当前结论

从 2026-03-12 起，`Sunset` 中任何关于线程 / worktree / UI / Unity 验收的判断，都不应再只围绕单一状态字段或单一 Git 现场展开。

必须先按本协议明确：

- 谁是一等真相；
- 谁只是投影；
- 当前完成到哪一层；
- 回答用户时先讲什么、后讲什么。
