# Codex现场判定与完成态协议

**日期**: 2026-03-12  
**文档定位**: 用于统一回答“现在该信哪个现场”“哪些线程属于 Sunset 内部治理对象”“什么才算做完了”“联合迁移目前到底能做到哪一层”。

---

## 1. 现场定义

### 1.1 根仓库现场

指项目根目录仓库的真实 Git 现场，例如：

- `D:\Unity\Unity_learning\Sunset`
- `git branch --show-current`
- `git rev-parse HEAD`

它适用于：

- 项目治理线程；
- 根仓库轻任务线程；
- 绑定 `main` 的治理、总览、规则、记忆同步工作。

### 1.2 线程 worktree 现场

指某条长期功能线程对应的真实物理 worktree 现场，例如：

- `D:\Unity\Unity_learning\Sunset_worktrees\NPC`
- `D:\Unity\Unity_learning\Sunset_worktrees\farm-10.2.2-patch002`

它适用于：

- 长线功能线程；
- 独立 worktree 功能线程；
- 任何“这条线程实际应该在哪个目录和分支继续工作”的判断。

### 1.3 UI 激活工作区现场

指客户端当前激活工作区根与相关前端状态，例如：

- `.codex-global-state.json` 中的 `active-workspace-roots`
- `electron-saved-workspace-roots`
- `electron-workspace-root-labels`

它适用于：

- 解释客户端为什么显示成这样；
- 判断为什么看起来像“回到了根仓库”；
- 评估 UI / workspace 承接是否有漂移。

它**不是**当前真实 Git 现场的最高真相源。  
它是**带时间戳的全局 UI 状态样本**，会随着当前激活工作区或当前查看线程变化，不能被当成永久静态事实。

像 `D:\迅雷下载\开始` 这样的路径，也只能按“本轮是否命中实时值”来解释：

- 命中实时值时，按 `系统全局监督线程` 现场处理；
- 未命中实时值时，只保留为历史样本或可重现实例；
- 不能被写成永久固定的 `Sunset` 当前事实。

### 1.4 用户验收现场

指用户当前真实打开、真实观察、真实点击和真实验收的那份工程与那块视图。

在 Unity 项目里，它至少包括：

- 用户当前打开的工程路径；
- 当前打开的场景 / 资源 / Inspector；
- 用户眼前是否真的能看见结果；
- 是否真的能点、能交互、能验收。

它是可视化任务的一等公民。

## 2. 优先级协议

### 2.1 默认优先级

当任务涉及真实交付、线程归属或用户混乱时，默认优先级如下：

1. **用户验收现场**
2. **线程 worktree 现场**
3. **根仓库现场**
4. **UI 激活工作区现场**
5. **状态库 / 线程记忆 / 历史标题等投影线索**

### 2.2 为什么这样排

- 用户验收现场优先，因为“用户眼前是否成立”决定当前是否真的交付；
- 线程 worktree 现场次之，因为它决定当前线程应该在哪工作；
- 根仓库现场再次之，因为它是治理主路，但不必然等于当前线程执行现场；
- UI 激活工作区现场更像“客户端显示现场”，重要但不能压过真实 Git 和用户验收；
- 状态库与记忆更多是证据与投影，不应替代当前真相。

### 2.3 特殊说明

如果当前任务明确是治理 / 审计 / 规则任务，而不是功能线程实现，则：

- 根仓库现场与治理线程现场通常合并为第一执行现场；
- 但一旦用户反馈“我眼前看到的不是你说的状态”，仍应把用户验收现场提升回第一优先级。

## 3. 线程类型模型

### 3.1 总原则

后续不再先看某个字段“像不像 Sunset”，而是先回答：

1. 这条线程属于哪一类线程；
2. 它是否属于 `Sunset` 内部治理对象；
3. 它当前默认应该跟哪一层现场；
4. 只有在这个前提下，某个字段才可能被判定为冲突。

### 3.2 六类线程定义

| 线程类型 | 是否属于 Sunset 内部治理对象 | 默认主现场 | Git / worktree 策略 | memory / workspace 策略 | 是否纳入 `active-workspace-roots` 冲突判断 |
|---|---|---|---|---|---|
| 系统全局监督线程 | 否 | 外部全局工作区 | 不默认绑定 Sunset 仓库；可读取 Sunset 证据，但不是 Sunset 内部功能现场 | 记忆留在外部全局线程自己的根目录 | 否，除非它真的改坏了 Sunset 内部线程承接 |
| 项目治理线程 | 是 | Sunset 根仓库 | 默认留在根仓库 `main`；不默认申请独立 worktree | 线程记忆与治理文档都落在 Sunset 根仓库体系 | 是，期望命中根仓库现场 |
| 根仓库轻任务线程 | 是 | Sunset 根仓库 | 默认不建独立 worktree；若进入真实实现仍必须先建 `codex/` 分支 | 线程记忆留在项目根目录；工作区记忆按子后父同步 | 是，期望命中根仓库现场 |
| 长线功能线程 | 是 | 先在根仓库做计划，获批后迁入独立现场 | 触发 branch / worktree 申请；未获批前不直接污染 `main` | 先记录预期分支 / worktree，获批后更新承接口径 | 是，但要在获批目标明确后再判断 |
| 独立 worktree 功能线程 | 是 | 对应功能 worktree | 已获批独立 `codex/` 分支与 worktree；后续默认在该现场继续 | 线程记忆目标路径仍归项目根规则管理；历史观测路径可暂存为事实现场 | 是，但必须结合“当前正在看的线程是谁”一起判断 |
| 临时只读排查线程 | 视排查对象而定 | 以当前排查目标为准 | 默认不建独立 worktree；不擅自进入修改态 | 只记录证据、判断与恢复点，不抢占长期线程现场 | 默认否；只有明确负责 Sunset 内部排查时才纳入 |

### 3.3 这条修正对后续判断的硬约束

- 系统全局监督线程不属于 `Sunset` 内部执行线程；
- `D:\迅雷下载\开始` 这类外部全局工作区根，不能直接被当成 `Sunset` bug 样本；
- `active-workspace-roots` 的判断必须先过线程类型过滤，再讨论是否属于 `Sunset` 内部冲突。

## 4. Sunset 新线程生命周期规范

### 4.1 启动先分类

任何新的 `Sunset` 线程启动时，先回答三件事：

1. 我属于哪一类线程；
2. 我当前是治理、轻任务，还是长线功能；
3. 我需不需要独立施工现场。

在这一步完成前，不默认直接读一堆代码、切分支或建 worktree。

### 4.2 何时触发 branch / worktree 申请

只要满足以下任意多项，就应触发独立施工现场申请：

- 预计持续较久；
- 会改多个文件或多个系统；
- 会改资源 / UI / 场景 / Prefab；
- 会与其他功能线并行；
- 不适合继续污染根仓库 `main`；
- 需要单独测试、回滚或审计。

### 4.3 用户审批点

涉及 branch / worktree 的动作，必须保留用户审批点。  
后续统一口径如下：

- **需要用户审批**：建分支、建 worktree、改变长期线程默认施工现场；
- **批准后线程自动完成**：生成建议分支名、创建 worktree、更新治理文档与记忆、产出迁移后基线；
- **当前仍属受限能力**：不经用户确认就改高风险状态库字段、假装 UI 一定会无重启刷新。

### 4.4 批准后的联合迁移最小闭环

批准后，不再把迁移理解成“只建 branch + worktree”。  
最小联合迁移闭环至少包含：

1. 创建新的 `codex/` 分支；
2. 创建新的 worktree 目录；
3. 锚定后续执行 `cwd`；
4. 明确子工作区 / 父工作区 / 线程记忆的承接位置；
5. 更新线程路由映射；
6. 写清 UI / workspace 承接策略；
7. 写清根仓库与 worktree 的职责边界；
8. 生成迁移完成后的最新基线。

### 4.5 迁移后的稳定验收

迁移完成后，至少验证：

- 当前目录；
- 当前分支；
- 当前 worktree 根；
- 线程记忆写入位置；
- provider 切换后是否仍在正确现场；
- 重启后是否仍在正确现场；
- 重新打开旧线程时是否还能回对现场。

### 4.6 当前实现边界

上面这套是**目标工作流**，不是“Codex 当前原生已经自动完成的一切”。

当前已经能做的是：

- 用文档与脚本先把分类、申请口径、迁移清单和基线生成固定下来；
- 在用户批准后由线程完成 Git / worktree / 路由 / 记忆层的白名单动作；
- 用只读基线持续解释“现在成立到哪一层”。

当前还不能夸大成已经原生支持的是：

- 新线程自动识别全部类型；
- 不经审批自动申请分支 / worktree；
- 迁移后 UI / workspace 必定无重启刷新。

## 5. 冲突时的决策树

### 5.1 第一步：先判线程类型

必须先回答：

- 这是系统全局监督线程，还是 `Sunset` 内部线程；
- 这是治理线程、轻任务线程，还是功能线程；
- 这是只读排查，还是要落地产物。

如果答案是“系统全局监督线程”，则不能直接把它的 `cwd` 或 `active-workspace-roots` 当作 `Sunset` bug。

### 5.2 第二步：锁定用户当前验收现场

优先确认：

- 用户当前打开的是哪份工程；
- 用户当前是看根仓库还是某个 worktree；
- 用户当前到底在哪个路径和分支上验收。

### 5.3 第三步：核对线程真实 worktree 现场

固定核对：

- 当前 `cwd`
- 当前 `git branch --show-current`
- 当前 `git rev-parse --show-toplevel`
- `git worktree list`

### 5.4 第四步：再读 UI 激活工作区与状态投影

只在前面三步完成后，再补读：

- `.codex-global-state.json`
- `state_5.sqlite`
- `session_index.jsonl`
- 线程路由映射

### 5.5 第五步：给结论时必须分层

当冲突存在时，回答不能只给一句“你现在在 X”。

必须拆成：

- 根仓库现场是什么；
- 当前线程 worktree 现场是什么；
- 用户当前验收现场是什么；
- UI 激活工作区现场是什么；
- 当前主线应跟哪一层；
- 最小下一步是什么。

## 6. 只读核查顺序

### 6.1 最小核查顺序

1. `Get-Location`
2. `git branch --show-current`
3. `git rev-parse --show-toplevel`
4. `git worktree list`
5. 用户当前打开工程 / 当前可见视图
6. `.codex-global-state.json`
7. `state_5.sqlite`
8. `session_index.jsonl`
9. 线程路由映射与线程记忆路径

### 6.2 本轮新增硬约束

- `active-workspace-roots` 先看“当前是不是哪类线程的正常现场”，再看它是否构成 `Sunset` 内部冲突；
- 外部系统全局监督现场只保留为历史样本和解释材料，不再直接上提为 `Sunset` 当前实时 bug；
- 每次继续执行，都以**最新直读现场**为准，不沿用旧截图、旧样本或旧聊天瞬时值。

## 7. UI / workspace / 线程承接能力边界

### 7.1 A 层：今天就能安全做到的

- 把线程类型、迁移口径、完成态和回滚边界写进协议；
- 用只读脚本生成最新执行基线；
- 在用户批准后，把 Git / worktree / 路由 / 记忆承接写成显式步骤；
- 让后续命令明确锚定到目标 `cwd` 和目标 worktree。

### 7.2 B 层：今天可以半自动做到，但有前提和风险的

- 某些 UI / workspace 根变化可以通过重新打开线程、切换视图后反映出来；
- 某些承接信息可以靠最新基线再次核对，而不是靠 UI 直觉判断；
- 某些联合迁移可以做到 Git 与文档层闭环，但 UI 层仍需用户二次确认。

### 7.3 C 层：今天做不到，仍属客户端黑箱 / 受限能力

- 不能证明迁入新 worktree 后一定无需重启就稳定刷新；
- 不能证明 provider 切换后一定无漂移；
- 不能把客户端黑箱字段写成“已原生自动修好”；
- 不能在没有备份与验证前，粗暴改高风险状态层字段。

### 7.4 当前 fallback

当 UI / workspace 承接不够稳定时，当前默认 fallback 是：

1. 先信最新直读基线；
2. 再信真实 `cwd / branch / worktree`；
3. 必要时重新打开线程、切换视图或重启 `Codex`；
4. 回答用户时明确说“这是当前 fallback，不是最终无感体验”。

## 8. 自动纠偏边界

### 8.1 可以自动修的

- `threads.cwd`
  - 仅限有显式线程路由映射，且目标目录真实存在；
- `electron-saved-workspace-roots`
  - 仅限补齐，不做危险删除。

### 8.2 只能谨慎修的

- `threads.git_branch`
  - 必须满足：
    1. 线程真实 worktree 已确定；
    2. 当前真实分支可唯一确认；
    3. 有状态库备份；
    4. 本轮明确不是临时线程；
    5. 不会覆盖用户当前验收现场。
- `active-workspace-roots`
  - 必须先完成线程类型判定；
  - 必须先确认当前值到底是正常治理现场、正常系统全局监督现场，还是 Sunset 内部线程的错误现场；
  - 只能在协议已经明确、回归样本已经通过时谨慎修。

### 8.3 默认绝不能动的

- 聊天原文；
- `sessions/*.jsonl`；
- `archived_sessions/*.jsonl`；
- `thread_name`；
- `title`；
- `rollout_path`；
- `WindowsApps` 下的 Codex 本体。

## 9. 完成态分级协议

### 9.1 L0：存在态

定义：

- 某内容在某处存在；
- 可能是代码、资源、文档或快照。

不等于：

- Git 已记录；
- 当前线程可见；
- 用户可验收。

### 9.2 L1：Git 记录态 / 状态投影未收口

定义：

- 内容已被某个分支或提交记录下来；
- 可追溯、可比较、可回退；
- 但线程元数据、状态投影或承接口径仍未收口。

不等于：

- 当前线程现场已经完全对上；
- 用户当前打开工程可见。

### 9.3 L2：线程现场态

定义：

- 当前线程所在目录、worktree、真实分支与必要文件视图已对齐；
- 当前线程真的能在自己应该工作的现场继续推进。

不等于：

- 用户当前打开工程已经可见。

### 9.4 L3：用户验收态

定义：

- 用户当前打开的工程和视图里，结果已经可见、可点、可验收。

这是 Unity 可视任务的最低“可交付”标准。

### 9.5 L4：文档同步态

定义：

- 子工作区、父工作区、线程记忆都已更新；
- 文档里明确写清来源分支、同步状态、恢复点与风险边界。

### 9.6 L5：最终闭环态

定义：

- 用户验收成立；
- 文档同步完成；
- 回滚口径明确；
- 当前主线已能稳定交接；
- 不再依赖单次偶然状态维持。

### 9.7 禁止混说

后续禁止再把：

- “存在态”
- “Git 记录态 / 状态投影未收口”
- “线程现场态”
- “用户验收态”
- “文档同步态”

混成一句“做完了”。

## 10. 默认回答模板

### 10.1 当用户问“我现在到底在哪”

默认按这个顺序回答：

1. **先分类**：这条线程属于哪一类线程；
2. **先现场**：用户当前打开的是哪份工程 / 我当前真实 `cwd` 是什么；
3. **再主线**：当前主线绑定哪个 worktree / 分支；
4. **再分层**：根仓库、线程现场、UI 激活根分别是什么；
5. **再下一步**：现在应该继续哪里，不该继续哪里；
6. **最后才讲规则**：为什么以后要这么判断。

### 10.2 当用户问“是不是已经做完了”

默认按完成态分级回答：

1. 目前成立到哪一层；
2. 还没成立哪一层；
3. 当前最大缺口是什么；
4. 下一步最小动作是什么。

### 10.3 当用户已经混乱或焦虑

默认不要先讲抽象治理原则。

先给：

- 当前真实现场；
- 当前该信哪一层；
- 当前主线有没有恢复；
- 当前最小下一步。

---

## 11. 当前结论

从 2026-03-12 起，`Sunset` 中任何关于线程 / worktree / UI / Unity 验收的判断，都不应再只围绕单一状态字段或单一 Git 现场展开。

必须先按本协议明确：

- 这条线程属于哪一类线程；
- 谁是一等真相；
- 谁只是投影；
- 当前完成到哪一层；
- 联合迁移今天能做到什么、做不到什么；
- 回答用户时先讲什么、后讲什么。
