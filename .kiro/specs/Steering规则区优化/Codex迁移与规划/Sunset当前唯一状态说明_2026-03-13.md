## 4. MCP 当前真实定性
- 当前项目安装的是 `com.coplaydev.unity-mcp`，并且 Unity 侧日志明确出现 `Successfully registered with Claude Code using HTTP transport.`。
- 这条提示说明：Unity 包内置配置器当前成功登记的是 `Claude Code` 的 HTTP MCP 客户端，不是对 `Codex` 会话成功回执的证明。
- 当前 `C:\Users\aTo\.codex\config.toml` 仍并存：
  - `[mcp_servers.unityMCP]`：HTTP 入口
  - `[mcp_servers.mcp-unity]`：旧 `stdio` 入口
- 当前 `Codex` 侧仍持续出现 `Connection failed: Unknown error`，说明当前 Sunset 现场仍处于“Unity 侧已向 Claude Code 注册成功，但 Codex 侧工具链未完成统一切换”的混用态。
- 当前治理结论固定为：
  - Sunset 默认客户端应统一到 `Codex`
  - 默认只维护 `unityMCP` 这条 HTTP 入口
  - `Claude Code` 注册只保留为明确切换客户端时的例外配置，不再作为 Sunset 默认口径

## 6. 当前默认开发规则
- 小改动、低风险、单线程、当天可验收的改动：
  - 允许直接在 `main` 开发
  - 提交时必须白名单
  - 可直接推送到 `origin/main`
- 中高风险、需要独立提交链、需要明显隔离 review 面的改动：
  - 先开 `codex/xxx` 分支
  - 默认仍在同一个主项目目录内开发，不默认开 worktree
- 只有以下场景才允许 worktree：
  - 高风险隔离
  - 故障修复
  - 特殊实验
- 线程启动前必须先读取本说明，再开始判断现场与默认流程。
