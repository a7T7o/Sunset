因此，整体不能再笼统表述为“接近可用闭环”或“整体到 L4”。

### 12.2 四层问题拆解

#### A. Git / worktree 真实层
- `NPC` 与 `farm-10.2.2-patch002` 的真实 worktree 都仍存在，真实分支也仍存在；
- 根仓库里出现 `fatal: ''<branch>'' is already used by worktree at ...`，反而证明这些功能分支已经被正确绑定到独立 worktree；
- 这一层当前不是“Git 丢了”，而是“客户端线程没有回到该去的 worktree 现场”。

#### B. 状态库投影层
- `state_5.sqlite` 中的 `cwd / git_branch / git_sha / rollout_path` 仍在影响线程恢复投影；
- `NPC` 当前最关键的投影漂移仍是：真实 worktree 已在 `codex/npc-generator-pipeline`，但状态库仍写 `git_branch = main`；
- `farm` 样本也呈现同类结构：`cwd` 已落到 worktree，但 `git_branch` 仍是 `main`；
- 这层属于“单线程最小修正候选”，但目前还不能粗暴批量改库。

#### C. UI / workspace / 线程承接层
- 用户现象表明：重启后线程会短暂回到正确项目文件夹，但点击线程后又掉回 `Sunset/main` 视角；
- 这说明 UI / workspace 承接不是完全失效，而是在线程打开时重新读取了错误或过时的投影来源；
- 当前最保守判断是：线程元数据旧值、状态库投影旧值、UI 激活工作区刷新机制三者并存失配。

#### D. 用户验收层
- 用户眼里的“恢复”标准不是脚本跑通，而是点开线程后：
  - 回到正确项目组；
  - 回到正确 worktree 语境；
  - 不再误回 `Sunset/main`；
  - 不再误触发根仓库切功能分支；
  - 不需要靠重启暂时恢复。
- 当前离这个标准仍有明显距离。

### 12.3 两个核心问题的统一回答

#### 问题 1：到哪一阶段才能验收到“重启/不重启都正常”
- 只读治理阶段：不能，只能解释问题；
- 状态投影修正准备阶段：不能，只能建立可执行修复入口；
- 单线程真实修复阶段：有机会让“单线程点击即正确”成立，但仍不能承诺所有线程、所有 provider、无需重启全恢复；
- 用户真实闭环验收阶段：只有当 `NPC` / `farm` / 治理线程样本都通过用户点击验证，且重启与不重启两条路径都成立，才允许对外宣称“线程正常了”。

#### 问题 2：是否存在“一跑即恢复 UI”的脚本式能力
- 当前不能这样承诺；
- 不是因为一定做不到，而是当前证据只支持“建立最小修复入口 + 单线程最小字段修正预案 + 用户点击验收矩阵”；
- 当前最接近这个目标的方案，是单线程受控修正状态投影字段后，再用用户点击验证矩阵做验收；
- 失败模式包括：
  - 状态字段改了，但 UI 仍缓存旧现场；
  - 重启后暂时对，不重启仍漂；
  - provider 切换后再次回落到根仓库语境。
