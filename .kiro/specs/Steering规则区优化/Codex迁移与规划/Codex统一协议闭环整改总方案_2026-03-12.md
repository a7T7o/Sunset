# Codex统一协议闭环整改总方案

**日期**: 2026-03-12  
**适用范围**: `Sunset` 项目中 `Codex` 的线程 / worktree / 用户验收 / 成果分层 / 文档治理闭环整改  
**文档定位**: 这不是一份新的 Git 规则，也不是新的客户端实现说明；它用于把当前整改主线从“状态修补工程”升级成“统一协议建设工程”。

---

## 1. 为什么旧问题定义不够

上一轮整改抓到的是真实问题，但只修到了中层。

已成立的中层判断包括：

- 问题不只是 Git，而是 `Codex` 客户端线程 / UI / worktree 承接未闭环；
- `state_5.sqlite`、`session_index.jsonl`、`.codex-global-state.json`、`active-workspace-roots` 确实影响线程承接；
- 审计表、路由映射、审计脚本、字段边界文档这些阶段性产物都是真实有效的。

但它们仍然不够，因为它们主要在回答：

- 哪个字段错了；
- 哪个线程半迁移；
- 哪个 worktree 没命中；
- 该不该自动纠偏。

它们还没有先回答更深的一层：

- 当根仓库、线程 worktree、UI 激活工作区、用户当前打开工程互相冲突时，到底谁才是一等真相；
- 对 Unity 这种强可视验收项目，“做完了”到底是 Git 有记录，还是用户当前眼前已经可见可点；
- 正式成果、治理文档、线程记忆、跨线程索引之间到底怎么分层、怎么映射、怎么收口。

所以旧问题定义只够支撑“状态投影层修补”，不够支撑“使用语义层闭环”。

## 3. 四份家访材料分别推进了什么理解

### 3.1 `NPC.md`

它把问题从“根仓库看起来是什么”推进到“当前线程真实现场到底是什么”。

它提供的核心推动是：

- 线程级现场必须优先于根仓库默认视角；
- 回答“我现在在哪”时必须分层；
- 根仓库与 worktree 冲突时必须先停下来重锚；
- memory 的连续性路径必须跟随 active worktree，而不是沿用根仓库想当然路径。

它推进出来的是：

**线程真实现场判定协议。**

### 3.2 `spring-day1.md`

它把问题从“功能存在”推进到“用户当前打开工程是否真的可见、可点、可验收”。

它提供的核心推动是：

- Unity 可视任务的交付标准不能只看 Git 或线程；
- 真正的完成必须落在用户当前打开并验收的那份工程上；
- “某个分支里有内容”和“用户眼前已经能验收”必须明确分开。

它推进出来的是：

**用户验收现场判定协议。**

### 3.3 `Skills和mcp.md`

它把问题从单点 Git 治理推进到 `Skills / MCP / 文档 / memory / 配置 / 主线恢复` 的一体治理。

它提供的核心推动是：

- 工具价值在于降低切换成本，而不是堆数量；
- 文档、配置、环境、Git 不是分离问题；
- 不是每个小动作都即时进 `main`，而是形成稳定闭环就尽快主线化；
- 协作线程的职责应该是帮助收口，而不是增加心理负担。

它推进出来的是：

**主线恢复与短周期闭环协议。**

### 3.4 `项目文档总览.md`

它把问题从“成果是不是散了”推进到“成果分层与总索引是否建立起来”。

它提供的核心推动是：

- 不是成果没了，而是四层没有明确分工；
- 正式成果层、治理文档层、线程记忆层、跨线程总索引层当前互相越权；
- 真正缺的是总索引，不是先大迁移；
- 线程记忆不该继续承担过量索引职责。

它推进出来的是：

**成果分层与总索引协议。**

### 3.5 `对话丢失修复/memory_0.md`

它把问题从“看起来线程丢了”推进到“状态投影、原始会话、缓存、数据库、守护式兼容层必须分层处理”。

它提供的核心推动是：

- 先修状态投影，不动聊天原文；
- `run-once` 只能补救，守护式方案才是长期策略；
- 修库前必须先备份，且回滚路径要明确；
- 要区分根因层与表现层，不要把显示异常直接误判成数据消失。

它推进出来的是：

**兼容层建设与可回滚边界协议。**

## 4. 重新定义后的总问题：五大缺口

### 缺口 1：线程真实现场判定缺口

当以下对象冲突时，当前没有统一优先级协议：

- 根仓库现场；
- 当前线程 worktree 现场；
- UI 激活工作区现场；
- 线程状态库记录；
- 线程 memory 路径。

### 缺口 2：用户验收现场判定缺口

尤其在 Unity 项目里，当前缺少明确协议回答：

- 用户此刻打开的是哪份工程；
- 这份工程是不是本轮应该出现结果的那份；
- “做完了”到底要在谁眼前成立。

### 缺口 3：完成态分级缺口

当前缺少统一定义去区分：

- 存在态；
- Git 记录态；
- 线程现场态；
- 用户验收态；
- 文档同步态；
- 最终闭环态。

### 缺口 4：成果分层与总索引缺口

当前缺少明确边界去区分：

- 正式成果层；
- 治理文档层；
- 线程记忆层；
- 跨线程总索引层。

### 缺口 5：协作响应协议缺口

当前缺少一套默认响应顺序，导致用户越乱时，系统越容易继续讲抽象规则，而不是先帮用户定锚、拆层、给最小下一步。

## 6. 从“状态修补”升级到“统一协议建设”的总路线

### 阶段 A：先建协议

先完成三类文档建设：

1. 统一现场判定与完成态协议；
2. 成果分层与总索引模型；
3. 总体整改总方案与风险边界。

目标不是再多写文档，而是先把：

- 谁是一等真相；
- 什么叫做完；
- 哪层负责什么；
- 冲突时怎么回答；

这几件事写死。

### 阶段 A2：把协议推进到线程类型与生命周期层

在统一现场协议之上，继续补三件事：

1. 明确六类线程类型模型；
2. 明确 `Sunset` 新线程的生命周期规范；
3. 明确联合迁移今天能自动做到什么、只能半自动做到什么、还做不到什么。

这一层完成前，不能再把：

- 系统全局监督线程；
- 项目治理线程；
- 长线功能线程；
- 独立 worktree 功能线程

混成同一种线程来讨论。

### 阶段 B：再做最小技术落地

只有协议明确后，才进入：

- 激活工作区根的最小纠偏；
- 线程默认 `cwd` / `git_branch` 的可控修正；
- 守护式兼容层最小实现。

### 阶段 C：最后做真实闭环验证

验证必须从“脚本成功”升级为“用户验收现场成功”，至少覆盖：

- 治理线程；
- NPC 线程；
- 农田线程；
- Unity 可视验收线程；
- 文档治理线程。

## 7. 风险边界

### 7.1 当前明确高风险的对象

- `state_5.sqlite`；
- `session_index.jsonl`；
- `.codex-global-state.json`；
- `active-workspace-roots`；
- 任何会被守护器自动改写的线程状态字段。

### 7.2 当前明确不能越界的对象

- 聊天原文；
- `sessions/*.jsonl`；
- `archived_sessions/*.jsonl`；
- `thread_name` / `title` 这类历史语义字段；
- `WindowsApps` 下的 Codex 本体。

### 7.3 当前不能再犯的错误

- 把阶段性产物包装成总体闭环；
- 把用户的判断负担重新丢回给用户；
- 只讲规则，不先定锚；
- 只修状态，不建协议。

## 8. 当前明确不做的事项

本轮总方案明确不做以下事情：

- 不直接修改 `Codex` 安装本体；
- 不提前大规模改数据库或启动自动守护器；
- 不重构现有全部目录结构；
- 不删除历史线程记忆、外部交叉材料或治理证据；
- 不提前宣称“已经闭环修好”。

## 9. 回滚边界

### 9.1 文档层回滚

- 本轮新增的统一协议文档与任务补记，均可通过 Git 白名单提交回退；
- 旧任务定义与旧记忆不删除，保留为历史证据。

### 9.2 状态层回滚

进入任何自动纠偏前，必须先：

- 备份 `state_5.sqlite`；
- 记录本轮开始基线；
- 记录本轮协议对应的目标线程样本；
- 准备回滚命令或人工恢复步骤。

### 9.3 兼容层回滚

任何守护器或桥接脚本上线前，必须保证：

- 可禁用；
- 可回退；
- 可重跑审计验证；
- 可恢复到只读模式。

## 10. 验收标准

### 10.1 阶段性通过，不等于总体闭环

阶段性通过的最低标准是：

- 五大缺口被正式定义；
- 协议文档落盘；
- 旧任务口径被重构；
- 后续技术动作有明确先后顺序。

### 10.2 最终闭环通过的最低标准

只有同时满足下面这些条件，才允许宣称“统一协议闭环整改通过”：

- 线程真实现场判定协议可执行；
- 用户验收现场判定协议可执行；
- 完成态分级已成为默认口径；
- 成果分层与总索引已有实际落点；
- 三类线程与至少一类 Unity 验收线程完成真实回归；
- 回滚口径与对外说明已经能稳定复用。

## 11. 当前结论

当前最准确的口径应是：

- 前一轮整改：阶段性产物成立，但主要修到了状态投影层；
- 当前这一轮：统一协议已经继续推进到“线程类型分流 + 新线程生命周期规范 + 联合迁移能力边界”这一层；
- 当前 `active-workspace-roots` 的旧冲突样本已被重定义为“动态全局 UI 状态 + 线程类型前置过滤”问题；
- 当前 Sunset 内部最该先动的真实冲突，已经收敛为 **NPC 线程状态库 `git_branch` 旧值**，其次才是 NPC 线程记忆归位策略；
- 所以现在仍不能宣称“已经闭环修好”；
- 但可以明确宣称：**整改主线已经从状态修补升级为统一协议建设。**
