# Unity MCP 迁移试装方案（2026-03-10）

> 适用现场：
>
> - Unity 版本：`6000.0.62f1`
> - 项目：`D:/Unity/Unity_learning/Sunset`
> - 当前状态：旧 `mcp-unity` 可作为过渡方案继续使用，但关键读操作与验证链路已出现不稳定

---

## 1. 本方案要解决什么

这不是“换一个看起来更高级的 MCP”。

本方案真正要解决的是：

1. 让 Unity MCP 至少稳定支撑：
   - 场景层级读取
   - console 错误读取
   - EditMode tests
2. 保持当前业务工作继续推进，不因为基础设施升级把节奏打断。
3. 给 Sunset 留出后续扩展空间：如果未来要做项目专用 MCP Tool，不必推倒重来。

---

## 2. 当前结论

### 2.1 当前 MCP 的定位

当前环境里的 `mcp-unity` 继续可用，但只适合：

- 过渡期
- 少量基础调用
- 帮当前回合不被完全卡死

它不适合作为长期主力，原因是我本轮现场已经遇到：

- `timeout`
- `connection closed`
- 关键资源可枚举，但读取不稳定

这会直接伤害 Sunset 最重要的验证链路。

### 2.2 迁移优先级

基于最新 GitHub 官方仓库信息与当前 Unity 版本，推荐顺序如下：

1. **第一试装候选：`CoplayDev/unity-mcp`**
2. **第二试装候选：`notargs/UnityNaturalMCP`**
3. **第二阶段扩展底座：`IvanMurzak/Unity-MCP`**
4. **观察项：`nurture-tech/unity-mcp-server`**

---

## 3. 候选选择理由

## 3.1 `CoplayDev/unity-mcp`

### 为什么先试它

它最贴近“先求稳定”的目标。

根据其 GitHub README，可直接确认：

- 支持 `Unity 2021.3 LTS+`，因此覆盖当前 `6000.0.62f1`
- 公开强调：
  - stale connection detection
  - tool state persistence across reloads
  - domain reload resilience
  - IPv6/IPv4 loopback fixes
  - `batch_execute`
- 工具覆盖包含：
  - `read_console`
  - `run_tests`
  - `manage_scene`
  - `manage_prefabs`
  - `manage_components`
  - `manage_scriptable_object`

这些点和 Sunset 当前痛点高度吻合。

### 适合 Sunset 的原因

1. 你当前最需要的是“稳定读”和“稳定验证”
2. 它支持多 Unity 实例路由
3. 它已经给出 Windows 下的 `uvx`/HTTP 配置范式，迁移路径清晰

### 关键信息来源

- GitHub: [CoplayDev/unity-mcp](https://github.com/CoplayDev/unity-mcp)

---

## 3.2 `notargs/UnityNaturalMCP`

### 为什么升到第二优先

这次和之前相比，最大的变化是：**你的 Unity 版本已经明确是 `6000.0.62f1`**。

而这个候选的 README 明确写了：

- 要求 `Unity 6000.0 or later`
- 支持通过 Unity Project Settings 配置端口
- 支持 Streamable HTTP
- 支持在 Unity C# 中定义自定义 MCP tool

这使它从“前瞻候选”升级为“值得并行比较的版本原生候选”。

### 现在不放第一的原因

1. 它仍然明确标注为 `preview stage`
2. 在 Windows + Claude Code 场景下，README 还特别提到：
   - 需要 WSL2
   - 可能需要 mirrored networking
   - 绑定 `localhost` 可能失败

这对 Sunset 当前“先稳住日常工作流”的目标来说，风险高于 Coplay。

### 适合你后续关注的点

1. 版本匹配度高
2. 自定义 C# MCP tool 路径清晰
3. 如果后面你要做 Unity 6000 深度定制，它很值得继续看

### 关键信息来源

- GitHub: [notargs/UnityNaturalMCP](https://github.com/notargs/UnityNaturalMCP)

---

## 3.3 `IvanMurzak/Unity-MCP`

### 为什么不作为第一枪

它很强，但它更像“扩展底座”，不是“先把稳定性救回来”的最小替代。

根据其 README，可直接确认：

- 50+ built-in tools
- 支持 custom MCP Tool / Resource / Prompt
- 支持 Editor 与 Runtime
- 支持本地和 HTTP

### 它真正适合 Sunset 的阶段

当你后面想做这些事情时，它会非常合适：

1. 做 Sunset 项目专用 MCP Tool
2. 把项目工作流嵌进 Unity 编辑器 / 游戏内
3. 深度扩展自动化能力，而不是只解决“连接稳不稳”

### 当前不放第一的原因

1. 体系更重
2. 引入后治理成本更高
3. 它更像“平台建设”，而不是“快速修复当前验证链路”

### Sunset 额外注意点

它 README 明确要求 **项目路径不能包含空格**。  
当前 `D:/Unity/Unity_learning/Sunset` 不含空格，所以这条对你是满足的。

### 关键信息来源

- GitHub: [IvanMurzak/Unity-MCP](https://github.com/IvanMurzak/Unity-MCP)

---

## 3.4 `nurture-tech/unity-mcp-server`

### 为什么暂时只观察

这个方案有几个很吸引人的点：

1. 视觉能力强
2. 有 `save_scene`、`open_scene`、`test_active_scene`
3. 启动方式现代，适合更强自动化

但它 README 也明确强调：

- 最好通过 agent 启动 Unity
- 不建议从 Unity Hub 启动再接
- 不要用 `-logFile`

这意味着它对日常工作方式的改动更大。

对于 Sunset 当前阶段，这不是最优先的风险承担方向。

### 关键信息来源

- GitHub: [nurture-tech/unity-mcp-server](https://github.com/nurture-tech/unity-mcp-server)

## 4. 迁移原则

### 原则 1：不阻塞当前开发

在新的 MCP 没验证稳之前：

- 不移除旧 MCP
- 不把当前工作流全部押到新 MCP 上

### 原则 2：并行试装，不一刀切替换

新的 Unity MCP 先以“试装候选”身份进入，不立刻成为唯一主力。

### 原则 3：只看真实验证结果，不看功能表幻想

最终是否替换，依据 Sunset 现场验证，不依据 GitHub Stars 或 README 丰富程度。

### 原则 4：优先稳定读，不优先华丽功能

只要这些先稳住，就已经值回迁移成本：

1. 场景层级读取
2. console 读取
3. EditMode tests
4. 重编译后不易断

## 5. 试装方案总览

## Phase 0：保底不动主线

当前旧 MCP 继续保留，保证本轮业务工作不被升级动作打断。

## Phase 1：优先试装 `CoplayDev/unity-mcp`

目标：验证它能不能稳定替代“读场景 / 读日志 / 跑测试”。

## Phase 2：并行比较 `UnityNaturalMCP`

目标：验证 Unity 6000 原生候选，在你的版本上是否有更好连接稳定性或可扩展性。

## Phase 3：评估是否引入 `IvanMurzak/Unity-MCP`

目标：如果未来进入“Sunset 专用工具平台化”，再把它作为扩展底座。

---

## 6. Phase 1：`CoplayDev/unity-mcp` 试装步骤

### 6.1 安装方式

按其 README，Unity 包安装方式为：

`Window > Package Manager > Add package from git URL...`

推荐先试：

`https://github.com/CoplayDev/unity-mcp.git?path=/MCPForUnity#main`

### 6.2 最小配置验证

在 Unity 内：

1. 打开 `Window > MCP for Unity`
2. 启动服务
3. 确认本地 HTTP 服务启动
4. 仅在测试窗口里接入，不替换旧 MCP 配置

### 6.3 验证项目

只验证 Sunset 最核心 5 项：

1. 读取场景层级
2. 读取 console errors
3. 运行 EditMode tests
4. 脚本变更后 recompile 再调用是否稳定
5. 连续调用 5 轮是否出现断连

### 6.4 通过标准

满足以下条件才算过：

- 3 次以上连续场景读取成功
- 3 次以上 console 读取成功
- EditMode tests 至少连续成功触发 2 次
- Unity 编译后不会立即失联
- 不出现当前这种“枚举有，读取挂”的半可用状态

---

## 7. Phase 2：`UnityNaturalMCP` 比较步骤

### 7.1 目的

不是立刻替换，而是回答一个问题：

> 在 Unity `6000.0.62f1` 这个现场里，版本原生候选是否比 Coplay 更稳？

### 7.2 安装方式

按其 README，可通过 UPM 指向：

`https://github.com/notargs/UnityNaturalMCP.git?path=/UnityNaturalMCPServer`

### 7.3 核心风险

要特别观察：

1. Windows 环境网络配置是否复杂化
2. 是否需要 WSL2 / mirrored networking
3. `localhost` 是否稳定
4. 预览阶段是否会出现不可预期行为

### 7.4 比较项目

与 Coplay 用同一组 Sunset 验证项比较：

1. 场景层级读取稳定性
2. console 稳定性
3. tests 稳定性
4. reload 后连接恢复
5. 自定义 tool 可扩展性

---

## 8. Phase 3：`IvanMurzak/Unity-MCP` 评估条件

只有当下面其中至少一条成立时，再认真引入它：

1. 你要开始做 Sunset 专用 MCP Tool
2. 你要把 Unity 编辑器与 Runtime 都纳入 AI 工作流
3. 你要把“工具平台化”作为正式项目来做

如果当前只是想把验证链路稳住，它不是最小成本答案。
