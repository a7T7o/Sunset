## 一、核心差异与融入原则

### 1.1 Kiro 与 Cursor 的核心差异

| 维度 | Kiro | Cursor |
|------|------|--------|
| **自动继承对话** | ✅ 有（>80%自动压缩，Hook 生成继承快照） | ❌ 无 |
| **规则加载** | Hook 自动智能加载 | Rules 配置 + globs |
| **Memory 更新** | 自动（Hook 触发） | 手动执行 |
| **任务结束 Hook** | 自动触发 | 无此机制 |

### 1.2 不需要融入 Cursor 的 Kiro 特性

- `000-context-recovery.md` 的继承恢复流程
- `memory-update-check.kiro.hook` 的自动压缩检测
- 自动生成的继承会话快照机制

### 1.3 需要融入 Cursor 的 Kiro 特性

- Steering 规则体系（禁止事项、编码规范，工作流）
- Hook 的"可手动执行的部分"：
  - 智能规则加载思路 → 通过 Cursor Rules 的 `globs` 实现
  - Memory 更新流程 → 通过规则约束手动执行
- Skills/Subagents 的设计理念

## 二、迁移目标

1. **保持规则一致性**：所有规范仍以 `.kiro/steering` 中 `isCanonical: true` 的文件为真理来源
2. **适配 Cursor 工作流**：将 Kiro 的自动化特性转化为 Cursor 可以执行的规则和手动操作
3. **明确差异处理**：明确哪些自动化特性因 Cursor 限制需要改为手动执行

## 四、具体融入方案

### 4.1 智能规则加载（对应 smart-assistant.kiro.hook）

**Kiro 行为**：promptSubmit 时自动分析关键词并加载对应领域规则

**Cursor 方案**：通过 Rules 的 `globs` 配置实现文件级别自动加载

```markdown
# .cursor/rules/farming.md
---
description: 农田系统规则
globs: ["**/*Farm*.cs", "**/*Crop*.cs"]
alwaysApply: false
---
```

### 4.2 Memory 更新（对应 memory-update-check.kiro.hook）

**Kiro 行为**：agentStop 时自动检查并更新 memory

**Cursor 方案**：通过规则约束，要求每次任务结束后提醒用户或用户手动触发

### 4.3 继承恢复（对应 000-context-recovery.md）

**Kiro 行为**：自动检测上下文压缩并生成快照

**Cursor 方案**：不需要自动执行，用户可通过输入"执行对话快照操作"手动触发快照

## 五、风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| Memory 更新遗漏 | 项目历史不可追溯 | 规则约束提醒 + 用户监督 |
| 规则加载不生效 | 开发规范失效 | 测试验证 + 手动确认 |
| 手动快照遗漏 | 重要讨论丢失 | 用户触发机制 |
