# Sunset 当前规范快照（2026-03-22）

## 1. 文件身份
- 本文件是 Sunset 当前唯一 live 规范快照文件。
- 从 2026-03-22 起，凡是涉及：
  - 默认 Git 开发模型
  - 线程默认如何提交
  - branch / worktree 何时才是例外
  - 当前应遵守哪份活规范
  统一以本文件为准。
- 如果旧文档仍写着 `main-common + branch-task + checkpoint-first + merge-last`，按历史背景理解，不再视为当前默认口径。

## 1A. 当前 live 规则层级
- 当前 Sunset 的 live 执行优先级固定为：
  1. 用户当前裁定
  2. `D:\Unity\Unity_learning\Sunset\AGENTS.md`
  3. 本文件
  4. `D:\Unity\Unity_learning\Sunset\.kiro\specs\Codex规则落地\` 下当前命中的治理规范正文
  5. `.kiro/steering/` 中的通用结构、领域细则与历史通用基线
- `.kiro/steering/` 仍然重要，但它不再单独定义当前 shared-root / main-only / dispatch / startup-guard 的 live 默认。

## 2. 当前默认开发模型
- 当前默认模型不是 branch-first。
- 当前默认模型是：
  - `main-only + whitelist-sync + exception-escalation`
- 直白说就是：
  - 普通线程默认都在 `D:\Unity\Unity_learning\Sunset @ main` 语义下开发
  - 完成一刀后，默认只提交自己修改的内容
  - 默认只提交自己的白名单路径
  - 默认一线程一次只收一刀自己的 checkpoint
- 治理线程默认不代替业务线程提交。
- 治理线程主要负责：
  - 高危打断
  - 顺序裁定
  - 例外收口

## 3. 只有这些情况才升级治理介入
1. 撞到同一个高危目标
2. 同一个 Scene / Prefab / 热脚本被多人改
3. 跨线程强耦合，无法按白名单拆刀
4. 成果还停在 branch carrier / worktree，尚未迁入 `main`
5. 需要 Unity / MCP live 写，且已有人占用

## 4A. 当前两类 hot target 的实战差异
- `Primary.unity`
  - 按单 scene writer 接盘面处理
  - `dirty + unlocked + ownerless` 表示“无人接盘的 stale mixed scene”，不等于永久不可写
  - 当前批准的接盘顺序为：`NPC -> spring-day1`，直到用户改写
- `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`
  - 按“可并发但必须记触点”的共享热点处理
  - 不再因为文件本身已经 dirty 就自动要求所有其他线程停工
  - 只有当两个线程撞到同一入口、同一方法或同一语义顺序时，才升级治理 / integrator

## 5. 当前 Git 收口口径
- 普通线程完成一刀后，默认直接做白名单 `main` 收口。
- 当前允许：
  - `git-safe-sync.ps1 -Action sync -Mode task -OwnerThread <线程名> -IncludePaths ...`
  - 在 `main` 上执行
  - 只收本线程白名单
  - 保留 unrelated dirty
- 当前不再允许把“shared root 里还有 unrelated dirty”当成默认一刀切阻断理由。
- 但从现在起，`git-safe-sync.ps1` 会继续往前多挡一层：
  - 如果当前白名单所属 `own roots` 下还有未纳入本轮的 remaining dirty / untracked
  - 即使这些改动仍属于你自己
  - 也必须直接阻断，不能继续 sync
- 真正要阻断的是：
  - 你想把别人的改动也带进去
  - 你在碰高危目标
  - 你在做无边界提交

## 5B. 当前收口与回执硬闸门
- 普通线程的 own dirty / untracked 默认必须在当前这一刀里自己收口；不能再把自己的尾账拖到下一轮，再等治理线程专门发 cleanup。
- 有效回执现在必须额外回答：
  - `当前 own 路径是否 clean`
- 如果 `当前 own 路径是否 clean != yes`：
  - 这条回执只能算 blocker / owner 报实
  - 不能算“当前这一刀已闭环”
  - 也不能直接进入下一轮 feature prompt
- 治理线程默认不再为“常规 own dirty 尾账”反复开 cleanup 批次；只有命中 cross-thread mixed / hot-file incident、owner 不清或 shared root 现场已被卡死时，才升级治理收口。
- 当前强制报实的 unresolved hot / mixed 目标固定为：
  - `Primary.unity`
  - `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`
  - `ProjectSettings/TagManager.asset`
  - `Assets/Editor/StaticObjectOrderAutoCalibrator.cs`
- 如果 `changed_paths` 包含 `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`：
  - 回执除固定最小字段外，还必须额外回答 `touched_touchpoints`
  - 用来说明这轮到底触碰了哪些入口 / 方法 / 行为链

## 5D. 当前并发登记与退出口径
- 从 2026-03-31 起，Sunset 普通线程的当前并发模型补强为：
  - `Begin-Slice -> 施工 -> Ready-To-Sync -> sync`
  - 如果中途暂停或放弃当前一刀，则改走：
  - `Begin-Slice -> 施工/分析 -> Park-Slice`
- 这 3 个动作当前对应的 live 脚本是：
  - `D:\Unity\Unity_learning\Sunset\.kiro\scripts\thread-state\Begin-Slice.ps1`
  - `D:\Unity\Unity_learning\Sunset\.kiro\scripts\thread-state\Ready-To-Sync.ps1`
  - `D:\Unity\Unity_learning\Sunset\.kiro\scripts\thread-state\Park-Slice.ps1`
- 直白说：
  - 线程不是“想提交时才想起自己这轮改了什么”
  - 而是开工时先登记、收口前先过闸、停工时先合法停车
- `Begin-Slice` 什么时候必须跑：
  - 不是纯只读分析时
  - 而是一旦开始做这一刀真实施工、准备改 tracked 内容、接盘热文件或形成本轮白名单切片时
- `Ready-To-Sync` 什么时候必须跑：
  - 不是“提交以后补记”
  - 而是任何白名单 `sync` 之前都必须先跑
- `Park-Slice` 什么时候必须跑：
  - 当前这轮不再继续
  - 卡住等待别的线程
  - 需要把现场让出来
  - 不允许线程人已经停下，但 live 状态层仍显示自己在施工
- 当前治理位 / integrator 看现场时，默认不再只靠 memory 和追问；先看：
  - `D:\Unity\Unity_learning\Sunset\.kiro\scripts\thread-state\Show-Active-Ownership.ps1`
- 过渡规则：
  - 已经在跑的旧线程不要求全部废弃重开
  - 但从现在起，最晚必须在下一次真实继续施工前补 `Begin-Slice`
  - 最晚必须在第一次准备 `sync` 前补 `Ready-To-Sync`
  - 如果准备先停，则直接补 `Park-Slice`
- 治理 prompt 的默认接线规则：
  - 凡是 Sunset 治理位、典狱长位、唤醒位、或续工 prompt 生成位，只要这条 prompt 会让线程进入真实施工，正文末尾默认必须追加统一 `thread-state` 接线尾巴
  - 统一尾巴文件：
    `D:\Unity\Unity_learning\Sunset\.kiro\specs\Codex规则落地\线程续工统一尾巴_thread-state补登记_2026-04-01.md`
  - 只有当这条线程本轮始终停留在只读分析，才允许不追加
- 业务线程直聊口径：
  - 这套规则同样适用于“用户直接和 Sunset 业务线程对话”的高频场景
  - 也就是说，业务线程自己就必须自发执行 `Begin-Slice / Ready-To-Sync / Park-Slice`
  - 不以“是否收到治理位尾巴”作为触发前提
  - 第一条 `commentary` 就应报实：本轮只读，还是现在先跑 `Begin-Slice` 再继续施工

## 5C. 当前用户可读汇报硬闸门
- 现在必须明确区分：
  - `给治理看的最小回执`
  - `给用户看的用户可读汇报`
- 这两种文本不再允许混为一谈。
- 只要汇报对象是用户，或用户问的是：
  - `现在做到哪了`
  - `你实际做了什么`
  - `还剩什么`
  - `下一步做什么`
  - `到哪个阶段了`
  线程就必须先交 `用户可读汇报层` 的 `保底六点卡`，而且必须按固定顺序逐项显式回答：
  - `当前主线`
  - `这轮实际做成了什么（按功能点，不按文件）`
  - `现在还没做成什么`
  - `当前阶段`
  - `下一步只做什么`
  - `需要用户现在做什么（若无写无）`
- `保底六点卡` 不允许合并、省略或留空；某项没有内容时，也必须明确写 `无 / 尚未 / 不需要 / 仍待验证`，不能让用户自己从后文猜。
- 用户可读回复的固定顺序是：`A1 保底六点卡 -> A2 用户补充层（可选） -> B 技术审计层`；首先是给用户看懂，其次才是给治理看清。
- `changed_paths`、参数名、checkpoint、日志、dirty/lock/owner 报实等技术信息，只能放在最后的 `技术审计层`，不能顶替前面的 `用户可读汇报层`，也不能在 `保底六点卡` 写齐前提前插入。
- 如果回复里使用：
  - `best-known stable checkpoint`
  - `detour`
  - `release / recover`
  - `blockOnsetEdgeDistance`
  - `HardStop`
  - `stuck cancel`
  - `blocked-hotfile`
  这类术语，必须立刻翻译成玩家可感知结果，而且应优先出现在 `保底六点卡` 或 `用户补充层`；不能只给工程术语。
- 对用户宣称“已完成 / 已修好 / 已过线”时，必须显式标明验证状态：
  - `用户已测`
  - `线程自测已过`
  - `静态推断成立`
  - `尚未验证`
- 以后线程若出现以下任一情况，即使工程事实没错，也视为回执不合格：
  - 缺任一项 `保底六点卡`
  - 把多项用户信息揉成一段，用户必须自己拆
  - 先贴技术 dump，后补人话
  - 只说做了什么，不说还没做成什么
  - 只说 blocker，不说下一步只做什么
  - 不写 `需要用户现在做什么`
- 治理线程和当前执行线程审这类回复时，必须先按 `保底六点卡` 判是否说明白，再看技术审计层；技术层充分，不等于用户层达标。

## 5A. 当前代码正确率闸门
- 自 2026-03-23 起，只要 `task` 模式白名单里包含 `.cs` 文件，`git-safe-sync.ps1` 会在收口前自动触发代码闸门。
- 代码闸门当前固定做 3 件事：
  - 目标文件 UTF-8 检查
  - `git diff --check`
  - 程序集级编译检查
- 代码闸门未通过：
  - 不允许继续收口
  - 不应再把“等 Unity / MCP / 用户贴报错后再修”当作默认流程
- 代码闸门通过，只代表“代码层 / 文本层 / 程序集层暂未见阻断”，不代表 Unity Console 已干净，也不代表运行时红错已经验收完成。
- 从 2026-04-04 起，Sunset 爆红处理的默认主入口固定为：
  - `CLI first, direct MCP last-resort`
  - 高频 fresh compile / console / own-external 判断默认先走 CLI
  - 只有 CLI 当前覆盖不到、明确落 `unity_validation_pending / blocked`、或任务进入 scene / play / inspector / live flow 这类低频高风险取证时，才升级 direct MCP
- 从现在起，只要本轮改动触及运行时代码、scene、prefab、asset、UI、剧情、交互、输入或管理器链，并且线程准备对外宣称：
  - `无红错`
  - `红错已清`
  - `Unity 可直接交接`
  - `可直接提交`
  就必须同时补一份运行态侧的最新证据，至少包括：
  - 默认先给 CLI 侧的 fresh recompile 或等价程序集级编译证据
  - 默认先给 CLI 侧的 fresh console 读取结果
  - 若 CLI 当前覆盖不到，再补 direct MCP / live 证据
  - 若任务本身涉及 scene/UI/runtime flow，再补最小 live 取证，或明确标成 `live 待验证`
- `validate_script`、`CodexCodeGuard`、`git diff --check` 都只属于代码层自检，不允许再被写成“红错已经验收完毕”。
- 如果当前 CLI 与 direct MCP / Unity 基线都不可用，唯一允许的口径只能是：
  - `代码闸门通过，但 Unity 红错验证未闭环`
  - 或 `被 CLI / MCP / Unity blocker 卡住`
  不允许把这种状态包装成“已无红错”或“可放心交接”。

## 6B. 典狱长模式
- 从现在起，用户明确说出 `典狱长`、`典狱长上货`、`上货` 时，治理线程进入“典狱长模式”。
- 典狱长模式的第一动作不是继续群发 prompt，而是先审最新回执并把线程判成四类：
  - `继续发 prompt`
  - `停给用户验收`
  - `停给用户分析 / 审核`
  - `无需继续发`
- 只有第一类才继续写新的 prompt 文件；其余三类都必须停下并向用户交代原因。
- 典狱长模式下，给用户的转发格式固定为：
  1. `对应文件在：`
  2. 平铺列出最新 prompt 文件路径
  3. `你可以直接这样发给 XXX：`
  4. 一个 `text` 代码块，内部直接包含 `请先完整读取 [最新 prompt 路径]`
- 详细 prompt 正文继续留在文件里，不再回贴到聊天正文。
- 当前 live 规范源：

## 6C. 看守长模式
- 从现在起，用户明确说出 `看守长`、`看守长上场`、`喊看守长` 时，默认进入“看守长模式”。
- `看守长模式` 不是 `典狱长模式` 的别名：
  - `看守长` = 线程完工后的用户验收交接
  - `典狱长` = 治理线程的审回执 / 判停发 / 再分发
- 如果当前是在 Sunset 业务线程自己的对话里触发 `看守长`，且用户没有明确要求它去审别的线程，默认就以“当前线程自己最近完成或准备交接的这一刀”为输入：
  - 主动回看当前工作区 / 线程 memory / 当前 slice / 最新验证
  - 直接交验收包
  - 不允许先让用户补贴“线程回执 / prompt 文件 / 线程名”
- 这里的默认输入进一步收紧为：
  - 上一轮刚完成
  - 且已经向用户汇报过结果
  的那一刀
- 如果用户是在完成汇报后直接喊 `看守长`，线程必须直接沿刚才那轮完成项生成验收包，不允许把验收对象漂成整条大线，也不允许重新向用户确认“要验哪一轮”
- 如果这次不是紧跟在完成汇报后，但上下文仍明显是当前业务线程自己的验收场景，线程也必须自己回看当前工作区 / 线程 memory / 当前 slice / 最近完成汇报，选最近一次“已完成且已向用户交代过结果”的切片来交接；最多只允许用一句话报明这次按哪一刀交接，仍不允许把材料索取工作甩回给用户
- `看守长` 回复第一屏不应先讲模式切换或治理说明，而应直接进入验收交接正文
- 也不要只回 `在 / 看守长到了 / 看守长到位` 这种空占位再等用户继续补材料
- 当前交付顺序固定为：
  1. `看守长总判断`
  2. `线程已自验`
  3. `本轮仍需用户终验的点`
  4. `统一前提`
  5. `建议顺序`
  6. `验收清单矩阵`
  7. `最少必测包`
  8. `快捷回执单`
  9. `完整版回执单`
- Sunset 下默认叠加：
  - `acceptance-warden-mode`
  - `sunset-acceptance-handoff`
