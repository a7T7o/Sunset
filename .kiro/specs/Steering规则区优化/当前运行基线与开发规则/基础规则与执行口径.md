## 3. Git 执行口径

### 治理类改动
- 典型范围：
  - 规则
  - 索引
  - `memory`
  - Hook 文档
  - 脚本说明
  - 治理续办代办
- 允许直接留在 `main`
- 收尾方式固定为：
  - `git-safe-sync.ps1 -Action sync -Mode governance`
  - 显式白名单提交
- 补充硬规则：
  - `governance` 默认自动允许的只剩 `.gitattributes`、`.gitignore`、`AGENTS.md`、`scripts/`、`.kiro/steering/`、`.kiro/hooks/`
  - 活文档目录、`.kiro/specs/000_代办/codex/`、`.codex/threads/`、工作区 / 线程 `memory` 虽然属于治理语义，但同步时仍必须显式传 `-ScopeRoots` 或 `-IncludePaths`

### 真实任务改动
- 典型范围：
  - `Assets/`
  - `Packages/`
  - `ProjectSettings/`
  - 业务脚本
  - 场景
  - Prefab
  - ScriptableObject
  - 资源
- 不直接在 `main` 上收尾
- 收尾方式固定为：
  1. 从默认入口读取现场
  2. 必要时查锁
  3. 切到 `codex/xxx`
  4. 用 `git-safe-sync.ps1 -Mode task` 白名单同步

### 白名单原则
- 白名单只解决“本轮带哪些文件”。
- 白名单不等于放宽分支规则。
- 仍禁止无边界 `git add -A`。

## 4. 多线程协作口径
- 当前真正的第一优先级，不是格式漂亮，而是共享状态清楚。
- 所有线程推进前，都要先看：
  - 当前状态入口
  - 当前 Git 规则入口
  - 锁 / 认领 / dirty / 线程状态等共享层文档
- 如果本轮改动会改变认领范围、WIP 边界或线程状态，必须同步共享层，不允许只写在本线程自己的 `memory` 里。
- 用户不应继续充当线程之间的人工中继站。

## 5. 文档身份与命名口径
- 如果文件职责是“持续描述当前状态 / 当前规则 / 当前默认入口”，就不能继续沿用已经过时的日期文件名承接最新内容。
- 如果文件职责是“描述某次事件、某次重组、某次阶段收口”，则可以保留事件日期，但必须明确它是索引、底册、阶段记录或历史锚点。

  - `Sunset当前唯一状态说明_2026-03-13.md` 的正文已归档到 `06_L5解冻与阶段快照_2026-03-16\01_03-13历史状态锚点退役_2026-03-16`；
