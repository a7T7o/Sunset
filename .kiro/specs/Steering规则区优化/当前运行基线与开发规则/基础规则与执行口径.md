# Sunset 基础规则与执行口径

## 1. 当前唯一主口径
- 默认现场：
  - `D:\Unity\Unity_learning\Sunset @ main`
- 默认开发模型：
  - `main-only + whitelist-sync + exception-escalation`
- shared root 占用状态：
  - `D:\Unity\Unity_learning\Sunset\.kiro\locks\shared-root-branch-occupancy.md`
- 是否真的可作为中性入口，以 shared root 占用文档里的实时裁定为准，不能把“当前在 `main`”直接等同于“现场已干净可切分支”。
- worktree 口径：
  - 只作例外，不作日常
  - 若历史 worktree 仍暂时存在，脚本与规则副本不自动视为 live；执行闸机时优先使用 shared root 的现行脚本与入口文档

## 2. 目录职责
- `D:\Unity\Unity_learning\Sunset\AGENTS.md`
  - live 路由与执行约束层
- `Sunset当前规范快照_2026-03-22.md`
  - 当前唯一 live 规范快照文件

  - 当前治理裁定、分发与回执 live 规则
- `.kiro/steering/`
  - 通用结构、领域细则与历史通用基线
  - 不再单独定义当前 shared-root / main-only / dispatch / startup-guard 的 live 默认
- `Steering规则区优化/当前运行基线与开发规则`
  - 现行入口和当前口径
- `.kiro/specs/Codex规则落地`
  - 正式治理工作区
- `.kiro/specs/000_代办/codex`
  - TD 镜像区，不是工作区

## 3. 线程默认进入方式
1. 先从 shared root 进入
2. 先过 `sunset-startup-guard`
3. 先核：
  - 当前主线
  - 当前工作目录 / 分支 / HEAD
  - shared root 是否中性
4. 治理默认留在 `main`
5. 业务默认也留在 `main` 语义下推进
6. 完成一刀后默认按白名单直接收口
7. 只有命中例外时才升级到 `codex/...` branch carrier 或 worktree

## 4. Git 口径

### 治理类改动
- 允许留在 `main`
- 收尾使用：
  - `git-safe-sync.ps1 -Action sync -Mode governance -OwnerThread <线程名>`
- 仍需显式给出：
  - `-IncludePaths`
  - 或 `-ScopeRoots`

### 业务类改动
- 默认允许在 `main` 上按白名单收尾
- 收尾使用：
  - `git-safe-sync.ps1 -Action sync -Mode task -OwnerThread <线程名>`
- 当前脚本现行口径：
  - `main` 允许 `task` 模式白名单 sync
  - 允许保留 unrelated dirty
  - 禁止无边界自动提交
  - 如果位于 `codex/` 分支，仍要求分支与 `OwnerThread` 语义匹配
- 当前脚本已补上第二波最小 shared root 闸机：
  - 若 shared root 占用文档声明 `is_neutral = false`，则 `task` 模式仍必须同时匹配 `owner_thread + current_branch`
  - `main-only` 白名单 sync 不再因 unrelated remaining dirty 一刀切阻断
  - 只有命中 branch carrier / worktree 例外场景时，才继续走 `ensure-branch`

## 5. Unity / MCP 单实例口径
- Git 现场安全，不等于 Unity / MCP 现场安全。
- 只要任务涉及 Unity、MCP、Play Mode、Compile、Domain Reload、Console、Scene / Prefab / Inspector 读写，必须额外核：
  - `D:\Unity\Unity_learning\Sunset\.kiro\locks\mcp-single-instance-occupancy.md`
  - `D:\Unity\Unity_learning\Sunset\.kiro\locks\mcp-hot-zones.md`
  - `D:\Unity\Unity_learning\Sunset\.kiro\locks\mcp-single-instance-log.md`
- 当前默认口径：
  - Unity 视为单实例共享资源
  - 写操作默认不假设天然排队安全
  - 读操作也不假设一定能读到稳定最终态
- 凡是为了取证、调试或验收进入 Play Mode，完成当前步骤后都必须主动退回 Edit Mode，再允许汇报、切分支或把现场交给其他线程。
- 出现以下任一信号时，先停止继续写入并退回只读核查：
  - Play Mode 切换
  - 编译 / Domain Reload
  - 对象失效、端口占用、MCP 返回中间态
  - 场景 / Prefab / Inspector 状态突然跳变

## 6. 表现与样式验收口径
- 只要任务涉及 DialogueUI、NPC 气泡、字体、UI 布局、样式、材质、截图验收画面或其他玩家直接可见的表现层资源，就不能只以“功能通了”为完成标准。
- 启动闸门后，至少还要补读：
  - `D:\Unity\Unity_learning\Sunset\.kiro\steering\ui.md`
  - 必要时再读 `D:\Unity\Unity_learning\Sunset\.kiro\steering\scene-modification-rule.md`
- 最低验收项包括：
  - 字体与字号是否合适
  - 气泡 / 面板样式是否协调
  - 锚点、对齐、留白、层级遮挡是否合理
  - 可读性、专业感与场景一致性是否成立
- 发现“测试味过重、位置别扭、字体不协调、遮挡明显、样式粗糙”时，不得直接宣称完成。
