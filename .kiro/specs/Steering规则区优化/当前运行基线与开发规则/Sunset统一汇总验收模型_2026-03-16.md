# Sunset 统一汇总验收模型（2026-03-16）

## 1. 文件身份
- 本文件从 `2026-03-16` 起承接当前现行的项目级统一验收模型。
- `Sunset统一汇总验收模型_2026-03-15.md` 降级为第一轮收口分析阶段的历史首版，不再承担现行验收入口职责。
- 后续如果项目的“准入门槛”发生阶段性变化，应新建新的当前文件，不继续把最新准入定义叠写进旧日期文件。

## 2. 目的
- 本模型只回答一件事：
  - 什么时候可以把某一阶段、某一组线程、或整个项目，定性为“已达到当前阶段的正常开发准入”。
- 它不替代线程自己的完成态定义，而是站在项目级共享现场上做最终裁决。
- 后续不再允许：
  - 用“某个线程说自己完成了”代替整个项目的收口判断；
  - 用“没有新报错”代替共享现场已经健康；
  - 用“用户口头理解了”代替现行入口已经可独立接手。

## 3. L1-L5 验收层

### L1. 规则与入口一致性层
- 必须满足：
  - 当前唯一状态入口是 `Sunset当前唯一状态说明_2026-03-16.md`
  - 当前 Git 规则入口是 `Sunset Git系统现行规则与场景示例_2026-03-16.md`
  - 当前共享表入口是 03-16 版线程总表、锁表、dirty/WIP 归属表
  - 当前工作区结构和代办口径能在 `Sunset工作区四件套与代办规范_2026-03-16.md` 中直接找到
  - 上述文档之间没有互相打架的现行口径
- 不通过时的输出：
  - 明确指出冲突文档
  - 指定谁是现行入口
  - 指定谁应降级为历史锚点、首版快照或兼容页

### L2. Git、锁与工作树归属层
- 必须满足：
  - 默认基线 `main -> origin/main` 的定义清楚
  - 当前实际工作树所在分支、`HEAD` 与 dirty 归属清楚
  - 线程自有 WIP 与默认基线边界清楚
  - A 类共享热文件遵守查锁、上锁、checkpoint、释放与交接纪律
  - 不存在“来源不明、owner 不明、谁都能顺手带走”的 dirty
- 不通过时的输出：
  - 逐项列出未归属路径
  - 明确 owner 线程
  - 明确是否为 `Committed`、`Parked`、`Abandoned`、`共享只读` 或 `待线程收口`

### L3. 编译与 Console 层
- 必须满足：
  - 没有新的项目级红编译
  - Console 没有新的阻断性错误
  - warning 已按责任线程或责任系统分流，不再被泛化成“项目没恢复”
- 注意：
  - 工具链 warning、初始化日志、已知非阻断 warning，不自动等于项目级失败
  - 但若影响本轮主线验收，必须被提升到对应线程或治理层单独处理
- 不通过时的输出：
  - 明确指出它是项目级阻断、线程级阻断、工具链问题，还是只读核查噪音

### L4. Live 场景与关键链路层
- 必须满足：
  - `unityMCP` 或同级现场读取能力稳定可用
  - 当前主场景与关键链路能被独立核到
  - 关键组件空引用能被判定为：
    - 自动补线设计
    - 实际缺口
    - 非阻断历史残留
  - 关键行为验收不再只停留在“代码看起来有逻辑”
- 不通过时的输出：
  - 直接列出缺失对象、空引用对象、行为级缺口，或尚未取得的现场证据

### L5. 正常并行开发准入层
- 只有在 L1-L4 都达到“无项目级阻断”后，才能进入 L5 判定。
- 额外必须满足：
  - 新线程只读当前活文档和共享表，就能知道现在怎么开工
  - 用户不再需要手动充当线程之间的中继站
  - A 类热文件写锁机制已经真实可用，而不是只写在文档里
  - “线程先收遗留状态，再谈新功能”的口径已经进入当前流程
- 通过后的含义是：
  - 项目回到按规则运行的常态开发
  - 不等于回到无分支、无白名单、无锁、无边界的旧状态
- 不通过时的输出：
  - 必须明确写出唯一剩余准入缺口
  - 不允许同时列出一长串分散问题，把真正门槛说模糊

## 4. 当前最小达标定义
- 当前不是要求：
  - 所有线程零 warning
  - 所有线程零 WIP
  - 所有历史文档都完全重写完毕
- 当前要求的是：
  - 规则入口唯一
  - 共享状态可独立读取
  - Git 与锁边界清楚
  - 项目不被新的红编译和共享热文件混写卡死
  - live 场景与关键链路可被现场核查
  - 用户不再承担人工同步成本

## 5. 每次项目级裁决时的固定输出
1. 当前裁决对象是什么
2. L1-L5 哪些已通过
3. 第一层未通过的是哪一层
4. 当前唯一剩余缺口是什么
5. 现在允许继续做什么
6. 现在仍禁止继续做什么

## 6. 2026-03-16 当前示例
- 按 2026-03-16 的现行现场：
  - `L1` 已通过：当前状态、Git 规则、共享表与代办入口已切到 03-16 版
  - `L2` 已通过：`Primary.unity` 与 `DialogueUI.cs` 的 hot-file handoff 缺口已关闭，当前锁池为空
  - `L3` 已通过：当前没有新的项目级红编译
  - `L4` 已通过：`Primary`、`DialogueCanvas`、`NPC001`、对话最小链路已有 live 证据
  - `L5` 已通过：项目可恢复全面正常开发
- 这里的“已通过”只表示：
  - 准入门槛已经满足；
  - 不表示所有线程的尾账已经自动消失。
