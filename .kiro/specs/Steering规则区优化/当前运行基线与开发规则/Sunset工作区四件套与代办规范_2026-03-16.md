# Sunset 工作区四件套与代办规范（2026-03-16）

## 1. 文件身份
- 本文件从 `2026-03-16` 起承接 Sunset 当前的“工作区四件套与代办结构”专项口径。
- 它是 Sunset 在不违背 `.kiro/steering/workspace-memory.md` 的前提下，对 `requirements / design / tasks / memory` 如何落地的项目级细化规则。
- 本文件不替代 steering 正文；当两者存在冲突时，以 steering 正文为上位规则，本文件负责 Sunset 的具体用法与减负方式。

## 2. 为什么必须重构
- 过去的主要失真点不是“文档不够多”，而是：
  - 旧全局 `tasks.md` 长期膨胀，已经不适合继续承接新增治理续办；
  - 一些工作区把 `requirements / design / tasks / memory` 机械四连建，导致空壳文档和重复入口越来越多；
  - `memory.md` 长期不分卷，已经出现 1000 行以上的超长卷；
  - “后续再做”长期靠上下文与线程记忆硬扛，没有形成稳定的代办接力层。

## 3. 四件套的责任边界

### 3.1 `memory.md`
- 永远是第一核心文档。
- 负责记录：
  - 主线目标
  - 本轮子任务 / 阻塞
  - 关键决策
  - 修改文件
  - 验证结果
  - 恢复点 / 下一步
- 只要本轮有实质性工作，就必须更新。

### 3.2 `tasks.md`
- 不是“工作区的一本总账”，而是“某一阶段的执行清单”。
- 默认只挂在阶段目录里，不再默认由整个工作区长期共用一个全局 `tasks.md`。
- 一个阶段结束后：
  - 该阶段 `tasks.md` 保留为历史执行记录；
  - 新阶段应开新目录、新 `tasks.md`。

### 3.3 `design.md`
- 只有在下列情况之一成立时才创建：
  - 存在多个可行方案，需要比较
  - 会影响多个线程、多个系统或共享热文件
  - 有明显架构权衡，后续需要回看“为什么这样设计”
- 如果只是：
  - 单点修 bug
  - 纯文档整理
  - 简单规则改写
  - 一次性检查清单
  则不默认创建 `design.md`。

### 3.4 `requirements.md`
- 只有在需求尚未稳定，或必须把“用户想要什么”与“我们决定怎么做”分开时才创建。
- 如果用户目标已经稳定，且当前只是在执行既定治理续办或阶段收口，不必机械补一份 `requirements.md`。

## 4. Sunset 当前推荐结构

### 4.1 治理续办 / 代办型工作区
```text
.kiro/specs/000_代办/codex/
├── memory.md
├── 01_阶段A/
│   └── tasks.md
├── 02_阶段B/
│   ├── tasks.md
│   └── design.md（按需）
└── ...
```

- 根层默认只保留：
  - `memory.md`
- 分阶段目录各自持有：
  - `tasks.md`
- `design.md`、`requirements.md` 只有在确实需要时才新增。

### 4.2 功能 / 业务型工作区
```text
.kiro/specs/某个业务工作区/
├── memory.md
├── 001_阶段说明/
│   ├── tasks.md
│   └── design.md（按需）
└── 002_下一阶段/
    └── tasks.md
```

- 业务主工作区也应逐步走“根层 memory + 分阶段 tasks”的结构。
- 不再鼓励在同一层长期维护一个无限膨胀的总 `tasks.md`。

## 5. 什么时候该新建代办工作区
- 满足任一条时，优先考虑进入 `000_代办/...`：
  - 待办横跨多个工作区
  - 它不是单个功能线程的直接下一步，而是治理续办或历史尾账
  - 需要长期排期，而不是当前线程一次做完
  - 用户明确要求“先收成代办，不要继续压在当前工作区里”
- 当前 Sunset 的治理续办统一承接点已经固定为：
  - `D:\Unity\Unity_learning\Sunset\.kiro\specs\000_代办\codex`

## 6. 什么时候留在原工作区推进
- 满足全部条件时，优先留在原工作区：
  - 当前只有单一 owner
  - 主线目标没有换
  - 验收标准仍属于该工作区
  - 不是跨工作区的治理问题
- 典型例子：
  - `spring-day1` 新增剧情功能
  - `NPC` 固化自己手里的 Prefab / 脚本 WIP
  - `farm` 继续推进农田交互新需求

## 7. `tasks.md` 的阶段化规则
- 一个阶段只维护一份 `tasks.md`。
- 阶段结束后：
  - 不再回头把新的任务塞进旧阶段 `tasks.md`
  - 应新开下一阶段目录
- 每份 `tasks.md` 只回答三件事：
  - 这个阶段要达成什么
  - 还差哪些事
  - 什么结果算这一阶段完成

## 8. `memory` 分卷规则
- 继续遵守 steering 的分卷上限：
  - 主工作区活跃卷 `memory.md` 建议不超过 `200` 行
  - 子工作区活跃卷 `memory.md` 建议不超过 `300` 行
- Sunset 现行补充口径：
  - 超限检查必须在“写入前”做，而不是超很多后再补救
  - 工作区活跃卷始终叫 `memory.md`
  - 工作区旧卷按 `memory_0.md`、`memory_1.md` 递增归档
  - 新卷开头必须写前情提要与当前恢复点
- 对已经超过上限很多的旧卷：
  - 不要求现在立刻全量重构历史；
  - 但禁止继续无边界往超长卷里叠写。

### 8.1 工作区记忆
- 适用于 `.kiro/specs/.../memory.md`。
- 分卷动作固定为：
  1. 将旧活跃卷完整迁为下一份 `memory_N.md`
  2. 新建精简 `memory.md`
  3. 在新卷写清分卷索引、承接摘要、当前恢复点

### 8.2 线程记忆
- 适用于 `.codex/threads/.../memory_0.md`。
- 线程卷与工作区卷不同：
  - 线程默认活跃入口必须继续保留为 `memory_0.md`
  - 超长时应把旧长卷迁为 `memory_1.md`、`memory_2.md` 这类历史卷
  - 然后重建新的精简 `memory_0.md`
- 原因：
  - `memory_0.md` 是当前线程的固定默认入口
  - 后续线程接手时不应猜测“当前活跃卷到底叫 `memory.md` 还是别的名字”
