# Sunset 线程统一汇总模板（2026-03-16）

## 1. 文件身份
- 本文件从 `2026-03-16` 起承接当前现行的线程统一汇报模板。
- `Sunset线程统一汇总模板_2026-03-15.md` 降级为第一轮冻结汇总阶段的历史首版，不再承担现行模板职责。
- 后续如果线程汇报结构发生阶段变化，应新建新的当前模板文件，不继续在旧日期模板上叠写。

## 2. 使用目的
- 让治理线程、功能线程、审计线程、文档线程都按同一结构汇报。
- 降低用户在多个线程之间来回搬运上下文的成本。
- 让共享表、锁表、dirty/WIP 归属表能接收统一格式的增量信息。

## 3. 汇报硬规则
- 必须先写“当前主线目标”，不能只回答最后一句。
- 必须把“已验证事实”和“当前判断 / 推测”分开。
- 必须写清当前工作目录、当前分支、当前 `HEAD`。
- 涉及文件时必须给绝对路径。
- 如果本轮没有修改文件，也必须明确写“本轮未修改文件”。
- 如果本轮涉及共享热文件，必须交代：
  - 当前是否查锁
  - 当前是否持锁
  - 当前是否释放锁
- 如果本轮只是只读复核，也必须写清：
  - 读了什么
  - 没改什么
  - 为什么当前判断仍成立或不成立

## 4. 所有线程都要覆盖的 12 项
1. 当前线程名称
2. 当前主线目标
3. 当前子任务 / 当前阻塞
4. 当前现场锚点
5. 本轮读取了哪些现场
6. 本轮实际修改文件
7. 本轮共享热文件 / 锁状态
8. 本轮已验证事实
9. 当前判断 / 仍待验证
10. 当前阻塞点
11. 下一步动作与验收点
12. 是否需要用户动作

## 5. 日常进展模板

```markdown
1. 当前线程名称
- `线程名`

2. 当前主线目标
- 用一句话写当前真正要完成的主任务。

3. 当前子任务 / 当前阻塞
- 说明本轮是在推进主线，还是在清理哪个阻塞。

4. 当前现场锚点
- 工作目录：`绝对路径`
- 当前分支：`分支名`
- 当前 `HEAD`：`提交号`
- Unity 现场：`活动场景 / Console / MCP 状态`（如无则写“本轮未读”）

5. 本轮读取了哪些现场
- `Git / 场景 / Console / memory / 只读代码 / 截图 / 资产` 中本轮实际读过的对象

6. 本轮实际修改文件
- `绝对路径 A`
- `绝对路径 B`
- 如无改动，写：`本轮未修改文件`

7. 本轮共享热文件 / 锁状态
- 是否涉及 A 类热文件：`是 / 否`
- 涉及对象：`绝对路径`（如无则写“无”）
- 锁状态：`未查锁 / 已查锁未申请 / 已持锁 / 已释放`

8. 本轮已验证事实
- 只写已经通过 Git、代码、文档、Unity live、MCP 或脚本直接确认的内容。

9. 当前判断 / 仍待验证
- 写推断，但必须和事实分开。

10. 当前阻塞点
- 只写当前真正阻断下一步的 1-3 个点。

11. 下一步动作与验收点
- 动作 1：`要做什么`
- 验收 1：`什么结果算通过`
- 动作 2：`要做什么`
- 验收 2：`什么结果算通过`

12. 是否需要用户动作
- `不需要`
- 或明确写：`需要用户做什么`
```

## 6. 冻结 / 快照模板
- 适用于：
  - 统一冻结汇总
  - 线程复工前快照
  - 用户要求“一次性交出当前现场”

```markdown
1. 当前线程名称
- `线程名`

2. 当前主线目标
- `一句话主线`

3. 当前工作目录 / 分支 / HEAD
- 工作目录：`绝对路径`
- 分支：`分支名`
- `HEAD`：`提交号`

4. 当前 dirty / untracked 范围
- tracked dirty：`绝对路径列表或归类`
- untracked：`绝对路径列表或归类`

5. 当前认领或占用的文件
- `绝对路径`
- 如无，写：`当前无占用`

6. 当前是否涉及 A 类热文件
- `是 / 否`
- 如是，写清是否持锁、锁名、当前状态

7. 本轮只读核对了什么
- `Git / Console / 场景 / memory / 资产 / Prefab / 脚本`

8. 本轮未做什么
- 明确写出未修改代码、未改场景、未提交、未释放以外的行为边界。

9. 当前已验证事实
- 只写已核到的现场结论。

10. 当前仍不确定的点
- 写尚未闭环的内容，不要装作已完成。

11. 如果现在复工，第一动作是什么
- `一句话`

12. 当前是否需要用户动作
- `不需要`
- 或明确写：`需要用户批准什么`
```

## 7. 写锁申请 / 归还模板
- 适用于：
  - 申请写 `Primary.unity`
  - 申请写 `GameInputManager.cs`
  - 归还任何 A 类热文件锁

```markdown
1. 当前线程名称
- `线程名`

2. 当前主线目标
- `一句话主线`

3. 申请或归还对象
- 文件：`绝对路径`
- 动作：`申请写锁 / 归还写锁`

4. 当前现场锚点
- 工作目录：`绝对路径`
- 分支：`分支名`
- `HEAD`：`提交号`

5. 当前为什么必须动这个文件
- `一句话理由`

6. 本轮最小 checkpoint
- `改到什么程度就停`

7. 当前依赖的前置条件
- `已查锁 / 已读共享表 / 已确认无他人占用 / 已完成上一 owner 收口`

8. 如果只是归还写锁
- 本轮实际改了哪些文件
- 当前 checkpoint 是否已通过
- 当前是否已提交、已停车，还是仅本地验证完成

9. 风险边界
- 明确写不碰哪些文件、不扩到哪些范围。

10. 验收点
- `什么结果算锁可以交接`

11. 是否需要用户动作
- `不需要`
- 或明确写：`需要批准申请 / 需要人工手验`
```

## 8. 不允许的写法
- 不允许只写“我这边好了”。
- 不允许把旧结论原样复述成当前真相。
- 不允许不写当前主线目标。
- 不允许不写分支和 `HEAD`。
- 不允许把“猜测”写成“已验证事实”。
- 不允许碰了共享热文件却不写锁状态。

## 9. 与共享状态层的关系
- 汇报前优先看：
  - `Sunset活跃线程总表_2026-03-16.md`
  - `Sunset文件认领与锁定表_2026-03-16.md`
  - `Sunset当前工作树dirty与WIP归属表_2026-03-16.md`
- 如果本轮让认领范围、锁状态或 dirty 归属发生变化，必须同步共享表，而不是只在自己线程里说。
