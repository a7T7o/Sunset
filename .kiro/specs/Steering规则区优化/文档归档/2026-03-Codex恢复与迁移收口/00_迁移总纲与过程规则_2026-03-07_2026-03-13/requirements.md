# Codex迁移与规划 - 需求文档

**创建日期**: 2026-03-07

---

## 1. 背景与目标

当前项目已经完成两类重要迁移探索：

1. **Kiro 原生规则与 Hook 工作流**：以 `.kiro/steering/` 为唯一规则源，以 `.kiro/specs/` 工作区为正式产物层。
2. **Claude 适配治理**：通过 `CLAUDE.md`、`.claude/hooks` 和相关治理文档，明确了哪些能力可以迁移、哪些只能降级为提醒或审计。

本子工作区的目标不是重复造一套规则，而是：

- 基于现有 Steering 体系，总结 **Codex 在本项目中的真实能力边界**。
- 设计一套适合 Codex CLI / Codex Desktop 的 **执行协议、规则路由方式与实操技巧**。
- 为后续是否上提到项目级说明文档（如仓库级 Codex 指南）提供稳定草案。

---

## 2. 核心需求

### 2.1 单一真理源

- `.kiro/steering/` 继续作为规则唯一真理源。
- `Codex迁移与规划` 只能做“解释、映射、SOP 固化”，不能成为新的业务规则权威来源。

### 2.2 工作区先行

- Codex 相关的方案、流程、技巧、风险边界、迁移计划，必须落在 `.kiro/specs/Steering规则区优化/Codex迁移与规划/` 内。
- 不依赖全局 plans 路径承载正式产物。

### 2.3 证据先行

- Codex 侧规则设计必须建立在已读文件、已知工具能力、当前运行环境事实之上。
- 禁止把 Kiro Hook 语义、Claude hook wiring、Codex 工具能力混为一谈。

### 2.4 Memory 手动收尾

- Codex 当前没有 Kiro 那种自动 `agentStop` 语义来兜底本项目的 memory 更新。
- 因此需要把“先子后父、append-only、写后复查”的 memory 收尾流程，内化为 Codex 明确 SOP。

### 2.5 工具映射必须真实可执行

- Codex 文档中提到的工具、能力、验证链路，都必须是当前 Codex 环境真实存在或可明确转交用户的。
- 禁止写入不可执行的幻想型指令。

### 2.6 面向实际使用的技巧沉淀

- 除规则说明外，还需要沉淀一份适用于 Codex 的工作技巧手册，覆盖：
  - 任务启动与规则路由
  - `update_plan` 的使用边界
  - 工具分组与并行读取
  - Unity MCP 与 Shell 的边界
  - Web 检索触发条件
  - 最终回复与路径引用习惯

---

## 3. 非目标

- 不在本阶段直接修改 `.kiro/steering/` canonical 文档。
- 不在本阶段直接设计或实现新的 Codex 平台 Hook 机制。
- 不尝试 1:1 复刻 Kiro 的自动继承恢复、自动 prompt 注入或自动 memory 更新机制。
- 不顺手重构 `.claude/`、`.cursor/` 现有配置。

---

## 4. 成功标准

满足以下条件，视为本阶段达到目标：

1. 已创建独立子工作区 `Codex迁移与规划`。
2. 已形成 `requirements.md`、`design.md`、`tasks.md`、`memory.md` 的基础骨架。
3. 已形成一份可直接指导未来 Codex 会话的《Codex工作流与技巧手册》。
4. 文档中已明确区分：
   - Kiro 的原生自动机制
   - Claude 的治理经验
   - Codex 的现实能力边界与可执行 SOP
5. 后续如需把 Codex 规则下沉到项目级文档，已经有可复用蓝本。
