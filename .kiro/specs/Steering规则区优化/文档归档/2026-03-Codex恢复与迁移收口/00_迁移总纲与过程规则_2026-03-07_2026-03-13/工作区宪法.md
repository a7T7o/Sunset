# Codex迁移与规划工作区宪法

**版本**: v1.0
**创建日期**: 2026-03-10
**状态**: 生效中
**定位**: 本文档用于固化当前工作区的长期执行约束，确保后续 Codex 或其他智能体继续维护 `.kiro/about/` 三份主文档时，不会偏离用户已经反复确认的目标、方法和边界。

---

## 1. 宪法性质与适用范围

### 1.1 本文档是什么

- 本文档是 `Codex迁移与规划` 工作区的执行宪法。
- 它不是新的 Steering，也不是第四份项目总报告。
- 它只负责回答一件事：在这个工作区里，后续应该如何持续正确地推进 `.kiro/about/` 文档工程。

### 1.2 本文档不是什么

- 不是 `.kiro/steering/` 的替代品。
- 不是 `AGENTS.md` 的替代品。
- 不是 `.kiro/about/00/01/02` 三份主文档的替代品。
- 不是随时可重写的临时笔记。

### 1.3 生效边界

- 在不与用户当前明确指令、项目 `AGENTS.md`、`.kiro/steering/` 正文规则冲突的前提下，本宪法是当前工作区的直接执行口径。
- 若上位规则与本宪法冲突，以上位规则为准；但应尽量把冲突点显式记录进 `tasks.md` 与 `memory.md`，不能默默漂移。

---

## 2. 最高目标

本工作区的最高目标只有一个：

- 持续建设并维护 `.kiro/about/` 下的三份主文档，使后来者或其他智能体只看这三份文档，就能快速建立对 `Sunset` 的整体理解，并具备继续接手的能力。

这个最高目标拆开后包含三层含义：

- 不是只做“项目总览”，而是做“可接手版认知底座”。
- 不是只读文档，还必须读代码、场景、Prefab、ScriptableObject、测试入口和 live memory。
- 不是写完一版就冻结，而是允许在新证据出现后持续纠偏和迭代。

---

## 3. 用户长期要求总表

以下要求视为当前工作区的长期硬约束：

1. 三份主文档方案固定：
   - `00_项目总览与阅读地图.md`
   - `01_系统架构与代码全景.md`
   - `02_当前状态、差异与接手指南.md`
2. 不新增新的主报告替代三文档；允许新增辅助治理文档，但辅助文档不能抢走三主文档的主入口地位。
3. 普通说明文字一律中文。
4. 默认直接做，不要一步一步频繁向用户确认。
5. 必须先维护 `tasks.md`，再迭代回写文档与记忆。
6. 允许边读边写、边证据化边修正文档旧结论；不允许“全部读完再一次性成稿”。
7. 发现新证据时，优先修正旧结论，再补新内容。
8. 场景 / Prefab / ScriptableObject / Inspector 属于高风险配置：
   - 默认只做审视与证据整理
   - 未获明确授权，不修改现有配置
9. 仓库或工作区可能是脏的：
   - 绝不回滚用户改动
   - 绝不把别人的脏改误当作可以清理的噪音
10. 记忆必须落盘，不能假设线程记忆天然持续。
11. 任务清单必须持续更新，不能只在第一次建表时写一版。
12. 任何“已完成”都必须有相应证据、文档口径与 memory 同步，不允许假闭环。

---

## 4. 固定主交付形态

### 4.1 三份主文档职责固定

- `00`
  - 负责入口、阅读顺序、真源判断、误读预警、角色化阅读路径
- `01`
  - 负责系统结构、代码入口、场景 / Prefab / SO 映射、测试与验证入口、证据索引
- `02`
  - 负责当前状态、漂移点、风险、接手顺序、SOP、闭环标准

### 4.2 辅助文档职责

当前工作区允许存在辅助治理文档，例如：

- `tasks.md`
- `memory.md`
- 本宪法
- 其他治理说明、技能清单、MCP 方案、故障报告

但必须遵守：

- 辅助文档服务于三主文档维护，不替代三主文档。
- 若后来者只需要“理解项目并接手”，优先仍应读三主文档。
- 若后来者需要“继续维护这套文档工程”，才额外读本宪法与 `tasks.md`。

---

## 5. 真源与证据优先级

后续继续维护时，证据优先级固定为：

1. 用户当前明确指令
2. 项目 `AGENTS.md`
3. `.kiro/steering/`
4. 活跃工作区 `memory.md`
5. 当前 live 代码 / 场景 / Prefab / ScriptableObject / 测试入口
6. `History/` 交接索引
7. `Docx/` 历史规划与总结

继续维护时必须明确区分以下四类结论：

- `代码已核对`
- `现场证据已核对`
- `历史快照`
- `基于证据的推断`

禁止把第 6、7 层内容直接写成当前现实。

---

## 6. 固定执行回路

每一轮正常推进都必须遵守以下顺序：

1. 回看 `tasks.md`
2. 判断本轮属于哪一项主线 / 子任务 / 阻塞处理
3. 如有新工作，先把任务写进 `tasks.md`
4. 读新证据
5. 先修正文档旧结论
6. 再补新内容
7. 核对三主文档口径是否一致
8. 按顺序同步 memory：
   - 子工作区
   - 父工作区
   - 线程记忆

这意味着：

- `tasks.md` 不是事后记录，而是每轮推进的入口锚点。
- 文档更新不是“追加越多越好”，而是“旧结论与新证据必须重新对齐”。
- memory 不是可选项，而是收尾硬步骤。

---

## 7. tasks 治理条款

### 7.1 tasks 的角色

- `tasks.md` 是当前工作区的真实看板。
- 它既记录待办，也记录完成状态、阶段目标和恢复点。
- 后续任何人接手本工作区，都必须先读它，再决定继续做什么。

### 7.2 tasks 的更新规则

- 每次新增一类明确工作时，先写进 `tasks.md`。
- 每次阶段闭环后，必须收口状态，不能让任务表停留在“实际上已完成，但仍显示进行中/待执行”。
- 长期纪律类任务可以保留为持续执行中，例如：
  - T24 这类“持续维护与纠偏”

### 7.3 tasks 的违宪信号

以下情况视为当前工作区的典型跑偏信号：

- about 文档已经更新，但 `tasks.md` 完全没动
- 正文已经判定完成，但看板仍写未完成
- 新增了大量内容，却没有明确挂到任何任务上
- 只记结果，不记恢复点

---

## 8. about 文档迭代条款

### 8.1 固定写法

继续维护 `.kiro/about/` 三文档时，必须遵守：

- 优先修正旧结论
- 再追加新结论
- 明确写出漂移点
- 明确写出证据来源层级
- 明确区分“当前现实”和“历史设计意图”

### 8.2 触发更新的条件

出现以下任一情况，就应继续回写 about 文档：

1. 找到新的 live 代码入口
2. 找到新的场景 / Prefab / SO 现场证据
3. 发现旧结论和实仓现实不一致
4. 发现三主文档之间出现口径不一致
5. 用户纠正了某条关键判断
6. 某个系统的完成度或风险等级需要重判
7. 某个高频误读点被识别出来

### 8.3 最小自检清单

每轮写完后，至少自检以下问题：

1. 场景数量、主路径、关键资源数量有没有写错
2. 有没有把历史设计误写成当前实现
3. Day1 / 制作台 / 背包容量 / `V3` 命名漂移这些高风险口径是否一致
4. 有没有残留“后续仍要补齐”这类过期占位
5. 三主文档的成熟度表述是否与真实完成度相符

---

## 9. 高风险内容边界

以下对象默认属于高风险配置对象：

- Unity 场景
- Prefab
- ScriptableObject
- Inspector 配置
- 序列化引用

默认处理原则：

- 先审视
- 先给证据
- 先写判断
- 未获明确授权不修改

换句话说：

- 可以核对 `Primary.unity`
- 可以核对 `DialogueValidation.unity`
- 可以核对 `PackagePanel.prefab`
- 可以核对 `MasterItemDatabase.asset`
- 但不能因为看到了问题就顺手改掉

---

## 10. 记忆同步条款

### 10.1 固定顺序

任何产生稳定结论的工作，都必须按以下顺序同步：

1. 当前子工作区 `memory.md`
2. 受影响的父工作区 `memory.md`
3. 当前线程记忆 `memory_0.md`

### 10.2 每次追加至少包含

- 用户目标
- 当前主线目标
- 本轮子任务 / 阻塞
- 已完成事项
- 关键决策
- 涉及文件
- 验证结果
- 恢复点 / 下一步

### 10.3 禁止事项

- 不允许只改 about 文档，不同步 memory
- 不允许只同步父记忆，不同步子记忆
- 不允许把“模型记住了”当作替代方案

---

## 11. 明确禁止事项

在当前工作区内，以下行为一律视为违宪：

1. 未经用户授权新增第四份主报告并试图替代三主文档
2. 用历史 `Docx` 直接覆盖 live 代码与场景结论
3. 把“有代码”写成“已闭环”
4. 把“有 package / 有 asset / 有设计稿”写成“已经可运行”
5. 因为着急收口而跳过 `tasks.md`
6. 因为着急结束而跳过 memory 同步
7. 在未获授权时修改高风险场景或资产配置
8. 回滚或覆盖用户已有改动
9. 为了省事不再修正文档旧结论
10. 把三主文档写成维护日志，失去“后来者可直接接手”的面向

---

## 12. 当前验收标准

当以下条件同时成立时，可视为本工作区一次迭代真正闭环：

1. `tasks.md` 已更新且状态真实
2. 三份主文档已与本轮新证据对齐
3. 三文档之间没有关键口径冲突
4. 关键结论能追溯到代码、场景、Prefab、SO、测试或工作区记忆
5. 已完成子工作区 → 父工作区 → 线程记忆同步
6. 当前恢复点已明确写出，后来者无需再重新讨论方向

---

## 13. 后续接手最小入口

如果未来有新的智能体或新的会话继续维护本工作区，应按以下顺序起手：

1. 读本宪法
2. 读 `tasks.md`
3. 读 `.kiro/about/00_项目总览与阅读地图.md`
4. 读 `.kiro/about/02_当前状态、差异与接手指南.md`
5. 读 `.kiro/about/01_系统架构与代码全景.md`
6. 再下钻对应 `memory.md` 与 live 代码 / 场景 / Prefab / SO

---

## 14. 当前固定恢复点

截至 2026-03-10，本工作区的固定恢复点为：

- `.kiro/about/` 三份主文档已经达到“可接手版”交付要求。
- 后续继续推进的正确方式不是重开方向讨论，而是：
  - 发现新 live 证据
  - 更新 `tasks.md`
  - 修正文档旧结论
  - 补充新结论
  - 同步多层 memory

这就是当前工作区的长期稳定运行方式。
