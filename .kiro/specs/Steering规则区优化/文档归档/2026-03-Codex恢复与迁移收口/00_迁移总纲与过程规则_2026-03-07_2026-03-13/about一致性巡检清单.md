# about 一致性巡检清单

**版本**: v1.0  
**创建日期**: 2026-03-11  
**定位**: 给后续维护 `.kiro/about/00/01/02` 的人或智能体一个固定巡检顺序，避免再次出现“看板先于正文”“搜证已做但文档没落盘”“把显示乱码误判成文件损坏”。

---

## 1. 使用时机

以下情况应直接使用本清单：

- 继续维护 `.kiro/about/` 三主文档
- 发现三文档之间可能口径不一致
- 新读了代码 / 场景 / Prefab / ScriptableObject 后，需要判断是否回写
- 怀疑某份 `memory.md` 或 `memory_*.md` 出现乱码
- 准备把本轮分析正式收口到 `tasks.md` 和多层 `memory`

如果本轮没有形成新的稳定结论，不要为了“过清单”而硬改文档。

---

## 2. 起手前检查

1. 确认当前主线仍是 `.kiro/about/` 文档工程，而不是中途某个支撑子任务。
2. 先读：
   - `工作区宪法.md`
   - `tasks.md`
   - `00_项目总览与阅读地图.md`
   - `02_当前状态、差异与接手指南.md`
   - `01_系统架构与代码全景.md`
3. 明确本轮属于哪一类：
   - 补新证据
   - 修旧结论
   - 做一致性巡检

4. 记录当前恢复点：
   - 主线目标是什么
   - 本轮子任务是什么
   - 完成后要回到哪一步

## 7. `memory` 乱码巡检

### 7.1 判断原则

- 不要把 PowerShell 裸 `Get-Content` 的显示结果直接当成文件本体编码结论。
- 先判断是不是“显示乱码”，再判断是不是“文件真的坏了”。

### 7.2 建议核对方式

先用 UTF-8 字节读取核对真实内容：

```powershell
$path = Resolve-Path '.kiro/specs/Steering规则区优化/memory.md'
[System.Text.Encoding]::UTF8.GetString([System.IO.File]::ReadAllBytes($path))
```

### 7.3 真损坏的典型信号

- 会话标题或字段名变成 `### �...`
- `tasks` 被制表符吃掉首字母
- `ahead 4` 被控制字符污染成 `head 4`
- 同一份文件里只有局部会话块坏掉，而不是整文件统一编码

### 7.4 收口要求

- 修完后复扫全部 `memory*.md`
- 确认没有：
  - 替换符乱码块
  - 异常控制字符
  - `tasks` / `ahead` 等局部吃字问题

## 8. 收尾前自检

在准备结束本轮前，至少确认以下 8 项：

1. 本轮新增结论是否真的有证据支撑
2. 旧结论若被推翻，是否已先修正文而不是只追加新话
3. `00 / 01 / 02` 是否仍互相对得上
4. `tasks.md` 是否反映了最新真实状态
5. 子工作区 `memory.md` 是否已追加
6. 父工作区 `memory.md` 是否已追加
7. 线程记忆是否已追加
8. 如果这轮涉及乱码判断，是否已做 UTF-8 字节复核，而不是只看终端显示

---

## 9. 一句话执行口径

先核事实，再修旧结论，再补新证据，最后同步 `tasks + 子 memory + 父 memory + 线程 memory`；任何一步超前，都会重新制造漂移。
