# OpenClaw 终端闪窗排查与止血报告

## 1. 报告目的

给后续会话一个可直接复用的结论：本次桌面反复出现的“终端窗口一闪而过”问题，真实根因是什么，如何定位，如何止血，以及主线该恢复到哪里。

## 2. 现象与用户反馈

### 2.1 现场现象

- Windows 桌面持续出现可见终端窗口一闪而过。
- 该问题会明显打断用户操作，已经构成工具层阻塞。

### 2.2 用户关键反馈

> “现在屏幕一直不断出现莫名其妙的终端窗口一闪而过，是你的问题？？？”

> “现在已经没有任何弹窗情况出现了，你做的很棒，应该就是你关掉后的结果。”

## 3. 最终结论

### 3.1 根因结论

本次持续闪窗的真实根因，不是 OpenClaw 网关整体不可用，也不是当前 `playwright` MCP 还在自动重启，而是：

- `D:\1_AAA_Program\OpenClaw\openclaw.mjs gateway run --verbose`
- 在 Windows 上触发了 Bonjour / 局域网探测分支
- 该分支会周期性拉起可见的 `cmd.exe`
- 典型命令为：

```powershell
cmd.exe /d /s /c "arp -a | findstr /C:\"---\""
```

也就是说，真正导致“闪窗”的，是 OpenClaw 网关的 Bonjour 探测子进程，而不是主网关监听本身。

### 3.2 已确认排除的旧来源

此前另一个已知弹窗来源是 Codex 自动拉起：

```toml
[mcp_servers.playwright]
command = "npx"
```

对应的 `npx @playwright/mcp@latest` 在 Windows 上会弹出可见控制台。

但本轮复核时已确认：

- `C:\Users\aTo\.codex\config.toml` 中 `playwright` MCP 自动启动项仍为禁用状态
- 当前闪窗已不是这条旧来源

## 4. 排查过程

### 阶段 A：先排除 Codex 自身旧来源

先复核：

- `C:\Users\aTo\.codex\config.toml`

确认结果：

- `[mcp_servers.playwright]` 已保持注释禁用
- 当前 Codex 侧仅保留：
  - `[features] rmcp_client = true`
  - `[mcp_servers.unityMCP] url = "http://localhost:8080/mcp"`

这一步的意义是先切掉“旧已知问题”的干扰，避免误判。

### 阶段 B：现场抓进程链

随后对以下进程做现场采样：

- `cmd.exe`
- `conhost.exe`
- `powershell.exe`
- `node.exe`

抓到的关键链路为：

- 父进程：`node.exe`，实际命令行为 `D:\1_AAA_Program\OpenClaw\openclaw.mjs gateway run --verbose`
- 子进程：`cmd.exe /d /s /c "arp -a | findstr /C:\"---\""`

这一步把“谁在弹窗”从猜测变成了证据。

### 阶段 C：回看 OpenClaw 本地源码

继续读取本地源码：

- `D:\1_AAA_Program\OpenClaw\src\infra\bonjour.ts`
- `D:\1_AAA_Program\OpenClaw\src\gateway\server-discovery-runtime.ts`

得到的关键结论：

- `OPENCLAW_DISABLE_BONJOUR=1` 可以关闭 Bonjour 广播 / 探测路径

这意味着不需要停掉整个 OpenClaw，也不需要粗暴卸载，只要切掉问题分支即可。

## 5. 采取的处理动作

### 5.1 止血策略

采用“最小止血”而不是“整体停服”：

- 保留 OpenClaw 网关
- 保留 `18789` 端口能力
- 仅禁用 Bonjour 探测分支
- 同时改为隐藏启动，避免再弹出可见控制台

### 5.2 处理后的运行状态

处理后确认到：

- OpenClaw 网关进程：`node.exe` PID `8088`
- 监听端口：
  - `127.0.0.1:18789`
  - `[::1]:18789`

对应日志文件：

- `D:\1_AAA_Program\OpenClaw\openclaw-gateway.stdout.log`
- `D:\1_AAA_Program\OpenClaw\openclaw-gateway.stderr.log`

## 6. 验证结果

### 6.1 现场验证

止血后连续监控，未再抓到：

- `openclaw -> cmd.exe`
- `openclaw -> conhost.exe` 的新可见弹窗链

同时复核到：

- `playwright` MCP 没有重新被自动拉起
- `18789` 仍正常监听

### 6.2 用户确认

用户最终反馈：

> “现在已经没有任何弹窗情况出现了，你做的很棒，应该就是你关掉后的结果。”

因此本轮可以视为：

- 闪窗阻塞已解除
- 当前处理已在用户现场得到正向确认

## 7. 对主线的影响判断

这不是换线，只是主线下的工具阻塞处理。

当前主线仍然是：

```text
Unity -> MCP -> Codex
```

也就是继续完成 `CoplayDev/unity-mcp` 的接入与验证闭环。

### 当前主线恢复点

闪窗问题止血后，主线恢复到这里：

1. 继续确认 Unity 侧 `CoplayDev/unity-mcp` 是否真正启动
2. 当前 `http://localhost:8080/mcp` 仍未监听
3. 已知阻塞不是项目代码，而是 Unity MCP 依赖未齐
4. 需要先补齐：
   - Python 3.10+
   - `uv`
5. 然后回到 Unity 面板 `Refresh` / `Start Server`
6. 再继续 `sunset-unity-validation-loop`

## 8. 给后续会话的可复用操作顺序

如果以后再次出现“终端窗口一闪而过”，建议按这个顺序处理：

1. 先查 `C:\Users\aTo\.codex\config.toml`，排除 `playwright` MCP 自动启动
2. 再抓 `cmd.exe` / `conhost.exe` / `node.exe` 父子链
3. 若父进程是 `openclaw.mjs gateway run --verbose`
4. 优先检查 Bonjour 探测是否被关闭
5. 采用“保留网关，仅禁用 Bonjour”的最小止血策略
6. 止血后一定再验证端口监听与用户现场反馈
