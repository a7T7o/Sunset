# Codex客户端线程承接失配复盘与整改计划

**日期**: 2026-03-11  
**定位**: 本文档用于正面复盘本轮 `Sunset` Git / worktree 治理为什么仍没有解决用户真正的使用痛点，并给出后续真正要完成的底层整改方向。  
**目标**: 不再把问题停留在“规则已经补齐”或“Git 已经治理完”，而是明确：要让 `Codex` 真正好用，必须把客户端线程/UI 对 worktree 的承接体验也闭环起来。

---

## 1. 先给出总判断

本轮我前半段最大的误判，不是某个 Git 命令打错，而是**把“Git/worktree 已理顺”误当成了“用户体验已闭环”**。

这导致我虽然完成了这些事：

- 把 `main`、`codex/npc-generator-pipeline`、`codex/farm-10.2.2-patch002` 分开；
- 给 `NPC` 和 `farm` 建了独立 worktree；
- 把线程分支对照表、使用说明、Git 安全基线都落进了项目；

但用户仍然会遇到这些核心痛点：

- 点进旧线程后，界面又像回到了 `main`；
- 新 worktree 明明存在，但客户端看上去像“无线程”；
- 线程标题、底部分支提示、真实目录、真实 Git 分支不一致；
- 最终体感仍然是“没接上”，工作被迫停摆。

**所以这轮真正没有闭环的，不是 Git 文件层，而是 Codex 客户端线程/UI 承接层。**

---

## 2. 我这轮到底错在哪

## 2.1 错误一：我把“房子修好了”误当成“人已经住进去了”

我解决的是：

- Git 分支怎么隔离；
- 文件应该放在哪个 worktree；
- 哪些线程理论上应该归哪条功能线。

但用户实际卡住的是：

- 点这个线程后，Codex 客户端到底把我带到哪里；
- 左侧线程、当前工作区、底部分支提示为什么互相打架；
- 为什么真实文件在 NPC worktree，但界面还像在根仓库。

也就是说，我修的是“文件和分支的房子”，用户卡的是“Codex 客户端到底认不认这间房子”。

## 2.2 错误二：我一度把 `.codex/threads/...` 看得太重

`.codex/threads/.../memory_0.md` 很重要，它是我们自己定义的线程记忆层。  
但它不是 Codex 客户端 UI 线程的唯一真源。

真正影响客户端线程承接体验的，还包括：

- `C:\Users\aTo\.codex\state_5.sqlite`
- `C:\Users\aTo\.codex\session_index.jsonl`
- `C:\Users\aTo\.codex\.codex-global-state.json`
- Electron 本地缓存与当前激活工作区状态

如果这几层没闭环，只靠 `.codex/threads/...` 和记忆规则，是不可能让客户端体验稳定下来的。

## 2.3 错误三：我过早把“治理完成”说得太满

正确说法应该是：

- **Git / worktree 治理已大幅推进**
- **线程记忆与项目规则已初步稳定**
- **但客户端线程/UI 承接还没有彻底打通**

如果把前两者直接说成“现在已经完全好了”，那就会和用户实际使用体验直接冲突。

## 2.4 错误四：我在重心上仍然偏向“让用户理解”，而不是“让我自己把底层继续修完”

用户现在缺的不是再来一份“你该怎么判断”的教程。  
用户真正要的是：

- 让我承认问题还在；
- 让我把后续要修的底层问题讲明白；
- 让我自己列出整改任务，而不是继续把判断负担交还给用户。

这一点用户的指正完全正确。

## 3. 现在已经确认的真实技术模型

当前必须用四层模型理解 `Sunset` 的线程混乱问题：

## 3.1 第一层：Git 真实分支层

这是最底层的代码事实。

- `D:\Unity\Unity_learning\Sunset` → `main`
- `D:\Unity\Unity_learning\Sunset_worktrees\NPC` → `codex/npc-generator-pipeline`
- `D:\Unity\Unity_learning\Sunset_worktrees\farm-10.2.2-patch002` → `codex/farm-10.2.2-patch002`

这一层目前已经基本理顺。

## 3.2 第二层：worktree 物理目录层

这是“哪条业务线看哪份文件”的物理落点。

例如 NPC 当前真实文件就在：

- `D:\Unity\Unity_learning\Sunset_worktrees\NPC\Assets\Editor\NPCPrefabGeneratorTool.cs`
- `D:\Unity\Unity_learning\Sunset_worktrees\NPC\.kiro\specs\002_After_26.1.21\NPC系统\0_NPC生成器与预制体流水线`
- `D:\Unity\Unity_learning\Sunset_worktrees\NPC\.codex\threads\Sunset\NPC\memory_0.md`

这一层目前也已经基本理顺。

## 3.3 第三层：Codex 线程实体元数据层

这里主要由以下对象构成：

- `state_5.sqlite`
- `session_index.jsonl`

这一层当前仍然存在明显漂移：

- 同一个 NPC 线程 id，`cwd` 已对到 NPC worktree；
- 但 `git_branch` 还可能留着旧值 `main`；
- 同一个线程 id 甚至可能出现多个历史标题，如 `Interpret malformed prompt` 和 `NPC`。

这说明：**线程实体是真实存在的，但它的元数据并不总可靠。**

## 3.4 第四层：Codex 客户端 UI / 激活工作区 / 缓存层

这里主要由以下对象参与：

- `.codex-global-state.json`
- Electron 当前工作区根
- 本地缓存 / 会话恢复行为

当前已经确认：

- 保存过的工作区根里包含 NPC worktree；
- 但当前激活工作区仍可能停留在 `D:\Unity\Unity_learning\Sunset`；

这就会造成一个非常典型的用户体验错位：

- 真实业务已经在 NPC worktree；
- 但界面当前主语境还像在 Sunset 根仓库；
- 用户点来点去时，会觉得所有线程和分支都乱了。

## 4. 必须吸收的“对话丢失修复”经验

`D:\迅雷下载\开始\.codex\threads\系统全局\对话丢失修复` 这条线，已经给了我几个非常重要的经验，我必须明确继承，而不能再重犯。

## 4.1 经验一：不要动聊天原文，先修“状态投影层”

那条线已经证明：

- 真正的聊天原文在 `sessions` / `archived_sessions`
- 很多“丢线程”“无线程”“显示错乱”问题，本质不是原文丢了
- 而是索引、状态库、provider、工作区根、缓存这些“投影层”出了问题

这对当前 Sunset 的启发是：

**不要把“线程承接错乱”继续理解成 Git 或 memory 问题，真正要修的是客户端状态投影。**

## 4.2 经验二：`run-once` 只能补救，守护器才是长期方案

对话丢失修复线已经验证过：

- 单次修复可以把状态拉正；
- 但 Codex 运行态可能又把状态写歪；
- 所以只有“守护器持续纠偏”才能形成真正稳定的兼容层

这直接推导出当前 Sunset 的下一步：

**如果要把线程/UI 与 worktree 真正接上，不能只靠一次性改库或一次性切目录，必须有长期的状态纠偏层。**

## 4.3 经验三：任何修库前都必须先备份，并且保持可回滚

对话丢失修复线已经形成了很清晰的底线：

- 先备份 `state_5.sqlite`
- 明确哪些字段可修、哪些字段不能乱动
- 不直接改聊天 `.jsonl` 原文

这套底线必须继续沿用到后续 Sunset 的客户端闭环整改。

## 4.4 经验四：要分清“根因层”和“表现层”

那条线真正可借鉴的不是某个具体脚本，而是分析方式：

- 先分清是数据库层、索引层、provider 层、缓存层还是进程层；
- 再决定修哪一层；
- 不要把表层症状都归给同一个原因。

当前 Sunset 的线程错乱也必须照这个方法走。

## 5. 现在我对“后续要修什么”的重新定义

后续目标不应该再写成笼统的“继续完善 Git 治理”。  
正确目标应该是：

**在不修改 `WindowsApps` 下 Codex 安装本体的前提下，补齐本机状态层、线程元数据层、工作区激活层与缓存层的兼容闭环，让 Sunset 的长期线程能稳定承接到正确 worktree。**

这句话里有两个边界必须说死：

## 5.1 我能修的

- 本地状态库
- 本地索引
- 本地全局状态文件
- 本地守护器 / 兼容层
- 本地工作区激活与线程路由策略
- 项目内线程-分支-worktree 映射

## 5.2 我不能承诺直接修的

- `WindowsApps` 里的 Codex 客户端安装包本体
- 官方未暴露的产品内部路由逻辑
- 纯粹由客户端 UI 自己决定但不经本地状态层落盘的黑箱行为

所以“把底层的 Codex 修好”在当前可控范围内，应该落成：

**把可控的本地兼容层做完整，而不是声称我能直接改 Codex 产品内核。**

## 6. 后续必须执行的整改任务清单

下面这份清单不是给用户做的，而是给我自己后续执行的。

## A. 真相建模阶段

### A1. 建立唯一的线程承接真相表

必须把以下字段统一映射到同一张审计表里：

- 线程 id
- `session_index.jsonl` 当前线程名
- `state_5.sqlite` 当前标题
- `state_5.sqlite` 当前 `cwd`
- `state_5.sqlite` 当前 `git_branch`
- 当前激活工作区根
- 线程记忆路径
- 线程应归属的项目 / worktree / 分支

目的：

- 不再靠口头判断线程属于哪里；
- 让每个线程当前到底“在哪里、应该在哪里、为什么不一致”一眼可查。

### A2. 定义 Sunset 的机器可读线程路由映射

当前只有 Markdown 对照表还不够。  
后续需要再有一份机器可读映射源，例如：

- `thread_id -> expected workspace root`
- `thread_id -> expected branch`
- `thread_id -> expected project thread memory path`

目的：

- 让后续脚本和守护器能够自动核验，不再只靠人工读文档。

## B. 状态审计阶段

### B1. 新建客户端状态审计脚本

需要一份新的审计脚本，专门检查：

- `state_5.sqlite`
- `session_index.jsonl`
- `.codex-global-state.json`
- 当前各 worktree 真实分支

输出内容必须直接回答：

1. 哪些线程是完全对齐的；
2. 哪些线程 `cwd` 对了但 `git_branch` 漂了；
3. 哪些线程工作区根已保存但当前 UI 激活根不对；
4. 哪些线程标题存在历史混乱。

### B2. 明确哪些字段允许被兼容层纠偏

需要把字段分成三类：

- 可自动纠偏：如 `cwd` 规范化、工作区根映射
- 谨慎纠偏：如 `git_branch`
- 默认不动：如聊天原文、线程历史事件流

目的：

- 避免后续为了“看起来顺眼”而把线程历史语义抹掉。

## C. 兼容层扩展阶段

### C1. 不再把 `provider-bridge` 只当 provider 工具

当前 `provider-bridge` 已经解决了：

- `cwd` 规范化
- `model_provider` 桥接
- `has_user_event` 回补

但它还没有解决：

- `git_branch` 漂移
- 激活工作区根与线程默认目录失配
- 线程标题 / 线程名 / 路由信息多源漂移
- worktree 建好后，客户端 UI 仍停在旧根目录

后续必须决定两条路之一：

1. 扩展现有 `provider-bridge`，让它升级成更广义的 `Codex state bridge`
2. 新建第二层专用桥，例如 `thread-workspace-bridge`

### C2. 引入“守护式线程路由纠偏”

后续不能只靠一次性写库。  
必须做成：

- 发现线程状态漂移 → 自动比对映射 → 仅修允许修的字段 → 记录日志与备份

否则用户一重开客户端、切一次工作区、恢复一次线程，状态又会重新漂。

## D. 客户端激活层修复阶段

### D1. 处理“保存过工作区根”与“当前激活工作区根”不一致

当前已经确认这是导致 UI 继续像在 `Sunset/main` 的一个关键原因。  
后续要补一套策略，用来处理：

- 线程默认 `cwd` 在 NPC worktree；
- 但客户端当前激活工作区根还是 Sunset 根目录；
- 这种情况下到底如何安全桥接、何时桥接、桥接后如何验证。

### D2. 评估缓存刷新与最小重载策略

如果状态层已经正确，但 UI 仍不跟，后续必须进一步区分：

- 是状态层还没闭环；
- 还是需要最小缓存刷新 / 重启客户端 / 重载工作区。

这一层必须做成有证据的判断，而不是再次靠猜。

## E. 验证阶段

### E1. 为 Sunset 定义三类线程的闭环验收

至少要验证三类线程：

1. 治理类线程：如 `Codex规则落地`
2. 功能类线程：如 `NPC`
3. 功能类线程：如 `农田交互修复V2`

每类线程都要验证：

- 打开线程后当前目录是否正确；
- 当前分支是否正确；
- 当前工作区根是否正确；
- 文件视图是否对应正确 worktree；
- 是否还会误回 `main`。

### E2. 验证“旧线程恢复”与“新线程新建”两条路径

因为这两条路径很可能不是一个逻辑：

- 旧线程恢复时，会吃历史状态；
- 新线程新建时，会吃当前激活工作区；

如果只验证其中一条，后续还会继续出乌龙。

## F. 交付阶段

### F1. 输出“客户端闭环已完成/未完成清单”

后续每推进一轮，都必须明确汇报：

- 已彻底闭环的层
- 仍未闭环的层
- 当前 workaround 是什么
- 还有哪些是 Codex 本体黑箱，暂时只能规避

### F2. 输出回滚口径

任何涉及本地状态库、索引、全局状态、缓存策略的修复，都必须给出：

- 修改前备份位置
- 回滚命令或回滚路径
- 失败时如何恢复到修复前状态
