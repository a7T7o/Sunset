# Sunset 文件认领与锁定表（2026-03-15 首版）

## 1. 使用说明
- 本表用于解决“谁正在改什么、谁当前只读、哪些路径改动前必须先同步”的问题。
- 认领状态定义：
  - `占用中`：未经 owner 同步，不直接改
  - `协同改动`：涉及多线程，改前必须在统一快照里留痕
  - `共享只读`：允许读取，不默认改动
  - `待释放`：本轮 checkpoint 后可重新评估归属

## 3. 当前执行口径
- 没在本表里、且属于高风险共享路径的文件，不应由任何线程“默认顺手改”。
- 认领状态为 `占用中` 或 `协同改动` 的路径，必须先在统一快照里说明再改。
- 后续若某线程完成 checkpoint，应同步更新本表，把路径状态从 `占用中` 调整为 `待释放` 或 `共享只读`。

## 4. 2026-03-16 锁转交补充口径
- A 类共享热文件的写锁转交，判断单位改为“文件”，不是“整个工作树”。
- `只读复工` 不触发写锁转交，因此不要求先把整个 shared worktree 清空。
- 但如果下一线程要继续写同一个 A 类文件，上一 owner 在该文件上的改动必须先满足以下三种状态之一：
  - `Committed`：该文件改动已白名单固化到上一线程自己的任务分支；
  - `Parked`：该文件改动已导出为 patch / 快照等可回放证据，且 shared worktree 中该文件已恢复到可安全交接状态；
  - `Abandoned`：该文件上的本轮改动已明确放弃并恢复。
- 明确禁止 `Mixed-in-place`：上一线程把同一热文件的 dirty 留在 shared worktree 中，下一线程再直接接着写。
- 当前直接裁决：
  - `spring-day1` 的首批 checkpoint 已通过，`Primary.unity` 与 `DialogueUI.cs` 两把锁也已释放；
  - 但这两份文件在后续若要转交新的“写锁”，仍必须先通过上述文件级固化/停车检查；
  - `farm` 当前只允许执行“零写入、只读复工 checkpoint”；如果它后续申请写 `Primary.unity`，治理线程必须先检查 `spring-day1` 在该文件上的上一轮改动是否已 `Committed`、`Parked` 或 `Abandoned`。
