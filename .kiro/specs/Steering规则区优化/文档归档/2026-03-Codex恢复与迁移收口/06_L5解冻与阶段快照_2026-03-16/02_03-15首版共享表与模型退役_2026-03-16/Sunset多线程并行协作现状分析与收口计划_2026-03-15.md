## 4. 当前最关键的问题模型

## 4.1 不是“每个线程不会工作”，而是“线程之间没有共享状态层”

当前规则已经解决了以下问题：

- 默认活文档入口在哪里；
- `main` / `codex/xxx` / worktree 在 Git 层怎么分工；
- 白名单同步怎么做；
- 哪些改动不能直接在 `main` 上收尾。

但当前规则没有解决以下问题：

- 某个线程开始修改 `Assets/YYY_Scripts/Story/UI/DialogueUI.cs` 时，其他线程如何立刻知道；
- 某个线程把项目改到临时红编译时，其他线程如何知道“现在不要继续推进运行态验收”；
- 某个线程只看到自己那一份局部现场时，如何知道自己的汇报是否已经过期；
- 某个文件当前究竟属于“默认基线”“线程私有 WIP”“待合并 checkpoint”中的哪一类。

这就是当前最核心的结构性缺口：  
**没有共享状态层，就只能靠线程自己记忆和用户手动转述。**

---

## 4.2 当前不存在“线程锁 / 文件认领 / 解锁”机制

本轮核查结果：

- 现行活文档目录中，没有线程级锁定规则；
- `.kiro/steering` 中，没有当前有效的跨线程占用/解锁规范；
- `scripts/git-safe-sync.ps1` 只处理：
  - 分支合法性
  - dirty 分类
  - 白名单同步
  - `main` / `codex/xxx` / worktree 进入边界
- 它不处理：
  - 文件认领
  - 线程占用登记
  - 互斥编辑预警
  - 过期锁释放

因此当前真实定性必须写死为：

**Sunset 现在有 Git 安全基线，但没有线程协作锁基线。**

## 4.4 当前线程汇报会快速过期，不能再被当作最终真相

本轮已经出现一个直接证据：

- `spring-day1` 汇报里曾写“当前有 4 条真实编译错误”；
- 但本轮我直接用 Unity MCP 读取 Console，结果为 `0` 条日志；
- 这说明在线程并行推进期间，线程汇报的时间窗口极短，稍后就可能被后续提交或现场刷新覆盖。

因此必须明确：

**线程汇报不能再直接当成最终基线，只能当成“某一时刻、某一线程看到的局部快照”。**

后续必须新增一个统一动作：

- 以当前 `HEAD` 为锚点；
- 让所有活跃线程按同一模板回报；
- 将它们压成一次总汇总快照；
- 再基于该快照判断是否允许进入正常开发。

## 4.6 当前回复模板重要，但不是第一优先级

统一回复模板当然仍然重要，但它现在是第二层问题。

### 第一优先级

- 共享状态层
- 锁 / 认领机制
- 当前唯一规则冲突收口
- 活跃线程汇总快照

### 第二优先级

- 统一回复模板
- 统一“事实 / 推断 / 待验证 / 用户动作 / 验收点”表达方式

原因很简单：

- 如果没有共享状态层，再好的模板也只是让每个线程更整洁地汇报局部真相；
- 只有先把共享状态层做出来，模板才真正有落点。

## 5. 当前项目应如何重新分层理解

为了后续收口，当前建议把 `Sunset` 多线程协作结构拆成 4 层：

### A 层：默认主项目基线层

只回答以下问题：

- 当前默认开发目录是什么；
- 默认 Git 入口是什么；
- 当前唯一状态说明是什么；
- 哪些内容属于稳定基线；
- 哪些内容属于公共规则。

### B 层：线程协作共享层

这是当前缺失的层，后续必须新建。它至少要回答：

- 当前有哪些活跃线程；
- 每个线程当前工作在哪个目录 / 分支 / commit；
- 每个线程当前认领了哪些路径；
- 哪些文件正在被谁改；
- 哪些 blocking 是项目级，哪些只是线程局部；
- 哪些工作已经完成，哪些只是待合并 checkpoint。

### C 层：线程私有工作层

每个线程自己的：

- memory
- 局部计划
- 局部实现
- 局部审计
- 局部试验

### D 层：最终汇总验收层

只在某一轮准备收口或允许恢复常态开发时使用，回答：

- 当前是否可编译；
- 当前是否有阻断性 Console 错误；
- 当前场景关键链路是否可用；
- 当前剩余 dirty 是否都已归属；
- 当前是否允许“恢复正常开发”。

## 6. 当前最应该做的事情顺序

## 阶段 1：先做一次彻底的线程总同步

目标不是继续改代码，而是让所有活跃线程在同一时间窗口内交一次统一格式的状态快照。

本阶段应覆盖至少这些线程：

- `Codex规则落地`
- `spring-day1`
- `农田交互修复V2`
- `NPC`
- `导航检查`
- `遮挡检查`
- `项目文档总览`
- `skills和mcp`（如果当前仍在活跃）

本阶段交付物应是：

- 一次统一时间窗的线程汇总快照
- 每个线程当前目录 / 分支 / HEAD / 改动范围 / 阻断 / 下步 / 验收点

### 阶段 1 验收标准

- 所有活跃线程都按同一模板汇报；
- 每条汇报都锚定同一类事实字段；
- 能一眼看出哪些问题是全局阻断，哪些只是局部线程问题。

---

## 阶段 2：收口唯一规则冲突

重点是把当前互相打架的规则统一掉。

必须收口的对象：

- `Sunset当前唯一状态说明_2026-03-13.md`
- `Sunset Git系统现行规则与场景示例_2026-03-15.md`
- `线程分支对照表.md`
- `线程分支对照表.json`

这一步必须明确回答：

1. `NPC/farm` 现在到底是不是默认回到主项目目录；
2. 如果真实任务必须先开 `codex/xxx` 分支，那么默认进入 `main` 的准确含义是什么；
3. 线程映射表是否还需要保留 feature thread -> worktree 的旧口径；
4. 哪些线程是治理/审计线程，哪些线程是实现线程；
5. 当前长期线程到底默认绑定根目录还是绑定分支还是绑定 worktree。

### 阶段 2 验收标准

- 不再存在两份同时生效、但互相矛盾的线程入口规则；
- 任何一个线程启动时，都能从文档里得到唯一一致的起点。

---

## 阶段 3：建立共享协作状态层

这是当前真正缺失、但最需要补的机制。

建议至少建立两份共享文档或共享数据：

### 3.1 活跃线程总表

字段至少包括：

- 线程名
- 类型：治理 / 实现 / 审计 / 只读
- 当前工作目录
- 当前分支
- 当前基线提交
- 当前状态：进行中 / 阻塞 / 待汇总 / 待验收 / 已完成未合并
- 当前主线目标
- 当前阻断

### 3.2 文件认领 / 锁定表

字段至少包括：

- 路径或路径根
- 当前认领线程
- 认领原因
- 认领时间
- 预计释放条件
- 当前状态：占用 / 可共享只读 / 已释放待合并
- 冲突处理规则

### 这一步要解决的问题

- 一个线程修改业务脚本时，其他线程不再靠猜；
- 一个线程临时改到红编译时，其他线程知道应暂停哪类动作；
- 一个线程只做只读审计时，不需要误伤其他线程；
- 用户不再需要手工在对话里充当状态总线。

### 阶段 3 验收标准

- 当前所有活跃线程都已登记；
- 当前所有高风险文件/目录都有明确归属；
- 新线程启动前必须先看总表和认领表。

---

## 阶段 4：把当前工作树 dirty / WIP 彻底归属

这一步不一定要求立即清空工作树，但要求：

- 每一项 dirty / untracked 都知道是谁的；
- 每一项都知道属于：
  - 默认基线
  - 线程私有 WIP
  - 待提交 checkpoint
  - 本地噪音 / Git 外对象

至少需要覆盖：

- `Primary.unity`
- `DialogueUI.cs`
- `DialogueChinese Pixel SDF.asset`
- `Assets/100_Anim/NPC/`
- `Assets/222_Prefabs/NPC/`
- `Assets/Sprites/NPC/*.meta`
- 各线程 memory 变更

### 阶段 4 验收标准

- 当前工作树中不存在“未知归属”的 dirty；
- 没有线程再因为“别人刚好也动了这个文件”而被动中断但又不知道原因。

---

## 阶段 5：建立统一汇总验收模型

后续必须不再用“某线程说完成了”来代表整体完成。

建议统一分 5 层验收：

### L1 文档与规则层

- 现行规则唯一；
- 活跃线程总表存在；
- 认领表存在；
- 线程入口不冲突。

### L2 Git 与现场层

- 当前工作树 dirty 已全部归属；
- 当前默认基线与线程私有 WIP 分界明确；
- 分支 / worktree 使用口径唯一。

### L3 编译与控制台层

- 项目无新的项目级红编译；
- Console 无新的阻断性错误；
- warning 已分类，不再混写成“项目没恢复”。

### L4 Live 场景与交互层

- MCP 可稳定读取场景；
- 关键主场景链路存在；
- 关键组件、资源、引用缺口已补齐或已明示。

### L5 正常开发准入层

只有当前面四层都达到最低标准后，才允许正式写“进入正常开发”。

- 也不是“完全收尾完成”

而是：

**恢复已基本完成，但多线程协作基线尚未建立完成的过渡期。**

在这个阶段里，最重要的不是继续堆新开发，而是先把：

1. 共享状态层；
2. 唯一规则口径；
3. 线程认领机制；
4. 汇总验收模型；

四件事补上。

只有这四件事成立后，后续的统一模板、常态开发、并行推进，才不会再次把项目拖回“用户充当中间总线”的状态。

---

## 10. 最终验收标准

只有当以下条件同时成立，才能正式写：

**“Sunset 已完成多线程治理收尾，允许恢复正常并行开发。”**

### 10.1 规则唯一

- 当前唯一状态说明、Git 规则、线程映射表、活跃线程总表完全一致

### 10.2 共享层存在

- 活跃线程总表已建立
- 文件认领 / 锁定表已建立

### 10.3 当前现场可解释

- 当前所有 dirty / untracked 都已归属
- 没有未知来源的阻断改动

### 10.4 当前项目可验证

- 项目无新的项目级红编译
- Console 无新的阻断性错误
- MCP 能读 live 场景与控制台

### 10.5 当前线程可接轨

- 所有活跃线程都已按统一模板完成一次同步汇报
- 每条线程都知道自己当前主线、当前阻断、下步动作、退出条件

### 10.6 用户可脱离中间转述

- 用户不再需要人工把 A 线程的话转给 B 线程才能维持协作

达到以上六条，才算真正进入正常开发。
