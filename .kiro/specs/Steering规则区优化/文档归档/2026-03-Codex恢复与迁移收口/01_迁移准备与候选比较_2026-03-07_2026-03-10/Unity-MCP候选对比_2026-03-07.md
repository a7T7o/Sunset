# Unity MCP候选对比（2026-03-07）

> 适用前提：当前项目 Unity 版本为 `6000.0.62f1`。
>
> 目标：在不影响当前业务推进的前提下，为 Sunset 项目选出更稳定、更适合长期工作流的 Unity MCP 方案。

---

## 1. 当前判断

当前正在使用的 `mcp-unity` **可以凑合使用，但不适合作为长期主力**。

原因不是“完全不可用”，而是它在当前项目里的关键场景已经暴露出稳定性问题：

- 读取 Unity 资源列表能枚举，但关键资源读取会出现超时或连接关闭。
- 我本轮实测读取 `packages`、`scenes_hierarchy`、`tests/EditMode` 时出现：
  - `Request timed out`
  - `Connection closed`
- 这会直接破坏你最看重的链路：
  - 场景审视
  - 控制台核查
  - EditMode tests
  - 自动验证闭环

因此当前优先级不是“继续忍着用”，而是：

1. 选出更靠谱的候选
2. 设计并行试装与回退方案
3. 用真实 Sunset 场景验证后再替换主力

---

## 2. 选型标准

本次不是按“功能越多越好”选，而是按 Sunset 项目真实痛点选。

### P0 标准

1. **连接稳定性**
   - 编辑器重载后不易断
   - 测试 / 编译 / 日志调用不易挂
2. **只读能力强**
   - 场景层级
   - 组件字段
   - 资源/测试/日志
3. **Unity 6000 兼容性**
4. **适合 Codex / Claude / Cursor 这类 MCP 客户端**

### P1 标准

1. **批量操作能力**
2. **可扩展为 Sunset 专用工具**
3. **对场景/Prefab/脚本工作流友好**

### P2 标准

1. 视觉理解
2. 反射/方法调用
3. 高级编辑器工具生态

---

## 3. 候选清单总览

| 候选 | 适配 Unity 6000 | 优势 | 风险 | 当前建议 |
|---|---|---|---|---|

## 4. 各候选分析

### 4.1 `CoplayDev/unity-mcp`

#### 选择理由

这是当前最适合作为 **Sunset 主力 Unity MCP 候选** 的方案。

#### 优点

1. 明确强调稳定性增强，公开更新点包括：
   - stale connection detection
   - tool state persistence across reloads
   - IPv6/IPv4 loopback 修复
   - domain reload resilience
2. 支持批量执行（`batch_execute`），适合后续把 Sunset 的“多步审视/验证”收束为低轮次调用。
3. 覆盖范围较完整：
   - 场景
   - Prefab
   - 脚本
   - UI
   - 测试
   - 控制台
   - 资源
4. 对你当前最关键的“稳定读 + 稳定验证”问题，方向最对口。

#### 风险

1. 需要真实试装后确认：
   - 在 `6000.0.62f1` 下的 reload 稳定性
   - EditMode tests 的连续调用稳定性
2. 需要确认是否足够适配你这种“场景审视先于修改”的工作方式。

#### 结论

**最推荐先试装。**

---

### 4.2 `IvanMurzak/Unity-MCP`

#### 选择理由

这是当前最适合作为 **扩展型底座** 的候选。

#### 优点

1. 功能覆盖非常广，包含大量 Unity 操作、反射、脚本、资源、测试相关能力。
2. 最大价值不是“现在就全用”，而是它支持进一步做项目专用 MCP Tool。
3. 对 Sunset 这类规则重、工作区重、验证重的项目，后续可扩展出：
   - 场景审视工具
   - memory 追踪工具
   - 工作区索引工具
   - Story 验证工具

#### 风险

1. 系统更大，工具治理成本更高。
2. 不适合作为“先求稳定上线”的最小迁移目标。
3. 如果一开始就重度引入，容易把“工具选择问题”扩大成“平台治理项目”。

#### 结论

**适合第二阶段引入，不适合替代测试的第一枪。**

---

### 4.3 `nurture-tech/unity-mcp-server`

#### 选择理由

这是一个偏现代协议与视觉分析路线的候选。

#### 优点

1. 强调官方 MCP C# SDK 和较新的协议兼容。
2. 有视觉/视图/分析能力，对复杂场景理解有潜力。
3. 如果你后续非常重视“让智能体看懂场景”，它值得继续跟踪。

#### 风险

1. 更像增强候选，不是当前最稳妥的替换选项。
2. 是否适合你当前以日志、测试、场景安全审视为核心的工作流，还需真实验证。

#### 结论

**保留观察，不作为当前第一替换方案。**

---

### 4.4 `notargs/UnityNaturalMCP`

#### 选择理由

这是因为你当前 Unity 已经是 `6000.0.62f1`，所以它突然从“不适配”变成了“版本上可用”。

#### 优点

1. 明确针对 Unity 6000+。
2. 从方向看，和你的版本兼容性心理负担更低。

#### 风险

1. 当前仍处于 preview 阶段。
2. 对你这种项目，不只是要“能跑”，还要“连续稳定地服务工作流”，preview 阶段不够安心。

#### 结论

**可做技术前瞻验证，但不建议立刻替主力。**

---

### 4.5 当前 `mcp-unity`

#### 已知优点

1. 已接入当前环境。
2. 迁移成本最低。
3. 一部分基础工具已经能工作。

#### 已知问题

1. 关键读取与验证链路不够稳定。
2. 当前现场里已经出现：
   - timeout
   - connection closed
3. 这意味着：
   - 它不适合作为唯一主验证通道
   - 也不适合作为“后续所有 skill 都依赖的底层”

#### 结论

**只适合过渡，不适合长期主力。**

---

## 5. 推荐顺序

### 第一名：`CoplayDev/unity-mcp`

理由：

- 最贴合当前痛点：连接稳定性、reload 稳定性、批处理、验证链路。
- 能以最小风险验证“换 MCP 能不能立刻变稳”。

### 第二名：`IvanMurzak/Unity-MCP`

理由：

- 更像长期扩展底座。
- 适合后续做 Sunset 专用 MCP Tool。

### 第三名：`notargs/UnityNaturalMCP`

理由：

- 由于你已经是 Unity 6000，这个候选有现实测试价值。
- 但现在更适合做对照实验，不适合直接顶主力。

---

## 6. 建议迁移策略

### Phase 1：保持当前 MCP 继续干活，但不再把它当长期方案

当前这不影响继续做：

- Skill 建设
- History 索引
- 文档落盘
- 工作区治理

也不影响先完成你现在要求的一条龙内容。

### Phase 2：并行试装 `CoplayDev/unity-mcp`

只验证 5 个 Sunset 核心动作：

1. 读取场景层级
2. 读取 console logs
3. 跑 EditMode tests
4. 脚本 recompile 后是否掉连接
5. 多次连续调用是否稳定

### Phase 3：如果稳定，再决定是否引入第二套扩展型 MCP

如果后续你真要把 Sunset 做成“AI 深度参与开发工作流”的长期工程，再评估 `IvanMurzak/Unity-MCP` 作为扩展底座。

---

## 7. 对当前优先级的影响

**不影响。**

你刚才列的优先级顺序仍然成立：

1. 先稳现有工作流可直接落地的内容
2. 先建 skills
3. 先建索引与路由层
4. MCP 替换是后续基础设施升级，不阻塞当前可立即交付的文档和 skill 产物

换句话说：

- **MCP 很重要，但不是现在这一轮交付的阻塞项**
- **它是下一轮基础设施升级项**

---

## 8. 当前结论

一句话总结：

> 当前 `mcp-unity` 不够格做长期主力；`CoplayDev/unity-mcp` 是最值得优先试装的主替代候选，`IvanMurzak/Unity-MCP` 更适合作为第二阶段的扩展底座，`UnityNaturalMCP` 因为你已是 Unity 6000 也值得关注，但还不该直接上主线。
