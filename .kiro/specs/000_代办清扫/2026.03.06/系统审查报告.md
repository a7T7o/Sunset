# 000_代办清扫 / 系统审查报告

## 审查目标
本报告用于清扫 `.kiro/specs/000_代办/` 的历史代办。目标不是简单复述 TD，而是结合当前代码、资源、场景与工作区文档，判断：

1. 哪些代办现在可以直接执行。
2. 哪些代办被其他迭代覆盖、已经过时或需要重写。
3. 哪些代办因为系统缺口、配置缺失或架构问题暂时不能执行。
4. 哪些问题尚未进入 TD，但已经在当前实现中暴露。

## 当前审查策略
- 先做系统级总览，明确优先级与依赖关系。
- 后续按系统增量补充“文档证据 + 代码证据 + 审查结论 + 建议动作”。
- 每轮优先写入高价值结论，保证中断时已保存核心成果。

## 第一轮总览结论（系统级）

### 1. 当前最值得优先审查的系统

#### 农田系统
- 代办数量最多，且 `TD_000_农田系统.md` 中仍挂有 P0/P1 级遗留项。
- 代码耦合度最高：农田工具预览、农田网格、作物控制器、放置系统彼此交叉。
- 最近仍在活跃开发，说明旧代办很可能已经部分失效，需要重新核对。
- 目前最像“总代办仍在，但细项已经被代码演进改写”的系统。

#### 物品放置系统
- 最近仍在修改，且与农田系统直接耦合。
- 总代办只剩两项，但从近期修改文件看，存在“总代办过于简化，实际风险点更多”的可能。
- 需要核对多格对齐、预览表现与放置验证逻辑是否已经分流到其他系统。

#### 存档系统
- 代办表面不多，但属于全局基础设施。
- 当前总代办里“农作物存档集成验证”大概率仍有效，但必须结合当前 Crop / Placement / Chest 等数据流复核。
- 存档类问题容易出现“文档说待验证，代码实际上已经偏移”的情况。

#### 箱子系统
- 代码看起来比较完整，但总代办区仍保留“终极清算”类历史任务。
- 这类系统常见情况是：主流程能跑，但交互边角、存档、拖拽、排序等问题遗留在文档里未清理。

#### 制作台系统
- 当前像“代码基础已在，但配置与资源层未闭环”。
- 如果 TD 仍把它视为纯开发任务，可能已经过时，应拆成“资源配置缺口”和“功能验证缺口”。

### 2. 初始分类判断

#### A. 看起来仍可执行的代办类型
- 需要用户/场景内验证的集成测试项。
- 明确缺少 Prefab / ScriptableObject / Inspector 配置的补齐项。
- 明确尚未实现的动画接入、数据流接入、UI 预制体创建项。

#### B. 高概率已过时或需要重写的代办类型
- 早期工作区对系统边界的描述，如果当前代码已经经过 9.x/10.x 多轮重构。
- “主系统未完成”的旧表述，但当前其实已经有完整代码目录和资源落地。
- 只描述现象、不包含现状约束的旧 bug 单，可能已被后续修复或转移。

#### C. 可能无法直接执行的代办类型
- 依赖旧架构、旧字段、旧 SO 结构的任务。
- 需要先确认 Inspector / Scene / Prefab 当前配置才能继续的任务。
- 代办本身缺乏验收条件，只保留一句历史结论的任务。

### 3. 第一批系统现状摘录

#### 农田系统
- 现有总代办仍记录：作物存档验证、Collect 动画接入、交互通病修复、视觉/规则写入 steering 等。
- 这说明农田系统已经不是“缺主功能”，而是“进入收尾整合阶段但遗留大量局部问题”。
- 审查重点应转向：哪些项属于真实缺口，哪些项只是旧分析未清理。

#### 物品放置系统
- 总代办只剩“预览透视问题”和“多格子物品对齐规则”。
- 但近期 `PlacementManager.cs` 仍在修改，说明总代办可能没有覆盖最新风险。
- 需要直接核对代码与最近工作区，确认是否还有未入 TD 的回归问题。

#### UI 系统
- 当前总代办集中于制作台 UI 预制体和验证任务。
- 更像资源与验收问题，而不是核心代码从零实现问题。
- 需要警惕：TD 可能过旧，只代表 old 工作区，不代表整个 UI 系统现状。

#### 存档系统
- 主代办聚焦“农作物存档验证、季节渐变问题、异步存档/读档”。
- 其中至少第一项仍值得保留；后两项则需要核对近期实现与当前优先级。

### 4. 当前已暴露的系统性风险
- 代办区与实际代码演化不同步：越活跃的系统，TD 越可能失真。
- “总代办汇总”粒度过粗，容易把资源配置问题、代码缺口、用户验收问题混写在一起。
- 高耦合系统（农田 / 放置 / 存档）如果只看某一侧 TD，容易得出错误结论。

## 第二轮细化：农田系统 + 物品放置系统

### 农田系统

#### 文档证据
- `TD_000_农田系统.md` 当前仍保留 10 项全局待办，其中高优先项集中在：Collect 动画、交互通病、浇水配置确认、Controller 与 SO 数据流完善。
- `TD_10.0.3收获DropTable与动画.md` 已经把该阶段任务压缩到 3 项：
  1. Controller 与 SO 数据流完善
  2. Collect 动画集成
  3. CropController Prefab Inspector 配置
- 这说明 10.0.3 之前的大部分“收获机制改造”已经结束，主问题不再是功能从无到有，而是收尾整合与配置闭环。

#### 代码证据
- `Assets/YYY_Scripts/Farm/CropController.cs` 已经直接持有 `dropItemData` / `witheredDropItemData` / `needsWatering` / `persistentId` 等字段，说明收获掉落、生长规则与持久化入口都已经下沉到当前控制器，而不是停留在设计阶段。
- `CropController.Harvest(...)` 已经实现成熟作物和枯萎成熟作物的掉落处理：成熟分支调用 `WorldSpawnService.SpawnMultiple(dropItemData, ...)`，枯萎分支在 `witheredDropItemData` 非空时也会掉落，表明“DropTable/掉落化”这类旧代办已不是空白状态，而是已落入现行代码流程。
- `CropController.Save()` / `Load()` 已经把 `CropSaveData` 写入 `WorldObjectSaveData.genericData` 并恢复 `layerIndex`、`cellPosition`、`grownDays`、`daysWithoutWater`、`harvestCount` 等关键字段，说明“作物存档接入”从代码层面已经存在，剩下的更像集成验证而非实现缺口。
- `Assets/YYY_Scripts/Farm/FarmToolPreview.cs` 中多次明确写出“锁定机制已移除”“种子走放置系统”“永远更新实时数据”，说明旧代办里关于预览锁定、种子预览的一部分语义已经被后续补丁改写，不能继续按旧问题原样执行。

#### 审查结论

##### 可直接执行
1. **Collect 动画集成**
   - 仍然成立，且是明确剩余核心需求。
   - 原因：当前 `CropController.OnInteract()` 仍直接调用 `Harvest(context)`，没有看到“导航后播放 Collect 动画、关键帧再触发收获”的链路，因此该项不是过时代办。

2. **CropController Prefab Inspector 配置**
   - 仍然成立。
   - 原因：控制器虽然已有 `dropItemData` 与 `witheredDropItemData` 字段，但字段是否在各作物 Prefab 上完整配置，必须靠资源侧补齐，代码本身无法证明配置已完备。

3. **作物存档系统集成验证**
   - 仍然成立，但性质应改写为“验证项”而不是“待实现项”。
   - 原因：`Save/Load` 已在代码存在，当前缺的是游戏内全链路验证。

##### 需要重写/拆分表述
4. **Controller 与 SO 数据流完善**
   - 原代办成立，但描述过宽，需要拆成多个更可执行的小项。
   - 原因：从 `CropController.cs` 看，`needsWatering` 等规则已经在 Controller 上实际生效；真正还未闭环的不是“数据流整体未做”，而是“哪些字段应保留在 Prefab Inspector，哪些应由 SeedData/CropData 驱动”仍未统一。继续保持大而笼统的表述会导致代办无法收敛。

5. **预览锁定视觉同步规则写入 steering**
   - 高概率要先重判再决定是否保留。
   - 原因：`FarmToolPreview.cs` 已明确声明“锁定机制已移除”，说明这个 TD 很可能对应旧机制下的问题。如果现象仍在，就应该以“新机制下的预览跳变/实时同步问题”重写，而不是延续旧锁定语义。

##### 当前无法直接执行 / 依赖外部确认
6. **浇水水渍 Inspector 配置确认**
   - 当前仍依赖场景/Inspector 证据，不能仅凭脚本定案。
   - 需要后续读取相关管理器和场景配置再判断是“真实缺配置”还是“文档滞后”。

##### 已被代码演进部分覆盖
7. **旧种子预览相关代办**
   - 在当前实现下应视为已迁移而非原样继续。
   - 原因：`FarmToolPreview.cs` 已多次标注“种子走放置系统”，说明一部分旧农田预览问题已经被系统边界调整吸收。

#### 新发现的系统风险
- `CropController` 同时承载生长规则、掉落、交互、持久化、成熟特效和旧版兼容字段，职责已经偏重，后续任何“再补一点”都容易继续膨胀这个脚本。
- 农田总代办把“实现项 / 配置项 / 验证项 / steering 规则项”混在一起，执行优先级容易失真，建议后续清扫时拆栏。

### 物品放置系统

#### 文档证据
- `TD_000_物品放置系统.md` 只保留“预览透视问题”“多格子物品对齐规则”两项。
- 但 `TD_8_放置系统多问题修复.md` 实际还写着“代码修复重新执行”，说明总代办摘要已经遗漏了一个更大的事实：该系统并不是纯剩余两条小尾巴，而是存在一次“文档改了，代码修复未完整落地”的历史中断。

#### 代码证据
- `Assets/YYY_Scripts/Service/Placement/PlacementManager.cs` 已经实现完整状态机：`Idle / Preview / Locked / Navigating`，并且在 `EnterPlacementMode(...)`、`ExitPlacementMode()`、`CheckInterruptConditions()`、`HandleInterrupt()` 之间维护放置过程，说明放置主链路早已存在，不属于“系统未完成”。
- 该脚本还直接订阅 `HotbarSelectionService.OnSelectedChanged` 与 `InventoryService.OnSlotChanged`，说明放置模式与快捷栏/背包联动是当前高风险点，而总代办并没有把这类联动中断问题显式记录出来。
- `PlacementManager.Update()` 中对“背包打开时隐藏预览但不退出 Locked/Navigating”的处理，说明当前系统已经有较强状态保持逻辑；这类复杂状态机通常比“预览透视问题”更值得审查。
- `Assets/YYY_Scripts/Service/Inventory/HotbarSelectionService.cs` 中，可放置物品在 `GameInputManager.Instance.IsPlacementMode` 为真时会重新进入 `PlacementManager.EnterPlacementMode(itemData, slot.quality)`；空槽或非放置物品又会调用 `ExitPlacementModeIfActive()`。这说明放置模式切换不只受放置系统自身控制，还受热键槽装备逻辑驱动，存在跨系统状态竞争风险。

#### 审查结论

##### 可直接执行
1. **代码修复重新执行**
   - 这条比总代办里的两条表述更真实，应该保留并前置。
   - 原因：从当前文档看，8_修复工作区自己已经承认“文档已修正但代码待修复”，而近期 `PlacementManager.cs` 仍在修改，说明这不是历史垃圾项，而是当前仍可能未彻底闭环的任务。

2. **多格子物品对齐规则确认**
   - 仍成立，但必须先读 `PlacementPreview` / `PlacementValidator` 后再细化，不适合直接动手。

##### 需要重写/升级表述
3. **预览透视问题**
   - 不能只保持为单一视觉问题描述。
   - 原因：按当前实现，放置预览牵涉状态机、Sorting Layer、背包开关、导航锁定、快捷栏切换。即使最终表现是“透视”，根因也未必只是渲染层级，因此应重写为“预览显示与状态同步问题”，再拆视觉层与状态层。

##### 可能遗漏但应补入 TD 的问题
4. **放置模式与快捷栏/背包联动中断风险**
   - 这是当前代码直接暴露的真实风险点，但我在 TD 摘要里还没看到对应条目。
   - 原因：`PlacementManager` 与 `HotbarSelectionService` 双向影响放置模式，任何一个在错误时机触发 `EnterPlacementMode/ExitPlacementMode`，都可能造成预览错位、状态丢失或中断异常。这个问题比“透视”更核心。

#### 新发现的系统风险
- 放置系统总代办明显“过度乐观”。文档看上去像只剩两个小问题，但代码显示它已经发展成一个高状态复杂度系统。
- `PlacementManager` 同时负责输入状态、导航、预览、背包联动、热键联动、音效与特效，后续代办需要按风险源重新分类，否则会持续把大问题压缩成小标题。

## 第三轮细化：存档系统 + 箱子系统 + 制作台/UI 交叉区

### 存档系统

#### 文档证据
- `TD_000_存档系统.md` 当前只保留 3 项：农作物存档集成验证、季节渐变存档问题修复、异步存档/读档。
- 其中第一项明确写的是“`CropSaveData` 已定义，`CropController Save/Load` 已实现，需要实际游戏内验证完整性”，这和上一轮对农田代码的判断一致，说明该 TD 至少没有严重失真。

#### 代码证据
- `Assets/YYY_Scripts/Data/Core/SaveManager.cs` 的 `SaveGame()` / `LoadGame()` 已经完整跑通“收集全局数据 → 收集 Registry 世界对象 → JSON 写盘 / 读盘 → 恢复世界对象 → 刷新 UI”的主流程。
- `SaveManager.LoadGame()` 在读档前显式调用 `PersistentObjectRegistry.PruneStaleRecords()`，并注明不能 `Clear()`，否则会断开场景实例与存档数据之间的桥梁，说明这里已经经历过一次真实 bug 修复，而不是草稿状态。
- `SaveManager.InitializeDynamicObjectFactory()` 已经支持 `PrefabDatabase` 优先、`PrefabRegistry` 回退的双通路初始化，动态对象重建链路是存在的，不属于“未来规划”。
- `DynamicObjectFactory.TryReconstruct(...)` 已经覆盖 `Drop / Tree / Stone / Chest / Crop` 五类对象，说明当前存档系统已进入“支持范围不断扩展后的集成维护期”。
- `SaveManager` 中没有异步接口，也没有 `Task` / `async` 实现痕迹，说明“异步存档/读档”确实尚未开始，而不是 TD 过时。

#### 审查结论

##### 可直接执行
1. **农作物存档集成验证**
   - 仍成立，而且应该保留在存档系统主视野中。
   - 原因：`CropController.Save/Load` 与 `DynamicObjectFactory.TryReconstructCrop(...)` 已经是现行链路的一部分，现在缺的是实际运行链路验证，不是代码空白。

2. **季节渐变存档问题修复**
   - 当前仍可执行，但前置条件是先读取 3.7.6 对应分析文档并确认用户之前未否决方案。
   - 原因：TD 明确写着“V2 分析文档已完成，等用户确认后实施”，这不是过时项，而是决策被挂起的项。

##### 暂时不能直接执行 / 需要额外上下文
3. **异步存档/读档**
   - 仍是真待办，但不适合直接开做。
   - 原因：它会牵涉保存流程、加载反馈、可能的 UI loading 动画与对象恢复时序，已经超出“补一处逻辑”的范围。当前 TD 只记录需求意图，没有给出验收边界。

##### 需要改写表述
4. **存档系统完成度 80%**
   - 这个判断本身应视为过时表述，而不是稳定事实。
   - 原因：从代码看，存档系统已经不是 MVP 草图，而是包含世界对象注册、动态重建、UI 刷新、旧路径回退的多阶段实现。更准确的说法应改为“主链路已成型，但仍存在若干集成验证项和异步化需求”。

#### 新发现的系统风险
- `SaveManager` 仍是同步读写 JSON 文件，未来一旦对象规模继续增大，异步化需求会从“体验优化”升级为“稳定性/卡顿风险”问题。
- `SaveManager` 通过 `FindFirstObjectByType` 刷新 UI、查找 Player 等，说明当前存档恢复仍较依赖运行时场景结构稳定性，后续如果 UI/Player 结构再改，旧 TD 可能无法覆盖新风险。

### 箱子系统

#### 文档证据
- `TD_000_箱子系统.md` 只保留两类任务：
  1. 终极清算测试检查点
  2. Sort 功能实现验证
- `TD_7_终极清算.md` 则进一步展开为 5 类未执行测试：导航+开箱、排序、拖拽、日志、全面验收。
- 这说明箱子系统当前的主要未闭环点确实集中在“测试与验证”，而不是“缺少主功能实现”。

#### 代码证据
- `Assets/YYY_Scripts/World/Placeable/ChestController.cs` 已实现 `IPersistentObject`，并在 `Save()` 中调用 `SyncInventoryToV2()`，在 `Load()` 后调用 `SyncV2ToInventory()`，说明箱子存档的核心同步坑位已经被正面修过。
- `ChestController` 同时实现 GUID 持久化、prefabId 保存、排序层恢复、Load 先于 Start 的兼容说明，显示该系统已经经过多轮线上式补丁，而不是只做到 demo。

- `BoxPanelUI.Open()` / `Close()`、`SubscribeToChest()` / `UnsubscribeFromChest()`、`ReturnHeldItemsBeforeClose()` 以及拖拽垃圾桶逻辑都已经存在，说明拖拽、关闭归位、事件刷新这些高风险点都已写入代码，只是缺系统验收。

#### 审查结论

##### 可直接执行
1. **终极清算测试检查点**
   - 明确成立，而且优先级应保持较高。
