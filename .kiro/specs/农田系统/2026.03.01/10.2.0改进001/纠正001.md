# 10.2.0改进001 纠正001

## 一、这次纠正的核心结论

本次验牌后确认：此前把 `V` 键过度收缩成“只控制农田工具模式”的理解是错误的。

正确语义应为：

- `V` 是**统一的放置 / 施工模式总开关**。
- 它统一约束两大类交互：
  1. 农田施工类：`Hoe`（锄头）、`WateringCan`（水壶）
  2. 可放置物类：`SeedData`（种子）、`SaplingData`（树苗）、`PlaceableItemData`（箱子/家具等）
- 但是两类交互的**执行器不同**：
  - 农田工具链由 `GameInputManager` 负责分流与执行
  - 可放置物链由 `PlacementManager` 负责预览 / 锁定 / 导航 / 放置状态机
- 因此正确关系不是“V 完全替代 PlacementManager”，也不是“PlacementManager 与 V 完全脱钩”，而是：
  - **V 负责统一授权**
  - **PlacementManager 负责 placeable 的具体状态机执行**

---

## 二、纠正后的正式需求

### 2.1 总需求

新增 `V` 键作为统一放置模式（toggle 开关）。

该模式同时控制：
- 耕地工具：锄头 / 水壶
- 放置物品：种子 / 树苗 / 可放置物品

### 2.2 默认状态（V 关闭）

- `Hoe`：
  - 不能耕地
  - 不显示农田施工预览
  - 但仍保留普通工具身份，可正常挥动，可做非耕地原有交互（如清作物/树苗等）
- `WateringCan`：
  - 不能浇地
  - 不显示浇水预览
- `SeedData / SaplingData / PlaceableItemData`：
  - 不应处于有效放置模式
  - 不应显示 placeable 预览
  - 不应允许继续点击放置

### 2.3 放置模式（V 开启）

- `Hoe`：
  - 可以耕地
  - 显示农田施工预览
  - 允许进入农田 FIFO / 导航 / 动画收尾链
- `WateringCan`：
  - 可以浇地
  - 显示浇水预览
  - 允许进入浇水施工链
- `SeedData / SaplingData / PlaceableItemData`：
  - 允许进入 / 维持 `PlacementManager` 放置状态机
  - 显示 placeable 预览
  - 允许锁定 / 导航 / 放置

---

## 三、关键设计边界

### 3.1 V 统一授权，但不直接重写 PlacementManager

`PlacementManager` 仍然负责：
- `Preview`
- `Locked`
- `Navigating`
- `Idle`
- 预览刷新
- 放置导航
- 放置执行
- 右键 / ESC 退出放置

但 `PlacementManager` 不能在 `V` 关闭时继续维持“有效可放置态”。

所以正确关系是：
- `GameInputManager.IsPlacementMode` = 统一授权标志
- `PlacementManager` = 被授权后执行 placeable 状态机的系统

### 3.2 农田链与 placeable 链必须同时受 V 约束

#### 农田链
由 `GameInputManager` 直接控制：
- `Hoe` / `WateringCan` 是否允许进入施工链
- 农田预览是否显示
- FIFO / 导航 / 动画回调是否继续

#### Placeable 链
由 `PlacementManager` 执行，但是否允许活跃，必须取决于 `V`：
- `V` 开：允许 `PlacementManager` 进入 / 维持放置模式
- `V` 关：`PlacementManager` 应退出或拒绝进入有效放置模式

## 四、行为矩阵（验收标准）

### 4.1 手持 Hoe

#### V 关
- 不显示农田施工预览
- 左键不耕地
- 仍可挥动
- 仍可执行非耕地原有交互（如清作物/清树苗等）

#### V 开
- 显示农田预览
- 左键允许进入 Till / RemoveCrop / Harvest 边界链
- 可走队列 / 导航 / 动画收尾

### 4.2 手持 WateringCan

#### V 关
- 不显示浇水预览
- 左键不能浇地

#### V 开
- 显示浇水预览
- 左键允许进入浇水施工链

### 4.3 手持 Seed / Sapling / PlaceableItem

#### V 关
- 不应处于有效放置模式
- 不应显示 placeable 预览
- 不应允许继续放置

#### V 开
- 应允许进入 / 维持 placeable 预览
- 应允许锁定 / 导航 / 放置

### 4.4 切换 Hotbar

- 已开 `V` 时切到可放置物：应立即进入 placeable 预览
- 已关 `V` 时切到可放置物：不应进入有效预览
- 切换工具不应偷偷改写 `V` 状态

### 4.5 按 V 时的即时反馈

#### V 从关 → 开
- 手持 `Hoe / WateringCan`：农田预览立即出现
- 手持 `Seed / Sapling / PlaceableItem`：placeable 预览立即出现

#### V 从开 → 关
- 手持 `Hoe / WateringCan`：农田预览立即消失
- 手持 `Seed / Sapling / PlaceableItem`：placeable 状态应立即退出 / 失效，不允许残留继续放置

## 五、其他按键语义

### 5.1 WASD

`WASD` 的职责是：
- 中断当前锁定 / 导航 / 队列
- 但**不直接改写 V 状态**

也就是：
- `V` = 当前是否处于放置 / 施工模式
- `WASD` = 当前这一次动作是否被打断

二者语义必须分离。

### 5.2 ESC / 右键

`ESC / 右键` 应优先处理“当前链路退出”，而不是偷改 `V`：

- 农田链：取消当前队列 / 导航 / 锁定
- placeable 链：退出或回退 `PlacementManager` 当前状态
- 是否关闭整个 `V` 模式，不应再隐式绑定到这些中断键

当前纠正口径下，推荐维持：
- `V` 只由 `V` 自己切换
- `ESC / 右键 / WASD` 只处理本次交互链的中断与退出

## 六、本次代码修复方向（纠正方案）

### 6.1 必须纠正的错误方向

此前错误方向是：
- 把 placeable 从 `V` 语义里剥离
- 让 `PlacementManager` 与 `GameInputManager.IsPlacementMode` 基本脱钩

这会导致：
- `V` 关闭时，placeable 仍可能继续显示预览或继续放置
- 与“V 键统一放置模式”的总需求冲突

### 6.2 正确修复方向

#### 方向 A：恢复 V 对 placeable 的授权控制
- `V` 关闭时：
  - 当前手持 placeable 时，`PlacementManager` 应退出或失效
- `V` 开启时：
  - 当前手持 placeable 时，`PlacementManager` 应进入或恢复有效预览

#### 方向 B：保留 PlacementManager 原有状态机职责
- 不重写 `Preview / Locked / Navigating / Idle`
- 不新增管理器
- 不重构放置架构
- 只在进入条件、维持条件、V 切换同步点上补约束

#### 方向 C：保留已修复正确的 Hoe 双态逻辑
- 非放置模式下 Hoe 仍是普通工具
- 不再让 Hoe 因为 V 关闭而整体失效

## 七、最小兼容实施方案

### 7.1 `GameInputManager.cs`

需要调整：
- `V` 切换时，除了控制农田预览，还要同步处理当前手持 placeable 的授权状态
- 若当前手持的是 placeable 且 `V` 关闭：应通知 `PlacementManager` 退出有效放置模式
- 若当前手持的是 placeable 且 `V` 开启：应通知 `PlacementManager` 进入或刷新当前预览

### 7.2 `HotbarSelectionService.cs`

需要核对：
- 当前切到 placeable 时，是否无条件 `EnterPlacementMode()`
- 若是，则应在入口增加 `V` 授权判断：
  - `V` 开启才允许自动进入放置模式
  - `V` 关闭时不自动进入

### 7.3 `PlacementManager.cs`

需要保留：
- 状态机本体
- 预览 / 锁定 / 导航 / 放置执行逻辑
- `RefreshCurrentPreview()` 可继续保留，作为 V 切换后的即时刷新入口

但需要避免：
- 与 `V` 完全脱钩
- 在 `V` 关闭时仍能独立活跃

## 九、本次纠正结论（对话口径）

最终统一口径：

> `V` 控制“是否允许进行放置 / 施工类交互”；
> `Hoe / WateringCan` 走 `GameInputManager` 农田链；
> `Seed / Sapling / PlaceableItem` 走 `PlacementManager` 放置链；
> 但这两条链都必须受 `V` 统一约束。

这就是本轮纠正后的正式需求与实现基线。
