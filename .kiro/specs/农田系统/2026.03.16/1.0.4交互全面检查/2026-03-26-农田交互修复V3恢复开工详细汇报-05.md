  - 现行逻辑是先 `CreateTile(...)`，成功后才 `CommitCurrentToolUse(...)`
  - 这说明“锄头 / 普通工具运行时提交”并没有真正被前置成事务入口，而仍然是地块成功后的后置提交

### 用户可验收点

- 如果用户只想看当前主分支已经具备了哪些表层表现，可以做现状观察；但这不是本轮可 claim 的最终闭环。

### 测试前提

- 打开 `Sunset`
- 进入可正常锄地的农田场景
- 手持一把有耐久的锄头或其他耐久工具
- 准备一把接近耐久归零的工具，便于观察损坏反馈

### 入口路径

- 运行游戏后，切到工具栏里的锄头 / 普通工具
- 面向可交互农田或普通工具的成功作用目标

### 操作步骤

1. 连续用工具对合法目标执行成功动作，直到耐久接近归零。
2. 观察每次成功动作后：
   - 是否扣精力
   - 是否扣耐久
   - UI 槽位耐久条是否同步变化
3. 在最后一次成功动作后，观察工具归零时是否：
   - 从槽位移除
   - 触发 shake
   - 播放损坏音效
   - 出现 burst 特效
   - 头顶出现 3 秒损坏气泡

### 预期结果

- 现状层面，成功动作后应能看到：
  - 精力减少
  - 耐久减少
  - 耐久归零后工具移除
  - 损坏反馈链触发
- 但从实现正确性口径看，这一项本轮仍不能 claim “完全闭环”，因为事务起点仍在热区文件里没有前置

### 失败判读

- 如果出现“成功作用了，但耐久不减 / 工具不坏 / 没有气泡特效”，更像是工具反馈链本身未接通
- 如果出现“世界结果已经成立，但工具提交失败仍然没有阻止世界变化”，更像是 `GameInputManager.cs` 的执行顺序问题

### 本轮卡点

- 卡点：`GameInputManager.cs`
- 是否因为 scope spill 到 hot 目标：是
- 最小 blocker：
  - 把 `ExecuteTillSoil(...)` 从“先 `CreateTile` 再提交工具消耗”改成“先提交成功工具使用，再落地世界结果，必要时提供失败回滚”

  - 这一点更像输入层闸门，不是 `TreeController` 单点就能完全兜住

### 用户可验收点

- 这一项当前属于“部分可验、但不能 claim 全闭环”。

### 测试前提

- 打开 `Sunset`
- 准备至少两把不同等级的斧头：
  - 一把低于目标树要求等级
  - 一把满足目标树要求等级
- 场景里存在一个高等级树目标

### 入口路径

- 运行游戏后，切到低等级斧头
- 面向高等级树执行挥砍

### 操作步骤

1. 用低等级斧头对高等级树挥砍一次。
2. 观察：
   - 树木是否掉血
   - 精力是否被扣
   - 是否出现不足等级气泡
   - 是否播放不足等级反馈音效
3. 保持 30 秒内，继续用同类低等级斧头或另一把仍不足等级的斧头对同类高树尝试挥砍。
4. 在冷却仍存在时，切换到足够等级的斧头，再次砍同类高树。
5. 观察气泡是否切换为“还是这把斧头锋利！”

### 预期结果

- 当前代码口径下应至少成立：
  - 不足等级时不扣精力、不掉血
  - 第一次触发冷却和气泡
  - 冷却窗口内再次命中同类失败提示时，不应不断刷新同类气泡时长
  - 切换到足够等级斧头成功砍树时，可切换为正向气泡
- 当前仍不能宣称必然成立的点：
  - 是否已经在输入层彻底挡掉 30 秒内的再次挥砍动作

### 失败判读

- 如果不足等级时仍掉精力或树木掉血，更像是 `TreeController` 等级判定或工具提交顺序缺口
- 如果正向切换气泡不出现，更像是 `HandleAxeTierRecovered()` 触发链未被正确命中
- 如果仍能频繁出完整挥砍动作，更像是缺少输入层冷却闸门，而不是 `TreeController` 文案链没写

### 本轮卡点

- 卡点：高树失败冷却目前只在命中侧有计时与提示，尚未看到输入层前置拦截证据
- 是否因为 scope spill 到 hot 目标：大概率是
- 最小 blocker：
  - 需要输入层在“低级斧头 -> 高级树 -> 冷却窗口”上前置拒绝再次挥砍动作；当前这类入口不在本轮允许的 non-hot 文件里

## 功能点 4：hover 遮挡范围收紧

### 做了什么

- 已核对 `FarmToolPreview.cs` 当前给 `OcclusionManager` 上报 hover 预览范围的代码。

### 改了什么

- 本轮未改代码。

### 自验证据

- 当前 blocker 证据：
  - `FarmToolPreview.cs:1125-1167`
    - `NotifyOcclusionSystem()` 通过 `TryGetHoverPreviewBounds(...)` 向 `OcclusionManager` 上报 bounds
  - `FarmToolPreview.cs:1177-1202`

    - 这意味着耕地 hover 仍按整组 preview tile 的联合 bounds 上报，而不是只按中心格或主格上报

- 这和用户要求“中心块没被挡住就不应该触发隐藏”仍不一致。

### 用户可验收点

- 当前不建议把这一项写成“已收紧完成”。

### 测试前提

- 打开 `Sunset`
- 进入存在前景遮挡物的农田区域
- 手持锄头或能触发农田 hover 预览的工具

### 入口路径

- 运行游戏后，切到锄头
- 把鼠标悬停到靠近遮挡物边缘的农田格上

### 操作步骤

1. 把鼠标移动到一个中心格周围存在 1+8 预览范围、但中心格本身并未被遮挡的耕地位置。

3. 再把鼠标移动到中心格本身确实被前景遮挡的位置。
4. 对比两种情况下透明触发范围。

### 预期结果

- 理想完成定义应是：
  - 只有中心格 / 主格真的被遮挡时才触发透明
  - 不再因为外围 1+8 的任意边界 tile 命中就整组触发
- 当前主分支仍不能把这个预期写成“已成立”

### 失败判读

- 如果中心格没被挡住，外围格一贴边就开始透明，更像是 `FarmToolPreview` 上报给 `OcclusionManager` 的 bounds 仍然过大
- 如果中心格确实被挡住却完全不透明，则更像是 `OcclusionManager` 或 preview 上报失联

### 本轮卡点

- 卡点：`FarmToolPreview.cs` 仍按整组 `currentPreviewPositions` 联合计算 hover bounds
- 是否因为 scope spill 到 hot 目标：否，这一项本身仍属于 non-hot
