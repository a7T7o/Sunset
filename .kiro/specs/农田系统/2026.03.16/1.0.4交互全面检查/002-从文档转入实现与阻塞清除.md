- `Assets/YYY_Scripts/Service/Inventory/ChestInventory.cs`

## 五、第二刀：放置系统必须真正统一“到达判定 / 可放置距离 / 预览边界”

箱子递归清掉之后，第二刀直接进入放置系统。

你要同时处理两类症状：

1. 树苗连续快速放置时的“幽灵占位”
2. 箱子放置时玩家导航 retry 3 次后原地停下

这两条不能再当成两个互不相干的小问题。  
它们背后高度可能共享同一类边界失配：

- 导航到达判定
- 放置成功判定
- 预览碰撞体边界
- 玩家最终停下的位置
- 失败回滚或表现刷新时机

你必须把这几件事统一到一条清晰口径上：

- 玩家走到哪里算“已经到达可放置距离”
- 这个距离到底由什么定义
- 预览碰撞体边界和最终放置成功边界是否完全一致
- 快速点击 + 玩家移动时，是否存在“逻辑已占位但表现未落盘”的中间态

如果存在这种中间态，就要消灭，不准继续留“幽灵”。

## 六、第三刀：Toolbar 与抖动锁定必须按真正边界收口

这一刀不要再泛谈“背包联动复杂”。  
你要正面把边界写死并落到代码里：

1. 背包关闭后不允许出现 Toolbar 双选中
2. 只有真正正在交互中的工具和可放置 / 可使用物品，才允许进入抖动锁定
3. 单纯“手上拿着但没有实际进入导航执行和交互执行”的树苗、种子等内容，不允许被错误锁死

换句话说：

- “被选中”
- “被持有”
- “正在执行交互”
- “正在导航去执行交互”

这四种状态你必须分开，不准再糊成一团。

## 七、第四刀：耐久 / 精力 / Tooltip 必须从半残状态进入统一参数链

这一刀不允许再停在“测试按钮才能触发 UI”的病态状态。

你最终至少要做到：

1. 所有工具自动具备耐久测试入口，不依赖 `DurabilityTestCanvas` 的按钮作为唯一触发源
2. 工具使用时，精力与耐久都真正消耗
3. 精力和耐久消耗都走 SO 参数
4. Console 必须输出：
   - 当前工具名称
   - 当前耐久
   - 本次消耗耐久
   - 本次消耗精力
5. 鼠标悬停工具超过 `0.6s` 时，显示：
   - 工具名称
   - 工具简介
   - `剩余耐久/总耐久`
6. 鼠标悬停精力条超过 `0.6s` 时，显示：
   - `当前精力/总精力`

这里的重点不是“补一个小 UI”，而是要把：

- 数据来源
- 消耗触发
- UI 表现
- Tooltip 时序

整条链收成一致。
