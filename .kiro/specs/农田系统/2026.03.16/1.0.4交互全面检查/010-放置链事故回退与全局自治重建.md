# 010-放置链事故回退与全局自治重建

这轮我先把前一轮口径彻底纠正到位：

1. 我不接受把 `009-hover遮挡链闭环` 外推成“农田这条线整体已经绿了”。
2. 用户最新现场已经明确否定当前整体放置体验，而且否定得非常重：
   - 现在的放置根本没办法执行；
   - 玩家距离很远就停；
   - 也没办法放置成功；
   - 现在全是幽灵；
   - 整体比之前更粗糙、bug 更多。
3. 所以这轮不是继续 `hover-only`，也不是继续拿单条 live pass 充总完成。

这轮的真实定性已经变成：

> 当前 placeable / 放置交互链发生了整链回归事故。  
> 先恢复到最后一个用户至少能接受、至少能工作的基线，  
> 再在这个基线上自治重建。

---

## 一、当前已接受的基线

当前农田线我只接受这些事实：

1. `1.0.4` 的原始要求不是 hover 单点，而是整套全局交互处理：
   - 放置成功判定
   - 放置导航与到达边界
   - 幽灵占位
   - 背包 / Toolbar / 锁定边界
   - 工具 runtime 参数链
   - 箱子链
2. 用户对“之前至少还能用”的下限描述已经很清楚：
   - 之前的放置系统虽然有特殊边界幽灵
   - 也有放置物品导航目的地位置不对、判定不一致
   - 但还没有坏到“距离很远就停 + 根本放不下去 + 到处都是幽灵”
3. 当前 `009` 只证明了：
   - hover-only runner 在特定取样窗口下能跑通
   - 不代表 placeable 链整体可用
4. 用户现在要的不是再一刀一刀拖，而是：
   - 线程自己顾全全局
   - 自己理解代码
   - 自己决定回退 / selective restore / 重建路径
   - 最终把整条放置链重新拉回可用

---

## 二、这轮唯一主刀

### 这轮唯一主刀固定为：

> 先把当前放置链从“比旧基线更差的事故态”拉回到可工作基线，  
> 然后在同一轮里按原始需求重新建立正确推进面。

更直白一点：

- 这轮不是 hover-only；
- 这轮不是 docs-only；
- 这轮也不是“只再补一个小 live runner”。

这轮你被明确授权在线内自治：

- 先找最后一个用户至少能接受 / 至少能工作的放置基线；
- 再决定到底是 selective rollback、selective restore，还是局部保留当前好改动后重建；
- 但无论你选哪条路，目标都只有一个：
  - 先把 placeable / 放置交互从事故态救回来。

---

## 三、这轮必须先回答的核心问题

你这轮开工后，第一优先不是再写 hover 诊断，而是先查清这 4 件事：

1. 当前“距离很远就停、无法放置成功、到处都是幽灵”到底是从哪一批改动开始明显恶化的。
2. 最后一个用户至少能接受 / 至少能工作的放置基线是谁：
   - 具体 commit
   - 或具体文件集合状态
3. 当前已进入 `main` 的哪些改动是真正有价值、应该保留的。
4. 当前 dirty / 最近提交里，哪些改动就是导致整链回归的第一责任点。

这轮不要求你先来问我怎么回退。

我明确授权你在线内自己判断：

- 该回退就回退；
- 该 selective restore 就 restore；
- 该保留好改动、只撤坏改动，就按这个做。

但你必须把判断依据讲清楚。

## 五、这轮明确禁止

### 不允许：

1. 不准再把这轮缩成 `hover-only`。
2. 不准再拿某一条单独 live pass 充当整条放置链恢复完成。
3. 不准停在“我已经分析出问题了”，却不真正恢复基线。
4. 不准把系统继续留在“比旧基线更差”的状态。
5. 不准做没有依据的全量大回退，把已经明确有效的改动也一起抹掉。
6. 不准把这轮再次漂去无关系统，只允许围绕当前 placeable / 放置交互事故处理。

## 六、这轮完成定义

只有满足下面任一结局，这轮才算完成。

### 结局 A：放置链已恢复到可工作基线

你要明确给出：

1. 你认定的“最后一个至少可工作基线”是谁；
2. 你最后采用的恢复策略是什么；
3. 你实际恢复了哪些行为；
4. 当前至少这几类现场已经回正：
   - 玩家不会在很远处就停
   - 代表性 placeable 可以真正放置成功
   - 不再全场到处都是幽灵预览
   - placeable 成功 / 失败边界至少回到用户可理解的状态
5. 哪些之前确实好的链被你保住了，没有因为回退一起做坏。

### 结局 B：整链已先恢复到不比旧基线更差，并继续收敛出单一剩余点

如果你判断这一轮来不及把全部 placeable 体验完全做漂亮，也可以接受。

但前提是你必须先做到：

1. 当前状态已经不再比旧基线更差；
2. 用户最痛的事故态已经压掉：
   - 远停
   - 根本放不下
   - 全场幽灵
3. 剩余问题已经被你压成单一、可继续追的剩余点；
4. 你已经说明：
   - 保留了什么
   - 回退了什么
   - 接下来唯一剩余点是什么

如果当前仍然是“整体比旧基线更差”，这轮不算完成。

## 七、这轮 live 验证至少覆盖这些代表场景

1. 一个代表性 placeable：
   - 走到边上即可放下
   - 不会远停
2. 一个箱子类 placeable：
   - 不会 retry 三次后原地停
   - 不会明明到边上却失败
3. 一个树苗 / 树相关代表场景：
   - 连点 / 移动中点击不再轻易出幽灵
4. hover 遮挡 sanity：
   - 不能再出现“为了修 hover，整条 placeable 链更糟”的反向事故

你不必拘泥于旧 runner 名字，但必须覆盖这些真实行为。
