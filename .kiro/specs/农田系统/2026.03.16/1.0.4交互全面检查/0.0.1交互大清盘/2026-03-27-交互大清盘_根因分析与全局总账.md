## 1. 当前 3 个必须重开的硬问题

### 1.1 树木 1/2 阶段被砍倒后又“复活立起来”

#### 现象

- 不是树苗阶段。
- 是树木还没进入树桩阶段的那两个中间阶段。
- 树已经被砍倒了，倒下动画快播完时如果玩家继续挥砍，会看到树的 sprite 又重新站起来。
- 但这个“站起来”的东西又碰不到，像个幽灵壳子。

#### 根因，用大白话说

这棵树在“开始倒下”的那一刻，并没有真正被系统判成“已经死了”。

现在的实现更像是：

1. 先把掉落、经验、音效这些东西处理掉。
2. 然后临时造一个“倒下用的 sprite”去播表现。
3. 但原树对象自己并没有立刻切到“不可再命中”的单向状态。

结果就是，玩家在倒下动画那几百毫秒里继续挥砍时，系统还会把它当成一棵正常活树再吃一次命中。

用户看到的“死而复生”，本质上大概率不是真树复活了，而是第二个临时倒下表现又从直立姿态重新起播了一遍。

#### 真正卡点在哪

- 当前 `TreeState` 只有 `Normal / Withered / Frozen / Melted / Stump`，没有“正在倒下”这种中间锁死状态。
- 无树桩阶段走的是 `FallAndDestroyCoroutine(...)`，这条路只是播表现，不是事务封口。
- `ChopDown()` 开始后，原树对象没有立刻失去“可被再次命中”的资格。
- 碰撞体也不是在开始倒下时就先撤掉，而是一直拖到最后销毁。

#### 如果不修，会带来什么额外风险

这不只是视觉 bug。

它还有机会带出这些问题：

- 重复掉落
- 重复经验
- 重复砍倒音效
- 树对象生命周期和场景表现不同步

#### 我准备怎么修

这条不能再当动画问题修，必须当“死亡事务”修。

修法应该是：

1. 树一旦进入 `ChopDown()`，立刻切到一个单向的“已死亡 / 正在倒下”状态。
2. 从这一刻起，原树对象不再接受新的 `OnHit(...)`。
3. 同时撤销原树的命中资格和碰撞资格，不要等动画播完才算死。
4. 临时倒下 sprite 只负责表现，不再代表业务对象本体。
5. 掉落、经验、音效都只允许提交一次。

### 1.2 连点放置时，鼠标不动就不重新判断当前格子

#### 现象

用户把鼠标停在离玩家很近的位置不动，然后连续点放置。

按正常体验，玩家往前走一点，鼠标实际指向的世界格子就已经变了，预览应该自己刷新到新的候选格。

但现在不会。

必须要玩家故意晃一下鼠标，预览才重新判断。

#### 根因，用大白话说

这条问题不是“判格慢了一点”，而是“刷新条件从一开始就写错了”。

现在放置成功后，系统为了防抖，故意把 preview 暂时钉在刚放下的位置。

这个思路本身不是错。

真正错的是它后面用来恢复刷新的条件。

它现在看的不是：

- 鼠标现在对应的世界格子有没有变

而是：

- 鼠标在屏幕上的像素坐标有没有动

这两个不是一回事。

玩家自己走动、相机跟随、世界相对关系变化时，哪怕鼠标屏幕坐标没动，鼠标实际指向的世界格子也早就变了。

但代码还在认定“鼠标没动”，于是 preview 就继续死钉在上一个格子。

#### 真正卡点在哪

- 放置成功后，`ResumePreviewAfterSuccessfulPlacement()` 会开启一个“先停在原格”的 hold。
- `ShouldHoldPreviewAtLastPlacement()` 解除 hold 的条件，是 `Input.mousePosition` 有没有超过阈值。
- 也就是说，它把“屏幕像素位移”误当成了“世界候选格变化”。

#### 为什么这条很严重

这条不是纯视觉小瑕疵。

它会直接放大两类问题：

- 用户以为下一格应该已经能种了，但系统还在拿上一格做判断
- sapling ghost / occupied cell 的体验会被这条错误冻结继续放大

也就是说，这条问题本身就属于树苗连续放置主链，不是附带小毛病。

#### 我准备怎么修

这条要保留“放置后短暂稳一下”的目标，但不能继续拿屏幕像素位移做唯一条件。

应该改成：

1. 放置成功后，允许 preview 只做极短暂稳态。
2. 但重新刷新时，要看“当前鼠标对应的世界候选格 / 格心”有没有变化。
3. 玩家走动导致候选格变化时，即使鼠标屏幕坐标没动，也必须重新判格。
4. 放置成功后还要主动按“当前世界候选格”立刻重刷一次，而不是傻等用户晃鼠标。

### 1.3 工具坏掉了，但动作动画还在继续

#### 现象

现在会出现这种违和状态：

- 工具耐久归零了
- 损坏特效出了
- 气泡也出了
- 物品槽也清掉了
- 控制台日志也对

但玩家动作还在继续播，好像手里还拿着那把工具。

#### 根因，用大白话说

这里不是单点逻辑错，而是两条状态链没有在关键节点重新汇合。

当前至少有两条链在各跑各的：

1. 工具运行时链
   - 负责扣耐久
   - 负责清槽
   - 负责气泡和特效
2. 动作动画链
   - 负责计时
   - 负责动作是否结束
   - 负责工具 sprite 什么时候隐藏

现在工具坏掉时，第一条链已经知道“这把工具没了”。

但第二条链还不知道。

于是就出现：

- 物品世界观已经没了
- 动作状态机却还按固定时长继续播放

#### 真正卡点在哪

- `ToolRuntimeUtility` 在耐久归零时会清槽并发反馈。
- `PlayerInteraction` 仍然让当前动作按固定动画时长自然跑完。
- `LayerAnimSync.ForceHideTool()` 这类收尾逻辑当前主要还是挂在正常的 `OnActionComplete()` 分支上。
- 系统里缺少一个“当前动作依赖的手持物已经被移除，因此允许立刻强制收尾”的合法打断点。

#### 这条最关键的统一口径

这里真正该作为“正当打断理由”的，不是抽象的“耐久 == 0”。

而是：

**这次动作依赖的那把手持物，在动作提交后已经不存在了。**

这个判断才足够干净。

因为：

- 斧头坏掉是“物品没了”
- 水壶没水不是“物品没了”
- 低级斧头砍高树失败也不是“物品没了”

所以合法打断点不该写成一个含糊的“资源变 0”，而该写成“当前动作依赖的手持物已被移除 / 已失效”。

#### 我准备怎么修

这条应该这样收：

1. 保留“普通动作不被中途打断”的原则。
2. 但补一个明确例外：当前动作依赖的手持物已在提交后被移除时，允许立即强制收尾。
3. 这个强制收尾不能只藏工具图。
4. 必须一起处理：
   - 动画计时
   - tool sprite 隐藏
   - action lock
   - 长按续挥砍
   - 队列与后续动作衔接

不然只是把图藏掉，状态机还在跑，问题只会换一种形态继续出现。

## 2. 这次大清盘拉出来的全局总账

## 2.1 已被用户点名、现在仍不能算通过的项

### A. 树苗连续放置主链

这条现在不能再被简化成“只剩 runner 稳定性”。

它至少还包含两层真实问题：

- 幽灵占位本体，用户已经亲口否过
- 放置成功后 preview 只认屏幕鼠标位移、不认世界候选格变化

换句话说，这条现在仍是严重未收口项。

### B. 树木倒下事务

这条就是本轮新增的“1/2 阶段被砍倒后又立起来”。

它说明树木生命周期事务现在依然没有真正封死。

### C. 工具坏掉后的动作尾巴

当前反馈链和动画链还没统一到一个收尾口径。

这条也不能算通过。

### D. 箱子开启距离 / 到位手感

这条仍应列为未闭环项。

原因很简单：

- 用户明确提过“右键箱子根本没走到能开的距离”
- 当前代码里箱子还是固定 `InteractionDistance = 1.5f`
- `GameInputManager` / `PlayerAutoNavigator` 仍在用一套固定比例去推 stop radius

这说明它至少还没有拿到“手感真的对了”的最新现场证据。

### E. 打开箱子不应退出放置模式

这条也不能判完成。

从当前代码搜索结果看，`GameInputManager` 和 `HotbarSelectionService` 里仍然存在多处自动 `ExitPlacementMode()` 路径。

也就是说，这条需求虽然在池子里，但还没有可靠证据证明它已经彻底被封住。

### F. Toolbar 双选中 / 错误锁槽

这条历史上反复复发过。

虽然结构上做过整理，但当前没有新的用户“通过”回执。

所以现在不能按“应该没事了”处理。

### G. hover 遮挡

这条必须单独写实：

- 结构上后来确实朝“中心格 / 主格优先”改过
- 但用户最后一次明确反馈仍然是“不对”
- 之后没有新的用户终验回执把它转正

所以它现在的真实状态只能写成：

**做过改动，但还没有被用户重新确认通过。**

### H. 玩家气泡样式

功能链不再是空白，但表现层还没过用户那关。

用户已经明确说过：

- 不是不能弹
- 是“不好看”
- 必须去学 NPC 气泡

所以这条不能按“功能有了就过”处理。

### I. 低级斧头砍高树的 30 秒冷却

当前更像是“结构做了一半”。

`TreeController` 已经有不足等级冷却计时和气泡切换入口。

但“输入层是否真的在 30 秒内挡住再次挥砍动作”这件事，当前还缺可靠现场证据。

### J. 无碰撞体可脚下放置

这条是刚刚补进去的新规则。

需求入口和验证代码都已经写了。

但用户还没有给现场回执。

所以现在只能叫：

**规则已固化，终验未完成。**

## 2.3 这次彻查后，我认为整条交互线真正反复坏掉的共性

现在问题看起来很多，但共性其实很集中。

### A. 放置链缺的不是一个 if，而是“成功事务”

现在放置相关问题反复出现，本质上是：

- 什么时候算真的放下了
- 什么时候该把占位当真
- 什么时候该刷新下一格

这些事情还不是同一个事务。

### B. 树生命周期缺的不是动画，而是“单向死亡状态”

树一旦开始倒下，就不该再有第二种身份。

现在它还能同时处在“开始死了”和“还能再吃一次命中”的叠态里，这就是问题根。

### C. 工具链缺的不是反馈资源，而是“动作前检 -> 动作提交 -> 物品失效 -> 强制收尾”的完整闭环

只要这条闭环没收齐，就一定会继续出现：

- 能不能挥砍和什么时候扣账对不上
- 物品没了但动作还在
- 水壶 / 斧头 / 锄头每条线各说各话

### D. UI 和交互链缺的不是局部刷新，而是“谁是事实源”

包括：

- Toolbar 选中态
- 背包第一行
- 箱子 UI
- 放置模式是否保持

这些问题反复冒出来，本质都是“显示层”和“事实层”还没彻底分开。

## 3. 我建议的后续施工顺序

### 第一刀：先把树苗放置主链真正收成一刀

不要再把它拆成：

- 一部分叫 sapling ghost
- 一部分叫 preview refresh
- 一部分叫 occupied cell

这三件事其实是同一条链。

应该一次收成：

- 放置成功后立刻按当前世界候选格重刷
- 占位判断按真实单格 / 真实落地状态走
- 不再出现“必须晃鼠标才刷新”的冻结

### 第二刀：收树木 1/2 阶段的倒下事务

这条不要混到气泡或耐久里。

它就是单独的树生命周期问题。

目标只有一个：

树一旦开始倒，就再也不是活树。

### 第三刀：收工具损坏后的强制收尾

这条要围绕一个唯一断点来做：

**当前动作依赖的手持物已被移除。**

不要继续拿抽象的“耐久为 0”做所有情形的混合判断。

### 第四刀：回头清历史尾账

这一批更像第二层问题：

- 箱子开启距离 / 手感
- 开箱不退出放置模式
- Toolbar 双选中 / 错误锁槽
- hover 最终终验
- 玩家气泡样式终修
- 低级斧头高树 30 秒冷却输入层前置拦截

### 第五刀：最后再做一轮用户向整体验收

这一轮不要再靠“结构 pass”替代“用户 pass”。

要直接按用户真实体验看：

- 视觉和逻辑是不是终于统一
- 动画和物品状态是不是终于统一
- 放置和交互手感是不是终于统一

## 4. 这次大清盘后的总判断

### 4.1 当下判断

当前这条线不是“很多独立小 bug 同时冒出来”。

而是几个关键事务边界一直没真正关门。

### 4.2 历史判断

之前这条线最大的问题，不是没有做事，而是太多时候把：

- 结构 pass
- runner pass
- 局部 log pass

过早当成了用户 pass。

所以后面才会不断被现场重新打回来。

### 4.3 全局判断

后面再继续做交互，不能再只用“代码看起来合理”来判断完成。

至少要同时满足这三层：

1. 事务口径统一
2. 表现口径统一
3. 用户现场体验统一

只要这三层里少一层，后面还会继续返工。
