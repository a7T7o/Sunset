## 2. 全局执行纪律

后续整条线必须统一遵守下面 8 条，不允许再回到“每轮局部看起来像过了，但用户体感一直没过”的旧循环。

### 2.1 先关事务门，再做表现层抛光

优先级必须固定为：

1. 放置事务
2. 树生命周期事务
3. 工具失效事务
4. 其余交互一致性与表现层

### 2.2 不允许只拿 runner pass 当完工

以下任一情况都不能单独作为“完成”：

- 自动化 runner pass
- 单次 `Editor.log` 漂亮
- 代码结构看起来合理
- `git diff --check` 通过
- 程序集级 preflight 通过

这些只能算“可以继续往下走”，不能算“用户这条已经过了”。

### 2.3 完成定义必须同时覆盖三层

每个任务都必须同时满足：

- 代码层：逻辑真正闭环，没有留下半事务状态
- live 层：至少有代表性 Unity 运行时证据
- 体验层：用户在屏幕上看到和手上按出来的结果与口径一致

### 2.4 不允许偷换事实源

后续修复必须尽量把事实源统一到真正的业务真源上，禁止继续出现：

- 占位看一套位置，预览看另一套位置
- 动作链认一套状态，工具链认另一套状态
- UI 表现和业务状态不同步

### 2.5 Hot / Mixed 文件可以碰，但必须有刀口

后续如果必须重开 `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`，可以重开。

但要求是：

- 只为当前主刀服务
- 必须在回执里明确写出“为什么必须碰它”
- 不能借机把 unrelated WIP 一起吞进去

### 2.6 不允许把表现问题降级成“只是视觉 bug”

这条线里很多问题表面看是视觉，实质上是事务没封口。

尤其以下几类：

- 幽灵占位
- 死而复生 sprite
- 工具坏了但动作还在播
- hover 透明范围过大

后续一律按“视觉和逻辑统一”处理，不再当成小瑕疵。

### 2.7 每一阶段都要有明确出口

后续施工必须按阶段过闸，不能无边界并行扩写。

只有前一阶段达到“完成定义”，下一阶段才允许 claim 开始。

### 2.8 最终交付必须带专业验收包

整条线全部做完后，不能只说“你再测一下”。

必须补一份给用户直接看的详细验收文件，按功能点写清：

- 测试前提
- 入口路径
- 操作步骤
- 预期结果
- 失败判读

## A1. 树苗连续放置事务

### 4.1 目标

彻底修掉树苗连续放置主链的“逻辑占位、视觉占位、预览刷新”三件事不一致的问题。

### 4.2 当前问题范围

这条不能再被理解成单点 bug。

它至少包含 4 层问题：

1. 连续放置后仍会出现幽灵占位
2. 鼠标不动时，玩家走近后预览不会自动重判下一格
3. 放置成功后，preview hold 逻辑仍可能拿旧格子持续做判断
4. “无碰撞体可脚下放置”规则不能在修这条时被回归掉

### 4.3 高概率涉及文件

- `Assets/YYY_Scripts/Service/Placement/PlacementManager.cs`
- `Assets/YYY_Scripts/Service/Placement/PlacementValidator.cs`
- `Assets/YYY_Scripts/Service/Placement/PlacementNavigator.cs`
- 必要时才涉及：`Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`

### 4.4 必做改动

1. 把“放置成功事务”收成统一闭环，明确先后关系：
   - 成功落地
   - 占位真源更新
   - 预览短暂稳态
   - 立即基于世界候选格重算下一格
2. 重新定义 preview 恢复条件：
   - 不能再只认 `Input.mousePosition`
   - 必须认“当前鼠标对应的世界候选格 / 格心是否变化”
3. 树苗占位口径必须统一使用树根格或最终落地格真源，不能再在不同步骤混用偏移世界坐标、宽 AABB 或经验半径
4. “无碰撞体可脚下放置”规则必须保留：
   - 树苗允许脚下放置
   - 播种允许脚下进行
   - 其他无碰撞体 placeable 也不应被玩家自身挡红
5. 有实体碰撞体的 placeable 继续阻挡，不允许被这轮误放开

### 4.5 完成定义

同时满足以下 6 条才算完成：

1. 鼠标停住不动，玩家走近后，预览会自己刷新到新的候选格
2. 连续种植同一方向多格树苗时，不再出现“刚种完还判上一格”的冻结感
3. 同格或已占格不会出现“视觉能放 / 逻辑其实不能放”的幽灵状态
4. 新种下的树苗既不会被错误二次挡住，也不会让下一个候选格错误残留占位
5. 树苗和播种的脚下放置仍成立
6. 箱子等有实体碰撞体的对象继续阻挡

### 4.6 必跑验证

至少要做这 5 组 live：

1. 固定鼠标不动，玩家向前走，连续种 3 到 5 格树苗
2. 刚放下一棵树苗后，鼠标仍停原处，观察 preview 是否自动切到下一格
3. 在已占格边缘连续点放置，确认不会出现视觉可放但最终失败的幽灵格
4. 玩家脚下种树苗
5. 玩家脚下播种 + 有碰撞体物件脚下仍阻挡

### 4.7 禁止漂移

- 不允许因为这条又回头全面重审 `012`
- 不允许只修 runner，不修真实放置主链

## A2. 树木倒下事务

### 4.8 目标

彻底修掉树木 1/2 阶段在倒下期间还能再次吃命中、导致 sprite “复活立起来”的问题。

### 4.9 当前问题范围

当前问题不是单纯动画异常，而是树生命周期没有单向封口。

### 4.10 高概率涉及文件

- `Assets/YYY_Scripts/Trees/TreeController.cs`
- `Assets/YYY_Scripts/Trees/TreeEnums.cs`
- 必要时涉及：掉落、经验或特效触发链相关脚本

### 4.11 必做改动

1. 为树的倒下过程建立明确的单向状态：
   - 已死亡
   - 正在倒下
   - 不可再次命中
2. `ChopDown()` 一开始就要撤销原树对象的再次命中资格
3. 需要的话同步撤销原树碰撞资格，不再拖到 coroutine 最后
4. 临时倒下 sprite 只做表现，不再继续代表业务对象本体
5. 掉落、经验、砍倒音效、后续树桩生成只能提交一次
6. 0 阶段树苗、1/2 阶段中树、树桩阶段三条路径都要重新确认，不允许只修一种阶段

### 4.12 完成定义

同时满足以下 5 条才算完成：

1. 1/2 阶段树在倒下后半程继续挥砍，不再“重新立起来”
2. 倒下期间不会再次吃命中
3. 不会重复掉落、重复经验、重复砍倒反馈
4. 树桩路径和无树桩路径都没有生命周期打架
5. 用户看到的表现和系统里的状态一致

### 4.13 必跑验证

1. 1 阶段树砍倒，倒下快结束时继续挥砍

3. 0 阶段树苗正常流程回归
4. 树桩生成路径与无树桩路径各跑一次

### 4.15 目标

统一“动作前检查 -> 动作后扣账 -> 工具失效 -> 当前动作强制收尾”这整条工具事务。

### 4.16 当前问题范围

1. 最后一刀挥完后，工具坏了但动画还在继续
2. 工具链和动作链没有在“手持物已被移除”这一事实点重新汇合

### 4.18 必做改动

1. 动作起播前统一做一次可用性前检：
   - 当前槽位是否有工具
   - 当前耐久 / 水量 / 精力是否满足起播条件
2. 扣账时机统一放到“这次动作完成后提交”
3. 当次动作若正好把工具扣到损坏：
   - 允许这次动作完整成立
   - 提交后立刻清槽
   - 立即触发损坏特效、气泡、音效
4. 加入明确例外：
   - 如果当前动作依赖的手持物已在提交后被移除 / 失效，则允许立刻强制收尾
5. 强制收尾必须一次性统一处理：
   - 动画计时
   - 手持工具 sprite
   - action lock
   - 长按续挥砍
   - 后续输入衔接
6. 这条事务不能只对斧头成立；锄头 / 水壶也必须确认不会因为资源链改动出现新的起播漏洞

### 4.19 完成定义

1. 耐久为 0 时，动作根本不起播
2. 只剩最后一次耐久时，允许完整挥完这一次
3. 这次挥完后立即损坏、清槽、出反馈
4. 工具消失后，当前动作不会继续假播尾巴
5. 下一次挥砍被前置检查拦下
6. 空水壶、精力不足等场景不会被错误当成“工具已损坏”去强制打断

### 4.20 必跑验证

至少要做这 6 组 live：

2. 下一次立刻再挥，动作不起播
3. 锄头只剩最后一点耐久
