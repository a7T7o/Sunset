## 1. `1.0.3` 收尾与 `1.0.4` 接管边界

### 1.1 `1.0.3` 已经完成的内容

`1.0.3基础UI与交互统一改进` 已经完成的，不应再回头重复开发：

- Tooltip 入口已经从“几乎没有真实接线”推进到背包 / Toolbar / 装备位可显示。
- `ItemTooltip.cs` 已不再只显示 `itemData.description`，而是消费 `ItemTooltipTextBuilder.Build(...)`。
- `InventoryItem` 实例态已经能够进入 Tooltip，至少工具耐久、种袋状态、剩余种子数、保质期可以显示。
- 掉落与拾取链已经补齐 `InventoryItem` 实例态跟随，不再只剩静态 `ItemStack`。
- 食物 / 药水已经接到 `EnergySystem` 与 `HealthSystem`。
- 背包第一行与 Toolbar 的实时同源同步已经补上。
- 箱子 UI 已经开始优先走 `RuntimeInventory`。
- 农田 hover 预览已经接到遮挡系统。
- `1~5` 数字键直选、滚轮 `1~12` 循环这一组 Toolbar 输入边界已经重新钉死。

### 1.2 `1.0.3` 没有真正完成，但已经不能再当局部尾项处理的内容

这些内容虽然是在 `1.0.3` 验收过程中暴露出来的，但本质上已经超出“基础 UI 与交互统一改进”的边界：

- 放置系统的成功判定、导航到达判定、视觉显示、占位提交并不是一个统一状态机。
- 箱子问题已经不是“某个槽位没刷新”，而是旧库存与新库存双向同步存在结构性递归风险。
- 背包 / Toolbar / 当前活跃手持物品保护不只是 UI 同步，而是“交互中的动作权”和“UI 可变更权”没有被统一定义。
- 工具耐久、精力、Tooltip、SO 参数链不是单个 UI 缺口，而是“参数源、运行时状态、日志、显示、消耗”没有形成单一事实源。
- 用户当前关心的不是几个分散 bug，而是“为什么每次局部修完，又会在相邻边界重新坏掉”。

### 1.3 为什么这些问题必须升级到 `1.0.4`

因为它们已经具备三个典型特征：

- 问题跨系统：同一条现象同时涉及输入、导航、预览、运行时数据、UI、容器桥接。
- 问题跨状态：不是某个静态配置错，而是 `Preview -> Locked -> Navigating -> Executing -> UI调整 -> 返回` 这类状态切换边界没有统一。
- 问题跨语义：用户说的是“我只是手持，不等于正在交互”“走到边上就该能放”“显示出来才算真正成功”“箱子应该只有一个真库存”，这些都是语义边界，而不是某个 if 条件。

### 1.4 `1.0.3` 的最终边界

`1.0.3` 到此为止的最终边界应明确为：

- 它已经完成基础 Tooltip / 实例态 / 掉落链 / 食物药水状态 / 背包与 Toolbar 同步 / 箱子运行时容器接线 / 农田 hover 遮挡的第一轮代码闭环。
- 它不再继续承接“全局交互边界重新定义”。
- 它剩余的现场验收问题，从本轮起全部转入 `1.0.4交互全面检查` 统一处理。

### 1.5 `1.0.4` 的接管边界

`1.0.4` 接管的不是“再补几个 bug”，而是以下五条系统性主线：

1. 放置系统与“幽灵占位”
2. 箱子放置时导航 retry 3 次停下
3. 背包关闭后 Toolbar 双选中，以及不该触发的抖动锁定
4. 工具耐久 / 精力 / Tooltip / SO 参数链
5. 箱子 `StackOverflowException` 与双库存同步递归

## 3. `1.0.2 -> 1.0.3 -> 1.0.4` 演进脉络

### 3.1 `1.0.2纠正001` 解决的是“农田交互正确性”

`1.0.2` 的重点是把农具队列、预览清理、导航取消、手持保护等农田交互边界拉回正确口径，核心关注点仍然是：

- 农具自动队列什么时候中断
- 手持槽位什么时候不能切
- 预览残留如何清
- 背包第一行怎样和 Toolbar 同步
- 农田预览怎样不误伤种子和树苗

也就是说，`1.0.2` 主要还是“农田交互修正”。

### 3.2 `1.0.3基础UI与交互统一改进` 解决的是“UI 与实例态出口”

`1.0.3` 已经把主题上提了一步：

- Tooltip 不再只看静态描述，而要进实例态
- 掉落与拾取不能再丢运行时数据
- 食物 / 药水不能再只是日志
- 背包第一行和 Toolbar 必须真的是同一个热槽事实源
- 箱子 UI 不再只认旧 `ChestInventory`

但 `1.0.3` 仍然默认这些问题可以通过一批“局部收口”解决。

### 3.3 `1.0.4交互全面检查` 必须上提为“全局交互处理”

用户在这一轮明确给出的不是“再补一个缺口”，而是直接指出：

- 放置是否成功
- 导航为什么停
- 手持为什么被锁
- UI 为什么双选中
- 精力/耐久为什么只半通
- 箱子为什么递归爆栈

这些已经不是单系统能自证的局部问题，而是整个项目里“交互行为由谁定义、UI 只负责显示还是还能偷偷改变事实、导航边界与执行边界谁说了算、容器数据谁是单一事实源”的总问题。

`1.0.4` 因此不再是某条农田支线的补丁，而是：

- 农田交互
- 放置系统
- 背包与 Toolbar
- 工具运行时状态
- 箱子容器桥接

五条链的统一认知收口。

## 4. 当前全局认知对齐

### 4.1 用户真正不接受的东西是什么

用户这轮最核心的不满，不是“有 bug”这么简单，而是以下三种失真反复出现：

- **行为失真**：逻辑上说成功了，视觉上却像失败；或者视觉看着能切，系统却又把你锁住。
- **边界失真**：同一个动作，在导航、执行、UI、拖拽里各自有一套判断。
- **事实源失真**：箱子到底认旧库存还是新库存、工具耐久到底认按钮、SO、实例态还是 Tooltip、背包第一行到底认 Toggle、认热槽服务还是认关闭面板瞬间的残余状态，这些都没有被明确成单一事实源。

### 4.2 历史上反复理解错过的地方

过去已经发生过、而且在 `1.0.4` 最容易再次漂移的误解主要有四类：

- 把“正在交互”误写成“只要手里拿着就算锁定”。
- 把“到达可放置距离”误写成“走到一个大致差不多的位置就行”，而不是严格统一到同一个边界。
- 把“Tooltip 有显示”误当成“工具运行时参数链已统一”，忽略了实际消耗链没有接上。
- 把“箱子 UI 现在能显示运行时容器”误当成“底层双库存同步结构已经安全”。

### 4.3 当前最容易再次认知漂移的点

- 树苗 / 种子 / 普通 placeable 不能再混成同一条放置语义。
- “UI 关闭后双选中”不能只看 Toggle 视觉，必须看单一热槽事实源是否始终唯一。
- “工具耐久已经有条了”不能等于“耐久链已经工作”，因为当前确认到的是显示链先通，消耗链未统一。
- “箱子现在能拖东西”不能等于“同步递归问题已解决”，因为结构性爆栈在当前代码里仍然能直接解释。

### 4.4 当前应该统一的总边界

本阶段必须统一的，不是某个函数，而是以下五个全局边界：

1. **交互状态边界**：`Preview / Locked / Navigating / Executing / UI Open / Dragging / Container Syncing`
2. **成功定义边界**：成功必须同时意味着“验证通过 + 对象落地 + 显示正确 + 占位提交 + 运行时数据完成登记”
3. **保护槽位边界**：只有真正处于进行中的自动操作 / 放置导航 / 执行态，才保护当前手持槽位；“只是手持”不保护
4. **参数事实源边界**：工具精力、耐久、Tooltip、日志、UI 都必须从 `ToolData + InventoryItem` 这一组事实源导出
5. **库存事实源边界**：箱子必须只有一个可写真源，另一层只能做受控桥接或显示适配

## 5. 五大主题的完整分析

### 5.1 放置系统与“幽灵占位”

#### 5.1.1 历史设计过程

历史上放置系统已经做过多轮纠偏，重点大致经历了三步：

- 先统一 `Preview -> Locked -> Navigating -> Execute` 这条基础状态机。
- 再把种子从农田队列剥离，交给放置系统处理；树苗继续走 `PlaceableItemData / SaplingData` 语义。
- 再把预览和导航尽量统一到 `PlacementPreview.GetPreviewBounds()` 与 `PlacementNavigator` 的同一套 `ClosestPoint` 距离上。

这套历史思路的优点是：它已经意识到“预览边界”和“执行边界”必须统一。问题在于：当前“放置成功”“视觉落地”“真实占位”“后续重验”仍然没有变成一个真正原子化的事务。

#### 5.1.2 已确认事实

- `PlacementManager.LockPreviewPosition()` 会在锁定前做一次重验，随后根据 `navigator.IsAlreadyNearTarget(...)` 判断是直接执行还是先导航。
- `PlacementNavigator.CalculateNavigationTarget(...)` 与 `CheckReached(...)` 都基于 `previewBounds.ClosestPoint(playerCenter)` 计算距离，说明导航目标和到达判定已经尽量同源。
- `PlacementManager.ExecutePlacement()` 的执行顺序是：
  1. 验证快照
  2. 读取 `placementPreview.LockedPosition`
  3. 实例化预制体
  4. 同步层级与 sorting
  5. 扣除背包物品
  6. 广播事件 / 树苗特殊处理
- 树苗仍然走通用放置链，最后通过 `HandleSaplingPlacementWithSavedData(...)` 再去 `InitializeAsNewTree()` 和 `SetStage(0)`。

#### 5.1.3 当前真实问题

用户描述的“树苗连续快速点击时，逻辑上像成功了，但现场没有出现已经种下的内容，像幽灵占位”对应的不是单一点错误，而是这四个状态没有被绑定成单个事务：

- 成功判定是否通过
- 实例化是否成功
- 排序 / 层级 / 可见性是否完成
- 占位与后续重验是否已经提交

只要其中某一步延迟、缺登记、被后续点击覆盖、或和玩家移动中的下一轮重验发生竞争，就会出现“上一轮好像成功，但画面和占位已经脱钩”的体感。

#### 5.1.4 历史推断

当前最可疑的结构问题不是“树苗完全没生成”，而是“放置成功后没有一个统一的 post-commit 状态确认”。尤其树苗走的是：

- 通用放置实例化
- 随后再做 `TreeController.InitializeAsNewTree()`
- 然后再靠排序层与事件链完成剩余接线

这意味着树苗的“落地成功”和“后续可见 / 可判定 / 可阻挡”并不是一步完成。

#### 5.1.5 最终需求口径

- 对用户来说，只有当树苗真正出现在场景里、排序正确、后续验证也把它当作已占位时，才叫“放置成功”。
- 不允许出现“逻辑成功但视觉没落地”或“视觉消失但占位已经提交”的中间态。
- 连点、移动中点击、快速重新预览，都不能把上一轮成功变成幽灵态。

#### 5.1.6 最终设计口径

`1.0.4` 应把放置成功统一为单一事务：

1. `ValidationPassed`
2. `Spawned`
3. `VisualConfigured`
4. `OccupancyCommitted`
5. `InventoryCommitted`
6. `StateReturnedToPreviewOrIdle`

任何一个步骤失败，都必须显式回滚，不允许只回滚一半。树苗、普通 placeable、种子虽可保留不同业务分支，但都必须服从同一套事务式成功定义。

#### 5.1.7 当前风险点

- 树苗目前仍走“通用放置 + 树控制器补初始化”的双阶段路径。
- 快速连点 + 玩家移动中点击，是最容易把“预览成功、实例化成功、占位提交成功”打散的组合。
- 当前还缺少一份能证明“放置对象何时真正进入成功态”的统一日志。

#### 5.1.8 仍待代码验证

- 是否真的存在“对象已生成但因为排序 / 父节点 / layer 同步时序导致视觉像没生成”的路径。
- 是否存在“对象已经生成，但下一轮点击前的重验还没把它纳入障碍集合”的短暂窗口。
- 树苗是否还存在 `TreeController` 初始化完成前后的注册空窗。

### 5.2 箱子放置时导航 retry 3 次停下的问题

#### 5.2.1 历史设计过程

这个问题历史上已经被讨论过很多次，核心始终是：

- 玩家走到哪里算“已经可以放”
- 导航目标点是算到预览中心、边缘、还是碰撞体外扩边界
- 到达判定和最终执行判定是不是完全同一个标准

当前用户再次明确给出的主观口径非常清楚：在人类看来，正确行为就是“走过去，走到预放置物体边上，够得到它的碰撞边界时停下，然后放置成功”。

#### 5.2.2 已确认事实

- `PlacementNavigator.CalculateNavigationTarget(...)` 现在返回的是 `previewBounds` 边缘上的最近点，而不是中心点。
- `PlacementNavigator.CheckReached(...)` 与 `IsAlreadyNearTarget(...)` 都用 `CalculateDistanceToPreview(playerCenter, previewBounds) <= triggerDistance`。
- 也就是说，导航目标和到达检测已经统一到了“玩家中心与 `previewBounds` 的最近距离”。
- 但 `previewBounds` 来自 `PlacementPreview.GetPreviewBounds()`，它本质上是“格子预览联合 Bounds”，不是显式由最终 `placementPrefab` 的实际碰撞边界推出来的统一 reach envelope。

#### 5.2.3 当前真实问题

当前系统虽然已经把“导航目标点”和“到达检测”统一到了同一套数学公式，但还没有把“到达检测所依据的边界”和“最终执行真正依据的可放置边界”统一成同一个业务对象。

所以现在仍然会发生：

- 数学上已经抵达 `previewBounds`
- 但业务上对用户而言，箱子还没有真正到达“碰撞体边上”
- 导航器反复重试几次，最后停下

这说明“到达边界”和“可放置成功边界”只是在公式上接近，还没有在语义上完全合一。

#### 5.2.4 历史推断

当前最像根因的不是导航器本身，而是 `previewBounds` 的来源仍偏向“网格预览”，而不是“最终物体的碰撞包络”。当箱子是多格、底部对齐、Collider 形状和格子联合 Bounds 不完全一致时，这个差异就会被放大成 retry。

#### 5.2.5 最终需求口径

- 玩家到达“预放置物体的实际可放置边界”就应该停下，并立即能成功放置。
- 不允许导航到达、放置判定、执行重验各用一套不同标准。
- 用户当前倾向的设计基准是：以预放置物体预览时的碰撞边界作为可放置距离标准。

#### 5.2.6 最终设计口径

`1.0.4` 应明确引入统一的 `PlacementReachEnvelope` 概念：

- `CalculateNavigationTarget(...)` 用它
- `IsAlreadyNearTarget(...)` 用它
- `CheckReached(...)` 用它
- `LockPreviewPosition()` 的“近距离直接放置”判定用它
- 导航到达后的最终执行重验也用它

这个 envelope 不能再只是“预览格子的联合 Bounds”，而必须能代表最终放置物体对玩家来说真正的可达边界。

#### 5.2.7 当前风险点

- 如果继续沿用“格子联合 Bounds 大致够用”的思路，箱子、树苗、多格 placeable 都会继续在边缘位置出现体感错误。
- 只修 retry 次数、只调 `triggerDistance`，都只能算掩盖，不是统一标准。

#### 5.2.8 仍待代码验证

- `PlacementPreview.GetPreviewBounds()` 是否已经足以代表所有 placeable 的真实可达边界。
- 箱子的 `placementPrefab` Collider 与当前 grid preview bounds 之间到底偏差多大。
- 当前 retry 3 次究竟发生在导航器层，还是发生在到达后重验层。

### 5.3 背包关闭后 Toolbar 双选中，以及不该触发的抖动锁定

#### 5.3.1 历史设计过程

这条线的历史迭代已经非常明确：

- Toolbar 和背包第一行本质上是同一个热槽模型的两个视图。
- 数字键 `1~5`、滚轮 `1~12`、Toolbar 点击三种输入边界已经重新钉死。
- 保护手持槽位的目的，是阻止“进行中的自动操作 / 放置导航 / 执行动作”被 UI 交换破坏。
- 用户已经多次纠正过：只是手里拿着，但没有正在执行、没有正在导航、没有正在交互的情况，绝对不应该被错误锁定。

#### 5.3.2 已确认事实

- `InventorySlotUI` 现在已经直接订阅 `HotbarSelectionService.OnSelectedChanged`，并用 `RefreshSelection()` 驱动自身 Toggle 与 `selectedOverlay`。
- `ToolbarSlotUI` 也在用 `RefreshSelection()` 和 `ForceRestoreToggleState()` 维护自身选中态。
- `GameInputManager.HandleHotbarSelection()` 当前边界是：
  - 面板打开时禁用数字键直选
  - 滚轮允许在 `InventoryService.HotbarWidth` 内循环
  - 数字键只处理 `InventoryService.HotbarDirectSelectCount = 5`
- `GameInputManager.TryRejectActiveFarmToolSwitch(...)` 只有两类情况下会拒绝切换：
  - `HasActiveAutomatedFarmToolOperation()`
  - 当前手持是 `isPlaceable`，且 `HasActivePlacementNavigationSession()`
- `TryRejectActiveFarmToolInventoryMove(...)`、`TryRejectProtectedHeldInventoryMutation(...)`、`TryRejectProtectedHeldInventoryReshuffle(...)` 都是围绕“保护当前进行中的活跃手持槽位”在做反馈。
- `PlayProtectedHeldRejectFeedback(...)` 会同时打 Toolbar 与 Inventory 第一行同槽位 shake。

#### 5.3.3 当前真实问题

从代码上看，双选中问题已经被修过一轮，但用户这次把它重新上提，是因为他要的不是“这次正好看起来正常”，而是以下最终边界：

- 永远只有一个真实热槽
- 背包只是 Toolbar 第一行的另一个视图
- 关闭背包后不允许遗留第二个被选中的视觉态
- 保护逻辑只能锁“正在进行中的动作”，不能锁“只是手持”

也就是说，`1.0.4` 必须把“事实源”和“保护条件”同时收成统一口径，而不能只看某次是否又红框了。

#### 5.3.4 历史推断

过去之所以反复出错，是因为系统里同时存在：

- Toggle 自己的视觉状态
- `selectedOverlay` 的脚本显式状态
- `HotbarSelectionService.selectedIndex`
- 面板打开 / 关闭时的刷新时机

只要其中任何一个不是唯一真源，就会再次出现“两个框都亮”或者“第一次打开没同步，第二次才同步”的问题。

#### 5.3.5 最终需求口径

- Toolbar 与背包第一行是同一个热槽事实的双视图，不允许出现双选中。
- 只有正在交互、正在导航执行、正在放置执行中的物品，才可以触发 shake 与拒绝交换。
- 只是手持但未进入进行中状态的树苗、种子、工具，不允许被错误锁定。

#### 5.3.6 最终设计口径

- `HotbarSelectionService.selectedIndex` 必须是唯一热槽事实源。
- `ToolbarSlotUI` 与 `InventorySlotUI` 只做投影，不再各自暗藏一套真状态。
- “受保护槽位”的判断必须抽成统一策略：只认 active operation / active placement navigation / active execution。
- `TryReject...` 系列只负责拒绝与反馈，不负责偷偷推进动作完成或偷偷改变状态。

#### 5.3.7 当前风险点

- 视觉层仍存在 Toggle、Overlay、面板刷新时机三套来源，稍有新改动就可能复发双选中。
- 如果保护槽位判断仍散在 `GameInputManager`、`InventoryInteractionManager`、`InventorySlotInteraction` 和 UI 点击链里，后续仍会出现“第一次拒绝，第二次却能拖”的漂移。

#### 5.3.8 仍待代码验证

- 背包关闭时是否还存在没有统一走 `HotbarSelectionService` 的补刷新路径。
- 当前“只是手持”的放置物品，是否还有被错误判定成受保护槽位的极端路径。
- `ToolbarSlotUI` 与 `InventorySlotUI` 是否仍会在 ToggleGroup / prefab 配置变化后再次漂移。

### 5.4 工具耐久 / 精力 / Tooltip / SO 参数链

#### 5.4.1 历史设计过程

这条线的历史演进其实已经把“数据层”和“显示层”都做出一半了：

- `ToolData` 已经有 `energyCost`、`hasDurability`、`maxDurability`
- `InventoryItem` 已经有 `currentDurability / maxDurability / UseDurability()`
- `ItemTooltip` 和 `ItemTooltipTextBuilder` 已经能把运行时耐久与种袋状态显示出来
- `EnergySystem` 和 `HealthSystem` 已经有实时状态、日志与 UI Slider
- 食物 / 药水已经能通过 `ItemUseConfirmDialog` 改状态

问题在于：工具使用这一条真正消耗链还没有被收束成统一运行时事实源。

#### 5.4.2 已确认事实

- `ToolData.GetTooltipText()` 已经会输出 `消耗精力` 与 `最大耐久`。
- `InventoryItem` 已经定义了 `HasDurability`、`CurrentDurability`、`MaxDurability`、`UseDurability()`。
- `ItemTooltip.Show(...)` 会立即显示 Tooltip，没有内置 `0.6 秒延迟显示` 机制。
- `ItemTooltipTextBuilder.Build(...)` 已经能追加：
  - 当前耐久 `CurrentDurability / MaxDurability`
  - 种袋开袋状态
  - 剩余种子数
  - 剩余保质期
- `ItemUseConfirmDialog` 已接到 `EnergySystem` 和 `HealthSystem`，但它只负责食物 / 药水，不负责工具使用。
- 当前代码全局检索结果表明：`InventoryItem.UseDurability(...)` 只有定义，没有找到实际消费调用。
- `energyCost` 已经在 `TreeController`、`StoneController`、`PlayerToolHitEmitter` 等资源节点链里被读取和日志化，但并未证明农田工具链、全部工具链都已经统一消费。
- 当前没有查到“鼠标停留 UI 0.6 秒后显示精力条 Tooltip”的现成入口，也没有查到精力条专用 Tooltip driver。

#### 5.4.3 当前真实问题

用户这条诉求的实质不是“加个耐久条”，而是要统一下面四件事：

- 工具初始化时如何自动获得运行时耐久
- 工具使用时如何从 SO 读取精力 / 耐久消耗
- Tooltip 如何显示当前剩余量 / 总量
- 控制台如何打印真实消耗证据

当前系统的问题是：显示链先通了，但消耗链没有统一；部分资源节点会消耗精力，但全局工具耐久没有真正跟着使用；Tooltip 可以显示已有的运行时耐久，但没有证明所有工具都会自动获得并持续维护这个运行时状态。

#### 5.4.4 历史推断

当前最容易“看起来像完成了、实际上没完成”的原因是：

- `ToolData`、`InventoryItem`、`Tooltip`、`EnergySystem` 都存在
- 但“工具被使用一次”这件事，并没有一个全局唯一的 runtime consumption pipeline

于是就会出现：

- 有些工具消耗精力
- 有些工具只在测试按钮下出现耐久条
- Tooltip 能显示某些耐久，但整理背包后耐久丢失
- 丢掉再捡起来又因为实例态链不同而表现不一致

#### 5.4.5 最终需求口径

- 所有工具必须自动进入耐久测试链，不再依赖测试按钮。
- 每次工具使用都必须从 SO 读取精力 / 耐久参数。
- 每次工具使用都必须输出当前耐久、消耗耐久、消耗精力的日志。
- 鼠标进入工具 UI 超过 `0.6` 秒后，显示名称、简介、当前耐久。
- 鼠标进入精力条 UI 超过 `0.6` 秒后，显示当前精力 `剩余量/总量`。

#### 5.4.6 最终设计口径

- `ToolData` 是静态参数真源：决定工具类型、精力消耗、是否有耐久、最大耐久。
- `InventoryItem` 是运行时状态真源：决定当前耐久、动态属性。
- 任意工具使用事件，必须统一进入“读取 `ToolData` -> 校验 / 初始化 `InventoryItem` -> 消耗精力 -> 消耗耐久 -> 输出日志 -> 更新 UI/Tooltip”的同一条流水线。
- Tooltip 不再自己拼业务真相，而是展示这条运行时流水线已经提交过的状态。
- `0.6 秒` 延迟显示必须抽成通用 UI hover driver，而不是分散写在每个槽位脚本里。

#### 5.4.7 当前风险点

- 如果继续只修 Tooltip，而不统一消耗链，用户会再次看到“显示有了，行为还是不对”。
- 如果继续让不同工具各自读自己的参数，不做统一 runtime 入口，后续日志、耐久、精力会继续各走各的。
- 如果精力条 Tooltip 不抽成通用 hover 组件，后续生命条、耐久条、食物状态条都会重复造轮子。

#### 5.4.8 仍待代码验证

- 当前所有工具在进入背包时，是否都已经自动初始化 `InventoryItem` 耐久。
- 农田工具链（锄头 / 水壶等）当前是否完全没有调用耐久消耗。
- 整理背包导致耐久丢失，到底是运行时容器排序、实例态复制还是 UI 刷新链在覆盖。

### 5.5 箱子 `StackOverflowException` 与双库存同步递归

#### 5.5.1 历史设计过程

箱子这条线其实已经暴露出一个非常典型的迁移阶段遗留问题：

- 旧系统里 UI 主要依赖 `ChestInventory`
- 新系统里运行时与存档逐步转向 `ChestInventoryV2`
- 为了兼容，`ChestController` 同时保留了 `_inventory` 和 `_inventoryV2`
- `BoxPanelUI` 又开始优先读 `RuntimeInventory`

这使得项目进入了一种“功能半迁移、结构双真源”的危险阶段。

#### 5.5.2 已确认事实

- `ChestController.OnInventoryChangedHandler()` 里直接调用 `SyncInventoryToV2()`。
- `ChestController.OnInventoryV2ChangedHandler()` 里直接调用 `SyncV2ToInventory()`。
- `SyncInventoryToV2()` 内会调用 `_inventoryV2.ClearItem(...)` / `_inventoryV2.SetItem(...)`。
- `ChestInventoryV2.ClearItem(...)` / `SetItem(...)` 都会继续 `RaiseInventoryChanged()`。
- `SyncV2ToInventory()` 内会调用 `_inventory.ClearSlot(...)` / `_inventory.SetSlot(...)`。
- `ChestInventory.ClearSlot(...)` / `SetSlot(...)` 也会继续 `RaiseInventoryChanged()`。
- 当前代码中没有看到同步方向屏蔽、无事件写入 API、也没有看到显式 reentrancy guard。

#### 5.5.3 当前真实问题

这不是“某个方法写错了”，而是结构上出现了最危险的情况：

- 旧库存变化会触发同步到新库存
- 新库存变化又会触发同步回旧库存
- 双方同步时用的还是会继续发事件的普通写接口

这天然就会导致递归回环。用户现场报出的 `StackOverflowException`，和当前代码链完全一致。

#### 5.5.4 历史推断

哪怕当前某些场景因为数据恰好相同而暂时没炸，这个结构也仍然是危险的。因为只要一个槽位发生：

- Clear
- Set
- Merge
- Runtime item 变更

就可能触发事件风暴。

#### 5.5.5 最终需求口径

- 箱子必须能稳定放入和取出物品，不能再递归爆栈。
- UI、存档、运行时容器只能有一个单一真源。
- 如果必须保留兼容桥接，也必须是受控的，而不是双向实时互踢。

#### 5.5.6 最终设计口径

`1.0.4` 的箱子最终口径应明确为：

- `ChestInventoryV2` 作为运行时与存档的单一真源更合理。
- 旧 `ChestInventory` 只能做受控兼容层，不能继续作为另一份会主动发事件的平级真源。
- 同步必须满足以下至少一条：
  - 单向同步
  - 显式桥接阶段禁用回调
  - 无事件写入 API
  - reentrancy guard

如果这些都没有，结构上就不安全。

#### 5.5.7 当前风险点

- 当前 `BoxPanelUI` 已经优先读 `RuntimeInventory`，但底层桥接仍然是双向事件互踢。
- 只修某个 `ClearItem` 或 `ClearSlot` 的局部判断，不能从根上解决双真源问题。

#### 5.5.8 仍待代码验证

- 哪些外部系统仍然硬依赖旧 `ChestInventory` 写接口。
- 迁到 `ChestInventoryV2` 单一真源后，旧 UI / 保存链 / 交互链会不会出现新的兼容缺口。

## 6. 当前阶段的总收束结论

### 6.1 已确认事实

- `1.0.3` 的第一轮代码闭环已经完成，不能再把当前问题当成“基础 UI 尾项”。
- 放置系统、背包 / Toolbar、工具运行时参数链、箱子同步，都已经跨过局部修补边界。
- 箱子递归爆栈在当前代码里有直接结构性证据。
- 工具耐久显示链存在，但统一消耗链没有被确认完成。
- `0.6 秒` Tooltip 和精力条 Tooltip 入口当前没有现成实现证据。

### 6.2 历史推断

- 树苗“幽灵占位”高概率是成功、显示、占位、后续重验没有被事务化统一，而不是单纯一条 sorting order 漏写。
- 箱子 retry 3 次停下高概率是“导航边界”和“实际可放置边界”依然不是同一个 envelope。

### 6.3 最终需求口径

`1.0.4` 后续代码必须围绕四个统一目标推进：

1. 成功判定统一
2. 交互保护统一
3. 参数事实源统一
4. 容器事实源统一

### 6.4 当前风险点

- 如果直接开始写代码，不先统一这些口径，很容易继续出现“修好一个点，旁边又坏”。
- 如果只修最显眼的报错，不先处理单一事实源，后续还会在其他入口重复爆。

### 6.5 仍待代码验证的问题

- 树苗快速连点与玩家移动并发时，到底是哪一步先脱节。
- 箱子导航 retry 的具体 stop 条件是否来自 `previewBounds` 偏差还是到达后重验偏差。
- 整理背包导致耐久丢失的真实链路。

### 6.6 本文档的最终定位

从本轮开始，后续 `1.0.4` 的所有代码工作都应以这里写清的五条主线、四种证据口径和最终建议边界为总入口。任何后续实现如果偏离这里，必须在 `1.0.4` 工作区文档与 memory 中显式回写原因，而不能静默改口。
