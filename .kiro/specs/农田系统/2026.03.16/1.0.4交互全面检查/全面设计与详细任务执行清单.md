# 1.0.4 交互全面检查｜全面设计与详细任务执行清单

> 当前线程语义：`全局交互V3（原：农田交互修复V2）`
>
> 本轮主线：`全局交互处理`
>
> 本文档身份：`1.0.4 的设计总纲、实施顺序与验收总表`

## 0. 本阶段总目标

`1.0.4` 的总目标不是继续打零散补丁，而是在 `main` 语义下，把下面五个系统性问题统一收口：

1. 放置系统与“幽灵占位”
2. 箱子放置时导航 retry 3 次停下
3. 背包关闭后 Toolbar 双选中，以及不该触发的抖动锁定
4. 工具耐久 / 精力 / Tooltip / SO 参数链
5. 箱子 `StackOverflowException` 与双库存同步递归

交付标准不是“改了几个脚本”，而是：

- 交互边界统一
- 事实源统一
- 日志与证据统一
- 验收脚本统一
- 每一刀都可回退

## 1. 设计原则

### 1.1 单一事实源原则

- 热槽选中只认 `HotbarSelectionService.selectedIndex`
- 工具静态参数只认 `ToolData / FoodData / PotionData / EquipmentData`
- 工具运行时状态只认 `InventoryItem`
- 箱子运行时库存只认一个真源，不能双真源互踢

### 1.2 交互状态机先于视觉补丁

后续所有实现必须先解决“当前处于什么状态、谁有权改变状态”，再解决视觉表现。不能再用视觉修口来掩盖状态机不一致。

### 1.3 成功必须可见、可登记、可回滚

任何“成功”都必须同时满足：

- 行为上成功
- 视觉上可见
- 运行时已登记
- 后续重验已纳入

否则视为失败或半提交状态，必须回滚。

### 1.4 文档级设计先于代码扩写

`1.0.4` 的每一块都要先用当前文档定义清楚：

- 根因假设
- 代码链
- 状态边界
- 验收标准
- 回滚点

之后才允许进入代码。

### 1.5 一刀一收原则

除非同一条逻辑没有闭环，否则每次实现都应是：

- 一个最小主题
- 一个可验证 checkpoint
- 一次白名单收口

不能顺手把相邻高风险主题混成一次大改。

## 2. 统一交互边界

### 2.1 统一状态边界

本阶段涉及的状态至少有：

- 放置：`Idle / Preview / Locked / Navigating / Executing`
- 农田工具：`Idle / Preview / Locked / Navigating / Executing`
- UI：`Closed / Open`
- 容器拖拽：`Idle / Holding / Swapping / ProtectedReject`
- 箱子桥接：`Idle / SyncingFromLegacy / SyncingFromV2`

后续任何实现都不能再默认“只看当前功能自己的状态”。

### 2.2 统一“受保护槽位”边界

受保护槽位必须只在以下场景成立：

- 自动农具队列正在执行或导航
- 放置物品正在导航或执行
- 当前动作已经进入不会被 UI 随意交换的进行中状态

以下场景绝不保护：

- 只是手持
- 只是预览跟随
- 只是把树苗 / 种子拿在手里但没有实际开始导航执行

### 2.3 统一 Tooltip 边界

- 进入 UI 后 `0.6s` 才显示
- 离开立即消失
- Tooltip 只展示静态参数真源和运行时状态真源组合出来的结果
- Tooltip 本身不负责修补错误数据链

### 2.4 统一导航 / 到达 / 执行边界

放置导航相关的以下四处必须用同一套 reach envelope：

- 目标点计算
- 已在附近判定
- 到达判定
- 执行前最后重验

## 4. 五大问题的根因假设与设计方案

### 4.1 放置系统与“幽灵占位”

#### 4.1.1 根因假设

- `PlacementManager.ExecutePlacement()` 已经把“验证、实例化、层级、扣物品、事件广播”串起来了，但这些步骤还不是一个可确认成功 / 失败的事务对象。
- 树苗额外走 `HandleSaplingPlacementWithSavedData(...)`，说明树苗落地还带二段初始化。
- 快速连点 + 玩家移动中点击时，极有可能出现上一轮成功尚未完全提交、下一轮预览/重验已经开始的竞争窗口。

#### 4.1.2 目标设计

- 抽出统一的 `PlacementExecutionTransaction` 或等价结构。
- 让普通 placeable、树苗、种子都在统一的提交/回滚框架里运行。
- 对树苗额外补一条“落地后可见性 / 排序 / 注册完成确认”。

#### 4.1.3 需要核实的代码链

- `PlacementManager.LockPreviewPosition()`
- `PlacementManager.ExecutePlacement()`
- `PlacementManager.ExecuteSeedPlacement()`
- `PlacementManager.HandleSaplingPlacementWithSavedData()`
- `PlacementManager.SyncLayerToPlacedObject()`
- `PlacementManager.SetSortingOrder()`

#### 4.1.4 需要补的结构改造

- 放置执行结果结构：记录 `Spawned / VisualReady / OccupancyCommitted / InventoryCommitted`
- 失败回滚钩子：销毁对象、恢复库存、撤销占位、恢复状态机
- 树苗落地后的 post-spawn confirm

#### 4.1.5 需要补的日志与证据

- 一次放置的事务编号
- 锁定位置
- reach envelope
- 实例化结果
- 层级 / sorting 配置结果
- 占位提交结果
- 回滚原因

#### 4.1.6 验收标准

- 树苗连续快速点击时，不再出现“成功但没显示 / 像幽灵占位”的情况
- 失败时会明确失败，不留下半提交残留
- 下一轮重验一定能识别上一轮已经成功的对象

### 4.2 箱子放置时导航 retry 3 次停下

#### 4.2.1 根因假设

- `PlacementNavigator` 已经统一了数学公式，但 `previewBounds` 仍然偏向格子预览 bounds，而不是最终物体的真实 reach envelope。
- 到达判定与最终执行判定用的是同套距离公式，但可能不是同一个业务边界。

#### 4.2.2 目标设计

- 引入 `PlacementReachEnvelope`。
- 让 `PlacementPreview` 在预览阶段就能算出真正的 envelope。
- 放置系统全链复用该 envelope，而不是各处各算一遍。

#### 4.2.3 需要核实的代码链

- `PlacementPreview.GetPreviewBounds()`
- `PlacementNavigator.CalculateNavigationTarget()`
- `PlacementNavigator.CheckReached()`
- `PlacementNavigator.IsAlreadyNearTarget()`
- `PlacementManager.LockPreviewPosition()`

#### 4.2.4 需要补的结构改造

- `PlacementPreview` 增加“用于交互 reach 的 bounds”与“用于遮挡 / 视觉的 bounds”区分
- `PlacementNavigator` 改为只收 envelope，不再直接猜 grid bounds

#### 4.2.5 需要补的日志与调试证据

- 导航开始时记录 envelope min/max
- 已在附近判定的距离值
- 到达判定触发的距离值
- 执行前最后重验是否仍使用同一 envelope

#### 4.2.6 验收标准

- 箱子放置不再 retry 3 次后原地停下
- 玩家走到边上就能稳定放置
- 多格 placeable 不再靠调 `triggerDistance` 临时修补

### 4.3 背包关闭后 Toolbar 双选中，以及不该触发的抖动锁定

#### 4.3.1 根因假设

- `Toggle`、`selectedOverlay`、`HotbarSelectionService`、面板刷新时机之间仍然可能存在多真源。
- 受保护槽位判断目前虽然已经收敛，但仍散在 `GameInputManager` 与多个 UI 脚本调用点。

#### 4.3.2 目标设计

- 把“热槽选中”明确成单一真源。
- 把“受保护槽位”明确成单一策略函数。
- UI 只负责显示、恢复和反馈，不再藏业务判断。

#### 4.3.3 需要核实的代码链

- `InventorySlotUI.RefreshSelection()`
- `ToolbarSlotUI.RefreshSelection()`
- `ToolbarSlotUI.ForceRestoreToggleState()`
- `GameInputManager.TryRejectActiveFarmToolSwitch()`
- `GameInputManager.TryRejectActiveFarmToolInventoryMove()`
- `GameInputManager.TryRejectProtectedHeldInventoryMutation()`
- `InventoryInteractionManager` / `InventorySlotInteraction` 的 reject 调用点

#### 4.3.4 需要补的结构改造

- `ProtectedHeldSlotPolicy` 或等价 helper
- UI 层的统一 reject feedback adapter
- 面板开关后的统一热槽 refresh hook

#### 4.3.5 需要补的日志与调试证据

- 当前热槽索引
- 当前受保护槽位
- 拒绝原因：自动农具 / 放置导航 / 执行态
- reject shake 触发来源

#### 4.3.6 验收标准

- 关闭背包后不会出现两个选中框
- 只是手持但未进入进行中状态时，允许正常交换
- 真正进行中的动作仍会拒绝交换，并只抖当前真实受保护槽位

### 4.4 工具耐久 / 精力 / Tooltip / SO 参数链

#### 4.4.1 根因假设

- 静态参数层、运行时状态层、UI 展示层都存在，但没有全局统一的工具使用流水线。
- `InventoryItem.UseDurability()` 没有接入真实工具使用主链。
- `0.6 秒` Tooltip 延迟和精力条 Tooltip 入口目前并不存在。

#### 4.4.2 目标设计

- 抽一条统一工具运行时消费流水线。
- 工具进入背包时自动初始化 runtime durability。
- 工具使用时统一读取 SO、消费精力、消费耐久、输出日志、刷新 UI。
- Tooltip 入口统一改为延迟 hover driver。

#### 4.4.3 需要核实的代码链

- `InventoryItem.SetDurability()` / `UseDurability()`
- `ToolData.energyCost` / `hasDurability` / `maxDurability`
- `PlayerToolHitEmitter.BuildContext()`
- `TreeController.GetEnergyCost()` / `TryConsumeEnergy()`
- `StoneController.GetEnergyCost()` / `TryConsumeEnergy()`
- `ItemTooltip.Show()`
- `ItemTooltipTextBuilder.Build()`
- `EnergySystem` / `HealthSystem`

#### 4.4.4 需要补的结构改造

- `ToolRuntimeStateService` 或等价辅助层
- `DelayedTooltipPresenter` 通用 hover 组件
- 精力条 / 生命条 Tooltip presenter
- 工具使用统一日志出口

#### 4.4.5 需要补的日志与调试证据

- 工具名称 / itemId
- 当前耐久
- 消耗耐久
- 消耗精力
- 剩余精力

#### 4.4.6 验收标准

- 所有工具自动具备耐久运行时状态
- 每次使用都能看到耐久和精力日志
- Tooltip 在 `0.6s` 后显示 `剩余量/总量`
- 整理背包、丢弃再拾取都不会丢失耐久

### 4.5 箱子 `StackOverflowException` 与双库存同步递归

#### 4.5.1 根因假设

- `ChestController` 当前仍让 `_inventory` 和 `_inventoryV2` 互相监听并互相调用带事件的写接口。
- 这构成了最典型的递归桥接结构。

#### 4.5.2 目标设计

- 确立一个唯一真源，优先建议 `ChestInventoryV2`。
- 旧库存只做只读适配或单向镜像。
- 同步期间禁止回调反弹。

#### 4.5.3 需要核实的代码链

- `ChestController.SyncInventoryToV2()`
- `ChestController.SyncV2ToInventory()`
- `ChestController.OnInventoryChangedHandler()`
- `ChestController.OnInventoryV2ChangedHandler()`
- `ChestInventory.ClearSlot()` / `SetSlot()`
- `ChestInventoryV2.ClearItem()` / `SetItem()`

#### 4.5.4 需要补的结构改造

- reentrancy guard
- 无事件写入 API 或 bridge silent mode
- UI 对旧库存接口的残余依赖清理
- 最终落到单向桥接或单真源适配

#### 4.5.5 需要补的日志与调试证据

- 同步方向
- 触发事件来源
- guard 是否生效
- 是否使用 silent 写入

#### 4.5.6 验收标准

- 箱子可稳定放入、取出、交换、排序
- 不再出现 `StackOverflowException`
- UI、运行时、存档对箱子内容的认知一致

## 8. 实施顺序

### 8.1 串行优先顺序

必须优先串行的顺序如下：

1. **箱子单真源 / 递归桥接治理**
   - 原因：它当前直接阻断箱子所有交互测试
2. **工具运行时参数链统一**
   - 原因：没有统一消费链，Tooltip 和日志验收都不可信
3. **背包 / Toolbar / 受保护槽位统一**
   - 原因：它依赖前两者提供清晰的运行时状态边界
4. **放置 reach envelope 统一**
   - 原因：要先统一“到达边界”，再去处理“幽灵占位”
5. **放置事务与树苗幽灵占位治理**
   - 原因：它建立在 reach envelope 与 post-commit 结构上

### 8.2 可并行部分

可以谨慎并行的只有：

- Tooltip 延迟显示层
- 精力条 / 生命条 Tooltip presenter
- 日志输出格式统一

但并行前提是不要与高危共享文件写冲突，尤其要注意：

- `GameInputManager.cs`
- `InventorySlotUI.cs`
- `ToolbarSlotUI.cs`
- `ChestController.cs`
- `PlacementManager.cs`

## 9. 风险与回滚点

### 9.1 主要风险

- 继续在双真源箱子结构上做局部修补，会把递归暂时压住但不解决根因。
- 继续在 UI 层单独修 Toggle 或 Overlay，可能再次制造热槽双真源。
- 继续只做 Tooltip 展示，不改运行时消费链，会再次出现“看起来有了、实际上还是假的”。

### 9.2 回滚点

每个主题都必须保留单独 checkpoint，优先按下面粒度切：

1. 箱子桥接治理
2. 工具运行时参数链
3. 背包 / Toolbar 统一
4. reach envelope 统一
5. 放置事务 / 树苗幽灵占位

如果某一刀导致新的系统性回归，必须能单独回退，不允许一次大提交把五块绑死。
