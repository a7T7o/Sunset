# Codex规则落地工作区设计说明

## 1. 文件身份
- 本文件用于定义 `D:\Unity\Unity_learning\Sunset\.kiro\specs\Codex规则落地` 的工作区边界。
- 它回答三个问题：
  - 什么内容属于真正的治理工作区。
  - 什么内容只属于 `000_代办` 的代办镜像层。
  - 这一轮为什么要把旧的 `000_代办/codex` 迁离。

## 2. 当前最终口径
- `Codex规则落地` 是 Sunset 当前正式治理工作区。
- `000_代办/codex` 不再承载阶段正文、根层 `memory`、阶段 `tasks` 或执行证据。
- `000_代办/codex` 现在只保留：
  - TD 镜像
  - 当前待办总览
  - 指向正式工作区的读取入口

## 3. 为什么必须迁移
- 用户已经明确裁定：
  - `000_代办` 只能用于记录和读取代办，不能再充当工作区。
  - `worktree` 是 Sunset 的红线例外，不得继续通过复杂结构把现场重新绕乱。
- 旧结构的问题不是“文档太多”，而是职责混层：
  - 代办区同时承载工作区正文。
  - 根层 `memory` 与阶段目录同时被当作治理主入口。
  - live 入口把“代办镜像区”误写成“正式工作区”。

## 4. 新结构原则
- 正式治理工作区固定在：
  - `D:\Unity\Unity_learning\Sunset\.kiro\specs\Codex规则落地`
- 工作区根层默认只保留：
  - `memory.md`
  - `memory_0.md`
  - `design.md`
- 具体执行动作固定落在分阶段目录：
  - `01_...`
  - `02_...`
  - `...`
  - `12_治理工作区归位与彻底清盘`
- 每个阶段目录各自持有：
  - `tasks.md`
  - `memory.md`
  - `design.md` 按需存在
- 不再恢复根层“大一统 tasks”。

## 5. 与 000_代办 的关系
- `000_代办/codex` 是读取层，不是执行层。
- 代办区里的 TD 文件只做三件事：
  - 说明当前是否仍有待办。
  - 指向正式工作区和对应阶段。
  - 标记是“进行中 / 已完成 / 已封板 / 仅历史”。

## 6. 与 Steering 的关系
- `.kiro/steering/` 仍是规则正文源。
- `Steering规则区优化/当前运行基线与开发规则/` 仍是现行活文档入口层。
- `Codex规则落地` 负责承接跨线程、跨工作区、跨规则的治理执行与收口。
- 因此现在的三层关系固定为：
  - `steering`：规则正文
  - `当前运行基线与开发规则`：live 入口与现行口径
  - `Codex规则落地`：治理工作区正文

## 7. 当前清盘后的硬规则
- shared root 默认为：
  - `D:\Unity\Unity_learning\Sunset @ main`
- 日常开发默认为：
  - `shared root + branch`
- `worktree` 仅保留给：
  - 事故隔离
  - cleanroom
  - 特殊实验
- 代办区不再承载工作区正文。

## 8. 一句话口径
- 从本轮开始，Sunset 的治理正文进入 `Codex规则落地`，而 `000_代办/codex` 只保留为 TD 镜像和读取入口。
