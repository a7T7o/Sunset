# 2026-04-04_Codex规则落地自续工_Town常驻链与场景健康闭环_01

当前唯一主线固定为：

- 把 `Town` 自己的常驻链 / bootstrap / 场景健康收成“别的线程敢继续消费”的基础设施基线。

## 一、为什么这刀由治理线程自己接

这轮不是在抢别线 owner，而是在补一个当前没有更合适 owner 的 Town 主闸：

1. `PersistentManagers.cs` 与 `PersistentObjectRegistry.cs` 是上次 `Town` 承接稳定化 slice 自己带进来的运行时兜底。
2. 用户最新 live 仍然坐实：
   - 常驻链自举还有残留 warning / root 问题
   - `Town` 进入时 scene health 还没有干净证明
3. 这条线如果不先闭环，`spring-day1 / NPC / 场景承载` 都只能拿到半悬空的 Town。

## 二、你这轮唯一切片

你这轮只做下面这一刀：

- `Town` 常驻链与场景健康最小闭环

合法范围：

1. `Assets/YYY_Scripts/Service/PersistentManagers.cs`
2. `Assets/YYY_Scripts/Data/Core/PersistentObjectRegistry.cs`
3. `Assets/000_Scenes/Town.unity` 仅在确有必要时
4. 与 `audio listener` / bootstrap / duplicate instance / root 挂载直接相连的最小现场核查

明确禁止：

1. 不要回到 `spring-day1` 前半段剧情
2. 不要碰 `Primary.unity`
3. 不要碰 `DialogueUI.cs`
4. 不要碰 `CameraDeadZoneSync.cs`
5. 不要顺手扩成 Town 美术、NPC 群像或字体底座

## 三、这轮必须拿到的结论

1. `PersistentManagers` 的重复实例 / 自举降级，当前究竟是不是 Town 现场真 blocker。
2. `PersistentObjectRegistry` 的挂根与 `DontDestroyOnLoad` 警告，当前是不是仍然存在真实路径。
3. `There are no audio listeners in the scene` 当前到底属于：
   - `Town.unity` scene wiring
   - 进入链时机问题
   - 还是别的链路副作用
4. 最终给出 `Town` 场景健康裁定：
   - `可进 sync-ready`
   - 或 `仍需外线继续接`

## 四、完成定义

只有下面几条同时成立时，这刀才算过：

1. 已给出最小 live 或等价低负载证据，不再只靠静态 YAML。
2. Town 常驻链主 warning 已被清掉，或已被证明不是 blocker 且原因说死。
3. `audio listener` 现场已被修掉，或其真实 owner 已被精确钉出。
4. 没有把 Town 总闸偷换成别的线。
5. Town blocker matrix 已被更新到可用于下一轮裁定。

## 五、收尾要求

1. 如果这刀完成且 Town 仍未完全放行，要把剩余 blocker 精确压成：
   - owner

   - 类型

   - Town 自身基线已站住
