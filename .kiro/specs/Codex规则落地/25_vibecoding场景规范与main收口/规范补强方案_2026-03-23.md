# 规范补强方案 2026-03-23

## 1. 先说结论
- 当前改革不是失败了，而是已经把主干从“branch-first + 大一统治理”扳回到了“main-only + 白名单收口”。
- 现在暴露出来的问题，不是方向错，而是还有几类高频场景没有被制度真正吃掉。
- 所以当前不该回到旧重治理模型，也不该继续放任“靠项目经理肉眼追责”；正确动作是补 4 层窄而硬的规则。

## 2. 这轮已经证明有效的部分
- `main-only + whitelist-sync + exception-escalation` 已经比旧 branch-first 更贴近真实开发。
- `git-safe-sync.ps1` 上的 `.cs` 代码闸门已经有效，能拦住“改完代码不自查，等用户贴报错”的低级问题。
- `scene-build` 这类特殊 worktree 可以作为例外长期存在，但必须被明确标记为例外，而不是默认开发模型。
- 快速出警通道已经能用，说明“责任线程速查”这条路是对的。

## 3. 这轮真正暴露出来的漏洞

### 3.1 规则更新了，但线程入口没同步到底
- 典型表现：
  - 有些线程还把旧 branch carrier 当 blocker。
  - 有些线程不知道自己现在已经可以直接在 `main` 上白名单收口。
  - prompt 更新晚一步，就会继续沿旧认知梦游。
- 根因：
  - 规范正文更新了，但“线程实际会读到的入口”没有同轮强制同步。
- 补强：
  - 以后任何规则变更，必须同轮同步 4 个入口：
    - `AGENTS.md`
    - 当前规范快照
    - 批次 `README.md`
    - 通用分发前缀 prompt
  - 四者没同步齐，不算规则生效。

### 3.2 热文件卫生还没变成线程硬义务
- 典型表现：
  - `Primary.unity` 被多个线程留下残留，最后还要靠治理线追责谁来扫地。
  - 线程容易误把“我这条线的调试残留”留在 scene 里。
- 根因：
  - 现在只有“高危目标要注意”，但还没有把“退出前必须扫自己留的热文件残留”写成线程默认动作。
- 补强：
  - 新增 `热文件卫生规则`：
    - 只要线程碰过 `Scene / Prefab / Primary.unity / 热脚本`，在收口前必须额外回一段：
      - 我留了什么
      - 哪些保留
      - 哪些清掉
      - 当前是否已对我这条线 clean
  - 这段不是可选说明，而是热文件线程的固定回执字段。

### 3.3 现在只有 `.cs` 闸门，缺少 Scene / Prefab / Asset 侧的轻量闸门
- 典型表现：
  - 代码已经有闸门，但 Scene 残留、TMP 字体副产物、Prefab 脏改、测试条挂场景里，这些还会漏。
- 根因：
  - 当前闸门主要覆盖代码正确率，还没有覆盖 Unity 资源卫生。
- 补强：
  - 新增 `资源卫生闸门`，但保持轻量，不走重验证：
    - 如果白名单里包含 `.unity` / `.prefab` / `.asset`
    - 收口前必须做一次 `git status --short` 对照
    - 明确分类：
      - 本轮有效内容
      - 自动副产物
      - 调试残留
      - 他线脏改
  - 自动副产物默认不得混入 checkpoint，除非线程显式声明“这是本轮必须保留的有效资源改动”。

### 3.4 “例外机制”还存在，但退场机制不够硬
- 典型表现：
  - worktree / branch carrier 的历史成果容易长期挂着，线程自己也会把它们继续当 blocker。
  - 用户和治理线会反复为“这条线到底该不该还停在分支上”消耗注意力。
- 根因：
  - 现在定义了例外进入条件，但没有把“例外完成后必须如何退场”写成一张固定清单。
- 补强：
  - 新增 `例外退场四步`：
    1. 先判断例外现场里还有没有未迁入的真实有效成果。
    2. 有就先迁最小 checkpoint 到 `main`。
    3. 再把旧分支 / worktree 遗留从 blocker 清单里显式移除。
    4. 最后要求线程以 `main` 语义继续下一刀，禁止继续把旧 carrier 当默认起点。

### 3.5 Unity / MCP live 窗口有基线，但没有线程级占窗模板
- 典型表现：
  - 遮挡线程验证时，项目里会反复问“现在能不能给他一个清净时刻”。
  - 线程容易擅自扩张验证策略，比如复制副本、乱开第二实例。
- 根因：
  - 现在有 MCP 基线，但“申请 live 验证窗口时该怎么说”还不够模板化。
- 补强：
  - 新增 `live 窗口四字段`：
    - 需要什么实例
    - 最多占多久
    - 只做什么
    - 做完退回到什么状态
  - 没有这四字段，不批准 live 写或 live 验证。

### 3.6 回执格式还不够稳定，导致治理线仍要人工二次解释
- 典型表现：
  - 同一个状态，有的线程回“我做完了”，有的线程回“我这里还有 dirty”，但不说 dirty 是谁的。
  - 还有回执和聊天记录散落的问题。
- 根因：
  - 当前回执格式虽有模板，但还没完全压成所有线程默认遵守的最小字段集。
- 补强：
  - 把线程回执最小字段正式收紧为：
    - 当前在改什么
    - changed_paths
    - 是否触碰高危目标
    - 是否需要 Unity / MCP live 写
    - 当前是否可直接提交到 `main`
    - 提交 SHA（如有）
    - 当前 `git status` 是否 clean
    - blocker_or_checkpoint
    - 一句话摘要
  - 如果碰过热文件，再追加一组 `热文件卫生字段`。
  - 所有回执默认直接写回批次固定文件，不再让项目经理手动建新文件夹。

### 3.7 缺少“每刀结束就收”的时间纪律
- 典型表现：
  - 有些线程能做很多，但收口晚，导致现场长期挂 dirty，责任越来越难分。
- 根因：
  - 当前有白名单收口原则，但还没有把“多大算一刀”讲得足够硬。
- 补强：
  - 明确一条新纪律：
    - 只要形成可复核 checkpoint，就优先先收这一刀，不继续顺手叠第二刀。
  - 例外只限：
    - 同一逻辑未闭环
    - 同一 live 验证窗口内必须连跑
    - 用户明确要求“一次性做到某个更完整 checkpoint”

## 4. 建议立刻升级成现行规则的内容
- 规则入口四同步，未同步齐不算生效。
- 热文件卫生字段，成为所有 Scene / Prefab 线程必填项。
- 资源卫生闸门，补到 `.unity` / `.prefab` / `.asset` 收口前。
- 例外退场四步，写死到当前规范快照。
- live 窗口四字段，作为 Unity / MCP live 验证的固定申请格式。
- 回执最小字段收紧，并固定写回批次文件。
- 一刀一收，默认不叠刀。

## 5. 不该继续加重的部分
- 不回退到 `branch-first`。
- 不恢复 queue / grant / ensure-branch 作为普通开发前置。
- 不把所有线程重新拉回治理审批。
- 不为了资源卫生问题发明第二套重型闸门。
- 不把自动 hook 当当前最高优先级；当前更重要的是把入口、回执和卫生规则压实。

## 6. 推荐落地顺序
1. 先把上述 7 条补强项写进当前规范快照和治理阶段正文。
2. 再把“并发线程当前版本更新前缀”统一改成新字段版。
3. 然后只观察一轮真实线程使用反馈。
4. 如果一轮后仍频繁出现 Scene / Prefab 脏残留，再决定是否上第二层自动化检查。

## 7. 一句话判断
- 当前规范需要加强，但不是推翻重来。
- 正确方向是：继续坚持 `main-only`，同时把“入口同步、热文件卫生、资源卫生、例外退场、live 占窗、固定回执、一刀一收”这 7 根钉子彻底钉死。
