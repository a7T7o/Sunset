# 并发线程统一回执与 main 收口机制

## 目标
- 解决现在这个真实问题：
  - 线程都在并发干活
  - 但没有一套简单办法判断“谁现在可以进 `main`、谁要再等等、谁在撞车”
- 这份机制是给项目经理直接用的，不是再造 queue 帝国。

## 核心原则
- 可以做“一次统一广播、一次统一收件、一个统一收口窗口”。
- 但不建议做“所有线程揉成一个大杂烩 commit”。
- 推荐做法是：
  - 同一波次统一回执
  - 同一波次统一裁定
  - 同一波次顺序进 `main`
  - 每线程仍保留自己的提交边界和 diff 归属

## 默认提交规则
- 默认就是：
  - 一线程只提交自己修改的内容
  - 一线程只提交自己的白名单路径
  - 一线程一次只收一刀自己的 checkpoint
- 这条规则现在应视为主流程，而不是例外。
- 只有命中下面这些情况时，才不走“线程自己白名单提交”：
  - 撞到同一个高危目标
  - 同一个 Scene / Prefab / 热脚本被多线程同时改
  - 改动跨线程耦合，必须人工拆刀
  - 成果还停在 branch carrier / worktree，尚未迁入 shared root `main`

## 关于“提交 id”
- Git 真正的提交 ID 是 SHA，不能手动改成你想要的格式。
- 但可以标准化两种东西：
  - checkpoint 名称
  - commit message
- 当前推荐统一成：
  - `YYYY.MM.DD_<线程名>_<编号>`
- 例如：
  - `2026.03.22_农田_01`
  - `2026.03.22_spring-day1_01`
  - `2026.03.22_导航检查_01`
- 这样做的作用是：
  - 人类可读
  - 便于按线程追历史
  - 不影响 Git 自己生成 SHA

## 为什么不建议一个超大 commit
- 回退太痛。
- 以后查“这是谁改的”会很差。
- 某一个线程出问题时，没法只拿掉它那一刀。
- 用户后续想让某个线程继续补改，也会失去清晰边界。

## 推荐机制：一波收件，顺序入 main

### 1. 治理线程发固定群发 prompt
- 所有当前活线程先暂停扩写，只做一次收口回执。

### 2. 每个线程只回固定字段
- 当前在改什么
- changed_paths
- 是否触碰高危目标
- 是否需要 Unity / MCP live 写
- 代码闸门是否通过
- Unity / MCP 红错验证是否闭环
- 当前是否可以提交到 `main`
- 如果不能提交，阻塞是什么
- blocker_or_checkpoint
- 一句话摘要

## 红错验收层级
- 从现在起，必须把“代码闸门”和“运行态红错验收”拆成两层事实：
  1. 代码闸门
     - `UTF-8`
     - `git diff --check`
     - 程序集级编译
  2. 运行态红错验收
     - 默认先走 CLI 的 fresh recompile
     - 默认先走 CLI 的 fresh console
     - CLI 覆盖不到时，再补 direct MCP / live 取证
- 第一层通过，不等于第二层已经通过。
- 当前默认顺序固定为：`CLI first, direct MCP last-resort`
- 凡是改动触及运行时代码、scene、prefab、asset、UI、剧情、交互、输入、管理器链，就不允许再把：
  - `validate_script`
  - `CodexCodeGuard`
  - `git diff --check`
  单独写成“无红错”。
- 如果 CLI 与 direct MCP / Unity 当前都不可用，可以形成本地 checkpoint，但回执只能写：
  - `代码闸门通过`
  - `Unity 红错验证未闭环`
  - `live / console 待补`
  不能写成“已可放心交接”。

### 3. 治理线程做一张批次表
- A组：可以直接进 `main`，且路径不冲突
- B组：可以进 `main`，但要调整顺序
- C组：暂时不能进 `main`
- D组：撞到同一高危目标，必须人工裁定

### 4. 收口窗口里顺序提交
- 不是大家同时乱提。
- 是同一波次里按裁定顺序连续收口。
- 可以是：
  - 线程自己按裁定顺序提交
  - 或治理侧统一白名单收口
- 但原则都是：
  - 一线程一刀
  - 一线程一份 changed_paths
  - 一线程一条 checkpoint 说明

### 5. 收口后保留一份批次记录
- 记录每个线程：
  - 收件时间
  - changed_paths
  - 是否入 main
  - 提交 SHA
  - 如果没进 main，为什么

## 这套机制和“main-only”是什么关系
- `main-only` 不是“任何时候都可以乱提交”。
- 它的正确含义是：
  - 大家都在 `main` 语义下开发
  - 但真正入 `main` 的窗口要有一个最小裁定
- 所以这套机制不是回到旧 queue。
- 它只是给 `main-only` 加一个最小收口层。

- 或你准备发起一轮集中收口
- 或 shared root 里 dirty 已经开始混线
