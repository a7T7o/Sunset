# 2026-04-05_CLI主导爆红规范_日志可判定补强_01

## 1. 这次补强到底补什么

2026-04-04 那轮已经把 Sunset 的 no-red 方向改成：

- `CLI first, direct MCP last-resort`
- `代码闸门 != Unity 红错闭环`

但还缺最后一刀：

**线程就算口头上照新规说了，治理层仍然不能只看日志就判断它到底有没有按规矩跑。**

这轮补的不是“再发明一套新 no-red 规则”，而是把现有规则收成：

1. 日志里必须出现哪些字段
2. 这些字段允许写什么值
3. 看日志的人怎样一眼判断这条回执合不合格

目标只有一个：

**以后看线程日志、看 memory、看技术审计层，就能直接判断它有没有真的按 CLI-first 的 no-red 规范执行。**

## 2. 新的硬标准：`No-Red 证据卡 v2`

从现在起，只要线程在回执、聊天、memory 或 handoff 里对代码任务写出以下任一类结论：

- `无红错`
- `红错已清`
- `Unity 可直接交接`
- `可直接提交`
- `checkpoint 可验`
- 任何同义的 no-red / handoff-ready 说法

技术审计层都必须额外原样补一张 `No-Red 证据卡 v2`，最少包含：

1. `cli_red_check_command`
2. `cli_red_check_scope`
3. `cli_red_check_assessment`
4. `unity_red_check`
5. `mcp_fallback`
6. `mcp_fallback_reason`
7. `current_owned_errors`
8. `current_external_blockers`
9. `current_warnings`

少一项，都算“日志不可判定”。

## 3. 每个字段允许写什么

### 3.1 `cli_red_check_command`

写这轮真正用的 Sunset CLI 主命令，例如：

- `validate_script`
- `errors`
- `compile`
- `no-red`

如果你这轮根本没跑 Sunset CLI，这一项就不能写成空白；你必须直接承认这轮没拿到 CLI 证据。

### 3.2 `cli_red_check_scope`

写这轮 CLI 真正检查的目标范围，例如：

- 单个脚本路径
- 一组明确脚本路径
- 这次 `errors` 的 fresh console 样本范围

不允许只写“当前脚本”“我的改动”“已检查”这种不可判定口径。

### 3.3 `cli_red_check_assessment`

只允许写 CLI 当前原值：

- `no_red`
- `own_red`
- `external_red`
- `unity_validation_pending`
- `blocked`

不允许自由翻译成：

- `pass`
- `基本没问题`
- `应该可交接`

这类会破坏可判定性的词。

### 3.4 `unity_red_check`

只允许写：

- `pass`
- `blocked`
- `live-pending`
- `not-required`

解释：

- `pass`：这轮 Unity 红错验证已真正跑完并拿到结果
- `blocked`：Unity / MCP / bridge / fresh validation 这层被卡住
- `live-pending`：代码层证据有了，但 Unity/live 这一层还没闭环
- `not-required`：这轮任务本质上不要求 Unity 红错验收

### 3.5 `mcp_fallback`

只允许写：

- `not-needed`
- `required`
- `blocked`

解释：

- `not-needed`：CLI 已经给出足够答案，不需要 direct MCP
- `required`：CLI 当前不够，下一步必须升级 direct MCP / live
- `blocked`：理论上该 fallback，但当前连 fallback 也被基线或现场阻断

### 3.6 `mcp_fallback_reason`

如果 `mcp_fallback != not-needed`，只允许写：

- `baseline_fail`
- `unity_validation_pending`
- `blocked`
- `scene_live_flow_required`
- `playmode_required`
- `inspector_required`

不允许写“有点复杂”“我感觉需要 live”“情况特殊”这类模糊理由。

### 3.7 `current_owned_errors / current_external_blockers / current_warnings`

这 3 项必须分开写，不允许把 warning、owned red、external red 混成一句。

## 4. CLI 主流程默认顺序

以后高频 no-red 核查默认只按这条顺序走：

1. `doctor`
   - 用于看桥、基线、推荐动作是否健康
2. `validate_script`
   - 默认主入口
   - 优先显式脚本路径；需要原生兼容时再用 `--name / --path / --level`
3. `errors`
   - 看 fresh console
4. `manage_script validate`
   - 只作更轻的原生补充验证
   - 不替代 `validate_script`
5. direct MCP / live
   - 只在 CLI 当前覆盖不到时升级

## 5. 看日志时怎么判这条回执合不合格

### 合格

必须同时满足：

1. 有 `No-Red 证据卡 v2`
2. `cli_red_check_command + scope + assessment` 都写了
3. `unity_red_check` 明确
4. 如果用了 fallback，`mcp_fallback_reason` 也明确
5. `owned / external / warnings` 三项分开写了

### 不合格

出现下面任一项就直接判不合格：

1. 只写“无红错 / 可交接 / 可提交”，没有证据卡
2. 只写 `validate_script 过了`
3. 只写 `CodexCodeGuard 过了`
4. 只写 `git diff --check` 干净
5. 不写 `cli_red_check_assessment`
6. `mcp_fallback_reason` 自由发挥
7. 把 warning 和 error 混写
