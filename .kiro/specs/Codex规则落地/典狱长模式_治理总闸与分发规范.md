# 典狱长模式_治理总闸与分发规范

## 文件身份
- 本文件是 Sunset 治理线程进入“典狱长模式”时的 live 规范源。
- 它补充的是“先审回执、再判是否继续发 prompt”的总闸口径。
- 它不替代业务工作区正文；它只约束治理线程怎样判断该不该继续推进。

## 触发词
- 用户明确说出以下任一触发词时，默认进入典狱长模式：
  - `典狱长`
  - `典狱长上货`
  - `上货`
- 如果用户虽然没说触发词，但任务实质上就是“审线程回执、判停发、按需给下一轮 prompt”，也按典狱长模式处理。

## 核心目标
- 不再把“收到回执”自动等同于“继续发下一轮 prompt”。
- 先判断线程当前到底属于：
  - 继续施工
  - 应停给用户验收
  - 应停给用户分析 / 审核
  - 已无需继续发
- 只有真正还该继续施工的线程，才继续写 prompt 文件。

## 固定裁定顺序
1. 回读最新用户裁定、最新线程回执、最新 prompt 文件、相关工作区记忆。
2. 先判断上一轮完成定义是否真的命中，不允许把 checkpoint 冒充总完成。
3. 先核这条回执是否有效：
   - 是否已经显式回答 `当前 own 路径是否 clean`
   - 若属于 cleanup / white-list sync / owner 报实轮次，`当前 own 路径是否 clean` 是否为 `yes`
4. 再把线程判入四类之一：
   - `继续发 prompt`
   - `停给用户验收`
   - `停给用户分析 / 审核`
   - `无需继续发`
5. 只有第一类才生成新的 prompt 文件。
6. 其余三类一律停下，并向用户明确说明：
   - 为什么现在不该继续发
   - 当前应由谁出手
   - 后续恢复点是什么
7. 如果线程这轮虽然说了很多工程事实，但仍没把“做成什么 / 还剩什么 / 下一步”说成人话，典狱长模式应把它视为“汇报不合格”，而不是直接默认用户已经被解释清楚。

## 回执有效性与 own-path-clean 闸门
- 只要本轮回执里带有 cleanup、white-list sync、owner 报实或 shared-root 尾账收口语义，`当前 own 路径是否 clean` 就是硬闸门。
- 如果线程没有回答这项，或答案不是 `yes`：
  - 这条回执不能算“当前这一刀已闭环”
  - 不能直接进入下一轮 feature prompt
  - 只能继续作为 blocker / owner 报实输入
- 典狱长模式要先拦这种“自己尾账还没收干净，却想直接进入下一轮 feature”的漂移。

## 用户可读汇报闸门
- 典狱长模式除了核 `own-path-clean`，还必须核另一层：
  - 当前回执是不是已经把结果说成用户能拿来下令的话。
- 若用户当前关心的是：
  - 现在做到哪了
  - 实际产出是什么
  - 还剩什么
  - 下一步要做什么
  则线程回执至少要明确回答：
  - `当前主线`
  - `这轮实际做成了什么`
  - `现在还没做成什么`
  - `当前阶段`
  - `下一步只做什么`
- 如果线程只给：
  - `changed_paths`
  - 技术 checkpoint
  - 参数名
  - 实验日志
  - compile / live 噪音
  却没有先回答上面 5 件事，那么这条回执对用户仍然是不合格的。
- 典狱长模式下，治理线程不应代替线程长期做这种翻译；应把它作为下一轮 prompt 或当前轮退回补充的硬要求。

## 四类裁定口径

### 1. 继续发 prompt
- 线程上一轮没有命中完成定义。
- 当前剩余问题仍然属于线程自己可继续施工、自验、取证的范围。
- 不需要用户先亲手体验、亲手分析或亲自拍板。

### 2. 停给用户验收
- 当前剩余点已经进入用户体验判断、观感判断、手感判断、最终玩法判断。
- 线程自己该做的 compile / test / live / 自验已经做到头了。
- 下一步最重要的不是再补一刀，而是用户先确认“是否真的过线”。

### 3. 停给用户分析 / 审核
- 当前需要用户做的是判断方向、判断方案、判断设计、判断是否接受某种取舍。
- 或者用户已经明确说了“这轮先讨论，不继续施工”。
- 这种情况下只能做解释、整理、审查，不应擅自继续发施工 prompt。

### 4. 无需继续发
- 该线程这一轮已经完成且当前不需要继续推进。
- 或者当前主线已被切走，线程暂时不在活跃施工清单里。

## 典狱长模式下的停发条件
- 用户明确说“这轮先停下来讨论 / 分析 / 审核”。
- 线程已经进入用户终验阶段。
- 当前阻塞点本质上需要用户亲自判断，而不是线程继续做代码或 live。
- 线程上一轮只做到 checkpoint，还没命中完成定义，但当前剩余动作不该由它继续做。
- 线程 own 路径还没 clean，或回执没能证明它已经对自己 clean。

## 典狱长模式下的 prompt 生成规则
- 默认仍然采用“详细正文写进文件”的两层交付。
- 给用户的聊天转发格式固定为：
  1. `对应文件在：`
  2. 平铺列出最新 prompt 文件路径
  3. `你可以直接这样发给 XXX：`
  4. 一个 `text` 代码块
- `text` 代码块里必须直接包含：
  - `请先完整读取 [最新 prompt 路径]`
  - 当前唯一主刀
  - 禁止漂移或停发要求
- 不再使用旧的“路径 + 极短短句”分离格式。
- 不要把 prompt 全文重新粘回聊天。

## 完成定义与健康口径
- 典狱长模式的完成定义不能只看“规则和 skill 已经写出来”。
- 当前统一拆成 3 层：
  1. `文档层`
     - `AGENTS.md`、治理规范、SKILL 正文、注册表已经对齐
  2. `审计层`
     - `skill-trigger-log.md`、线程记忆、用户可见转发格式已经对齐
     - 不再混用旧的自由发挥格式
  3. `运行层`
     - 至少有 1 次干净的显式命中样本
     - 样本里真的执行了“四类裁定”
     - 最终转发也真的落成“对应文件在 + 可直接复制 text 代码块”
- 在拿到第 3 层证据前，默认只能说“文档层已落地”或“审计层已补齐”，不能对外宣称“典狱长模式已经完全闭环且有效”。

## 与其他治理 skill 的关系
- `skills-governor`：负责前置核查、skill 路由、收尾审计。
- `sunset-warden-mode`：负责典狱长模式本体。
- `sunset-governance-dispatch-protocol`：负责治理线程的分发与回收协议。
- `sunset-prompt-slice-guard`：负责把下一轮压成单轮可验硬切片。
- `sunset-acceptance-handoff`：负责进入用户终验阶段时的验收指南与回执单。

## 2026-03-26 追加补充：cleanup 不再作为常规兜底
- 普通线程的 own dirty / untracked 默认必须在当前这一刀里自己收口；治理线程不再默认替大家周期性发 cleanup。
- 只有命中 cross-thread mixed dirty、hot-file / singleton config asset owner 不清、或 shared root 现场被 hot / mixed 目标卡死时，才允许升级治理 cleanup / owner 报实批次。
- 当前 Sunset 的强制报实目标至少包括：
  - `Primary.unity`
  - `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`
  - `ProjectSettings/TagManager.asset`
  - `Assets/Editor/StaticObjectOrderAutoCalibrator.cs`

## 2026-03-27 追加补充：两类 hot target 的裁定差异
- `Primary.unity`
  - 继续按单 scene writer 处理
  - 当它处于 `dirty + unlocked + ownerless` 时，治理线程不应再把它描述成“有人正在写”
  - 更准确的判词是：“这是无人接盘的 stale mixed scene，等待当前指定 writer 接盘”
  - 当前批准的接盘顺序为：`NPC -> spring-day1`，直到用户改写
- `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`
  - 继续是强制报实热点，但不再默认按单 writer 停工
  - 典狱长模式下应先审是否撞到同一触点 / 同一行为链
  - 若只是不同触点并发，可继续推进，并把最终合并责任交给 integrator

## 一句话口径
- 典狱长模式不是“看到回执就继续发”，而是“先审回执判停发，再决定要不要发下一轮 prompt”。
