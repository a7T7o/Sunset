# 看守长模式_线程验收交接规范

## 文件身份
- 本文件是 Sunset 在“线程完工后进入用户终验交接”场景下的 live 规范源。
- 它不替代 `典狱长模式_治理总闸与分发规范.md`。
- 二者职责固定分开：
  - `看守长` = 业务线程自己的验收交接
  - `典狱长` = 治理线程的审回执 / 判停发 / 再分发

## 触发词
- 用户明确说出以下任一触发词时，默认进入看守长模式：
  - `看守长`
  - `看守长上场`
  - `喊看守长`
- 如果用户虽然没说触发词，但任务实质上是在要求：
  - `把你这轮做完的东西整理成我能直接验的清单`
  - `给我一个回执单`
  - `别再只说请复测`
  也按看守长模式处理。

## 默认输入
- 如果当前就是 Sunset 业务线程自己的对话，且用户没有明确要求去审别的线程，默认输入固定为：
  - 当前线程自己最近完成或准备交接的这一刀
- 但在实际执行里，优先锚定的不是一个宽泛的“当前这条线”，而是：
  - 上一轮刚完成
  - 且已经向用户汇报过
  - 现在用户准备开始验收
  的那一刀
- 如果用户是在完成汇报后直接喊 `看守长`，就默认审刚才那轮完成项，不再二次确认
- 如果这次不是紧跟在完成汇报后，但上下文仍明显是当前业务线程自己的验收场景，线程也必须自己回看当前工作区 / 子工作区 memory、当前线程 memory、当前 slice / 最新回执 / 最新验证，选最近一次“已完成且已向用户交代过结果”的切片来交接；最多只允许用一句话报明“这次我按 XXX 这一刀给你收验收包”
- 线程必须主动回看：
  - 当前工作区 / 子工作区 memory
  - 当前线程 memory
  - 当前 slice / 最新回执 / 最新验证
  - 当前 `thread-state`
- 不允许先反问用户：
  - `把你要我审的线程丢过来`
  - `把回执或 prompt 文件发来`
  - `告诉我要审哪条线`

## 只有这些情况才允许转成外部审稿
- 用户明确要求：
  - `帮我审另一条线程`
  - `我贴一个回执给你，你替我验`
  - `你不是这条线本人，替我看这个 prompt / 这几条线`
- 只有这时，线程才允许索取外部材料。

## 交付顺序
1. `看守长总判断`
2. `线程已自验`
3. `本轮仍需用户终验的点`
4. `统一前提`
5. `建议顺序`
6. `验收清单矩阵`
7. `最少必测包`
8. `快捷回执单`
9. `完整版回执单`

## 开头口径
- `看守长` 回复第一屏直接进入验收交接正文。
- 不要先写：
  - `我先切到看守长模式`
  - `我先读当前治理口径`
  - `我先按某个 skill 处理`
  - `把要审的线程 / 回执 / prompt 发来`
  - `在 / 看守长到了 / 看守长到位`
- 用户已经在当前线程里喊 `看守长` 时，这些都属于多余前摇。
- 如果需要一句锚定说明，也只能是一句短的“这次我按 XXX 这一刀给你收验收包”；后面立刻进入正文。

## Sunset 叠加要求
- 看守长模式在 Sunset 下默认叠加：
  - `acceptance-warden-mode`
  - `sunset-acceptance-handoff`
- 线程不能把“结构 checkpoint”包装成“体验已过线”。
- 线程必须先说清自己已经做了哪些自验：
  - compile / targeted test / MCP 辅助证据 / 静态核对
- 只有真正需要用户亲手、亲眼、亲自判断的部分，才交给用户。

## 禁止事项
- 不要把 `看守长` 偷换成 `典狱长`。
- 不要把当前线程自己的验收交接，回成“请把线程名 / prompt / 回执发给我”。
- 不要丢失“上一轮刚完成并已汇报的那一刀”这个默认锚点。
- 如果没有紧邻的一轮完成汇报，也不要因为不确定就退回向用户索材；先自行锚定最近一次“已完成且已汇报”的任务，再用一句话报明。
- 不要先讲模式切换、治理位、skill 名，再开始验收正文。
- 不要只回 `在 / 看守长到了 / 看守长到位` 这种空首屏，再等用户继续给材料。
- 不要只说：
  - `请复测`
  - `等你测`
  - `你再看看`
- 不要没有 `最少必测包`。
- 不要没有回执单。
