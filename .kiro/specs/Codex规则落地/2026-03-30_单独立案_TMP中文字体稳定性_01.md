## 这轮重新钉死的硬证据

### 1. 生成器本身就在造动态字体

`Assets/Editor/Story/DialogueChineseFontAssetCreator.cs` 当前明确把字体资产创建为：

- `AtlasPopulationMode.Dynamic`
- `isMultiAtlasTexturesEnabled = true`

这意味着它不是“烘焙完就基本稳定”的静态字体，而是可能继续长 atlas / glyph 的动态资产链。

### 2. 当前资产序列化结果已经呈现出动态膨胀特征

这轮回读当前资产 YAML，至少可确认：

- `DialogueChinese SDF.asset` 已有大块 `m_GlyphTable` / `m_CharacterTable` / `m_AtlasTextures`
- `DialogueChinese V2 SDF.asset` 已有大块 `m_GlyphTable` / `m_CharacterTable` / `m_AtlasTextures`
- `DialogueChinese Pixel SDF.asset` 同样已经大块膨胀
- 多个资产都开着 `m_IsMultiAtlasTexturesEnabled: 1`

### 3. 当前日志快照已经明确打出 importer 不一致

`C:\Users\aTo\AppData\Local\Unity\Editor\Editor-prev.log` 当前至少明确记录到：

- `Importer(NativeFormatImporter) generated inconsistent result for asset ... "Assets/TextMesh Pro/Resources/Fonts & Materials/DialogueChinese SDF.asset"`
- 栈里出现 `TMP_EditorResourceManager:DoPostRenderUpdates()`

同一份日志里也能看到：

- `DialogueChinese V2 SDF.asset` 被重复导入

这轮最新日志快照虽然没有把所有历史症状都重新打印一遍，但最低限度已经足够证明：  
当前问题不是“某个业务线程改坏了一份字体引用”，而是这组共享 TMP 中文字体资产本身就存在 importer 稳定性风险。

## 当前裁定

1. 这组字体 dirty 一律不再按业务 checkpoint 吞并。
2. `Day1 / UI / NPC / 导航 / Story` 任何线程都不应再把这组资产混进自己的“已完成”里。
3. 从现在开始，所有与 `DialogueChinese*` 相关的 churn 都应先按：
   - `共享字体稳定性案`
   - `不可信运行/导入副产物`
   处理，而不是先按业务内容处理。

## 这轮故意不做什么

1. 不直接重建字体资产。
2. 不在业务线程里继续追 `V2` 风格问题。
3. 不在 `Primary.unity`、`Story/UI` 或业务 prefab 里顺手继续扩修字体底座。

## 下一步唯一正确切法

开一条独立的 `TMP 中文字体稳定化` 切片，只做下面顺序：

1. 先冻结这组共享字体从业务线剥离出来，不再混进业务 sync
2. 以最小隔离环境复现 importer 是否稳定
3. 确认哪几份资产必须回到已提交基线
4. 在独立切片里二选一：
   - 重建为更确定性的静态字体资产
   - 承认动态版本只适合作为非默认实验资产并降级

## 本案完成定义

1. 业务线程不再把字体 churn 误报成业务 blocker
2. `DialogueChinese*` 资产的默认链路和实验链路被明确切开
3. importer 不一致问题被隔离到独立 owner 切片，不再污染其他业务线验收
