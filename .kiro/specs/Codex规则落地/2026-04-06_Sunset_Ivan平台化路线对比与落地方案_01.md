# Sunset｜Ivan 平台化路线对比与落地方案

## 0. 文档定位

- 本文不是一次“换包建议”。
- 本文是针对 Sunset 当前 Unity MCP 基础设施，给出的：
  - `IvanMurzak/Unity-MCP` vs `CoplayDev/unity-mcp` 平台级对比
  - 当前现网与未来平台方向的拆分判断
  - 是否值得立项、怎么立项、先做什么、别先做什么
- 本文目标不是证明某一方“绝对更先进”，而是把三件事说清楚：
  1. Sunset 当前已经深度绑定了什么
  2. Ivan 真正吸引人的地方到底是什么
  3. 如果我们要认真走 Ivan 方向，正确姿势是什么

---

## 1. 先说结论

### 1.1 一句话结论

`IvanMurzak/Unity-MCP 值得作为 Sunset 的下一代 AI 平台实验线正式立项，但不应被当成“今天直接替换现网 CoplayDev/unity-mcp”的动作。`

### 1.2 更展开一点的结论

- `CoplayDev/unity-mcp` 更适合 Sunset 当前已经跑起来的工程化工作流：
  - Editor 控制
  - 结构化工具面
  - 多实例
  - compile / console / scene / script 自动化
  - 当前 CLI 和治理体系已经全面绑在它上面
- `IvanMurzak/Unity-MCP` 更适合 Sunset 下一阶段你真正开始要的方向：
  - AI 平台化
  - Skills
  - Reflection
  - Runtime / In-Game
  - 更强的“AI 开发伙伴”想象空间
- 所以最合理的动作不是二选一，而是双线：
  - `现网生产线继续用 Coplay`
  - `平台实验线正式立 Ivan`

### 1.3 本文最终主张

`不建议今天直接切主线；建议今天正式立项。`

## 3. 两套仓库各自是什么

## 3.1 CoplayDev/unity-mcp 是什么

从仓库定位和 MCP Registry 信息看，Coplay 这套更像：

- Unity Editor bridge
- local Python server
- 明确的 structured tools / resources
- multiple Unity Editor instances
- 更偏“稳定的 Unity Editor 自动化底座”

它的强项是：

- 工具面清晰
- 资源面清晰
- Editor 自动化很强
- 多实例支持明确
- 对于“脚本、console、scene、editor state”这类能力非常适合工程化封装

当前 Sunset 用它做出来的典型东西就是：

- compile-first / no-red 流程
- `sunset_mcp.py`
- `errors` / `manage_script validate`
- 当前治理里对 live 验证、CLI fallback、No-Red 证据卡的结构化表达

## 3.2 IvanMurzak/Unity-MCP 是什么

从公开 README 看，Ivan 这套更像：

- `AI Skills, MCP Tools, and CLI for Unity Engine`
- 强调 `100+ built-in tools`
- 强调 Runtime / In-Game
- 强调 Reflection
- 强调 CLI + setup + Skills
- 更偏“AI Game Developer 平台”

它吸引人的点，不是“比 Coplay 多几个工具”这么简单，而是：

- 它在叙事上更像一个完整 AI 平台，而不是单纯 Editor bridge
- 它把：
  - Skills
  - Tooling
  - Reflection
  - Runtime
  - CLI
  这些东西压到了同一个方向里

这恰好就是你现在明显已经开始心动的原因。

## 4. 核心对比表

| 维度 | IvanMurzak/Unity-MCP | CoplayDev/unity-mcp | 对 Sunset 的现实意义 |
|---|---|---|---|
| 产品定位 | AI 平台 / Skills / Runtime / Reflection / CLI | Unity Editor bridge / Python server / structured tools | Ivan 更像未来平台，Coplay 更像当前底座 |
| 核心能力风格 | 广、强、平台化、灵活 | 结构化、稳定、Editor-first | Ivan 更强，Coplay 更稳 |
| Runtime / In-Game | 强调支持 | 不是主叙事 | Ivan 在“游戏内 AI”方向更香 |
| Reflection | 很强 | 不是主卖点 | Ivan 更适合做平台层 |
| Editor 工具标准化 | 有，但不是唯一中心 | 非常强 | Coplay 更贴 Sunset 当前高频流程 |
| 多实例 | 有多模式思路 | 明确支持并有 `set_active_instance` | Coplay 更贴当前实际运行治理 |
| CLI 叙事 | 很强 | 有 CLI，但不是平台化 CLI 叙事中心 | Ivan 更符合你“我要一个平台”的直觉 |
| 当前 Sunset 绑定度 | 0 | 很高 | 切 Ivan 不是轻量替换 |
| 当前 no-red 适配 | 中 | 高 | 现网继续 Coplay 更稳 |
| 下一代平台潜力 | 高 | 中 | Ivan 更值得立平台实验线 |

而是更大的东西：

换句话说，你心动的不是 “Ivan 比 Coplay 多几个按钮”，而是：

`Ivan 更像一个 AI 平台起点。`

这点我认为你看得很准。

## 6. 为什么我仍然不建议今天直接替换现网

不是因为 Ivan 不强，而是因为 Sunset 现在已经不是一张白纸了。

### 6.1 替换动作的真实成本不在“装包”

真实成本在这些地方：

1. CLI 绑定要改
   - 现在的 [sunset_mcp.py](/D:/Unity/Unity_learning/Sunset/scripts/sunset_mcp.py) 绑定的是 Coplay 工具语义

2. 基线要改
   - 当前是 `unityMCP + 8888 + MCPForUnity`
   - 这套运行口径和自恢复逻辑都要重写

3. Unity 侧桥和自恢复逻辑要改
   - 当前 [CodexMcpHttpAutostart.cs](/D:/Unity/Unity_learning/Sunset/Assets/Editor/CodexMcpHttpAutostart.cs) 明确是围绕 Coplay 当前桥语义在写

4. 治理规则要改
   - current no-red、fallback、baseline、thread-state、handoff 都是围绕现有能力面压出来的

5. 现场验证口径要重写
   - 当前很多语义例如 `errors`、`manage_script validate`、`refresh_unity`，已经被实际工作流依赖

### 6.2 所以问题不是“值不值得”，而是“什么时候值”

我的判断不是：

- “Ivan 不值得”

而是：

- “Ivan 值得，但不该用直接替换现网的方式起步”

## 7. 正确的立项方式

## 7.1 项目名建议

建议正式命名为：

`Sunset AI Platform 0.1｜Ivan 实验线`

## 7.2 立项目标

不是：

- “把 Coplay 换成 Ivan”

而是：

`验证 IvanMurzak/Unity-MCP 是否值得成为 Sunset 下一代 AI 平台底座。`

## 7.3 立项不做什么

- 不直接替换现网 Coplay
- 不破坏当前 no-red CLI 生产链
- 不在没有 PoC 的情况下重写治理体系
- 不先做“大迁移”，再来验证是否真有收益

## 7.4 成功定义

只要达到下面 4 条，这个实验线就算成功：

1. Ivan 能在 Sunset 相关环境里稳定跑通
2. 能完成至少 3 类典型任务
3. 在至少 1 个平台维度上明显优于 Coplay
4. 迁移收益可以被量化，不再只靠感觉

---

## 8. 我建议的推进路线

## 8.1 第 0 期：立项与边界冻结

目标：

- 正式承认 Ivan 是平台实验线
- 同时冻结一个原则：
  - `实验线 != 现网替换`

产出：

- 本文
- 一份实验目标清单
- 一份成功标准
- 一份“不碰现网”的边界说明

## 8.2 第 1 期：独立沙盒 PoC

目标：

- 不在当前 Sunset 主现场直接换包
- 单独做一个最小实验环境

推荐两种方式：

1. 独立 Unity sandbox 项目
2. Sunset 的单独实验副本 / 独立 worktree / 单独试验环境

要验证的最小能力：

1. 基础连接与安装
2. CLI / MCP / Skills 跑通
3. Reflection 能否真为 Sunset 提供价值
4. Runtime / In-Game 能力是否值得长期投资

## 8.3 第 2 期：只测 3 类 Sunset 典型任务

### A 类：高频工程任务

目标：

- 看它对当前高频 no-red 链路有没有真实帮助

任务建议：

- 读 console
- 读编译状态
- 验证脚本
- 触发局部工具动作

### B 类：结构化 Editor 自动化任务

目标：

- 看它做 Editor 自动化的组织方式是否优于当前 Coplay

任务建议：

- 修改 scene object
- 调 inspector / 资源
- 调自定义工具
- 触发 editor command

### C 类：平台化任务

目标：

- 这才是 Ivan 最可能真正胜出的地方

任务建议：

- 把某个 Sunset 业务 C# 方法暴露给 AI
- 做 Reflection 驱动的业务工具
- 做 Runtime / In-Game 调试与控制

## 8.4 第 3 期：形成裁定

这一期不再谈感觉，只看 5 条：

1. 性能
2. 稳定性
3. 可控性
4. 平台收益
5. 迁移成本

最终只允许三种裁定：

1. `继续只用 Coplay`
2. `Coplay 生产 + Ivan 平行平台`
3. `计划性迁移到 Ivan`

我当前预判，最可能的结果是：

`Coplay 生产 + Ivan 平行平台`

## 9. Sunset 视角下，最值得先测的能力

## 9.1 最值得优先测的，不是“替代 errors”

不要把第一刀浪费在最像 Coplay 的地方。

真正值得优先测的是 Ivan 更有机会赢的地方：

1. Reflection 暴露业务方法
2. Runtime / In-Game 能力
3. Skills 驱动的工程工作流
4. 平台式 CLI 与项目能力组织

### 9.2 当前最不值得优先测的

1. 先去复刻现有 `errors`
2. 先去复刻现有 `manage_script validate`
3. 先去复刻现有 `sunset_mcp.py`

原因：

- 这些是 Coplay 当前已经跑通的地方
- 就算 Ivan 也能做，不代表它更值得
- 先复刻只会把实验线拉回“替代品比较”，而不是“平台能力验证”

## 10. 如果用户坚持“成本几乎不计”，我怎么理解这句话

这句话不能理解成：

- “迁移真的没成本”

更准确的理解应该是：

`用户愿意接受为了平台方向而额外付出工程代价。`

这会改变优先级，但不会改变事实。

也就是说：

- 迁移成本依然客观存在
- 只是你愿意为未来平台收益买单

在这个前提下，我的建议会从：

- “暂时别动”

变成：

- “可以立项，但要用实验线方式动”
