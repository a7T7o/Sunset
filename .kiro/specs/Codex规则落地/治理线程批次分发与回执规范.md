# 治理线程批次分发与回执规范

## 文件身份
- 本文件只约束 `Codex规则落地` 这条治理线程。
- 它不约束业务线程如何写功能。
- 它只回答一件事：
  - 当治理线程需要向多个线程分发 prompt，并要求回写收件单时，应该怎么做。

## 适用场景
- 同一轮里要向多个 Sunset 线程分发专属 prompt。
- 需要线程自己去领取对应文件，而不是让我手工逐条转抄。
- 需要把线程回复压缩到最小格式，减少聊天上下文污染。
- 需要从固定回收卡里收件，而不是靠聊天里长回复做判断。
- 需要先审回执、再决定哪些线程该继续发 prompt，哪些线程该停给用户。

## 典狱长模式先判后发
- 如果用户明确说出 `典狱长`、`典狱长上货`、`上货`，或本轮实质上是在做治理总闸裁定，这一轮先进入典狱长模式，再决定是否分发。
- 典狱长模式下，治理线程必须先把线程判成四类：
  - `继续发 prompt`
  - `停给用户验收`
  - `停给用户分析 / 审核`
  - `无需继续发`
- 只有 `继续发 prompt` 的线程，才进入本文件后续的批次分发或单条分发。
- 如果线程已经进入用户验收、用户分析或用户审核阶段，治理线程必须停下并说明原因，不要机械继续发 prompt。

## 硬规则
1. 这类“批次分发文件”必须按轮次新建。
2. 批次分发文件必须放在：
   - `D:\Unity\Unity_learning\Sunset\.kiro\specs\Codex规则落地\`
3. 不要把批次分发文件固化在某个阶段目录里。
4. 阶段目录只保留：
   - 线程专属 prompt
   - 固定回收卡
   - 阶段文档与治理裁定
5. 批次分发文件不是长期维护资产。
6. 一轮分发结束后：
   - 可以保留为本轮证据
   - 也可以在后续收口时删除或归档
   - 但不能把它表述成“以后一直复用的固定 prompt”

## 批次分发文件必须包含什么
- 本轮批次名称或用途
- 线程要去哪个目录领取自己的专属文件
- 线程必须把结果回写到哪个固定回收卡
- 线程在聊天里只能回复哪些最小字段
- 本轮禁止事项

## 推荐最小回复格式
- 已回写文件路径
- 是否完成本轮要求
- 当前 own 路径是否 clean
- 一句话摘要

不要让线程在聊天里长篇复述回收卡内容。

## 最小回执与用户汇报的边界
- 上面的“推荐最小回复格式”只适用于：
  - 线程回治理线程
  - 固定回收卡
  - 典狱长模式下的最小聊天回卡
- 它不等于线程直接对用户汇报时的完整格式。
- 只要线程的回复对象是用户，或用户在问：
  - `现在做到哪了`
  - `你实际做了什么`
  - `还剩什么`
  - `下一步做什么`
  - `到哪个阶段了`
  线程就必须先给一层 `用户可读汇报`，至少回答：
  - `当前主线`
  - `这轮实际做成了什么`
  - `现在还没做成什么`
  - `当前阶段`
  - `下一步只做什么`
  - `需要用户现在做什么`
- `changed_paths`、checkpoint、参数、dirty/lock/owner 报实等技术信息，放在这层之后，不能拿来顶替上面的用户可读层。
- 如果线程只交技术 dump、不交用户可读层，治理线程应直接视为“回执信息不足”，不能当成对用户已说明白。

## 回执有效性闸门
- 只要本轮涉及 cleanup、white-list sync、owner 报实或 shared-root 尾账收口，线程回执必须显式回答：
  - `当前 own 路径是否 clean`
- 如果 `当前 own 路径是否 clean != yes`：
  - 这条回执只能算 blocker / owner 报实
  - 不能算“当前这一刀已闭环”
  - 不能直接进入下一轮 feature prompt
- 治理线程收件时，必须优先核这件事，而不是只看“有 commit SHA”或“git status overall 仍不 clean”。

## 给用户转发时的固定格式
- 如果这轮不是生成批次入口，而是要直接给用户几条可复制的转发话术，默认聊天格式固定为：
  1. `对应文件在：`
  2. 平铺列出最新 prompt 文件路径
  3. `你可以直接这样发给 XXX：`
  4. 一个 `text` 代码块
- `text` 代码块里必须直接包含 `请先完整读取 [最新 prompt 路径]`，不要把路径与转发语拆成两段，也不要回退到旧的“路径 + 极短短句”格式。

## 治理线程自己的动作顺序
1. 先确认当前命中的正式工作区与本轮阶段。
2. 如果本轮属于典狱长模式，先完成回执审查与四类裁定。
3. 如果要群发，先在 `Codex规则落地` 根层生成一份本轮批次分发文件。
4. 再让线程去各自阶段目录领取同名专属 prompt。
5. 收件时优先读取固定回收卡，不依赖聊天长回复。
6. 做完裁定后，再决定是否进入下一轮串行准入。

## 用户看得懂时的追加要求
- 如果用户最终会直接看线程汇报，治理线程在 prompt 里除“最小回复格式”外，还应额外写明 `用户可读汇报层` 的强制字段。
- 如果用户这轮已经明确抱怨“看不懂、说不清、我不知道下一步怎么发令”，后续 prompt 必须显式要求线程补交：
  - `当前主线`
  - `这轮实际做成了什么`
  - `现在还没做成什么`
  - `当前阶段`
  - `下一步只做什么`
  - `需要用户现在做什么`
- 不能继续只收最小回执。

## 配套技能口径
- 这条规范应由治理线程专用 skill 触发。
- 典狱长模式下，优先组合：
  - `sunset-warden-mode`
  - `sunset-governance-dispatch-protocol`
  - `sunset-prompt-slice-guard`
- 如果当前会话里该 skill 还不可见，治理线程也必须手工执行等价流程。

## 一句话口径
- 批次分发文件是治理线程的“本轮临时入口”，不是阶段资产，更不是永久模板。
- 典狱长模式下，先判“该不该继续发”，再做分发。

## 2026-03-26 追加补充：cleanup 批次使用边界
- 线程自己的 own dirty / untracked，默认应在当前这一刀内自行收口，不应依赖治理线程下一轮专门发 cleanup。
- 治理线程默认不再为“常规 own dirty 尾账”重复开 cleanup 批次。
- 只有命中以下情况，才升级成治理 cleanup / owner 报实批次：
  - cross-thread mixed dirty
  - hot-file / singleton config asset owner 不清
  - shared root 物理现场被 hot / mixed 目标卡死
- 当前 Sunset 的强制报实目标至少包括：
  - `Primary.unity`
  - `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`
  - `ProjectSettings/TagManager.asset`
  - `Assets/Editor/StaticObjectOrderAutoCalibrator.cs`
- 不要把“own 路径还没 clean”的回执误判成“cleanup 已完成”。
- 不要把治理 cleanup 批次当作普通开发的默认收尾手段。

## 2026-03-27 追加补充：`GameInputManager.cs` 与 `Primary.unity` 的分发口径
- 如果线程要碰 `Assets/YYY_Scripts/Controller/Input/GameInputManager.cs`：
  - prompt 或回执里必须额外写 `touched_touchpoints`
  - 治理线程不再因为它已 dirty 就机械停发所有其他线程
  - 只有当不同线程撞到同一入口 / 同一方法 / 同一行为链时，才升级打断或 integrator
- 如果线程要碰 `Primary.unity`：
  - 继续按单 scene writer 分发
  - `dirty + unlocked + ownerless` 不再口头误判成“有人占着写”
  - 当前 live 口径下，它表示“等待指定 writer 接盘的 stale mixed scene”
  - 当前批准的 scene writer 顺序为：`NPC -> spring-day1`

## 2026-03-19 追加补充：并发 queue 实盘的 tracked 写入禁令
- 如果本轮批次属于 shared root 上的并发 `request-branch / queue` 实盘：
  - waiting 线程不得在 `main` 上写任何 tracked 文件。
  - 包括但不限于：
    - 阶段收件卡
    - `.codex/threads/.../memory_0.md`
    - 其他 tracked 治理文档
- 原因：
  - 这类写入会把 `main` 写脏，直接污染 queue 准入现场。
- 因此该类批次默认改为：
  - 线程只做最小聊天回执
  - 固定收件箱仍保留
  - 但由治理线程根据聊天最小回执统一回填
- 只有真正拿到 `GRANTED / ALREADY_GRANTED` 并进入自己的任务分支后，线程才允许在自己的 branch carrier 范围内做受控写入。

## 2026-03-19 追加补充：执行层改题后的分发口径
- 从这一天开始，治理线程的批次文件只负责：
  - 发入口
  - 收最小聊天回执
  - 在事后做审计与记忆同步
- 治理线程不再把 tracked 回收卡当成运行时消息总线。
- waiting 线程在 shared root 上的恢复点，优先通过以下方式保留：
  - `CheckpointHint`
  - `QueueNote`
  - queue runtime json
  - 最小聊天回执
- `wake-next`、`return-main`、`request-branch` 的脚本输出已经会携带：
  - checkpoint hint
  - queue note
  - 推荐持槽窗口
  治理线程应优先读取这些输出与 runtime，而不是再要求线程回写一大段 tracked 说明。
