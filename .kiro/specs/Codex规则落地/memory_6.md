# Codex规则落地 memory_6

## 2026-03-19：整工作区方向重审结论
- 当前主线目标：重新回答“`Codex规则落地` 整个工作区的方向到底对不对”。
- 最终判断：
  - 前半段方向是对的，而且必要：
    - 它解决了 Sunset 最危险的失控问题。
  - 但继续沿这条工作区无限膨胀下去，方向会越来越不对：
    - 因为它开始把治理工作区变成运行时协调总线。
- 这条工作区当前最严重的结构性问题有三个：
  1. 根层入口已经不再可靠：
     - `memory.md` 仍停留在 `21` 语境，说明工作区根层单一活入口已经失真。
  2. 运行时协调过度依赖 repo tracked 文档：
     - 批次
     - 回收卡
     - memory 镜像
     - 审计补卷
     都在 shared root 语境中发生写入。
  3. 目标函数已经漂移：
     - 本来要恢复开发
     - 后来逐渐变成“继续优化治理流程自身”
- 因此这条工作区现在正确的定位应是：
  - 它是“安全治理与事故恢复工作区”
  - 不是“高吞吐多线程执行工作区”
- 现在真正该解决的核心问题已经不叫“再把 queue 做强一点”，而叫：
  - 如何把执行和调度解耦
  - 如何缩短持槽期
  - 如何把运行时状态从 tracked repo 写入中移出去
  - 如何让多个线程持续推进，而不是一个线程占槽、其他全部等待
- 推荐的下一步：
  - 新开后续阶段，主题不要再叫 queue / grant / return-main。
  - 应直接叫：
    - 执行模型重构
    - 或 shared root 调度大厅与多执行舱解耦
- 恢复点 / 下一步：
  1. 不再把 `Codex规则落地` 的扩写当成项目进度本身。
  2. 先承认：这一工作区已经完成“安全层”主体使命。
  3. 后续真正要提速，必须切到新的执行层架构问题。
