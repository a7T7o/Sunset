## 当前规则的真实口径

### 一、讨论 / 分析是不是实质性工作

是。

当前 Sunset 规则已经明确：

1. 任何产生可复用结论的实质性工作，都不能只写在线程记忆里；
2. 只读分析如果产生稳定治理结论、路由结论、风险判断，也要更新记忆；
3. 架构决策、问题分析、设计变更、技术债识别、锐评执行，都属于即使没有代码修改也必须触发 `memory` 记录的讨论类场景。

所以“这轮只是讨论”最多只能说明：

- 这轮不一定改了业务代码

不能推出：

- 这轮就不算实质性工作
- 这轮就不需要记忆
- 这轮就天然不需要收口

### 二、什么时候必须继续 Git 收口

当前规则也已经明确：

1. 实质性工作完成记忆更新后，只要达到可提交状态，就应继续 `git-safe-sync`
2. `main-only` 白名单 sync 不再因为 shared root 里还有 unrelated dirty 而一刀切阻断
3. 真正会阻断的是：
   - 本轮白名单所属 `own roots` 还有未纳入的 remaining dirty / untracked
   - 或者命中 hot / mixed blocker

因此，规范允许的正确说法只能是下面两类：

1. `这轮已经达到可提交状态，我按白名单 sync 收口`
2. `这轮还不能 sync，真实原因是 own roots 未 clean / 命中 hot-mixed blocker`

不允许再用下面这类模糊说法代替：

1. `因为只是讨论，所以先不收口`
2. `因为 shared root 里还有别人 dirty，所以我这轮不适合 sync`

## 当前 Sunset 现场下的正确回答框架

### 一、用户原题该怎么答

用户问的是：

> 多线程并行时，为什么会出现“我只做了 A，但回看像 A+B，所以线程自己不敢提交”的 `dirty` 归因混乱？  
> 正确解法是不是彻底解除 Git 规约？

正确答案应只包含 4 层：

1. `own diff / foreign diff / same-file contamination`
2. 普通路径并行允许
3. hot target 继续独占
4. integrator 收盘保留，但不是 hot 资源混写许可证

### 二、这轮额外必须补上的规则辨析

还必须再补一句：

> “讨论性更新”不等于“不需要收口”；真正该看的不是 shared root 里有没有别人脏，而是这条线程自己的 `own roots` 是否已经 clean、这轮是否达到可提交状态。

### 三、落到 `NPC` 与导航的并行场景

1. `NPC` 与导航可以并行，只要不同时 tracked 写同一个 hot target
2. 如果只是普通代码路径并行，就不该再因为 foreign dirty 自动停工
3. 如果碰的是 `Primary.unity`、Prefab、关键 shared `.asset` 或当前强制报实目标，就不能用“foreign dirty 无所谓”糊过去
4. 如果一条线说自己这轮不能 sync，必须回答：
   - 是 own roots 还没收干净
   - 还是被 hot / mixed blocker 卡住

## 必须撤回的错误判断

1. 撤回“导航唯一总主线”
2. 撤回“导航唯一主实现”
3. 撤回“`NPCV2` 支持线程化”
4. 撤回“`Primary.unity` / 字体 / owner / cleanup blanket 后置”
5. 撤回“直接新增一整套治理基础设施”
6. 撤回“讨论性更新天然不用收口”
7. 撤回“shared root 有别人 dirty 就足以说明这轮不适合 sync”

## 当前应保留的有效结论

1. 普通线程不该再对全仓 `overall dirty` 背责
2. 应区分 `own diff / foreign diff / same-file contamination`
3. 普通路径可以并行
4. hot target 继续独占
5. 最后需要 integrator 收盘
6. 但 integrator 收盘不能被误解成 hot serialized 资源可以先长期混写

## 一句话结论

- 真正该保留的是并发 dirty 的归因模型；真正必须打掉的是两种借题发挥：
  - 一种是把问题偷换成 current-wave 改题
  - 另一种是把“讨论性更新”和“Git 收口条件”混成一句模糊挡箭牌
