# 线程用户可读汇报失真分析与强制规约方案

- 日期：`2026-03-28`
- 状态：`当前有效`
- 作用：
  - 回答“为什么线程越来越会写给治理看、不会写给用户看”
  - 把“用户可读汇报”从软建议升级成 live 硬规则
  - 给后续所有线程和治理线程一套统一回复契约

---

## 1. 问题不是“线程没做事”，而是“线程没把做的事翻译成用户要的语言”

用户这轮对 `导航V2` 的不满，不是因为它完全没产出。

它真实做了这些事情：

1. 继续缩小导航主线热区
2. 清掉了两个会污染验证面的编译阻塞
3. 找到了当前最好的一组稳定参数口径
4. 把更激进但失败的试验回退掉，没有把脏实验留在现场
5. 让 Unity 回到 Edit Mode，没有留下新的 compile red

这些内容对工程是有价值的。

但用户仍然会觉得“你在说废话”，原因是它的汇报缺了用户真正想要的 5 件东西：

1. 你这轮实际做成了什么
2. 现在玩家看到的情况到底改善到哪
3. 现在还没做完什么
4. 当前到了哪个阶段
5. 下一步你只打算继续做什么

所以这不是“线程没干活”，而是：

> 线程在用工程视角报事实，但没有先把这些事实翻译成用户能拿来做决策的语言。

---

## 2. 当前 live 规范为什么没拦住这件事

现有规范已经很重视这些东西：

1. `changed_paths`
2. `own-path-clean`
3. `blocker_or_checkpoint`
4. `提交 SHA`
5. `是否触碰高危目标`

这些字段对治理、审计、白名单 sync 都很重要。

但问题在于：

- 它们更像“治理可审计字段”
- 不是“用户决策字段”

结果就是线程学会了：

1. 如何证明自己没乱碰
2. 如何证明自己没留编译红
3. 如何证明这轮有 checkpoint

却没有被强制要求先回答：

1. 现在用户能不能验
2. 这轮体验上到底前进了什么
3. 还差哪一刀
4. 下一条命令该怎么发

---

## 3. 真正的结构性原因

### 3.1 受众错位

线程默认把汇报对象当成：

- 父线程
- 审核者
- 另一个工程师

而不是：

- 最终下令的人
- 不看代码、只看结果和下一步的人

### 3.2 字段错位

现有最小回执字段主要解决：

- 可审计
- 可收口
- 可判断是否继续发 prompt

但它没有强制线程先交：

- 用户可读的阶段判断
- 用户可读的功能结果
- 用户可读的下一步

### 3.3 顺序错位

线程现在很容易按这个顺序汇报：

1. 我改了哪些文件
2. 我试了哪些参数
3. 我压到了哪个技术 checkpoint
4. 我回写了哪些 memory

最后才隐约提一句：

- “还剩什么”
- “下一步做什么”

对用户来说，这个顺序是反的。

### 3.4 “最小回执”被过度外推

最小回执原本是给：

- 治理线程
- 固定收件卡
- 批次回收

用的。

但实际运行里，它被错误地外推成了：

- 线程直接对用户汇报时也照样用最小回执思路

---

## 4. 现在必须明确区分的两种文本

### 4.1 给治理看的最小回执

它回答的是：

1. 你改了什么
2. 你有没有碰高危目标
3. 你 own 路径 clean 不 clean
4. 这刀能不能进入下一轮

### 4.2 给用户看的用户可读汇报

它必须先回答：

1. 当前主线一句话
2. 这轮实际做成的内容
3. 现在还没做成的内容
4. 当前阶段和进度判断
5. 下一步只做什么
6. 需要用户做什么，或者明确“不需要”

这两种文本都可以存在。

但从现在起：

> 不能再用“给治理看的最小回执”去顶替“给用户看的用户可读汇报”。

---

## 5. 新的强制口径：双层汇报契约

从现在起，所有 Sunset 线程直接对用户汇报时，必须至少交两层。

## A 层：用户可读汇报层

必须放在最前面，而且必须用大白话。

### A1. 保底六点卡

固定至少回答 6 项，而且必须逐项显式输出、顺序固定、不得合并：

1. `当前主线`
2. `这轮实际做成了什么`
3. `现在还没做成什么`
4. `当前阶段`
5. `下一步只做什么`
6. `需要用户现在做什么`

补充硬规则：

1. 这 6 项必须逐行出现，不能揉成一大段让用户自己拆。
2. 某项没有内容时，也必须明确写：
   - `无`
   - `尚未`
   - `不需要`
   - `仍待验证`
   这类直接答案，不能省略。
3. 这 6 项在任何情况下都必须先于技术信息出现。

### A2. 用户补充层

这层是可选的。

只有当线程认为下面这些内容会直接帮助用户理解和下令时，才允许追加：

1. 一句更白的话解释玩家体感
2. 一句“这轮为什么先做这刀”
3. 一句“如果现在要测，测什么最值”
4. 一句“当前别被哪些工程词吓到”

这层仍然是给用户看的，不是给治理看的。

## B 层：技术审计层

这层才允许写：

1. `changed_paths`
2. 参数名
3. 技术 checkpoint
4. compile / live / diff 细节
5. dirty / lock / owner 报实

顺序固定为：

1. 先 `A1 保底六点卡`
2. 如有必要，再 `A2 用户补充层`
3. 最后才 `B 技术审计层`

---

## 6. 新的术语翻译硬规则

如果线程用了下面这类术语：

- `best-known stable checkpoint`
- `detour`
- `release / recover`
- `blockOnsetEdgeDistance`
- `HardStop`
- `stuck cancel`
- `blocked-hotfile`

那么它必须在同一段里立刻翻译成：

- 玩家现在会看到什么
- 玩家现在还看不到什么
- 这说明阶段前进到了哪

不能只交技术词。

---

## 7. 新的验证状态硬规则

以后所有线程对用户汇报时，任何“已完成”都必须带验证状态标签。

固定只能用下面 4 类：

1. `用户已测`
2. `线程自测已过`
3. `静态推断成立`
4. `尚未验证`

---

## 8. 新的拒收机制

从现在起，出现下面任一情况，治理线程应直接视为“回执不合格”：

1. 只讲技术术语，不讲用户结果
2. 只讲改了哪些文件，不讲功能点
3. 只讲 best checkpoint，不讲当前阶段
4. 只讲 blocker，不讲下一步
5. 只讲“做了很多”，不讲“还剩什么”
6. 把最小回执当成给用户的完整汇报
7. `保底六点卡` 缺任一项
8. 先贴技术审计层，再回头补用户层
9. `需要用户现在做什么` 不写，导致用户不知道这轮自己该不该接手

---

## 9. 以后所有线程对用户汇报的固定模板

### 用户可读层：A1 保底六点卡

- 当前主线：
- 这轮实际做成了什么：
- 现在还没做成什么：
- 当前阶段：
- 下一步只做什么：
- 需要你现在做什么：

### 用户补充层：A2（可选）

- 如果还有一句必须先让用户知道的补充，再写在这里；没有就不写。

### 技术审计层：B

- changed_paths：
- 验证状态：
- 是否触碰高危目标：
- blocker_or_checkpoint：
- 当前 own 路径是否 clean：

如果线程当前只需要给治理线程回卡，而不是直接对用户说话，才允许只发技术审计层的最小版。

---

## 10. 把 `导航V2` 那段话翻译成用户真正需要的样子

用户真正需要看的版本，应该更像这样：

### 当前主线

- 继续修“玩家右键自动走路时，靠近 NPC 不要卡死或直接取消”这条导航主线。

### 这轮实际做成了什么

1. 我把两个会让验证结果失真的编译阻塞先清掉了，所以现在测到的结果更可信。
2. 我找到了一组目前最稳的参数口径，玩家现在已经能更稳定地贴近单个静止 NPC，不再像之前那样经常一靠近就直接取消。
3. 我试过更激进的参数，但那些方案都会把情况重新打回“硬停 / 卡死 / 直接取消”，所以我没有把那些坏实验留下来。

### 现在还没做成什么

1. 还没有真正做到“玩家能自然绕过静止 NPC 并顺利通过”
2. 当前还会太早触发 close-constraint 和 stuck cancel

### 当前阶段

- 现在不是编译问题，也不是 scene 问题。
- 当前已经进入“导航行为微调阶段”。

### 下一步只做什么

- 下一步继续只打 passive NPC 附近的 close-constraint 和 stuck cancel 衔接，不扩到别的导航议题。

### 需要用户现在做什么

- 暂时不需要你做决定；等我把这条衔接继续压一刀后，再给你新的可读汇报。

---

## 11. 最终结论

当前 Sunset 不是没有汇报规范，而是：

- 有治理规范
- 没有足够硬的用户可读汇报规范

所以这轮要做的不是“提醒大家以后说清楚点”，而是：

1. 把“用户可读汇报层”变成 live 硬规则
2. 把“技术审计层只能放后面”变成强制顺序
3. 把“只交技术 dump”变成可拒收项

一句话总结：

> 以后线程不是只要说真话就够了，还必须先说成用户能拿来直接做决策的话。
