## 当前现场总况

### 1. 仓库不是“小脏”，而是明显进入 mixed 事故面

当前 `D:\Unity\Unity_learning\Sunset` 的工作树是：

- `128 files changed`
- `36315 insertions`
- `100920 deletions`

这里面最大事故面不是普通代码 diff，而是场景与目录结构一起动了。

### 2. `Primary.unity` 已经不能再按“单线程普通 scene 脏改”理解

当前最危险的事实是：

- tracked 的 `Assets/000_Scenes/Primary.unity` 被删除
- untracked 的 `Assets/222_Prefabs/UI/Spring-day1/Primary.unity` 新出现
- 多条 navigation / UI / Day1 memory 都在引用这个新路径

这更像“同 GUID 主场景搬家或现场分裂”，而不是某一条线顺手改了个 UI prefab。

因此第一轮不允许任何线程直接把整张 `Primary.unity` 吞成自己的 ordinary own。

### 3. UI 工作区发生了真实迁移，不是单个文件增删

当前 UI 侧不是只多了几张图，而是旧目录被整批删除，新目录被整批建立：

- 旧 `UI系统/*` 多个目录是 `D`
- 新增 `UI系统/0.0.1 SpringUI/`
- 新增 `UI系统/Before03.28/`

这意味着 UI 父线程和 UI-V1 实施线之间，至少有一次真实的工作区迁移或改组。第一轮必须让它们各自举证，不允许后面靠口头回忆。

### 4. 当前 active set 里至少有 3 类容易互相顶账的面

- `Primary.unity` same-GUID / 主场景搬家事故面
- `GameInputManager.cs` mixed hot-file 面
- `DialogueChinese*` 共享字体稳定性面

只看聊天，很容易误把“我碰过附近”说成“这就是我的 own”；第一轮要专门把这类误报拆开。

## 第一轮为什么不直接拍最终责任

原因不是我不敢判，而是现在直接硬拍，错误成本太高：

1. 多条线程的聊天叙事已经与真实工作树脱节。
2. 同一个文件附近经常同时出现“我碰过”“我引用过”“我验证过”“我真正 own 了”四种不同层级。
3. 有些线程已经停表，但旧 blocker 叙事还挂在它头上。
4. 有些线程明明在做父线程审核，却被 working tree 表象误看成实现 owner。

所以第一轮的目标不是“谁先认得多谁就赢”，而是先逼每条线程把三份证据对齐：

- 自己聊天里怎么说的
- 自己 memory / 工作区文档怎么记的
- working tree 里真实留下了什么

## 第一轮回执必须回答什么

每条线程都必须明确回答：

1. 当前自己仍然明确 own 的路径清单

3. 当前只能报 mixed、还不能拍板的路径清单
4. 哪些旧 blocker / 旧叙事 / 旧 owner claim 现在必须撤回
5. 当前 own roots 是否 clean
6. 第二轮若要正式分配清扫，自己建议怎么领
