# 2026.03.19 queue-aware业务准入 01 事故复盘

## 结论先行
- 这轮失败不是业务线程集体违规，而是三类基建问题叠加：
  1. stable launcher 参数转发有真实 bug。
  2. queue runtime 的自愈逻辑会在 `grant -> ensure-branch -> task-active` 交接瞬间误杀票据。
  3. 持锁线程进入 task branch 后，shared root 上继续堆治理 dirty，导致它无法完成自己的 `task sync / return-main`。

## 已核实事实
- live shared root 当前不在 `main`，而是在 `codex/npc-roam-phase2-003 @ 7385d123`。
- occupancy 当前登记：
  - `owner_mode = task-branch-active`
  - `owner_thread = NPC`
  - `current_branch = codex/npc-roam-phase2-003`
  - `lease_state = task-active`
- queue runtime 当前却登记：
  - `NPC ticket=1 -> cancelled`
  - `农田 ticket=2 -> waiting`
  - `导航 ticket=3 -> waiting`
- 这说明 queue runtime 与 occupancy 已经分裂，当前 shared root 不具备继续发新准入的条件。

## 事故链重建
1. `NPC` 与 `遮挡检查` 先使用 stable launcher 发起 `request-branch`。
2. launcher 实际没有把 `-BranchName` 等可选参数稳定转发给 canonical script，于是两边都报 `request-branch 必须提供 -BranchName`。
3. `NPC` 随后改用仓库里的 canonical 脚本等价执行，成功拿到 `GRANTED` 并完成 `ensure-branch`，进入 `codex/npc-roam-phase2-003`。
4. `Ensure-TaskBranch` 会先把 occupancy 改成 `task-active`，再调用 queue 更新；但 queue 读取时会先跑 `Repair-SharedRootQueueRecord`，它看到当前条目还是 `granted`、现场又不再是 `main + granted`，于是直接把 `NPC ticket=1` 改成 `cancelled`。
5. `导航检查` 随后用 canonical `request-branch` 成功进入 `LOCKED_PLEASE_YIELD`，被排到 `ticket=3 / queue_position=2`，这一步本身是对的。
6. `NPC` 在 task branch 上只做了最小 continuation checkpoint，并没有进入 Unity / MCP / Play Mode；但此时 shared root 上又被写入了治理回收卡与 memory dirty，这些文件不在 `NPC` 的 task sync 白名单里。
7. 结果就是：`NPC` 合法占槽、合法进入分支，却无法 `task sync`，也就无法 `return-main`；shared root 长时间卡在 `NPC active + foreign dirty`。

## 根因拆解

### 1. stable launcher 参数转发 bug
- 文件：`C:\Users\aTo\.codex\tools\sunset-git-safe-sync.ps1`
- 现象：`New-ForwardArgumentList` 在函数内部使用自己的 `$PSBoundParameters`，导致 `-BranchName`、`-CheckpointHint`、`-QueueNote` 这些脚本级参数没有稳定传入 canonical script。
- 结果：`request-branch` / `grant-branch` 这类依赖 `-BranchName` 的动作会在 launcher 层失真。

### 2. queue repair 抢先取消 granted 条目
- 只读核对对象：`main:scripts/git-safe-sync.ps1`
- 现象：
  - `Get-SharedRootQueueRecord` 每次读 queue 都会先调用 `Repair-SharedRootQueueRecord`。
  - `Repair-SharedRootQueueRecord` 会把“不是 `main + granted` 的 granted 条目”直接降成 `cancelled`。
  - `Ensure-TaskBranch` 又是在 `Set-SharedRootTaskActive` 里先改 occupancy，再去做 `Set-SharedRootQueueTaskActive`。
- 结果：`NPC` 在 `grant -> task-active` 的合法过渡期，被 repair 逻辑先一步改成 `cancelled`，因此 queue runtime 与 occupancy 脱钩。

### 3. 持锁线程收口模型未隔离治理 dirty
- 证据：`NPC.md` 明确写到 `task sync` 不是被 NPC 自己的白名单挡住，而是被 shared root 上新增的治理 dirty 挡住。
- 结果：持锁线程即使只是最小 checkpoint，也可能因为别的线程或治理线程写入回收卡 / memory，而被卡死在 `task-active`，无法归还 shared root。

### 4. live branch 脚本版本漂移放大了混乱感
- 当前 live branch 上的 `D:\Unity\Unity_learning\Sunset\scripts\git-safe-sync.ps1` 仍是旧版，只包含 `preflight / ensure-branch / grant-branch / return-main / sync`。
- 而 stable launcher 默认转发的是 `main` 上的 queue-aware canonical script。
- 结果：同一轮里出现了“launcher 用的是新逻辑、工作树文件看见的是旧逻辑、回执里又写 canonical 直接执行”的三层错位，极易让人误判现场。

## 当前裁定
- 这一轮是一次非常有价值的失败样本。
- 它证明：
  - queue-aware 方向是对的，`导航检查` 的 waiting 结果就是有效样本。
  - 但当前版本还不能进入“全线程自由排队开发”。
- 当前 shared root 仍是 `NPC` 占用且带多线 dirty 的现场，不能继续发新一轮准入。

## 恢复前提
1. 先修 stable launcher 的参数透传。
2. 再修 canonical queue 的 `granted -> task-active` 交接，避免被 repair 误杀。
3. 再补“持锁线程收口期间的治理写入隔离”或 rescue runbook，确保 slot holder 能完成 `task sync / return-main`。
4. 回正 occupancy / queue runtime 后，才能重发 batch。
