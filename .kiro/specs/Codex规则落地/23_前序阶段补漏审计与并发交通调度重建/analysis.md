## 6. 对 `gemini.md` 的审核结论

### 路径判断
- 结论：**Path B**
- 原因：
  - Gemini 对“物理单写不等于逻辑单线程”的问题定性是对的
  - 但它给出的“立刻全员并发唤醒、去抢租约、抢不到就自动挂起等待”的执行建议，当前对 Sunset 来说仍然过早

### 已验证为正确的部分
- 当前真正缺失的不是入口闸机，而是：
  - 排队
  - 等待态
  - 中断不丢进度
  - 多 checkpoint 持续推进
- 当前治理线程确实不能继续停留在“人工三选一发牌”的口径上
- `23` 应该承接“并发交通调度器”而不是继续重复“单写者串行就是终局”

### 必须纠正的部分
- 不能在调度层尚未落地前，就直接群发“你们现在全部开始并行抢租约”的业务 prompt
- 当前仓库里还没有真正的：
  - 队列状态载体
  - 等待态回执格式
  - 失败后自动转挂起的脚本逻辑
  - 锁释放后的标准唤醒路径
- `git-safe-sync.ps1` 当前仍然以 `FATAL` 为主，不存在真正的 `yield / wait / queue` 自动调度能力

### 因此采纳的方式
- 采纳 Gemini 的**方向性纠偏**：
  - single-writer 不是终局
  - 应向“并发调度器”升级
- 不采纳 Gemini 的**立即执行建议**：
  - 不在本轮直接切到“全员并发唤醒 + 抢租约”
  - 先由 `23` 把调度模型、等待态、回执和验证路线写成正式正文

## 7. queue runtime 兼容性 bug 与当前裁定
- 这轮在 `git-safe-sync.ps1` 新增 queue-aware 入口后，首次真实运行暴露出 PowerShell 兼容性问题：
  - `ConvertFrom-Json -Depth 10`
  - 当前 live PowerShell 不支持该参数，导致 `request-branch` 在读取 queue runtime 时直接崩溃
- 本轮已完成的修补：
  - 去掉 `ConvertFrom-Json -Depth`
  - 为 queue runtime 增加空文件回退
  - 为坏 JSON 增加显式 `FATAL`
- 本轮最小验证结果：
  - 在当前治理 dirty 现场执行 `request-branch`
  - 已稳定返回 `LOCKED_PLEASE_YIELD`
  - 说明“抢不到槽位时活着挂起”这条新路径已经真正可运行
- 当前仍未完成的层：
  - 自动唤醒
  - 明确取消排队
  - 完整的等待态回执格式
  - 真正的队首消费 / 越序审批 / 负例矩阵
- 因此当前口径应收紧为：
  - queue 入口已从纸面进入脚本可运行态
  - 但 Sunset 还没有到可以全员并发实盘抢租约的阶段
