# 阶段 23 memory_3

## 2026-03-19：跳出阶段 23 后的整工作区重判
- 当前主线目标：不再只从 `23` 的 queue-aware 修复视角看问题，而是跳出整个 `Codex规则落地` 工作区，重新判断这条治理主线整体方向是否仍然正确。
- 本轮重判结论：
  - `Codex规则落地` 到目前为止做对了前半段：
    - 把 Sunset 从“线程乱闯、错分支写入、shared root 脏循环、事故不可回正”里拉了出来。
    - 它已经建立起：
      - AGENTS / skills 闸门
      - stable launcher
      - `git-safe-sync.ps1`
      - shared root occupancy
      - queue-aware admission
      - 批次分发与治理审计
  - 但它也在后半段逐渐走偏：
    - 它开始把“治理工作区”同时当成：
      - 宪法层
      - 调度塔
      - 事故法医室
      - 运行时消息总线
      - 收件箱
      - 线程记忆镜像层
    - 这导致 repo 内 tracked Markdown 被过度用作运行时协调介质。
- 当前真正的方向性问题不是 queue 细节，而是架构分层错误：
  1. 把 shared root 同时当成：
     - 代码入口
     - 运行时调度大厅
     - receipt / memory 写入面
  2. 把 `Codex规则落地` 同时当成：
     - 稳态治理基线
     - 高频运行时 orchestration 面
  3. 结果就是：
     - 安全性上来了
     - 吞吐下来了
     - 工作区正文越来越多
     - 但真正的持续并发执行模型没有长出来
- `遮挡检查` 这次 11 分钟持槽，是一个关键证据：
  - 不是“queue 坏了”
  - 而是 shared root 单槽里混入了太多不该在持槽期做的动作：
    - 只读核对
    - 回执
    - memory
    - 治理写回
- 因此当前对整个工作区的最终定性应改为：
  - `Codex规则落地` 目前已经基本完成“安全恢复层”
  - 但不适合继续膨胀成“高吞吐并发执行层”
- 下一步方向建议：
  - 把 `Codex规则落地` 收紧成：
    - 稳态治理基线
    - 事故 runbook
    - 少量批次级调度入口
  - 不再让它承载过多运行时写入和高频 orchestration 明细。
  - 后续真正该新开的阶段，应该解决的是：
    - 执行模型重构
    - shared root 调度大厅与长耗时执行解耦
    - 运行时状态从 tracked Markdown 中抽离
- 恢复点 / 下一步：
  1. 新阶段不应再只是 `23` 的子延长，而应明确改题为“执行模型与吞吐重构”。
  2. 继续扩写 queue / 收件卡 / 批次文档，只会让治理层更重，不会让开发更快。
