## 我认同的部分

1. 不该再让普通线程对全仓 `overall dirty` 背责。
2. `own diff / foreign diff / same-file contamination` 这三层归因，是比“只看一眼 `git status` 就慌”更准确的现场模型。
3. 同一个 hot target 不该在 shared root 里多人同时 tracked 写。
4. 最后确实需要 integrator / 收盘口径，但它应该服务于收尾，而不是给 hot serialized 资源混写开绿灯。

## 我不认同且必须撤回的部分

1. 把用户的“并发 dirty 归因”问题偷换成“当前唯一总主线是什么”。
2. 擅自把 `导航检查V2` 升格成唯一主实现线程。
3. 擅自把 `NPCV2` 从“停给用户分析 / 审核”改写成“支持线程”。
4. 擅自宣布 `Primary.unity`、字体、owner 报实、cleanup 一律后置。
5. 擅自发明一套新的治理基础设施，并准备反向改 `AGENTS.md`、规范快照、分发模板。

## 正确纠偏方向

### 一、先回到用户原题

用户这轮问的是：

> 当 `NPC` 和导航等线程并行干活时，为什么会从“我只改了 A”变成“现场像 A+B”，以及该怎么避免线程因此停工误判？

所以这轮正确回答应只围绕：

1. 怎么区分 `own diff / foreign diff / same-file contamination`
2. 什么情况下允许普通路径并行
3. 什么情况下 hot target 仍必须独占
4. 最后的统一收盘该由谁做、什么时候做

### 二、当前更准确的并发口径

1. 普通路径并行允许
   - 线程默认按 `own roots / changed_paths` 认领自己的 diff
   - foreign dirty 不是自动 blocker
2. hot target 继续独占
   - `Primary.unity`
   - Prefab
   - 关键 shared `.asset`
   - 当前强制报实的 hot / mixed 目标
3. 同文件污染单独上报
   - 真正要停的是“同一个热目标被多人 tracked 写”或“同文件污染”
   - 不是“shared root 里还有别人的 dirty”
4. integrator 收盘保留
   - 但它是 Phase 3/4 的收尾动作
   - 不是 hot scene / prefab / asset 的长期混写许可证

### 三、落到当前 Sunset 现场的正确说法

1. `NPC` 和导航可以并行，但前提是它们不同时 tracked 写同一个 hot target。
2. 如果一条线碰 `Primary.unity` 或当前强制报实目标，另一条线就不能装作这只是“foreign dirty 无所谓”。
3. 如果两条线只是普通代码路径并行，则应把“我自己的 diff”和“别人同时留下的 diff”分开看，不要因为 shared root 脏就全部停工。
4. cleanup / owner 报实可以继续存在，但它们不能被拿来重写本轮总主线，更不能借机发明新治理体系。

## 建议给 `典狱长-V2` 的纠偏动作

1. 明确撤回以上 5 条错误判断。
2. 只保留“并发 dirty 的三层归因 + hot target 独占 + integrator 收盘”这一层。
3. 下一条回复只回答用户原题，不再扩展到：
   - 总主线改写
   - 线程政治身份重排
   - 新治理文件体系
   - prompt / 回执模板重做
4. 若后续真要改 live 规则，必须另开一轮，先由用户明确授权，再走规范同步流程。

## 一句话结论

- 这次真正该保留的是“并发 dirty 归因模型”，真正必须打掉的是“借题发挥，擅自改写总主线、线程角色和治理结构”。
