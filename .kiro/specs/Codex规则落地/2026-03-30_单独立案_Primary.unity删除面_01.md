# 2026-03-30｜单独立案：`Primary.unity` 删除面

## 一句话结论

本案当前裁定是：`恢复旧 canonical path`。  
不是 `重建`，也不是 `确认删除`。

## 这轮钉死的硬证据

1. 当前 working tree 的直接删除面是：
   - `Assets/000_Scenes/Primary.unity`
   - `Assets/000_Scenes/Primary.unity.meta`
2. `ProjectSettings/EditorBuildSettings.asset` 仍把 build scene 指向：
   - `Assets/000_Scenes/Primary.unity`
   - `guid = a84e2b409be801a498002965a6093c05`
3. `Assets/Editor/NPCAutoRoamControllerEditor.cs` 里仍硬编码：
   - `private const string PrimaryScenePath = "Assets/000_Scenes/Primary.unity";`
4. `HEAD` 里当前同时跟踪了两份 `Primary.unity`：
   - `Assets/000_Scenes/Primary.unity`
   - `Assets/222_Prefabs/UI/Spring-day1/Primary.unity`
5. 这两份 scene 的 `.meta` 当前是同一个 GUID：
   - `a84e2b409be801a498002965a6093c05`
6. 当前这次删除，只删除了旧路径那一份，并没有解决 same-GUID 双场景问题；它只是把事故显性化了。

## 为什么不是“确认删除”

1. 旧路径仍是当前构建设置和编辑器代码默认认的 canonical path。
2. 新路径挂在 `Assets/222_Prefabs/UI/Spring-day1/` 下，语义上更像测试/迁移现场，不像主场景的长期落点。
3. 在 same-GUID 双场景仍未正式裁定前，直接确认删除旧路径，只会把一个未闭环迁移强行固化成事实。
4. 这会把后续问题从“恢复 canonical path”升级成“全仓改路径、改引用、改 build scene、改工具硬编码”的更大事故。

## 为什么也不是“重建”

1. 真实 scene 内容并没有消失到需要从零重建的程度。
2. 当前问题不是“场景坏到只能重做”，而是“canonical path 和 same-GUID duplicate 现场没收干净”。
3. 所以这案子该走的是“恢复 canonical path + 再清 duplicate”，不是“从零新建第三份 Primary”。

## 当前执行裁定

1. `Assets/000_Scenes/Primary.unity` 旧路径应恢复。
2. `Assets/222_Prefabs/UI/Spring-day1/Primary.unity` 当前不应直接视为新的 canonical `Primary`。
3. 新路径这份 scene 应视为：
   - same-GUID duplicate scene 事故面
   - 需要后续单独决定是迁成新 GUID 的专项场景，还是在差异回收后删除
4. 本案在没有单 scene writer 接盘前，不进入 live scene 改写。

## 这轮故意不做什么

1. 不直接改 `Primary.unity` 内容。
2. 不直接改 `EditorBuildSettings.asset`。
3. 不直接改 `NPCAutoRoamControllerEditor.cs`。
4. 不让任何业务线程顺手吞这案子。

## 下一步唯一正确切法

开一条独立的 `Primary.unity` 单 writer 治理/修复切片，只做四件事：

1. 从 `HEAD` 恢复 `Assets/000_Scenes/Primary.unity` 与 `.meta`
2. 把旧路径重新立回 canonical path
3. 对 `Assets/222_Prefabs/UI/Spring-day1/Primary.unity` 做 same-GUID duplicate 裁定
4. 在 canonical path 被立稳后，再决定是否需要做路径迁移或 duplicate 清理
