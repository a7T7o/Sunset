# Codex规则落地 memory_5

## 2026-03-19：当前治理主线的真实定性
- 当前主线目标：回答“我们现在到底在实现什么”，避免继续把安全恢复层误当成并发执行层。
- 已完成内容的真实定性：
  - 我们已经做成的是：
    - `Git shared-root 安全恢复 + 准入闸机 + queue-aware 调度入口`
  - 它解决的是：
    - 错分支写入
    - shared root dirty 循环
    - 无租约切分支
    - 无法回正与无法审计
- 还没有做成的内容：
  - 持续高吞吐的多线程开发执行模型
  - 多线程长期并行推进真实工作，而不是只有一个线程持槽、其他线程等待
- 当前新的核心问题：
  - shared root 单槽的生命周期设计错误
  - 分支占用开始得太早、释放得太晚
  - 持槽期里混入了：
    - 只读核对
    - memory / 回执 / 治理写回
    - 非必要的总结动作
  - 结果就是：
    - 安全性提升了
    - 吞吐和并发推进能力没有提升
- 我的方向判断：
  - 接下来不应继续只优化 queue 细节
  - 而应转向“执行模型重构”
  - 这意味着要正式回答一个架构问题：
    - 我们到底接受“单 shared root + 串行短事务”
    - 还是引入“调度大厅 + 多执行舱”的受控并发模型
- 推荐方向：
  - 目标若是“多个线程持续完成任务”，就应走后者。
  - shared root 保留为统一调度与回正大厅。
  - 真正耗时的业务执行，不再挤在 shared root 单槽里完成。
- 恢复点 / 下一步：
  1. 以这条判断为依据，新立后续阶段。
  2. 该阶段不再讨论“怎么再发一轮 queue prompt”，而是讨论“怎么把执行与调度解耦”。
