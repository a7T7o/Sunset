```text

- 只有你亲自复核为 `main + occupancy neutral`，阶段一口径才成立
- 如果 live 现场不是中性，不要硬写“通过阶段一”，要先如实回写现场

你这一轮默认身份仍是只读审计线程，不是直接进入写入线程。
你的第一条 commentary 必须显式说明你正在使用 skills-governor、sunset-workspace-router，并先完成只读 preflight。

这次不要只在聊天里回复。请把完整结果写入：
D:\Unity\Unity_learning\Sunset\.kiro\specs\Codex规则落地\22_恢复开发分发与回收\线程回收\导航检查.md
只填写：
- `## 阶段一｜线程回收区（由 导航检查 填写）`

请只读完成以下动作：
1. 在 shared root 复核 cwd / branch / HEAD / git status --short --branch。
2. 读取你的线程 memory、相关导航工作区文档、当前治理入口文档。
3. 在 main 上只做只读审计：
   - 读取导航相关现状
   - 分层问题
   - 判断哪些结论只读即可成立
   - 判断若要整改，是否需要单独开写入分支

本轮禁止：
- 禁止写文件
- 禁止 ensure-branch
- 禁止进入 Play Mode
- 禁止直接把 shared root 切成自己的长期现场

- 是否建议进入阶段二
- 一句话摘要
```

## 阶段二｜条件式准入
```text
【阶段二｜条件式准入｜导航检查】

仅当你在阶段一已明确判定“必须进入真实写入整改”，且治理线程明确批准后，才执行本轮。

当前前提仍必须满足：
- D:\Unity\Unity_learning\Sunset
- main
- clean
- occupancy = neutral

发送本 Prompt 不等于替你跳过 live preflight；你在执行 `grant-branch` 前仍必须先自查一次上述条件。

默认建议的新分支名：
- codex/navigation-audit-001

请把完整结果写入：
D:\Unity\Unity_learning\Sunset\.kiro\specs\Codex规则落地\22_恢复开发分发与回收\线程回收\导航检查.md
只填写：
- `## 阶段二｜线程回收区（由 导航检查 填写）`

请严格执行：
1. 先执行 grant：
   git-safe-sync.ps1 -Action grant-branch -OwnerThread "导航检查" -BranchName "codex/navigation-audit-001"
2. 再执行准入：
   git-safe-sync.ps1 -Action ensure-branch -OwnerThread "导航检查" -BranchName "codex/navigation-audit-001"
3. 进入分支后只做阶段一已说明过的最小整改 checkpoint。
4. 如果要碰 Unity / MCP / Play Mode 或共享热区，做完后必须退回 Edit Mode。
5. checkpoint 完成后 return-main 归还 shared root。

本轮禁止：
- 禁止无阶段一结论直接进入写入
- 禁止扩大整改范围
- 禁止在 main 上直接写导航整改

写完回收文件后，在聊天里只回复：
- 已回写文件路径
- grant / ensure-branch 是否成功
- 一句话摘要
```
