# 2026-03-20 事故反思与主线重对齐

## 1. 这次问题到底为什么会发生
- 表面现象是：
  - `NPC` batch03 后场景一度恢复
  - `farm` 收口回到 `main` 后，`Primary.unity` 里的 `NPC 001/002/003` 又变成红色缺失
- 真正原因不是 `farm` 覆盖了 NPC。
- 真正原因是：
  - `main` 的 [Primary.unity](D:\Unity\Unity_learning\Sunset\Assets\000_Scenes\Primary.unity) 已经引用了 `codex/npc-roam-phase2-003` 才有的 prefab、anim、profile、meta 和 runtime 脚本
  - 但这些资产从未真正进入 `main`
  - 所以用户之前看到 NPC 正常，其实只是 shared root 暂时停在 NPC 分支；一旦 `return-main`，断链就必然复现

## 2. 这暴露了什么治理缺口
- 我们之前把很多注意力放在：
  - 是否拿到 grant
  - 是否成功 `ensure-branch`
  - 是否成功 `sync`
  - 是否成功 `return-main`
- 这些都重要，但它们只证明了“交通系统正常”，没有证明“生产场景真的恢复了”。
- 这次事故说明，真实业务批次还缺一个关键验收维度：
  - `main-ready`

## 3. 我从中得到的核心发现
1. `carrier-ready` 不等于 `main-ready`
2. “branch 里有成果”不等于“用户此刻在 main 里就能正常开发”
3. 治理线程不能只做交通警察，还必须做最终基线裁判
4. 对生产场景有直接依赖的任务，必须验证 `main` 的真实可运行性，而不是只看 branch 收口是否漂亮

## 4. 当前主线现在是什么
- 当前治理线程的主线不是继续做抽象交通实验。
- 当前主线是两件事并行推进：
  - 维护 shared root 的高吞吐、安全执行模型
  - 让这个模型真正服务真实业务开发，而不是制造“看似闭环、实则 main 不可用”的假恢复

## 5. 这轮之后主线应如何继续
1. 把 `main-ready` 正式并入后续真实业务批次模板与回收卡
2. 继续恢复 `NPC / 农田` 真实业务开发，而不是重复做只读探测
3. 把 `dirty 容忍 / 清扫推送机制` 作为独立治理议题讨论，不混进这次事故结论里偷渡上线
4. 把重度 MCP 场景搭建线程单独归类，避免误用普通共享根短事务模型

## 6. 一句话结论
- 这次事故不是 shared root 交通系统失效，而是我此前的“业务恢复验收口径”还不够硬；从现在开始，治理线程必须同时对“交通正确”和“main 真正可用”负责。
