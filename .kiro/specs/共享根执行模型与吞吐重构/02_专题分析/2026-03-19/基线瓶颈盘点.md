# 基线瓶颈盘点（2026-03-19）

## 1. 盘点目的
- 把“为什么现在要从治理层切题到执行层”说清楚。
- 这份盘点不复述旧事故全史，只抓当前真正拖慢开发吞吐的物理瓶颈。

## 2. 已解决的问题
- shared root 错分支写入
- 无租约就切分支
- launcher 参数丢失
- queue 状态机误杀
- 现场回不正与无人可审计

这些问题依然重要，但它们已经不再是当前第一主矛盾。

## 3. 当前三大瓶颈

### 3.1 持槽期过长
- 证据：`遮挡检查` 曾在 shared root 内持续占槽约 11 分钟。
- 问题本质：
  - 不是线程不会排队。
  - 而是“长时间只读分析 + 回执 + memory + 治理补写”被塞进了持槽期。
- 直接后果：
  - 其他线程全都只能等待。
  - queue 即使存在，也只是把等待排队显性化，没有提升吞吐。

### 3.2 tracked 文档被过度用于运行时协调
- 典型表现：
  - waiting 线程试图在 `main` 上写回回执或 memory。
  - 治理线程为了收件与裁定，在运行中不断往 tracked 文档里写内容。
- 直接后果：
  - `main` 容易再次被治理动作写脏。
  - request / wake / grant 的现场被文档层反向污染。

### 3.3 治理线程角色过重
- 当前风险：
  - 治理线程不仅维护规则，还承担高频运行时编排。
  - 这会把“安全层”拖成“人工调度中心”。
- 直接后果：
  - 一切并发最终都卡在人。
  - 工作区正文越来越厚，但交付速度没有上升。

## 4. 新主线的判断
- 继续补 queue 细节，并不能解决上述三大瓶颈。
- 真正要改的是：
  - shared root 只做最小写事务
  - waiting 状态尽量进入 ignored runtime
  - 线程在未拿到槽位前继续并发做只读准备
  - 事后证据与运行态消息分离

## 5. 一句话结论
- 现在的核心问题已经不是“怎么把门锁做得更厚”，而是“怎么让大家在只有一扇门的前提下更快、更稳地轮转进出”。
