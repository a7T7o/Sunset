# 2026-03-20 dirty 容忍度与清扫推送机制讨论稿

## 1. 为什么会提出这个议题
- 用户的真实不满是合理的：
  - 如果每次只要出现 dirty 就必须整轮停工、回退、等待治理窗口，开发吞吐会很差
- 这说明当前模型虽然安全，但在某些场景下可能过于保守。
- 但“保守”不等于可以直接摆烂成“只要能跑就都带脏推进”。

## 2. 这个议题真正要解决的是什么
- 不是让 shared root 永远带脏开发。
- 而是回答：
  - 哪些 dirty 对项目几乎无害、且可被下一位稳定接手
  - 哪些 dirty 一旦放行就会制造隐形炸弹
  - 在什么前提下，dirty 可以被视为“已清扫、可推送、可接手”

## 3. 当前我的客观判断
- 现在还不能直接批准“完全不清理也不认领，只要能跑就继续”。
- 原因很简单：
  - 我们还没有 dirty 分级
  - 还没有 takeover 合法条件
  - 还没有定义谁来为脏改负责
  - 还没有定义哪些类型一律禁止放行
- 在这些前提缺失时，放开 dirty 会直接把当前刚稳定下来的 shared root 模型重新拉回混乱状态。

## 4. 我认为可以讨论的分级方向

### 4.1 可优先考虑容忍的类型
- gitignored runtime
- Draft 沙盒
- 明确 owner 的治理文档草稿
- 不影响运行语义、且已具备交接说明的轻量 tracked 记忆补写

### 4.2 需要强约束才可能容忍的类型
- 纯文档型 tracked 改动
- 已形成明确白名单、且下一位可以直接承接的业务代码脏改
- 已有最小 checkpoint 说明、且回滚成本可控的 branch 内收口脏改

### 4.3 当前仍应绝对禁止放行的类型
- `Primary.unity`
- `GameInputManager.cs`
- 共享 Prefab / ScriptableObject / Sprite meta 的未说明脏改
- ownership 不清的 mixed dirty
- 需要 Unity / MCP 连续验证但尚未完成验证的 scene / inspector 脏改

## 5. 如果未来真要落地，至少需要什么前置
1. dirty 分级表
2. takeover 条件表
3. owner 与责任归属字段
4. 白名单同步与最小验收模板
5. 失败时的回退/搁置(runbook)

## 6. 当前裁定
- 这条方向值得继续做，但今天还只是讨论稿，不是已批准机制。
- 在正式设计、脚本与回收模板都落地前：
  - `main clean`
  - `shared root neutral`
  仍然是默认硬门槛。
