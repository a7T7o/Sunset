# 2026-03-20 重度 MCP 场景搭建线程执行方案

## 1. 这类线程是什么类型
- 这不是普通代码线程。
- 它更像：
  - 长寿命 Unity 编辑器工作流
  - 高频场景读写
  - 高频截图回读与视觉修正
  - 强依赖连续上下文的单线程创作任务

## 2. 在 2D 项目里，AI 能做到什么，不能做到什么

### 2.1 能做到的
- 扫描 prefab / sprite / material / ScriptableObject 资产
- 基于命名、引用链、尺寸感和序列化字段建立可检索索引
- 在参考图与现有资源命名关系较清晰时，搭出高质量初稿
- 通过截图回读做二轮、三轮修正

### 2.2 不能神化的
- 它不能像人类美术一样凭空理解所有审美意图
- 它不能从单张复杂透视图 100% 还原全部遮挡结构
- 如果 prefab 命名混乱、资源没有对应关系、参考图信息不足，效果会明显下降

## 3. 为什么这类任务不适合拆线程
- prefab 语义、尺度、pivot、构图主次这些信息一旦在线程间切断，就很容易失真
- Unity / MCP 本身也是单实例高热区
- 所以这类任务更适合：
  - 一个线程独占完成整条链
  - 而不是多个线程分拆“索引 / 摆放 / 修图 / 回读”

## 4. worktree 在这里的真实作用
- `worktree` 能解决的是：
  - Git WIP 隔离
  - 长时间未收口场景改动的独立承载
- `worktree` 解决不了的是：
  - Unity / MCP 并行冲突
  - 单实例 Editor 的热区争用
- 所以是否使用 `worktree` 的判断标准不是“想不想并行”，而是：
  - 这条线是否会形成长寿命场景 WIP
  - 是否需要避免 shared root 被长期占成脏工作台

## 5. 推荐执行模型
1. 先做 prefab 普查与资产索引
2. 先做一次 MCP 写链路沙盒验证
3. 优先在独立验证场景或临时搭建场景中工作，而不是一上来直写 `Primary.unity`
4. 在单线程内完成：
   - 参考图拆解
   - 候选 prefab 匹配
   - 初稿摆场
   - 截图回读
   - 二轮修正
5. 只有在初稿稳定后，才考虑迁移到生产场景

## 6. 当前治理裁定
- 这类任务应该被视为“特殊执行类型”。
- 推荐口径是：
  - 单线程独占
  - Unity / MCP 单写者
  - Git 隔离按需启用 `worktree`
  - 真碰 `Primary.unity` 时另开独立波次

## 7. 一句话结论
- 对 2D 场景搭建而言，AI 不是全能美术，但在资源命名清晰、参考图明确、线程独占连续工作的条件下，完全可以承担“Prefab 普查 -> 索引 -> 初稿搭建 -> 截图回读 -> 二轮修正”的整条链。
