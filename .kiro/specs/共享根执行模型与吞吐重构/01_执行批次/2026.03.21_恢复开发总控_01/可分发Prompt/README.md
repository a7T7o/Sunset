# 可分发Prompt 使用说明

## 这批文件到底是什么
- 这批文件默认不是“先测试一下有没有准备好”的空转 prompt。
- 这批文件默认是“当前就可以直接发出去执行”的本轮 prompt。
- 也就是说，你把文件发给对应线程后，对方不是只回答“我准备好了没有”，而是应该直接：
  - 按 prompt 里的准入动作进入
  - 在允许边界内做本轮真实 checkpoint
  - 做完后按最小回执格式回给治理线程

## 为什么 prompt 里同时有准入和任务
- 因为 shared root 线程不是先聊半天再开发，而是“拿到放行 prompt 后，直接在同一轮里完成准入 + 最小真实工作 + 收口”。
- 这样做是为了减少项目经理手工发两轮 prompt 的负担。
- 所以这里的 prompt 是“执行 prompt”，不是“问询 prompt”。

## 哪些是直接开发 prompt
- `NPC_当前开发放行.md`
- `农田交互修复V2_当前开发放行.md`
- `导航检查_当前开发放行.md`
- `遮挡检查_当前开发放行.md`
- `scene-build_结构层继续施工.md`

这些文件发出去以后，对方应直接进入本轮真实工作，而不是只回复“已准备好”。

## 哪个不是直接开发 prompt
- `spring-day1_集成波次前只读准备.md`

这份是例外。
它当前就是只读准备 prompt，不允许进入真实开发。

## 项目经理最简单判断法
1. 文件名里如果是“当前开发放行”或“继续施工”，默认就是现在直接执行。
2. 文件内容里如果要求：
   - `request-branch`
   - `ensure-branch`
   - 做一个真实 checkpoint
   - `sync`
   - `return-main`
   那它就是执行 prompt，不是测试 prompt。
3. 文件内容里如果明确写着“只读准备 / 不允许 request-branch / 不允许 ensure-branch / 不允许真实实现”，那才是准备 prompt。

## 当前这轮的口径
- `scene-build`：直接施工，但在自己的 worktree 里做。
- `NPC / 农田交互修复V2 / 导航检查 / 遮挡检查`：都是 shared root 真实执行 prompt，按队列一条一条发。
- `spring-day1`：当前只允许只读准备，不进真实开发。

## 一句话口径
- 除了 `spring-day1` 这一份例外，当前目录下这些 prompt 你都可以理解成：“发过去就开干，不是先试探。”
