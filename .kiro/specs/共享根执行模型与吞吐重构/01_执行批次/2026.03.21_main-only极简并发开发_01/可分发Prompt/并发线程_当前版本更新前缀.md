# 并发线程 当前版本更新前缀

```text
当前制度已更新，你先按新口径重对齐，然后直接推进，不要再沿用旧的 branch-first、排队制、或“shared root 还有别人的 dirty 所以我默认不能动”的说法。

当前唯一真实基线：
- 普通线程默认就在 `D:\Unity\Unity_learning\Sunset @ main`
- 默认模型是：`main-only + whitelist-sync + exception-escalation`
- 完成一刀后默认就是：
  - 只提交你自己修改的内容
  - 只提交你自己的白名单路径
  - 一次只收一刀自己的 checkpoint
  - 默认执行纪律是：`一刀一收`

新补强规则：
- 如果你碰过 `Scene / Prefab / Primary.unity / 热 Unity 资源`，回执必须补：
  - 我留了什么
  - 保留什么
  - 清掉什么
  - 当前是否已对我这条线 clean
- 如果你的 `changed_paths` 含 `.unity / .prefab / .asset`，回执必须补资源卫生分类：
  - `有效内容 / 自动副产物 / 调试残留 / 他线脏改`
- 如果你要做 Unity / MCP live 写或 live 验证，先报四字段：
  - `需要什么实例 / 最多占多久 / 只做什么 / 做完退回什么状态`
  - 先按 `mcp-live-baseline.md` 核：`unityMCP + 8888 + pidfile + 当前会话暴露`
  - 如果基线脚本已 pass，但你这条旧线程还在报旧端口 / 旧桥名 / `resources/templates` 为空，不要直接把锅甩给服务端；先按“旧会话 MCP 路由缓存未刷新”处理，必要时换新线程 / 新会话再验
  - 进入 live 前还要先说清：需要什么证据、最多跑几轮、看到什么信号就 `Pause / Stop`
  - 一旦拿到当前步骤需要的证据，立刻 `Pause / Stop` 并退回 `Edit Mode`，不要继续刷成千上万条日志
- 如果你这轮已经推进到“接下来需要用户终验”：
  - 先把你自己能做的 compile / test / MCP 自验做完
  - 再给出专业验收指南：步骤、预期结果、失败判读
  - 再附验收回执单
  - 不要只说“等用户复测”
- 如果你的旧成果还在 branch carrier / worktree，不要继续把它当默认 blocker：
  - 先判断有没有真实有效成果没迁回 `main`
  - 有就先迁最小 checkpoint
  - 再移除旧 blocker
  - 然后回到 `main` 语义继续

如果你这轮改了任何 `.cs` 文件，再补一条：
- 收口前 `git-safe-sync.ps1` 会自动触发代码闸门
- 代码闸门会做：
  - 目标文件 UTF-8 检查
  - `git diff --check`
  - 程序集级编译检查
- 所以不要再把“等用户贴编译报错后再修”当默认流程
- 代码闸门通过，只代表代码层自检通过，不代表 Unity 红错已经验收完成

如果你后面准备对外写“无红错 / 可交接 / 可直接提交”，并且本轮改动触及运行时代码、scene、prefab、asset、UI、剧情、交互、输入或管理器链，再补一条：
- 必须同时给出：
  - 默认先给 CLI 侧的 fresh recompile
  - 默认先给 CLI 侧的 fresh console
  - CLI 覆盖不到时，再补 direct MCP / live 取证，或明确标注 `live 待验证`
- `validate_script`、`CodexCodeGuard`、`git diff --check` 不得单独冒充 Unity 红错闭环
- Sunset 当前默认顺序是：`CLI first, direct MCP last-resort`
- 如果 CLI 与 direct MCP / Unity 当前不可用，只能写：
  - `代码闸门通过`
  - `Unity 红错验证未闭环`
  - `live / console 待补`
- 如果你对外要写任何同义的 `无红错 / 可交接 / 可直接提交` 结论，技术审计层还必须原样补一张 `No-Red 证据卡 v2`，至少包含：
  - `cli_red_check_command`
  - `cli_red_check_scope`
  - `cli_red_check_assessment`
  - `unity_red_check`
  - `mcp_fallback`
  - `mcp_fallback_reason`
  - `current_owned_errors`
  - `current_external_blockers`
  - `current_warnings`
- `cli_red_check_assessment` 只能直接沿用 CLI 原值：
  - `no_red`
  - `own_red`
  - `external_red`
  - `unity_validation_pending`
  - `blocked`
- 如果 `mcp_fallback != not-needed`，`mcp_fallback_reason` 只允许：
  - `baseline_fail`
  - `unity_validation_pending`
  - `blocked`
  - `scene_live_flow_required`
  - `playmode_required`
  - `inspector_required`
- 缺任一项 `No-Red 证据卡 v2`，或只写“通过 / 没问题”却不给命令和 assessment，就视为“日志不可判定”。

只有命中下面这些例外你才停：
1. 撞到同一个高危目标
2. 同一个 Scene / Prefab / 热脚本被多人改
3. 你要做 Unity / MCP live 写，但已经有人在写
4. 你的成果还在 branch carrier / worktree，且尚未迁入 `main`
5. 你已经把现场写坏了，需要先收烂摊子

你先不要泛泛回复“收到”，而是完整回忆并汇报：
1. 你上次真实做到哪里，已经收了什么，还没收的是啥
2. 你当前剩余任务详细清单，哪些必须继续，哪些可以直接判废
3. 你下一步准备一次性推进到哪里，最多做到哪个 checkpoint
4. 你现在是否可直接开工；如果不能，唯一 blocker 是什么

聊天只回这些字段：
- 当前在改什么
- 上次 checkpoint / 当前恢复点
- 剩余任务清单
- 本轮下一步最多推进到哪里
- changed_paths
- 是否触碰高危目标
- 是否需要 Unity / MCP live 写
- `hot_file_hygiene: pass / fail / not-applicable`
- `asset_hygiene: pass / fail / not-applicable`
- `code_self_check: pass / fail / not-applicable`
- `pre_sync_validation: pass / fail / not-run`
- `unity_red_check: pass / blocked / live-pending / not-required`
- `cli_red_check_command`
- `cli_red_check_scope`
- `cli_red_check_assessment: no_red / own_red / external_red / unity_validation_pending / blocked`
- `mcp_fallback: not-needed / used / unavailable`
- `mcp_fallback_reason: baseline_fail / unity_validation_pending / blocked / scene_live_flow_required / playmode_required / inspector_required`
- `current_owned_errors`
- `current_external_blockers`
- `current_warnings`
- 当前是否可以直接提交到 `main`
- 提交 SHA（如有）
- 当前 `git status` 是否 clean
- 如果进入用户终验阶段：验收指南路径 / 验收回执单
- blocker_or_checkpoint
- 一句话摘要
```
