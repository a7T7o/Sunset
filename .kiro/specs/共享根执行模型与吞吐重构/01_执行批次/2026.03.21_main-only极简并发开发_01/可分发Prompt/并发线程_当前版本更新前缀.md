# 并发线程｜当前版本更新前缀

```text
当前版本更新，你先按新制度重对齐，然后直接推进，不要再按旧的 branch / 排队 / “shared root 有 dirty 所以我先不动” 口径回复。

当前唯一真实基线：
- 普通线程默认就在 `D:\Unity\Unity_learning\Sunset @ main`
- 当前默认模型是：`main-only + whitelist-sync + exception-escalation`
- 完成一刀后，默认就是：
  - 只提交你自己修改的内容
  - 只提交你自己的白名单路径
  - 一次只收一刀自己的 checkpoint
- checkpoint 名称 / commit message 统一格式：
  - `YYYY.MM.DD_<线程名>_<编号>`

如果你这轮改了任何 `.cs` 文件，再补一条新制度：
- 收口前 `git-safe-sync.ps1` 会自动触发代码闸门
- 代码闸门会做：
  - 目标文件 UTF-8 检查
  - `git diff --check`
  - 程序集级编译检查
- 所以你不能再把“等 Unity / 用户贴报错后再修”当默认流程

只有命中下面这些例外你才停：
1. 撞到同一个高危目标。
2. 同一个 Scene / Prefab / 热脚本被多人改。
3. 你要做 Unity / MCP live 写，但已经有人在写。
4. 你的成果还在 branch carrier / worktree，尚未迁入 `main`。
5. 你已经把现场写坏了，需要先收口。

你先不要泛泛回答“收到”，而是先完整回忆并汇报：
1. 你上次真实做到哪里，已经收口了什么，还没收口的是什么。
2. 你当前剩余任务的详细清单，哪些必须继续，哪些可以直接判废。
3. 你下一步准备一次性推进到哪里，最多做到哪个 checkpoint 为止。
4. 你现在是否可直接开工；如果不能，唯一阻塞是什么。

回完这轮对齐后，不要继续 docs-first 空转，也不要再等治理线程代提。
除非中途命中上面的硬例外，否则就直接做到你刚刚承诺的那个 checkpoint 再停。

聊天只回这些字段：
- 当前在改什么
- 上次 checkpoint / 当前恢复点
- 剩余任务清单
- 本轮下一步最多推进到哪里
- changed_paths
- 是否触碰高危目标
- 是否需要 Unity / MCP live 写
- `code_self_check: pass / fail / not-applicable`
- `pre_sync_validation: pass / fail / not-run`
- 当前是否可以直接提交到 `main`
- blocker_or_checkpoint
- 一句话摘要
```
