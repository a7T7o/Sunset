# 并发线程 当前版本更新前缀

```text
当前制度已更新，你先按新口径重对齐，然后直接推进，不要再沿用旧的 branch-first、排队制、或“shared root 还有别人的 dirty 所以我默认不能动”的说法。

当前唯一真实基线：
- 普通线程默认就在 `D:\Unity\Unity_learning\Sunset @ main`
- 默认模型是：`main-only + whitelist-sync + exception-escalation`
- 完成一刀后默认就是：
  - 只提交你自己修改的内容
  - 只提交你自己的白名单路径
  - 一次只收一刀自己的 checkpoint
  - 默认执行纪律是：`一刀一收`

新补强规则：
- 如果你碰过 `Scene / Prefab / Primary.unity / 热 Unity 资源`，回执必须补：
  - 我留了什么
  - 保留什么
  - 清掉什么
  - 当前是否已对我这条线 clean
- 如果你的 `changed_paths` 含 `.unity / .prefab / .asset`，回执必须补资源卫生分类：
  - `有效内容 / 自动副产物 / 调试残留 / 他线脏改`
- 如果你要做 Unity / MCP live 写或 live 验证，先报四字段：
  - `需要什么实例 / 最多占多久 / 只做什么 / 做完退回什么状态`
- 如果你的旧成果还在 branch carrier / worktree，不要继续把它当默认 blocker：
  - 先判断有没有真实有效成果没迁回 `main`
  - 有就先迁最小 checkpoint
  - 再移除旧 blocker
  - 然后回到 `main` 语义继续

如果你这轮改了任何 `.cs` 文件，再补一条：
- 收口前 `git-safe-sync.ps1` 会自动触发代码闸门
- 代码闸门会做：
  - 目标文件 UTF-8 检查
  - `git diff --check`
  - 程序集级编译检查
- 所以不要再把“等用户贴编译报错后再修”当默认流程

只有命中下面这些例外你才停：
1. 撞到同一个高危目标
2. 同一个 Scene / Prefab / 热脚本被多人改
3. 你要做 Unity / MCP live 写，但已经有人在写
4. 你的成果还在 branch carrier / worktree，且尚未迁入 `main`
5. 你已经把现场写坏了，需要先收烂摊子

你先不要泛泛回复“收到”，而是完整回忆并汇报：
1. 你上次真实做到哪里，已经收了什么，还没收的是啥
2. 你当前剩余任务详细清单，哪些必须继续，哪些可以直接判废
3. 你下一步准备一次性推进到哪里，最多做到哪个 checkpoint
4. 你现在是否可直接开工；如果不能，唯一 blocker 是什么

聊天只回这些字段：
- 当前在改什么
- 上次 checkpoint / 当前恢复点
- 剩余任务清单
- 本轮下一步最多推进到哪里
- changed_paths
- 是否触碰高危目标
- 是否需要 Unity / MCP live 写
- `hot_file_hygiene: pass / fail / not-applicable`
- `asset_hygiene: pass / fail / not-applicable`
- `code_self_check: pass / fail / not-applicable`
- `pre_sync_validation: pass / fail / not-run`
- 当前是否可以直接提交到 `main`
- 提交 SHA（如有）
- 当前 `git status` 是否 clean
- blocker_or_checkpoint
- 一句话摘要
```
