# NPC规划001 - 自动移动与环境社交草案

## 1. 用户当前基础设想

- 为 NPC 增加自动移动脚本，并且必须回顾现有导航系统后再适配。
- NPC 自动移动策略以“随机间隔移动”作为基础。
- 每一次移动完成后，NPC 会进入短暂停留，区间约为 `0.5s ~ 3s`。
- 连续短暂停留 `3 ~ 5` 次后，进入一次长停留。
- 长停留期间，NPC 会弹出气泡对话框。
- 气泡内容可能是 NPC 自言自语，也可能是与身边的 NPC 进行对话聊天。
- 本文档作为后续 NPC 脑洞风暴与实现规划的第一份基线文档。

## 2. 真实需求理解

- 这次要做的不是“让 NPC 随机走两步”，而是让 NPC 看起来像真的活在场景里。
- 自动移动必须接到当前项目已有的导航能力上，不能绕开阻挡、不能穿墙、不能和现有导航口径打架。
- NPC 的节奏感要成立：移动、短停、长停、冒泡说话要形成可观察的生活流。
- 行为系统必须兼容当前 NPC 的 `Idle / Move` 双状态动画口径，不能额外依赖 `Death` 或复杂动作树。
- 需要保留后续可拓展空间，让我们之后可以继续往“日程”“分时段行为”“关系系统”“多人聊天”上叠。

## 3. 当前项目可复用的现成基础

### 3.1 导航基础

- `NavGrid2D`：当前项目已有 A* 2D 网格寻路与动态障碍刷新能力。
- `PlayerAutoNavigator`：已有完整的路径构建、路径跟随、卡住检测、ClosestPoint 思路，可作为 NPC 导航代理的参考来源。
- `INavigationUnit` / `NavigationUnitBase`：项目里已经预留了“玩家 / NPC / Enemy / StaticObstacle”统一导航单位接口，NPC 自动移动适合直接接这套抽象。

### 3.2 NPC 现有运行时基础

- `NPCAnimController`：当前只维护 `Idle / Move`，并负责左右 `Side + flipX` 桥接。
- `NPCMotionController`：当前负责移动检测、朝向驱动与停驻表现。
- 这意味着 NPC 自动移动系统可以优先只解决“怎么找到点并移动过去”，动画层已有基础桥接。

### 3.3 对话与气泡基础

- `DialogueNode` 已经存在 `isBubble` 字段。
- 这说明项目的剧情/文本数据结构里已经预留了“气泡表现”概念。
- 当前仍未证实是否已有现成的 NPC 头顶气泡运行时组件，因此 V1 规划中应把“气泡表现层”单独列出来，不假装它已经完整可用。

## 4. 第一版建议的行为闭环

### 4.1 核心状态

- `短停留`
- `随机移动中`
- `长停留`
- `自言自语气泡`
- `双人聊天气泡`

### 4.2 基础循环

1. NPC 出生后先处于停留状态。
2. 到达下一次移动触发时，选择一个随机可达点。
3. 通过导航系统移动到目标点。
4. 到达后进入一次短暂停留，时间随机落在 `0.5s ~ 3s`。
5. 记录本轮短停次数。
6. 当短停累计达到随机阈值 `3 ~ 5` 次后，切换到一次长停留。
7. 长停留期间触发一次“气泡说话决策”：
   - 附近没有合适 NPC：走自言自语
   - 附近有合适 NPC：尝试双人聊天
8. 长停留结束后，清空短停计数，重新进入下一轮随机移动循环。

## 5. 面向实现的第一层拓展

### 5.1 随机移动不是纯随机点瞬发

- 每个 NPC 需要一个“活动锚点”概念，通常是出生点或手动指定的 home 点。
- 随机目标应优先从“活动半径内的可达点”里挑选，而不是全图乱飞。
- 随机目标必须经过导航可达性校验；不可达则重新抽点或放弃本轮移动。

### 5.2 长停留不是单纯挂机

- 长停留是“生活感展示窗口”。
- 长停留阶段最适合承载：
  - 自言自语
  - 看向附近 NPC
  - 与近邻 NPC 形成双人聊天
  - 未来扩展为看风景、浇花、观察玩家等环境反馈

### 5.3 双人聊天的最小规则

- 只在长停留阶段尝试触发，避免 NPC 一边赶路一边硬聊。
- 优先寻找附近处于空闲/停留状态的 NPC。
- 若目标 NPC 当前正在移动、正在聊天、或不可打断，则退回自言自语。
- V1 可以先做“面对面站住 + 一到两句气泡”的轻量版，不必一步到位接完整剧情对话树。

### 5.4 动画与移动适配原则

- 自动移动只驱动位移与朝向，不额外引入新的动画状态。
- 运动中：`Move`
- 停驻中 / 聊天中：`Idle`
- 这样能与当前 NPC 动画模板保持一致，也能减少第一版接线成本。

## 6. 推荐的系统拆分方向

- `NPCAutoRoamController`
  - 管理随机移动、短停、长停、状态切换。
- `NPCNavigationAgent`
  - 负责把“想去哪里”接到当前 `NavGrid2D` 和路径跟随逻辑上。
- `NPCSocialBehavior`
  - 负责附近 NPC 搜索、可聊天对象判定、聊天打断规则。
- `NPCBubblePresenter`
  - 负责把一句话显示成头顶气泡，后续再决定是否直接复用现有 Dialogue 数据。
- `NPCRoamProfile`（可后续再上）
  - 用来配置每个 NPC 的活动半径、停留时长、聊天频率、语料风格。

## 7. V1 建议边界

- V1 先不做完整日程系统。
- V1 先不做复杂关系值或社交网络。
- V1 先不做多人围聊。
- V1 先不做跨楼层复杂导航策略。
- V1 先不把 NPC 聊天直接接成正式剧情流程。
- V1 目标应聚焦为：
  - 一个 NPC 能随机走动
  - 能按节奏短停和长停
  - 长停时能冒泡
  - 有机会和附近 NPC 形成轻量聊天

## 8. 后续脑暴优先问题

- 长停留的时长区间要不要也做成随机范围。
- 活动半径是圆形范围、矩形范围，还是基于可编辑区域。
- 双人聊天时是否需要双方都暂停自己的计时器。
- 聊天气泡文本来源是硬编码、配置表、还是直接走 Dialogue 资产。
- 是否需要“固定站岗 NPC”这种关闭随机移动但保留冒泡的特例。
- 当玩家靠近时，NPC 是否需要优先看向玩家或暂停聊天。

## 9. 当前建议的第一落点

- 第一阶段先做“单 NPC 随机移动 + 短停 / 长停 + 自言自语气泡”。
- 第二阶段再接“附近 NPC 配对聊天”。
- 这样能先把导航、状态机、节奏感、气泡表现四件核心骨架跑通，再补社交互动。
