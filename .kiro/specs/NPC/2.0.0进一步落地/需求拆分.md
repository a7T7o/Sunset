# NPC 2.0.0 需求拆分

- 版本：v1.0
- 日期：2026-03-25
- 状态：当前有效
- 目的：完整记录用户本轮提出的详细需求、约束和扩展方向，并给出后续可直接对照执行的专业拆分

## 1. 这份文档解决什么问题

这份文档不是实现稿，也不是单个系统设计稿。

它解决的是：

- 把用户本轮关于 NPC 的大需求完整收拢
- 把“导航契约、场景化、人设、气泡、对话、好感度、受击 / 工具命中 / 反应”这些方向拆成可持续推进的结构
- 避免后续再回到“零散聊天理解”和“文件越长越多、但主线不清”的状态

这份文档是 `2.0.0进一步落地` 的入口说明。

## 2. 用户本轮需求原意整理

用户本轮提出的真实要求，不是单点修补，而是 NPC 产品线升级。

核心要求有三组：

### 2.1 文档治理要求

- `NPC-导航接入契约与联调验收规范` 之前已经写过多个版本，后续不能再继续堆同类文件
- 要求合并成一个最终版本，以后只维护这一个
- 文件数量要少，但每一份都要够精、够稳、够能长期用

### 2.2 NPC 场景化与角色化要求

- `NPC 场景化真实落点设计` 必须开始
- 不能只看最近一轮测试内容，要回看前后剧情、人设、关系线索
- 之前的测试型气泡文案不够用，要开始做真实角色气泡
- 当前 3 个 NPC 已有明确角色原型：
  - `001`：村长
  - `002`：村长女儿
  - `003`：村长二儿子，偏研究型
- 三人的环境气泡、相遇对话、对玩家态度、互相之间的说话内容都不应再是同一种口气

### 2.3 NPC 交互与反应系统要求

- `NPC 受击 / 工具命中 / 反应系统兼容设计` 必须开始进入架构
- 好感度要纳入后续 NPC 体系
- 每个 NPC 的单独对话要开始落地
- 非剧情聊天可以通过气泡做
- 后续需要支持玩家气泡与 NPC 气泡的双向表达
- 玩家气泡需要独立配色和更合适的字体选择
- 但不能破坏当前已经定下的尺寸、边距、动效规范
- 所有表现层前提都是：美观、自然、专业

## 3. 当前总目标的专业定义

这轮需求的总目标不是“继续修 NPC 若干问题”，而是：

- 把 NPC 从“功能验证对象”升级成“可扩展的角色系统”

这个总目标包含 4 个层面的升级：

1. 从多份平行回执，升级到唯一主文件治理
2. 从测试型角色，升级到有人设、有关系、有空间归属的角色
3. 从单向气泡展示，升级到包含玩家/NPC 双向表达的轻交互体系
4. 从“不可被玩家系统触发的背景单位”，升级到能兼容命中、工具、关系变化和反应层的角色对象

## 4. 需求拆分总览

这次需求应拆成 4 条主线，而不是同时摊平成几十个零碎任务。

### 4.1 主线 A：文档与协作口径治理

目标：

- 收敛 NPC 对导航和后续扩展的文档结构

当前必须产出：

- `NPC-导航接入契约与联调验收规范`
- `需求拆分.md`

这条线的成功标准：

- 导航相关只认一个正式契约文件
- 后续扩展方向只认一个总拆分入口
- 不再继续长出“新版回执”“补充回执”“最终回执二次修订版”这类平行文件

### 4.2 主线 B：NPC 场景化真实落点与角色化设计

目标：

- 让 NPC 在村庄中有真正的身份、活动区、相遇逻辑和日常表达

这条线后续要覆盖：

- 角色人设摘要
- 场景归属点
- 活动范围
- 日常站位 / 路线倾向
- 相遇上下文
- 单人环境气泡
- NPC / NPC 相遇对话

### 4.3 主线 C：NPC 轻交互、好感度与双气泡体系

目标：

- 建立正式剧情对话之外的“日常轻交互层”

这条线后续要覆盖：

- 非剧情聊天气泡
- 玩家靠近时的短响应
- 玩家气泡与 NPC 气泡的视觉区分
- 好感度对轻交互语气的影响
- 每个 NPC 的单独日常话术池

### 4.4 主线 D：受击 / 工具命中 / 反应兼容框架

目标：

- 让 NPC 进入“能被玩家行为作用”的系统层，而不是永远只是路人单位

这条线后续要覆盖：

- 命中来源分类
- 反应层级
- 不同行为的反馈差异
- 好感度联动
- 剧情保护 / 特殊状态兼容

## 5. 角色设计方向拆分

基于当前用户已经明确给出的 3 名角色原型，后续角色表达至少要做到“同一套系统，不同的角色语言”。

### 5.1 `001` 村长

当前设计方向：

- 村务视角
- 稳重
- 有判断感
- 对村庄秩序、节奏、生产和人的状态更敏感

后续气泡与对话更适合偏向：

- 观察
- 提醒
- 安排
- 经验式表达
- 对玩家带一点审视和认可递进

### 5.2 `002` 村长女儿

当前设计方向：

- 更偏生活感
- 更有人际观察和情绪波动
- 对环境和人之间的关系更敏感

后续气泡与对话更适合偏向：

- 情绪
- 日常
- 好奇
- 轻微吐槽或关心
- 对玩家的态度更容易随好感变化

### 5.3 `003` 村长二儿子 / 研究型

当前设计方向：

- 偏研究癖
- 观察型
- 对异常、现象、试验结果、工具和材料更敏感

后续气泡与对话更适合偏向：

- 推测
- 记录感
- 研究碎片
- 新发现
- 对玩家可能表现出“拿你当观察对象”的兴趣

## 6. 场景化真实落点的具体需求

这里的“场景化真实落点”不能被误解成只摆一个 `Home Anchor`。

它实际包含 5 层：

1. 空间归属
- 每个 NPC 属于哪里
- 哪些点是他常驻点、停留点、工作点、观察点

2. 时间语义
- 不同时段更可能在哪
- 不同状态下会不会改变活动区

3. 相遇逻辑
- 谁更容易遇到谁
- 为什么会聊
- 见面时说的内容为什么合理

4. 场景内容挂钩
- 哪些气泡是“这个地点才会说”
- 哪些话是“这个时间 / 这个地点 / 这个对象都满足才会说”

5. 正式 NPC 与验证样本的边界
- `003` 当前仍承担验证样本职能
- 但它的人设内容不能永远被测试句占住
- 必须保留“验证模式”和“正式角色表达”之间的边界

## 7. 气泡与对话系统的具体需求

### 7.1 NPC 气泡不再允许继续停留在测试口吻

后续 NPC 气泡要明确区分：

- 单人环境气泡
- 看见玩家时的轻交互气泡
- NPC / NPC 相遇对话
- 状态型气泡

### 7.2 每个 NPC 需要自己的话术风格

这意味着后续不能只维护一个共用句库。

至少要拆：

- 公共基础句型
- `001` 专属句型
- `002` 专属句型
- `003` 专属句型
- 两两相遇时的差异句型

### 7.3 正式对话与气泡表达要双轨并存

后续系统应明确区分：

- 正式剧情 / 正式交互对话
- 非剧情轻量气泡表达

两者不能互相替代，但应互相服务。

## 8. 好感度与关系系统的需求含义

用户虽然没有要求这一轮直接实现完整好感度系统，但已经明确要求：

- 好感度必须纳入后续设计
- 同一个 NPC 对玩家的气泡和轻交互不能永远固定不变

这意味着后续设计里，至少要预留：

- 关系阶段
- 好感度对语气的影响
- 对话内容的解锁层次
- 命中 / 冒犯 / 帮助行为对关系的反馈路径

## 9. 受击 / 工具命中 / 反应兼容的具体需求

这条线不是“小补丁”，而是 NPC 正式进入交互系统的开始。

至少要区分：

### 9.1 触发来源

- 玩家普通碰撞
- 玩家攻击
- 工具挥击
- 特殊交互触发
- 误伤或范围影响

### 9.2 反应层级

- 无视
- 注意到
- 不满
- 警惕
- 抗议
- 关系下降
- 特殊剧情反应

### 9.3 兼容目标

- 不和当前导航 / 漫游状态机直接打架
- 不要求后续推翻 NPC 当前运动基线
- 能按角色差异给出不同反馈
- 能与好感度挂钩

## 10. 玩家气泡系统的需求含义

用户要求玩家气泡：

- 单独设计配色
- 字体可以单独选择
- 但沿用当前已经定下来的尺寸、边距、动效和整体规范

这意味着后续正确方向不是“重做一套 UI”，而是：

- 在同一套气泡体系里做玩家 / NPC 双视觉语言

后续玩家气泡至少要满足：

- 一眼能和 NPC 气泡区分
- 又不会像两个完全不同的产品
- 与当前 NPC 气泡的边距、尺寸逻辑、动效节奏兼容

## 11. 后续实施顺序建议

这次需求不能乱序推进，合理顺序应是：

### 阶段 0

- 收敛唯一导航契约文件
- 建立 `需求拆分.md`

### 阶段 1

- 开始 `NPC 场景化真实落点与角色日常设计`
- 优先把 001 / 002 / 003 的角色表达框架做出来

### 阶段 2

- 开始 `NPC 轻交互、好感度与双气泡体系`
- 把测试气泡逐步替换成角色化内容

### 阶段 3

- 开始 `受击 / 工具命中 / 反应兼容框架`
- 为后续交互系统落地打基础

### 阶段 4

- 等导航交付后，把行为、场景、相遇和角色表达真正接起来做联调整验

## 12. 本工作区建议维护的核心文件

为了满足“文件少而精”，`2.0.0进一步落地` 后续建议只维护少量主文件：

1. `需求拆分.md`
- 总入口
- 记录用户新增需求与阶段拆分

2. `NPC-导航接入契约与联调验收规范.md`
- 唯一导航协作文档

3. 后续择机新增：
- `NPC场景化真实落点与角色日常设计.md`
- `NPC交互反应与关系成长设计.md`

不建议再继续增加：

- 大量同主题“补充回执”
- 重复说明同一边界的“最新版”
- 只是在聊天里整理过一遍就再起一份新文档

## 13. 当前一句话结论

本轮用户需求的真实含义，不是继续给 NPC 打补丁，而是正式把 NPC 推进到 2.0.0 阶段：文档治理收口、角色化落地、场景化设计、轻交互与关系层建立，以及受击 / 工具命中 / 反应兼容框架启动。
