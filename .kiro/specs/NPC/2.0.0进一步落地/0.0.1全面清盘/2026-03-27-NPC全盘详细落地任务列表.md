## 1. 使用规则

这份列表不是宽泛规划，而是执行账本。

后续任何“开始做 NPC 下一刀”的动作，都必须先回答 4 件事：

1. 当前做的是哪一个任务编号
2. 该任务当前状态是什么
3. 它的前置条件是否已满足
4. 完成后要交什么证据

如果上面 4 件事答不清，就不应宣称“已经进入下一刀”。

## 2. 状态字段定义

- `pending`
  - 还没开始做，但前置条件理论可满足
- `in-progress`
  - 已进入施工
- `blocked-hotfile`
  - 被 `Primary.unity`、共享根热区或同根 dirty 阻塞
- `blocked-external`
  - 被导航线程、外部 owner、Unity 会话或联合依赖阻塞
- `done`
  - 已完成并有明确证据
- `parked`
  - 当前故意不做，等待更高优先级闭环

## 3. 任务类型定义

- `可直接施工`
  - 当前 NPC 线自己就能推进
- `联合前置后施工`
  - 需要先拿到 scene 写窗口、联合准入或外部交付
- `外部阻塞项`
  - 不应由当前 NPC 局部线程单独吞并，但必须列入总账

## 4. 总体验收口径

只有当以下 6 个层面都完成，才可以称为“NPC 这条线完成到当前版本可验收”：

1. `P0` 基线闭环完成
2. `P1` 场景化真实落点最小可玩切片完成
3. `P2` 角色化内容正式成型
4. `P3` 轻交互与双气泡完成首版
5. `P4` 关系成长完成最小接线
6. `P5 + P6` 的反应系统与联调 / 测试闭环完成

如果其中任何一层仍处于 `pending / blocked-* / parked`，都不能把整条 NPC 线说成“只差一点点就做完”。

## 5. 总执行纪律

后续所有任务都必须遵守：

1. 不得把 `Primary.unity` mixed cleanup 偷换成业务推进
2. 不得顺手吞 `DialogueChinese*` 字体
3. 不得把导航核心动态避障 / 会车 / 礼让框架偷换成 NPC 自己修
4. 在 `HomeAnchor` 运行态基线没闭环前，不得扩大量 scene 行为
5. 后续文档应优先继续维护当前主文件，不要继续长平行“终版 / 二版 / 补充版”
6. 只要进入 Unity / MCP live 写或 live 验证，必须先说明：
   - 只做什么
   - 最多占多久
   - 做完退回什么状态
