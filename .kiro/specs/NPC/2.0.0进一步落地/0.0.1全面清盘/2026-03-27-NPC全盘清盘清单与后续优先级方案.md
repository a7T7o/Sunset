## 1. 这份文档解决什么问题

这份文档不重复讲 NPC 历史，而是基于历史总表，把“现在到底该先做什么、后做什么、哪些暂时不能吞、哪些是外部前置”一次性梳理成可执行清单。

目标只有两个：

1. 把 NPC 总线所有未完成项重新拉平，不再被 `NPCV2` 眼前这一刀带偏
2. 给出一份后续能直接执行的优先级顺序，避免再陷入“想到哪修到哪”

## 5. 后续优先级重排

这里的优先级不是“想做什么”，而是按依赖关系、验收价值和风险排序后的结果。

## P0｜先把当前基线闭环，不再让 `HomeAnchor` 卡在半空

这是当前最高优先级。

### 为什么必须排第一

1. `HomeAnchor` 是 2026-03-26 已经正式开过刀的现场
2. 这刀如果不闭环，后续所有“场景化真实落点”都缺可信基线
3. 当前用户已经明确盯住“运行中 Inspector 还是空”这个现实问题

### 这一优先级包含什么

1. 运行中 `Home Anchor` 非空的最终实证
2. 若仍为空，把断点继续压窄到唯一剩余事实
3. `NPCV2` own 文档与白名单收口
4. 补一轮与当前工具链直接相关的最小验证

### 完成标准

1. 在不停 Unity 的前提下，`001 / 002 / 003` 的 Inspector 能明确看到 `Home Anchor` 非空
2. 如果仍有一个为空，必须能精确说明卡在赋值链哪一环
3. 当前这把刀的 own 路径可以安全收口

## P1｜做“场景化真实落点”的最小可玩切片，而不是继续停在锚点占坑

这条是 NPC 总线下一阶段最值得优先推进的业务主刀。

### 为什么排第二

1. 历史需求里，`HomeAnchor` 从来不是最终目的，只是进入真实场景化的最低门槛
2. `NPC场景化真实落点与角色日常设计.md` 已经把结构写明，当前缺的是最小实现切片
3. 这部分仍主要属于 NPC 自己的业务责任，不必等全部重型交互系统做完

### 这一优先级包含什么

1. 为 `001 / 002 / 003` 落一套最小真实空间表达：
   - 常驻区
   - 1 到 2 个常见停留点
   - 1 个公共相遇点
   - 1 个边缘自言自语点
2. 把“只会围着一个 `HomeAnchor` 漫游”升级成“有空间倾向的最小活动组织”
3. 让 scene 点位与角色方向开始对应起来

### 完成标准

1. `001 / 002 / 003` 不再只是从一个点随便散开
2. 玩家能看出三人在哪类区域更常出现
3. 不需要先碰导航核心，也能形成第一版“像真的住在村里”的感觉

## P2｜稳定角色化内容，不再让系统层和内容层脱节

这条是第二个业务主刀。

### 为什么排第三

1. 场景化真实落点如果没有角色化内容支撑，最后看起来仍像测试对象
2. `001 / 002 / 003` 的角色方向已经明确，继续拖只会造成系统先行、内容永远欠账
3. 这部分能直接提升用户体感，不需要等命中反应系统完成

### 这一优先级包含什么

1. 公共基础句型与三人专属句型正式拆层
2. `001 <-> 002`、`001 <-> 003`、`002 <-> 003` 两两相遇句型
3. 与场景点位、时间语义、状态语义挂钩的环境气泡
4. `003` 验证样本文案继续向正式角色表达靠拢，但保留验证模式边界

### 完成标准

1. 三个 NPC 的话术听起来不再像同一个人换壳
2. 同一类场景下，三人会说出不同立场与语气的话
3. 相遇内容不再是统一模板

## P3｜建立“轻交互 + 双气泡”的最小成型版本

这是玩家开始真正感到“NPC 会回应我”的转折点。

### 为什么排第四

1. 双气泡和轻交互是 2.0.0 的正式要求，但它依赖角色内容与表现风格先稳
2. 如果在角色化内容没稳定前先做，容易变成一套空 UI
3. 这部分能明显提升体验，但不必抢在场景化和角色内容之前

### 这一优先级包含什么

1. 玩家 / NPC 双气泡视觉规范正式落地
2. 玩家靠近时的最小轻响应
3. 正式对话轨与日常气泡轨的实际分工
4. 不破坏当前已定下的边距、尺寸、动效节奏和整体专业感

### 完成标准

1. 玩家气泡与 NPC 气泡一眼可区分，但仍像同一产品体系
2. 玩家靠近 NPC 时能拿到短反馈，而不是只有环境自言自语
3. 气泡表现仍符合“美观、自然、专业”的硬验收项

## P4｜把关系成长做成“最小可接线结构”，而不是一开始就铺大系统

### 为什么排第五

1. 历史需求明确要求好感度进入体系，但并没有要求先上完整数值大系统
2. 关系成长应该服务于轻交互内容差异，而不是先长一套空数据
3. 正确顺序是先有角色内容和轻交互，再把关系变化接进来

### 这一优先级包含什么

1. 最小关系阶段结构
2. 同一句轻交互在不同阶段的语气差异
3. 内容解锁层次的最小钩子

### 完成标准

1. 同一个 NPC 对玩家的反馈不再永远固定
2. 关系变化能影响轻交互语气或内容层级
3. 不需要先把整个剧情系统一起拉进来

## P5｜最后再接受击 / 工具命中 / 反应兼容

### 为什么排第六

1. 这是重型系统接入，风险最高
2. 它同时依赖：
   - 角色差异
   - 关系阶段
   - 当前漫游 / 导航基线稳定
3. 过早做这条，最容易把 NPC 现有运动与表现层全部打乱

### 这一优先级包含什么

1. 命中来源分类
2. 反应层级接线
3. 与关系成长联动
4. 与当前漫游 / 导航状态机兼容

### 完成标准

1. NPC 能对玩家行为作出差异化反馈
2. 不会直接打穿当前运动基线
3. 不同角色、不同关系阶段能有不同反应

## P6｜导航联调与正式测试实跑

这是必须完成的后段主线，但它不应该抢在前面的 NPC 自有主线之前。

### 为什么排最后

1. 部分内容本来就不是 NPC 当前局部线程自己能吞的
2. 导航线程交付前，提前硬接只会反复返工
3. 更合理的是先把 NPC 自己的内容、场景和轻交互基线定稳，再做联调整体验收

### 这一优先级包含什么

1. 导航交付后的 NPC 行为联调
2. 玩家绕 NPC、NPC / NPC 避让、会车 / 礼让后的整体验收
3. Unity Test Runner 中 NPC 工具链与关键流程的正式实跑
4. 最终用户体验验收

### 完成标准

1. NPC 自己的表现层与导航基线不打架
2. 真实场景里不会因为导航缺口反复把 NPC 体验拖回测试态
3. 工具链与运行态至少有一组正式测试与手动验收闭环

## 6. 当前建议执行顺序

如果从现在开始继续做，推荐顺序固定如下：

1. 先完成 P0：
   - 把 `HomeAnchor` 运行态闭环
   - 收当前 `NPCV2` own 刀口
2. 再做 P1：
   - 用最小 scene 切片把“真实空间归属”做出来
3. 再做 P2：
   - 让三人真正说出不同的话
4. 再做 P3：
   - 玩家 / NPC 双气泡和轻响应正式成型
5. 再做 P4：
   - 关系阶段最小接线
6. 再做 P5：
   - 命中 / 工具反应兼容
7. 最后做 P6：
   - 导航联调
   - 正式测试
   - 用户终验

## 7. 现在立刻该做的第一步

如果只说“下一步到底先干嘛”，答案非常明确：

### 当前第一步

先把 P0 做完。

### 具体含义

1. 不再扩刀到 `Primary.unity` mixed cleanup
2. 不去碰字体
3. 不去吞导航 runtime
4. 只把当前 `HomeAnchor` 运行态实证做闭环
5. 把这一刀 own 路径收口

### 为什么必须这样

因为当前最危险的不是“功能少”，而是“基线没闭环就急着往后堆功能”。

一旦 P0 还没闭环就继续往 P1/P2 扩，后面所有真实场景化、角色化和双气泡推进都会继续踩在不稳定现场上。

## 8. 当前一句话结论

NPC 这条线现在最合理的清盘顺序不是“哪里有反馈就补哪里”，而是：

先闭当前基线，再做真实场景化，再稳角色内容，再上轻交互与双气泡，再接关系成长，最后才进入命中反应和导航联调。
