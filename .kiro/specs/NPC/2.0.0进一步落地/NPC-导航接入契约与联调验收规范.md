## 3. 当前冻结的 NPC 稳定语义

### 3.1 状态语义

`NPCAutoRoamController` 当前稳定状态：

- `Inactive`
- `ShortPause`
- `Moving`
- `LongPause`

当前冻结口径：

- `ShortPause`：短暂停留阶段，可承载短气泡、自言自语、轻量停顿
- `Moving`：NPC 正在执行路径推进
- `LongPause`：长停留阶段，可承载更明显的停顿、聊天或观察

导航线程当前不能默认 NPC 会静默改掉这组状态语义。

### 3.2 `IsMoving` 语义

当前存在两层 `IsMoving`：

- `NPCAutoRoamController.IsMoving`：状态层语义
- `NPCMotionController.IsMoving`：动画桥接层语义

当前冻结口径：

- 两者都视为稳定承诺
- 导航联调期间，NPC 线程不会无通知改掉其含义

### 3.3 最终位移语义

当前 NPC 的运动职责分层仍是：

- 路径推进主循环在 `NPCAutoRoamController`
- 动画和速度桥接在 `NPCMotionController`
- 最终位移主语义仍以 `rb.MovePosition(...)` 为主

这条基线在导航主线未收口前，不应被导航线程反向要求 NPC 先推翻。

### 3.4 当前 prefab / 碰撞基线

当前共同样本基线：

- `Assets/222_Prefabs/NPC/001.prefab`
- `Assets/222_Prefabs/NPC/002.prefab`
- `Assets/222_Prefabs/NPC/003.prefab`

当前冻结的物理输入：

- `BoxCollider2D`
- `Rigidbody2D`
- `isTrigger = false`
- 下半身实体碰撞
- `mass = 6`
- `linearDamping = 8`
- `tag = Untagged`

### 3.5 `003` 的正式定位

`003.prefab` 当前正式定位：

- 默认保留 `NPCBubbleStressTalker`

同时明确：

- `NPCBubbleStressTalker` 是长期测试入口
- 它不是正式生产 NPC 的常驻业务行为组件

### 3.6 当前不能被误当成长期承诺的内容

下面这些不属于本契约的长期冻结承诺：

- `NPCBubbleStressTalker` 的具体文案和节奏

## 4. 导航 -> NPC 的最小接入契约

### 4.1 NPC 提供给导航的稳定输入

NPC 当前稳定提供给导航的输入：

- 当前 agent 是 NPC
- 当前 agent 采用下半身实体碰撞基线
- 当前 agent 具有明确 pause / move 状态语义
- 当前最终位移消费语义仍围绕 `Rigidbody2D` 与 `MovePosition`
- 当前可以区分正式 NPC 与验证样本 NPC，其中 `003` 是验证样本

导航线程当前不应额外要求 NPC 先改：

- pause 状态定义
- `IsMoving` 定义
- 最终位移语义
- prefab 基础碰撞结构

### 4.2 导航必须提供给 NPC 的最小输出

在 NPC 真正进入主集成前，导航至少要能稳定给出这些语义等价输出：

- 当前移动意图是否可继续执行
- 当前交通裁决结果：继续 / 让行 / 等待 / 侧绕 / 恢复
- 当前路径是否仍有效，还是进入重规划 / 重建 / recover
- 当前阻塞是否来自动态 agent
- 到达、卡住、恢复完成是否可清晰区分

如果导航暂时不愿暴露正式 API，也至少必须有等价可观测证据：

- 结构化日志
- debug overlay
- Inspector 可见状态
- 或专用 live validation 输出

### 4.3 导航回给 NPC 的联调证据字段

导航线程未来回包至少必须包含：

- 本轮 live 基线：`branch / HEAD / scene`
- `changed_paths`
- 本轮参与联调的 agent 样本：玩家、`001/002/003` 中哪些对象
- 每个场景下的交通裁决证据：谁让谁、谁等、谁重规划、谁 recover
- 最终行为结果：通过 / 不通过 / 部分通过
- 当前仍失败的边界
- 如果需要 NPC 配合，明确到接口、字段或执行点

## 5. 联调验收清单

| 场景 | 导航线程必须回的证据 | NPC 线程会看的现象 | 通过标准 | 不通过判定 |
| --- | --- | --- | --- | --- |
| 玩家右键自动导航，横切移动中的 NPC | live 录屏或等价证据 + 交通裁决状态 + changed_paths | 玩家是否自然减速、侧绕、等待，而不是直推 NPC | 玩家不再把移动中的 NPC 顶着走；行为可重复复现 | 仍出现“玩家推着 NPC 走”或靠纯碰撞硬顶通过 |
| NPC 漫游迎面遇到玩家 | 让行 / 侧绕 / 等待 的状态证据 + 失败样本说明 | NPC 是否不再傻撞玩家 | NPC 至少能表现出稳定绕行或等待策略 | 仍是迎面硬顶、反复 stuck、无语义区分 |
| NPC / NPC 会车 | 两个 NPC 的交通裁决证据 + recover 证据 | 是否还会互顶、互卡、互穿 | 两个 NPC 能通过且不会长时间死锁 | 仍明显互卡、互撞、互穿 |
| 路口礼让 | 十字或岔路样本 + 优先级说明 | 是否像先后通过而不是乱撞 | 至少能看到稳定礼让趋势 | 谁都不让、或全靠物理硬挤 |
| 卡住恢复 | stuck 检测与 recover 证据 + 恢复后的结果 | 卡住后是否能恢复 | recover 后能重新进入有效推进 | recover 只存在于日志，运行态仍卡死或抽搐 |

### 5.1 NPC 线程对联调证据的最低要求

- 至少一份真实运行态证据，不接受只贴 synthetic baseline
- 至少一份真实业务场景下的证据，不接受只在脱离业务语义的测试环境里自证
- 至少说明仍失败的边界，不接受只报喜不报忧

### 5.2 NPC 线程不会把什么当成“已经过线”

下面这些不能当成进入 NPC 主集成的通行证：

- 只跑通了静态 A* 求路
- 只证明了 detour / synthetic baseline 绿灯
- 只证明了局部 steering 偶尔能侧一下
- 只证明了某个日志打印过 `Proceed / Yield`

NPC 线程只认真实运行态里“不能推着 NPC 走、不能互撞成团、能让、能 recover”的行为闭环。

## 6. 当前还不配进入 NPC 集成的红线

只要命中以下任一条，NPC 线程就不应进入主集成：

- 玩家右键导航时仍能把移动中的 NPC 顶着走
- NPC / NPC 仍会高概率互撞、互卡或互穿
- 交通裁决只存在于日志层，运行态没有对应行为结果
- 还无法说明失败落在裁决层、求解层还是最终执行层
- 还需要 NPC 线程先改 `IsMoving`、pause 语义或最终位移方式，导航结果才勉强成立
- 只有 synthetic / unit / probe 证据，没有真实 live 业务场景证据
- 还拿不出明确的失败边界和下一步缺口说明
