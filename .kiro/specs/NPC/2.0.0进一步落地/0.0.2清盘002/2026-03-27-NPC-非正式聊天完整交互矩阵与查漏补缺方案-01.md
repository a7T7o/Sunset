# 2026-03-27 NPC 非正式聊天完整交互矩阵与查漏补缺方案

## 1. 文档目的

这份文档不是“理想脑暴稿”，而是把当前 `NPCV2` 这条线已经真实写进代码的行为、已经被用户实测确认的问题、以及后续应该收敛到的理想矩阵，一次性钉成后续施工标准。

后续 `NPC` 非正式聊天不再允许只看“能跑起来”，而必须按这份矩阵逐项补齐。

## 2. 当前分析依据

本文件的“当前实现”全部来自以下真实代码与已确认现象：

- `D:\Unity\Unity_learning\Sunset\Assets\YYY_Scripts\Service\Player\PlayerNpcChatSessionService.cs`
- `D:\Unity\Unity_learning\Sunset\Assets\YYY_Scripts\Story\Interaction\NPCInformalChatInteractable.cs`

- `D:\Unity\Unity_learning\Sunset\Assets\YYY_Scripts\Data\NPCDialogueContentProfile.cs`

同时吸收以下已经被用户亲测确认的现象，视为当前真实 bug，而不是“猜测”：

- 玩家聊到一半突然跑开时，玩家气泡可能卡在第一个字。
- 该卡字异常会导致体验上像“重新靠近 NPC 才继续”。
- 玩家离开时，NPC 也可能出现被卡住或过渡割裂。
- 当前虽然已经补了自动续聊，但离开链依然不完整、不丝滑。

## 3. 当前真实状态机

当前 `PlayerNpcChatSessionService` 的主状态为：

| 状态 | 当前含义 | `E` 键当前语义 |
|---|---|---|
| `Idle` | 没有会话 | 可发起聊天 |
| `PlayerTyping` | 玩家当前句正在打字 | 跳过当前玩家打字动效 |
| `WaitingNpcReply` | 等 NPC 回复前的延迟 | 加速 NPC 回复到来 |
| `NpcTyping` | NPC 当前句正在打字 | 跳过当前 NPC 打字动效 |
| `AutoContinuing` | 两轮之间的自动续聊空档 | 当前仍可跳过这段空档 |
| `Interrupting` | 已进入离场中断链 | 跳过当前中断动效 |
| `Completing` | 会话结束前的收尾停留 | 跳过收尾停留 |

当前主链已经不是旧版“再按一次 `E` 才能进下一轮”，而是：

`玩家句打字 -> NPC 等待 -> NPC 句打字 -> AutoContinuing -> 下一轮`

这条链当前已经有 live 证据，不再是推测。

## 4. 当前真实“玩家离开”矩阵

### 4.1 先按触发源分类

| 离开触发源 | 当前命中代码路径 | 当前 endReason | 当前有没有玩家离场句 | 当前有没有 NPC 反应 | 当前问题 |
|---|---|---|---|---|---|
| 玩家距离超过 `SessionBreakDistance` | `Update() -> StartWalkAwayInterrupt()` | `WalkAwayInterrupt` | 有 | 有 | 是当前唯一真正做了内容分流的离开路径 |
| 打开阻塞 UI | `Update() -> CancelConversationImmediate()` | `Cancelled` | 没有 | 没有 | 太硬，像被砍断 |
| 正式 Dialogue 抢占 | `Update() -> CancelConversationImmediate()` | `Cancelled` | 没有 | 没有 | 没有“被打断”专用收束 |
| 当前 NPC 失活 / 被禁用 | `Update() -> CancelConversationImmediate()` | `Cancelled` | 没有 | 没有 | 没有过渡，直接蒸发 |
| 玩家侧 `PlayerMovement` 丢失 | `Update() -> CancelConversationImmediate()` | `Cancelled` | 没有 | 没有 | 系统硬断 |
| 会话服务自身 `OnDisable` | `OnDisable() -> CancelConversationImmediate()` | `Cancelled` | 没有 | 没有 | 系统硬断 |
| 会话已进入 `Completing` 后玩家离开 | 不再触发离场链 | `Completed` | 没有 | 没有 | 逻辑上算聊完，但观感可能割裂 |

### 4.2 再按“玩家跑开时机”分类

这张表只讨论当前唯一真正有内容的离开路径，也就是“距离越界触发 `WalkAwayInterrupt`”。

| 玩家跑开发生时机 | 当前理论代码行为 | 当前真实体验结论 | 当前是否理想 |
|---|---|---|---|
| `PlayerTyping` 期间跑开 | 停掉原会话协程，进入 `RunWalkAwayInterrupt()`，开始播 `playerExitLine` | 用户已实测存在严重 bug：玩家气泡可能卡在第一个字，离场句切换不稳定 | 不理想 |
| `WaitingNpcReply` 期间跑开 | 直接切入中断链，播玩家离场句，再播 NPC cue / reaction | 当前这条相对最稳，live trace 也已拿到 | 部分理想 |
| `NpcTyping` 期间跑开 | 切断 NPC 正在说的句子，转入玩家离场句，然后 NPC 反应 | 逻辑能走，但情绪上突兀，因为 NPC 原句会被硬腰斩 | 不理想 |
| `AutoContinuing` 期间跑开 | 结束自动续聊空档，转入离场链 | 逻辑上可行，但当前没有“刚准备继续时被叫停”的细化反应 | 不理想 |
| `Interrupting` 期间继续跑 | 不重复触发，继续当前中断链 | 合理 | 基本理想 |
| `Completing` 期间跑 | 不再走离场链，只等完成收尾 | 逻辑成立，但观感可能像“人都走了还在原地聊完” | 不理想 |

### 4.3 当前 `WalkAwayInterrupt` 内部真实内容结构

当前距离越界后，真实内容顺序是：

1. 取当前关系阶段的 `walkAwayReaction`
2. 播放 `playerExitLine`
3. 等待 `0.18s`
4. 如果有 `reactionCue`，先播一个紧凑 cue
5. 再等待 `npcReactionDelay`
6. 如果有 `npcReactionLine`，播 NPC 反应句
7. 应用 `relationshipDelta`
8. 进入 `Completing`
9. 收尾结束，记为 `WalkAwayInterrupt`

当前已经做成的分流维度只有一个：

| 维度 | 当前是否已实现 | 当前说明 |
|---|---|---|
| 按 `NPCRelationshipStage` 分流 | 已实现 | 通过 `ResolveWalkAwayReaction(_activeRelationshipStage)` |
| 按“离开的原因”分流 | 未实现 | 现在没有“有事先走 / 敷衍逃跑 / 温柔告辞 / 故意气人”这样的 cause |
| 按“离开发生时机”分流 | 未实现 | 早跑、晚跑、打断 NPC 说话，目前都走同一套 reaction |
| 按“玩家操作强度”分流 | 未实现 | 轻微越界和高速逃跑没有区别 |
| 按“是否还能回聊”分流 | 未实现 | 现在结束后只是 idle，没有 continuity 设计 |

## 5. 当前代码已经有内容，但为什么仍然不理想

### 5.1 当前最大问题不是“没内容”，而是“离开 cause 与离开 phase 没正交”

现在的离开逻辑只有一个粗暴判断：

- 只要玩家越界了
- 并且当前还没在 `Interrupting / Completing`
- 就统一走 `WalkAwayInterrupt`

这会直接导致：

- 玩家是轻轻走出去一步，还是突然拔腿就跑，当前系统不区分。
- 玩家是在自己说话时走，还是在 NPC 正在说关键句时走，当前系统不区分。
- 玩家是礼貌抽身，还是故意敷衍，当前系统不区分。

所以现在“内容是有的”，但“矩阵是不成立的”。

### 5.2 当前第二大问题是“中断接管不原子”

用户已经实测到：

- 玩家在 `PlayerTyping` 阶段突然跑开时，玩家气泡会卡在第一个字。

这说明当前 `PlayerTyping -> WalkAwayInterrupt` 的接管虽然在代码上看似连上了，但在表现层没有做到原子切换。

换句话说，当前缺的不是“又加一句对白”，而是：

- 当前句如何被终止
- 离场句如何无缝接手
- 旧 partial text 如何被清理
- 玩家 / NPC bubble 的 renderer 生命周期如何切换

这一层现在没有被设计完整。

### 5.3 当前第三大问题是“`Cancelled` 太粗暴”

当前所有系统接管类离开：

- UI 打开
- 正式 Dialogue 抢占
- 目标 NPC 失活
- Service disable

全部直接 `CancelConversationImmediate()`

结果是：

- 没有玩家离场句
- 没有 NPC 反应
- 没有“被系统打断”的过场
- 也没有 cause-specific 的结束语义

这在代码上省事，但在体验上像会话被砍断。

### 5.4 当前 `E` 的语义仍然没有完全收干净

用户原始要求是：

- `E` 跳过动效
- `E` 不是跳过对话

当前虽然已经不需要第二次 `E` 才能进下一轮，但 `E` 仍然会：

- 加速 `WaitingNpcReply`
- 跳过 `AutoContinuing`
- 跳过 `Interrupting` 的等待
- 跳过 `Completing` 的等待

这比旧版好很多，但距离“只跳动效，不改时序语义”还差一步。

## 6. 理想的完整交互矩阵

### 6.1 理想矩阵的四个正交维度

后续必须把离开拆成这四个维度，而不是继续堆 if：

| 维度 | 理想定义 | 当前状态 |
|---|---|---|
| 离开原因 `LeaveCause` | 为什么离开 | 缺 |
| 离开发生时机 `LeavePhase` | 在哪一拍离开 | 只有隐式，没有显式模型 |
| 关系阶段 `RelationshipStage` | NPC 当前关系档位 | 已有 |
| 结果策略 `ExitOutcome` | 播什么、掉不掉好感、能否续回 | 缺 |

### 6.2 理想 `LeaveCause` 分类

后续理想上至少应拆成：

| `LeaveCause` | 说明 | 理想体验 |
|---|---|---|
| `StepOutProbe` | 玩家只是轻微越界，可能马上回来 | 不立刻判离开，给 grace window |
| `SoftExit` | 玩家明确但礼貌地结束闲聊 | 玩家有完整告辞句，NPC 正常回应 |
| `AbruptRunAway` | 玩家聊到一半突然跑掉 | 玩家句更短更急，NPC 反应更受伤 / 更困惑 |
| `NpcSpeakingInterrupted` | 玩家打断 NPC 当前句离开 | NPC 应有“句子被截断”的特化反应 |
| `SystemTakeover` | 正式剧情 / UI / 演出强制接管 | 不该像蒸发，应有系统接管收束 |
| `TargetInvalid` | NPC 被禁用、切场景、对象失效 | 走静默 abort，但要清晰收尾 |
| `PostCompletionRelease` | 已聊完，玩家只是离开 | 不应再触发跑开惩罚 |

### 6.3 理想 `LeavePhase` 分类

| `LeavePhase` | 理想定义 | 当前是否显式存在 |
|---|---|---|
| `PlayerSpeaking` | 玩家当前句动画中 | 否 |
| `NpcThinking` | NPC 回复前等待中 | 否 |
| `NpcSpeaking` | NPC 当前句动画中 | 否 |
| `BetweenTurns` | 两轮之间空档 | 否 |
| `InterruptSequence` | 已在离开收束中 | 否 |
| `CompletionHold` | 聊完收尾中 | 否 |

后续应在状态机里把这层从“隐含在 `_state` 里”升级成可供 reaction 选择的显式上下文。

### 6.4 理想离开结果矩阵

| `LeaveCause` x `LeavePhase` | 玩家气泡 | NPC 气泡 / cue | 好感变化 | 收尾语义 |
|---|---|---|---|---|
| `StepOutProbe x any` | 不立刻切离场句 | NPC 不立刻反应 | 0 | grace 后若未回来再升级 |
| `SoftExit x PlayerSpeaking` | 先收掉当前 partial，再播完整告辞句 | NPC 礼貌回应 | 0 或微升 | `SoftExitCompleted` |
| `SoftExit x NpcThinking` | 播简短告辞句 | NPC 轻回应 | 0 | `SoftExitCompleted` |
| `AbruptRunAway x PlayerSpeaking` | 玩家句直接被替换成急促离场句 | NPC 困惑 / 不满 cue + reaction | 微降 | `WalkAwayInterrupt` |
| `AbruptRunAway x NpcSpeaking` | 玩家不一定说整句，只可有短促甩句 | NPC 应有“被打断”反应 | 微降或中降 | `WalkAwayInterrupt` |
| `NpcSpeakingInterrupted` | 玩家可无额外句 | NPC 有被截断专属 cue | 视关系而定 | `InterruptedWhileNpcSpeaking` |
| `SystemTakeover` | 可不播完整句，但要有系统接管收束 | NPC 不应像蒸发 | 0 | `SystemCancelled` |
| `TargetInvalid` | 直接静默收尾 | 无 | 0 | `TargetInvalidCancelled` |
| `PostCompletionRelease` | 不再播离场句 | 不再播惩罚反应 | 0 | `Completed` |

## 7. 从当前实现到理想矩阵的缺口

### 7.1 P0 必补

| 缺口 | 当前问题 | 必须补的方向 |
|---|---|---|
| 中断接管原子化 | 玩家 `PlayerTyping` 跑开会卡字 | 引入稳定的 interrupt handoff，保证 partial text 先结算或清空再接离场句 |
| `LeaveCause` 缺失 | 现在只有“越界”和“一刀砍取消” | 引入显式离开原因枚举 |
| `Cancelled` 过粗 | UI / 正式对白 / 失活全都一样 | 拆 `SystemTakeover` 与 `TargetInvalid` |
| `E` 语义未完全收干净 | 仍会跳过 AutoContinue / 收尾等待 | 明确 `E` 只跳当前动效，是否允许催答要单独定义 |
| 离开 grace 缺失 | 轻微越界立刻算跑开 | 增加 grace timer 和回到范围内撤销 |

### 7.2 P1 应补

| 缺口 | 当前问题 | 建议方向 |
|---|---|---|
| 按 phase 分 reaction | 打断 NPC 说话和打断等待回复没区别 | reaction 读取时附带 `LeavePhase` |
| 按 cause 分文案包 | 现在所有离场文案都混在 `walkAwayReaction` | 扩成 `InterruptReactionSet` |
| cue / 表情独立层 | 当前 cue 仍是简化版文字气泡 | 把 cue、emoji、数值漂字从对白气泡里拆出去 |
| 续聊 continuity | 离开后再回来没有“刚才聊到哪儿”感 | 保留最近一次中断上下文 |

### 7.3 P2 体验深化

| 缺口 | 方向 |
|---|---|
| 玩家离开方式人格化 | 根据 NPC 人设和关系阶段生成更有角色性的离场句 |
| NPC 反应更外显 | cue、表情、停顿、转身、微动作 |
| 玩家 / NPC 双方打断逻辑统一 | 对话双方都能被自然打断，而不是只有 NPC 这边补缝 |

## 8. 具体落地方案

### 8.1 `PlayerNpcChatSessionService.cs`

后续应新增这些显式模型：

- `ConversationAbortCause`
- `LeaveCause`
- `LeavePhase`
- `ExitSequencePlan`
- `PendingInterruptSnapshot`

同时改造这几个关键点：

1. `Update()` 中不要直接“越界就立刻中断”。
2. 先进入 `LeaveProbe`。
3. 到达 grace 超时后再决定升级成：
   - `SoftExit`
   - `AbruptRunAway`
   - `NpcSpeakingInterrupted`
4. `StartWalkAwayInterrupt()` 前必须先做：
   - 当前打字协程稳定落点
   - partial bubble 清理 / 收束
   - 当前 phase snapshot
5. `CancelConversationImmediate()` 要拆成：
   - `CancelBySystemTakeover()`
   - `CancelByTargetInvalid()`
   - `CancelByServiceShutdown()`

### 8.2 `NPCDialogueContentProfile.cs`

当前的 `InformalChatInterruptReaction` 不够，后续应升级为：

- `InterruptReactionSet`
- 按 `LeaveCause`
- 按 `LeavePhase`
- 按 `RelationshipStage`

至少拆出：

- `softExitReaction`
- `abruptExitReaction`
- `npcSpeakingInterruptedReaction`
- `systemTakeoverReaction`

### 8.3 `NPCBubblePresenter.cs` 与 `PlayerThoughtBubblePresenter.cs`

后续要支持三个独立频道，而不是都往一个对白 bubble 塞：

- `DialogueLine`
- `ReactionCue`
- `EmotionIndicator`

这样才能避免：

- cue 抢掉正文
- 正文打断 cue
- 中断时旧 bubble 残留

### 8.4 `NPCInformalChatValidationMenu.cs`

后续验证不该只测：

- closure
- interrupt

还必须补成：

- `PlayerTyping` 时离开
- `NpcThinking` 时离开
- `NpcTyping` 时离开
- `BetweenTurns` 时离开
- `SystemTakeover` 时取消
- `TargetInvalid` 时取消

每条都要记：

- `LeaveCause`
- `LeavePhase`
- `EndReason`
- `PlayerExitSeen`
- `NpcCueSeen`
- `NpcReactionSeen`
- `BubbleStateCleaned`

## 10. 当前结论

我认同用户的判断：

- 当前代码不是“没有内容”。
- 当前代码的问题是“内容没有形成一套完备矩阵”。
- 尤其玩家离开这件事，虽然已经有 `WalkAwayInterrupt`，但它还远远不是一个成熟系统。

所以后续正确方向不是继续零碎修一句对白，而是严格按这份文档把：

- 当前真实实现

- 理想完整矩阵
- 明确施工顺序
