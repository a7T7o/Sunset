## 允许 scope

只允许处理关系页玩家面这一层，例如：

- 分区顺序
- 标题层级
- 副标题 / 标签位的视觉秩序
- 正文区块的留白与阅读顺序
- 哪一块更像“身份与位置”
- 哪一块更像“今日你会看到的样子”
- 哪一块更像“你为什么会记住他”

如果你需要微调现有关系页里这些承载区的布局、字号、间距、容器顺序，可以做。

## 明确禁止

不要碰：

- `NpcCharacterRegistry` 的人物真值
- `roleSummary / presenceNote` 的内容编写
- formal 对白
- `DialogueUI`
- `NPC_Hand`
- Prompt / Workbench / Town / Primary
- 广义 UI 壳扩题

不要把这轮偷换成“顺手把人物内容也一起改了”。

## 完成定义

你这轮做完时，关系页应当能稳定承载下面这 5 块，而且玩家第一眼顺序清楚：

1. 顶部识别区
   - 头像
   - 名字
   - 身份锚点 / 今日存在感
   - 小标签
2. 身份与位置区
3. 今日你会看到的样子
4. 你为什么会记住他
5. 后续预留区

并且要满足：

- 不拥挤
- 不像系统调试面
- 不像把几段字硬塞进卡片
- 第一眼能先认人，再认位置，再知道今天为什么会注意到他

## 结构线与体验线分开报

回执时请分开：

### 结构线

- 你把哪几个区块收出来了
- 页面顺序如何
- 哪些字段落在哪个区

### 体验线

- 玩家第一眼先看到什么
- 阅读顺序是否自然
- 还有哪些点仍然需要用户看体感

## 固定回执格式

1. 当前在改什么
2. changed_paths
3. 这轮新做成了什么
4. 还没做成什么
5. 当前最需要用户终验的点
6. 一句话摘要
