## 四、最新新问题：关于“NPC 移民 / 驻村部署”的思考

用户刚刚现场指出了一个很关键的问题：

- 这些新 NPC 看起来不像本来就在村里
- 而像是到某个阶段、对话或时机才突然蹦出来

我读完代码后，结论是：

这个判断是对的，而且这不是小观感问题，是一层更上面的部署契约问题。

## 1. 真正干这件事的文件是谁

核心文件不是我刚改的 crowd 对话资产，而是：

- `Assets/YYY_Scripts/Story/Managers/SpringDay1NpcCrowdDirector.cs`

## 2. 它现在到底在干什么

我已经确认的事实是：

1. 它当前只在 `Primary` 场景工作  
2. 非 `Primary` 时会 `TeardownAll()`，也就是把这批 crowd 全拆掉  
3. 在同步 crowd 时，如果 state 不存在，就会：
   - `Instantiate(entry.prefab, spawnPosition, Quaternion.identity)`
4. 所以它的当前契约更像是：
   - `Primary-only`
   - `runtime spawn`
   - `phase/beat 到点才生成或显隐`

而不是：

- “这批人本来就常驻在村里，只是导演层临时调他们”

## 3. 这意味着什么

这意味着用户现在看到的体验问题，根本上不是：

1. 台词不够好
2. 群像内容太空
3. 某个 crowd NPC 没写到位

而是：

1. 这批 NPC 的“存在方式”不对
2. 它们更像 runtime 临时演员，不像村里本来就活着的人

## 4. 这件事我当前该不该直接吞

按当前边界，我认为不该由我这条 `NPC 内容线` 默认直接吞。

原因不是我不想做，而是这件事已经越过了我上一刀被授权的范围：

1. 它会碰 `SpringDay1NpcCrowdDirector.cs`
2. 它已经进入 `Town runtime contract`
3. 如果真要改成“本来就在村里”，大概率还会碰：
   - `Primary / Town` 场景级常驻布置
   - crowd 常驻 root
   - 常驻 -> 导演接管的切换契约

而这些都不是我上一刀拿到的 own 范围。

## 六、我建议你现在怎么接这件事

我认为你现在最值钱的不是马上把锅甩回给 `NPC`，而是先在你这边做一个明确判断：

## 方案 A：继续保持 runtime spawn，但把“突然蹦出来”的体感修掉

这是偏保守、改动面更小的方案。

方向会是：

1. crowd actor 提前预热
2. 更早生成
3. 先 offscreen / 先 inactive / 先占位
4. 到 cue 时只做激活、让位、接管、路径和朝向变化

这样做的好处：

1. 不需要立刻把 crowd 全迁成场景常驻
2. 还能保留导演线当前的可控性
3. 但玩家感知到的“凭空蹦出来”会少很多

## 方案 B：正式改成“村里本来就有这批人”

这是更重、但也更贴近用户直觉的方案。

方向会是：

1. 这批 crowd 先常驻在 `Town/Primary` 的合适位置
2. 导演层只接管已有角色
3. 不再把“人有没有存在”交给 runtime instantiate 决定

这样做的代价是：

1. 要重新定义 crowd 常驻 root
2. 要重新定义 phase 切换时谁关谁开
3. 要重新定义常驻态和导演态之间的 ownership

## 我更建议的顺序

如果你想快速止血体验，我建议先走 `A`，把“突然蹦出来”的感受压掉。

如果后面你确认这批人确实就是要当“村里本来活着的人”，再逐步走 `B`。
