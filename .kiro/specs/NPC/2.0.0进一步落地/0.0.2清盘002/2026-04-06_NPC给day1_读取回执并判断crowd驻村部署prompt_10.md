4. 不要继续泛写新边界文档，不要写成新的空方案

你这轮真正要给出的，是一个清楚判断：

## 你要先在下面二选一，或者给出非常明确的过渡口径

### A. 继续 runtime spawn，但先把“突然蹦出来”的体感压掉

意思是：

1. crowd 仍由导演/runtime 管
2. 但改成更早生成 / 预热 / 预占位 / 只在 cue 时激活与接管

### B. 这批 crowd 后续要往“村里本来就有的人”去走

意思是：

1. 要开始思考常驻居民化
2. 导演层以后更多是接管已有 actor，而不是现场 instantiate

如果你当前认为既不能直接 A，也不能直接 B，那你必须给出：

- 一个非常明确的中间态口径  
- 说明下一刀到底先修哪一层，才能最快消掉“突然生成”的违和感

## 你这轮回复至少要讲清 6 件事

1. 你读完回执后，对 `NPC` 线当前完成度的判断  
2. 你是否接受“突然生成是 runtime deployment contract 问题，而不是内容层问题”  
3. 你当前更倾向 `A / B / 中间态` 哪条路  
4. 如果走你选的路，第一刀该落在哪个 exact file / exact contract  
5. 这件事你是否继续自己接，还是需要重新授权给 `NPC`  
6. 如果需要 `NPC` 接，你准备授权到哪一层
