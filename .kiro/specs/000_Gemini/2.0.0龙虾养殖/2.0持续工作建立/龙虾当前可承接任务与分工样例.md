# 龙虾当前可承接任务与分工样例

**创建日期**: 2026-03-10

---

## 1. 文档目的

这份文档用于回答两个很实际的问题：

1. 现在的龙虾，已经可以承接哪些 `Sunset` 任务？
2. 如果用 `直接 PM`、`main -> pm`、`pm -> reader/reviewer/builder` 这几种方式来跑，实际会长什么样？

这份文档不是未来设计图，而是当前阶段可以直接拿来试跑的样例集。

---

## 2. 当前可承接任务范围

当前阶段，龙虾优先承接下面 5 类任务。

### 2.1 工作区文档任务

例如：

- 补操作手册
- 补验收指南
- 补最小闭环说明
- 补阶段结论文档

### 2.2 工作区治理任务

例如：

- 判断当前主线是什么
- 识别本轮是主线还是支撑子任务
- 重新锚定目标
- 梳理后续推进顺序

### 2.3 记忆与恢复任务

例如：

- 回读当前工作区与父工作区 `memory.md`
- 补写恢复点
- 追加当前结论
- 保证下一轮不丢主线

### 2.4 验收与审视任务

例如：

- 输出“通过 / 部分通过 / 未通过”判断
- 补可勾选验收项
- 给出测试样例
- 给出风险与边界说明

### 2.5 低风险执行任务

例如：

- 修明确小问题
- 做限定范围验证
- 补一份局部报告
- 在明确规则下做小范围文档或代码调整

---

## 3. 分工原则

当前阶段的分工不是为了“显得像多 agent”，而是为了避免任务混乱。

### 3.1 `pm`

负责：

- 定义本轮任务目标
- 收束范围
- 指定交付物
- 指定验收标准
- 最后统一收口

### 3.2 `reader`

负责：

- 读工作区 `memory`
- 读 `steering`
- 读相关文档或代码
- 提炼事实与上下文

### 3.3 `reviewer`

负责：

- 判断风险
- 检查是否偏离主线
- 给出验证口径
- 指出哪些不能乱做

### 3.4 `builder`

负责：

- 执行最小必要动作
- 产出文档、修改、验证结果
- 把动作控制在本轮任务边界内

### 3.5 当前阶段的重要现实口径

> 这些角色当前首先是工作分工，不等于必须已经变成长期常驻、完全自治的独立 agent 集群。

换句话说，当前看的是结果是否专业，而不是形式上拉起了几个名字。

## 4. 测试样例一：`直接 PM`

### 适用任务

- 单一工作区
- 单一交付物
- 边界很清楚

### 推荐测试输入

> 请在 `2.0持续工作建立` 工作区补一份“最小运行闭环说明”，并告诉我现在怎么验收。

### 这条链路里龙虾要做什么

1. 定位到 `2.0持续工作建立`
2. 读取 `memory.md` 与相关规则
3. 产出闭环文档
4. 明确验收口径
5. 更新子工作区、父工作区、线程记忆
6. 回复“已完成什么、怎么验、下一步是什么”

### 预期产物

- `龙虾最小运行闭环.md`
- 相关 `memory` 追加记录

### 通过条件

- 文档存在
- 闭环流程清楚
- 验收口径清楚
- 记忆已写回

---

## 5. 测试样例二：`main -> pm`

### 适用任务

- 用户从总入口给出一个还没完全收束的 `Sunset` 任务
- 需要先定位工作区，再决定如何执行

### 推荐测试输入

> 请你看看 `Sunset` 里当前这条龙虾主线下一步最该做什么，并给我一个能直接验的交付。

### 这条链路里龙虾要做什么

1. `main` 接收任务
2. `pm` 判断当前主线仍然是 `2.0持续工作建立`
3. `pm` 收束出本轮交付物，例如“补最小闭环”或“补任务样例”
4. 执行对应动作
5. 返回明确产物，而不是只说分析过程

### 预期产物

- 一份新文档，或一份明确可验收的执行清单
- 对主线状态的清楚说明

### 通过条件

- 没有因为入口模糊就跑到别的项目
- 最终交付是具体产物
- 主线与恢复点清楚

---

## 6. 测试样例三：`pm -> reader/reviewer/builder`

### 适用任务

- 任务已经进入明确工作区
- 但需要先读、先审，再做

### 推荐测试输入

> 请在 `Sunset` 当前相关工作区里，挑一个明确且低风险的小任务，按读规则、做风险判断、执行、验证的方式完整跑一遍。

### 这条链路里龙虾要做什么

1. `pm` 明确任务边界与交付物
2. `reader` 读取 `memory`、`steering`、相关文件
3. `reviewer` 输出风险边界、验证方式、不能碰的地方
4. `builder` 执行最小修改或产出
5. `pm` 汇总结果并写回记忆

### 预期产物

- 一项真实的小交付
- 一段清晰的验证结果
- 一次完整的恢复点说明

### 通过条件

- 不是只读不做
- 不是只做不验
- 不是只验不写回

## 7. 测试样例四：飞书入口分派

### 当前阶段如何理解

飞书当前应被视为“任务入口形式”，而不是另一套独立工作逻辑。

也就是说，本质仍然应该是：

`飞书 -> main -> pm -> 执行闭环`

### 推荐测试输入

> 请回顾 `2.0持续工作建立` 当前状态，告诉我这轮已经完成了什么、还差什么，并给我一个下一步能直接验收的交付。

### 这条链路里龙虾要做什么

1. 从飞书入口接收任务
2. 把任务收束到 `Sunset` 当前主线
3. 返回明确结论与交付物
4. 不因为入口变化就改变闭环标准

### 当前阶段的诚实口径

当前阶段真正要验的，不是“飞书 UI 自动点击得多炫”，而是：

- 飞书来的任务有没有被正确收束
- 有没有进入同一套专业闭环
- 最终有没有交出可验收结果

---

## 8. 当前最推荐先跑的 3 个基础样例

如果现在就要开始验收，我建议先从下面 3 个开始：

1. `直接 PM` 跑一个文档型任务
2. `main -> pm` 跑一个主线收束任务
3. `pm -> reader/reviewer/builder` 跑一个低风险执行任务

原因是：

- 这 3 个最能验证龙虾是否已经像“工具”而不是“聊天壳”
- 它们都可以直接在 `Sunset` 当前主线下进行
- 不需要先把整个大系统都搭满

---

## 9. 当前阶段的验收重点

验收这几种样例时，不要先看“是否像多 agent”，而要先看下面 5 件事：

1. 是否始终围绕 `Sunset` 主线
2. 是否有明确交付物
3. 是否有验证结果
4. 是否写回记忆
5. 是否说清下一步恢复点

只要这 5 条成立，说明龙虾已经开始具备可持续承接任务的基础。

---

## 10. 一句话结论

> **现在最应该看的不是龙虾能不能把链路跑得更长，而是它能不能把一个 `Sunset` 任务按专业分工稳稳做完。**
