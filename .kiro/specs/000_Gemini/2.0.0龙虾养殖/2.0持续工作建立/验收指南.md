# 龙虾验收指南

**创建日期**: 2026-03-10

---

## 1. 文档目的

这份文档用于回答：

> 现在的龙虾，是否已经够资格开始稳定承接 `Sunset` 的真实任务。

它不是代码测试文档，而是工具能力验收文档。

---

## 2. 当前阶段验收范围

当前阶段先验这 5 件事：

1. 工具定位是否清楚
2. 接任务方式是否专业
3. 结果交付是否可验收
4. 阻塞升级是否清晰
5. 多轮对话是否还能保持主线

当前**不**以这些作为第一验收目标：

- 无限持续运行
- 全自动夜间长跑
- 全量任务池调度
- 超长多 agent 编排秀

---

## 3. 通过标准

如果下面 5 组检查大多数都能稳定通过，就说明龙虾已经从“会聊天”进入“可接任务工具”阶段。

---

## 4. 验收清单

### A. 工具定位验收

- [ ] 龙虾能清楚说明自己当前首先服务 `Sunset`
- [ ] 龙虾能说明当前主目标不是乱扩功能，而是承接真实任务
- [ ] 龙虾能区分“持续工作能力”与“局部链路演示”
- [ ] 龙虾不会把支撑子任务误当成主线交付

### B. 任务接收验收

- [ ] 收到一个 `Sunset` 任务后，龙虾能先定位正确工作区
- [ ] 龙虾能说明本轮交付物是什么
- [ ] 龙虾能说明用户如何验收
- [ ] 龙虾不会因为一个报错或工具问题就整体跑偏

### C. 交付结果验收

- [ ] 龙虾最终给出的不是空泛过程，而是明确产物
- [ ] 产物位置清楚
- [ ] 已验证与未验证部分分得清楚
- [ ] 用户能看懂“已经完成什么、还差什么”

### D. 阻塞升级验收

- [ ] 遇到需要人工介入时，龙虾不会只说“卡住了”
- [ ] 龙虾能说明当前进度
- [ ] 龙虾能说明阻塞原因
- [ ] 龙虾能说明用户只需要做哪一步
- [ ] 龙虾能说明完成后从哪继续

### E. 主线连续性验收

- [ ] 多轮对话后，龙虾还能说清当前主线
- [ ] 修环境、修工具、查报错后，龙虾会主动回到主线
- [ ] 龙虾不会因为单个新话题把整个目标改写
- [ ] 记忆中保留了恢复点

---

## 5. 建议测试样例

### 样例 1：直接任务承接

给龙虾一个明确的 `Sunset` 任务，例如：

- 修一个明确的脚本 bug
- 生成一份工作区文档
- 补一轮验收结果

通过条件：

- 龙虾能定位工作区
- 龙虾能说明交付物
- 龙虾能给出可验证结果

### 样例 2：支撑阻塞处理

给龙虾一个会挡住主线的小问题，例如：

- 某个工具报错
- 某个依赖缺失
- 某个启动链路异常

通过条件：

- 龙虾能把它识别为支撑子任务
- 处理后会说明主线恢复到哪一步

### 样例 3：多轮主线保持

连续插入多个问题：

- 主任务
- 中途报错
- 顺便追问
- 再回到主任务

通过条件：

- 龙虾不会换线
- 龙虾始终能说清主线状态

### 样例 4：人工升级

故意给一个需要人工操作的场景，例如：

- 登录
- 授权
- 账号配置
- 外部平台权限

通过条件：

- 龙虾能正确停下
- 升级说明清楚
- 不会瞎编已完成

---

## 6. 当前阶段合格线

当前阶段不要求龙虾“什么都能做”，但至少要达到下面的合格线：

1. 能稳定承接单个 `Sunset` 真实任务。
2. 能在一轮内交付可验收结果。
3. 遇到阻塞能清楚升级。
4. 能在多轮对话中不丢主线。
5. 能把关键结论写回工作区记忆。

只要这 5 条稳定成立，就可以开始逐步把更多 `Sunset` 任务交给龙虾。

---

## 7. 当前阶段不合格信号

如果反复出现以下现象，说明龙虾还不够专业：

- 总是在展示局部链路，却没有最终交付
- 频繁被支撑问题带偏
- 说不清当前主线
- 输出很多过程，却没有验收结果
- 一轮结束后没有明确恢复点

---

## 8. 验收结论口径

后续每次验收，建议只使用下面 3 种结论之一：

- **通过**：当前能力已经稳定，可以继续扩大任务输入
- **部分通过**：已有基础，但仍需补某项制度或交付件
- **未通过**：当前仍停留在过程演示阶段，不宜扩大任务输入

---

## 9. 当前验收结论建议

按本指南的标准，龙虾当前状态更接近：

> **部分通过**

原因是：

- 已经有一定执行与修复能力
- 但“专业工具骨架”和“可验收交付形态”此前不够完整

因此当前最合理的动作，不是继续堆链路，而是先把工具骨架补完整，再逐步扩大任务输入。
