# 龙虾最小运行闭环

**创建日期**: 2026-03-10

---

## 1. 文档目的

这份文档回答的是：

> 现在的龙虾，至少要怎样工作，才算在 `Sunset` 里形成了一个真正可用的最小闭环。

它不是“未来愿景”，也不是“无限运行蓝图”，而是当前阶段可以直接开始验收的工作闭环定义。

---

## 2. 一句话定义

> **龙虾最小运行闭环 = 接收一个真实 `Sunset` 任务，完成定位、执行、验证、记忆回写，并交付用户可直接验收的结果。**

只要这个闭环稳定成立，龙虾就开始具备“专业任务工具”的基本资格。

---

## 3. 当前阶段的边界

### 3.1 当前要做到的

- 稳定承接单个 `Sunset` 真实任务
- 不丢主线
- 交出明确产物
- 产物可验证
- 结果可回写、可恢复、可继续

### 3.2 当前还不作为主验收目标的

- `24x7` 无人值守长跑
- 超长多 agent 自动编排展示
- 一次性吞下大量任务池
- 在多个项目之间自由漂移
- 只靠入口链路跑通就算完成

---

## 4. 最小闭环的标准流程

当前阶段，龙虾每接一个任务，都至少要完成下面 8 步。

### 第 1 步：接收任务并判断类型

先判断本轮输入属于：

- 主线延续
- 支撑子任务
- 明确换线
- 纯澄清

这一步的目标不是分析得很花，而是防止一上来就跑偏。

### 第 2 步：定位工作区与规则入口

龙虾必须先回答：

- 当前任务属于哪个 `Sunset` 工作区
- 需要先读哪些 `steering`
- 这轮属于文档、治理、代码、验证、场景，还是评审型任务

如果用户给了明确路径，以用户路径为准。

### 第 3 步：明确本轮交付物与验收口径

正式执行前，龙虾至少要内部收束出这 3 件事：

- 本轮最终产物是什么
- 用户怎么验
- 若中途阻塞，恢复点在哪里

如果这一步说不清，后面大概率会出现“过程很多但没有结果”。

### 第 4 步：选择工作模式

当前阶段允许的最小工作模式有 3 类：

1. `直接 PM`
2. `main -> pm`
3. `pm -> reader/reviewer/builder`

注意：

> 这里的重点不是“必须有多少独立 agent”，而是任务是否被按职责拆清并形成闭环。

### 第 5 步：执行最小必要动作

只做与本轮交付直接相关的最小动作，例如：

- 读取必要文档
- 修改必要文件
- 生成报告或清单
- 修一个明确 bug
- 做一轮审计或验收

不允许因为“顺手”就把任务扩散成另一条线。

### 第 6 步：做验证

没有验证，不算完成。

验证形式视任务类型而定：

- 文档任务：内容是否齐全、结构是否清楚、路径是否正确
- 治理任务：是否与当前规则一致、是否能支撑后续执行
- 代码任务：是否通过编译、测试、日志检查
- 场景任务：是否完成审计、是否排除高风险引用问题、是否通过验证流程
- 报告任务：是否能直接用于下一轮执行或验收

### 第 7 步：写回记忆

写回顺序必须正确：

1. 当前子工作区 `memory.md`
2. 父工作区 `memory.md`
3. 当前线程 memory

每次至少写清：

- 当前主线目标
- 本轮做了什么
- 解决了什么阻塞
- 下一轮从哪继续

### 第 8 步：交付结果并给出恢复点

最终回复里至少要说清楚：

- 已完成什么
- 产物在哪
- 验到了什么
- 还差什么
- 下一步从哪继续

只有做到这一步，闭环才算真正收口。

---

## 5. 当前阶段可接受的 3 种工作模式

## 5.1 `直接 PM`

适用场景：

- 单个边界清楚的任务
- 不需要复杂分工
- 本轮重点是快速交付明确结果

典型任务：

- 生成一份工作区文档
- 补一轮验收清单
- 修一个明确的小问题
- 回顾当前主线并给出下一步

闭环要求：

- 直接完成接收、执行、验证、记忆、回复
- 不因为模式简单就省略验收和写回

## 5.2 `main -> pm`

适用场景：

- 用户从总入口给任务
- 任务需要先被正确路由到 `Sunset` 工作区
- 需要先收束任务目标，再决定是否继续拆分

当前阶段的真实含义：

- `main` 是入口
- `pm` 是任务收束与路由角色
- 重点在于“先把任务收准”，而不是“入口看起来很炫”

闭环要求：

- `main` 不能只转发一句模糊话
- `pm` 必须把工作区、交付物、验收方式说清

## 5.3 `pm -> reader/reviewer/builder`

适用场景：

- 任务已经进入一个明确工作区
- 需要读规则、做风险判断、再执行动作
- 任务复杂度已经超出单个动作，但仍希望保持收束

角色职责：

- `pm`：定义本轮目标、分工、收口标准
- `reader`：读规则、读 memory、提炼上下文
- `reviewer`：指出风险、边界、验证口径
- `builder`：执行最小必要修改或产出

当前阶段的重要说明：

> 这组角色当前首先是一套专业分工方法，而不强制要求已经变成长期常驻、独立自治的多 agent 编队。

也就是说，当前验收重点是“分工有效、闭环成立”，不是“同时拉起越多 agent 越好”。

## 6. 当前阶段龙虾应优先承接的任务类型

### 6.1 文档与治理任务

例如：

- 工作区定位文档
- 操作手册
- 验收指南
- 规则对齐说明
- 最小闭环说明

这是当前最适合优先跑稳的任务类型。

### 6.2 记忆与恢复任务

例如：

- 回顾当前工作区主线
- 整理恢复点
- 补写工作区 `memory.md`
- 追加父工作区摘要

这是让龙虾具备持续工作的底座能力。

### 6.3 验收与审视任务

例如：

- 对当前阶段做可验收总结
- 补充测试样例
- 输出“通过 / 部分通过 / 未通过”判断
- 生成风险说明或审视报告

### 6.4 低风险执行任务

例如：

- 文档修订
- 小范围配置调整
- 明确 bug 排查与修复
- 一轮限定范围验证

### 6.5 场景与资产相关任务

可以做，但必须更谨慎。

前提是：

- 先读 `scene-modification-rule.md`
- 先做审计
- 先说明风险边界
- 改后必须验证

---

## 7. 当前阶段不应默认承接的任务

下面这些事，不应在当前阶段默认直接开做：

- 没有明确工作区的大而泛目标
- 高风险的大规模场景批改
- 没有验收口径的“你先全做完”
- 需要账号登录、验证码、付费权限的外部平台操作
- 需要跨多个项目同时推进的任务
- 以“证明多 agent 很强”为唯一目的的链路展示

这些不是永远不能做，而是不能抢在最小闭环之前做。

---

## 8. 当前阶段的合格闭环长什么样

只要一次任务执行后，同时满足下面 5 条，就可以认定“形成了一次合格闭环”：

1. 已经定位到正确的 `Sunset` 工作区。
2. 已经交付用户可直接查看的明确产物。
3. 已经说明验证结果，而不是只说“我做了”。
4. 已经按顺序写回工作区与线程记忆。
5. 已经给出下一步恢复点。

如果缺其中任意两条以上，就仍然只能算“做了一些过程动作”。

---

## 9. 当前阶段可以怎么验收这份闭环定义

建议直接用 3 类任务试：

1. 文档型任务：例如补一份工作区说明或验收清单。
2. 治理型任务：例如回顾当前主线并重构执行口径。
3. 低风险执行型任务：例如修一个明确的小 bug 并做验证。

只要这 3 类任务都能稳定形成上面的 8 步闭环，龙虾就已经开始具备“可持续接任务”的基础。

---

## 10. 一句话结论

> **龙虾当前阶段不是要证明自己能跑多久，而是要证明自己能把一个 `Sunset` 真实任务稳稳做完。**
