# Sunset 像素美术 V1 工具路线裁定与 Aseprite 边界重写

## 当前主线

当前主线不是“继续幻想一个会自动做漂亮像素美术的总系统”，而是把 `美术生` 这条线收紧成一个**足够诚实、足够可决策的 V1 工具路线**：

- 先明确外部工具各自只补哪一层
- 先明确哪些现在不该装
- 先明确为什么 V1 第一批准批次应写死为 `Gold_Standard + ArtBatch + DNA extractor 输入/输出字段草案`

这条线的目标仍然是：**限制 AI 不乱来、降低基础资产生产与导入审核成本**，不是突然解决 Sunset 当前全部 UI 审美和场景叙事问题。

## 这轮实际做成了什么

这轮我做成了四件事：

1. 吸收了全局调研对 `Aseprite / pixel-mcp / aseprite-mcp / Figma / interface-design / Blender` 的最新收口，不再沿用“泛候选列表”口径。
2. 吸收了全局偏好里和这条线直接相关的约束：
   - UI 不要测试味
   - Workbench / Prompt 先守壳体秩序
   - 玩家可见证据必须来自玩家真实可见面
   - 场景先做可见构图，再补 invisible 语义
3. 把 V1 工具路线重写为更窄的裁定：
   - `Aseprite` 先定义为**像素源资产格式与后续增强项**
   - `pixel-mcp / aseprite-mcp` 先定义为**未来可选的局部执行器**
   - V1 主轴继续留在 **本地可控、无需外部安装的 DNA / ArtBatch / Review 链**
4. 把第一批准批次强制裁成单一答案，并显式纠偏旧信息污染：
   - 当前批次我选 `B`
   - 即：`Gold_Standard` 目录规范 + `ArtBatch` schema + DNA extractor 输入/输出字段草案
   - 同时把旧的 helper 失败说法收口为：该描述已过时，当前最新口径是 helper 可用于 `manual-equivalent`，不再按损坏不可用表述

## 现在还没做成什么

现在还没做成这些：

1. 还没有任何实现落地。
2. 还没有建立 `Gold_Standard` 基准素材库。
3. 还没有 `ArtBatch` manifest。
4. 还没有 Python DNA extractor 原型。
5. 还没有真实跑通一轮“参考图 -> DNA -> Review 资产 -> Unity 阅兵”的试跑。

所以当前只能说：**路线判断更稳了**，不能说“V1 已验证可用”。

## 当前阶段

当前阶段是：**立项前的工具路线裁定收口阶段**。

更具体地说：

- 第一轮架构骨架已经站住
- 这一轮把“该不该押宝外部工具”收紧到了可决策程度
- 还没进入实现阶段

因此现在属于：

- `结构 / checkpoint`：成立
- `targeted probe / 局部验证`：尚未开始
- `真实入口体验`：不成立

## 第一批次边界裁定

这轮我明确选 **B**：

**当前同一批准批次就是：`Gold_Standard` 目录规范 + `ArtBatch` schema + DNA extractor 输入/输出字段草案。**

这里要特别写死两件事：

1. 这仍然是文档与边界批次，不是实现批次。
2. 被纳入当前批次的是 **DNA extractor 输入/输出字段草案**，不是 Python DNA extractor 代码或原型实现。

## 外部工具 / skills / MCP 裁定表

| 对象 | 当前裁定 | 在 V1 的角色 | 为什么不是主路线 |
| --- | --- | --- | --- |
| `Aseprite` | 保留，先不当 V1 前置依赖 | 像素源资产格式、人工可编辑源文件、后续增强项 | 项目里有 `com.unity.2d.aseprite` 和 `.aseprite` 资源，不等于本机已有 `aseprite` CLI 生产链；当前本机 CLI 缺失 |

| `Blender` | 直接排除 | 无 | 对当前 2D 像素 UI / 场景问题帮助极弱，只会扩大工具面和复杂度 |

## 更收紧的 V1 工具路线裁定

## 1. Aseprite 在 Sunset V1 里到底是什么角色

我现在对 `Aseprite` 的正式口径收紧为：

- 它是**项目内已经存在的像素源资产生态的一部分**
- 它是**后续增强项的自然落点**
- 它**不是当前 V1 必须先打通的 CLI 自动化前提**

当前更诚实的说法是：

1. 项目里有 `com.unity.2d.aseprite` 和现成 `.aseprite` 资源，这说明 Sunset 对 `.aseprite` 资产并不陌生。
2. 但当前本机没有 `aseprite` CLI，所以不能把“项目支持 Aseprite”顺滑偷换成“现在已经能直接跑 Aseprite CLI 产线”。
3. 因此 V1 应按：
   - `PNG-first`
   - `.aseprite` 作为可接受参考源或后续增强项
   - 不把 CLI 集成作为第一刀阻塞

一句话说，`Aseprite` 在 V1 里是**可兼容的像素源格式与后续增强点**，不是当前必须先贯通的自动化主干。

## 2. `pixel-mcp / aseprite-mcp` 现在值不值得装

当前裁定：**现在不装。**

不是因为它们完全没价值，而是因为它们的价值层级和 `美术生` 当前主线不匹配。

它们能补的是：

- 小型像素资产
- 小图标
- 提示气泡
- 小边框
- 小面板素材

它们补不了的是：

- Workbench / Prompt 正式面的壳体秩序
- UI 的层级、留白、比例与正式感
- 场景构图、视觉重心与叙事组织
- “会动但丑、能摆但没叙事”的主问题

所以当前正确口径只能是：

- `pixel-mcp / aseprite-mcp` 未来可能成为**受限资产生产链的局部执行器**
- 它们不是 Sunset 的“审美总监”
- 它们不是当前 V1 的第一刀

现在不装的原因也很实际：

1. V1 还没把 `Gold_Standard -> DNA -> ArtBatch -> Review` 这条本地主干站住。
2. 过早安装外部执行器，只会把注意力从“先定义规则”转移到“先折腾工具接入”。
3. 当前最缺的不是一个外部像素生成接口，而是 Sunset 自己认可的参考资产、批次模型和审核入口。

## 3. 哪些外部工具只是思路参考，哪些可能成为后续局部执行器

### 3.1 只是思路参考

- `Figma 系`
- `interface-design`
- `Blender`

它们最多提供：

- 设计语义对齐
- 审稿方法论
- 反模板化提醒

但都不适合作为当前 V1 的生产底座。

### 3.2 真可能成为后续局部执行器

- `Aseprite`
- `pixel-mcp / aseprite-mcp`

但前提是：

1. `Gold_Standard` 已经建好
2. `ArtBatch` 已经定义好
3. DNA extractor 已经先跑通
4. Unity Review 链已经存在
5. 真实瓶颈已经证明卡在“小型像素资产生产效率”，而不是卡在审美判断缺位

在这些前提没成立前，它们都不该抢主轴。

## 4. 为什么第一批准批次我选 B，而不是把 DNA 字段草案延后

因为这三件东西一起解决的是**规则来源、事务边界和后续实现契约**，而不是工具幻觉。

### 4.1 `Gold_Standard`

解决“谁来定义 Sunset 的风格真值”。

没有它，任何外部工具都只能围绕模糊参考自由发挥。

### 4.2 `ArtBatch`

解决“这一批资产如何被追踪、摆场、审核、驳回和清理”。

没有它，Reject 只能退化成粗暴回滚或手工扫垃圾。

### 4.3 `DNA extractor 输入/输出字段草案`

解决“后续 DNA 提取器到底要吃什么、吐什么、如何和 `Gold_Standard` 与 `ArtBatch` 接口对齐”。

把它放在当前同一批次里，有三个好处：

1. 它仍然是边界设计，不是实现漂移。
2. 它能避免出现“`ArtBatch` 先写了，但 DNA 元数据到底长什么样还在下一轮摇摆”的断层。
3. 它能让后续真正进入实现前，输入、输出和批次关系已经有统一契约。

所以当前同一批准批次应明确写成：

1. `Gold_Standard` 目录规范
2. `ArtBatch` schema
3. DNA extractor 输入/输出字段草案

而 **Python DNA extractor 原型实现** 明确延后到下一刀，而不是混在这批里。

## 5. `Artist_Temp.unity` 还是新建 `Staging_Art_Review.unity`

当前我继续维持原判断：**V1 优先复用 `Artist_Temp.unity`。**

原因很简单：

1. 这轮主线是收紧路线，不是扩张场景资产。
2. V1 先验证“能否低成本阅兵”，不是先追求一套更漂亮的专用审核场。
3. 在玩家可见证据必须对准真实画面的偏好下，先用现有场景跑通 Review 面，比继续新增验证场更稳。

只有在下面任一条件成立时，才值得新建 `Staging_Art_Review.unity`：

- `Artist_Temp.unity` 已经承载了别的高耦合用途
- 阅兵布局、批次隔离或 UI 预览形态无法在现有场景里低风险实现
- Review 资产开始需要独立的长期维护场

在这之前，V1 不该先增场。

## 6. 这条线能解决什么，明确解决不了什么

### 6.1 能解决的

这条线能解决：

1. AI 在基础像素资产上乱色、脏色、模糊、乱比例的问题
2. 基础 UI 底板、九宫格、建筑白盒和占位图的生产边界问题
3. Unity 导入规范容易失守的问题
4. 审核入口分散、Reject 成本高的问题

### 6.2 明确解决不了的

这条线解决不了：

1. Workbench / Prompt 正式面的整体审美定稿
2. UI 壳体秩序、布局节奏、层级与留白是否高级
3. Scene 的叙事组织、视觉焦点和语义顺序
4. “会动但丑、能摆但没叙事”这类更高层产品问题

所以它是**基础资产自动化线**，不是“美术总监自动化线”。

## 下一步只做什么

如果继续推进，我下一步只做这一批：

**`Gold_Standard` 目录规范 + `ArtBatch` schema + DNA extractor 输入/输出字段草案。**

原因是：

- 它们定义的是 Sunset 自己的规则来源和批次边界
- 这是所有后续自动化与实现契约的地基
- 也是最不容易因为外部工具变动而返工的部分

## 需要用户现在做什么

你现在只需要决定是否批准这一完全相同的第一批次：

**`Gold_Standard` 目录规范 + `ArtBatch` schema + DNA extractor 输入/输出字段草案。**

这轮**不需要**你去装工具，也**不需要**你先挑外部 MCP。

## 自评

这轮我给自己 `8.5/10`。

比上一轮更好的地方是：

- 把“外部工具只是局部执行器，不是审美解决方案”讲硬了
- 把 `Aseprite` 从“项目已有资源基础”进一步收紧成“当前不是 CLI 主干”
- 把 V1 主轴重新压回本地可控路线

我最不满意的地方是：

- 这轮仍然停在路线裁定层，还没有真实原型证据
- `Aseprite` 后续是否值得进入第二阶段，仍要等 V1 主干跑通后再判断

## 最可能看错的点

我这轮最可能看错的点有两个：

1. 如果项目里其实很快就会出现稳定可用的 `Aseprite` CLI 环境，那么我当前对它的“后续增强项”定位可能会偏保守。
2. `Artist_Temp.unity` 是否真的足够承载后续阅兵场，仍需要等第一轮 Review 布局时再验证。

但即便这两点后续需要调整，也不会改变当前主结论：

**V1 不该把希望押在外部工具安装上，而该先把 Sunset 自己的规则、批次和审核主干站住。**
