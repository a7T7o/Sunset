# Sunset 像素美术“暴君大闸”架构评估

## 1. 我对需求的理解

你要的不是“再给 AI 一个画图按钮”，而是把 AI 的美术权限收窄成：

1. 必须先从人类认可的参考资产里提取规则
2. 生成时不能越过调色板、透明度、尺寸和网格边界
3. Unity 导入后必须自动变成像素游戏可接受的配置
4. 最后只给人看一个可快速拍板的阅兵面

换句话说，这不是“让 AI 当美术总监”，而是“把 AI 变成一个只会在铁轨上运货的资产工人”。

这个方向我认可，而且和 Sunset 当前“AI 不该冒充已经懂审美”的偏好是一致的。

## 2. 可行度判断

### 2.1 整体判断

- 作为 **基础资产生产管线**：可行度高
- 作为 **终版美术自动生成系统**：可行度低到中

我给这条线的当前判断是：

| 子阶段 | 可行度 | 说明 |
| --- | --- | --- |
| DNA 提取 | 高 | 参考图读色板、尺寸、透明度规则都能稳定落地 |
| 受限生成 | 中 | 只做九宫格底板、白盒建筑、占位底图是现实的；做终版美术不现实 |
| Unity 合规导入 | 高 | 当前项目已有现成 Point / PPU / Uncompressed 工具土壤 |
| 阅兵与人工审核 | 高 | 项目里已有可复用的验证场景 |

### 2.2 我不愿意虚报的边界

当前不能把下面这些话当成已经成立：

- “只要接上这套管线，AI 就能稳定做出好看的 Sunset 像素美术”
- “只要有色卡，风格就统一了”
- “只要自动 9-slice，一切 UI 框都会自然高级”

原因很直接：色板和尺寸只能锁住“不要太脏、不要太糊、不要太错位”，锁不住真正的审美判断。

## 4. 当前真正缺的东西

### 4.1 缺的是 Gold Standard，不是缺一个模型

你设计里最关键的前提，其实现在还不存在：

- `Assets/Art_Reference/Gold_Standard`

当前资产里当然有 tile palette、素材包、`.aseprite` 源文件，但它们不等于“人类已经批准过的风格 DNA 库”。

如果没有这层筛选，DNA 提取器抽出来的东西只会是“项目里某批资源的统计结果”，而不是“Sunset 应该长成这样”的规则。

### 4.2 缺的是批次事务模型

你的设计里说“驳回就 `git reset --hard` 并清除该批次文件”。这在概念上很爽，但在 Sunset 里不够稳。

我反对直接把它写成默认方案，原因有三条：

1. 它太粗，会误伤同工作树里别的未提交改动。
2. 它和当前项目的 Git 安全口径冲突。
3. 真正需要回滚的不是整个工作树，而是“这一个 art batch 的输出集合”。

更好的模型是：

- 独立 art worktree
- `ArtBatch/<batch_id>/manifest.json`
- 所有生成资产都挂到该 batch
- 审核拒绝时按 manifest 定向删
- 如果这个 worktree 就是一次性试验台，直接废弃整个 worktree

### 4.3 缺的是真实生成边界

如果“受限生成”这一步理解成“给模型一句 prompt，生成一个新 UI 框或新房子”，那这条线会很快重新失控。

V1 真正应该做的是：

- 九宫格底板
- 边框/角块模板拼装
- 建筑/Tile 白盒底图
- 轮廓剪影
- 网格占位图

也就是“先把比例、栅格、边框、色板和占位做对”，而不是直接做 hero art。

## 5. 我建议的 V1 工程方案

## 5.1 语言选择

我倾向于：

- **DNA 提取器**：Python
- **Unity 侧**：C# Editor 工具 + AssetPostprocessor
- **批次调度**：Python CLI 驱动 + JSON manifest

理由：

1. 本机已有 [python.exe](/D:/Unity/Unity_learning/Sunset_worktrees/scene-build-5.0.0-001) 可用，版本是 3.11.9。
2. 图像处理、颜色聚类、透明度扫描、边框分析这类工作，Python 写起来更稳更短。
3. Node 虽然也能做，但这类“像素层约束 + 批处理图像分析”我更信任 Python 工具链。

### 5.1.1 依赖判断

当前本机：

- Python 可用
- Node 可用
- `aseprite` CLI 不可用
- `Pillow` 还没装

所以我建议的 V1 是：

1. 先做 `PNG-first`
2. 用 Python + `Pillow`
3. 把 `.aseprite` 支持做成可选增强，不作为第一阶段前提

## 5.2 DNA 提取器应该提什么

我不同意只提：

- 16-32 色色卡
- 宽高
- 9-slice 边界

因为这还不够。

我建议 DNA JSON 至少包含：

1. `palette`
   - 经过压缩后的允许颜色列表
2. `alpha_policy`
   - 允许透明度集合，V1 直接收成 `{0,255}`
3. `ppu`
   - 当前参考图的像素单位
4. `grid_size`
   - 16x16 / 32x32 等
5. `border_candidates`
   - 9-slice 候选，不是最终真理
6. `ramp_groups`
   - 明暗色阶分组，帮助后续限制生成时的高光/阴影
7. `component_stats`
   - 轮廓、边框厚度、内填充厚度等简单几何统计
8. `source_assets`
   - DNA 来源，方便追责和复验

## 5.3 生成器不该“画”，而该“拼”

V1 里我最想压住的一件事是：**不要把生成器设计成会自由作画的系统。**

更好的结构是：

### UI 路线

- 从参考图里切：
  - corner
  - edge
  - fill
  - ornament candidate
- 用 JSON spec 组装出：
  - 基础框
  - 按钮底板
  - 面板底板

### 建筑 / Tile 路线

- 只生成：
  - 网格占位底图
  - 单色或双色 silhouette
  - 简化块面
- 保证：
  - 尺寸是网格整数倍
  - 颜色只来自 palette
  - 透明度只有 0/255

也就是说，V1 不是 text-to-art，而是 reference-to-constrained-template。

## 5.4 Unity 侧怎么串

我建议新增一个小型目录规范：

```text
Assets/
├─ Art_Reference/
│  └─ Gold_Standard/
├─ ArtPipeline/
│  ├─ Review/
│  ├─ Approved/
│  └─ Metadata/
```

同时定义 `ArtBatch`：

```json
{
  "batchId": "20260329-ui-frame-001",
  "kind": "ui-frame",
  "dna": "Assets/ArtPipeline/Metadata/.../dna.json",
  "outputs": [
    "Assets/ArtPipeline/Review/20260329-ui-frame-001/frame_a.png"
  ],
  "importPolicy": {
    "textureType": "Sprite",
    "spriteMode": "Single",
    "ppu": 16,
    "filterMode": "Point",
    "compression": "None",
    "alphaIsTransparency": true
  },
  "reviewScene": "Assets/000_Scenes/Artist_Temp.unity",
  "status": "PendingReview"
}
```

然后：

1. Python 生成输出
2. 写 batch manifest
3. Unity AssetPostprocessor 按路径或 manifest 套导入规则
4. Editor 工具把资产摆进 staging scene
5. 人工拍板

## 5.5 Unity 导入器怎么做

这部分可行度最高。

我建议不是做一个“全局乱扫”的 AssetPostprocessor，而是：

- 只盯 `Assets/ArtPipeline/Review/**`
- 或者只盯带 batch manifest 的输出

这样可以避免误伤项目里别的贴图。

Importer 应强制：

- `TextureImporterType.Sprite`
- `SpriteImportMode.Single/Multiple`
- `spritePixelsPerUnit = project standard`
- `FilterMode.Point`
- `TextureImporterCompression.Uncompressed`

如果是 UI 单图，还应读 batch manifest 里的 `border_candidates` 或人工确认值回写 border。

## 5.6 阅兵和阻断怎么做

这里我赞同你的方向，但要改实现。

### 我赞同的部分

- 必须有独立阅兵场
- 必须有批次等待人拍板
- 通过前不进入主资产区

### 我不同意的部分

- 不要默认 `git reset --hard`

### 我建议的实现

1. `Review` 资产只进：
   - `Assets/ArtPipeline/Review/<batch_id>/`
2. 阅兵场只引用 `Review` 资产
3. 人工点 Reject：
   - 删除该 batch 目录
   - 清 staging scene 的该 batch 实例
   - 更新 manifest 为 `Rejected`
4. 人工点 Approve：
   - Promote 到 `Approved`
   - 写批准记录
   - 允许后续人工再精修

如果整条线放在独立 art worktree 里，那么 Reject 的更强版本可以是：

- 直接废弃这个 worktree

这比在混合现场里做 `reset --hard` 安全得多。

## 6. 我会怎么分三期

## Phase 1

目标：先让“不会脏、不会糊、不会乱色”成立

- 建 `Gold_Standard`
- 做 DNA 提取器
- 做 palette JSON
- 做 Unity 导入强校验
- 用 `Artist_Temp.unity` 做阅兵场

## Phase 2

目标：先做出能用的基础资产，而不是好看的成品美术

- UI 九宫格底板模板生成
- 建筑白盒底图生成
- batch manifest
- 批次审核与清理

## Phase 3

目标：才去碰“受限生成更复杂图案”

- 边框纹样候选
- 简单 ornament 拼装
- 更多 DNA 特征
- 更细的 rejection taxonomy

## 7. 最关键的意见

如果你让我只说一句最重要的话，那就是：

**这个方案的价值，不在于“自动画得多好”，而在于它能把 AI 的错误自由度压缩到人能承受的范围。**

所以我支持你继续做，但我会强烈建议你把 V1 的完成定义改成：

- 生成基础资产
- 锁死像素规范
- 审核成本极低
- 拒绝成本极低

而不是：

- 直接自动产出真正漂亮的终版像素美术
