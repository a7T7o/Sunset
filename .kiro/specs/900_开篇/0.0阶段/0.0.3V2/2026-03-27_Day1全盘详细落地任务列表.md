## 1. 使用规则

这份列表不是宽泛规划，而是 Day1 线后续真正执行时的标准对照表。

后续任何“开始做 Day1 下一刀”的动作，都必须先回答 5 件事：

1. 当前做的是哪一个任务编号
2. 该任务当前状态是什么
3. 它的前置条件是否已经满足
4. 它允许碰哪些文件、明确不能碰哪些文件
5. 完成后要交什么证据

如果上面 5 件事答不清，就不应宣称“已经进入下一刀”。

## 2. 状态字段定义

- `pending`
  - 还没开始做，但理论上可以推进
- `in-progress`
  - 已经进入施工
- `blocked-hotfile`
  - 被 `Primary.unity`、`GameInputManager.cs`、同根 mixed dirty 或其他热面阻塞
- `blocked-external`
  - 被外部 owner、联合依赖、Unity 会话、字体 owner 或非本线 blocker 阻塞
- `done`
  - 已完成并有明确证据
- `parked`
  - 当前故意不做，等待更高优先级闭环

## 3. 任务类型定义

- `可直接施工`
  - 当前 Day1 线自己就能推进
- `联合前置后施工`
  - 需要先拿到 scene 写窗口、联合准入或外部 blocker 解除
- `文档/资产先行`
  - 先整理资产、配置、计划或验收材料，后续再进入实现
- `外部阻塞项`
  - 不应由当前 Day1 局部刀口单独吞并，但必须列入总账

## 4. 总体验收口径

只有当以下 6 个层面都完成，才可以称为“Day1 当前版本真正可验收”：

1. `G0` 施工闸门完成
2. `P0` 前半链 + 工作台 / 任务体验纵切完成
3. `P1` 后半链整日闭环完成
4. `P2` 开场与返村氛围完成
5. `P3` 必要通用系统与 formalization 完成
6. `P4` 验收、回写、尾账清扫与快照完成

如果其中任何一层仍处于 `pending / blocked-* / parked`，都不能把 Day1 说成“只差一点点就做完”。
