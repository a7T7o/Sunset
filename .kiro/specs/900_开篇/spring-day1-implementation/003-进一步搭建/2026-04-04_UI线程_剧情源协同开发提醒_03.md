# UI 线程剧情源协同开发提醒

## 1. 这份提醒解决什么问题

这份文档不是让 `UI` 线程替 `spring-day1` 接剧情 owner。

它只解决一件事：

把 `spring-day1` 接下来会改到、并且一定会影响玩家面显示结果的 **剧情源变化面** 说清楚，让 `UI` 在线继续做任务栏、对话框、工作台正式面、悬浮框和提示卡时，不会误把当前字串和当前节奏当成永久真值。

---

## 2. 当前真实分工

### 2.1 `spring-day1` own

只负责：

1. `CrashAndMeet / EnterVillage` 的剧情扩充
2. 正式对白资产
3. 阶段推进语义
4. 谁什么时候出场、谁只是背景层
5. 任务/提示的剧情源文本

### 2.2 `UI / SpringUI` own

只负责：

1. 玩家屏幕上真正看到什么
2. 任务栏、正规对话框、继续提示、工作台大面板、制作悬浮框、左下角提示卡
3. 所有排版、自适应、滚动、动画、弹性、静止锚定、显隐切换
4. 工作台交互的玩家面体验闭环
5. 缺字、空栏、重叠、漂移、提示不对、层级不对这些玩家可见问题

---

## 3. 为什么 `spring-day1` 仍然会影响 `UI`

虽然这轮我不碰 `UI` 文件本体，但我接下来会补的是 **早期剧情源**。

这会直接影响 `UI` 线程当前正在处理的 5 类东西：

1. 开场任务栏的文案长度
2. 村长前两段正式对话的节点数、字数和节奏
3. `Prompt / Focus / Progress / Footer` 这些源文本
4. `CrashAndMeet / EnterVillage` 里同一个阶段内部的子步数
5. “谁此刻该说话、谁此刻只做背景”的剧情语义

所以正确口径不是“我不碰 UI 就完全不影响 UI”，而是：

1. 我不碰 UI 的文件和视觉 owner
2. 但我会改变 UI 所读取的剧情源内容
3. 因此必须先把变化边界告诉 UI

---

## 4. 接下来一定会变化的源头

下面这些内容，`UI` 线程必须假设 **还会继续变化**：

### 4.1 开场任务与提示文案

当前开场不再会长期停留在“首段对话 -> 简短后续 -> 直接疗伤”的旧压缩版。

`spring-day1` 接下来会把前半段补回到：

1. 醒来
2. 语言错位
3. 危险逼近
4. 跟村长撤离
5. 进村围观
6. 闲置小屋安置

这意味着：

1. 开场任务卡会更长
2. 任务条目可能变多
3. `Focus / Progress / Footer` 的语义会更细

### 4.2 村长前两段正式对话

当前村长相关的早期对话，后续会增加：

1. 醒来时的错位感
2. 怪物逼近时的撤离指令
3. 进村时的围观语义
4. 到闲置小屋后的安置说明

这意味着：

1. 对话框正文长度会变
2. continue 提示出现次数和时机会更密
3. 不能按“当前只有两三句”的短链去写死布局

### 4.3 Prompt / 任务源的字数和节奏

后续我会改的是 **剧情源语义**，不是只换同长度的字。

所以 `UI` 线程必须默认：

1. 当前短句以后可能变长
2. 当前一拍以后可能拆成两拍
3. 当前一条任务以后可能扩成“主目标 + 小目标”

### 4.4 背景 NPC 的出现语义

后续进村阶段会更强调：

1. 围观村民
2. 小孩视线
3. 闲置小屋被安置

因此某些提示和文案的归属，会更偏“背景围观/氛围”，而不是所有内容都压给村长一个人。

## 6. UI 线程这轮应该按什么思路兼容

### 6.1 先做“稳结构”，不要围当前文案打死

优先把下面几件事做成对源文本长度更稳的结构：

1. 开场任务栏
2. 正规对话正文与 continue 提示
3. Workbench 左栏/右栏/制作条/悬浮框
4. 左下角提示卡

不要假设：

1. 开场只会有一小句
2. 村长前两段只会短短几句
3. `Focus / Progress` 永远只有现在这点长度

### 6.2 如果发现是“缺状态”，就精确回报缺什么

如果 UI 这边发现真正缺的是剧情状态，而不是排版能力，请只精确回报：

1. 缺哪个字段
2. 缺哪个状态
3. 缺哪个事件
4. 缺哪个阶段标签

不要自己吞：

1. `SpringDay1Director` owner
2. 正式对白资产 owner
3. NPC 出场时机 owner

### 6.3 如果只是玩家面问题，就继续算 UI own

下面这些问题继续都算 `UI` own，不要反抛给剧情线程：

1. 文字不显示
2. continue 提示为空
3. 任务栏超出边界
4. 左栏可点但无字
5. 正式面对不齐
6. 悬浮框排版不顺
7. 漂移、抖动、重叠

---

## 7. `UI` 最该重点兼容的 4 个变化点

### 7.1 开场任务会更像“分步推进”

因为我会把前半段扩回：

1. 醒来
2. 危险
3. 撤离
4. 进村
5. 安置

所以任务栏不该再默认只吃“当前这一个很短的教学句”。

### 7.2 村长正规对话会更长

continue 提示、正文高度、页内留白，都要准备承受更长的前半段。

### 7.3 Prompt / Focus / Progress 会换更准确的语义

如果 UI 现在还是围绕旧的短语句硬调高度，后面会再次炸版。

### 7.4 Workbench 不会由剧情线程背锅

这轮我不会回改 Workbench 正式面。

所以 `Workbench` 的：

1. 左栏空白
2. 进度条语义
3. 追加/中断/领取排版
4. 悬浮框布局
5. toggle 与静止锚定

仍然全部继续归 `UI` own。

## 9. 如果剧情源合同真的变化，怎么同步

后续如果我不仅改了字，还改了 `UI` 正在读取的稳定合同，我必须同轮补 2 件事：

1. 更新这份“剧情源协同开发提醒”
2. 在 `spring-day1` 的任务单或 memory 里明确写出：
   - 哪个合同变了
   - 为什么变
   - `UI` 应该怎样兼容

也就是说：

不能只改代码，不补协同文档。

---

## 10. 一句话结论

`UI` 这轮不是和一个静止的剧情源并行开发，而是在和一个即将补回前半段原案的剧情源并行开发。

所以正确做法不是“先按今天这几个字把 UI 调死”，而是“把 UI 做成能承受这条剧情源继续扩充的正式玩家面”。
