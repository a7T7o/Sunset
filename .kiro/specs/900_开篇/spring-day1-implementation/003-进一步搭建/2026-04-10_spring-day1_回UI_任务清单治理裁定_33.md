# `spring-day1` 给 UI 的任务清单治理裁定

## 1. 我的裁定结论

我这边的最终裁定是：

- `任务清单 / SpringDay1PromptOverlay` **不应该继续保留现在这种“独立 runtime overlay 自治模型”**。
- 最终目标应当是：
  - **A. 像 toolbar / state 一样收回固定 HUD 治理**
  - `PromptOverlay` 只保留“任务清单显示壳”职责
  - 不再同时兼任：
    - runtime parent 选择器
    - modal/page/dialogue 抑制 owner
    - 手动 bridge prompt 容器
    - formal task card 与 manual prompt 的混合状态机

换句话说：

- `任务清单` 不是一个“会自己找爹、自己决定何时让位、自己兼容多种内容源”的自治 overlay。
- 它最终应该是 **day1 语义真源驱动下的固定 HUD 组件**。

## 2. 我为什么这么判

因为从 `day1` owner 角度，**正式任务清单的语义真值其实已经在 `SpringDay1Director` 里了**，不是 UI 线程缺这层真值：

- [SpringDay1Director.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/Managers/SpringDay1Director.cs#L660)
  - 已经给了 `GetTaskListVisibilitySemanticState()`
  - 明确区分：
    - `managedByDay1`
    - `forceHidden`
    - `allowRestore`
    - `semanticKey`
- [SpringDay1Director.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/Managers/SpringDay1Director.cs#L889)
  - 已经给了 `BuildPromptCardModel()`
  - 也就是正式任务页该显示什么内容，day1 已经能给

但 `PromptOverlay` 当前仍然没有真正收成“被动显示壳”，它还在自己做太多事：

- [SpringDay1PromptOverlay.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/UI/SpringDay1PromptOverlay.cs#L234)
  - 仍然保留 `Show(string text)` / `Hide()`
  - 这代表它继续承接大量手动 bridge prompt
- [SpringDay1PromptOverlay.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/UI/SpringDay1PromptOverlay.cs#L1504)
  - `BuildCurrentViewState()` 里会把 `manualPromptText` 混进正式任务卡
  - 本质上是 **formal task card** 和 **manual prompt** 还在同一个组件里混跑
- [SpringDay1PromptOverlay.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/UI/SpringDay1PromptOverlay.cs#L314)
  - 还在自己接 `SetExternalVisibilityBlock(bool blocked)`
  - 说明 page/modal suppress 责任还没完全收回统一真链
- [SpringDay1PromptOverlay.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/UI/SpringDay1PromptOverlay.cs#L2865)
  - 还在自己 `ResolveParent()`
  - 还在自己找 UI 根和 runtime parent

所以 packaged live 里“任务清单还是不像固定 HUD 那样稳定”，**第一真实 blocker 不是排序值不够，也不是再调一次透明度**，而是：

- **同一个组件还在混着两套 owner**
  - `day1 formal task-list truth`
  - `manual bridge prompt / overlay runtime autonomy`

只要这层没拆开，UI 再继续修 `sorting order / fade / page shell`，也还是会返工。

## 3. 这问题该谁主刀

我的裁定是：

- **现在必须由 `spring-day1` 接回主刀**
- UI 线程这轮不该继续单独主刀把它修到底

原因不是 UI 线程之前方向全错，而是：

- UI 线程已经把“它不是普通 HUD 小卡片，而是治理模型混乱”这件事看出来了
- 再往下，已经不是纯 UI 壳问题
- 下一刀必须先由 `day1 owner` 把内容源和显示职责拆开

也就是说：

- UI 线程之前那刀“固定 sorting / 独立 overlay”不是白做
- 但它已经到边界了
- 再往前走，就会踩进 `day1` 自己的语义责任面

## 4. 下一刀唯一切片

下一安全切片只建议砍这一刀：

- **由 `spring-day1` 把 `SpringDay1PromptOverlay` 从“混合内容 owner”收成“正式任务页显示壳”**

这刀的唯一目标：

- 正式任务清单只再吃两类 day1 真值：
  - `BuildPromptCardModel()`
  - `GetTaskListVisibilitySemanticState()`

这刀明确要做的事：

- 停止把 `SpringDay1PromptOverlay.Instance.Show(...) / Hide()` 当成 day1 各阶段通用桥接提示的主通道
- 停止让 `manualPromptText` 继续覆盖或混入正式任务页内容
- 让“正式任务清单”与“临时剧情桥接提示”不再共用同一状态机

这刀明确不要做的事：

- 不扩到 Workbench
- 不扩到 Placement
- 不扩到 Status
- 不扩到 Toolbar
- 不扩到 NPC/Town 泛 UI
- 不顺手做第二轮视觉 polish

如果这刀做完后 packaged live 仍有纯显示异常，那时 UI 再回来只收“显示壳”才是对的。

## 5. UI 线程要不要继续碰 `SpringDay1PromptOverlay.cs`

我的结论是：

- **这轮不要继续碰**

更准确地说：

- UI 线程现在应该先停在 `PromptOverlay` 当前壳层，不再继续追加治理性修补
- 等 `spring-day1` 先把“formal task card vs manual prompt”拆开
- 拆完后，如果还存在纯 HUD 表现问题，UI 线程再回到：
  - [SpringDay1PromptOverlay.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/UI/SpringDay1PromptOverlay.cs)
  - 只做固定 HUD 壳的最后一刀

当前 UI 线程此刻最正确的动作不是继续补，而是：

- **冻结对 `SpringDay1PromptOverlay.cs` 的进一步治理性修改**
- 等 `day1` 提供新的、单一 owner 的消费口径

## 一句话收口

这不是“UI 再补一刀就行”的问题。  
当前 packaged 仍异常，根因是 **`PromptOverlay` 还没从“自治 overlay”退回“day1 真语义驱动的固定 HUD 组件”**。  
所以下一刀应由 `spring-day1` 主刀，UI 线程本轮先不要再继续碰 `SpringDay1PromptOverlay.cs`。
