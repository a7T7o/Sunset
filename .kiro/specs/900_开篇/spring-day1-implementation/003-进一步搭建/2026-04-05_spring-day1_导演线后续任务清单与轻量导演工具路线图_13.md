## B 组：轻量导演工具 MVP

这部分不是做“完整导演编辑器”，而是做最值钱的最小能力。

### B1. 剧情 beat 数据化

先把最容易被写死的内容，从代码/脑内约定中抽出来：

1. beat ID
2. 当前所属 phase
3. 当前所属分场
4. 参与 NPC 列表
5. 每个 NPC 的：
   - 初始站位
   - 朝向
   - 目标点
   - 是否移动
   - 停顿时长
   - 看向目标

### B2. 单 NPC 手动排练记录能力

最小版只做：

1. 选一个 NPC
2. 进入排练模式
3. 用 `WASD` 一样的方式移动它
4. 记录位置与朝向
5. 保存为当前 beat 的路径点

注意：

- 一开始不做多 NPC 同步录制
- 一开始不做时间线编辑器
- 一开始不做复杂事件轨道

### B3. 路径点/站位点回放

录完之后必须能做最小回放，不然工具没有价值。

最小回放能力包括：

1. NPC 从起点移动到记录点
2. 按记录的朝向转身
3. 到点后停住
4. 保持看向目标或看向玩家

### B4. 剧情段挂接

这个 MVP 必须能接入 Day1 自己，而不是一个独立玩具。

最小接入方式应该是：

1. 每个 beat 可引用一组站位/路径数据
2. 当前 phase 进入该 beat 时可加载
3. 当前 beat 结束时可释放或切换

### B5. 编辑器内可调，而不是只能改配置文件

最小工具一定要做到“你看得见自己在调什么”。

所以至少要有：

1. 选择当前 phase / 分场 / beat
2. 选择当前要排练的 NPC
3. 记录当前位置/朝向
4. 保存
5. 回放

## 四、明确不做清单

这是止损线，后面每轮都要拿来对照。

## 1. 不做完整版导演编辑器

当前不做：

1. 多 NPC 同时录制
2. 可视时间线轨道
3. 全量撤销/重录系统
4. 复杂事件曲线
5. 完整导演 UI 套件
6. 像 Timeline/DCC 那样的豪华控制台

## 2. 不做 runtime 全自动智能编排

当前不做：

1. 自动避障智能重算
2. 自动根据场地变化重生成群像站位
3. 自动导演所有 crowd 行为

## 3. 不为了工具反向拖慢 Day1 本体

如果工具开发开始反过来卡住剧情推进，就要立即止损。

止损口径是：

1. 保留已有数据结构
2. 保留最小录制/摆位能力
3. 暂停继续做豪华功能
4. 先把剧情本体推进完

## 五、推荐执行顺序

这部分是后续每一轮都优先遵守的顺序。

## 阶段 1：先把“剧情本体最缺的正式稿”继续压下去

优先顺序：

1. `CrashAndMeet`
2. `EnterVillage`
3. `HealingAndHP / WorkbenchFlashback / FarmingTutorial`
4. `DinnerConflict / ReturnAndReminder / FreeTime / DayEnd`

原因：

- 这些内容是 Day1 正式体验的主骨架；
- 就算工具暂时还没做，它们也必须继续推进。

## 阶段 2：并行补“最小导演工具数据层”

这一阶段只做：

1. beat 数据结构
2. 站位点/朝向/目标点存储
3. 最小编辑入口

目标不是立刻让工具很酷，而是：

- 先把“硬编码死位置”这件事止住。

## 阶段 3：补单 NPC 排练录制

这一阶段只做：

1. 选一个 NPC
2. WASD 操控
3. 录位置/朝向
4. 保存/回放

优先只接 2 类场景：

1. `EnterVillage` post-entry
2. `NightWitness_01`

原因：

- 这两类最容易受场景微调影响
- 也是最适合先验证工具收益的地方

## 阶段 4：用工具反哺剧情段

这一阶段的目标是：

1. 把 `EnterVillage` crowd 先从手写站位改成导演数据驱动
2. 把 `NightWitness_01` 先从手写位置改成导演数据驱动
3. 视效果决定是否继续推广到：
   - `DinnerBackgroundRoot`
   - `DailyStand_01~03`

## 六、第一批最适合吃导演工具的场景

不是所有场景都值得第一时间吃工具。

当前最适合的，是这 4 个：

### 1. `EnterVillageCrowdRoot`

原因：

1. 围观层最容易反复调
2. 站位密度与视线关系最敏感
3. 用硬编码很快会脏

### 2. `KidLook_01`

原因：

1. 单点观察位很适合用最小工具验证
2. 只需要一个 NPC，就能看出录制/回放值不值

### 3. `NightWitness_01`

原因：

1. 夜间见闻位天然适合“站位 + 朝向 + 一句话”
2. 不需要复杂路径，也能体现工具收益

### 4. `DailyStand_01`

原因：

1. 次日站位预示适合用静态摆位先吃工具
2. 风险低

当前不建议第一批就吃工具的，是：

1. `DinnerBackgroundRoot`
   - 因为背景层人数更多，容易一开始把工具复杂度拉高
2. `CrashAndMeet`
   - 因为前半段还牵涉更强的正式剧情链和临时承载链

## 七、验收标准分层

这部分后面每轮都要拿来防止装过线。

## 1. 剧情本体层

算完成，不代表体验全过线，只代表：

1. 正式文本稿成形
2. phase/beat 顺序清楚
3. 台词语义不再漂

## 2. 工具结构层

算完成，只代表：

1. 数据结构站住
2. 可保存
3. 可读取
4. 可最小回放

## 3. 工具体验层

算真正有价值，至少要做到：

1. 我或你真能拿它调一个 NPC 的站位
2. 录一次后能在剧情里复现
3. 改场景后确实比硬编码好改

如果只做到结构层，还不能吹成“导演工具已经好用了”。

---

## 八、风险评估与止损阈值

### 1. 当前最大风险

不是不会写，而是：

- 工具如果一开始做太大，会反过来拖死 Day1 本体。

### 2. 第二大风险

- 剧情稿和工具并行时，容易两边都开头，结果都没收口。

### 3. 正确止损阈值

只要出现下面任一情况，就必须停扩工具：

1. 为了工具连续两轮没推进剧情本体
2. 工具开始要求多 NPC 同录/复杂时间线才“感觉能用”
3. 单 NPC 摆位都还没跑通，就开始扩很多 fancy 功能

### 4. 正确推进判断

如果出现下面这些信号，说明工具值得继续：

1. `EnterVillageCrowdRoot` 改位置比以前明显更快
2. `NightWitness_01` 可以快速排练并回放
3. 同一 beat 改站位不用再改多处逻辑

## 九、后续每轮执行模板

以后我继续这条线时，每一轮都应先回答这 5 个问题：

1. 这轮推进的是剧情本体，还是导演工具，还是两者各一小步？
2. 这轮新增的内容是否真的减少了后续硬编码？
3. 这轮有没有越界去做 UI / Town scene / NPC own 底座？
4. 这轮是否还守住了“轻量工具”边界？
5. 这轮结束后，Day1 离正式可落资产更近了吗？
