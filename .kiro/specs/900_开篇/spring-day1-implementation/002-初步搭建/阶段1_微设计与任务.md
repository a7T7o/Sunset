## 🎯 系统设计概述

### 核心职责
对话系统负责：
1. **显示对话内容**：在屏幕下方显示对话框，包含说话者名字和对话文本
2. **控制对话流程**：支持多条对话顺序播放，玩家点击"继续"按钮推进
3. **禁用玩家输入**：对话显示时禁用玩家移动和交互
4. **提供查询接口**：其他系统可以查询对话是否正在进行

### 设计原则
1. **单一职责**：对话系统只负责显示对话，不负责触发逻辑
2. **事件驱动**：通过事件系统与其他模块解耦
3. **配置驱动**：对话内容通过 ScriptableObject 配置，不硬编码
4. **性能优先**：避免 GC（内存垃圾），使用对象池和 StringBuilder

## 📦 数据结构设计

### 3.1 DialogueSequenceSO.cs (ScriptableObject)

**职责**：存储完整对话序列（单资产），避免单句资产碎片化。

**为什么不拆成多个 DialogueData 资产？**
- ❌ 每句一个 `.asset` 会导致资产数量膨胀、维护成本急剧上升
- ❌ 配置时需要在 Inspector 中频繁拖拽引用，易错
- ✅ 一个 `DialogueSequenceSO` 承载整段对话，配置连续、可读性强
- ✅ 与 `OUT_design.md` 的 `DialogueSequenceSO + DialogueNode` 架构一致

**数据结构**：
```csharp
[CreateAssetMenu(fileName = "NewDialogueSequence", menuName = "Story/Dialogue Sequence")]
public class DialogueSequenceSO : ScriptableObject
{
    [Header("对话元信息")]
    public string sequenceId;

    [Header("节点列表（按顺序播放）")]
    public List<DialogueNode> nodes = new();

    [Header("序列配置")]
    public bool canSkip = true;
    public float defaultTypingSpeed = 30f; // 字/秒
}

[System.Serializable]
public class DialogueNode
{
    public string speakerName;
    [TextArea(3, 10)] public string text;
    public Sprite speakerPortrait;
    public bool isInnerMonologue;
    public bool isBubble;
    public float typingSpeedOverride; // <=0 时使用序列默认速度
}
```

**设计要点**：
1. `DialogueNode` 使用 `[System.Serializable]`，作为序列内部节点而非独立资产
2. 单资产序列结构便于策划一次性配置整段剧情
3. 字段已预留内心独白/气泡对话样式分支

## 🎮 管理器设计

### 4.1 DialogueManager.cs (单例管理器)

**职责**：控制对话的播放流程和状态管理

**为什么用单例？**
- 对话系统在整个游戏中只需要一个实例
- 其他系统需要方便地调用对话功能（如：`DialogueManager.Instance.PlayDialogue(...)`）
- 避免多个管理器同时播放对话导致冲突

**核心接口**：
```csharp
using Sunset.Events;

public class DialogueManager : MonoBehaviour
{
    public static DialogueManager Instance { get; private set; }

    public bool IsDialogueActive { get; private set; }

    public void PlayDialogue(DialogueSequenceSO sequence);
    public void AdvanceDialogue();
    public void StopDialogue();
}
```

**设计要点**：

1. **单例模式**：
   - 使用 `Awake()` 初始化单例
   - 检测重复实例并销毁
   - 使用 `DontDestroyOnLoad()` 保证场景切换时不被销毁

2. **状态管理**：
   - `IsDialogueActive`：其他系统查询对话状态
   - 对话开始/结束通过 `EventBus.Publish(new DialogueStartEvent/DialogueEndEvent)` 广播

3. **事件驱动**：
   - 对话开始：发布 `DialogueStartEvent`
   - 节点切换：发布 `DialogueNodeChangedEvent`
   - 对话结束：发布 `DialogueEndEvent`

4. **与 UI 的解耦**：
   - `DialogueManager` 不直接操作 UI 组件
   - `DialogueUI` 订阅 `DialogueNodeChangedEvent` 更新显示
   - 管理器仅负责流程与事件发布

---

### 4.2 核心流程设计

**播放对话序列的流程（含打字机）**：
```
1. 调用 PlayDialogue(DialogueSequenceSO sequence)
2. 校验 sequence/nodes 有效，设置 IsDialogueActive = true
3. 发布 EventBus.Publish(new DialogueStartEvent { SequenceId = sequence.sequenceId })
4. 进入播放协程，按节点顺序处理：
   a. 发布 DialogueNodeChangedEvent（携带当前节点）
   b. 使用 StringBuilder + 协程逐字刷新文本（打字机效果）
   c. 打字完成后等待玩家点击“继续”
5. 所有节点播放完毕，调用 StopDialogue()
6. 设置 IsDialogueActive = false
7. 发布 EventBus.Publish(new DialogueEndEvent { SequenceId = sequence.sequenceId })
```

**关键设计决策**：
- 使用协程（Coroutine）控制“逐字显示 + 等待继续”链路
- 打字阶段按字速参数驱动（节点覆盖速度优先于序列默认速度）
- 继续逻辑只改状态标记，不在 `Update()` 中做轮询分支堆叠

## ⚡ 性能优化设计

### 5.1 为什么要关注性能？

对话系统虽然看起来简单，但如果实现不当，会产生严重的性能问题：
- ❌ 每次更新文本都 `new string`，产生大量 GC（内存垃圾）
- ❌ 频繁的字符串拼接导致内存碎片
- ❌ UI 重建导致 Canvas 重绘，造成卡顿

### 5.2 优化策略

**策略一：使用 StringBuilder + 协程实现打字机（阶段1立即实现）**

```csharp
private readonly StringBuilder textBuilder = new StringBuilder(512);

private IEnumerator TypeNodeText(string fullText, float charsPerSecond)
{
    textBuilder.Clear();
    float interval = 1f / Mathf.Max(charsPerSecond, 1f);

    for (int i = 0; i < fullText.Length; i++)
    {
        textBuilder.Append(fullText[i]);
        dialogueText.text = textBuilder.ToString();
        yield return new WaitForSeconds(interval);
    }
}
```

**策略二：缓存 UI 组件引用**

```csharp
// ❌ 错误做法：每次都查找组件
void UpdateUI()
{
    GameObject.Find("DialoguePanel").GetComponent<TextMeshProUGUI>().text = "...";
}

// ✅ 正确做法：在 Awake 中缓存引用
private TextMeshProUGUI speakerNameText;
private TextMeshProUGUI dialogueText;

void Awake()
{
    speakerNameText = transform.Find("SpeakerName").GetComponent<TextMeshProUGUI>();
    dialogueText = transform.Find("DialogueText").GetComponent<TextMeshProUGUI>();
}
```

**策略三：对象池（阶段1暂不实现，但预留接口）**

未来如果需要显示多个对话气泡（如：NPC 头顶对话），可以使用对象池：
```csharp
// 预留接口
public interface IPoolable
{
    void OnSpawn();
    void OnDespawn();
}
```

### 5.3 阶段1的性能目标

- ✅ 无 GC 分配（除了首次创建 UI）
- ✅ UI 更新不超过 1ms
- ✅ 内存占用小于 1MB

### 6.4 DialogueUI.cs (UI控制脚本)

**职责**：监听 EventBus 对话事件并更新 UI 显示。

**核心接口**：
```csharp
using TMPro;
using Sunset.Events;

public class DialogueUI : MonoBehaviour
{
    [Header("UI组件引用")]
    public GameObject dialoguePanel;
    public TextMeshProUGUI speakerNameText;
    public TextMeshProUGUI dialogueText;
    public Button continueButton;

    private void OnEnable()
    {
        EventBus.Subscribe<DialogueStartEvent>(OnDialogueStart, owner: this);
        EventBus.Subscribe<DialogueNodeChangedEvent>(OnDialogueNodeChanged, owner: this);
        EventBus.Subscribe<DialogueEndEvent>(OnDialogueEnd, owner: this);
    }

    private void OnDisable()
    {
        EventBus.UnsubscribeAll(this);
    }

    private void OnDialogueStart(DialogueStartEvent _)
    {
        dialoguePanel.SetActive(true);
    }

    private void OnDialogueNodeChanged(DialogueNodeChangedEvent evt)
    {
        speakerNameText.text = evt.Node.speakerName;
    }

    private void OnDialogueEnd(DialogueEndEvent _)
    {
        dialoguePanel.SetActive(false);
    }
}
```

**设计要点**：
1. 统一订阅 EventBus，避免与管理器形成硬引用
2. 文本组件统一 `TextMeshProUGUI`
3. 按钮点击事件绑定到 `DialogueManager.Instance.AdvanceDialogue()`

## 💡 关键设计决策总结

### 为什么使用 ScriptableObject 而非 JSON？
- Unity 原生支持，Inspector 直接配置
- 类型安全，编译时检查
- 支持资源引用（Sprite、AudioClip）
- 避免手写 JSON 的错误

### 为什么使用事件系统而非直接调用？
- 解耦：对话系统不需要知道谁在监听
- 扩展性：未来可以轻松添加新的监听者
- 维护性：修改一个系统不影响其他系统

### 为什么使用单例模式？
- 对话系统全局唯一
- 方便其他系统调用
- 避免多个实例冲突

### 为什么预留扩展字段？
- 避免未来修改数据结构导致资源重建
- 数据结构一次设计到位
- 实现可以分阶段，但接口要完整
