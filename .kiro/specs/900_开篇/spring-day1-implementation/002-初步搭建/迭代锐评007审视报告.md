# 迭代锐评007审视报告

## 1) 锐评核心观点摘要
- 指控 DialogueUI 在 Update() 每帧写入 TMP 文本，导致“无论字符串是否相同都会 Dirty/Canvas Rebuild/GC 灾难”，要求加本地缓存拦截。
- 指控 root.SetActive(false) 可能连带禁用 DialogueUI → OnDisable 退订事件，导致系统“永久失聪”。
- 指控 UI 未绑定 NPC 头像（speakerPortrait）。
- 指控 UI 未实现“内心独白”视觉样式切换（italic/gray/隐藏名字头像）。
- 建议清理 StoryEvents.cs 的冗余 using。

## 2) 事实核查结果表（声明 / 代码事实 / 结论）
| 声明 | 代码事实（证据） | 结论 |
|---|---|---|
| Update 每帧赋值 dialogueText.text | DialogueUI.cs:53-67：Update 中每帧 `dialogueText.text = _dialogueManager.CurrentTypedText;` | ✅ |

## 3) 认同的部分（可直接采纳）
- DialogueUI 未覆盖“头像/内心独白”属于需求缺口（requirements.md:28-35）。
- root 引用的层级/挂载确实存在自毁风险，需通过场景层级约束或注释规避。
- StoryEvents.cs 的冗余 using 可清理。

## 4) 疑虑与异议（附代码证据）
- “相同字符串赋值也必定 Dirty/重绘/GC”与当前包内 TMP_Text 实现不符：TMP_Text.cs:127-136 已做相等短路。
  - 结论：Update 每帧赋值在“文本不变的等待阶段”不一定触发 Dirty；该段锐评属于过度绝对化。

## 5) 独立分析（补充风险）
- DialogueManager.TypeNodeText 每次 `StringBuilder.ToString()` 都会创建新 string（DialogueManager.cs:166-180），在打字阶段确实会产生分配；但这不等同于“等待阶段每帧重绘”。

## 6) 建议执行方向
- 将锐评拆分：需求缺口（头像/内心独白）与生命周期风险（root/挂载约束）优先修正；性能补丁需基于事实收敛，避免为错误断言做无意义改动。

## 7) 给锐评方的信息补充
- 当前项目使用的 TMP_Text setter 已包含“相同文本不触发 Dirty”的短路逻辑（TMP_Text.cs:125-136）。
