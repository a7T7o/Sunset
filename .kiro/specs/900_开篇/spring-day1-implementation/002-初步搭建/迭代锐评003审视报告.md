# 迭代锐评003审视报告

## 1. 锐评核心观点摘要
- 锐评003要求对前一版“仅供审阅的代码 Diff 方案”做三点修正：
  1) 打字机协程中 `WaitForSeconds` 的循环内 `new` 造成 GC 风险
  2) 不应在逐字阶段高频 `EventBus.Publish(...)`
  3) 输入锁应改为通用接口 `GameInputManager.SetInputEnabled(bool)`，避免 `_dialogueInputLocked` 专用变量扩散

## 2. 事实核查结果表

| 锐评声明 | 代码/文档事实 | 结论 |
|---|---|---|
| 设计文档明确要求“WaitForSeconds 缓存，避免每帧创建 YieldInstruction” | `OUT_design.md:875-885` 示例实际是 `yield return new WaitForSeconds(charInterval);`，未写缓存规范 | ❌ |
| 前版 Diff 在打字机循环中 `new WaitForSeconds(interval)` | 前版提交的 `DialogueManager` 评审 Diff 确有该写法（循环内创建） | ✅ |
| EventBus 用于跨系统广播，不适合高频逐字广播 | `OUT_design.md:1397-1429` 明确 EventBus 适合跨系统广播；未强制禁止高频，但方向上支持“低频状态事件” | ⚠️ |
| 前版 Diff 每字广播 `DialogueTypingProgressEvent` 会造成额外对象分配与广播开销 | 前版 Diff 确有每字 `EventBus.Publish(new DialogueTypingProgressEvent...)` | ✅ |
| 需求文档规定通用输入接口 `GameInputManager.SetInputEnabled(bool)` | `requirements.md:267-270` 明确列出该依赖接口与调用语义 | ✅ |
| 前版 Diff 使用 `_dialogueInputLocked` 专用字段，扩展性不足 | 前版 Diff 为对话特供标志位，未抽象为通用接口 | ✅ |

## 3. 认同的部分
- 认同“循环内 `new WaitForSeconds` 需要优化”的工程结论。
- 认同“逐字刷新不应走高频全局广播”的方向。
- 认同“输入锁应抽象成通用接口 `SetInputEnabled(bool)`”并由对话/CG共用。

## 4. 疑虑与异议（附证据）
- 锐评第一条中的“依据来源”有事实错误：
  - 锐评称设计文档已明确“必须缓存 WaitForSeconds”，但 `OUT_design.md:875-885` 并无该明确条款，示例反而是循环内 `new`。
- “EventBus 基于反射”并非当前仓库事实：
  - `Assets/YYY_Scripts/Core/Events/EventBus.cs:153-201` 使用泛型 `Action<T>` 委托调用链，非反射分发。

## 5. 独立分析
- 本次锐评属于“核心工程方向正确 + 局部依据表述失真”的混合类型。
- 依据 `.kiro/steering/code-reaper-review.md`：存在 ❌ 事实错误时，进入路径 C，先出审视报告再等待确认。

## 6. 建议的执行方向（待确认后实施）
1. **修正打字机等待对象**：将 `WaitForSeconds` 提前到循环外缓存（按速度变化点重建，而非每字重建）。
2. **移除逐字高频 EventBus 广播**：
   - EventBus 保留 `DialogueStartEvent / DialogueNodeChangedEvent / DialogueEndEvent`；
   - 逐字文本由 `DialogueManager.CurrentTypedText` 暴露，`DialogueUI` 本地协程/轮询读取。
3. **落地通用输入接口**：在 `GameInputManager` 增加 `SetInputEnabled(bool enabled)`，对话事件回调只调用该接口，不新增特供锁字段分支。

## 7. 给锐评方的反馈要点
- 采纳三项修正方向。
- 更正两处事实描述：
  1) 设计文档未明确写“WaitForSeconds缓存强制规范”；
  2) 当前 EventBus 不是反射分发。
- 后续将按通用输入接口和低频事件模型提交修正后完整 Diff。
