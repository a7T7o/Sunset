## 为什么这轮必须由你主刀

用户已经明确指出一个高风险事实：

- 现在不仅 fresh 运行会出问题
- `重新开始` 也会出问题
- `读档` 更大概率也会出问题

这说明问题已经不只是 `spring-day1` 自己的 runtime 开场链，而是：

`save/load/restart` 没把 Day1 晚段恢复所需的语义 contract 传全，或者没有把恢复动作交回正确的 day1 re-entry hook。

## 这轮必须站住的用户坏相

你要围绕下面这些真实坏相来做，不要只围绕“代码看起来合理”：

1. 晚饭无论是玩家主动找村长触发，还是用 `+` 快进到晚上，都会卡在 `0.0.6 晚餐冲突` 入口不继续接管。
2. `重新开始` 后，`001/002` 可能没有在晚饭剧情里正确就位，但对白却已经开始。
3. 用户合理怀疑：`读档` 只会比 `重新开始` 更容易留下：
   - 晚段 phase 恢复不完整
   - UI 状态残留
   - 时间夹取状态残留
   - 私有导演标记残留或丢失
   - 对话 / prompt / 模态 UI / pause 栈没有正确重建

## 你的职责边界

### 你必须做的

1. 明确 `StoryProgressPersistenceService` 这一侧，Day1 晚段恢复到底应该保存/恢复哪些语义。
2. 把这些语义恢复到 **“足够让 day1 runtime 自己接着往下跑”**，而不是只恢复一个静态 phase。
3. 明确恢复时要调用 day1 哪些 re-entry / rehydrate hook。
4. 补专门回归测试，证明：
   - 读档不会把晚段推进乱跳
   - 恢复后不会卡死在 dinner / reminder / free-time 入口
   - 恢复后不会留下 stale UI / pause / 对话状态

### 你不要做的

1. 不要越权自己去修 `001/002` 的剧情站位逻辑。
2. 不要自己去改 PromptOverlay 的视觉布局。
3. 不要替 `spring-day1` 线程接管晚饭 runtime staging / NPC 位置硬修。
4. 不要把 runtime own 问题包装成“存档已闭环”。

正确边界是：

- `spring-day1` own：晚段 runtime 开场、自愈、剧情 actor staging、prompt 展示语义
- `存档系统` own：把晚段恢复所需语义正确保存/恢复，并在恢复后把控制权交回 day1 runtime

## 你这轮至少要核清的恢复 contract

下面这些不是要求你全部都自己硬编码保存，而是要求你明确：

- 哪些必须持久化
- 哪些可以由 runtime 重新推导
- 哪些必须在 load 后主动调用 re-entry hook 才算恢复完成

最低要覆盖这些层：

### 1. 相位与对白消费层

- `StoryPhase`
- `completedDialogueSequenceIds`
- 晚段相关 sequence 的真实消费态：
  - `spring-day1-dinner`
  - `spring-day1-reminder`
  - `spring-day1-free-time-opening`

### 2. Day1 晚段导演私有语义层

至少核清这些是否必须恢复，或者能否安全重建：

- `_postTutorialExploreWindowEntered`
- `_dinnerSequencePlayed`
- `_returnSequencePlayed`
- `_freeTimeEntered`
- `_freeTimeIntroCompleted`
- `_dayEnded`
- 夜间 warning / final-call 这类晚段提示状态

### 3. 时间与门禁层

- 当前时间与 Day1 time guardrails 的关系
- load 后是否需要重新执行 `EnforceDay1TimeGuardrails() / SyncStoryTimePauseState()`
- 是否会因为只恢复了时钟、没恢复当前晚段门禁，而把剧情直接跳坏

### 4. UI / 对话 / 模态状态层

至少明确 load 后这几类状态是否必须强制清理或重建：

- `DialogueManager` 当前 active 状态
- `PromptOverlay` / manual bridge prompt 残留
- `PackagePanel / BoxPanel / Workbench / Prompt` 的模态阻塞残留
- time pause stack / input gate 残留

### 5. scene re-entry / day1 runtime 接线层

这层最重要：

你不能只把 phase 写回去，然后让 day1 自己“也许会在某一帧恢复”。

你必须明确：

- load 到 `DinnerConflict` 后要不要通知 day1 执行晚饭 re-entry
- load 到 `ReturnAndReminder` 后要不要通知 day1 执行 reminder re-entry
- load 到 `FreeTime` 后要不要通知 day1 执行 free-time re-entry

如果答案是“要”，就把这条 contract 明确钉住，并在测试里验证。
