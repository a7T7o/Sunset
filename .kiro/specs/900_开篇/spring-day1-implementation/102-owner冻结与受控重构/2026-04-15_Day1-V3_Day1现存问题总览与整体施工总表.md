## 2. Day1 语义基线

下面这些不是我主观猜测，而是当前后续所有施工都必须服从的 Day1 语义基线。

### 2.1 opening

1. `001/002/003/101~203` 这批 opening 参与者，先被放到你摆好的起点。
2. 再按 staged contract 走到终点。
3. 最多等 5 秒，超时 snap 到位。
4. 对白期间不再继续走位。

### 2.2 opening 结束后

1. `001/002` 继续吃 `house lead / Primary` 的剧情合同。
2. `003` 与 `101~203` 最终应该并回同一 resident release contract。
3. `Day1` 不该继续拖着普通 resident 的剧情后下半生。
4. resident 被 release 后：
   - 由 NPC 自己进入“回 anchor/home”或“恢复日常”的状态
   - 导航只负责把这段路走好

### 2.3 Healing / Workbench / Primary

1. `001/002` 只在 `Primary` 吃剧情特殊性。
2. Town resident 维持正常自由活动。
3. 不允许靠隐藏 Town、冻结 Town、阻止玩家回 Town 这类手法掩盖运行时问题。

### 2.4 dinner

1. `dinner` 与 `opening` 应共享同一类 staged contract 思想。
2. 不应再另起一套“表面像 staged、底层却是另一套 runtime”的逻辑。

### 2.5 通用时序

1. `19:30` 自由时段
2. `20:00` 回家
3. `21:00` snap / night-rest
4. `26:00` sleep

这组时序最终应是共享 schedule contract，不应继续深埋在 Day1 私房 runtime 中。

### 2.6 NPC movement 统一原则

我当前给出的明确主张是：

`只要 NPC 发生真实位移，原则上都应走统一的导航执行合同；剧情层只发“谁要动、去哪、何时放手、何时 snap”的语义命令。`

唯一允许的例外只有 3 类：

1. staged timeout 后的显式 snap
2. load/repair 场景下的显式 snap
3. 作者摆位失败时的强制对位兜底

换句话说：

1. `Day1` 可以决定“要不要动”
2. `导航` 负责“怎么动”
3. `NPC` 负责“动完后进哪个 resident state”

## 4. 我当前认可的最终边界

### 4.1 Day1 own

`Day1` 只该负责：

1. `opening / dinner / house lead / Primary`
2. phase / beat / dialogue queue
3. staged contract
4. release intent
5. 超时 snap 的语义裁定

`Day1` 不该再负责：

1. resident release 后怎么继续活
2. resident 的 return-home tick
3. resident 的 resume/restart roam
4. night-rest 运行时 owner

### 4.2 NPC own

`NPC` 只该负责：

1. resident state 自己怎么切
2. release 后是：
   - `回 anchor 再 roam`
   - `回 home`
   - `继续特殊工作`
3. 对外只暴露语义安全的 facade，不再裸露内部状态机原语

### 4.3 导航 own

`导航` 只该负责：

1. 真正的 movement execution
2. 去 anchor / 去 home / 去 staged target
3. stuck / fail / abort / replan
4. body clearance / static navigation / detour / local avoidance

`导航` 不该 own：

1. phase / beat
2. 谁该 release
3. `001/002/003` 的剧情语义

### 4.4 当前主导判断

我当前给出的主导判断是：

1. `Day1-V3` 继续主导 Day1 语义、owner 退权与集成收口。
2. `NPC` 线程应主导 facade 落地与低级 public API 收口。
3. `导航` 线程应主导“只要发生位移，就走统一导航执行合同”的执行层收口。

也就是说：

`Day1` 不是自己吞完所有代码，而是自己主导边界与 call-site 接入，把执行和身体收回到各自 owner。
