# 2026-04-15｜Day1-V3｜施工方向与分阶段清单

## 1. 本文件用途

这不是续工 prompt，也不是“分析已完成”的自我表扬。

这份文件只做 4 件事：

1. 把 `Day1` 这条长线程接下来到底怎么施工写成硬方向。
2. 把哪些东西现在能改、哪些东西现在不能碰写成硬边界。
3. 把后续切片拆成可执行阶段，而不是继续“想到哪修到哪”。
4. 把每阶段的完成定义、停步条件、验证口径写死，避免再次出现“结构说对了，但代码、测试、live 还在保护另一套真相”。

## 2. 当前唯一主线

当前唯一主线不是“继续把 `CrowdDirector` 调聪明”，也不是“先修一个坏相再看”。

当前唯一主线是：

`把剧情后 resident 生命周期从 SpringDay1NpcCrowdDirector 手里剥掉，并把 Day1 / Crowd / NPC / 导航 / schedule 的 owner 边界重新收成单一真相。`

## 3. 当前基线

### 3.1 已完成

`101-opening-handoff统一合同` 已经收掉 opening handoff 的第一条硬冲突：

1. opening release latch 后不再直接 `direct autonomy`
2. `003` 不再被导演从 opening 终点直接放回 roam
3. 对应 staging tests 已改成保护“先回 anchor/home，再恢复 roam”的 opening handoff 合同

### 3.2 仍未完成

下面这些现在仍然是代码真相，不允许误报为“已退权”：

1. `SpringDay1NpcCrowdDirector.Update()` 仍无条件跑：
   - `SyncResidentNightRestSchedule()`
   - `TickResidentReturns()`
2. `CrowdDirector` 仍持有：
   - `return-home`
   - `night-rest`
   - `recover missing residents`
   - `resume-roam after return`
3. `SpringDay1Director` 仍直接参与：
   - `003` release
   - story actor runtime policy
   - crowd release latch / cue hold
4. 测试层仍有一批 stopgap tests 在保护旧 owner 口径

### 3.3 当前准确阶段

当前阶段不是“最终体验已过线”，也不是“可以无边界总重构”。

当前准确阶段是：

`冻结完成前的受控重构准备期`

也就是：

1. 先把 owner 入口、003 残留、stopgap tests 全部冻结
2. 再按 owner 主线分批拆
3. 每一批只收一个语义层，不再散修

## 4. 目标分工

### 4.1 目标 owner matrix

| 组件 | 应该负责什么 | 不该再负责什么 |
|---|---|---|
| `SpringDay1Director` | Day1 phase / beat / dialogue queue / staged contract / release intent | resident 剧情后下半生、return-home tick、night-rest runtime |
| `SpringDay1NpcCrowdDirector` | roster / bindings / authored marker lookup / crowd cue 辅助 / probe/debug summary | resident release 后 locomotion owner、restart roam、night schedule、recover runtime |
| `NPC facade` | `AcquireStoryControl` / `ReleaseToAnchorThenRoam` / `ReleaseToHome` / `ResumeAutonomousRoam` / `SnapToTarget` | Day1 phase 判断、crowd cue 解析 |
| `导航` | 去 anchor / 去 home / stuck / fail / arrived | Day1 剧情语义、resident 是否该 release |
| `全日 schedule` | `20:00 return home` / `21:00 snap/rest` / `26:00 sleep` | Day1 私房 runtime owner |

### 4.2 当前代码与目标分工的核心差距

最大差距不是参数不对，而是：

1. `CrowdDirector` 还在既当 crowd layer，又当 resident 生活 owner
2. `Director` 还在既当剧情语义层，又当 runtime 释放协调器
3. 测试还在给这两条旧 owner 链亮绿灯

## 5. 本线程硬红线

后续施工必须遵守下面这些硬红线：

1. 不再把 opening cue 终点直接当成 free-roam 起点。
2. 不再允许 `CrowdDirector` 在 resident 已 release 后反复重抓、重放、重启 roam。
3. 不再把 `003` 的 opening 之后 runtime 当成单独 owner 分支长期保留。
4. 不再把 `20:00 / 21:00 / 26:00` 深埋成 `Day1` 私房 runtime。
5. 不再用 `StopRoam / RestartRoam / DebugMoveTo / fallback transform step` 组合拳掩盖 owner 问题。
6. 不扩：
   - `dinner`
   - `Primary`
   - `UI`
   - 导航 core
   除非当前批次的 owner 收口明确要求触达。

## 6. 分阶段施工图

### 6.1 Phase A｜冻结与单一真相

目标：

1. 列出 resident lifecycle 全入口冻结表
2. 列出 `003` 特殊化残留表
3. 列出 owner 级 stopgap tests 清单
4. 明确哪些保留、哪些重写、哪些删除

完成定义：

2. 后续每一刀都有明确落点和不落点
3. 不再允许“分析说退权，代码还在续命”

当前状态：

`进行中`

### 6.2 Phase B｜CrowdDirector daytime owner 收口

唯一目标：

把 daytime / opening release 后 resident 的持续 runtime 持有，从 `CrowdDirector` 身上切掉。

优先动作：

1. 停止 `Update()` 里对已 release resident 的持续 return-home / resume-roam 代驾
2. 收掉 `RecoverReleasedTownResidentsWithoutSpawnStates(...)` 这类恢复暗门
3. 让 `CrowdDirector` 在无显式 crowd beat 时尽量接近 idle

当前重点方法：

1. `TickResidentReturns()`
2. `TickResidentReturnHome(...)`
3. `TryBeginResidentReturnHome(...)`
4. `TryDriveResidentReturnHome(...)`
5. `FinishResidentReturnHome(...)`
6. `CancelResidentReturnHome(...)`
7. `RecoverReleasedTownResidentsWithoutSpawnStates(...)`

完成定义：

1. opening / daytime release 后，resident 不再被 `CrowdDirector` 每帧碰 runtime
2. `CrowdDirector` 关闭前后，不会再出现“关掉它反而 NPC 更正常”的反证
3. profiler 里 `SpringDay1NpcCrowdDirector.Update()` 不再因 resident 下半生 owner 占大头

### 6.3 Phase C｜Director 退回纯语义层

唯一目标：

把 `Director` 里对 resident runtime 的越界协调收回到语义层，只保留 staged contract / release intent。

优先动作：

1. 收缩 `ReleaseOpeningThirdResidentControlIfNeeded(...)`
2. 审 `ApplyStoryActorRuntimePolicy(...)` 与 story actor night-rest 是否继续混着 resident runtime
3. 保留 opening staged contract 必需的 authored actor 逻辑，移走剧情后的 resident lifecycle 协调

完成定义：

1. `Director` 不再对普通 resident 的 release 后 runtime 负责
2. `003` opening 之后不再靠导演特殊放手维持
3. `ShouldReleaseEnterVillageCrowd / ShouldLatch... / ShouldHold...` 只保留 cue/staged 语义，不再延伸成 resident runtime owner

### 6.4 Phase D｜全日 schedule 外移

唯一目标：

把 `20:00 / 21:00 / 26:00` 从 Day1 私房 runtime 迁出，统一为公共 schedule contract。

说明：

这一阶段优先级低于 Phase B / C。

原因：

1. 当前最痛的 live 反证仍是 daytime owner 错位
2. 先外移 schedule 容易误伤晚段稳定链

完成定义：

1. `ShouldResidentsReturnHomeByClock / ShouldResidentsRestByClock` 不再是 `CrowdDirector` 的长期 owner 入口
2. `Day1` 与 `NPC/导航` 对 schedule 只接合同，不再自持 runtime

### 6.5 Phase E｜测试真相统一

唯一目标：

把测试从“保护旧 owner stopgap”改成“保护新边界与真实合同”。

处理原则：

1. staging contract tests 尽量保留
2. owner stopgap tests 改写到 `NPC facade / schedule / 导演 release intent` 的新位置
3. 已经过时且只保护旧 owner 的测试直接删

完成定义：

1. 测试绿灯不再反向鼓励继续保留旧 owner
2. 本线程以后不会再出现“文档说对了，测试却仍在护旧合同”

### 6.6 Phase F｜live 终验

重点只看用户可感知结果：

1. opening 后 `003 / 101~203` 是否先回 anchor/home 再恢复自治
2. `001/002` 是否仍只保留 story actor 合同，不拖普通 resident 一起抖
3. `CrowdDirector` 开着时，resident 是否与关掉它时同样能正常回 anchor / roam
4. profiler 是否不再由 `SpringDay1NpcCrowdDirector.Update()` 热链主导

## 7. 当前阶段不做什么

下面这些即使看起来相关，这个阶段也不先做：

1. 不先调导航采样、避障、pathing 参数
2. 不先做 UI/Prompt/Primary 扩题
3. 不先扩大到 dinner 新逻辑
4. 不先做“看上去更流畅”的表层补丁

原因很简单：

`先改执行细节，后改 owner，只会继续换坏相，不会真正停掉错误 owner。`

## 9. 停步条件

如果出现下面任一情况，本线程应停下，不继续盲改：

1. 新批次需要改到 `dinner / Primary / UI / 导航 core`
2. `001/002` 的 story actor 合同和普通 resident 合同开始互相打架
3. 测试全面红，但红因其实是“新 owner 还没补新位置测试”
4. live 证据开始说明第一责任已不在 Day1 own
