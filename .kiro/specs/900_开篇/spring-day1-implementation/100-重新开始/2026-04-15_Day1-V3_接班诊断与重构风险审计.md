## 2. 我独立确认下来的核心事实

### 2.1 `CrowdDirector` 仍然是剧情后 resident runtime owner

3. 这意味着即使 opening handoff 已经进入所谓的 `stand-down`，`CrowdDirector` 仍然会继续：
   - 推 resident 回家
   - 接管夜间回家/睡觉

结论：
`stand-down` 目前只挡住了部分 `SyncCrowd()`，没有真正挡住 runtime owner。

   - `opening release latch -> direct autonomy`

这不是“还差一点点”，而是文档语义和代码语义还没统一。

## 3. 我认为真正的问题清单

### 3.1 第一问题：owner 没有单一真源

当前至少有 4 类 owner 在同一条 resident 生命周期上交错发言：

1. `SpringDay1Director`
2. `SpringDay1NpcCrowdDirector`
3. `NPCAutoRoamController`
4. 时钟/夜间调度

结果不是“每人负责一段”，而是同一 resident 会被：

1. 接管
2. 放手
3. 重抓
4. 重放
5. 重启 roam

所以它不是单个 bug，而是 owner 结构失真。

### 3.2 第二问题：事件驱动和轮询驱动混在一个控制器里

`SpringDay1NpcCrowdDirector` 现在既做：

1. beat/cue 事件型逻辑
2. `Update()` 轮询
3. return-home tick
4. 夜间 schedule
5. 恢复链

这会导致任何一处“看起来已经 release”之后，下一帧又从别的入口重新抓回。

### 3.3 第三问题：代码和测试在保护两个不同的真相

当前至少有 3 层真相并存：

1. 用户要的真相：
   - opening 后先回 anchor/home，再 roam
2. 文档声称的真相：
   - `CrowdDirector` 要退权
3. 代码/测试实际保护的真相：
   - opening handoff 之后仍可 direct autonomy
   - `CrowdDirector` 继续负责 return-home 与 resume-roam

只要这三层没有合一，线程就会一直“分析越来越对，现场越来越乱”。

### 3.4 第四问题：`NPCAutoRoamController` 本身有高成本路径

这不是第一根因，但是真风险。

[NPCAutoRoamController.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Controller/NPC/NPCAutoRoamController.cs) 当前至少有这些高成本特征：

1. `DriveResidentScriptedMoveTo(...)` 直接走 `DebugMoveTo(...)`
2. `DebugMoveTo(...)` 会触发建路
3. `TryBeginMove(...)` 在 autonomous roam 下会反复采样和建路
4. `HasAutonomousRoamDestinationAgentClearance(...)` 会全场扫描 `registeredRoamControllers`

所以：
owner 错误会先制造坏输入；
导航/NPC 再把坏输入放大成 profiler 风暴。

### 3.5 第五问题：当前测试覆盖的是“局部合同”，不是“真实 live 合同”

当前测试文件接近 2000 行，很多测试用的是：

1. `Track(new GameObject(...))`
2. 直接改私有字段
3. 直接调用单个方法
4. 断言 isolated state

这类测试有价值，但它保护的是：

1. 某个局部方法
2. 某个中间状态
3. 某个 stopgap 合同

它保护不了：

1. Town 真 scene
2. 多 resident 并发
3. opening 完整收尾后 5-30 秒内的 live 表现
4. profiler 热链是否真的降温

## 4. 为什么旧线程做了很多轮还没解决，反而一直在制造新问题

### 4.1 因为它一直在“换坏相”，没有真正拔掉 owner

1. 越来越意识到 `CrowdDirector` 太重

   - 锁死
   - 假 roam
   - 回不去
   - 高频抖动
   - profiler 风暴

### 4.2 因为每轮都在同一个巨型文件上叠加条件，而不是先缩 owner

当前文件规模已经非常危险：

1. [SpringDay1Director.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Scripts/Story/Managers/SpringDay1Director.cs)：约 `8509` 行

3. [SpringDay1DirectorStagingTests.cs](D:/Unity/Unity_learning/Sunset/Assets/YYY_Tests/Editor/SpringDay1DirectorStagingTests.cs)：约 `1980` 行

2. `442b9a40`（2026-04-14）

其中 `442b9a40` 单次就在这 3 个文件上新增/改动了 5000+ 行。

这意味着：
每次不是在做窄修，而是在继续给巨型 owner 叠补丁和叠测试。

### 4.3 因为“先说结构对了”经常领先于“代码真相已经统一”

旧线程后期最大的问题不是完全不会分析，而是：

1. 文档判断前进得比代码快
2. 代码又前进得比 live 验证快
3. 最后就形成：
   - 文档已经说成“退权”
   - 代码还在 direct autonomy / return-home owner
   - 测试还在保护旧 owner
   - live 继续打脸

### 4.4 因为它在“止血补丁”和“正式重构”之间来回摇摆

旧线程同时在做三件互相冲突的事：

1. 立刻止血
2. 写大边界文档
3. 暗中继续延长旧 owner 的生命周期

这会导致：

1. prompt 越写越清楚
2. 代码越改越复杂
3. 用户越来越难判断现在到底是“分析轮”还是“真修轮”

### 4.5 因为测试给了错误的安全感

当前并不是“完全没测试”，而是：

1. 测试很多
2. 但不少测试在保护错合同
3. 于是绿灯反而会把线程往错误 owner 上再推一步

这就是为什么它会反复出现：

1. “我这次真的收了一刀”
2. “fresh live 又出新问题”

## 5. 如果后面要重构，真正的风险在哪里

### 5.1 最大风险：先改执行细节，后改 owner

如果后面一上来先改：

1. 导航参数
2. avoidance
3. pathing 细节
4. autonomous roam 采样

但不先拆 owner，那么结果大概率还是：

1. 短期变好一点
2. 下一轮又被 Day1/CrowdDirector 重写掉

### 5.2 第二风险：`001/002` 和 `003` 被一起误伤

`001/002` 目前仍和：

1. `house lead`
2. `Primary`
3. story actor runtime

深度绑定。

`003` 又仍然残留单独入口。

如果拆的时候不先把：

1. `001/002` 的剧情 actor 合同
2. `003` 的特殊化残留
3. 普通 resident 合同

分开，后面很容易把稳定链一起拆坏。

### 5.3 第三风险：测试会把正确重构当成回归

只要不先处理测试口径，后续真正的退权重构会撞上两类假回归：

1. 旧测试失败
2. 失败原因却是“你不再让 CrowdDirector 当 owner 了”

所以后面真开工前必须先做一次：

1. 测试分层
2. 标出哪些是 stopgap 测试
3. 标出哪些是未来要删或重写的 owner 级测试

### 5.4 第四风险：恢复链与夜间链会偷偷把 owner 捡回来

即使 opening daytime 退了，如果不处理：

1. `ShouldRecoverMissingTownResidents(...)`
2. `SyncResidentNightRestSchedule()`
3. `TickResidentReturns()`

那 owner 很可能会从：

1. 恢复链
2. 夜间链
3. return-home tick

再次回流到 `CrowdDirector`。

### 5.5 第五风险：共享根现场太脏，容易把“分析结论”说成“代码已闭环”

当前 shared root 很脏，不能把任何只读分析误说成现场已经 clean。
这轮只站住一件事：

`问题结构已经看清，但我还没有开始拆它。`
