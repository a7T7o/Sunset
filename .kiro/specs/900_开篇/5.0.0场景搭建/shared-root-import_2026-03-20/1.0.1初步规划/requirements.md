# 需求文档：1.0.1 初步规划

## 1. 阶段目标

本阶段的任务不是直接开搭真实场景，而是把“开工前必须稳定的东西”一次性定下来：

- 资产入口是否够用；
- 场景骨架怎么搭；
- Tilemap 层怎么分；
- 新 scene 应该叫什么、放哪；
- 首版 prefab 候选池该细到什么粒度；
- 进入真实施工前还差哪些前置条件。

## 1.1 长期主线承接

虽然当前正文承载区是 `1.0.1初步规划`，但它承接的是整个“独立场景搭建”主线的起步阶段，因此必须同时覆盖后续主线的最小施工蓝图。

后续长期主线目标固定为：

- 避开 `Assets/000_Scenes/Primary.unity` 作为正式施工承载面；
- 建立独立 scene；
- 形成可继续精修的高质量场景初稿；
- 让后续微调建立在清晰骨架之上，而不是凌乱残骸之上。

## 2. 本阶段必须交付的内容

- 一套真实可用的正文文档基线；
- 一份首轮资产普查；
- 一份场景骨架与 Tilemap 分层方案；
- 一份新 scene 命名、路径与最小创建方案；
- 一份首版 prefab 候选池粒度方案；
- 一份下一阶段进入条件说明。

## 3. 本阶段硬约束

- 只做文档与只读普查，不直接改 `Assets/000_Scenes/Primary.unity`。
- 不把范围扩散成“全项目地图规范重构”。
- 不为了“层级看起来专业”而滥拆 Tilemap 层；分层必须服务施工与维护。
- 不把 Skills / MCP 本身继续当成主目标；它们现在只是场景搭建的支撑工具。
- 不为了做索引而做索引；首版只做足够开工的候选池。

## 4. 本阶段要回答的核心问题

### 4.1 资产层面

- 当前项目是否已有足够的 scene / prefab / tile / sprite 基础？
- 哪些目录可以作为后续优先候选池？

### 4.2 结构层面

- 新 scene 需要什么根层级？
- Tilemap 至少应拆成哪些层才清楚且不过度？

### 4.3 执行层面

- 从规划进入真实施工时，第一步是什么？
- 哪些内容先做，哪些内容后做？
- 如何避免重复返工和场景越搭越乱？

### 4.4 交付层面

- 后续交给用户的是“基础骨架”还是“高质量初稿”？
- 初稿需要达到什么水平才算节省后续微调成本？

## 5. 验收标准

- 存在完整的 `memory.md / requirements.md / design.md / tasks.md`。
- 已有一份首轮资产普查，可证明该主线具备真实资产基础。
- 已明确场景骨架、Tilemap 分层、scene 命名与候选池粒度。
- 已明确当前阶段不直接施工，下一阶段的进入条件清楚。
- 用户后续无需重新解释“你应该按什么顺序搭场景”。
