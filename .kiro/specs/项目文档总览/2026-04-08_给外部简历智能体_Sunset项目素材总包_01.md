# Sunset 对外简历/作品集素材总包

## 1. 文档定位

本文用于沉淀 `Sunset` 项目中可直接供外部简历智能体、作品集整理位、面试问答整理位使用的核心事实池。

重点不是代写简历，而是把以下内容一次性说清：

1. 用户在项目里的真实身份是什么。
2. 用户真实承担了哪些职责。
3. 用户到底做成了哪些事情。
4. 哪些内容适合写进简历，哪些更适合留到作品集和面试展开。

## 2. 项目基础事实

### 2.1 项目口径

- 项目名：`Sunset`
- 项目类型：Unity 6 2D 像素风奇幻生活模拟 RPG
- 项目主轴：生活经营 + 资源成长 + 剧情推进
- 叙事前提：失忆工匠在废弃聚落中重新建立秩序，并在经营过程中找回记忆
- 当前核心闭环：`资源积累 -> 经营生产 -> 能力成长 -> 事件 / 探索解锁 -> 聚落复兴反哺`

### 2.2 当前项目阶段

从当前主文、工作区和线程记忆来看，`Sunset` 当前已经不是只做单系统原型，而是进入了“跨场景主线切片 + 居民化 + 持久化 + 终验回拉”的正式推进阶段。

当前最核心的项目主线不是“系统数量很多”，而是：

1. 把 `spring-day1` 从概念型开篇剧情，推进成真正可玩的跨场景主入口。
2. 把 `Town / Primary / Home` 三个场景压进同一条体验链。
3. 把 NPC、背包、工具态、云影、存档和导演链逐步收束到连续世界体验里。

## 3. 用户在项目中的真实身份

### 3.1 最准确的定位

用户在 `Sunset` 里的真实身份，不是普通提需求者，也不是单纯让 AI 写代码的人，而是：

- 项目策划主控
- 设计边界拍板者
- 跨场景主线推进者
- AI 多线程协同开发组织者
- Unity 实机验收与体验回拉负责人

### 3.2 更具体一点的说法

如果要对外解释，最准确的理解应该是：

1. 用户负责决定项目当前主线是什么，不是什么都一起做。
2. 用户负责把模糊想法压成可落地的玩法规则、场景合同和验收目标。
3. 用户不是把实现完全外包给 AI 后就不管，而是持续做认知对齐、回归测试、返工裁定和继续/停发判断。
4. 用户自己会进入 Unity 做数据配置、场景检查、玩法试玩、手感判断和细节验收。

## 4. 用户真实承担的职责

### 4.1 项目主线与场景推进

用户持续把项目推进收束在 `spring-day1` 的跨场景体验链上，并明确拍板：

1. 当前 Day1 正式按 `Town -> Primary -> Home` 推进。
2. `Town` 负责入村围观、驻村承接与后续生活面入口。
3. `Primary` 负责旧屋安置、工作台闪回、农田教学和第一次真实制作。
4. `Home` 已纳入正式场景体系，但当前口径是室内承接壳，不是和 `Town / Primary` 同级的完整玩法场。

### 4.2 产品级边界拍板

用户不是只提建议，而是持续在做正式产品裁定，包括：

1. NPC 从一次性剧情承载改判为常驻居民化结构。
2. 正式剧情只消费一次，消费后回落到 resident / informal 日常结构。
3. 玩家改为跨场景唯一实例，背包、Hotbar、工具态与场景切换统一围绕连续世界重构。
4. `Home` 镜头按固定构图、黑底、尊重手摆的方向收口。
5. 高频验证口径固定为 `CLI first, direct MCP last-resort`。
6. `301` 当前退出 Day1 正式主链，主线收束到更稳的 `Town 开场围观 -> 村长带路 -> Primary 安置`。

### 4.3 需求拆解与执行切片

用户真实在做的不是写空泛策划案，而是把高耦合需求压成可执行切片，例如：

1. 把 `spring-day1` 拆成基础设施、扩展改造、内容配置的分层推进结构。
2. 把 `Town` 开场、围观、散场、带路、`Primary` 安置、`Home` 门合同分成独立可验切片。
3. 把 NPC、UI、存档、云影、导演工具、验证菜单等线程限制在清晰的 owner 边界内。
4. 把“什么算完成、什么不在本轮范围内、什么需要停给用户终验”写进真实推进口径。

### 4.4 AI 协同开发组织

用户负责的不是“调用一个 AI”，而是组织多线程、多角色的 AI 协作链，包括：

1. 用大白话长文本把复杂需求一次性讲透。
2. 让 AI 先复述理解，再进入方案和实现。
3. 自己做线程之间的信息转接、上下文统一和冲突裁定。
4. 审回执质量，而不是接受“做完了”“代码好了”这种空回执。
5. 判断线程该继续、该返工、该停给用户验收，还是该停给用户分析。

### 4.5 Unity 实操与终验

用户真实会进入 Unity 做策划侧落地和验收，包括：

1. 自己做 ScriptableObject 数据配置和内容填写。
2. 自己检查 Prefab、场景引用、对象结构与 Inspector 配置。
3. 自己做 Play 模式试玩，盯交互手感、状态异常和体验粗糙点。
4. 自己判断 UI、相机、门链、提示、工作台、围观、工具态这些玩家面是否真的成立。
5. 高频场景下，自己承担“AI 自测过了，但玩家面仍然不对”的最终纠偏责任。

## 5. 用户已经主导做成的核心成果

### 5.1 跨场景主线切片

当前最值钱的成果，不是“做了很多功能”，而是把 `spring-day1` 推进成了真实的跨场景可玩主入口：

1. `Town` 开场围观不再只是说明性场景，而是进入 resident 聚拢、散场、回家、恢复漫游的正式行为链。
2. `Primary` 不再只是一个地图，而是承接旧屋安置、工作台闪回、农田教学和第一次真实制作的教学主场。
3. `Home` 已接入 `Home <-> Primary` 门合同、入场锚点、固定镜头黑底和持久化玩家承接。

### 5.2 NPC 常驻居民化

用户主导把 NPC 从“任务点”推进到“剧情后仍然活在场景里的居民”：

1. formal 对话只消费一次。
2. formal 已消费后，同 NPC 自动回落到 resident / informal / nearby / selfTalk 等日常结构。
3. prompt、nearby 反馈和对话入口会跟 formal 是否已消费同步改口。
4. 当前 Day1 已先收束为 `7` 人正式 resident 阵容，而不是把全部历史角色都塞进同一轮可验范围。

### 5.3 连续世界与持久化

用户推动的不是孤立存档功能，而是连续世界体验：

1. `Primary / Town / Home` 现在围绕唯一玩家实例推进。
2. 背包、Hotbar、工具态和选中状态开始按“切场前捕获、切场后重绑”推进。
3. `cloudShadowScenes` 已被正式接入 `SaveDataDTOs / SaveManager` 的持久化链。
4. 云影、场景切换、存档和运行态恢复不再各自独立。

### 5.4 交互系统收口

用户主导的交互规则已经形成正式口径：

1. 背包为 `36` 格，前 `12` 格与 Toolbar / Hotbar 同源，不是两套库存。
2. 放置链形成 `Preview -> Locked -> Navigating -> Executing` 的四态主流转。
3. 农田链形成独立 `FarmNavState`，并与放置共享红色无效、绿色重定向、玩家保留撤回权的底层原则。
4. 箱子、工作台、NPC 等世界交互不再靠“射线扫到谁算谁”，而是按交互优先级和距离共同决策。

### 5.5 经营成长与资源博弈

用户不是只照搬模板，而是把一套真实经营成长线压进项目里：

1. 四季经营规则和五阶段植被过渡并行存在。
2. 资源成长不是“采到就结束”，而是围绕工作台链、工具等级、材料等级和资源门槛形成正反馈。
3. 矿物收益由工具等级与矿物层级匹配决定，低等级工具可命中高等级矿，但拿不到真正矿物收益。
4. 农田系统不是单格切换，而是三层结构加 `1 + 8` 邻接逻辑。

## 6. Unity、工具链与实际落地事实池

### 6.1 当前策划侧正式落地链路

`Sunset` 当前不是“少量手填数据 + 少量手摆场景”的开发方式，而是形成了明确的策划侧生产链：

`设计需求 -> SO 数据 -> WorldPrefab / Recipe / Placeable / Storage 等运行时载体 -> Tilemap / Sprite / Renderer / Prefab -> 场景落地 -> 验证菜单与运行态探针`

### 6.2 数据生产工具链

当前已进入正式口径的数据生产工具包括：

- `Tool_BatchItemSOGenerator.cs`
- `Tool_BatchItemSOModifier.cs`
- `Tool_BatchRecipeCreator.cs`

它们服务的真实问题是：

1. 物品 SO 手填成本高。
2. 字段结构调整后逐个返工成本太高。
3. 配方扩张如果完全靠手工，很快会拖垮内容生产效率。

### 6.3 世界资产接线工具链

当前已进入正式口径的世界资产与接线工具包括：

- `WorldPrefabGeneratorTool.cs`
- `BatchCreatePrefabs`
- `BatchSpriteRendererSettings`
- `PrefabDatabaseAutoScanner`
- `PrefabDatabaseEditor.cs`
- `TilemapToSprite.cs`

它们服务的真实问题是：

1. Sprite 资源如何快速变成可运行的世界玩法资产。
2. `WorldPrefab` 引用和数据库维护如何避免漂移。
3. 场景素材如何从“画面元素”转成“可交互、可生成、可掉落、可放置”的玩法对象。

### 6.4 动画生产工具链

当前已进入正式口径的动画生产工具包括：

- `LayerAnimSetupTool`
- `SliceAnimControllerTool`
- `ToolAnimationGeneratorTool`
- `ToolAnimationPipeline`
- `TriDirectionalAnimGenerator`
- `Tool_003_BatchAnimTransitions`

它们服务的真实问题是：

1. 三方向动画配置重复劳动重。
2. 玩家动作、工具动作、过渡状态接线容易错漏。
3. Animator 过渡和命名如果全靠手工，后续扩张会非常脆弱。

### 6.5 场景维护与验证工具链

当前已进入正式口径的场景维护与验证工具包括：

- `Tool_001_BatchProject`

- `Tool_004_BatchTreeState`

- `StoneControllerEditor.cs`
- `CloudShadowManagerEditor.cs`
- `OcclusionManagerEditor.cs`
- `OcclusionTransparencyEditor.cs`

1. 大量场景层级、排序、遮挡、树木阶段、石头阶段不能全靠手调。
2. 场景表现层不是单纯“好不好看”，还必须跟导航、碰撞和交互入口一致。
3. 树木成长、遮挡透明、动态排序、云影和场景连续性是一条表现问题链。

## 8. AI 原生策划推进事实池

### 8.1 用户的真实工作方式

用户的 AI 使用方式有几个关键特征：

1. 不是花哨 prompt 工程，而是高密度大白话长文本。
2. 不是“先让 AI 自己发挥”，而是先讲清楚需求，再要求 AI 复述理解。
3. 不是一个线程从头吞到尾，而是多线程分工，用户自己做信息转接和主线统一。
4. 不是只看产出文档，而是必须经过试玩、回归、no-red 和体验终验。

### 8.2 当前治理体系里的真实构件

用户已经围绕以下层级形成真实治理口径：

1. 规则栈：`AGENTS.md`、规范快照、`Codex规则落地`、`.kiro/steering`
2. 工作区栈：治理工作区 + 业务工作区并行
3. memory 栈：子工作区、父工作区、线程 memory、skill 审计层
4. hooks：规则路由、memory 检查、Git preflight 等前置动作
5. locks：shared root、MCP 单实例和高危现场约束
6. thread-state：`Begin-Slice / Ready-To-Sync / Park-Slice / Show-Active-Ownership`
7. skills：已被本地化进 Sunset 项目生产链

### 8.3 用户在 AI 治理中的真实主控权

用户真实做的事情包括：

1. 判断当前主线是什么，哪些线程必须服从主线。
2. 判断线程是否偏题、是否只停留在方案层、是否开始乱扩功能。
3. 审回执是否说清 `owner / blocker / no-red / 当前阶段 / 下一步`。
4. 判断线程是继续施工、停给用户验收、停给用户分析，还是无需继续。
5. 用真实玩家面、真实场景入口、真实体验反馈继续回拉线程质量。

3. 为什么 `301` 要退出当前 Day1 正式链。
