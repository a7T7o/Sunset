# Sunset 项目三维度母卷

## 1. 文档用途

本文用于作为 `Sunset` 项目对外简历、作品集、网申问答与面试展开的母卷材料。

全文只做一件事：

把 `Sunset` 项目里最值得被后续智能体继续压缩的真实事实，按：

1. 系统策划
2. 技术策划
3. AI 原生开发

三个维度重新组织清楚。

本卷不追求一句话投递成品，而是优先保证：

1. 事实完整。
2. 主控权清楚。
3. 主线收口清楚。
4. 每个亮点都能落回项目真实现场。

## 2. 项目总核心事实

### 2.1 项目基本口径

- 项目名：`Sunset`
- 项目类型：Unity 6 2D 像素风奇幻生活模拟 RPG
- 项目主轴：生活经营 + 资源成长 + 剧情推进
- 核心叙事前提：失忆工匠流落废弃聚落，通过采集、制作、建造与人际连接逐步恢复聚落秩序，并在经营过程中找回记忆
- 当前核心循环：`资源积累 -> 经营生产 -> 能力成长 -> 事件 / 探索解锁 -> 聚落复兴反哺`

### 2.2 用户在项目中的真实身份

用户在 `Sunset` 中的真实身份，不是普通参与者，也不是单纯给 AI 提需求的人，而是：

1. 项目策划主控。
2. 主线推进与设计边界的拍板者。
3. AI 多线程协同开发的组织者与验收者。
4. Unity 玩家面体验与实机回拉的最终责任位。

最准确的理解应该是：

用户不是在“做功能目录”，而是在持续把项目收束到一条真正能玩的主线切片上，并用规则、工具、验收和 AI 协同去保证这条主线不会被局部扩写拖散。

### 2.3 当前最值钱的项目成果

当前 `Sunset` 最值钱的项目成果，不是系统数量，而是三件事已经同时成立：

1. `spring-day1` 已从开篇设想推进成 `Town -> Primary -> Home` 的跨场景主线切片。
2. NPC 已从一次性剧情承载，推进到 formal 一次性消费后仍能继续活在场景里的 resident 结构。
3. 玩家、背包、Hotbar、工具态、云影与存档，开始围绕连续世界体验进入统一运行链。

这意味着 `Sunset` 当前已经不只是“做了很多生活模拟系统”，而是开始拥有一条真正可被体验、可被验证、可被持续扩张的项目主骨架。

## 3. 系统策划维度

### 3.1 主线切片与跨场景推进

用户在系统策划层最核心的工作，不是单独设计某个模块，而是持续主导 `spring-day1` 这条主线切片的推进方式。

当前已经形成的正式口径是：

1. `Town` 负责入村围观、驻村承接、村长带路与后续生活面入口。
2. `Primary` 负责旧屋安置、工作台闪回、农田教学与第一次真实制作。
3. `Home` 已被纳入正式场景体系，负责 `Home <-> Primary` 门合同、入场锚点、固定镜头黑底与室内承接壳。

这件事的价值不在“多了三个场景”，而在于：

1. 开篇剧情、教学、第一次经营行为和连续世界入口被压进了同一条体验链。
2. 剧情不再是脱离玩法的播片，教学也不再是硬塞的功能说明。
3. 后续 NPC 居民化、聚落复兴、自由时段和跨场景运行态，都有了稳定承载体。

### 3.2 需求拆解与切片推进

用户在系统策划上的真实强项，是把高耦合需求拆成能落地、能验收、能回拉的切片。

当前已经形成过的真实推进方式包括：

1. 把 `spring-day1` 拆成基础设施、扩展改造、内容配置三层推进结构。
2. 把 `Town` 开场围观、围观散场、村长带路、`Primary` 安置、工作台、农田、`Home` 门合同拆成独立施工切片。
3. 把“本轮做什么、不做什么、什么算完成、什么要停给用户终验”写进真实推进规则。

这说明用户不是只会提“大方向”，而是已经具备执行策划最核心的能力之一：

把混杂需求压缩成能被团队持续推进的明确结构。

### 3.3 交互系统与状态边界收口

用户在交互策划上的真实工作，不是简单列功能，而是持续定义状态边界和异常处理规则。

当前已经形成的交互正式口径包括：

1. 背包为 `36` 格，前 `12` 格与 Toolbar / Hotbar 同源，不是两套库存。
2. 放置链形成 `Preview -> Locked -> Navigating -> Executing` 的四态主流转。
3. 农田链形成独立 `FarmNavState`，与放置共享红色无效、绿色重定向与玩家保留撤回权的原则。
4. 箱子、工作台、NPC 等世界交互，不再按“谁先被射线扫到就算谁”，而是按交互优先级与距离共同决策。
5. 世界输入、面板输入、剧情输入与工具动作锁之间，已经被收成统一的中断、恢复和互斥逻辑。

这些内容最值钱的地方，不是术语，而是它们直接说明：

用户已经在做执行策划最难的那部分工作，也就是把“玩家实际怎么点、怎么错、怎么被打断、怎么恢复”真正写成规则。

### 3.4 经营成长闭环与资源博弈

用户在经营成长线上的工作，已经超出“照着星露谷搭基础系统”的程度，进入了资源节奏、工具门槛和长期成长链的组织层。

当前已形成的经营成长口径包括：

1. 四季规则与五阶段植被过渡并行存在。
2. 资源成长不是采到就结束，而是围绕工具等级、材料等级、工作台链与可处理资源范围形成正反馈。
3. 矿物收益由工具等级与矿物层级匹配决定，低等级工具可命中高等级矿，但拿不到真正矿物收益。
4. 农田系统不是单格切换，而是三层结构加 `1 + 8` 邻接逻辑。
5. 工作台不是单一 UI 容器，而是资源分流、成长门槛与长期配方扩张的正式装置。

这说明用户在经营策划上的真实能力，不只是“知道应该有农田、矿物、工作台”，而是：

已经能把资源获取、生产转化和成长门槛压回同一条长期循环。

### 3.5 NPC 常驻居民化与剧情消费结构

用户在剧情与 NPC 策划层最有辨识度的工作，是推动 NPC 从“剧情功能点”转向“剧情之后还能继续活在场景里的居民”。

当前真实成立的机制包括：

1. formal 对话只消费一次。
2. formal 已消费后，同 NPC 自动回落到 resident / informal / nearby / selfTalk 等日常结构。
3. prompt、nearby 反馈和对话入口会随着 formal 是否已消费同步改口。
4. 当前 Day1 主链已经收束为 `7` 人正式 resident 阵容，而不是把所有历史角色一股脑留在同一轮可验范围。

这组设计的价值非常高，因为它说明用户不是只在“做主线剧情”，而是在思考：

1. 剧情结束以后，世界还怎么继续活着。
2. 聚落复兴如何从“设施可用”推进到“人真的开始过日子”。
3. 玩家感知到的不是任务节点，而是一个逐渐恢复秩序的生活空间。

### 3.6 主线裁定与范围收缩能力

系统策划层还有一个非常值钱、但最容易被外部忽略的能力点：

用户会主动收范围，而不是一味加内容。

最典型的例子就是：

1. 当前 Day1 正式主链已经明确收束为 `Town 开场围观 -> 村长带路 -> Primary 安置`。
2. `301` 被正式裁定退出当前 Day1 主链。
3. `Home` 被纳入正式场景体系，但当前仍坚持“室内承接壳”口径，不把它硬写成完整玩法场。

这类裁定非常能体现执行策划成熟度，因为它说明用户清楚：

项目推进不靠一直加东西，而靠持续把当前主线收稳。

## 4. 技术策划维度

### 4.1 Unity 策划侧全链路落地意识

用户在技术策划维度最核心的价值，不是“自己写了多少代码”，而是已经建立了很完整的策划侧落地链路意识。

当前 `Sunset` 的策划侧正式落地链路可以概括为：

`设计需求 -> SO 数据 -> WorldPrefab / Recipe / Placeable / Storage -> Tilemap / Sprite / Renderer / Prefab -> 场景落地 -> 验证菜单与运行态探针`

这条链路的重要性在于：

1. 它说明用户不是停在概念设计层。
2. 它说明用户知道数据如何变成玩法对象，玩法对象如何进入场景，场景如何再进入玩家面验收。
3. 它说明用户已经能从“设计正确”继续追到“资源组织、场景承接、运行态反馈是否真的成立”。

### 4.2 数据资产治理与 SO 体系

用户在技术策划层的一个非常硬的事实，是项目已经形成了清晰的数据资产结构与 SO 治理意识。

当前真实存在的内容包括：

1. `111_Data` 下已经按 `Tools / Weapons / Materials / Seeds / Crops / Saplings / Placeable / Storage / NPC / Recipes / Story / Dialogue` 等方向细分。
2. 项目已经存在正式的 ID 分段规范、命名规范和字段调整治理口径。
3. 数据资产不是孤立摆放，而是围绕 `ItemData / ToolData / WeaponData / SeedData / CropData / PlaceableItemData / RecipeData / NPCDialogueContentProfile` 等类型进入运行时。

这类工作对策划岗位的价值在于：

它直接说明用户已经理解“设计内容要可扩张、可维护、可批量生产”，而不是停在一次性配置层。

### 4.4 场景、Tilemap 与空间可读性

用户在技术策划维度里，确实碰到了很实在的 Unity 场景组织问题，而不是只会做平面设计。

当前已形成正式口径的内容包括：

1. 多层 Tilemap 结构与场景功能分层。
2. 从 Sprite 资源到 Tilemap 的导入、配置和场景绘制。
3. 围绕功能层、装饰层、交互层、遮挡层的场景组织。
4. 导航、遮挡、动态排序与玩家视角可读性的联动。

这说明用户不是单纯“会摆图”，而是已经在处理生活模拟项目里很关键的一类技术策划问题：

让场景既能表达空间，又不破坏交互和导航。

### 4.5 动画、表现层与玩家面验收

用户在表现层上的真实工作，也明显超出了“会简单改改素材”的程度。

当前真实推进过的内容包括：

1. 三方向人物与工具动作的组织、接线与验收。
2. 帧索引硬锁同步、动作锁保护与人物 / 工具动作联动问题。
3. `DialogueUI`、`ItemTooltip`、`Workbench Overlay`、世界提示和状态条在玩家面上的可读性。
4. 遮挡透明、云影、动态排序和场景连续性的统一处理。

这类事实说明用户已经进入一种很像技术策划 / 实现策划的工作状态：

不是只关心规则写没写，而是会一直追到“玩家真正看到的东西是否清楚、是否不穿帮、是否能读懂”。

### 4.6 连续世界、持久化与运行态一致性

技术策划层最能体现项目成熟度的一块，是用户推动的不是单点存档，而是连续世界口径。

当前真实成立的内容包括：

1. `Primary / Town / Home` 正在围绕唯一玩家实例推进。
2. 背包、Hotbar、工具态和选中状态开始按切场前捕获、切场后重绑推进。
3. `cloudShadowScenes` 已进入 `SaveDataDTOs / SaveManager` 的持久化链。
4. 云影、场景切换、运行态恢复与存档不再各自为战。

这组事实的价值很高，因为它们证明用户在技术策划上已经开始处理：

“一个连续世界项目中，不同系统如何围绕同一批状态继续工作”。

这不是普通学生项目里很常见的能力点。

## 5. AI 原生开发维度

### 5.1 用户不是“会用 AI”，而是在主控 AI

这一维度最核心的一句话，不是“用户用了 AI”，而是：

用户已经把 AI 变成了自己作为策划主控推进项目的正式开发方式。

这件事之所以重要，是因为用户在项目里承担的，不是“让 AI 写点东西”，而是：

1. 用自己的需求理解定义项目主线。
2. 把复杂问题拆给不同线程。
3. 让 AI 先理解，再产出。
4. 自己持续审回执、做验收、裁定返工与停发。

### 5.2 真实工作方式：大白话长文本 + 认知对齐

用户在 AI 协作里的真实工作方式非常鲜明：

1. 不是依赖花哨 prompt 工程。
2. 而是用高密度大白话长文本把复杂需求一次性讲透。
3. 不是直接接受 AI 输出，而是先要求 AI 复述理解，再进入方案和实现。
4. 不是让 AI 自己决定项目方向，而是自己持续做设计边界与主线锚定。

这非常值钱，因为它说明用户的 AI 使用并不是“技巧性调用”，而是真正的策划主控行为。

### 5.3 多线程治理架构已经进入项目正式口径

当前 `Sunset` 的 AI 协同，已经形成相对完整的治理架构，而不是零散试验。

当前真实存在并已进入正式口径的层级包括：

1. 规则栈：
   - 用户裁定
   - `AGENTS.md`
   - 规范快照
   - `Codex规则落地`
   - `.kiro/steering`
2. 工作区栈：
   - 治理工作区
   - 业务工作区
3. memory 栈：
   - 子工作区 memory
   - 父工作区 memory
   - 线程 memory
   - skill 审计层
4. 现场约束层：
   - hooks
   - locks
   - thread-state
5. skills 本地化适配层：
   - `skills-governor`
   - `sunset-workspace-router`
   - `sunset-no-red-handoff`
   - `sunset-warden-mode`
   - `sunset-acceptance-handoff`
   - 以及其他项目专用能力

这组东西的价值，在于它们直接证明：

用户不是单纯“开几个聊天窗口让 AI 干活”，而是在搭一套能长期推进项目的协作秩序。

### 5.4 标准化推进流程已经形成

当前项目里的 AI 协同，已经形成一条相对稳定的推进流程：

1. 需求定义。
2. 项目级前置核查。
3. 命中工作区与规则。
4. 压成单轮可验切片。
5. `Begin-Slice` 进入真实施工。
6. CLI 快验证。
7. 必要时 direct MCP / live 取证。
8. `Ready-To-Sync` 或 `Park-Slice`。
9. 对用户做人话汇报。
10. 典狱长或看守长决定继续、停发或终验。

这条流程能体现的，不只是“AI 工作流成熟”，更重要的是：

用户已经知道怎样把 AI 的产出约束到真正服务项目推进，而不是无限展开。

### 5.5 典狱长、看守长与继续 / 停发裁定

用户在 AI 原生开发这一维度最独特的点之一，是已经形成了明确的治理角色分流：

1. 业务线程负责施工与对用户的人话汇报。
2. `典狱长` 负责审回执、判线程是继续施工、停给用户验收、停给用户分析还是无需继续。
3. `看守长` 负责把某一刀真正收成用户可执行的验收包。

这类设计的实际价值非常高，因为它们说明：

用户不仅让 AI 做事，还在持续解决一个更难的问题：

当多个线程并行推进时，谁该继续、谁该停、谁该交验，不能靠感觉，只能靠正式规则和正式裁定。

### 5.6 验证体系：CLI 优先，MCP 兜底

用户在 AI 原生开发上的另一条非常有辨识度的事实，是已经形成了明确的验证口径：

`CLI first, direct MCP last-resort`

这条规则的意义不只是工具偏好，而是项目治理思想：

1. 高频验证先追求快和稳定。
2. 先快速判断当前是 own red、external red 还是现场不稳。
3. 只有进入 Scene / Play / Inspector / runtime flow / GameView 这种必须 live 看真相的场景，才升级 direct MCP。

这非常能体现用户的成熟度，因为它说明：

用户不是盲目追求“工具越重越专业”，而是知道不同验证方式应该服务什么目的。

### 5.7 AI 治理已经真实改变了项目推进方式

当前最重要的不是用户建立了多少名词，而是这套 AI 协同方式已经真实改变了项目推进方式。

最直接的表现包括：

1. 项目主线不再由局部线程各自扩写，而是持续服从当前主线和场景合同。
2. resident 化、唯一玩家、`Home` 定位、`301` 退出主链等拍板，都会直接改变线程边界和后续实现。
3. no-red、回执质量、玩家面反馈与终验结果，已经进入正式推进链。
4. 项目推进不再只是“线程说自己完成了什么”，而是“策划主控先定义边界和完成标准，再由线程执行并通过终验继续收束”。

这一点非常值得后续对外材料重点保留，因为这正是用户和普通“会用 AI 的学生”之间最本质的差别。

## 6. 三个维度之间的共同主线

如果把系统策划、技术策划和 AI 原生开发三个维度放在一起看，用户当前在 `Sunset` 中最清晰的总体特征是：

1. 不是功能堆积型策划，而是主线收口型策划。
2. 不是纯概念型策划，而是会一直追到 Unity 场景、表现层和玩家面验收的落地型策划。
3. 不是普通 AI 使用者，而是以策划身份主控 AI 协同开发的人。

因此，后续无论是简历、作品集还是面试稿，都不应该把用户压成：

- 会很多功能的普通学生
- 会一点 Unity 的策划
- 会用 AI 提效的人

而应该始终围绕以下母本来继续压缩：

**一个正在用 Unity、规则收口、工具链和 AI 协同，把生活模拟 RPG 的第一条跨场景主线真正打通的策划主控。**
