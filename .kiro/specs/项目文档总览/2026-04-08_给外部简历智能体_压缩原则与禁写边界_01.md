# Sunset 对外压缩原则与禁写边界

## 1. 使用目标

本文用于指导外部简历智能体、作品集整理位和面试问答整理位，如何从 `Sunset` 项目事实池中做压缩，而不把用户的真实价值写偏、写弱或写假。

## 2. 总压缩原则

### 2.1 先立身份，再写模块

外部智能体写材料时，顺序必须是：

1. 先写用户在项目里的主控身份。
2. 再写用户主导做成的关键结果。
3. 最后才写支撑这些结果的系统、工具链和技能。

不能一上来就写“会 Unity、会 Tilemap、会 AI、会很多工具”，那样会把用户写成普通技术杂项型学生。

### 2.2 先写主线收口，再写系统数量

用户最值钱的地方，不是“做了很多系统”，而是：

1. 把 `spring-day1` 推成真正可玩的跨场景主线切片。
2. 把 `Town / Primary / Home`、NPC resident 化、唯一玩家、持久化和终验收口压进同一条体验链。

所以压缩时必须优先写：

- 主线
- 场景承接
- resident 化
- 连续世界
- AI 主控与验收治理

而不是平均平铺十几个系统名词。

### 2.3 先写主控权，再写协作

外部智能体必须把用户写成：

- 主导者
- 拍板者
- 验收者
- 回拉者

不能把用户写成：

- 辅助沟通的人
- 帮 AI 整理文档的人
- 单纯“和 AI 协作过”的普通使用者

## 3. 简历、作品集、面试三层压缩口径

### 3.1 简历层

简历层只保留最硬的 `3 ~ 4` 个点，优先级建议如下：

1. `spring-day1` 跨场景主线切片：`Town -> Primary -> Home`
2. NPC resident 化与正式剧情一次性消费后的日常回落
3. 唯一玩家跨场景连续体验与运行态持久化推进
4. 以策划身份主控 AI 协同开发，负责切片拆解、认知对齐、回归验收和继续/停发裁定

简历层不要塞太多黑话，不要把 `threads / specs / hooks / memory / steering` 原封不动堆进去。

### 3.2 作品集层

作品集层可以展开复杂度，但必须用 case 讲，不要写成治理流水账。

优先展开：

1. `spring-day1` 如何从概念切片变成跨场景正式主入口
2. NPC resident 化如何改变 Day1 的场景承接和居民结构
3. 连续世界如何把玩家、背包、工具态、云影和存档接成同一条链
4. 策划侧工具链如何服务数据扩张、资产接线、动画生产和验证回归
5. AI 治理如何把大白话需求压成真实切片，并把线程收口到验收标准

### 3.3 面试层

面试层必须保留真实工作方式和判断链，尤其要保留：

1. 大白话长文本需求如何做认知对齐
2. 为什么先做 `Town -> Primary -> Home`
3. 为什么要做 resident 化
4. 为什么要做唯一玩家
5. 为什么坚持 `CLI first`
6. 怎么判断线程该继续、该停发、该返工

面试层可以适度讲治理细节，但必须始终落回“你为什么这样判断”和“它怎样改变了项目推进”。

## 4. 可写、慎写、禁写

### 4.1 可直接写

以下内容可以直接进入对外材料：

1. Unity 6 2D 像素风生活模拟 RPG《Sunset》
2. `spring-day1` 跨场景主线切片
3. `Town / Primary / Home` 场景分工
4. NPC resident 化与正式剧情一次性消费
5. 唯一玩家跨场景承接
6. 背包 / Hotbar / 工具态连续体验
7. `cloudShadowScenes` 等运行态接入正式持久化链
8. 策划侧 Editor 工具链
9. 以策划身份主控 AI 协同开发
10. 需求拆解、回归验收、返工裁定、继续/停发判断

### 4.2 慎写

以下内容可以写，但要做翻译和降噪：

1. `Unity MCP`
2. `skills`
3. `hooks`
4. `memory`
5. `thread-state`
6. `steering`
7. `典狱长`
8. `看守长`

这些内容更适合翻译成：

- 协作规范
- 本地化能力模块
- 线程状态管理
- 验收与治理机制
- AI 协同工作流

除非在作品集或面试中，不要直接把内部术语原封不动塞进简历。

### 4.3 禁写

以下内容不能写：

1. 用户独立完成了全部程序开发。
2. 用户是成熟商业项目主策。
3. 用户主导完成了完整商业化项目。
4. 所有 Editor 工具都是用户一个人亲手写完全部代码。
5. `Unity MCP` 是项目默认主验证链。
6. resident 化只是普通内容扩写，没有改变项目主线。
7. `301` 退出当前 Day1 正式链只是无关紧要的小调整。
8. 用户只是“会用 AI 提高效率”。

## 5. 绝对不能削弱的重点

### 5.1 主控权

必须写出：

1. 用户拥有主线拍板权。
2. 用户拥有边界定义权。
3. 用户拥有终验权。
4. 用户拥有继续 / 停发 / 返工判断权。

### 5.2 主线收口

必须写出：

1. 用户做的不是功能堆积，而是主线收口。
2. 用户不是做模块目录，而是在把很多模块压进同一条连续世界体验链。

### 5.3 AI 原生策划

必须写出：

1. 用户不是把 AI 当黑盒工具，而是在组织一套推进方法。
2. 用户真实建立了需求定义、认知对齐、切片推进、回归验收、停发裁定这套链路。

## 6. 最容易把用户写弱的错误写法

### 6.1 平均写法

错误倾向：

- 会 Unity
- 会 AI
- 会写文档
- 会做游戏策划

这种写法会把用户压成普通实习生模板。

### 6.2 工具堆砌写法

错误倾向：

- 会 `threads / specs / memory / hooks / steering / skills / MCP`

如果没有翻译成项目价值，这会像黑话堆砌，反而削弱真实成果。

### 6.3 纯功能清单写法

错误倾向：

- 做了背包
- 做了放置
- 做了农田
- 做了工作台
- 做了箱子

这样会让人感觉只是把星露谷类系统抄了一遍，看不出主控权，也看不出收口能力。

### 6.4 纯夸张写法

错误倾向：

- 精通 Unity 工业化研发
- 独立完成多智能体架构研发
- 国内最早提出某某方法论

这类写法风险太高，和用户当前大三下、策划实习岗投递定位不匹配。

## 7. 高频追问与真实回答方向

### 7.1 你到底怎么主导 AI，而不是被 AI 带着走

回答方向：

1. 需求先由我用大白话长文本讲透。
2. 我先看 AI 是否真的理解，再让它复述。
3. 我把任务拆成不同线程，各自负责不同切片。
4. 我看回执质量、试玩反馈和 no-red 状态，再决定继续还是返工。

### 7.2 为什么要先做 `Town -> Primary -> Home`

回答方向：

1. 因为当前不是拼系统目录，而是要先把第一条跨场景可玩主线真正打通。
2. `Town`、`Primary`、`Home` 分别承接不同叙事和教学功能，合在一起才是 Day1 的真实入口。
3. 只有这条主线成立，后面居民化、聚落秩序、经营节奏和连续世界才有真正载体。

### 7.3 为什么要做 resident 化

回答方向：

1. 不想让 NPC 只是剧情播报器。
2. 正式剧情结束后，NPC 应该继续活在场景里，变成聚落秩序的一部分。
3. 这会直接改变 Day1 结束后的世界观感和后续生活面的真实感。

### 7.4 为什么要做唯一玩家跨场景承接

回答方向：

1. 因为背包、Hotbar、工具态和场景切换不能每次都重置。
2. 如果玩家实例不连续，很多体验会断掉，连续世界也会变假。
3. 这是把多个系统压回同一条体验链的基础。

### 7.5 为什么坚持 `CLI first, direct MCP last-resort`

回答方向：

1. 高频验证要先追求稳定和快。
2. CLI 更适合先判断当前是 own red、external red 还是现场不稳。
3. 只有 scene / play / inspector / runtime flow 这种必须 live 看的情况，才升级到 direct MCP。

## 8. 给外部智能体的推荐工作顺序

如果外部智能体要基于这批材料继续加工，建议顺序固定为：

1. 先读 `Sunset项目素材总包`
2. 再读本文的压缩原则和禁写边界
3. 先出一版“身份 -> 成果 -> 支撑能力”的简版摘要
4. 再分别压成：
   - 简历项目经历
   - 作品集 case 目录
   - 面试问答库

不建议一上来就直接写终稿简历，因为那样很容易把用户的真实价值压散。

## 9. 本文与素材总包的分工

本文负责：

1. 说明怎么压缩。
2. 说明哪些能写，哪些不能写。
3. 说明怎样避免把用户写偏、写弱或写假。

素材总包负责：

1. 提供真实事实池。
2. 提供真实职责和成果。
3. 提供可展开的 case 与问答话题。
