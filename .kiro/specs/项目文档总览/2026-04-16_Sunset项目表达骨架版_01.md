## 2. 一句话总判断

`Sunset` 当前最值钱的，不是“做了很多系统”，而是：

`你已经把导演剧情、NPC/玩家运行、交互优先级、背包/Hotbar、放置/农田、工作台/箱子、时间季节、正式存档与跨场景承接，压进同一条可以跨场景续跑的 Day1 主线。`

所以它不能再被写成“几个常见系统并列堆起来的学生 Demo”。

更准确的理解是：

`这是一个已经进入多系统运行底座 + Town -> Primary -> Home 主线承接 + 连续世界状态继承 + resident 聚落秩序收口阶段的 Unity 6 项目。`

## 3. 推荐项目简介

### 3.1 当前推荐版

《Sunset》是一款基于 Unity 6 开发的 2D 像素奇幻生活模拟 RPG。项目以“失忆工匠推动聚落复兴”为叙事核心，我以策划主控身份负责项目定位、系统设计、需求切片、规则裁定与验收回拉，AI 协同完成实现落地。当前已将导演剧情、NPC/玩家运行、背包与交互、放置与农田、工作台与配方、时间季节、存档与跨场景承接等系统接入同一条 `Town -> Primary -> Home` 的 Day1 可玩主线，并持续收口连续世界与居民运行时边界。

### 3.2 这段简介要承担的价值

这段简介不是用来“讲全项目故事”的，而是用来先钉死 5 件事：

1. 这是生活模拟 RPG，不是泛 2D Demo
2. 你是策划主控，不是辅助记录者
3. 项目已经从“单点系统可用”进入“多系统主线承接”
4. 项目难点在运行时承接和边界收口，不在单点功能有没有
5. AI 只是推进方式，不是项目主体

## 4. 推荐成果骨架

下面的结构，当前推荐按：

- `4 条主成果`
- `1 条备选展开`

来理解。

如果后面是一页偏紧版本，就优先保留前 4 条。

如果是两页或作品集展开版，再考虑第 5 条。

### 4.1 主成果一｜从 0 到 1 搭建多系统运行底座

【从 0 到 1 搭建多系统运行底座】围绕 `Town -> Primary -> Home` 的 Day1 主线，打通导演与剧情、玩家/NPC 运行、背包/UI 交互、放置与耕地、工作台与配方、时间季节、树石资源、正式存档与场景持久化等关键运行链；让原本分散的系统从“单点可用”进入同一条可连续承接的主线流程。

**这条真正要打的价值：**

1. 不是“我做了很多模块”
2. 而是“我把很多模块真正接到一起跑了起来”
3. 这是你从 0 到 1 的核心证明

### 4.2 主成果二｜把共享状态语言压进高频交互与成长链

【把共享状态语言压进高频交互与成长链】不是把背包、农田、放置、工作台分别做完，而是统一输入抢占、目标选择、导航执行、取消恢复与状态继承规则；落地 `36` 格背包 + `12` 格 Hotbar 同源结构、放置/农田的 `Preview -> Locked -> Navigating -> Executing` 执行链，以及工作台/箱子/手持态共享取材与恢复逻辑，减少多系统并联下的断链、抢状态与体验跳变。

**这条真正要打的价值：**

1. 体现你不是“功能堆叠型策划”
2. 而是“知道多系统怎么共享状态、怎么互不打架”
3. 这是系统设计能力最该亮的一条

### 4.3 主成果三｜建立连续世界与居民承接

【建立连续世界与居民承接】推动正式存档与跨场景 runtime 承接分层落地：`SaveManager` 负责正式槽位与 `scene-aware` 读写，`PersistentPlayerSceneBridge` 负责 `Town / Primary / Home` 间的玩家、inventory、hotbar、input、placement 与部分 resident/world runtime 续跑；让切场与读档后的世界不再频繁“重开”，也让 Day1 与聚落居民状态具备连续性。

**这条真正要打的价值：**

1. 体现项目不是“切场后重置”的拼接 Demo
2. 体现你理解正式存档层和跨场景承接层的差异
3. 这是连续世界感最硬的一条

### 4.4 主成果四｜以策划主控组织多 Agent 并行开发与验证闭环

【以策划主控组织多 Agent 并行开发与验证闭环】围绕需求定义、范围裁定、语义拍板、回执审核、黑盒回归与终验回拉，持续组织 AI 协同开发；并将导演验证、运行态探针、导航/NPC/存档校验与高频配置工具接入同一套生产/验证流程，使项目在高频迭代下保持可验、可改、可收口。

**这条真正要打的价值：**

1. 体现你不是“会用 AI”，而是“能主控 AI”
2. 体现你会把 AI 产出压回可交付项目
3. 这是你和普通应届生真正拉开差距的一条

### 4.5 备选展开｜收束高耦合 runtime owner 与系统边界

【持续收束高耦合 runtime owner 与系统边界】围绕 Day1 导演层、NPC resident、导航执行、story actor 与夜间 schedule 的职责边界持续做裁定与重构判断，把项目难点从“功能能不能搭起来”推进到“运行时 owner 是否清晰、问题是否能定位、系统是否能长期稳定扩展”的工程状态。

**这条适合什么时候用：**

1. 两页版空间够时
2. 面试官明显更关心“你怎么处理复杂系统边界”时
3. 作品集或面试展开时

**这条不适合什么时候硬塞：**

1. 一页简历空间很紧时
2. 前面 4 条已经够满时
3. 招聘方不一定能快速读懂 `owner` 这类表达时

## 5. 当前推荐顺序

如果后面直接进入简历正文，我当前推荐的顺序是：

1. 项目简介
2. 从 0 到 1 搭建多系统运行底座
3. 把共享状态语言压进高频交互与成长链
4. 建立连续世界与居民承接
5. 以策划主控组织多 Agent 并行开发与验证闭环

如果后面空间足够，再补：

6. 收束高耦合 runtime owner 与系统边界

这个顺序的原因很明确：

1. 先让人知道你把项目跑起来了
2. 再让人知道你不是堆功能，而是会做系统承接
3. 再让人知道你已经有连续世界和 resident 承接
4. 最后再抬你的 AI 主控方式

也就是说：

`先项目本体，后推进方式。`

## 6. 当前必须废掉的旧写法

下面这些写法，后面继续判定为禁写或慎写：

### 6.1 黑话类

1. `resident 化`
2. `formal 一次性消费`
3. `runtime 语义`
4. `placement 公共链`
5. `hooks / steering / memory / thread-state`
6. `Begin-Slice / Ready-To-Sync / Park-Slice`

### 6.2 漂移类

1. 把 `Day1` 写成“只剩一点终验的收尾项目”
2. 把 `存档` 写成“完整通用全世界态持久化”
3. 把项目写成“背包、农田、箱子、工作台、存档五个并列模块”
4. 把 AI 协同写成内部工具名和流程名堆叠

### 6.3 自嗨类

1. “做了很多系统”
2. “做了很复杂的架构”
3. “有完整工业化流程”
4. “效率提升很多”

这些都不是不能讲，而是不能这么空着讲。

后面必须全部落回：

1. 具体做成了什么
2. 解决了什么问题
3. 项目因此进入了什么阶段
