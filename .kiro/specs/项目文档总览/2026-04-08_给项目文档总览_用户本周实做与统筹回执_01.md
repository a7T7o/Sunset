## 2. 本周用户的真实主控角色

这一周里，用户在 `Sunset` 的角色已经明显不只是“提需求的人”，而是同时承担了四种实际职责：

### 2.1 主线导演

用户持续把全项目主线钉在：

- `spring-day1` 的真实可玩推进
- `Town / Primary / Home` 三场景承接
- 聚落从“开篇演示”走向“有后续可消费结构”的运行态

用户多次明确阻止线程漂回：

- UI 美化自嗨
- NPC 大而全扩写
- Primary 总清扫
- 只写分析不落地
- 只讲边界不进 runtime / scene-side

用户对主线的真实要求，是：

> 不是做更多，而是把“当前最值钱、最会影响后续消费”的那一段狠狠干到底。

### 2.2 典狱长 / 看守长式治理主控

本周用户不是被动看回执，而是反复对线程提出了很强的治理要求：

- 不允许漂移
- 不允许只交技术 dump
- 不允许用“线程很多”掩盖责任
- 不允许把 shared dirty 假装成自己完成
- 不允许把工具超时包装成“已经无红错”
- 不允许线程各说各话，必须把 owner / blocker / 提交边界说清楚

用户尤其推动了两条治理升级：

1. `爆红规范` 必须真正落地，不能只停在口号
2. `CLI first, MCP last-resort` 必须成为实际工作流，而不是一句建议

### 2.3 真实验收者

这一周很多关键推进不是代码自己宣布“完成”，而是用户通过真实 live 反馈把线程继续往正确方向压：

- `Town` 相机是否真正跟随
- `Home` 的镜头是否符合构图
- 玩家出门后的落点是否合理
- `Town` 耕地、放置、背包、工具是否真的能玩
- NPC 气泡是不是还是方泡
- Workbench 主条、悬浮卡、文案和颜色是不是玩家面真能接受
- 边界 HUD 是否遮挡内容

用户不是只说“有问题”，而是持续提供：

- 现象顺序
- 哪个场景、哪个入口
- 具体视觉结果
- 体验上“对/不对”的判定

这使很多线程从“静态上看像对”被继续推进成“玩家面真的过线”。

### 2.4 设计总控与系统策划拍板者

这一周里，多条关键设计方向都是用户亲自拍板后才真正稳定下来的。

这些拍板不是小修小补，而是直接改变了项目后续结构：

- `Town` 不再只是剧情说明位，而是要承担 Day1 后续生活面的承接层
- `spring-day1` 从 runtime 临时演员思路，转向“驻村常驻化”
- `Home` 不是临时屋内壳，而是要成为和 Town 同属可承接的场景
- 玩家应该走向跨场景唯一实例持久化，而不是每个场景各摆一个玩家
- 相机是否持久化要谨慎区分，不是所有东西都做成常驻
- `Home` 镜头改判为固定构图、黑底、尊重手摆，而不是默认跟随
- `Town/Home` 的 UI 结构不应是“所有 UI 全局化”，而应是“核心 UI 全局 + 场景本地扩展”
- `NPC 301` 先从当前 Day1 主链全面弃用，不允许继续半在半不在
- 中文字体链、验证链、红错链都必须收成能复用的项目级规则

## 3. 本周最重要的项目级成果，不是单点功能，而是 6 条新基线

### 3.1 Day1 已从“开篇孤立剧情”推进到跨场景承接主线

本周 `spring-day1` 这条线已经不再只是一个 `opening` demo。

它已经形成了明确的跨场景运行结构：

- `Town`：围观、驻村承接、生活面入口
- `Primary`：旧屋安置、工作台、农田教学、带路后续
- `Home`：屋内承接、休息、门链与后续居住空间

更关键的是，这条线不是只停在剧本文案，而是已经开始要求：

- 导演排练
- runtime 承接
- scene-side anchor / resident root
- 玩家真实门链
- HUD / Prompt / Dialogue / 工具态跨场景可继续工作

### 3.2 `Town / Primary / Home` 三场景开始被当成一个系统，而不是三个孤岛

这一周最明显的结构升级之一，是用户持续要求把三场景放到同一运行合同里看：

- 玩家是否唯一
- 切场后背包是否保留
- Hotbar / 工具状态是否保留
- 镜头逻辑是否合理
- UI 是否继续可用
- Door / EntryAnchor 是否明确
- 出门落点与朝向是否能解释

这让 `Town/Home/Primary` 从“各自能打开”推进到“是否属于同一连续世界”的思路。

### 3.3 NPC 从“先多做几个人”变成“formal consumed -> resident fallback”的长期结构

本周 NPC 线的真正升级不只是补了文本，而是明确形成了：

- formal 一次性消费
- resident 常驻 fallback
- phase-aware informal / nearby / selfTalk / walkAway
- resident-director bridge
- 气泡/提示/导演与 runtime 的接口分工

这意味着项目不再把 NPC 只当成一次性剧情演员，而是开始进入“会留在村子里继续被生活面消费”的结构。

### 3.4 Workbench / Prompt / Tooltip / 气泡进入了真正的“玩家面终验”阶段

这一周 UI 线不是简单修一个 prefab，而是被用户不断拉回玩家侧真实体验：

- 主进度条是否真有动画
- 条内文字是否真绑定正确
- 完成态颜色是否可读
- ready 未领取时是否真的阻断继续制作
- 悬浮卡是否在完成后继续保留
- NPC 气泡为什么还是方泡
- Tooltip 延迟/状态条分流是否合理

这说明项目 UI 已经从“功能在”走向“玩家看起来到底对不对”。

### 3.5 存档、云影、持久化玩家三条链开始汇入“运行态延续”这一个主题

这一周用户推动出的不是三个孤立修复，而是一条更高层的共同方向：

- `Persistent player`：唯一玩家跨场景
- `CloudShadow`：分场景运行态缓存，后续接正式存档
- `SaveManager / SaveDataDTOs / StoryProgressPersistence`：正式把跨系统状态接入保存

这让项目第一次比较清晰地朝“不是只在当前场景活着，而是世界状态能延续”这个方向走。

### 3.6 AI 治理与工具链从“能用”推进到“开始反过来改变开发节奏”

这周用户强行推进并坐实了以下几件事：

- `爆红规范` 必须收成高频动作，不允许继续“先写红一片再收”
- `UnityMCP -> CLI` 不是空研究，而是要真正服务高频 no-red
- `validate_script / compile-first / external_red / own_red` 的口径必须落地
- 线程的回执、验收、继续 prompt、停发，不再只靠聊天推断

换句话说，本周 AI 治理最大的变化不是“文档更完整”，而是：

> 它开始真实约束开发行为本身。

## 4. 按线程看，本周用户到底推动了什么

下面不是“线程做了什么”的流水账，而是“用户在这条线程上主导达成了什么”。

### 4.1 `Codex规则落地 / 典狱长 / Town-Home-Primary`

这是本周最强的主控位之一。

用户在这条线上做的事包括：

1. 逼着治理线程不再只说边界，而是真的去承担 `Town` 承接层的建设责任
2. 反复要求：
   - 先厘清 owner
   - 再拆 blocker
   - 再决定谁继续、谁停、谁无需继续发
3. 亲自拍板 `Town` 不只是说明位，而要承担 Day1 后续生活面承接
4. 亲自拍板 `Home` 也要进入这套承接系统，不只是屋内空壳
5. 强行把“每个场景一个玩家”的临时结构，往“唯一玩家跨场景保留”推进
6. 持续用 live 反馈把 Town/Home/Primary 的真实问题钉出来：
   - 相机
   - 门链
   - 入场落点
   - 朝向
   - HUD 遮挡
   - 背包延续
   - 工具链
   - scene baseline

这条线的结果是：

- `Town` 进入 resident / scene-side / runtime contract 语义
- `Home` 从“有房间”推进到“是场景合同的一部分”
- `Primary/Town/Home` 被拉进同一连续体验讨论

### 4.2 `spring-day1`

用户这周没有把 Day1 当成“写几段剧情”，而是把它当成整个项目第一条真实导演主线。

用户在这条线上的主导包括：

1. 一再阻止线程回漂到：
   - UI
   - NPC 大文件
   - Town 总历史
   - 只写 staging 结构不进 live
2. 要求导演工具不是摆着看，而是必须能：
   - 选 cue
   - 开始排练
   - 接管 NPC
   - 回写 stage book
3. 推动 Day1 从前半段开篇扩展到：
   - Town 开场围观
   - 带路
   - Primary 旧屋安置
   - Workbench
   - 晚饭 / 夜晚 / 回屋提醒
4. 后续又亲自拍板：
   - 不再按 runtime 临时演员思路推进 101~301
   - 改判为“驻村常驻化”
5. 对 `301` 做出明确产品判断：
   - 当前 Day1 先全面弃用
   - 不允许半退半留

结果不是“剧情更多了”，而是 Day1 真的开始成为：

- 主线教学入口
- 场景承接入口
- NPC resident 化入口
- 导演工具真实消费入口

### 4.3 `UI`

用户这周对 UI 的最大贡献，不是“提了几个审美意见”，而是强行把 UI 线程拉到玩家面终验层。

用户反复指出的不是抽象风格，而是具体体验断点：

- 主进度条没动画
- 条内文字没绑定对
- 完成态黄条不可读
- 悬浮卡完成后消失
- 气泡还是方泡
- 缺字
- 打包后中文/TMP 仍不稳

用户还不断强调：

- 不要假装过线
- 不要拿技术 dump 代替玩家面
- 不要把“结构成立”包装成“体验已过”

这让 UI 线真正从“做壳”进入“做玩家能接受的体验”。

### 4.4 `NPC`

用户在 NPC 线上的推动重点，不是“加人”，而是建立长期可消费结构。

用户推动形成的核心判断包括：

- `formal consumed -> resident fallback`
- resident 必须能继续在场景中被生活面消费
- 导演线与 NPC 线要协作，但不要互相吞 owner
- 气泡、interaction、relation、day1 正文要分层
- `301` 当前阶段明确退出 Day1 正式链

更重要的是，用户把 NPC 从“剧情演员”往“驻村居民”这个方向真正推过去了。

### 4.5 `存档系统`

用户在存档线上的主导，不是单纯说“加个保存”，而是明确要求它服务于跨场景连续体验。

这周和用户拍板直接相关的方向包括：

- 唯一玩家跨场景后，背包 / Hotbar / 工具态不能丢
- `cloudShadowScenes` 这种运行态缓存值得继续接到正式存档
- 存档不仅要面向玩家读档，也要支撑作者工作流
- 箱子默认内容不能只靠运行时临时塞，作者态要有真正入口

这让存档系统开始同时服务：

- 玩家连续性
- 运行态恢复
- 作者配置效率

### 4.6 `云朵与光影`

用户在这条线上的主导特别能体现“拍板 + 止损”能力。

最关键的不是修了几朵云，而是：

1. 用户明确否定了不合适的新光影试验
2. 要求线程马上回滚失败方向
3. 只保留真正值钱的目标：
   - `CloudShadow` 穿场
   - 场景级缓存
   - 死云 / 孤儿云清理
   - 后续再由存档线程正式接存档

这说明用户不是“谁动了我就看结果”，而是会明确决定：

- 什么该继续
- 什么必须止损
- 什么要移交给别线程

### 4.7 `UnityMCP -> CLI / AI治理 / 项目文档`

这三条看起来是工具线，但实际上非常能体现用户这周的主控强度。

用户在这三条线上的关键推动包括：

## 5. 本周用户拍板过的关键设计口径，建议正式写进项目总文档

这些不是临时聊天判断，而是已经反复影响线程走向的正式产品 / 结构口径：

### 5.1 `Town` 是承接层，不只是剧情说明位

- 不再把 `Town` 只当边界说明位
- 要让它能承接 Day1 后续生活面
- 但不能假装已经全部 live-ready

### 5.2 `Home` 必须被纳入正式场景体系

- 不是屋内占位房间
- 必须有 door contract
- 必须与 `Primary` 连起来
- 要继续走向真正可承接的室内场景

### 5.3 玩家走向跨场景唯一实例

- 不再长期接受“每个场景一只玩家”的结构
- 背包 / Hotbar / 工具态 / 入场锚点 / 朝向都要围绕唯一玩家重构

### 5.4 相机不能“一股脑全持久化”

用户不是简单说“相机也持久化吧”，而是推动团队去区分：

- 哪些是 global service
- 哪些仍应属于 scene-local 相机配置

这条判断很重要，因为它避免项目为了图省事把场景内 camera 配置全做成常驻混乱链。

### 5.5 `Home` 镜头最终改判为固定构图、黑底、尊重手摆

这不是技术细节，而是明确的玩家感知设计：

- 不跟随
- 画面居中构图靠手摆
- 黑底
- 镜头大小、位置由场景表现决定

### 5.6 UI 结构不是“全部全局化”，而是“核心 UI 全局 + 场景扩展本地”

这条口径已经直接影响了 `Home` 与 `Town/Home` 边界 HUD 的推进方式，建议进入正式策划案 / 交互系统卷。

### 5.7 Day1 的 NPC 方向已改判为“驻村常驻化”

这会直接影响：

- 场景承接
- NPC resident root / group root
- director 设计
- formal / resident fallback
- 后续日常与关系成长的消费方式

### 5.8 红错治理的正式口径是“高频责任簇自检”，不是“每两行验一次”，也不是“整轮最后再一起编”

这条是项目级开发习惯改变，建议写进 AI 治理与工具链卷。
