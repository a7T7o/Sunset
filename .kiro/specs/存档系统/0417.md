1. 先读本文件

   - 两者都必须给玩家可见提示
7. 当前工作台“半成品制作/中途产线”不扩成单独存档功能。
8. 技能经验未实装，不纳入当前存档范围。
9. 当前最严重问题不是 save UI，而是：
   - 跨场景 world-state
   - 背包 / toolbar / 选槽 / 输入恢复
   - load / restart / Day1 恢复 contract

## Part A｜语义梳理

这部分只回答一个问题：

`现在这套存档系统“应该怎样”才算正确。`

### A1. 正式存档的正确语义

正式存档不是“抄当前屏幕长什么样”，而是：

`把玩家在该时刻应当拥有的 canonical 进度状态存下来。`

它至少应该覆盖：

1. 玩家自身
   - 所在场景
   - 世界坐标 / 朝向
   - 背包 / 装备 / 当前可用物品
2. 世界状态
   - 箱子内容
   - 农地状态
   - 作物状态
   - 树 / 石头 / 掉落物等运行态对象
3. 叙事进度
   - `StoryPhase`
   - 已消费对话
   - Day1 关键导演推进态
4. 时间与门禁
   - 当前受控时钟
   - free-time / day-end / reminder / dinner 等恢复语义

正式存档不应直接保存的内容：

1. 某个 UI 当前正播到第几帧
2. PromptOverlay 当前 alpha 或某条瞬时文本
3. 某个 modal 当下壳体是否刚好打开
4. 某个剧情演员恰好站在过渡中的那一帧位置

### A2. 切场连续性的正确语义

切场连续性不是正式存档的替代品，而是：

`在玩家没有主动读档/重开的前提下，保证离开场景再回来时，这个场景的世界状态还能接上。`

它至少应该覆盖当前会被玩家明显感知的 world-state：

1. 箱子内容
2. 农地耕地 / 浇水 / 可种植状态
3. 作物实例
4. 掉落物
5. 树 / 石头等可采集对象

正确理解：

- 正式存档 = 长期进度真值
- 切场连续性 = 场景在本轮游玩内的离场后续接

二者不能再是“两套覆盖范围不同、互相打架的半系统”。

### A3. 重开 / 默认存档的正确语义

`默认存档` 当前应被视为：

`一个真实可读写、但受保护的默认槽位。`

它必须满足：

1. `F5` 保存到它
2. `F9` 从它读取
3. 可以复制，但不能粘贴 / 删除 / 走普通覆盖入口
4. 左侧摘要与普通存档同模板，不再单独做“开局介绍卡”

`重新开始` 当前应被视为：

`清掉当前游玩进度，并恢复到游戏定义的 Town 原生入口 / 进村开局。`

它必须满足：

1. 不读取旧的临时 baseline 壳
2. 不保留上轮游玩残留的背包 / 箱子 / 农地 / UI / 输入门控
3. 不把“上一次场景切换缓存”带进新开局
4. 不把 stale prompt / stale modal / stale dialogue / stale pause 栈带回来

### A4. UI 与事实源的正确语义

当前必须明确区分：

1. 真实事实源
   - 背包内容
   - 装备内容
   - 当前可操作槽位
   - 场景 world-state
2. 显示层
   - 背包高亮
   - toolbar 高亮
   - 箱子面板视觉选中态
   - 提示与弹窗

正确原则：

`显示层可以重建，但不能反过来改写真实事实源。`

也就是说：

1. toolbar 高亮不是唯一真值
2. 背包面板当前高亮也不是唯一真值
3. 箱子面板视觉选中态只是 UI 表现，不该成为跨场景真源
4. 真正的“当前可用槽位 / 当前使用来源”必须有统一真值

### A5. Day1 恢复的正确语义

这部分不由存档线程自己发明，必须继承 spring-day1 已定义的整条 contract。

当前正确口径是：

`save / load / restart / restore 必须把 Day1 恢复到玩家视角下应看到的 canonical state，再交还给 day1 runtime 自己继续接管。`

这意味着：

1. 相位必须正确
2. 已消费对话必须正确
3. stale UI / stale dialogue / stale modal / stale pause 必须清干净
4. 恢复后要让 day1 runtime 自己重新建：
   - task
   - prompt
   - gate
   - dinner / reminder / free-time / sleep 压力态

## Part B｜已知现状总图

这部分只讲：

`现在系统实际上坏在哪里。`

### B1. 已证实问题

#### 1. 正式存档目前不是“全世界统一入盘”

当前真实存档样本表明：

1. 当前所在场景的 world-state 更容易进档
2. 未加载场景的 runtime world-state 不会稳定一起进来

玩家体感：

- 我在 A 场景改了东西
- 去 B 场景存档
- 再回 A，像没改过

#### 2. 切场连续性当前不覆盖箱子 / 农地 / 作物

当前切场 continuity 只明显覆盖了少量 world object。

没有稳定覆盖：

1. 箱子
2. 农地
3. 作物

玩家体感：

- 离开场景前还对
- 回来就回档

#### 3. 箱子内容会复活

这条已经是铁证。

当前箱子在重新进场后，会重新按场景预设内容初始化。

玩家体感：

1. 箱子拿空后又满了
2. 隔天回去又恢复了
3. 场景内看似保存，跨场景或跨天就像没保存

#### 4. 农地 / 浇水 / 作物离场会丢

这不是单纯显示问题，而是：

1. 当前现场有变化
2. 但离场 continuity 没把它们续住

玩家体感：

1. 刚种下去
2. 刚浇完水
3. 切场回来像没种过

#### 5. 背包与 toolbar 只保住了“高亮”，没保住“真实操作来源”

跨场景后，系统只记得：

- 当前亮的是哪一格 hotbar

却没有稳定记住：

- 实际该从背包哪一格取种子 / 可放置物 / 当前真实操作槽

玩家体感：

1. 看起来还选着
2. 但实际用不出来
3. 种子、工具、放置都像不认了

#### 6. 重开 / 切场时，真实选槽又会被强压回 hotbar

这会进一步放大上面的分裂。

玩家体感：

1. 我之前在背包下排选的格子失效了
2. 系统偷偷又按快捷栏第一排来算
3. 高亮和实际使用来源彻底分家

#### 7. 剧情开始时只关箱子，不关背包；但输入门控又把“任何面板开着”都算阻塞

这会制造一种非常恶心的坏相：

1. 剧情接管前背包开着
2. 剧情后系统仍觉得“有 UI 占着输入”
3. 玩家感觉整套移动、切栏、用工具都像卡死

#### 8. 持久 UI 壳体跨场景活着，但 runtime 绑定对象不稳定

UI 不一定会消失，但它绑的对象可能已经不是当前该绑的那一个。

玩家体感：

1. 界面还在
2. 按钮也在
3. 高亮也还像活着
4. 但内容、操作、刷新、交互像假的

### B2. 高概率问题

#### 1. ToolbarUI 失活再激活后，可能不再继续听库存 / 选中广播

玩家体感：

1. 图标不刷新
2. 高亮不跟
3. 数量不动
4. 看起来像死掉了

#### 2. 背包面板和箱子面板重新打开时，会把视觉选中强行拉回 hotbar

玩家体感：

1. 我刚才明明点的是别的格
2. 重新打开面板后，高亮又跳回第一排对应格

#### 3. UI 重绑大量依赖“场上先找到哪个就先绑哪个”的兜底逻辑

这类问题不一定直接爆红，但最容易造成：

1. 绑错对象
2. 状态串场
3. 看着活着，行为不对

### B3. 已有基线，但不能误判为“整体已完成”

下列内容已经做过基础落地或清理，但仍需要放进最终回归保护：

1. `5_Settings` 内的存档 UI 基础框架已存在
2. 默认开局旧 baseline 壳已清理
3. `CloudShadowManager` 的正式存档接线已做过接入
4. Day1 restore hygiene 已经从“只有 story snapshot”推进到“load/restart 前先统一清场”
5. 部分 Inventory / toolbar / workbench rebind 合同已经补过测试护栏

正确理解：

`这些不是无效工作，但也绝不等于当前整条存档线已经闭环。`

1. 真正操作来源只有一个真值

1. 剧情开始和结束后，不再留下卡死型 stale gate
2. 切场 / 重开后，工具、放置、移动、切栏恢复一致

目标：

1. 严格按 spring-day1 已定义好的 contract 落地
2. 不越权去修 day1 own 的剧情 staging

   - 读档后 runtime 能继续
   - 重新开始后不会带旧壳

目标：

1. `5_Settings` 内存档功能继续可用
2. 读档 / 存档 / 复制 / 粘贴 / 覆盖 / 删除 / 重新开始 / 默认档语义不回归
3. 快速存档 / 快速读档提示正常

### R9｜交叉系统回归

1. 工作台材料识别
2. 箱子与背包之间转移后排序 / 刷新

### R10｜打包前 smoke 与可外发性底线

## Part E｜执行策略

### 总顺序

当前推荐顺序不是“想到哪修哪”，而是：

1. 先保测试能力
2. 再统一 world-state
3. 再收 Day1 合同
4. 最后做 build smoke

### 推荐执行顺序

#### Phase 0｜文档与基线冻结

目标：

1. 建立 `0417.md`
2. 冻结当前语义
3. 冻结当前问题树
4. 冻结本轮非目标

#### Phase 1｜恢复玩家可测试性

优先收：

1. 背包 / hotbar / toolbar 真值统一
2. 剧情 / 重开 / 切场后的输入门控清理
3. 面板重开后的错误视觉选中态

原因：

如果玩家连背包、工具、种子、toolbar 都没法稳定用，后面所有世界状态测试都会被噪音污染。

#### Phase 2｜收 world-state 主链

优先收：

1. 箱子
2. 农地
3. 作物
4. 掉落物 / 树 / 石头与 off-scene 合同对齐

原因：

这是当前“跨场景像回档”的真正主因。

#### Phase 3｜收 save / load / restart / Day1 contract

优先收：

1. stale UI / stale dialogue / stale gate 清理
2. 整条 Day1 phase canonical restore
3. free-time / dinner / reminder / sleep 压力态 re-entry

#### Phase 4｜收回归与 packaged smoke

优先收：

1. `5_Settings` 存档 UI 回归
2. 工作台 / 箱子 / 背包联动回归
3. build smoke
