**版本**: v0.3

## 1. 使用说明

本文件不是终稿，而是迭代式总览文档。

编写策略：
- 先写当前已确认内容
- 后续随着代码和文档阅读推进持续修正
- 如果旧结论被推翻，优先改正文，不保留误导性描述

阅读本文件时请始终记住：
- `.kiro/steering/` 是规则真源
- `.kiro/specs/` 是活跃工作区真源
- `History/` 是交接索引层
- `Docx/` 是历史设计与总结沉淀层

---

## 2. 项目一句话概述

`Sunset` 是一个基于 Unity 6 的 2D 农场模拟游戏项目，当前已落地时间、季节、库存、放置、箱子、存档、导航等大量基础系统，同时仍在推进农田修补、对话验证和文档治理线。

## 3. 文档源优先级

### 3.1 当前确认的优先级

1. 用户当前指令
2. `AGENTS.md`
3. `.kiro/steering/`
4. `.kiro/specs/` 活跃工作区文档
5. `History/` 交接索引与交接稿
6. `Docx/` 历史规划、总结、完整文档

### 3.2 当前已知注意点

- `.kiro/specs/README.md` 很有用，但目录映射带有旧命名痕迹，只能作为总表索引，不能直接当最新状态。
- `Docx/分类/全局/000_项目全局文档.md` 与 `Docx/分类/全局/全局功能需求总结文档.md` 信息密度很高，但属于历史快照，需要继续用当前代码和 `memory.md` 核对。
- `.kiro/steering/archive/progress.md` 的进度快照停在 2026-02-06，不能直接当“今天的真实完成度”。
- `物品放置系统/memory.md` 很有价值，但保留了 `PlacementManagerV3`、`PlacementValidatorV3` 的历史命名；当前代码文件名已经回到无 `V3` 后缀版本。
- `制作台系统/memory.md` 与 `Docx/Plan/*` 更多反映 2025-12 的设计与实现停点，路径大量仍写成 `Assets/Scripts/*`，不能直接当当前目录现实。
- `UI系统/memory.md` 价值主要在“背包交互教训、回滚原因、互斥规则、不要再踩的坑”，而不是单纯拿来当今天的 UI 结构索引。

## 4. 推荐阅读顺序

### 4.1 10 分钟速览

- `.kiro/steering/README.md`
- `.kiro/steering/archive/product.md`
- `.kiro/steering/archive/structure.md`
- `.kiro/steering/archive/progress.md`

### 4.2 1 小时建立全局理解

- 继续阅读 `.kiro/steering/archive/tech.md`
- 阅读 `Docx/分类/全局/000_项目全局文档.md`
- 阅读 `Docx/分类/全局/全局功能需求总结文档.md`
- 阅读 `History/2026.03.07-Claude-Cli-历史会话交接/总索引.md`
- 补读 `存档系统/memory.md`、`箱子系统/memory.md`、`物品放置系统/memory.md`

### 4.3 深入接手

- 按当前主线读取对应 `.kiro/specs/*/memory.md`
- 回头补读相关 `Docx/分类/*/000_系统完整文档.md`
- 最后核对代码和资源
- 如果是库存 / UI / 箱子链，建议把 `UI系统/memory.md`、`箱子系统/memory.md`、`存档系统/memory.md` 连读
- 如果是放置 / 农田链，建议把 `物品放置系统/memory.md`、`农田系统/memory.md` 连读

### 4.4 如果你是按任务类型接手

- 想继续业务修复：先读对应工作区 `memory.md`，再读 `01_系统架构与代码全景.md`
- 想继续剧情/对话：先读 `spring-day1-implementation/memory.md`，再看 `Assets/000_Scenes/DialogueValidation.unity`
- 想继续治理/文档：先读 `Codex迁移与规划/tasks.md`，再继续维护 `.kiro/about/`
- 想继续存档 / 动态重建：先读 `存档系统/memory.md`，再看 `SaveManager.cs`、`DynamicObjectFactory.cs`、`PrefabDatabase.asset`
- 想继续箱子 / 背包联动：先读 `箱子系统/memory.md` 与 `UI系统/memory.md`，再看 `ChestController.cs`、`BoxPanelUI.cs`、`PackagePanelTabsUI.cs`
- 想继续制作台 / 配方：先读 `制作台系统/memory.md`，再确认 `CraftingService.cs`、`CraftingPanel.cs` 与实际配方资产是否齐备

## 7. 当前已确认的漂移与谨慎点

- `.kiro/specs/README.md` 可以作为“旧索引”，但不能代表当前中文工作区全貌。
- `.kiro/steering/archive/progress.md` 的完成度快照已经滞后，当前状态必须联读各工作区 `memory.md`。
- 存档体系在文档上以 `PrefabDatabase` 为主，但代码与资产层仍保留 `PrefabRegistry` 兼容回退。
- `Packages/manifest.json` 里已经有 `com.coplaydev.unity-mcp`，但这不等于当前 Unity MCP 链路已经稳定可用。
- 历史文档常写 `Assets/Scripts/`，而当前真实代码主目录是 `Assets/YYY_Scripts/`，同时旧 `Assets/Scripts/Utils/` 仍残留。
- 旧 `Docx` 里背包常写成“20 格主背包 + 8 格快捷栏”，当前真实代码已经是 `36` 格背包，第一行 `12` 格与 Hotbar 直接映射。
- `物品放置系统` 的历史文档把当前版本叫作 `V3`，但 live 代码文件名已经回到 `PlacementManager`、`PlacementValidator`、`PlacementPreview`。
- `制作台系统` 的设计文档可帮助理解目标，但是否存在足够的 `RecipeData` 资产配置，目前仍需继续核对，不能先假设“数据已完整”。
- `DialogueValidation.unity` 当前仍是验证场景，不应误解为完整剧情场景；相关字体、布局、引用问题仍在验收链中。
